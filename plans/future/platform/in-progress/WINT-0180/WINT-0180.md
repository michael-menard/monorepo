# WINT-0180: Define Examples + Negative Examples Framework

## Context

The autonomous development workflow currently relies on agents making decisions based on:
- Static rules in agent instruction files
- KB precedent queries (when available)
- RAPID framework for gray areas (expert-intelligence.md)
- 5-tier decision classification (decision-handling.md)

However, there's no systematic framework for **examples** and **negative examples** that agents can reference when:
- Determining which pattern to follow among multiple valid options
- Identifying anti-patterns to avoid
- Learning from past successful/unsuccessful decisions
- Building intuition for gray area decisions

**Problem:**
Agents lack structured access to:
1. **Positive examples** - "This is how we do X well"
2. **Negative examples** - "This is what NOT to do when facing Y"
3. **Context for examples** - When/why an example applies
4. **Outcome tracking** - Whether following an example led to success

This forces repeated KB queries, increases token usage, and creates inconsistent decisions when agents encounter similar situations.

**Current State:**
- 100+ agents in `.claude/agents/` with varying sophistication levels
- Existing inline examples in `decision-handling.md` and `expert-intelligence.md`
- WINT database schemas support agent decisions (`wint.agentDecisions` table)
- KB integration pattern exists but no structured example retrieval

**Established Patterns:**
- YAML for structured agent metadata (frontmatter in `.agent.md` files)
- Zod schemas for all data structures (CLAUDE.md requirement)
- Inline examples in agent files (see decision-handling.md § Examples)
- Markdown-based documentation (agent instructions use `.md` format)
- KB-first queries (query before deciding from KB-AGENT-INTEGRATION.md)

---

## Goal

Define a framework for capturing, storing, and retrieving examples that:
1. **Integrates with existing systems** - Works with decision-handling.md, autonomy-tiers.md, and wint.agentDecisions
2. **Supports downstream stories** - Enables WINT-0190 (Patch Queue), WINT-0200 (User Flows), WINT-0210 (Role Pack Templates)
3. **Tracks effectiveness** - Links examples to outcomes for ML training
4. **Scales across agents** - Reusable by 100+ existing agents

**Success Criteria:**
- Zod schema defined for ExampleEntry with all required fields
- Storage strategy chosen (database table, filesystem YAML, or hybrid)
- Integration points documented in existing agent instruction files
- Lifecycle defined (created → validated → deprecated)
- Query pattern specified with function signatures
- Outcome tracking schema defined to measure example effectiveness
- Migration path created for existing inline examples

## Architecture Notes

### Storage Strategy Trade-offs

**Option A: WINT Database Table** (`wint.examples`)
- **Pros**: Queryable, versioned, supports cross-project access, ML pipeline ready
- **Cons**: Requires schema migration, adds database dependency
- **Best for**: Common examples shared across multiple agents

**Option B: Filesystem YAML/JSON**
- **Pros**: Simple, no migration, easy to edit, version control friendly
- **Cons**: Harder to query cross-project, no relational integrity
- **Best for**: Quick iteration, agent-local examples

**Option C: Hybrid (RECOMMENDED)**
- **Common examples**: WINT database table `wint.examples`
- **Agent-specific examples**: Inline YAML in `.agent.md` frontmatter or adjacent files
- **Pros**: Balance of queryability and simplicity
- **Cons**: Two storage mechanisms to maintain

### Integration Points

**decision-handling.md:**
- Add example query step to decision tiers 2-4
- Tier 1 (Documented) - No examples needed
- Tier 2 (Pattern Match) - Query examples by category
- Tier 3 (Judgment Call) - Query examples by scenario
- Tier 4 (Expert) - Query examples + precedents
- Tier 5 (Escalate) - No examples (escalate instead)

**KB-AGENT-INTEGRATION.md:**
- Extend precedent query pattern for examples
- Add example-specific query functions
- Document example vs. precedent differentiation

**wint.agentDecisions:**
- Add `example_id` foreign key column
- Track which example influenced each decision
- Enable outcome correlation for effectiveness metrics

### Schema Evolution Strategy

**Versioning:**
- Use semantic versioning (semver) for schema
- `schema_version` field in ExampleEntry (e.g., "1.0.0")
- Major version: breaking changes (field removal, type changes)
- Minor version: additive changes (new optional fields)
- Patch version: documentation/validation changes

**Backward Compatibility:**
- v1 examples load with v2 schema via default values
- Migration scripts provided for major version changes
- Deprecation warnings for old schema versions
