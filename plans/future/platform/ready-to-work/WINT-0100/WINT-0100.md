# WINT-0100: Create Context Cache MCP Tools

## Context

**What exists today:**

WINT-0030 (completed as part of WINT-0010) established the database schema for context cache management with three tables: `contextPacks`, `contextSessions`, and `contextCacheHits`. These tables enable token optimization across agent invocations by caching frequently-used context (codebase structure, story metadata, architecture decisions, lessons learned, etc.).

The context cache schema is deployed in the WINT schema namespace at `packages/backend/database-schema/src/schema/wint.ts` (lines 467-587) via migration `0015_messy_sugar_man.sql`. The schema includes:

- **contextPacks**: Stores cached content with pack type, key, JSONB content, version, TTL, and hit tracking
- **contextSessions**: Tracks agent sessions with token metrics (input, output, cached tokens saved)
- **contextCacheHits**: Join table tracking which packs were used by which sessions

A reference MCP server implementation exists at `apps/api/knowledge-base/src/mcp-server/` with established patterns for tool registration, validation, error handling, correlation IDs, and performance measurement. This server provides 20+ knowledge base tools using the MCP SDK v1.26.0 with stdio transport.

The `@repo/db` client package provides access to the WINT schema with auto-generated Zod schemas via drizzle-zod.

**The problem:**

Without context cache MCP tools:
- Agents cannot read/write context packs during workflow execution
- No way to track which context was used by which agent session
- Token savings from caching are invisible to agents
- Cache expiration must be managed manually
- Dependent stories (WINT-2030, 2040, 2050, 2060) are blocked - they need these tools to populate caches
- Cache hit rate optimization is impossible without query tools

**Why this matters:**

Workflow intelligence (WINT) aims to reduce agent token costs by caching frequently-accessed context. The database schema exists but is inaccessible to agents without MCP tools. This story unblocks the entire context caching value stream by providing the API surface agents need to interact with the cache.

## Goal

Create an MCP server (or extend the existing Knowledge Base MCP server) that provides tools for context cache operations, enabling agents to:

1. **Manage context packs**: Read, write, delete, and list cached context by pack type and key
2. **Track sessions**: Record agent session start/end with token usage metrics
3. **Record cache hits**: Log when a pack is accessed during a session and calculate token savings
4. **Query analytics**: Calculate cache hit rates, aggregate token savings, identify expired packs
5. **Perform maintenance**: Manually expire packs, check cache health

The tools must follow established MCP patterns from the Knowledge Base server: Zod-first validation, correlation IDs, error sanitization, performance logging, and dependency injection.

## Non-Goals

**Explicitly out of scope:**

- Data seeding for context packs (separate stories: WINT-2030, 2040, 2050, 2060)
- Cache warming strategies (separate story: WINT-2070)
- Agent integration to use cache (separate story: WINT-2110)
- Cache pre-population from KB (depends on KBAR-0030 Story Sync Functions)
- Cache compression for large packs (future enhancement)
- Multi-tenancy support (single workflow system for now)
- Custom cache eviction policies (LRU/LFU - future enhancement)
- Cache analytics dashboard UI (backend tools only)
- Migration of existing workflow context to cache (future migration story)

**Protected features (do not modify):**

- Context cache database schema (`packages/backend/database-schema/src/schema/wint.ts` lines 467-587)
- Migration `0015_messy_sugar_man.sql`
- `@repo/db` client package API surface
- Knowledge Base MCP server (can reference as pattern, but avoid unnecessary changes)

**Deferred work:**

- Performance optimizations for large-scale cache (defer to performance story)
- Cache replication/backup strategies (defer to infrastructure story)
- Cache versioning conflict resolution beyond optimistic locking (sufficient for MVP)
- Pack content schema validation beyond JSONB type (future enhancement)

### Pack Versioning Strategy

**Optimistic Locking via Version Field:**

```sql
-- Write operation
UPDATE context_packs
SET content = ?, version = version + 1, updated_at = NOW()
WHERE pack_type = ? AND pack_key = ? AND version = ?
RETURNING *;

-- If 0 rows updated â†’ version conflict
-- Client retries with latest version
```

**Conflict Resolution:**
1. Read current pack (including version)
2. Attempt update with expected version
3. If conflict (0 rows updated), re-read latest version and retry
4. Max 3 retries, then fail with "Concurrent update conflict"

**Version Semantics:**
- Version starts at 1 on first write
- Increments on every update
- Never decrements
- Delete does not affect version (pack removed)

### Session Tracking Flow

**Session Lifecycle:**
1. **Start**: Agent calls `context_cache_create_session` â†’ record created with `startedAt = NOW()`
2. **Execution**: Agent reads packs, calls `context_cache_record_hit` for each â†’ hits recorded, pack hitCount incremented
3. **Token Updates**: Agent periodically calls `context_cache_update_session` with token deltas â†’ counters incremented
4. **End**: Agent calls `context_cache_end_session` â†’ `endedAt = NOW()`, final metrics logged

**Token Metrics:**
- `inputTokens`: Sum of all input tokens consumed during session
- `outputTokens`: Sum of all output tokens generated during session
- `cachedTokens`: Sum of tokens saved by using cached context instead of re-fetching

**Hit Recording:**
- Each time agent accesses a pack, record hit with `sessionId` + `packId`
- Optionally include `tokensSaved` (estimated tokens avoided by cache hit)
- Update pack's `hitCount` and `lastHitAt`

### Cache Expiration Strategy

**TTL Configuration:**
- `ttlHours = null` â†’ never expires (for stable content like architecture decisions)
- `ttlHours = 24` â†’ expires in 24 hours (for volatile content like codebase snapshots)
- `expiresAt` calculated as `NOW() + INTERVAL '{ttlHours} hours'`

**Recommended TTLs:**
- `codebase`: 24 hours (code changes frequently)
- `story`: 1 hour (active stories change during elaboration)
- `feature`: 168 hours (7 days, features stable once deployed)
- `epic`: 720 hours (30 days, epics rarely change)
- `architecture`: null (never expires, ADRs are immutable)
- `lessons_learned`: null (never expires, lessons accumulate)
- `test_patterns`: null (never expires, patterns stable)

**Expiration Cleanup:**
- Manual: `context_cache_expire_packs` tool (dry-run mode for preview)
- Future: Automated cron job (separate story, not in this scope)
- Index on `expiresAt` ensures cleanup queries are fast

**Expired Pack Behavior:**
- `context_cache_get_pack` on expired pack â†’ return not_found (treat as cache miss)
- `context_cache_list_packs` excludes expired packs by default (unless `includeExpired=true`)

### Error Handling Philosophy

**Fail Fast, Fail Clear:**
- Invalid input â†’ immediate validation error with clear message
- Missing dependencies (session, pack) â†’ not_found error with helpful hint
- DB errors â†’ log full stack trace, return sanitized generic error

**Error Sanitization:**
```typescript
function sanitizeError(error: unknown): string {
  if (error instanceof Error) {
    // Strip sensitive data (DB credentials, internal paths)
    return error.message.replace(/password=\S+/gi, 'password=***')
  }
  return 'Internal server error'

```

**Correlation IDs:**
- Every request gets UUID v4 correlation ID
- Included in all logs (info, warn, error)
- Returned to client in error responses for debugging

**MCPError Types:**
- `InvalidRequest`: Input validation failed, schema mismatch
- `NotFound`: Session or pack not found
- `Conflict`: Duplicate session, version mismatch
- `InternalError`: DB connection failed, unexpected error

## UI/UX Notes

**Note:** This story is backend-only (MCP server tools). No UI components. However, MCP tool naming and error messages affect developer UX in Claude Code.

### Tool Naming UX

**Consistent Prefix:**
- Use `context_cache_*` prefix for all tools (not `cc_*` or `cache_*`)
- Ensures tools group together in tool list
- Clear namespace separation from other MCP servers

**Verb-First Naming:**
- `context_cache_get_pack` (not `context_cache_pack_get`)
- `context_cache_write_pack` (not `context_cache_pack_write`)
- `context_cache_record_hit` (not `context_cache_hit_record`)
- Aligns with common CLI conventions (git get, git write, etc.)

**Clear Distinctions:**
- `list_packs` returns array (many packs)
- `get_pack` returns single pack or not_found
- `create_session` initializes new session (fails if exists)
- `update_session` modifies existing session (fails if not found)
- `end_session` finalizes session (sets endedAt, logs summary)
