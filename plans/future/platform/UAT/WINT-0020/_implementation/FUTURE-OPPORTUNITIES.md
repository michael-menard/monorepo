Non-MVP gaps and enhancements tracked for future iterations.

## Categories

### Edge Cases

**Artifact synchronization edge cases**:
- **Concurrent artifact updates**: Multiple agents updating same artifact simultaneously. Mitigated by checksum-based detection but no locking mechanism. Application-layer concern (WINT-0090).
- **Orphaned checksums**: Artifact deleted from filesystem but database record remains. Cleanup strategy deferred to KBAR-0030+ sync features.
- **Checksum collision**: SHA-256 collision probability astronomically low (2^-256) but theoretically possible. No mitigation planned - acceptable risk.

**Phase history edge cases**:
- **Phase loops**: Story enters same phase multiple times (e.g., execute → review → execute → review). Handled via `iteration` field but no loop detection. Enhancement for WINT-3xxx telemetry.
- **Overlapping phases**: Database allows concurrent phase entries (e.g., in "execute" and "review" simultaneously for different iterations). LangGraph prevents this at application layer.
- **Negative duration**: `duration_seconds` can be negative if `exited_at` < `entered_at` due to clock skew. Consider CHECK constraint or application validation.

**Metadata versioning edge cases**:
- **Version gaps**: Versions not sequential (e.g., v1, v2, v5). Currently allowed. Consider enforcing sequential versioning with CHECK constraint or trigger.
- **Metadata schema evolution**: Metadata structure changes over time. JSONB flexible but no validation. Enhancement: add optional JSON Schema validation field.
- **Large metadata snapshots**: No size limit on metadata_snapshot JSONB. Risk of exceeding PostgreSQL TOAST limit (1GB). Mitigation: application-layer size validation.

**Assignment edge cases**:
- **Assignment without completion**: Active assignment never marked completed. No timeout mechanism. Enhancement: add `timeout_at` timestamp and automated cleanup job.
- **Completed assignment with null completed_at**: CHECK constraint prevents this but worth edge case testing.
- **Phase mismatch**: Assignment phase doesn't match story's current phase. Allowed by design - enables pre-assignment for future phases.

**Blocker edge cases**:
- **Circular blockers**: Story A blocks Story B, Story B blocks Story A (via storyDependencies). Blocker table doesn't prevent this - graph-level concern (WINT-4xxx).
- **Blocker without resolution notes**: Resolved blocker with null resolution_notes. Allowed by design but CHECK constraint could enforce this.
- **Stale blockers**: Blocker marked resolved but story still blocked. Application-layer validation needed.

### Performance

**Query performance optimizations**:
- **Index on checksum field**: Currently no index on storyArtifacts.checksum. Add B-tree index for duplicate checksum detection queries. Low priority - checksums nearly unique.
- **Partial indexes for active records**: Add partial indexes for `WHERE resolved_at IS NULL` (storyBlockers) and `WHERE status = 'active'` (storyAssignments). Medium impact for active record queries.
- **Index on iteration field**: Add index on storyPhaseHistory.iteration for iteration-specific queries. Low priority unless queries filter by iteration frequently.
- **JSONB GIN indexes**: Add GIN indexes on metadata_snapshot for frequent JSONB queries (e.g., searching by tag). Deferred until query patterns emerge.

**Storage optimizations**:
- **Partition storyMetadataVersions by story_id**: For stories with 1000+ versions, partitioning improves query performance. Deferred to scale (100K+ stories).
- **Compression for JSONB fields**: PostgreSQL native JSONB compression via TOAST. No action needed unless storage becomes issue.
- **Archive old phase history**: Move phase history older than 90 days to archive table. Deferred to WINT-2xxx retention policy work.

**Write performance**:
- **Batch inserts for phase history**: LangGraph may insert many phase entries rapidly. Consider batching inserts. Application-layer optimization.
- **Async metadata versioning**: Metadata versions written on every story update. Consider async writes to reduce latency. Enhancement for high-throughput scenarios.

### Observability

**Monitoring enhancements**:
- **Slow query alerts**: Alert on storyArtifacts queries >100ms, phase history queries >50ms. Requires WINT-3xxx telemetry integration.
- **Index usage tracking**: Monitor pg_stat_user_indexes for unused indexes. Cleanup unused indexes quarterly.
- **Table growth alerts**: Alert when storyMetadataVersions exceeds 1M rows (retention policy needed).
- **Checksum mismatch detection**: Track frequency of checksum changes per artifact type. Indicates sync issues or high churn.

**Metrics to track**:
- Phase success/failure rates by phase type
- Average phase duration by story type
- Assignment completion time by assignee type
- Blocker resolution time by blocker type and severity
- Artifact sync frequency by artifact type

**Alerting rules**:
- Foreign key violations (indicates application bug)
- Unique constraint violations on (story_id, artifact_type) - duplicate upload attempts
- High blocker creation rate (>100/hour) - workflow issue
- Long-running active assignments (>24 hours) - stuck agents

### Additional Considerations

**Security**:
- **Artifact path injection**: `file_path` field accepts arbitrary strings. Validate paths at application layer to prevent directory traversal.
- **JSONB injection**: Metadata snapshot accepts arbitrary JSON. Sanitize before display to prevent XSS if rendered in UI.
- **Assignment spoofing**: `assignee_id` is text field, no FK to users table. Application layer must validate assignee identity.

**Compliance**:
- **Audit trail retention**: Metadata versions and phase history provide audit trail. Consider retention policy for compliance (7 years for some regulations).
- **PII in metadata**: Metadata snapshots may contain user info (changed_by field). Consider encryption or anonymization for GDPR compliance.

**Data migration**:
- **Backfill from filesystem**: WINT-1030 will populate storyArtifacts from existing YAML/MD files. Consider migration script performance (10K+ stories).
- **Version 0 convention**: First metadata version should be v1 or v0? Document convention for consistency.

**Developer experience**:
- **Seed data for development**: Provide seed SQL for common artifact types, phases, blockers. Enhances local development.
- **Type safety**: Zod schemas auto-generated but consider stricter validation for enums (e.g., artifactTypeEnum values in sync with filesystem conventions).
- **Documentation**: Generate entity relationship diagram (ERD) from schema for onboarding. Tool: dbdocs.io or SchemaSpy.

**Testing**:
- **Mutation testing**: Verify constraints by attempting to violate them. Example: insert artifact with same (story_id, artifact_type) twice.
- **Load testing**: Test cascade delete performance with story having 1000+ related records. Deferred to integration testing.
- **Migration testing**: Test migration rollback on staging before production deploy.
