## Context

The orchestrator currently persists workflow artifacts as YAML files. INFR-0110 creates the database schemas for the 4 core workflow artifact types (story, checkpoint, scope, plan). This story (INFR-0120) completes the artifact schema foundation by adding the 3 review/QA artifact types (evidence, review, qa-verify).

**Problem**: Review and QA artifacts stored only as YAML files are difficult to query for quality analysis. We cannot efficiently answer questions like:
- "Show me all evidence records with failed test results"
- "What are the most common QA issues across stories?"
- "Which review findings lead to the most rework?"
- "Generate a trend report on QA verification pass rates"

**Solution**: Create Postgres schemas for the 3 review/QA artifact types, enabling dual persistence (YAML for human readability + Postgres for querying). This completes the foundation that INFR-0020 (Artifact Writer/Reader Service) will use to sync artifacts bi-directionally.

**Split Scope**: This story (INFR-0120) handles the 3 review/QA artifacts that track implementation quality. Core workflow artifacts (story, checkpoint, scope, plan) are handled in INFR-0110.

## Architecture Notes

### Enum Reuse

**Context**: INFR-0110 defined `artifact_type_enum` with all 7 values upfront, including 'evidence', 'review', 'qa-verify'.

**Benefit**: No ALTER TYPE statement needed in this story. Migration only creates tables.

**Validation**: Test that enum values 'evidence', 'review', 'qa-verify' work correctly in foreign key constraints and queries.

### Schema Alignment Strategy

**Same as INFR-0110**:
- Use snake_case in Postgres (PostgreSQL convention)
- Use camelCase in TypeScript (JavaScript convention)
- Drizzle ORM handles transformation automatically via column aliases
- Document mapping in SCHEMA-REFERENCE.md (update existing doc)

**Example**:
```typescript
// Drizzle schema
export const evidenceArtifactsTable = pgTable('evidence_artifacts', {
  storyId: uuid('story_id').notNull(), // DB: story_id, TS: storyId
  acEvidence: jsonb('ac_evidence').$type<AcEvidence[]>(),
})

// Auto-generated Zod schema
const EvidenceArtifactInsertSchema = createInsertSchema(evidenceArtifactsTable)
// Type: { storyId: string, acEvidence: AcEvidence[], ... }
```

### JSONB Denormalization Trade-offs

**Same rationale as INFR-0110**:

1. **Co-location**: Evidence ACs, review findings, QA issues always loaded together with parent artifact
2. **Size**: Max ~20 evidence items per story, ~10 review findings → fits JSONB performance profile (<100KB)
3. **Query patterns**: Most queries filter by story_id, not individual evidence/finding fields
4. **Schema evolution**: Zod schemas can evolve without migration for new JSONB fields

**Decision for INFR-0120**: Use JSONB denormalization. Normalize later if query patterns require (tracked in FUTURE-OPPORTUNITIES.md for INFR-0020+).

### Foreign Key Cascade Behavior

**Same as INFR-0110**: Add `ON DELETE CASCADE` to foreign key constraints.

**Rationale**:
- When a story is deleted from `wint.stories`, all related artifacts (including evidence/review/qa-verify) should be automatically deleted
- Prevents orphaned artifact records
- Consistent with INFR-0110 behavior
