# Provider Adapters (OpenRouter/Ollama/Anthropic)

## Context

The orchestrator currently has a hybrid Ollama/Claude LLM provider system implemented in `packages/backend/orchestrator/src/config/llm-provider.ts`. This implementation successfully supports local Ollama models and external Claude models, but the architecture is tightly coupled to these two specific providers.

**Current state**:
- `llm-provider.ts` contains hardcoded logic for Ollama and Claude initialization
- Model selection uses provider detection in `model-assignments.ts`
- LangChain integration exists via `@langchain/ollama` package
- Provider caching pattern established with `ollamaInstanceCache`
- Availability checking pattern exists with `isOllamaAvailable()`

**Limitations**:
- Cannot access OpenRouter's catalog of 200+ models (Claude, GPT, Llama, Mistral, Qwen, DeepSeek)
- Cannot use Anthropic direct API (currently relying on Claude Code invocation)
- Adding new providers requires modifying core provider factory code
- No uniform interface for provider capabilities and configuration

**Business need**:
To optimize model selection for different orchestrator tasks (code review, planning, testing), we need access to a diverse model catalog with varying cost/quality/latency profiles. OpenRouter provides this diversity, while Anthropic direct API gives us official Claude access with better rate limits and features.

## Goal

Create a pluggable provider adapter system that:
1. Abstracts provider-specific initialization and configuration behind a common interface
2. Enables access to OpenRouter's 200+ model catalog
3. Supports Anthropic direct API for official Claude access
4. Refactors existing Ollama implementation to fit the adapter pattern
5. Maintains backward compatibility with existing orchestrator nodes
6. Provides foundation for MODL-0020 (Task Contracts & Model Selector) to choose providers dynamically

**Success criteria**:
- Developer can instantiate any provider by calling factory with model string like `openrouter/anthropic/claude-3-5-sonnet`
- Existing Ollama usage continues working without modification
- Provider instances are cached to avoid re-initialization
- All provider configuration uses Zod schemas (no TypeScript interfaces)
- Integration tests validate real provider connections

## Non-Goals

This story focuses ONLY on provider adapter infrastructure. The following are explicitly OUT OF SCOPE:

- **Model selection logic** - Choosing which model for which task belongs in MODL-0020 (Task Contracts & Model Selector)
- **Quality evaluation** - Measuring model output quality belongs in MODL-0030 (Quality Evaluator)
- **Multi-armed bandit** - Selection algorithms belong in MODL-0020
- **Leaderboard tracking** - Model performance metrics belong in MODL-0040 (Model Leaderboards)
- **Modifying existing nodes** - LangGraph nodes continue using current model assignment system until MODL-0020 integration
- **Frontend changes** - This is backend infrastructure only
- **Database schema** - No persistence layer required
- **Provider-specific features** - Advanced features like OpenRouter routing preferences, Anthropic streaming modes deferred to MODL-0020
- **Multi-model fallback chains** - Fallback logic (try model A, then B) deferred to MODL-0020
- **Health checks for new providers** - OpenRouter/Anthropic can use lazy initialization (fail on first use). Only Ollama preserves existing availability check for backward compatibility.

## Architecture Notes

### Provider Interface Design

All providers implement a common interface defined in `base.ts`:

```typescript
import { z } from 'zod'
import type { BaseChatModel } from '@langchain/core/language_models/chat_models'

const ProviderConfigSchema = z.object({
  apiKey: z.string().optional(),
  modelName: z.string(),
  timeout: z.number().default(30000),
})

type ProviderConfig = z.infer<typeof ProviderConfigSchema>

interface IProvider {
  initialize(config: ProviderConfig): Promise<BaseChatModel>
  checkAvailability(): Promise<boolean>
  getCacheKey(modelName: string): string
}
```

### Model Prefix Format

Standard format: `{provider}/{model-identifier}`

Examples:
- `openrouter/anthropic/claude-3-5-sonnet`
- `ollama/qwen2.5-coder:7b`
- `anthropic/claude-opus-4`

**Full provider names only** in MVP (no short aliases like `or/`). Aliases deferred to post-MVP based on usage patterns.

### Provider Factory Routing

```typescript
function getProvider(modelString: string): BaseChatModel {
  const [provider, ...modelParts] = modelString.split('/')
  const modelName = modelParts.join('/')

  switch (provider) {
    case 'openrouter':
      return OpenRouterProvider.initialize({ modelName, ... })
    case 'ollama':
      return OllamaProvider.initialize({ modelName, ... })
    case 'anthropic':
      return AnthropicProvider.initialize({ modelName, ... })
    default:
      throw new Error(`Unsupported provider: ${provider}. Supported: openrouter, ollama, anthropic`)
  }
}
```

### Caching Strategy

Extend existing `ollamaInstanceCache` pattern:

```typescript
const providerCache = new Map<string, BaseChatModel>()

function getOrCreateProvider(modelString: string): BaseChatModel {
  const cacheKey = getCacheKey(modelString)

  if (providerCache.has(cacheKey)) {
    logger.info('Provider cache hit', { modelString })
    return providerCache.get(cacheKey)!
  }

  logger.info('Provider cache miss, initializing', { modelString })
  const instance = getProvider(modelString)
  providerCache.set(cacheKey, instance)
  return instance
}
```

### Backward Compatibility Strategy

Refactor existing `llm-provider.ts` to delegate to new provider system:

**Before** (current):
```typescript
function getLLMProvider(model: string) {
  if (model.includes('ollama')) {
    return new ChatOllama(...)
  } else {
    return new ChatAnthropic(...)
  }
}
```

**After** (refactored):
```typescript
import { getProvider } from './providers'

function getLLMProvider(model: string) {
  return getProvider(model)
}
```

This maintains the same public API while routing to new provider system internally.

## Infrastructure Notes

### API Key Management

**Validation strategy**: Provider factory MUST log warning if required API keys (OPENROUTER_API_KEY, ANTHROPIC_API_KEY) are missing from environment, but MUST NOT crash the application. This allows developers to work with only Ollama locally.

If a provider is requested and its API key is missing, throw error at request time with clear message:

```typescript
if (!process.env.OPENROUTER_API_KEY) {
  throw new Error(
    'OPENROUTER_API_KEY environment variable required for OpenRouter provider. ' +
    'See docs/setup.md for API key setup instructions.'
  )
}
```

### OpenRouter-Specific Configuration

OpenRouter uses OpenAI-compatible API but requires:
- Base URL: `https://openrouter.ai/api/v1`
- Headers:
  - `HTTP-Referer`: Application identifier (optional but recommended)
  - `X-Title`: Application name (optional but recommended)

Implementation with `@langchain/openai`:

```typescript
import { ChatOpenAI } from '@langchain/openai'

const model = new ChatOpenAI({
  openAIApiKey: process.env.OPENROUTER_API_KEY,
  modelName: modelIdentifier,
  configuration: {
    baseURL: 'https://openrouter.ai/api/v1',
    defaultHeaders: {
      'HTTP-Referer': 'https://github.com/your-org/your-repo',
      'X-Title': 'LEGO MOC Instructions Orchestrator',
    },
  },
})
```

### Ollama Availability Checking

Preserve existing async availability pattern with timeout:

```typescript
async function checkAvailability(): Promise<boolean> {
  try {
    const response = await fetch('http://localhost:11434/api/tags', {
      signal: AbortSignal.timeout(5000), // 5 second timeout
    })
    return response.ok
  } catch (error) {
    logger.warn('Ollama availability check failed', { error })
    return false
  }
}
```
