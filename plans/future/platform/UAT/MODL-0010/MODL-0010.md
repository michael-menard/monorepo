# MODL-0010: Provider Adapters (OpenRouter/Ollama/Anthropic)

## Context

The orchestrator currently has a hybrid Ollama/Claude LLM provider system that supports local Ollama models and external Claude models. However, this is hard-coded to two providers and doesn't support:
- OpenRouter's 200+ model catalog (Claude, GPT, Llama, Mistral, Qwen, DeepSeek)
- Anthropic direct API (for official Claude access)
- Easy addition of new providers in the future

The existing `llm-provider.ts` and `model-assignments.ts` provide a foundation for provider abstraction but need to be refactored into a pluggable adapter pattern.

**Current Reality (2026-02-13)**:
- Hybrid Ollama/Claude provider in `packages/backend/orchestrator/src/config/llm-provider.ts`
- Model assignment system in `packages/backend/orchestrator/src/config/model-assignments.ts`
- LangChain packages already installed: `@langchain/core`, `@langchain/ollama`
- Provider caching pattern exists: `ollamaInstanceCache`
- Availability checking pattern exists: `isOllamaAvailable()`

## Goal

Create a unified provider adapter system that:
1. **Abstracts provider interface** - Base contract all providers implement
2. **Enables OpenRouter integration** - Access to 200+ models via unified API
3. **Maintains backward compatibility** - Existing Ollama integration continues working
4. **Adds Anthropic direct access** - Official Claude API as alternative to Claude Code
5. **Supports future providers** - Easy addition of Groq, Together AI, etc.

This provides the foundation for MODL-0020 (Task Contracts & Model Selector) to choose providers dynamically based on task requirements.

## Non-Goals

- **Model selector logic** - That's MODL-0020 (Task Contracts & Model Selector). This story only provides the provider adapters.
- **Quality evaluation** - That's MODL-0030 (Quality Evaluator). This story doesn't evaluate model outputs.
- **Multi-armed bandit** - That's MODL-0020. This story doesn't implement selection algorithms.
- **Leaderboard tracking** - That's MODL-0040 (Model Leaderboards). This story doesn't track metrics.
- **Modifying existing nodes** - Existing LangGraph nodes continue using current model assignment system. MODL-0020 will integrate the new provider system.
- **Frontend changes** - This is backend infrastructure only.
- **Database schema changes** - No persistence layer in this story.

**Protected Features** (from seed):
- Existing Ollama integration must continue working unchanged
- Current model assignment logic preserved until MODL-0020
- No breaking changes to LangGraph node interfaces

## Architecture Notes

### Provider Abstraction Pattern

```typescript
// Base interface (base.ts)
interface ILLMProvider {
  getModel(config: BaseProviderConfig): BaseChatModel
  checkAvailability(timeout?: number): Promise<boolean>
  getCachedInstance(configHash: string): BaseChatModel | null
  loadConfig(): ProviderConfig
}

// Factory pattern (index.ts)
export function getProviderForModel(modelName: string): ILLMProvider {
  const prefix = modelName.split('/')[0]
  switch (prefix) {
    case 'openrouter': return new OpenRouterProvider()
    case 'ollama': return new OllamaProvider()
    case 'anthropic': return new AnthropicProvider()
    default: throw new Error(`Unsupported provider: ${prefix}`)
  }
}
```

### Configuration Schema Pattern

```typescript
// Example: OpenRouter config (openrouter.ts)
import { z } from 'zod'

const OpenRouterConfigSchema = z.object({
  apiKey: z.string().min(1),
  modelName: z.string().min(1),
  baseURL: z.string().url().default('https://openrouter.ai/api/v1'),
  timeout: z.number().int().positive().default(30000)
})

type OpenRouterConfig = z.infer<typeof OpenRouterConfigSchema>
```

### Caching Strategy

Reuse existing `ollamaInstanceCache` pattern:
```typescript
const providerCache = new Map<string, BaseChatModel>()

function getCachedInstance(configHash: string): BaseChatModel | null {
  const cached = providerCache.get(configHash)
  if (cached) {
    logger.debug('Provider cache hit', { configHash })
  }
  return cached ?? null
}
```

### Implementation Notes

**Provider Selection Strategy (Gap #1 resolution)**:
- MVP uses explicit prefix-based routing: `openrouter/*`, `ollama/*`, `anthropic/*`
- Provider is determined by model name prefix (e.g., `openrouter/claude-3-5-sonnet` routes to OpenRouter adapter)
- When multiple providers support the same model (e.g., Claude via OpenRouter vs Anthropic direct), the caller chooses via prefix
- Advanced selection logic (cost, latency, availability-based routing) deferred to MODL-0020 (Task Contracts & Model Selector)
- This design intentionally keeps provider abstraction simple for MVP

**Cache Invalidation Strategy (Gap #2 resolution)**:
- MVP cache invalidation: process restart only
- Provider configs loaded at startup from environment variables
- Cached instances persist for lifetime of process
- This is acceptable because provider configurations rarely change mid-execution
- Advanced invalidation (hot-reload, file watchers) deferred to MODL-0020 per Future Opportunities Gap #4
- No memory leak risk: cache is bounded to number of unique provider configs (typically 3-5)
