# WINT-1020: Flatten Story Directories

## Context

Currently, story directories are organized in a lifecycle-based directory structure under each epic:

```
plans/future/{epic}/
  ├── backlog/{STORY-ID}/
  ├── elaboration/{STORY-ID}/
  ├── ready-to-work/{STORY-ID}/
  ├── in-progress/{STORY-ID}/
  ├── ready-for-qa/{STORY-ID}/
  └── UAT/{STORY-ID}/
```

This structure was designed for filesystem-based workflow management, but creates several problems:

1. **Duplicate story directories** - Stories appear in multiple lifecycle directories (e.g., BUGF-020 appears in `backlog/`, `elaboration/`, `ready-to-work/`, `in-progress/`, `ready-for-qa/`, and `UAT/`)
2. **Manual file movements** - Moving stories between lifecycle stages requires filesystem operations via `/story-move`
3. **Complex lookups** - Finding a story requires searching across 6 possible directories
4. **Database-driven workflow mismatch** - WINT-0020 (Story Management Tables) will introduce a database schema where story status is a field, not a directory location

The current directory-based lifecycle model does not scale for database-driven workflow management. We need to transition to a flat directory structure where story status is stored in the database (and story YAML frontmatter), not inferred from directory location.

**Grounded in Reality Baseline (2026-02-13)**:
- Orchestrator YAML artifacts exist at `packages/backend/orchestrator/src/artifacts/`
- StoryFileAdapter exists at `packages/backend/orchestrator/src/adapters/story-file-adapter.ts`
- Story v2 schema supports backward compatibility (status vs state fields)
- Platform stories index just bootstrapped (no active platform stories in progress)
- All production DB schemas are protected (not touched by this story)

## Goal

Flatten all story directories to a single location per epic, transitioning from directory-based lifecycle management to database-driven status tracking.

**Before**: `plans/future/{epic}/{lifecycle}/{STORY-ID}/`
**After**: `plans/future/{epic}/{STORY-ID}/`

Story status will be tracked in:
1. **Database** - `stories` table with `status` field (via WINT-0020)
2. **YAML frontmatter** - `status` field in `{STORY-ID}.yaml` or `{STORY-ID}.md`
3. **Stories index** - `stories.index.md` with status column (temporary, until WINT-1070 deprecates it)

This change is a prerequisite for WINT-1030 (Populate Story Status from Directories), which will:
1. Scan existing lifecycle directories to infer current story status
2. Write status to database and YAML frontmatter
3. Migrate stories to flat structure
4. Update commands to use database queries instead of directory scans

## Non-Goals

- **NOT updating commands** - `/story-status`, `/story-move`, `/story-update` will be updated in downstream stories (WINT-1040, WINT-1050, WINT-1060) after WINT-1030 populates the database
- **NOT creating database tables** - That's WINT-0020 (Story Management Tables)
- **NOT populating the database** - That's WINT-1030 (Populate Story Status from Directories)
- **NOT deprecating stories.index.md** - That's WINT-1070 (Deprecate stories.index.md)
- **NOT running migration on production epics in this story** - Migration should only be tested on isolated test epics during WINT-1020. Production migration happens in WINT-1030.
- **NOT modifying protected features**:
  - Database schemas in `packages/backend/database-schema/`
  - Knowledge base schemas
  - `@repo/db` client package API
  - Orchestrator artifact schemas (story schema can be updated, but not the artifact persistence system)

## Architecture Notes

### Migration Script Structure

```
migrate-flatten-stories.js
├── CLI flags: --epic, --dry-run, --execute
├── Phase 1: Discovery
│   ├── scanAllEpics() → find all lifecycle directories
│   └── buildInventory() → list all stories, detect duplicates
├── Phase 2: Validation
│   ├── validateStories() → read frontmatter, check schema
│   └── reportValidationErrors() → output validation-report.json
├── Phase 3: Dry Run
│   ├── generateMigrationPlan() → determine moves, status inference
│   └── detectCollisions() → check for target directory conflicts
├── Phase 4: Backup (if --execute)
│   └── createBackupTarball() → tar -czf plans-backup-{date}.tar.gz
├── Phase 5: Execute (if --execute)
│   ├── moveDirectories() → fs.rename() for atomic moves
│   └── updateFrontmatter() → add/update status field via StoryFileAdapter
└── Phase 6: Verify
    ├── verifyFlatStructure() → check no lifecycle dirs remain
    └── validateStatuses() → all stories have status field
```

### Lifecycle Directory Priority

When story exists in multiple directories (duplicates), choose most advanced:

```
UAT/                  → status: uat          (Priority 1 - highest)
ready-for-qa/         → status: ready-for-qa (Priority 2)
in-progress/          → status: in-progress  (Priority 3)
ready-to-work/        → status: ready-to-work (Priority 4)
elaboration/          → status: elaboration   (Priority 5)
backlog/              → status: backlog       (Priority 6 - lowest)
```

### Error Handling Strategy

| Error Type | Behavior | Recovery |
|------------|----------|----------|
| Malformed YAML frontmatter | Skip story, log error | Manual review after migration |
| Missing frontmatter | Create minimal frontmatter with status | Continue migration |
| Target directory collision | Halt migration, log collision | Manual resolution required |
| Validation failure | Skip story, continue with others | Report in validation-report.json |
| Mid-migration failure | Stop execution, preserve partial state | Rollback from backup tarball |
| Disk full | Fail fast before any writes | Check disk space in Phase 0 |

### Data Safety Measures

1. **Backup tarball** created before ANY write operations
2. **Dry-run mode** mandatory for preview-before-execute
3. **Atomic operations** using `fs.rename()` (not copy + delete)
4. **Validation** with Zod schemas before writing frontmatter
5. **Rollback mechanism** tested with simulated failures
6. **Audit trail** via comprehensive migration log
