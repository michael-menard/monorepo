# WINT-0060: Create Graph Relational Tables

## Context

WINT-0010 established the foundational WINT database schemas including stub definitions for graph relational tables. These tables are critical for tracking feature relationships, capabilities, and cohesion rules that enable the workflow intelligence system to reason about codebase structure and feature dependencies.

The existing graph relational tables (`features`, `capabilities`, `featureRelationships`, `cohesionRules`) are incomplete stubs that lack the full schema design needed for production use. The workflow intelligence system requires a complete, production-ready graph schema to:

- Track feature relationships and dependencies across the codebase
- Model high-level capabilities provided by features
- Define and enforce cohesion rules for package/feature organization
- Enable graph-based queries for dependency analysis and impact assessment

**Current State:**
- WINT-0010 created stub tables for 4 graph tables (lines 935-1099 in `wint.ts`)
- `featureRelationshipTypeEnum` defined with 5 relationship types
- Test pattern established in `wint-schema.test.ts` (100% coverage achieved)
- Drizzle ORM v0.44.3 and drizzle-zod installed and operational

**Problem:**
The stub tables lack production-ready fields, indexes, constraints, and relations needed for:
- Feature identification and location tracking
- Capability lifecycle management
- Self-referencing relationship modeling (graph structure)
- Cohesion rule enforcement with flexible JSONB conditions

## Architecture Notes

### Self-Referencing Foreign Keys

**Pattern:**
```typescript
export const featureRelationships = wintSchema.table('feature_relationships', {
  id: uuid('id').primaryKey().defaultRandom(),
  sourceFeatureId: uuid('source_feature_id')
    .notNull()
    .references(() => features.id, { onDelete: 'cascade' }),
  targetFeatureId: uuid('target_feature_id')
    .notNull()
    .references(() => features.id, { onDelete: 'cascade' }),
  // ...
})
```

**Key Points:**
- Use forward references `() => features.id` to avoid circular dependency
- `onDelete: 'cascade'` ensures orphaned relationships are cleaned up
- Supports graph structures (A → B, B → C, A → C, etc.)

### JSONB Type Safety

**Pattern:**
```typescript
type FeaturePatterns = {
  include: string[]
  exclude?: string[]
}

export const cohesionRules = wintSchema.table('cohesion_rules', {
  featurePatterns: jsonb('feature_patterns').$type<FeaturePatterns>(),
  // ...
})
```

**Key Points:**
- `.$type<T>()` provides compile-time type safety for JSONB fields
- Zod schemas provide runtime validation
- Flexible schema allows rule evolution without migration

### Composite Index Cardinality

**Ordering Strategy:**
- High cardinality column first → Low cardinality column second
- Example: `(packageName, featureType)` — packageName has more unique values
- PostgreSQL uses leftmost prefix for index optimization

### Drizzle Relations (Graph Traversal)

**Pattern:**
```typescript
export const featuresRelations = relations(features, ({ many }) => ({
  outgoingRelationships: many(featureRelationships, {
    relationName: 'source_feature',
  }),
  incomingRelationships: many(featureRelationships, {
    relationName: 'target_feature',
  }),
}))

export const featureRelationshipsRelations = relations(featureRelationships, ({ one }) => ({
  sourceFeature: one(features, {
    fields: [featureRelationships.sourceFeatureId],
    references: [features.id],
    relationName: 'source_feature',
  }),
  targetFeature: one(features, {
    fields: [featureRelationships.targetFeatureId],
    references: [features.id],
    relationName: 'target_feature',
  }),
}))
```

**Key Points:**
- `relationName` disambiguates outgoing vs incoming relationships
- Enables lazy loading: `db.query.features.findFirst({ with: { outgoingRelationships: true } })`
- Prevents N+1 query problems
