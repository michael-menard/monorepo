# TELE-0010: Docker Telemetry Stack — Validate and Complete Local Observability Foundation

## Context

The baseline confirms that Prometheus, Grafana, and OTel Collector services are already defined in `infra/compose.lego-app.yaml` (added as part of the `@repo/observability` package work). The `lego-api` exposes a `/metrics` endpoint via prom-client when `ENABLE_METRICS=true`. Grafana provisioning YAML for the Prometheus datasource and dashboard folder provider already exist in `infra/grafana/provisioning/`. The `telemetry.workflow_events` table (INFR-0040 dependency) is defined in `packages/backend/database-schema/src/schema/telemetry.ts` and the INFR-0040 story is in-QA.

**The implementation is 80-90% complete.** The telemetry stack has never been smoke-tested end-to-end as a coherent observability foundation. The remaining gaps are:

1. No Grafana dashboard folder provisioning — the existing `default.yaml` uses an empty folder name, so TELE-003 dashboards would land in the Grafana root
2. `ENABLE_METRICS=true` is not documented in `apps/api/lego-api/.env.example` — without it, Prometheus shows `lego-api` target as `DOWN`
3. Path inconsistency: `PLAN.md` references `apps/telemetry/dashboards/` but Grafana mounts `infra/grafana/dashboards/` — this must be resolved before TELE-003 attempts to create dashboards
4. No end-to-end smoke test has confirmed all services start cleanly and all scrape targets are `UP`

---

## Goal

Validate the existing compose stack works end-to-end. Close the three remaining gaps: (1) add Grafana dashboard folder provisioning for the "Workflow Telemetry" folder, (2) document the `ENABLE_METRICS=true` environment variable, and (3) decide and document the canonical dashboard storage path. The deliverable is a fully verified, documented, and foundation-ready local telemetry stack that TELE-002 and TELE-003 can build on immediately.

**Dashboard path decision (resolved):** `infra/grafana/dashboards/` is the canonical location. The `apps/telemetry/dashboards/` reference in PLAN.md is superseded. No new pnpm workspace package is needed.

**Tempo scope (resolved):** Tempo is deferred to a separate story. AC-10 is explicitly out of scope for TELE-0010.

## Architecture Notes

### Dashboard Path Resolution

The conflict between `PLAN.md` (`apps/telemetry/dashboards/`) and the compose mount (`infra/grafana/dashboards/`) is resolved as follows:

**Decision: Option A — `infra/grafana/dashboards/` is canonical.**

Rationale:
- The directory already exists and is mounted by the Grafana container
- Creating `apps/telemetry/` as a new pnpm workspace package adds unnecessary complexity
- `apps/telemetry/` is not in `pnpm-workspace.yaml` and has no TypeScript code
- `infra/` is the correct home for infrastructure-as-code artifacts (per ADR-002)

Downstream impact:
- TELE-003 will create dashboard JSON files in `infra/grafana/dashboards/workflow-telemetry/`
- PLAN.md reference to `apps/telemetry/dashboards/` is superseded by `infra/grafana/dashboards/README.md`

### Grafana Folder Provisioning Strategy

Update `infra/grafana/provisioning/dashboards/default.yaml` to set `foldersFromFilesStructure: true`. This causes Grafana to automatically create folders based on subdirectory names in `infra/grafana/dashboards/`. Create `infra/grafana/dashboards/workflow-telemetry/.gitkeep` to establish the folder. The Grafana UI will show a folder named "workflow-telemetry" (lowercase, hyphenated) unless a display name override is configured.

Alternatively, set `folder: 'Workflow Telemetry'` on the provider and keep the flat directory structure. Either approach satisfies AC-5.

### ENABLE_METRICS Environment Variable

The `lego-api` server at `apps/api/lego-api/server.ts:80` conditionally registers the Prometheus metrics endpoint:

```typescript
if (process.env.ENABLE_METRICS === 'true') {
  app.get('/metrics', createMetricsEndpoint(registry))
}
```

This variable must be set in the local development environment for Prometheus to successfully scrape metrics. It is not a Docker Compose variable — it is set in the host process environment or in `apps/api/lego-api/.env.local`.
