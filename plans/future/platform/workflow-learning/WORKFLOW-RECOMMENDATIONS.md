## High Priority

### 1. Code Stories Severely Underestimated — Apply 4-8x Multiplier

**Pattern**: Stories delivering executable TypeScript code exceed token estimates by 148-834%
**Evidence**: WKFL-004 (+834%), WKFL-010 (+184%), WKFL-007 (+148%)
**Occurrences**: 3 stories — meets significance threshold
**Avg Variance**: +389%

**Root Cause Analysis**:
- Current token estimates appear to capture only a subset of the full pipeline (often just one phase)
- Elaboration alone often exceeds the full story estimate before implementation starts
- WKFL-004 estimate was 30,000 tokens; elaboration phases alone cost 73,500 tokens
- WKFL-007 estimate was 45,000 tokens; elaboration phases alone cost 73,000 tokens

**Recommendation**:
Apply a 4-8x token multiplier for stories tagged with `executable-code`, `typescript`, or `cli`:
- Simple CLI commands or schemas: multiply estimate by 4x
- Stories with agents + schemas + commands: multiply estimate by 6x
- Stories with complex algorithms (multi-phase, statistical): multiply estimate by 8x

Story type heuristics:
- Infrastructure/documentation-only (WKFL-001, WKFL-006): estimates accurate, no adjustment needed
- Agent + command + schema (WKFL-004, WKFL-010): 4-6x multiplier
- Complex algorithm + multi-phase (WKFL-007): 8x multiplier

**Proposed Story**: Create a token estimation calibration guide in `.claude/docs/token-estimation.md`
with per-story-type budgets based on this batch data.

### 2. KB and Task Tools Frequently Unavailable — Establish Deferred Pattern as Standard

**Pattern**: KB persistence and Task tool unavailability occurred in 4+ stories, requiring workarounds
**Evidence**: WKFL-002, WKFL-003, WKFL-008, WKFL-010 all note "KB tools unavailable — deferred write pattern" or "Task tool unavailable — workers executed sequentially"
**Occurrences**: 4 stories — meets significance threshold
**Correlation Rate**: 4/9 = 44% of stories affected

**Root Cause Analysis**:
- Both tools (`kb_*` MCP tools and the Task spawning tool) appear to be session-dependent
- When unavailable, agents correctly fall back to deferred writes (DEFERRED-KB-WRITES.yaml) and sequential execution
- However, the fallback patterns are re-discovered independently in each story rather than being standardized

**Recommendation**:
1. Make the deferred write fallback (DEFERRED-KB-WRITES.yaml) the documented first-class pattern, not a fallback
2. Add a Phase 0 check in all leaders for tool availability and branch to the appropriate path upfront
3. Create a `DEFERRED-KB-WRITES.yaml` template in `.claude/schemas/` with standard structure
4. Add a mechanism to batch-process deferred writes at session start

**Proposed Story**: WKFL-011 or equivalent — Standardize deferred KB write pattern with automated batch processing at session init.

## Medium Priority

### 3. Review Waived for Documentation-Only Stories — Formalize Waiver Criteria

**Pattern**: Review was waived or simplified for 3 stories delivering only agent/schema/markdown files
**Evidence**: WKFL-001 (review waived — infra story), WKFL-006 (review waived — docs only), WKFL-008 (review notes info-only findings with no blocking)
**Occurrences**: 3 stories — meets significance threshold

**Root Cause Analysis**:
- Agent/schema/command files cannot be linted or type-checked in the standard pipeline
- E2E gate exemption is consistently applied to these stories
- The review phase consumed tokens but produced no actionable findings for documentation stories

**Recommendation**:
1. Add a story type field to story.yaml: `story_type: code | documentation | schema | mixed`
2. For `documentation` and `schema` type stories, auto-waive the standard code review and substitute a documentation quality check (formatting, schema validity, cross-reference verification)
3. Update the review leader to read `story_type` and branch to the appropriate review workflow
4. Estimate token savings: ~20,000-40,000 tokens per documentation story (skip dead-weight review phases)

### 4. Token Logs are Inconsistent Across Stories — Standardize Coverage

**Pattern**: TOKEN-LOG.md files capture different phase subsets across stories, making cross-story comparison unreliable
**Evidence**: Across all 9 stories analyzed:
- WKFL-001: elab + QA phases only
- WKFL-002: PM generation only
- WKFL-003: PM validation path only
- WKFL-004: all major phases (elab + impl + QA)
- WKFL-005: QA phases only
- WKFL-006: elab + planning only
- WKFL-007: elab + QA only (missing implementation)
- WKFL-008: PM only
- WKFL-009: elab-setup only
- WKFL-010: PM + implementation + proof (most complete)
**Occurrences**: 9/9 stories — pervasive issue

**Root Cause Analysis**:
- Different agents log to different TOKEN-LOG.md files (_implementation/ vs root vs _pm/)
- Implementation phase logs are rarely captured
- There is no canonical "phase complete → log tokens" trigger

**Recommendation**:
1. Mandate that every phase leader appends to a single TOKEN-LOG.md in `_implementation/` at phase completion
2. Define a standard log row format: `| timestamp | phase | agent | input | output | total | cumulative |`
3. Add a token-logging requirement to the CHECKPOINT.yaml schema (must include cumulative_tokens field)
4. The `dev-documentation-leader` already has a hook for writing OUTCOME.yaml — wire token aggregation into that step

### 5. QA Elaboration Sometimes Adds ACs Mid-Flight — Create AC Freeze Policy

**Pattern**: QA found AC-scope gaps in WKFL-008, adding 2 new ACs (AC-6, AC-7) as part of a CONDITIONAL PASS
**Evidence**: WKFL-008 QA notes: "mvp_gaps_resolved: added AC-6 (insufficient data guard), added AC-7 (traffic routing integration point)"
**Occurrences**: 1 story — below threshold for KB logging, but notable for single high-priority story

**Root Cause Analysis**:
- Elaboration did not surface the minimum sample guard requirement as a testable AC
- QA agent identified AC-level gaps during verification, triggering scope expansion
- CONDITIONAL PASS allowed the story to proceed but required additional implementation cycles

**Observation** (below threshold — not logged to KB):
When QA identifies AC-level gaps, it creates rework cycles. Elaboration quality could be improved by
requiring explicit "How will this be tested with insufficient data?" review during elaboration for
stories with statistical or threshold-based features.
