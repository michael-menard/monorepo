# KBAR-0120: Unit Tests for Artifact MCP Tool Handlers

## Context

The KBAR epic is migrating workflow artifacts from file-system YAML files to database-first storage. As part of Phase 3 of this migration, KBAR-0090 established the canonical pattern for unit-testing already-implemented MCP tool handlers by producing `story-tools.test.ts` for the `kb_get_next_story` handler.

KBAR-0120 applies the same pattern to the four artifact tool handlers. All four handlers â€” `handleKbReadArtifact`, `handleKbWriteArtifact`, `handleKbListArtifacts`, `handleKbDeleteArtifact` â€” are fully implemented in production, including DB operations, MCP tool schemas, and dispatch registration (53 tools total). What is missing is a dedicated unit test file covering handler-level branches: null-return paths, authorization enforcement, input validation errors, and DB rejection handling.

The index entry originally framed KBAR-0120 as "implement artifact reading with fallback from DB â†’ file system." That framing is superseded by the seed's reality check: the implementation already exists, the `filePath` column is deprecated, and the tool description explicitly instructs agents to use `kb_read_artifact` "instead of reading YAML files." The scope is therefore unit tests only â€” paralleling exactly what KBAR-0090 did for story tools.

## Goal

Create `apps/api/knowledge-base/src/mcp-server/__tests__/artifact-tools.test.ts` following the `story-tools.test.ts` pattern (KBAR-0090). Provide handler-level unit test coverage for all 4 artifact MCP tool handlers, including the subtly-serialized null-return path for `handleKbReadArtifact`.

## Non-Goals

- **File-system fallback when DB returns null** â€” explicitly out of scope. The deprecated `filePath` column and tool description both confirm the DB-first direction. Any fallback is a separate story.
- **Adding new MCP tools** â€” no new tool schemas, handlers, or registrations. Tool count remains 53.
- **Modifying `artifact-operations.ts`** â€” the DB operations are correct; this story is handler-layer tests only.
- **Modifying `mcp-integration.test.ts`** â€” artifact tools already appear in the tool names list; no changes to tool count or name assertions.
- **DB schema changes or migrations** â€” the `storyArtifacts` table is protected and unchanged.
- **E2E or UAT tests** â€” no frontend impact, no UI involved.

## Architecture Notes

### Mock Isolation (Critical)

The mock must target the specific artifact operations module, not the broader CRUD index:

```typescript
// CORRECT â€” targets artifact-operations.js specifically
vi.mock('../../crud-operations/artifact-operations.js', async importOriginal => {
  const actual = await importOriginal<typeof import('../../crud-operations/artifact-operations.js')>()
  return {
    ...actual,
    kb_read_artifact: mockKbReadArtifact,
    kb_write_artifact: mockKbWriteArtifact,
    kb_list_artifacts: mockKbListArtifacts,
    kb_delete_artifact: mockKbDeleteArtifact,
  }
})

// WRONG â€” this is mcp-integration.test.ts pattern, do not use here
vi.mock('../../crud-operations/index.js', ...)
```

### null-Return Serialization (Critical)

`handleKbReadArtifact` returns the literal string `"null"` when `kb_read_artifact` returns `null` (DB miss):

```typescript
// In tool-handlers.ts lines 3397-3407:
if (result === null) {
  return {
    content: [{ type: 'text', text: 'null' }],  // string 'null', not JSON null
  }
}
```

Test must assert `result.content[0].text === 'null'` (string), not `null === null`.

### vi.hoisted Pattern

```typescript
const {
  mockKbReadArtifact,
  mockKbWriteArtifact,
  mockKbListArtifacts,
  mockKbDeleteArtifact,
} = vi.hoisted(() => ({
  mockKbReadArtifact: vi.fn(),
  mockKbWriteArtifact: vi.fn(),
  mockKbListArtifacts: vi.fn(),
  mockKbDeleteArtifact: vi.fn(),
}))
```

### Fixture Factory

```typescript

```

### Zod Default for iteration

`KbWriteArtifactInputSchema` has `iteration: z.number().int().min(0).optional().default(0)` â€” so `iteration` will be `0` if not supplied. `KbReadArtifactInputSchema` has `iteration` optional (no default). Tests should cover both explicit and implicit iteration values.
