# KBAR-0080: story_list & story_update Tools ‚Äî Validate, Complete Integration Tests, and Add State Transition Guard

## Context

The KB MCP server at `apps/api/knowledge-base/` already contains fully-implemented MCP tool handlers for `kb_list_stories`, `kb_update_story_status`, and `kb_update_story` (alongside the sibling `kb_get_story` from KBAR-0070). These are registered in the tool registry (`toolDefinitions` array in `tool-schemas.ts`), access control, and tool definitions. The DB operations exist at `story-crud-operations.ts`.

**Three gaps remain:**

1. **Stale integration test**: `mcp-integration.test.ts` asserts 52 tools but `kb_update_story` is defined and registered (actual count: **54**, verified 2026-02-23), yet absent from the expected tool names array.
2. **Missing unit tests**: No dedicated unit tests exist for `handleKbListStories`, `handleKbUpdateStoryStatus`, or `handleKbUpdateStory` handler behaviors (filter combinations, auto-timestamps, metadata updates, auth enforcement).
3. **No state transition guard**: `kb_update_story_status` accepts any valid `StoryStateSchema` value without checking current state, allowing logically invalid transitions (e.g., `completed ‚Üí in_progress`).

## Goal

Close all three gaps: fix the stale integration test, write unit tests using the established `vi.hoisted` mock pattern, and add a lightweight terminal-state guard inline in `kb_update_story_status`.

## Non-Goals

- Do NOT implement a full finite-state machine ‚Äî a terminal-state guard (block `completed/cancelled/deferred/failed_code_review/failed_qa ‚Üí non-terminal`) is sufficient.
- Do NOT add new DB schema columns or migrations.
- Do NOT implement `kb_get_next_story` (that is KBAR-0090).
- Do NOT modify sync functions in `packages/backend/kbar-sync/`.
- Do NOT add REST HTTP endpoints ‚Äî MCP tool protocol only.
- Do NOT re-implement `kb_update_story` from scratch ‚Äî it already exists.

## Architecture Notes

- **Tool handler pattern**: `enforceAuthorization ‚Üí validate input with Zod ‚Üí call DB operation ‚Üí log ‚Üí return JSON result`. Follow `handleKbGetStory` exactly.
- **State transition guard**: Add inline in `kb_update_story_status()` after fetching `existing` story and before building `updates` object. Terminal states: `completed`, `cancelled`, `deferred`, `failed_code_review`, `failed_qa`. Guard condition: `if (validated.state && TERMINAL_STATES.includes(existing[0].state) && existing[0].state !== validated.state) ‚Üí return error`.
- **Terminal-state guard return shape** _(AC-8 clarification)_: The guard MUST use the soft-return pattern consistent with the not-found branch (lines 284‚Äì290 of `story-crud-operations.ts`): `return { story: null, updated: false, message: "Cannot transition from terminal state '${currentState}' to '${targetState}'. Terminal states (completed, cancelled, deferred, failed_code_review, failed_qa) are final." }`. Do NOT throw ‚Äî the guard is a business rule soft-rejection, not an exceptional error. The caller (`handleKbUpdateStoryStatus`) surfaces this as a normal JSON tool result (not `isError: true`). Test assertions in ST-4 should verify `updated: false` and the presence of a `message` field, not `isError`.
- **Idempotency**: `startedAt` is only set when transitioning to `in_progress` AND `existing[0].startedAt` is null. `completedAt` is only set when transitioning to `completed` AND `existing[0].completedAt` is null. These checks already exist in the current implementation.
- **`blocked: false` clearing**: Already implemented ‚Äî when `blocked === false`, `blockedReason` and `blockedByStory` are set to null.
- **`states[]` precedence**: `validated.states?.length` check takes precedence over singular `state` filter in `kb_list_stories`. Already implemented ‚Äî do not change behavior.
- **No barrel imports**: Import directly from `story-crud-operations.ts` or `__types__/index.ts`, never from barrel files.
- **Zod-first types**: Any new constants should use Zod enums or `z.infer<>` where applicable.
- **Logging**: Use `createMcpLogger()` from `mcp-server/logger.ts` ‚Äî never `console.log`.
