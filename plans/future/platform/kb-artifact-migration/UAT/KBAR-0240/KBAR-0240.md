# KBAR-0240: Regenerate Index CLI — `regenerate:index` Command for DB-Driven Stories Index

## Context

KBAR-0230 (DB-Driven Index Generation) will implement `generateStoriesIndex()`, a TypeScript function that queries the database and generates a `stories.index.md` file to replace the manually maintained index. Once this function exists, there is no command-line way to invoke it.

Without a CLI entry point, `generateStoriesIndex()` can only be called from TypeScript code. Developers cannot trigger index regeneration from the terminal, CI pipelines cannot run it as a pre-push step, and workflow transition commands cannot invoke it to keep the index in sync after story state changes.

This story adds `regenerate-index.ts` to `packages/backend/kbar-sync/scripts/` — a thin CLI wrapper around `generateStoriesIndex()` — following the established `sync-epic.ts` / `sync-story.ts` pattern.

**Implementation Gate**: KBAR-0230 must be merged to main before implementation begins. The function signature of `generateStoriesIndex()` must be stable before the wrapper can be written. PM and elaboration artifacts for KBAR-0240 can be produced now.

## Goal

Provide a `regenerate:index` CLI command (registered in `package.json`) that:
- Accepts `--epic`, `--output`, `--dry-run`, `--verbose`, `--force`, and `--help` flags
- Delegates entirely to `generateStoriesIndex()` from `@repo/kbar-sync`
- Handles argument parsing, Zod validation, progress output, error reporting, and exit codes
- Is executable by developers, CI pipelines, and workflow transition scripts

## Non-Goals

- Implementing `generateStoriesIndex()` — that is KBAR-0230
- Wiring `regenerate:index` into the Husky pre-commit hook — stretch goal only; the story documents the integration path
- Automatically detecting which epics need regeneration — requires explicit `--epic` argument
- Supporting parallel regeneration of multiple epics in one invocation
- Modifying any existing story data or DB records — read-only from the DB perspective
- Frontend or API changes
- Replacing the existing `generate-stories.sh` script (for story generation, not index regeneration)
- Modifying `packages/backend/database-schema/`, `@repo/db` API surface, or orchestrator artifact schemas

## Scope

**Packages touched:**
- `packages/backend/kbar-sync/scripts/regenerate-index.ts` — new CLI entry point

## Architecture Notes

The CLI is a thin delegation layer. It must not contain any business logic — all index generation logic lives in `generateStoriesIndex()` in `@repo/kbar-sync`. The CLI's responsibilities:

1. Parse and validate CLI arguments (Zod)
2. Print `--help` without importing DB-touching modules
3. Call `generateStoriesIndex()` with derived options
4. Handle return value: write to file (`--output`) or print to stdout (`--dry-run`)
5. Catch errors and map to exit codes: 0 = success, 1 = logic/validation failure, 2 = DB connection failure
6. Log progress via `@repo/logger` when `--verbose` is set

**Key patterns from `sync-epic.ts` to copy verbatim:**
- `HELP_TEXT` inline constant at top of file
- `rethrowExitError` helper (required for `process.exit` mock in tests)
- Zod `safeParse` before any DB imports
- Dynamic import of `@repo/kbar-sync` after validation to allow `--help` without env vars

## Infrastructure Notes

No infrastructure changes required. This is a pure Node.js CLI script using `tsx` — already available in the package.

**Pre-commit hook integration (stretch goal, not required for this story):**
The `.husky/pre-commit` hook could be extended to run `regenerate:index --dry-run` when `plans/future/platform/*/stories.index.md` files are staged. This would enforce index freshness before commit. The conditional pattern (check `git diff --cached --name-only` for staged index files) already exists in `.husky/pre-commit`. This integration is documented here for reference but is NOT in scope for KBAR-0240.
