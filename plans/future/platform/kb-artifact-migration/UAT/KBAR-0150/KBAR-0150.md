# KBAR-0150: Artifact MCP Tools — Integration Tests via `handleToolCall`

## Context

The four artifact MCP tools (`kb_write_artifact`, `kb_read_artifact`, `kb_list_artifacts`, `kb_delete_artifact`) are fully implemented and registered in the MCP server. Unit-level tests for all four handlers exist in `artifact-tools.test.ts` (delivered by KBAR-0120), covering handler business logic at the direct-call level.

However, no tests exercise these tools through the `handleToolCall` dispatch layer — the MCP routing and error-wrapping infrastructure that connects the protocol wire format to individual handlers.

The `handleToolCall` dispatch layer is responsible for:
1. Routing tool names to handler functions
2. Enforcing access-control policies (authorization check before business logic)
3. Wrapping handler errors into the MCP error response format
4. Sanitizing database credentials from error messages

None of these behaviors are verified at the integration level for the artifact tools. If a tool name were misspelled in the registry, or the authorization rule misconfigured, existing unit tests would not catch it.

This story is structurally identical to KBAR-0100 (Story Tools Integration Tests) but targets the artifact tool domain.

## Goal

Write a new `artifact-tools-integration.test.ts` file in the existing `__tests__/` directory that calls `handleToolCall` for each artifact tool to verify:
1. Tool name routing correctly dispatches to artifact handlers
2. Validation errors bubble through `handleToolCall` with `isError: true` and `code: 'VALIDATION_ERROR'`
3. DB errors are sanitized before reaching the caller
4. Authorization roles are enforced: `kb_delete_artifact` is PM-only and returns `FORBIDDEN` for non-PM roles; the other three tools allow all roles
5. Happy-path responses are JSON-serializable and structurally correct via the MCP content array

## Non-Goals

- Do NOT re-implement unit tests already in `artifact-tools.test.ts` — those cover handler business logic; this story covers dispatch routing
- Do NOT test CRUD query logic (e.g., upsert behavior, DB query conditions) — mock at the `artifact-operations.js` boundary
- Do NOT add new MCP tool handlers, schemas, or access-control entries
- Do NOT add DB migrations, new schema columns, or Drizzle schema changes
- Do NOT add REST HTTP endpoints
- Do NOT add Playwright E2E tests (no UI surface)
- Do NOT add testcontainers or real-PostgreSQL integration tests
- Do NOT modify `mcp-integration.test.ts` tool count or names list (KBAR-0150 adds no new tools)
- Do NOT test the `artifact_write` dual-write tool (KBAR-0110) — this story tests the DB-only `kb_write_artifact` tool only
- Do NOT call individual handlers directly (e.g., `handleKbWriteArtifact(...)`) — use `handleToolCall` throughout

## Architecture Notes

**Integration vs Unit boundary**: This story tests the `handleToolCall` dispatch layer, not the individual handler logic:
- Unit tests (artifact-tools.test.ts) call `handleKbWriteArtifact(...)` directly — tests business logic
- Integration tests (this story) call `handleToolCall('kb_write_artifact', ...)` — tests routing, error wrapping, authorization

If a test would pass whether or not `handleToolCall` dispatches correctly, it belongs in `artifact-tools.test.ts`, not this story.

**Mock boundary**: Mock at `artifact-operations.js` CRUD boundary only. Do not mock at DB level, do not mock `tool-handlers.ts` itself.

**Authorization configuration** (from `access-control.ts`):
- `kb_write_artifact`, `kb_read_artifact`, `kb_list_artifacts` → `pm, dev, qa, all`
- `kb_delete_artifact` → `pm` only (admin destructive operation)

**ESM requirement**: `vi.mock('../../crud-operations/artifact-operations.js', ...)` — `.js` extension required, not `.ts`.

**ADR-005**: These are `vi.mock`-level integration tests. No real PostgreSQL required.
