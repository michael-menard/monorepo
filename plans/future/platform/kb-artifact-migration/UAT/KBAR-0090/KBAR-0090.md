# KBAR-0090 — `kb_get_next_story` Tool: Unit Tests and Integration Test Hygiene

## Context

The `kb_get_next_story` MCP tool is **already fully implemented** at every layer of the stack:

| Layer | Location | Status |
|-------|----------|--------|
| DB function | `apps/api/knowledge-base/src/crud-operations/story-crud-operations.ts` lines 369–496 | Fully implemented |

| Access control | `apps/api/knowledge-base/src/mcp-server/access-control.ts` | Granted to pm/dev/qa/all |

The `toolDefinitions` array currently has **53 entries** (as of 2026-02-23). The `mcp-integration.test.ts` file asserts 52 and is missing `kb_update_story` from the expected names list. These are hygiene issues, not feature gaps.

**The implementation gap is tests, not code.** No unit tests exist for `handleKbGetNextStory` or the underlying DB function.

This story was recharacterized from "implement story_ready_to_start tool" to "test coverage for the already-implemented kb_get_next_story tool" after codebase verification confirmed full implementation at all layers.

## Goal

Produce unit tests for `handleKbGetNextStory` covering all key scenarios (no candidates, all blocked, first unblocked wins, priority ordering, dependency resolution, `exclude_story_ids`, `include_backlog`, authorization, Zod validation), and fix the `mcp-integration.test.ts` hygiene gap (add `kb_update_story` to the expected names list, correct the tool count assertion to match `getToolDefinitions().length`).

## Non-Goals

- Do NOT implement `kb_get_next_story` — it is already fully implemented.
- Do NOT add circular dependency detection.
- Do NOT add new DB schema columns, migrations, or Drizzle schema changes.
- Do NOT add REST HTTP endpoints — MCP tool protocol only.
- Do NOT modify `tool-schemas.ts`, `tool-handlers.ts`, or `access-control.ts`.
- Do NOT add integration tests against a real PostgreSQL instance (testcontainers); unit tests with `vi.mock` are sufficient.
- Do NOT re-examine the `kb_update_story_status` terminal-state guard — that is KBAR-0080's scope.

## Architecture Notes

### Mock Return Shapes

Three key shapes to mock (matching the TypeScript return type of `kb_get_next_story`):

```typescript
// Shape 1: No candidates
{ story: null, candidates_count: 0, blocked_by_dependencies: [], message: 'No stories found in {epic} epic' }

// Shape 2: All blocked
{ story: null, candidates_count: N, blocked_by_dependencies: ['STORY-A (blocked by: STORY-B)'], message: 'All N candidates are blocked' }

// Shape 3: Successful find
{ story: { storyId: 'X', epic: '...', state: 'ready', priority: 'high', createdAt: new Date(), ... }, candidates_count: N, blocked_by_dependencies: [...], message: 'Found next story: X' }
```

Read `story-crud-operations.ts` lines 369–496 before writing mocks to ensure exact shape match.

### Handler Invocation Pattern

```typescript
// In handleKbGetNextStory (lines 3732-3778):
enforceAuthorization('kb_get_next_story' as ToolName, context)  // authorize first
const input = KbGetNextStoryInputSchema.parse(rawInput)          // then validate
const result = await kbGetNextStory(deps, input)                  // then DB call
return { content: [{ type: 'text', text: JSON.stringify(result, null, 2) }] }
// errorToToolResult(error) in catch block
```

### vi.mock ESM Pattern

```typescript
// ESM requires .js extension in mock paths:
vi.mock('../../crud-operations/story-crud-operations.js', async importOriginal => {
  const actual = await importOriginal()
  return {
    ...actual,
    kbGetNextStory: mockKbGetNextStory,
  }
})
```

### Circular Dependency Note

The current DB function does NOT implement explicit circular dependency detection. For circular references, it returns all candidates as blocked (safe behavior). Tests should not assert on specific circular dependency detection — only on the "all candidates blocked" outcome.
