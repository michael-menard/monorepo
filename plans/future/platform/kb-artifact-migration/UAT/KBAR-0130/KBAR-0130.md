# KBAR-0130: artifact_search Tool — Semantic Search Across KB-Indexed Artifacts Using Natural Language

## Context

The KB artifact migration epic (KBAR) is building an MCP tool layer so agents can read and write workflow artifacts (CHECKPOINT.yaml, EVIDENCE.yaml, REVIEW.yaml, etc.) through a stable API. KBAR-0110 (`artifact_write`) establishes dual-write, indexing artifacts into the KB `knowledgeEntries` table alongside the `storyArtifacts` table. KBAR-0120 (`artifact_read`) enables reading by story + type. KBAR-0130 closes the loop: it enables agents to find artifacts they didn't already know about via natural language.

**Problem**: Once hundreds of artifacts are stored in the KB across dozens of stories, agents have no way to query across them semantically. An agent running a new story cannot ask "what evidence artifacts mentioned OpenAI API timeout handling?" or "find all REVIEW artifacts that flagged Zod schema violations" without scanning the file system manually. The existing `kb_search` tool can answer these questions if called correctly, but it requires callers to know the right tag structure and `entry_type` filters — knowledge that is not codified in any tool API.

**Reality baseline** (2026-02-13): The `kb_search` function (`apps/api/knowledge-base/src/search/kb-search.ts`) is fully operational — hybrid semantic + keyword search with pgvector + RRF. The `handleKbSearch` handler in `tool-handlers.ts` is the direct structural template for `handleArtifactSearch`. The `SearchInputSchema` and `SearchResultSchema` in `search/schemas.ts` are the base types to compose from. The `ArtifactTypeSchema` and `StoryPhaseSchema` exist in `__types__/index.ts`. Current tool count is 53 (as of KBAR-0090 landing).

## Goal

Implement a new `artifact_search` MCP tool that accepts a natural language `query` plus optional artifact-domain filters (`story_id`, `artifact_type`, `phase`) and returns ranked, relevance-scored results from KB-indexed artifacts. The tool translates artifact filters into KB tag expressions and delegates all search logic to the existing `kb_search` function.

## Non-Goals

- Do not modify or extend `kb_search` — `artifact_search` is a wrapper, not a modification
- Do not implement `artifact_read` (KBAR-0120) or `artifact_write` (KBAR-0110) — those are separate stories
- Do not implement `artifact_summary_extraction` (KBAR-014) — that is the next phase story
- Do not add full-text search across artifact JSONB content directly in SQL — all search goes through KB embeddings via `kb_search`
- Do not add E2E/Playwright tests — no UI surface (`e2e: not_applicable`)
- Do not modify the `storyArtifacts` DB schema
- Do not modify the `knowledge_entries` schema or the `kb_search` function
- Do not modify existing `kb_write_artifact`, `kb_read_artifact`, or `kb_search` handlers

## Architecture Notes

**Filter-to-Tag Translation**:

```typescript
// Pattern: translate artifact-domain filters into KB tag filter array
// kb_search uses OR semantics for tags — callers should understand
// that multi-filter results include artifacts matching ANY provided tag
const tags = [
  'artifact',              // always present — scopes to KB-indexed artifacts
  story_id,               // optional — narrows to artifacts from a specific story
  artifact_type,          // optional — narrows to a specific artifact type (e.g., 'EVIDENCE')
  phase,                  // optional — narrows to a specific phase (e.g., 'dev_implementation')
].filter(Boolean) as string[]

// Compose with remaining SearchInput fields
const searchInput = {
  query: validated.query,
  tags,
  limit: validated.limit,
  min_confidence: validated.min_confidence,
  explain: validated.explain,
}
```

**Tag Convention Dependency**: The tag values used by `artifact_search` must match those written by `artifact_write` (KBAR-0110) when indexing artifacts into `knowledge_entries`. The convention must be confirmed before KBAR-0130 implementation begins. If KBAR-0110 writes `tags: ['artifact', story_id, artifact_type]`, then `artifact_search` must use the same values.

**Handler Structure** (following `handleKbSearch` template):
1. `ArtifactSearchInputSchema.parse(input)` — Zod validation
2. Build `SearchInput` — translate artifact filters to `tags` array
3. `kb_search(searchInput, deps)` — delegate, no timeout wrapping needed unless pattern requires it
4. Check `result.metadata.fallback_mode` — log warning if true
5. Return `{ content: [{ type: 'text', text: JSON.stringify(response) }] }` — always success for valid input (including empty results)
6. Catch errors — return structured error response (never unhandled exceptions at the tool boundary)

**No Database Access**: `artifact_search` goes through `kb_search` only — no direct DB queries. The `storyArtifacts` table is not queried directly.
