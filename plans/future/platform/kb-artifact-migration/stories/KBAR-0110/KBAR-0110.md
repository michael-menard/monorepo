# KBAR-0110: `artifact_write` Tool — Dual-Write Artifact to File + KB with Graceful Failure Isolation

## Context

The KB artifact migration epic (KBAR) is building an MCP tool layer over the knowledge-base DB so agents can read and write workflow artifacts (CHECKPOINT.yaml, EVIDENCE.yaml, etc.) through a stable API rather than raw file system operations. KBAR-0060 through KBAR-010 establish the story sync infrastructure and story tool layer. Phase 4 introduces artifact-specific tools.

Agents currently write artifacts directly to the file system using hardcoded paths. There is no mechanism to simultaneously index an artifact in the KB for semantic search. The existing `kb_write_artifact` tool (tool-handlers.ts line ~3298) writes only to the DB — it does not write to the file system. Agents need a single tool that (a) writes the YAML artifact to the canonical file path and (b) optionally indexes it in the KB for future retrieval via `kb_search`.

**Dependency gate**: Implementation cannot begin until both KBAR-010 (Story Tools Integration Tests) and KBAR-0090 (kb_get_next_story Unit Tests) are merged to main. Both stories modify `mcp-integration.test.ts` and `tool-schemas.ts`; merge conflicts on the tool count assertion are guaranteed if KBAR-0110 starts before they land.

## Goal

Implement a new `artifact_write` MCP tool that:
1. Accepts artifact content (YAML/JSON), artifact type, story ID, and an optional file path override
2. Writes the artifact to the canonical file location as its **primary** operation
3. Optionally writes to the KB (`storyArtifacts` table) as a **secondary, best-effort** operation
4. Returns a structured result indicating both write outcomes
5. Never lets KB write failure prevent the file write result from being returned

## Non-Goals

- Do not modify the existing `kb_write_artifact` or `kb_read_artifact` tools — `artifact_write` is a new, parallel tool
- Do not implement `artifact_read` (KBAR-012) or `artifact_search` (KBAR-013) — those are separate stories
- Do not implement async/fire-and-forget KB write — synchronous await-with-try/catch is sufficient
- Do not change orchestrator artifact YAML schemas in `packages/backend/orchestrator/src/artifacts/`
- Do not add E2E/Playwright tests — no UI surface (`e2e: not_applicable` in SCOPE.yaml)
- Do not modify the `storyArtifacts` DB schema

## Architecture Notes

**Dual-write pattern:**
```
1. Parse and validate input (ArtifactWriteInputSchema.parse(input))
2. Derive canonical file path (from input.file_path or computed from story_id + artifact_type)
3. Serialize content to YAML/JSON string
4. fs/promises.mkdir({ recursive: true }) + fs/promises.writeFile(path, content)
   → If writeFile throws: return error response immediately. Do NOT attempt KB write.
5. try {
     const result = await kb_write_artifact({ story_id, artifact_type, content, phase, iteration })
     kb_artifact_id = result.id
   } catch (err) {
     logger.warn('artifact_write: KB write failed', { error: err.message })
     kb_write_warning = err.message
   }
6. Return { file_path, file_written: true, kb_artifact_id, kb_write_skipped: false, kb_write_warning }
```

**Key invariant**: The caller ALWAYS receives the file write outcome. KB write failure is surfaced as a warning field, not a thrown error.

**skip_kb shortcut**: If `input.skip_kb === true`, skip step 5 entirely and include `kb_write_skipped: true` in response.

**Tool count hygiene**: At ST-1, run `getToolDefinitions().length` to get ground-truth count. Do NOT hardcode the expected count from this story document — the count may differ if KBAR-0090 and KBAR-010 change it. The ST-1 verification step resolves this.
