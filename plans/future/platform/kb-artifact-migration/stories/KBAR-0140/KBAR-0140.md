# KBAR-0140: Artifact Summary Extraction — Shared Utility for Auto-Populating `summary` on Write

## Context

The `storyArtifacts` jump table has a `summary` JSONB column (added in migration 009/015) that is populated by callers of `kb_write_artifact`. Today, no summary auto-extraction exists in production code — callers must either pass a `summary` object or leave it null. When agents call `artifact_write` or `kb_write_artifact`, the `summary` field in the database remains `null` unless the agent manually computes it.

A prototype `extractSummary()` function exists in `apps/api/knowledge-base/src/scripts/migrate-artifacts-simple.ts` (lines 100–203), covering 9 of 13 artifact types. It is not importable from production code (lives in `src/scripts/`).

The `artifact_search` tool (KBAR-0130, completed) surfaces `summary` fields from the jump table in search results. The quality of those results degrades when summaries are null. KBAR-0120 (artifact_read unit tests) is in UAT and establishes the test patterns this story follows. KBAR-0150 (Artifact Tools Integration Tests) is downstream and depends on this story.

## Goal

Extract `extractSummary()` into a shared, importable TypeScript module, expand coverage to all 13 artifact types, integrate into `artifact_write` so summary auto-extraction happens automatically at write time (with caller opt-out), and write Vitest unit tests covering all 13 types and the fallback case.

## Non-Goals

- Do NOT add a new MCP tool — this is a utility module consumed internally by existing tools
- Do NOT modify the `storyArtifacts` DB schema — `summary jsonb` column already exists
- Do NOT backfill existing null summaries in the database — separate migration concern
- Do NOT change the `kb_write_artifact` MCP tool's public input schema
- Do NOT touch `migrate-artifacts-simple.ts` — the script prototype is separate
- Do NOT update `mcp-integration.test.ts` tool count — no new MCP tool is registered
- Do NOT implement lesson or architectural decision extraction (KBAR-0250/0260)

## Architecture Notes

### Module placement

Place the new utility at `apps/api/knowledge-base/src/crud-operations/artifact-summary.ts`. This co-locates it with `artifact-operations.ts` (the only production consumer) and follows the existing cohesion of the `crud-operations/` directory.

### Function signature

```typescript
import { z } from 'zod'

const ExtractArtifactSummaryInputSchema = z.object({
  artifactType: z.string(),
  content: z.record(z.unknown()),
})

export function extractArtifactSummary(
  artifactType: string,
  content: Record<string, unknown>,
): Record<string, unknown>
```

### Integration into `artifact_write`

`ArtifactWriteInputSchema` must gain an optional `summary` field:

```typescript
export const ArtifactWriteInputSchema = z.object({
  // ... existing fields ...
  summary: z.record(z.unknown()).optional(),
})
```

Inside `artifact_write`, before the `kbWriteFn` call:

```typescript
const resolvedSummary =
  validatedInput.summary ?? extractArtifactSummary(validatedInput.artifact_type, validatedInput.content)
```

Pass `resolvedSummary` as `summary` to `kbWriteFn`. This satisfies AC-5 (auto-extraction) and AC-6 (caller override).

### Zod-first requirement

All types via Zod `z.infer<>`. No TypeScript `interface` or standalone `type` without Zod. The extraction function's I/O types are simple enough that `Record<string, unknown>` (already Zod-typed via `z.record(z.unknown())`) satisfies this.
