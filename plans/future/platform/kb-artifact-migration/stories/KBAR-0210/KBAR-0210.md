# KBAR-0210: Standardize kb_update_story_status Usage Across All Workflow Commands

## Context

The KBAR epic's Phase 5 migrates all workflow agents and commands to use the MCP tools for state management. KBAR-0210 is the command-layer counterpart to the agent-layer migrations in KBAR-0160 through KBAR-0200. The `kb_update_story_status` MCP tool (delivered in KBAR-0080, UAT'd) is already used by most workflow commands — but the call patterns are inconsistent.

Some commands have the full Step 0.6 claim + guard + abort recovery pattern (e.g., `qa-verify-story.md`, `dev-code-review.md`); others have only done-path updates (e.g., `dev-implement-story.md`); and `elab-story.md` uses a state string that requires verification against the actual `StoryStateSchema` enum.

**Critical finding (from feasibility review)**: The valid DB state strings come from `StoryStateSchema` in `apps/api/knowledge-base/src/__types__/index.ts`. The valid states include `'ready'` (not `'ready_to_work'`). The `story-update.md` shim mapping table maps the directory-level string `'ready-to-work'` to the shim's internal string `'ready_to_work'` — but the actual DB enum value is `'ready'`. This means `elab-story.md`'s use of `state: 'ready'` may already be correct. Dev must verify the actual call site at ST-1 before making corrections.

**KBAR-0210 closes Phase 5** by ensuring all workflow command markdown files are consistent, complete, and defensively documented. This is a **documentation-only** story — no TypeScript source changes.

## Goal

Audit all workflow command markdown files in `.claude/commands/` for `kb_update_story_status` call sites. For each command: verify state strings against the canonical `StoryStateSchema` enum; add or complete the Step 0.6 claim + guard + abort recovery pattern where missing; add graceful failure documentation for KB unavailability. Leave `story-update.md` and `story-move.md` unchanged unless a regression is found.

## Non-Goals

- Implementing a new MCP tool named `story_update` — `kb_update_story_status` is the correct tool
- Changing the `kb_update_story_status` or `kb_update_story` MCP tool implementations or signatures
- Migrating agents (`.claude/agents/*.agent.md`) — covered by KBAR-0160 through KBAR-0200
- Modifying state transition validation logic in the server — already implemented and tested
- Touching `story-update.md` or `story-move.md` unless a regression is found during the audit
- Adding unit tests for command markdown behavior (not testable at that granularity)
- Changing KB artifact content structure
- Modifying the `shimUpdateStoryStatus` compatibility shim

## Scope

- `.claude/commands/dev-implement-story.md` — add Step 0.6 claim if absent; verify Done-path state strings
- `.claude/commands/elab-story.md` — verify PASS/FAIL path state strings; add graceful failure note
- `.claude/commands/dev-fix-story.md` — verify Step 0.6 claim, guard, abort recovery pattern

**No TypeScript source files are modified.**

## Acceptance Criteria

- [ ] AC-1: All `kb_update_story_status` call sites in `.claude/commands/*.md` use state strings that match valid `StoryStateSchema` enum values: `backlog`, `ready`, `in_progress`, `ready_for_review`, `in_review`, `ready_for_qa`, `in_qa`, `completed`, `cancelled`, `deferred`, `failed_code_review`, `failed_qa`.
- [ ] AC-2: `elab-story.md` PASS path `kb_update_story_status` call uses the correct DB state string (verified against `StoryStateSchema` enum at implementation time — `'ready'` is a valid enum value; confirm whether the current call site already uses it correctly).
- [ ] AC-3: `dev-implement-story.md` either (a) adds a Step 0.6 DB claim with `state: "in_progress"`, guard, and abort recovery, or (b) explicitly documents why a Step 0.6 claim is not needed. The decision is documented in the story implementation notes.
- [ ] AC-4: All workflow commands that claim a story (Step 0.6) include: the `kb_update_story_status` call, a guard preventing double-claim (check if already in the claimed state), and an abort recovery instruction for interrupted runs.
- [ ] AC-5: All workflow commands that call `kb_update_story_status` document the graceful failure behavior: if the call returns null or throws, emit a WARNING and continue — do not block the workflow.
- [ ] AC-6: `dev-fix-story.md` Step 0.6 claim and Done-path state updates use correct state strings and have the guard + abort recovery pattern.
- [ ] AC-7: No changes are made to `story-update.md` or `story-move.md` unless a genuine regression is found. If changes are made, the reason is documented.
- [ ] AC-8: No TypeScript source files are modified. All changes are to command markdown files in `.claude/commands/`.

## Reuse Plan

- **Canonical pattern source**: `.claude/commands/qa-verify-story.md` and `.claude/commands/dev-code-review.md` — use these as the model for the Step 0.6 claim + guard + abort recovery pattern
- **State name reference**: `apps/api/knowledge-base/src/__types__/index.ts` `StoryStateSchema` enum — authoritative source of valid DB state strings
- **Graceful failure pattern**: Follow the same WARNING + continue semantics as `story-update.md` Step 3a for KB unavailability
- **Shim boundary**: `story-update.md` and `story-move.md` use `shimUpdateStoryStatus` — this is a different but correct pattern; do not change them to `kb_update_story_status`

## Architecture Notes

This story makes no architectural changes. It is a consistency audit and pattern-completion pass for existing command markdown files. The architecture is already correct; the goal is to ensure all commands follow it uniformly.

Key architectural decision to document for AC-3 (dev-implement-story.md Step 0.6):
- The worktree verification in dev-implement-story.md Step 1.3 already prevents parallel work conflicts at the filesystem level
- Adding a Step 0.6 DB claim (`state: "in_progress"`) would additionally ensure DB state reflects implementation start at the correct moment
- Recommendation: add the Step 0.6 claim for DB state consistency — this is the same reasoning applied to all other workflow commands
