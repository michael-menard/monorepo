# KBAR-0230: DB-Driven Index Generation — generateStoriesIndex() Utility

## Context

The `stories.index.md` files for each KBAR epic phase are currently maintained manually. Evidence across the KBAR epic history (KBAR-0150, KBAR-0170, KBAR-0180, KBAR-0210) shows repeated description drift: story sections accumulate stale titles and statuses as work progresses. This drift is a persistent maintenance burden and undermines the index's utility as a navigation and planning tool.

The `kbar` PostgreSQL schema already includes two tables designed to solve this problem:
- `kbar.index_metadata`: tracks generated index files (path, checksum, generation timestamp)
- `kbar.index_entries`: tracks per-story entries within each index (sort order, section)

These tables have been present since KBAR-0010 but unused — KBAR-0230 activates them by implementing the utility that writes to them.

This story depends on KBAR-0220 (Agent Migration Testing), which validates the full story workflow using updated agents. KBAR-0230 is a Phase 6 story and cannot begin implementation until KBAR-0220 completes.

## Goal

Implement `generateStoriesIndex(epic, db)` — a TypeScript utility function that queries the `kbar` PostgreSQL schema to produce a structurally valid, DB-authoritative `stories.index.md` file for a given epic. The generated output must be backward-compatible with existing parsers and agent tooling.

## Non-Goals

- CLI command for invoking the generator — that is KBAR-0240's scope
- Integration into pre-commit hooks or workflow transitions — that is KBAR-0240's scope
- Generating the global `platform.stories.index.md` — per-epic index only in this story
- Migrating existing story descriptions from filesystem to DB — earlier KBAR sync story scope
- Modifying any existing production DB schemas — the `kbar` schema already has required tables
- Real-time or event-driven regeneration — batch generation only
- E2E tests or UAT with real DB — unit tests only in this story

## Architecture Notes

### Implementation Location

The utility lives at `packages/backend/database-schema/src/seed/generate/generateStoriesIndex.ts`. This is parallel to the existing `parsers/` directory within the same package and consistent with the seeder utility pattern established by `capability-seeder.ts`.

### Frontmatter Serialization

Use a string template (not `gray-matter`) for generating the YAML frontmatter block. `gray-matter` is a parsing library and its serialization output does not reliably match the exact format of the manually-maintained files. A direct string template produces byte-for-byte identical output with lower friction.

### Dependency Label Resolution

`kbar.story_dependencies` stores `dependsOnStoryId` as a UUID (FK to `kbar.stories.id`). Human-readable story IDs must be resolved by joining back to `kbar.stories.storyId`. This must be a JOIN in the query layer — not N individual lookups — for performance. The join should produce tuples of `(dependingStoryId, dependsOnStoryId)` where both values are the human-readable string IDs.

### Checksum Computation

Use Node.js built-in `crypto.createHash('sha256').update(markdownString).digest('hex')`. No external dependency needed. Checksum is written to `kbar.index_metadata.checksum` for change detection and idempotent regeneration.

### DB Write Idempotency

Both `kbar.index_metadata` (unique on `indexName`) and `kbar.index_entries` (unique on `(indexId, storyId)`) must use `ON CONFLICT DO UPDATE` to support repeated calls without creating duplicate rows.

### "Ready to Start" Logic

A story qualifies for "Ready to Start" if:
1. It has no rows in `kbar.story_dependencies` pointing to it as the depending story, OR
2. All its dependency rows resolve to stories whose `status` in `kbar.stories` is `completed` or `uat`

The `resolved` boolean on `storyDependencies` is a cache hint and must be cross-referenced against actual target story status.

### Format Contract

The generated markdown must satisfy `parseStoriesIndex()` without errors. Read `index-parser.ts` in full before implementing any renderer functions. Implement the round-trip test (TC-07) early as a format contract test — failing this test reveals format deviations before completing the full implementation.
