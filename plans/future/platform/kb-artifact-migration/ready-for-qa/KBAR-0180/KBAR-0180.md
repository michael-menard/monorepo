# KBAR-0180: Migrate `review-aggregate-leader` to Use `artifact_write` for Dual-Write REVIEW Artifact Persistence

## Context

The KBAR epic's Phase 5 migrates all workflow agents from `kb_write_artifact` (KB-only writes) to `artifact_write` (dual-write: file system + KB). KBAR-0160 handles setup/plan leaders; KBAR-0170 handles execute/worker agents; KBAR-0180 handles code review agents.

The code review agent family consists of 10 worker agents (`code-review-syntax`, `code-review-security`, `code-review-react`, `code-review-typescript`, `code-review-style-compliance`, `code-review-reusability`, `code-review-accessibility`, `code-review-build`, `code-review-lint`, `code-review-typecheck`) and one aggregator (`review-aggregate-leader`). **Only the aggregator writes REVIEW artifacts to the KB** — the workers return YAML output to the orchestrator context only. This was confirmed by direct grep: `review-aggregate-leader.agent.md` is the only file in the code review agent family calling `kb_write_artifact`.

**Problem**: REVIEW artifacts written by `review-aggregate-leader` exist only in the KB database. They are not written to the file system (`_implementation/REVIEW.yaml`). This limits observability — humans cannot inspect past review findings without KB queries, and file-system tooling cannot discover them. REVIEW artifacts are high-value for pattern analysis, but the KB write is currently the only write.

**Proposed solution**: Update `review-aggregate-leader.agent.md` to replace the `kb_write_artifact` call (Step 5, "Write Review Artifact to KB") with `artifact_write`. Pass an explicit `file_path` argument pointing to `_implementation/REVIEW.yaml` (Option A: overwrite each iteration, consistent with KBAR-0160 convention). Update frontmatter `kb_tools`. Document graceful failure semantics. The 10 worker agents require no changes.

**Dependencies**:
- KBAR-0110 (`artifact_write` tool) — in UAT
- KBAR-0150 (Artifact Tools Integration Tests) — Created
- KBAR-0160 (Update Setup & Plan Leaders) — needs-code-review; **must merge before KBAR-0180 implementation starts**

## Architecture Notes

The `artifact_write` dual-write design:
- **File write is primary (blocking)**: If the file write fails, `artifact_write` returns `file_written: false` and an error. The workflow should treat this as a hard failure.
- **KB write is secondary (best-effort)**: If the KB write fails, `artifact_write` returns `file_written: true` and `kb_write_warning`. The code review workflow proceeds normally — the REVIEW result is preserved on the file system.
- **Iteration field in KB**: Even with Option A (REVIEW.yaml overwrite), the KB record stores `iteration` as metadata, enabling per-iteration queries via `artifact_search`.

**File path template**:
```
{feature_dir}/in-progress/{story_id}/_implementation/REVIEW.yaml
```

**Updated `artifact_write` call shape** (to replace current `kb_write_artifact`):
```javascript
artifact_write({
  story_id: "{STORY_ID}",
  artifact_type: "review",
  phase: "code_review",
  iteration: {iteration},
  file_path: "{feature_dir}/in-progress/{story_id}/_implementation/REVIEW.yaml",
  content: { /* REVIEW structure — unchanged from current */ }
})
```

**Response handling**:
```javascript
const result = artifact_write({...})
if (!result.file_written) {
  // Hard failure — REVIEW not persisted; block workflow
  throw new Error(`REVIEW artifact write failed: ${result.error}`)
}
if (result.kb_write_warning) {
  // Soft warning — log but continue
  console.warn(`REVIEW KB indexing failed: ${result.kb_write_warning}`)
}
```
