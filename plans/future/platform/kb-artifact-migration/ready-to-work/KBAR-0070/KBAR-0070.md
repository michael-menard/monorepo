# KBAR-0070: story_get Tool — MCP Story Retrieval with Optional Artifacts and Dependencies

## Context

The KB knowledge base MCP server provides a basic `kb_get_story` tool that retrieves a story record by ID from the `stories` database table. The current implementation (introduced in the KBAR story infrastructure) returns only the story row — it does not include the story's associated artifacts (from `story_artifacts`) or dependency edges (from `story_dependencies`).

Agents that need full story context — checking what artifacts have been written, or which stories a given story depends on — currently must chain multiple separate MCP tool calls: `kb_get_story` + `kb_list_artifacts` + an ad-hoc dependency query. This is token-inefficient and error-prone.

The `storyArtifacts` and `storyDependencies` tables already exist in the database schema (delivered via KBAR-001). The data is present; there is no query path to retrieve it in a single `kb_get_story` call.

**Blocking dependency:** This story is blocked until KBAR-0060 (Sync Integration Tests) passes UAT and merges to main. Story generation and elaboration can proceed now. Implementation must not begin until unblocked.

## Goal

Extend the existing `kb_get_story` MCP tool to support two optional boolean parameters — `include_artifacts` and `include_dependencies` — enabling agents to retrieve a complete story context in a single tool call while preserving full backward compatibility with existing callers.

## Non-Goals

- Adding new DB tables or columns — all data is in existing `stories`, `story_artifacts`, `story_dependencies` tables
- Implementing `kb_list_stories` eager loading — only single-story retrieval (`kb_get_story`) is in scope
- Returning artifact content JSONB field — `include_artifacts` returns metadata rows only (type, name, filePath, summary); full content is available via `kb_read_artifact`
- Dependency graph traversal — `include_dependencies` returns direct (1-hop) edges only; transitive closure is out of scope (handled by `story_ready_to_start`)
- Creating a new MCP tool named `story_get` — this is an extension of the existing `kb_get_story` tool
- Any UI or HTTP API surfaces
- KBAR-0080 (story_list & story_update) — separate story, delivered next

## Architecture Notes

**Return type design decision:**
The extended `kb_get_story` return type uses optional keys:
```typescript
// When include_artifacts: false (default) — backward compatible
{ story: Story | null, message: string }

// When include_artifacts: true or include_dependencies: true
{ story: Story | null, artifacts?: StoryArtifact[], dependencies?: StoryDependency[], message: string }
```

Use `KbGetStoryResultSchema = z.object({ story: ..., artifacts: z.array(...).optional(), dependencies: z.array(...).optional(), message: z.string() })` with `z.infer<>` for the return type.

**Query design (max 3 DB round-trips total):**
1. Primary story query: `SELECT * FROM stories WHERE story_id = $1 LIMIT 1`
2. Conditional artifact query (only when `include_artifacts: true` AND story != null): `SELECT * FROM story_artifacts WHERE story_id = $1`
3. Conditional dependency query (only when `include_dependencies: true` AND story != null): `SELECT * FROM story_dependencies WHERE story_id = $1 OR target_story_id = $1`

**Short-circuit guard:**
When the primary story query returns null, immediately return `{ story: null, artifacts: [], dependencies: [], message: '...' }` without executing queries 2 or 3. This is both correct behavior (no story to load relations for) and a performance guard.

**Tool schema update strategy:**
If `zodToMcpSchema(KbGetStoryInputSchema)` is called in `kbGetStoryToolDefinition`, the JSON Schema for the tool definition will automatically reflect the new optional fields after ST-1. The description string in `kbGetStoryToolDefinition` must be manually updated to document the new parameters.
