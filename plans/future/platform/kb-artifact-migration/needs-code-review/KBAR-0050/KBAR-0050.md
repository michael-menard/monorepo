# KBAR-0050: CLI Sync Commands

## Context

The `packages/backend/kbar-sync/` package contains a complete set of sync functions implemented in KBAR-0030 (story sync) and KBAR-0040 (artifact sync). These functions implement bidirectional filesystem-database synchronization with checksum-based idempotency, per-item fault isolation, sync event tracking, and batch operations with checkpoint resumption.

However, these functions have no command-line interface. Running a sync operation currently requires writing custom TypeScript that imports from `@repo/kbar-sync`, connects to the database, and calls the sync functions directly. There is no way for a developer or automation script to:
- Sync a single story from the terminal
- Run a full epic sync before a deployment
- Perform a dry-run to preview what would be synced
- Run an incremental sync as part of a CI pipeline

**Implementation gate**: KBAR-0040 is currently in UAT. KBAR-0050 PM and elaboration artifacts can be prepared now, but implementation must not begin until KBAR-0040 is merged to `main` and `@repo/kbar-sync` package exports are confirmed stable.

---

## Goal

Create two CLI commands as executable TypeScript scripts in `packages/backend/kbar-sync/scripts/`:

1. **`sync:story`** ظ¤ Syncs a single story (story metadata + optionally artifacts) to the database. Accepts `--story-id`, `--story-dir`, `--dry-run`, `--artifacts`, `--force`, `--from-db`, `--check-conflicts`, `--artifact-file`, `--artifact-type`, `--verbose` flags.

2. **`sync:epic`** ظ¤ Syncs all stories in an epic (or all epics in a base directory). Accepts `--base-dir`, `--epic`, `--artifact-type`, `--dry-run`, `--force`, `--checkpoint`, `--verbose` flags for incremental batch operations.

Both commands delegate entirely to `@repo/kbar-sync` functions. The CLI layer is responsible for: argument parsing, Zod input validation, human-readable progress output, error reporting, and exit code management.

The commands are registered as `pnpm` scripts in `packages/backend/kbar-sync/package.json` so they can be invoked as:
```
pnpm --filter @repo/kbar-sync run sync:story -- --story-id KBAR-0050 --dry-run
pnpm --filter @repo/kbar-sync run sync:epic -- --base-dir plans/future/platform --dry-run
```

- **Web UI or REST API for sync operations** ظ¤ out of scope; CLI only
- **Conflict resolution** ظ¤ detect conflicts and report them; resolution (filesystem_wins / database_wins) is out of scope for KBAR-0050
- **Schema changes** ظ¤ no DDL changes; `kbar.*` tables are protected (KBAR-0010 output)

- **Automated pre-commit or CI integration** ظ¤ KBAR-0050 delivers the commands; integration into CI pipelines is a separate concern
- **Interactive prompts** ظ¤ CLI is non-interactive (suitable for CI); no inquirer or readline prompts
- **Progress bars or terminal animations** ظ¤ plain text output only; structured for both human and script consumption

## Architecture Notes

### Thin CLI Layer (Non-Negotiable)

The CLI scripts follow the **parse ظْ validate ظْ delegate ظْ print** pattern with zero business logic:

```
process.argv
  ظْ parseRawArgs()       # minimal string parsing: process.argv.slice(2)
  ظْ SyncStoryCLIOptionsSchema.safeParse()  # Zod validation with human error
  ظْ syncStoryToDatabase(validatedInput)    # delegate to @repo/kbar-sync
  ظْ formatAndPrint(result)                 # human-readable output to stdout
```

No Drizzle queries, no filesystem scanning (for stories), and no checksum computation should exist in the CLI scripts ظ¤ all of this is handled by `@repo/kbar-sync` functions and (for dry-run) a dedicated read-only helper.

### Dry-Run Implementation Strategy

**Approach 2 is mandatory** ظ¤ confirmed that `SyncStoryToDatabaseInputSchema` (in `packages/backend/kbar-sync/src/__types__/index.ts`) does not include a `dryRun` or `force` parameter. Approach 1 is not viable.

**Required implementation** (Approach 2): CLI implements a read-only checksum comparison path:
- Compute filesystem checksum for target story/artifacts using the same checksum utility as `@repo/kbar-sync`
- Query `kbar.stories` / `kbar.artifacts` for current DB checksum (single batch query via `@repo/db`)
- Compare and report ظ¤ **zero sync function calls** in dry-run mode

This approach must use a **single batch query** for `sync:epic --dry-run` to avoid N+1 (lesson from KBAR-0040).

**`--force` flag**: Implemented by having the CLI skip its pre-sync checksum pre-check and always call `syncStoryToDatabase` ظ¤ which performs an upsert internally. There is no `force` parameter to pass to `syncStoryToDatabase`.

### Path Security (Defense in Depth)

The CLI layer validates `--story-dir`, `--base-dir`, and `--artifact-file` against:
1. Path traversal: reject any path containing `..` components
2. Symlinks: `fs.lstatSync(path).isSymbolicLink()` ظ¤ reject symlinks
3. Absolute paths outside working directory: restrict to relative or explicitly allowlisted base paths

This is defense-in-depth ظ¤ `@repo/kbar-sync` also validates internally (KBAR-0030 pattern). The CLI-layer validation ensures clear error messages at the boundary.
