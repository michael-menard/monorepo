# KBAR-0200: Update `knowledge-context-loader` to Use `artifact_search` for Semantic Pattern Discovery

## Context

The KBAR epic's Phase 4 delivered the `artifact_search` MCP tool (KBAR-0130) — semantic search over artifacts indexed in the Knowledge Base. Phase 5 migrates all workflow agents to use the new artifact tools. This story (KBAR-0200) is the final Phase 5 agent migration: updating the `knowledge-context-loader` to leverage `artifact_search` for discovering relevant past solutions and patterns before story creation, elaboration, or implementation begins.

The `knowledge-context-loader` (v2.1.0) currently loads context through two mechanisms:
1. `kb_search` with `tags: ["lesson-learned"]` — retrieves lesson entries about past failures, blockers, and patterns
2. Filesystem read of `ADR-LOG.md` — retrieves active Architecture Decision Records

These mechanisms are effective for lesson-level and policy-level knowledge, but do not surface the actual implementation artifacts from past stories (CHECKPOINT, PLAN, EVIDENCE, REVIEW, etc.). An agent working on a new story about MCP tool handlers would benefit from finding the EVIDENCE and PLAN artifacts from KBAR-0110 or KBAR-0130 — concrete implementations that demonstrate the patterns to follow.

**Problem**: The knowledge-context-loader cannot currently surface past implementation artifacts by semantic similarity. It relies solely on lessons (KB `lesson-learned` entries written by authors who explicitly chose to capture them) and hardcoded ADRs. If no lesson was written for a particular pattern, it goes undiscovered.

**Proposed solution**: Add a new "Pattern Discovery" query phase (Phase 1.5, after the existing lesson queries) to the knowledge-context-loader that uses `artifact_search` with domain-scoped natural language queries to find relevant artifacts from past stories. Results are surfaced as a new `pattern_discovery` section in the knowledge context output alongside the existing `lessons_learned` entries. Fallback is preserved: if `artifact_search` is unavailable or returns no results, the loader continues with lessons-only context and a warning.

The change spans two files: (1) `.claude/agents/knowledge-context-loader.agent.md` (documenting the new query phase for agent-mode invocations) and (2) `packages/backend/orchestrator/src/nodes/reality/load-knowledge-context.ts` (extending `KnowledgeContextConfigSchema` with an optional `artifactSearchFn` parameter so programmatic LangGraph orchestrator workflows also benefit). A third file, `packages/backend/orchestrator/src/artifacts/knowledge-context.ts`, gains an optional `pattern_discovery` field.

**Implementation note**: KBAR-0130 (`artifact_search` tool) must be in UAT or completed status before implementation begins. KBAR-0190 (QA & fix agents migration) must also be complete.

## Goal

Enable the `knowledge-context-loader` to surface relevant past implementation artifacts (EVIDENCE, PLAN, CHECKPOINT artifacts from past stories) via semantic `artifact_search` queries, providing agents with concrete past-implementation context in addition to the existing lessons-learned and ADR knowledge.

## Non-Goals

- Replacing the existing `kb_search` lesson-learned queries — they remain and continue to run as-is
- Updating the `ADR-LOG.md` filesystem read logic — ADR loading is unchanged
- Migrating other agents in the KBAR epic — KBAR-0190 covers QA/fix agents, KBAR-0170 covers execute/worker agents, KBAR-0180 covers code review agents
- Implementing the `artifact_search` tool — that is KBAR-0130 (Phase 4)
- Implementing `artifact_write` — that is KBAR-0110 (Phase 4)
- Changing the content structure of `KnowledgeContext` in ways that break existing consumers (`dev-execute-leader`, `dev-proof-leader`, `qa-verify-verification-leader`)
- Updating orchestrator commands — that is KBAR-0210 (next story in Phase 5)
- Adding E2E test coverage for the knowledge-context-loader
- Protected features: `kb_search` lesson-learned query phases (Phase 1 in current agent) must remain unchanged and unmodified

## Reuse Plan

- **DI pattern**: Mirror the existing `getLessonsFromKB()` function signature for `getPatternDiscoveryFromKB()`. Pass `artifactSearchFn` as an optional field in `kbDeps` within `KnowledgeContextConfigSchema`, not imported directly. Follow lines 378–492 of `load-knowledge-context.ts` as the template.

- **Schema extension**: Use `KnowledgeContextConfigSchema` with `kbDeps: z.object({ ..., artifactSearchFn: z.function().args(z.unknown()).returns(z.promise(z.unknown())).optional() }).optional()` — Zod, not TypeScript interface per CLAUDE.md.

- **Query construction template**: Use `"{story_domain} {story_scope} implementation"` as the primary query string, targeting `artifact_type: "evidence"` with `limit: 5`. Use `"{story_domain} {story_scope} plan solution"` for `artifact_type: "plan"`. Mirror how `getLessonsFromKB()` incorporates domain in its query strings.

- **Fallback pattern**: Use the same try/catch + warning pattern from `getLessonsFromKB()` (lines 483–491 in `load-knowledge-context.ts`): on failure, push to `warnings` array and return empty results. Do not rethrow.

- **Agent frontmatter template**: Follow KBAR-0160 `kb_tools` frontmatter update pattern — add `artifact_search` to the existing list.

- **Existing query block structure**: The three `kb_search` call blocks in Phase 1 of `knowledge-context-loader.agent.md` are the structural model for the new `artifact_search` call blocks in Phase 1.5.

## Architecture Notes

**Scope confirmed: Option B (both markdown + TypeScript)**

Dev Feasibility confirmed that the TypeScript LangGraph node (`load-knowledge-context.ts`) is actively used in the production orchestrator graph:
- `nodes/story/attack.ts` imports `KnowledgeContext` type from `load-knowledge-context.ts`
- `loadKnowledgeContextNode` and `createKnowledgeContextNode` are exported from `nodes/reality/index.ts`
- The node is registered in the LangGraph pipeline

Therefore both the agent markdown AND the TypeScript node must be updated. A markdown-only update (Option A) would leave the programmatic orchestrator path without `artifact_search` support.

**Output schema extension**: The `KnowledgeContextSchema` in `artifacts/knowledge-context.ts` gains an optional `pattern_discovery` field:
```
pattern_discovery: z.array(PatternDiscoveryResultSchema).default([])
```
where `PatternDiscoveryResultSchema` captures `{ story_id, artifact_type, content_summary, relevance_score, phase }`. Using `.default([])` ensures backward compatibility — existing consumers receive an empty array and can ignore it.

**Dual nature of knowledge context**: This story adds a third query mechanism to the loader:
1. `kb_search` with lesson-learned tags → `lessons_learned` section (existing, unchanged)
2. Filesystem read of ADR-LOG.md → `architecture_decisions` section (existing, unchanged)
3. `artifact_search` with domain-scoped queries → `pattern_discovery` section (new)

## Infrastructure Notes

No infrastructure changes. No new packages. No database migrations. No deployment changes required. This is a pure code + documentation change within existing packages.
