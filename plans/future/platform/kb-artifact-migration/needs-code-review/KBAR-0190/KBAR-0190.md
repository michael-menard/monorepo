# KBAR-0190: Migrate `qa-verify-*` and `dev-fix-fix-leader` Agents to Use `artifact_write` for Dual-Write Artifact Persistence

## Context

**Epic**: KBAR — KB Artifact Migration (Phase 5: Agent Migration)

The KBAR epic delivers the `artifact_write` MCP tool (KBAR-0110, in UAT) — a dual-write mechanism that simultaneously writes a YAML artifact to the filesystem and indexes it in the KB for semantic search. Phase 5 (KBAR-0160 through KBAR-0220) migrates all workflow agents to use `artifact_write` instead of the current KB-only `kb_write_artifact` calls.

**This story (KBAR-0190)** is the fourth Phase 5 migration, targeting the QA verification and fix agents. It depends on:
- KBAR-0110 (Phase 4): `artifact_write` tool — in UAT
- KBAR-0160 (Phase 5): dev-setup-leader and dev-plan-leader migration — ready-for-qa
- KBAR-0170 (Phase 5): dev-execute-leader, backend-coder, frontend-coder — pending (dependency)
- KBAR-0180 (Phase 5): code-review agents — pending (dependency)

**Current state of affected agents**:
- `qa-verify-verification-leader` (v4.1.0): KB tools include `kb_write_artifact`. Step 9 writes the `verification` artifact to KB only — no filesystem copy.
- `qa-verify-completion-leader` (v3.3.0): KB tools include `kb_write_artifact`. PASS Step 2 and FAIL Step 2 update the `verification` artifact in KB only.
- `dev-fix-fix-leader` (v5.0.0): KB tools include `kb_write_artifact`. Step 6 writes updated `evidence` artifact to KB only; Step 7 increments `review` artifact in KB only.
- `qa-verify-setup-leader` (v5.1.0): KB tools are `kb_search` only — **no `kb_write_artifact` calls**. Explicitly a non-goal for this migration.

**Problem**: The three affected agents write QA-VERIFY, EVIDENCE, and REVIEW artifacts exclusively to the KB database. These artifacts are not accessible via the filesystem without a KB query. Additionally, `dev-documentation-leader` and `dev-verification-leader` both read `FIX-CONTEXT.yaml` from `_implementation/FIX-CONTEXT.yaml` on the filesystem — after `dev-fix-fix-leader` migration, its `fix_summary` artifact write must set `file_path` explicitly so downstream filesystem reads continue to work.

## Goal

Update `qa-verify-verification-leader`, `qa-verify-completion-leader`, and `dev-fix-fix-leader` to replace `kb_write_artifact` write calls with `artifact_write` calls. Pass explicit `file_path` arguments pointing to canonical locations. Update agent frontmatter `kb_tools` sections. Document graceful failure semantics. Confirm `qa-verify-setup-leader` is explicitly a non-goal.

## Non-Goals

- Migrating `dev-execute-leader`, `backend-coder`, `frontend-coder` — that is KBAR-0170
- Migrating code-review agents — that is KBAR-0180
- Migrating `knowledge-context-loader` to use `artifact_search` — that is KBAR-0200
- Implementing the `artifact_write` tool itself — that is KBAR-0110 (in UAT)
- Changing `qa-verify-setup-leader.agent.md` — **explicitly a non-goal** (agent has no `kb_write_artifact` calls; AC-9 resolved as Option B)
- Changing the content structure of `verification`, `evidence`, `review`, or `fix_summary` artifacts
- Changing `kb_read_artifact` usage in any agent — only the write side migrates
- Touching any TypeScript source files — agent markdown files only
- Adding automated tests for agent behavior — validated via E2E workflow runs

## Architecture Notes

### Dual-Write Pattern (from KBAR-0110)

`artifact_write` performs two operations:
1. **File write** (primary, blocking): writes the YAML content to `file_path` on disk. `mkdir -p` is called automatically. Returns `file_written: true`.
2. **KB write** (secondary, best-effort): indexes the artifact in the `storyArtifacts` table. If KB write fails, returns `kb_write_warning` (non-null string). Does NOT throw. Does NOT block the calling agent.

**Migration pattern for each call site**:
```javascript
// BEFORE (kb-only write, no filesystem)
kb_write_artifact({
  story_id: "{STORY_ID}",
  artifact_type: "verification",
  phase: "qa_verification",
  iteration: 0,
  content: { /* QA-VERIFY structure */ }
})

// AFTER (dual-write: file first, KB best-effort)
artifact_write({
  story_id: "{STORY_ID}",
  artifact_type: "verification",
  phase: "qa_verification",
  iteration: 0,
  file_path: "{FEATURE_DIR}/UAT/{STORY_ID}/QA-VERIFY.yaml",
  content: { /* QA-VERIFY structure — unchanged */ }
})
// Response: { file_path, file_written: true, kb_artifact_id?, kb_write_warning? }
// If kb_write_warning is present: log it, continue. Do NOT block.
```

### FIX-CONTEXT.yaml Filesystem Dependency (Critical)

`dev-documentation-leader` and `dev-verification-leader` both read `FIX-CONTEXT.yaml` from the filesystem at `{FEATURE_DIR}/in-progress/{STORY_ID}/_implementation/FIX-CONTEXT.yaml`. Currently, `dev-fix-fix-leader` writes the `fix_summary` artifact only to KB via `kb_write_artifact`, meaning downstream agents cannot read it without KB access.

After migration, `dev-fix-fix-leader`'s `fix_summary` write (if any) and the `evidence` + `review` artifact writes (Steps 6 and 7) must include explicit `file_path` arguments. This ensures file-system consumers continue to work regardless of KB availability.

### Protected: Read Side Unchanged

`kb_read_artifact` calls in all three agents remain unchanged. This migration is write-side only. The read path stays DB-first as per KBAR decisions.

1. `KBAR-0170` (execute/worker agents) is merged — establishes patterns for evidence artifact migration
2. `KBAR-0180` (code review agents) is merged — avoids conflicts on agent file patterns; establishes review artifact migration pattern
