# Story MCP Tools — Integration Tests via `handleToolCall`

## Context

The five story MCP tools (`kb_get_story`, `kb_list_stories`, `kb_update_story_status`, `kb_update_story`, `kb_get_next_story`) are fully implemented in `apps/api/knowledge-base/src/mcp-server/tool-handlers.ts` (landed via KBAR-0070 and KBAR-0080). Unit-level tests covering handler business logic exist in `story-tools.test.ts` (extended via KBAR-0090 to include `handleKbGetNextStory` unit tests and `mcp-integration.test.ts` hygiene fixes).

However, no tests exist that exercise these tools through the `handleToolCall` dispatch layer — the MCP routing and error-wrapping infrastructure that connects the protocol wire format to individual handlers. The existing `mcp-integration.test.ts` tests the dispatch layer for core KB CRUD tools (`kb_add`, `kb_get`, `kb_list`); story tools have no equivalent integration coverage.

This story closes that gap by creating `story-tools-integration.test.ts` — a new file that calls `handleToolCall('kb_get_story', ...)` etc., mirroring the structure of the "Tool Invocation via Server" and "Authorization Integration" sections of `mcp-integration.test.ts`.

**Verified state as of 2026-02-24:**
- `story-tools.test.ts` exists and covers all 5 handlers at the unit level (`vi.mock` + direct handler calls)
- `mcp-integration.test.ts` has 53 tools registered and tested for discovery
- `access-control.ts` confirms all story tools allow `pm`, `dev`, `qa`, and `all` roles
- No `story-tools-integration.test.ts` exists yet — this story creates it

## Goal

Write integration tests at the `handleToolCall` dispatch layer for all five story MCP tools. These tests verify:

1. Tool name routing correctly dispatches to story handlers
2. Zod validation errors bubble through `handleToolCall` with `isError: true` and `code: 'VALIDATION_ERROR'`
3. DB errors are sanitized before reaching the caller (no credentials in error messages)
4. Authorization roles are accessible to all non-admin roles (`dev`, `all`, `pm`, `qa`)
5. Happy-path responses are JSON-serializable and structurally correct via the MCP content array

## Non-Goals

- Do NOT re-implement unit tests already in `story-tools.test.ts` — those cover business logic; this story covers dispatch routing
- Do NOT test DB logic (e.g., the terminal-state guard SQL) — mock at the `story-crud-operations.js` boundary
- Do NOT add new MCP tool handlers, schemas, or access-control entries
- Do NOT add migrations, new DB schema columns, or Drizzle schema changes
- Do NOT add REST HTTP endpoints
- Do NOT add Playwright E2E tests (no UI surface)
- Do NOT add testcontainers or real-PostgreSQL integration tests
- Do NOT modify `mcp-integration.test.ts` tool count or names list (already correct at 53 as of KBAR-0090)

## Architecture Notes

The `handleToolCall` function is the MCP dispatch layer — it receives a tool name string, validates inputs via Zod, routes to the appropriate handler, and wraps errors in a consistent MCP-compatible response format. Testing at this layer (rather than calling handlers directly) provides coverage of:

- Tool name dispatch routing (correct handler invoked for correct name)
- Zod input validation errors (schema rejection before handler is called)
- Error wrapping and sanitization (DB errors do not leak credentials)
- Authorization enforcement (`ToolCallContext` with `agent_role` tested against access-control matrix)

This is the same integration layer tested for core KB tools in `mcp-integration.test.ts`. The story tools were added in KBAR-0080 but only received unit tests (direct handler calls), not `handleToolCall`-level integration tests.

**Access control verification (confirmed in `access-control.ts`):**
All five story tools are registered as non-admin: `kb_get_story`, `kb_list_stories`, `kb_update_story_status`, `kb_update_story`, `kb_get_next_story` all use `new Set(['pm', 'dev', 'qa', 'all'])`. No FORBIDDEN responses are expected for any role when testing these tools.
