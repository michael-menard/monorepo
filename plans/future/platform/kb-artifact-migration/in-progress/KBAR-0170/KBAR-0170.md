# Migrate `dev-execute-leader`, `backend-coder`, and `frontend-coder` to Use `artifact_write` for Dual-Write Artifact Persistence

## Context

The KBAR epic's Phase 4 delivers the `artifact_write` MCP tool (KBAR-0110) — a dual-write mechanism that simultaneously writes a YAML artifact to the filesystem and indexes it in the KB for semantic search. Phase 5 migrates all workflow agents to use `artifact_write` instead of the current KB-only `kb_write_artifact` calls. KBAR-0160 migrates the setup and planning agents (`dev-setup-leader`, `dev-plan-leader`). KBAR-0170 migrates the execution agents: `dev-execute-leader`, `dev-implement-backend-coder`, and `dev-implement-frontend-coder`.

**Problem**: `dev-execute-leader` writes EVIDENCE.yaml and updates CHECKPOINT via `kb_write_artifact` — these artifacts exist only in the KB database. There is no `_implementation/EVIDENCE.yaml` file on the filesystem. This limits human observability and prevents the deferred-write fallback pattern from functioning correctly. `backend-coder` and `frontend-coder` write BACKEND-LOG.md and FRONTEND-LOG.md directly to the filesystem via the `Write` tool (not via `kb_write_artifact`) — they are not KB-indexed. The migration determines the correct scope for these coder agents.

**Proposed solution**: Update `dev-execute-leader.agent.md` to replace its two `kb_write_artifact` calls (EVIDENCE write in Step 8, CHECKPOINT update in Step 9) with `artifact_write` calls using explicit `file_path` arguments. Update the frontmatter `kb_tools`. Document graceful KB failure semantics. For `backend-coder` and `frontend-coder`: scope is resolved as Option (b) — direct `Write` calls remain unchanged for BACKEND-LOG and FRONTEND-LOG, avoiding KB round-trip overhead on high-frequency writes. Non-negotiables in both coder agents are updated to document this explicit decision.

**Baseline Note**: The baseline (2026-02-13) predates Phase 4 and 5 agent migration work. At time of story generation: KBAR-0110 (`artifact_write` tool) is in UAT; KBAR-0120 (artifact tool unit tests) is in UAT; KBAR-0130 (`artifact_search`) is in ready-for-qa; KBAR-0160 is in needs-code-review. The `artifact_write` contract is stable and verified from KBAR-0110 UAT.

## Goal

Update `dev-execute-leader.agent.md` to replace `kb_write_artifact` write calls (Steps 8-10) with `artifact_write` calls, using explicit `file_path` arguments pointing to canonical `_implementation/` paths. Update frontmatter `kb_tools`. Document graceful KB write failure semantics. For `backend-coder` and `frontend-coder`: update non-negotiables to explicitly document that direct `Write` tool calls for log files remain unchanged (AC-6 scope decision, Option b).

- KB-indexing BACKEND-LOG and FRONTEND-LOG (performance risk; belongs in separate story if desired)

## Architecture Notes

**AC-6 Scope Decision — Option (b): No log migration**

The index entry describes migrating backend-coder and frontend-coder "to use `artifact_write` for EVIDENCE, logs." However:
1. These agents do NOT write EVIDENCE — that is `dev-execute-leader`'s output.
2. These agents do NOT use `kb_write_artifact` — they write logs via the `Write` tool directly.
3. BACKEND-LOG and FRONTEND-LOG are high-frequency writes (5-10 per story per implementation chunk). Adding `artifact_write` KB round-trip overhead per write would meaningfully slow implementation phases.
4. Direct `Write` calls already achieve file persistence — the primary goal of this epic.

Option (b) is the correct choice: document the decision explicitly in non-negotiables, preserve direct `Write` calls, and scope KB-indexing of logs to a separate story if desired.

**Dual-write semantics (for execute-leader EVIDENCE and CHECKPOINT)**:
- File write is primary (blocking): if file write fails, `artifact_write` returns error
- KB write is secondary (best-effort): if KB write fails, `artifact_write` returns `file_written: true` with `kb_write_warning` — execute phase MUST NOT block
- `kb_read_artifact` calls in Steps 1-2 are unchanged (read path stays DB-first)
