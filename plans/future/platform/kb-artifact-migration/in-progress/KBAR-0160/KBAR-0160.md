# Migrate `dev-setup-leader` and `dev-plan-leader` to Use `artifact_write` for Dual-Write Artifact Persistence

## Context

The KBAR epic's Phase 4 delivers the `artifact_write` MCP tool (KBAR-0110) — a dual-write mechanism that simultaneously writes a YAML artifact to the filesystem and indexes it in the KB for semantic search. Phase 5 (this story and its siblings KBAR-0170 through KBAR-0220) migrates all workflow agents to use `artifact_write` instead of the current KB-only `kb_write_artifact` calls.

`dev-setup-leader` (v5.1.0) currently writes CHECKPOINT and SCOPE artifacts via `kb_write_artifact` (KB-only, no file write). `dev-plan-leader` (v1.2.0) similarly writes PLAN and updates CHECKPOINT via `kb_write_artifact`. Both agents have non-negotiables prohibiting direct file writes. After this migration, both agents will use `artifact_write`, which writes the file to `_implementation/` AND indexes it in the KB — providing file-system persistence for human inspection alongside KB-searchability.

**Problem**: Artifacts written by `dev-setup-leader` and `dev-plan-leader` exist only in the KB database. They are not accessible via file system (no `_implementation/CHECKPOINT.yaml` or `_implementation/PLAN.yaml`). This limits observability (humans cannot inspect artifacts without a KB query) and prevents the deferred-write fallback pattern from functioning correctly for these artifacts.

**Baseline Note**: The baseline (2026-02-13) predates Phase 4 agent migration work (KBAR-0110 through KBAR-0150). Those stories are now actively in-progress or ready-to-work. The baseline correctly documents the `kb_write_artifact` tool, which is the mechanism both agents currently use.

## Goal

Update `dev-setup-leader.agent.md` and `dev-plan-leader.agent.md` to replace `kb_write_artifact` calls (write operations only) with `artifact_write` calls. Pass explicit `file_path` arguments pointing to canonical `_implementation/` paths. Update agent frontmatter `kb_tools`. Update `dev-setup-leader` non-negotiables to reflect that `artifact_write` handles directory creation. Document graceful KB write failure semantics in both agents.

## Non-Goals

- Migrating `dev-execute-leader`, `backend-coder`, `frontend-coder` to use `artifact_write` (KBAR-0170)
- Migrating code review agents (KBAR-0180)
- Migrating QA agents (KBAR-0190)
- Implementing the `artifact_write` tool itself (KBAR-0110, Phase 4 dependency)
- Changing the content structure of CHECKPOINT, SCOPE, or PLAN artifacts
- Changing `kb_read_artifact` usage in either agent
- Touching any TypeScript source files in this story (agent markdown files only)
- Adding automated tests for agent behavior (agents are markdown; behavior is validated via E2E workflow runs)
- Removing `kb_write_artifact` from the KB MCP server (it remains and is used by other agents)

## Architecture Notes

**Dual-Write Semantics** (critical for agent documentation):
- File write is **primary** (blocking): `artifact_write` writes YAML to `_implementation/` before any KB operation
- KB write is **secondary** (best-effort): if KB write fails, `artifact_write` returns `file_written: true` + `kb_write_warning`
- Agents must document: KB write failure produces a warning, not a blocking error — setup/planning phase continues
- `kb_read_artifact` remains the read path — it reads from the KB database (not from the filesystem)

**File Path Convention** (Option A — explicit, recommended):
The agents should pass explicit `file_path` arguments rather than relying on `artifact_write`'s internal canonical path computation. This provides agent-side transparency and observability:

```
CHECKPOINT: {FEATURE_DIR}/in-progress/{STORY_ID}/_implementation/CHECKPOINT.yaml
SCOPE:      {FEATURE_DIR}/in-progress/{STORY_ID}/_implementation/SCOPE.yaml
PLAN:       {FEATURE_DIR}/in-progress/{STORY_ID}/_implementation/PLAN.yaml
```

**Non-Negotiable Update** (`dev-setup-leader` only):
The current constraint "Do NOT create `_implementation/` directories" must be updated. The new `artifact_write` tool creates the `_implementation/` directory as a side effect of the file write. The updated constraint should read: "MUST write artifacts via `artifact_write` (not direct file writes or `kb_write_artifact`)."

**Backward Compatibility**:
- `kb_write_artifact` remains available in the KB MCP server — other agents not in this story's scope continue to use it unchanged
- Stories already in-progress that have artifacts in KB (written by the old `kb_write_artifact`) can still be read via `kb_read_artifact` — the read path is database-first and works regardless of which tool wrote the artifact
- Only new artifact writes (after KBAR-0160 lands) will produce dual-write output (file + KB)
