# KFMB-5030 — Migrate _implementation/ Command Orchestrators to KB Artifact References

## Context

The implementation pipeline is driven by a set of command orchestrators (`.claude/commands/`) and shell scripts (`scripts/`) that coordinate the flow of stories through elaboration, implementation, code review, QA, and fix cycles. These orchestrators currently reference `_implementation/` artifacts by filesystem path:

- Shell scripts check for `_implementation/ELAB.yaml` to determine elaboration readiness (`is_elaborated()`), `_implementation/EVIDENCE.yaml` to detect completion (`is_implemented()`), and `_implementation/REVIEW.yaml` for review state (`is_reviewed()`)
- Command files validate artifact existence by checking filesystem locations via `--requires=_implementation/ELAB.yaml` and `--requires=_implementation/VERIFICATION.yaml`
- Agent spawn prompts in some orchestrators pass `_implementation/` filesystem paths rather than `story_id` as the primary artifact context key

KFMB-5010 and KFMB-5020 will have migrated all writer and reader agents to use `kb_write_artifact` and `kb_read_artifact` respectively, making the KB the authoritative store for all `_implementation/` artifacts. After those stories complete, the filesystem paths become stale references that no longer reflect where artifacts actually live.

**Phase in migration plan:** Phase 4 — Artifact Migration. This story is on the critical path (KFMB-1010 → KFMB-1030 → KFMB-2040 → KFMB-5010 → KFMB-5020 → **KFMB-5030** → KFMB-6010 → KFMB-6020).

**Baseline reference:** `plans/baselines/BASELINE-REALITY-2026-02-13.md`. The baseline records `packages/backend/orchestrator/src/artifacts/` as existing KB artifact schema infrastructure — the Zod-validated artifact types (checkpoint, scope, plan, evidence, review, qa-verify, audit-findings) are the canonical types this story's orchestrators will reference post-migration.

---

## Goal

Migrate the command orchestrators and shell scripts that drive the implementation pipeline to consume KB artifact state rather than filesystem state, so that artifact presence and story readiness are determined by KB tool calls — not by `_implementation/` directory or file existence checks.

---

## Non-Goals

- **NOT** modifying agent workers invoked by the orchestrators (e.g., `dev-setup-leader.agent.md`, `elab-analyst.agent.md`) — those are KFMB-5010 and KFMB-5020 scope
- **NOT** eliminating stage directories (`backlog/`, `elaboration/`, `ready-to-work/`, etc.) from the filesystem — that is KFMB-4010 and KFMB-4020
- **NOT** eliminating `stories.index.md` file references — that is KFMB-3020 and KFMB-3030
- **NOT** adding new MCP tool registrations to the KB server — tools are assumed to already exist from prior KFMB phases
- **NOT** removing the `_implementation/` directory structure from the repository — deferred to KFMB-6010/6020
- **NOT** migrating `_pm/` artifact reads/writes — that is KFMB-5040/5050
- **NOT** modifying `story-move.md` or `story-update.md` commands — those are KFMB-4010 scope
- **NOT** modifying `scripts/lib/resolve-plan.sh` — already KB-native, serves as the reference pattern
- **NOT** changing any production DB schemas in `packages/backend/database-schema/` or the `@repo/db` client package

## Architecture Notes

### Shell Script KB Query Architecture

Shell scripts run in bash environments outside Claude sessions — they **cannot** make MCP tool calls directly. The migration must delegate state queries to `claude -p` subprocess calls.

**Preferred strategy (Option B — batch query):** Call `kb_list_stories` once per run to fetch all story states for the plan, store results in an associative array, and use in-memory lookups for `is_elaborated()`, `is_implemented()`, `is_reviewed()`. This adds one `claude -p` call per run (not per story) and avoids the latency multiplication that per-story subprocess calls would cause in `implement-dispatcher.sh`'s polling loop.

```bash
# Pattern to follow (from resolve-plan.sh):
raw_result=$(claude -p \
  "Call kb_list_stories with plan_slug '${PLAN_SLUG}'. Output ONLY raw JSON array..." \
  --allowedTools "mcp__knowledge-base__kb_list_stories" \
  --output-format text 2>/dev/null) || true
```

**KB state mapping for story readiness:**
- Elaborated: `state == "ready"` or later (ready_to_work, in_progress, etc.)
- Implemented: `state == "ready_for_review"` or later (a story in `in_progress` is mid-flight, not yet implemented — using it as the `is_implemented()` signal would produce false positives)
- Reviewed: `state == "in_review"`

### KB Artifact Existence Check (Precondition Check Pattern)

```
# Template from dev-code-review.md Phase 0 (already KB-native):
artifact = kb_read_artifact({ story_id, artifact_type: "evidence" })
if artifact is null → FAIL with "evidence artifact not found in KB"
```

Apply the same pattern in `precondition-check.md` for `elaboration` and `verification` artifact types.

### Frontmatter kb_artifacts Atomicity (KBAR-0190)

Per lesson KBAR-0190: frontmatter `kb_artifacts` (or `kb_tools`) declarations in command files must be updated in the same commit as the call-site migration. All modified command files must have their `kb_artifacts` frontmatter updated to accurately list artifact types read/written after migration.

### Graceful Degradation for KB Unavailability

Scripts must have fallback behavior when `claude -p` KB subprocess fails (KB unavailable). Recommended: treat story as not-ready and skip (log to failure file), consistent with the "deferred write" pattern documented in the knowledge base. Do not crash the entire pipeline run.
