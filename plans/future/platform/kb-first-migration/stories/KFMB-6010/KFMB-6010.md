# KFMB-6010 — Script Modernization: Remove Filesystem Fallbacks and Consolidate KB-First Helpers

## Context

The KB-First Migration plan has, across Phases 3 and 4, progressively moved all automation script logic to KB-first patterns with graceful filesystem fallbacks. By the time KFMB-6010 starts, every script (`generate-stories.sh`, `implement-stories.sh`, `implement-dispatcher.sh`, `sync-work-order.py`) and shared library (`resolve-plan.sh`) will query the KB as the primary source of truth, with filesystem directory scans retained as resilience fallbacks for KB unavailability.

**Problem**: The fallback paths add maintenance burden and create split-brain risk — a script running during a KB outage can silently produce state based on stale filesystem data. As KB write stability is confirmed by the completion of KFMB-4010 and KFMB-4020 (which ensure the `stories.state` DB column is reliably written on every state transition), the filesystem fallbacks become safety-critical scaffolding that no longer needs to hold up the structure.

Additionally, the `kb_state_query()` bash helper function is currently duplicated across `generate-stories.sh` and `implement-stories.sh` (both added the same function independently in KFMB-4020). KFMB-4020 ELAB explicitly identified extraction to `scripts/lib/kb-state.sh` as a KFMB-6010 candidate (OPP-2).

Finally, `sync-work-order.py` has two dormant functions — `check_untracked_stories()` and `scan_all_index_stories()` — that became no-ops after KFMB-3030 removed the `*stories.index.md` glob from `discover_files()`. These should either be replaced with KB queries or removed cleanly.

**Dependency gate**: KFMB-6010 must not begin implementation until KFMB-3030, KFMB-4020, and KFMB-5030 are all at `needs-code-review` or later. Their script changes must be merged before fallbacks can safely be removed.

## Goal

After this story, all automation scripts operate against the KB exclusively for story state and artifact detection — no filesystem state reads for story state or artifact readiness. A shared `scripts/lib/kb-state.sh` library eliminates the duplicated `kb_state_query()` function. Dormant `sync-work-order.py` functions are cleaned up. KB unavailability produces a clear non-zero exit rather than silent fallback behavior.

## Non-Goals

- Do not remove `scan_directories()` or `scan_proof_files()` from `sync-work-order.py` — these detect real filesystem state (worktrees, PROOF files) that the KB does not replicate.
- Do not modify any `.claude/commands/` files — command file migration is KFMB-5030 and KFMB-4020 scope.
- Do not modify `packages/backend/database-schema/` or `@repo/db` client API — protected features.
- Do not add new MCP tool registrations to the KB server.
- Do not update usage documentation or comments in scripts that refer to deprecated filesystem behavior — that is KFMB-6020 scope.
- Do not delete `stories.index.md` files — deferred to KFMB-6020.
- Do not remove the `_implementation/` directory structure — deferred to KFMB-6020.
- Do not handle `sync-work-order.py` work order file update logic (`WORK-ORDER*.md` modifications) — out of scope.

## Architecture Notes

### kb-state.sh Library Design

The new `scripts/lib/kb-state.sh` follows the same structural pattern as `scripts/lib/resolve-plan.sh`:

```bash
#!/usr/bin/env bash
# Shared KB state query helpers for automation scripts.
# Source into calling scripts with: source "$(dirname "$0")/lib/kb-state.sh"

kb_state_query() {
  local story_id="$1"
  local raw
  raw=$(claude -p \
    "Call kb_get_story with story_id '${story_id}'. Output ONLY raw JSON with key: state. If not found output {\"error\":\"not_found\"}. No markdown fences." \
    --allowedTools "mcp__knowledge-base__kb_get_story" \
    --output-format text 2>/dev/null) || true

  if [[ -z "$raw" ]]; then
    echo ""
    return 0
  fi
  local json_line
  json_line=$(echo "$raw" | grep -o '{.*}' | tail -1) || true
  echo "$json_line" | jq -r '.state // empty' 2>/dev/null || true
}
```

### Filesystem Fallback Removal Pattern

The fallback removal replaces this pattern:
```bash
# [KB-FIRST] KB lookup
local state
state=$(kb_state_query "$STORY_ID")
if [[ -n "$state" ]]; then
  # use KB state
else
  # [KB-FIRST FALLBACK] filesystem scan
  for stage_dir in ...; do
    if [[ -d "${stage_dir}/${STORY_ID}" ]]; then ...
  done
fi
```

With this hardened pattern:
```bash
# KB-exclusive lookup
local state
state=$(kb_state_query "$STORY_ID")
if [[ -z "$state" ]]; then
  echo "Error: Could not query state for $STORY_ID from KB. Is KB available at port 5433?"
  exit 1
fi
# use KB state
```

### implement-dispatcher.sh Modernization

The dispatcher currently has inline plan resolution that does not use `resolve-plan.sh`. After KFMB-6010:
1. Source `scripts/lib/resolve-plan.sh` and `scripts/lib/kb-state.sh` at the top
2. Replace the inline `grep -rl "**Plan Slug**"` + `stories.index.md` existence check with `resolve_plan "$PLAN_SLUG"`
3. Use batch `kb_list_stories` per poll cycle (not per-story `claude -p` calls) — established by KFMB-5030

### sync-work-order.py Dormant Functions

`check_untracked_stories()` and `scan_all_index_stories()` were designed to detect stories in the filesystem that are not in the index. After KFMB-3030 removed the `*stories.index.md` glob from `discover_files()`, these functions receive no input and produce empty output. The KB covers story discovery natively via `kb_list_stories`. Removal with documentation is the recommended path.
