# KFMB-1020: kb_create_story MCP Tool and CRUD Update

## Context

The Knowledge Base MCP server (`apps/api/knowledge-base/`) provides story read/update tools (`kb_get_story`, `kb_update_story`, `kb_update_story_status`) but has no tool for creating stories. Story creation today happens through filesystem YAML files synced by a separate process.

As part of the KB-First Migration, bootstrap agents need to write new stories directly into the DB without touching the filesystem. Additionally, KFMB-1010 adds content columns (`description`, `acceptance_criteria`, `non_goals`, `packages`) to the `stories` table, but neither `kb_get_story` nor `kb_update_story` currently expose those new fields — callers cannot read or write story body content through MCP after KFMB-1010 lands.

**Dependency**: KFMB-1010 (Stories Table Content Columns Migration) must land first. This story consumes the new columns — it does not own the migration.

## Goal

1. Implement `kb_create_story` as an upsert-by-`story_id` MCP tool (insert or update on conflict) with partial merge semantics — omitted fields leave existing DB values untouched.
2. Extend `kb_update_story` to accept and write the four new content fields.
3. Verify `kb_get_story` returns the four new content fields (included automatically once present in the schema row type).
4. Register `kb_create_story` end-to-end: Zod input schema, CRUD operation, MCP handler, tool definition, access control.

## Non-Goals

- Do NOT modify the `stories` table schema — KFMB-1010 owns that migration; this story consumes the new columns
- Do NOT implement story dependency insertion as part of `kb_create_story` in the initial cut — `depends_on` wiring is deferred
- Do NOT touch filesystem YAML files or sync scripts — this is a pure DB/MCP layer story
- Do NOT implement `kb_delete_story` — out of scope
- Do NOT modify `storyArtifacts` or `storyDependencies` tables — only the `stories` table row is touched
- Do NOT create barrel files (index.ts re-exports) — import directly from source files

## Architecture Notes

**Upsert pattern (partial merge):**
```typescript
// Build set object dynamically — only include fields where caller provided a value
const setFields: Partial<typeof stories.$inferInsert> = {}
if (input.feature !== undefined) setFields.feature = input.feature
if (input.title !== undefined) setFields.title = input.title
if (input.description !== undefined) setFields.description = input.description
// ... etc for all optional fields

await db.insert(stories)
  .values({ story_id: input.story_id, title: input.title, ...setFields })
  .onConflictDoUpdate({
    target: stories.storyId,
    set: setFields,
  })
```

**Content column types**: Confirm exact column types from KFMB-1010 migration before writing Zod schemas for `description`, `acceptance_criteria`, `non_goals`, `packages`. Expected: `text` for string fields, `text[]` or `jsonb` for `packages`.

**Three-place registration rule**: Every MCP tool MUST be registered in exactly three places atomically:
1. `ToolNameSchema` enum in `access-control.ts`
2. `ACCESS_MATRIX` record in `access-control.ts`
3. `toolHandlers` dispatch object in `tool-handlers.ts`
4. `getToolDefinitions()` array in `tool-schemas.ts`

Missing any one causes a runtime error or silent mismatch.

**created flag**: Detect insert vs update by checking if a pre-existing row was found:
```typescript
const existing = await db.query.stories.findFirst({ where: eq(stories.storyId, input.story_id) })
const created = existing === undefined
// ... upsert ...
return { ...result, created }
```
