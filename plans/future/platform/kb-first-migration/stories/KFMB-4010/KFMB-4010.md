# KFMB-4010: Stage Directory Elimination — story-move and story-update Commands

## Context

The `/story-move` and `/story-update` commands currently treat filesystem directory moves and frontmatter writes as primary operations, with DB writes via `shimUpdateStoryStatus` as secondary non-blocking steps. This inverts the correct architecture established by the KB-first migration plan: the KB (via `kb_update_story_status`) should be the sole source of truth for story state.

KFMB-3010 eliminates `stories.index.md` references from agents. This story goes one layer deeper: it eliminates the filesystem directory move from `/story-move` and the frontmatter status write from `/story-update`, making `kb_update_story_status` the exclusive state mutation mechanism for both commands.

This story is part of Phase 3 (Index & Stage Elimination) and runs in Group 5 alongside KFMB-3020 and KFMB-3030. KFMB-3010 must complete first as it establishes the KB-query patterns that the rewritten commands must satisfy.

**Current behavior (before):**
- `/story-move`: executes `mv {FROM_STAGE}/{STORY_ID} {TO_STAGE}/{STORY_ID}` as primary operation; DB write via `shimUpdateStoryStatus` is secondary and non-blocking
- `/story-update`: updates frontmatter in story `.md` file as primary; DB write via `shimUpdateStoryStatus` is step 3a; index update step is step 4

**Target behavior (after):**
- `/story-move`: calls `kb_update_story_status` and returns success/failure based on DB response; no directory `mv` executed
- `/story-update`: calls `kb_update_story_status` as primary action (step 1); frontmatter write step removed; index step delegated to KFMB-3020

## Goal

Rewrite `.claude/commands/story-move.md` and `.claude/commands/story-update.md` so that story state transitions are exclusively driven by `kb_update_story_status`, with no filesystem side effects. Story state is tracked only in `core.stories.state` — directory location no longer implies state.

## Non-Goals

- Do NOT modify `/precondition-check` or `/context-init` — those are KFMB-4020 scope.
- Do NOT remove the `stories.index.md` update step from `/story-update` — that is KFMB-3020 scope. If KFMB-3020 has already landed, the step will already be gone; otherwise coordinate.
- Do NOT change the `SWIM_LANE_TO_STATE` constant or its location in `packages/backend/mcp-tools/src/story-compatibility/__types__/index.ts`.
- Do NOT modify `shimUpdateStoryStatus` or any TypeScript source files.
- Do NOT update calling agents to remove their `/story-move` or `/story-update` invocations — agents continue calling the commands; only command internals change.
- Do NOT add physical directory creation or deletion logic.
- Do NOT change the worktree cleanup logic (WINT-1150) in `/story-update`.
- Do NOT address the double-write issue (agents calling `kb_update_story_status` directly AND calling `/story-move`) — that cleanup is out of scope.
- Protected: `packages/backend/orchestrator/src/artifacts/` schemas — no modification.
- Protected: `packages/backend/mcp-tools/src/` — no TypeScript changes.

- [ ] **AC-9**: The result YAML returned by `/story-move` no longer includes `from_path` and `to_path` fields (no directory move occurs). The `db_updated` field is replaced by a mandatory `kb_updated: true | false` field.

- [ ] **AC-11**: Neither command contains any reference to filesystem directory operations (`mv`, `mkdir`, directory scan, frontmatter YAML write) for story state transitions after this story completes.

## Architecture Notes

### Behavioral Contract Change

The most significant change is the inversion of the failure model in `/story-move`:

**Before**: `mv` always executes as long as the filesystem is available. DB write failure is non-blocking (logged as warning, story directory still moves).

**After**: `kb_update_story_status` must succeed for the move to be recorded. DB unavailability returns `MOVE FAILED: DB unavailable`. The move is blocked — there is no fallback directory operation.

All callers of `/story-move` (dev-setup-leader, dev-execute-leader, qa-verify-completion-leader, dev-fix-fix-leader, elab-setup-leader) must handle this new failure mode. Updating those callers is explicitly out of scope — the command spec change creates the contract; caller updates are handled by follow-on work.

### SWIM_LANE_TO_STATE Mapping Preservation

The `SWIM_LANE_TO_STATE` mapping is preserved as an inline table in both command specs. It continues to translate human-readable stage directory names to DB state enum values. The mapping is NOT imported from TypeScript — it is copied inline. No changes to the mapping values.

### Worktree Cleanup Preservation (WINT-1150)

The `uat → completed` worktree cleanup step in `/story-update` is explicitly preserved and must run BEFORE the `kb_update_story_status` call. This ordering is required by WINT-1150 and must not change.

### Double-Write Advisory (Out of Scope)

Several agents (e.g., `dev-setup-leader`, `qa-verify-completion-leader`) currently call `kb_update_story_status` directly AND call `/story-move`. After this story, both calls will hit the DB. The guard clause that currently prevents double-writes will need evaluation in follow-on work, but that is explicitly out of scope here.
