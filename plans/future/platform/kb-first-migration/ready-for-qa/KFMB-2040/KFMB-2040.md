# KFMB-2040: KB-Native Story Generation Pipeline

## Context

The `/pm-story` command orchestrates a two-phase story generation pipeline: the `pm-story-seed-agent` produces a `STORY-SEED.md` file by reading `stories.index.md` from the filesystem; then, the `pm-story-generation-leader` reads that seed file, spawns worker agents, and synthesizes a complete story, persisting the result via a raw SQL INSERT into the KB `stories` table.

This filesystem-and-SQL approach creates three coupling points that the KB-first migration must eliminate:

1. **Story discovery**: The leader discovers which story to generate by reading `stories.index.md` — a filesystem file. After migration, it calls `kb_list_stories` with feature/state filters.
2. **Seed artifact output**: The seed agent writes `STORY-SEED.md` to `{output_dir}/_pm/` on disk. After migration, it calls `kb_write_artifact` with `artifact_type: 'story_seed'`, storing the seed in the KB.
3. **Story content persistence**: The generation leader's Phase 4.5 performs a raw SQL INSERT into the `stories` table. After migration (once KFMB-1020 delivers the content columns), it calls `kb_update_story` to write `description`, `acceptance_criteria`, and `non_goals` directly into the story row.

This story is part of Phase 2 (Bootstrap Migration) in the KB-first migration plan. It depends on:
- **KFMB-1020** — delivers extended `kb_update_story` with content field support
- **KFMB-1030** — registers `story_seed` artifact type in `ARTIFACT_TYPES` and creates detail table

No TypeScript packages are modified — all changes are to agent instruction files (`.claude/agents/`) and one command file (`.claude/commands/`).

**Baseline reference**: KB MCP server and `stories` table exist as confirmed infrastructure (baseline 2026-02-13). The `story_seed` artifact type does not yet exist (KFMB-1030 adds it). The `stories` table does not yet have content columns (KFMB-1010 adds them, KFMB-1020 exposes them).

---

## Goal

Migrate the full `/pm-story generate` pipeline so that story discovery, seed artifact output, and story content persistence all operate exclusively through KB MCP tools — eliminating direct `stories.index.md` reads, filesystem-only seed writes, and raw SQL inserts from the generation pipeline.

---

## Non-Goals

- Does NOT modify `pm-story-adhoc-leader`, `pm-story-followup-leader`, or `pm-story-split-leader` beyond ensuring they pass compatible inputs to the seed agent.
- Does NOT eliminate the filesystem `STORY-SEED.md` output until all downstream agents have switched to reading from KB; dual-write is acceptable as a transitional state.
- Does NOT modify the `stories` table schema — column additions are delivered by KFMB-1010.
- Does NOT add the `story_seed` artifact type to the DB — that is KFMB-1030.
- Does NOT add `kb_create_story` to the generation pipeline — scoped to KFMB-2010/2020 (bootstrap pipeline).
- Does NOT update `/elab-story` or `/dev-implement-story` commands — separate KFMB stories.
- Does NOT remove `stories.index.md` from the repository — that is KFMB-3020/3030.
- Does NOT touch stage directories (`backlog/`, `elaboration/`, etc.) in the repo.
- **Protected**: Production DB schemas in `packages/backend/database-schema/`; KB server API surface.

## Architecture Notes

### Atomic Constraint: Seed Write and Seed Read Must Ship Together

AC-2 (seed agent writes to KB) and AC-3 (generation leader reads from KB) must be implemented and deployed atomically. A mismatch — seed writes to KB but leader still reads from disk, or vice versa — will silently break story generation for the session where only one side is updated.

**Strategy**: Implement dual-write in the seed agent (KB + filesystem) and dual-read in the generation leader (KB first, fallback to filesystem). This eliminates the silent break risk during rollout.

### Caller Compatibility: index_path Must Remain Optional

Before removing `index_path` from the seed agent input signature (AC-5), an audit of all callers is required:
- `.claude/commands/pm-story.md` — primary caller
- `.claude/agents/pm-story-adhoc-leader.agent.md`
- `.claude/agents/pm-story-followup-leader.agent.md`
- `.claude/agents/pm-story-split-leader.agent.md`

If any sibling leader passes `index_path` to the seed agent, either update those leaders or keep `index_path` as an optional parameter in the seed agent signature. Do not remove it as a required parameter until all callers are confirmed.

### Lessons Applied

- **[KBAR-0190]**: `kb_tools` frontmatter must be updated atomically with call-site migration. Update frontmatter in the same commit that adds or removes tool call sites.
- **[KBAR-0200]**: Agent+TypeScript node pairs must be updated together. If any TypeScript migration script also performs the stories table INSERT, update it in lockstep with the agent Phase 4.5 migration.
- **[WKFL-010]**: Use Promise.allSettled() for resilient parallel KB lookups in the generation leader (if loading story + artifact data in parallel).
- **[WKFL-008]**: This story modifies `.claude/` agent and command files only — QA via direct file spot-checking (PROOF-based), not unit tests.

### Patterns to Avoid

- Do not perform raw SQL INSERTs/UPDATEs in agent instruction files — use MCP tools exclusively
- Do not reference `stories.index.md` file paths in KB Mode documentation after migration
- Do not collapse story discovery into a single `kb_get_story` call — the leader needs a list of stories awaiting generation, not just one
- Do not add new MCP tool dependencies not delivered by KFMB-1020 and KFMB-1030
