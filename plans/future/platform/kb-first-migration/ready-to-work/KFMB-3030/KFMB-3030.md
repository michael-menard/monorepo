# KFMB-3030: Eliminate stories.index.md ‚Äî Script Updates

## Context

The `scripts/` directory contains five scripts that currently discover story lists and read story
state from `stories.index.md` files on disk. As part of the KB-First Migration plan
(`kb-first-migration`), the knowledge base (via MCP tools `kb_list_stories` and `kb_get_plan`)
must become the authoritative source for story discovery and state tracking, eliminating the
dependency on static markdown files.

The groundwork is already partially in place. `scripts/lib/resolve-plan.sh` has two functions:
`resolve_plan()` ‚Äî which already queries KB as primary with a filesystem fallback ‚Äî and
`discover_stories()` ‚Äî which has a KB-only branch that activates when `stories.index.md` is
absent. The remaining work is to promote the KB branch in `discover_stories()` to the primary
path, refactor `implement-dispatcher.sh` (which has its own inline story discovery logic that
bypasses `resolve-plan.sh`), and update `sync-work-order.py` to replace its `scan_index_file()`
behavior with KB queries.

This story depends on KFMB-3010 (Eliminate stories.index.md ‚Äî Agent Updates) being in UAT or
done before the filesystem fallback in `discover_stories()` is fully removed. For now, the
fallback is demoted to secondary (tried when KB returns empty or fails), not deleted.

## Goal

All five scripts (`resolve-plan.sh`, `generate-stories.sh`, `implement-stories.sh`,
`implement-dispatcher.sh`, `sync-work-order.py`) can operate against plans that have no
`stories.index.md` file at all. The KB becomes the primary story discovery source.

## Non-Goals

- Do not remove or delete any `stories.index.md` files ‚Äî that is handled by KFMB-6020 (Dead Code Removal).
- Do not update any agents or Claude commands ‚Äî that is KFMB-3010.
- Do not eliminate the filesystem stage-directory scan in `sync-work-order.py` ‚Äî that belongs to KFMB-4020.
- Do not change how story state is written (directory-move-based state transitions are preserved until KFMB-4010/KFMB-4020).
- Do not add `kb_update_story_status` calls to the scripts ‚Äî this story only covers read/discovery paths, not write paths.
- Do not modify work order files (`WORK-ORDER*.md`) ‚Äî `sync-work-order.py`'s work order update logic is out of scope.
- Do not fully remove the filesystem fallback from `discover_stories()` ‚Äî that requires KFMB-3010 to be complete first.

## Scope

**Files modified:**
- `scripts/lib/resolve-plan.sh` ‚Äî promote KB branch in `discover_stories()` to primary
- `scripts/implement-dispatcher.sh` ‚Äî source `resolve-plan.sh`, remove inline story discovery
- `scripts/sync-work-order.py` ‚Äî replace `scan_index_file()` with KB query; update `discover_files()`

**Files audited (likely no functional changes needed):**
- `scripts/generate-stories.sh` ‚Äî already delegates to `resolve-plan.sh`
- `scripts/implement-stories.sh` ‚Äî already delegates to `resolve-plan.sh`

**No TypeScript, no API endpoints, no frontend changes.**

- [ ] **AC-5**: `scripts/sync-work-order.py` replaces `scan_index_file()` behavior with KB API queries for story status. The `discover_files()` function no longer searches for `*stories.index.md`. The directory scan and PROOF-*.md scan paths are preserved and continue to function as before.

## Reuse Plan

- **Components**: `resolve_plan()` and the existing KB-query subshell pattern in `scripts/lib/resolve-plan.sh` ‚Äî extend and harden rather than rewrite.
- **Patterns**: `claude -p "Call <tool> with ..." --allowedTools <tool> --output-format text` + `jq` parsing, already used in `resolve-plan.sh` lines 28‚Äì56.
- **Packages**: No TypeScript package changes; pure shell/Python scripting work.
- **Python KB query**: Use `subprocess.run(['claude', '-p', ...])` pattern (same approach as shell scripts) for the `scan_index_file()` replacement in `sync-work-order.py`. Cache the result at startup and refresh periodically to avoid per-cycle latency.

## Architecture Notes

- The `discover_stories()` function in `resolve-plan.sh` currently has filesystem as primary and KB as secondary (fallback when index absent). After this story, the order inverts: KB is tried first; filesystem is the fallback.
- `implement-dispatcher.sh` is the only script that doesn't source `resolve-plan.sh`. After refactor, it will source the library just as `generate-stories.sh` and `implement-stories.sh` do.
- `sync-work-order.py` uses three data sources for story status: (1) directory scan, (2) index file, (3) PROOF-*.md scan. After this story, source (2) becomes a KB query instead of an index file read. Sources (1) and (3) are untouched.
- **Bash compatibility constraint**: No associative arrays. macOS ships bash 3.2; all array patterns must use indexed arrays or positional args compatible with bash 3.2+.
- **Network constraint**: KB queries via `claude -p` are not guaranteed to succeed in all CI environments. Error handling must exit with a clear descriptive message ‚Äî no silent empty-list production.

## Infrastructure Notes

No infrastructure changes. These are developer-facing and CI-facing shell/Python scripts. No Lambda, no CDK, no database schema changes.
