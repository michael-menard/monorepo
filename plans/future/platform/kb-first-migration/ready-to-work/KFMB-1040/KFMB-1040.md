## Architecture Notes

The `kb_delete_artifact` handler follows the canonical MCP handler shape established by `handleKbWriteArtifact`:

1. Record `startTime` for latency logging
2. Log invocation with `correlation_id`
3. Call `enforceAuthorization('kb_delete_artifact', context)` — PM-only gate
4. `KbDeleteArtifactInputSchema.parse(input)` — validates UUID, throws on invalid
5. `await kb_delete_artifact(validated.artifact_id, { db: deps.db })` — delegate to CRUD (bare string, not object)
6. Return `{ content: [{ type: 'text', text: JSON.stringify({ deleted, artifact_id }) }] }`
7. Catch all errors → `errorToToolResult(error)` which sets `isError: true`

The CRUD function (`kb_delete_artifact` in `artifact-operations.ts`) performs a two-phase delete:
1. Read the `storyArtifacts` jump table row to get `detailTable` and `detailId`
2. Call `deleteDetailRow(db, detailTable, detailId)` to delete from the appropriate detail table (e.g., `artifact_checkpoints`)
3. Delete the jump table row from `storyArtifacts`
4. Return `true` if deleted, `false` if the jump row was not found

**Important**: The handler passes `validated.artifact_id` (a string) to the CRUD function, NOT the full `validated` object. This is verified by the existing bare-string dispatch test at `artifact-tools.test.ts` line 414.
