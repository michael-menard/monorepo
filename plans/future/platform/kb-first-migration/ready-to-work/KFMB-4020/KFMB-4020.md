# KFMB-4020 — Stage Directory Elimination: precondition-check, context-init, and Script State Detection

## Context

The story lifecycle pipeline uses filesystem stage directories (`backlog/`, `elaboration/`, `ready-to-work/`, etc.) as the authoritative source of story state. Two categories of consumers still read state exclusively from the filesystem:

1. **Workflow commands**: `/precondition-check` (Step 1 — Find Story; Step 2 — Check Stage) and `/context-init` (Step 1 — Locate Story) each scan all stage directories to determine where a story lives.

2. **Shell scripts**: `generate-stories.sh` derives `ALREADY_GENERATED` and `ALREADY_ELABORATED` flags from filesystem checks; `implement-stories.sh` contains `find_story_dir`, `is_elaborated`, `is_implemented`, `is_reviewed`, and `is_completed` functions that all iterate stage directories.

KFMB-4010 will have already converted `/story-move` and `/story-update` to write state to the `stories.state` DB column first (using `shimGetStoryStatus` / `shimUpdateStoryStatus`). After KFMB-4010 completes, the DB becomes the write-authoritative source for state transitions — but state reads in these consumers remain filesystem-only. This creates a split-brain risk: the DB may reflect a story as `in_progress` while `/precondition-check` scans directories and finds it elsewhere.

**Dependency note**: KFMB-4020 is in Parallelization Group 6 and must not be elaborated until KFMB-4010 (Group 5) is complete and has reached at least `ready-for-qa`.

## Goal

Update `/precondition-check` and `/context-init` to adopt the KB-first + directory-fallback pattern established by KFMB-4010 in `/story-move`. Update the state detection functions in `generate-stories.sh` and `implement-stories.sh` to query the KB (`storyGetStatus` via `claude -p`) with graceful fallback to the existing directory scan when KB is unavailable.

After this story: all state reads in workflow commands and automation scripts prefer the DB. The directory fallback remains as a resilience layer.

## Non-Goals

- Removing the directory fallback path entirely — that is Phase 5 (KFMB-6010/KFMB-6020) after writes are confirmed stable
- Migrating `uat-precondition-check.agent.md` — that agent checks MSW/API/Cognito health, not story state
- Updating `stories.index.md` progress counts from scripts — in scope for KFMB-3030
- Modifying any production DB schemas in `packages/backend/database-schema/` — `stories.state` column already exists via KFMB-1010
- Changing the KB MCP server infrastructure or pgvector setup
- Adding new DB columns — reads the existing `stories.state` column only
- Migrating `_implementation/` or `_pm/` artifact reads — that is Phase 4 (KFMB-5010 through KFMB-5050)

## Architecture Notes

### KB-First Guard Pattern (from story-move.md v2.1.0)

```
result = shimGetStoryStatus({ storyId: STORY_ID })

if result is non-null:
  # KB hit — use result.state to determine swim-lane via STATE_TO_SWIM_LANE
  current_state = result.state
  swim_lane = STATE_TO_SWIM_LANE[current_state] ?? null
  db_state_used = true
else:
  # KB miss or DB error — fall back to directory scan
  db_state_used = false
  swim_lane = <scan stage directories>
  log warning: "[KB-FIRST FALLBACK] using filesystem for {STORY_ID}"
```

### STATE_TO_SWIM_LANE Reverse Map

Derived from `SWIM_LANE_TO_STATE` in `packages/backend/mcp-tools/src/story-compatibility/__types__/index.ts`:

```typescript
export const STATE_TO_SWIM_LANE = Object.fromEntries(
  Object.entries(SWIM_LANE_TO_STATE).map(([lane, state]) => [state, lane])
) as Record<SwimLaneState, SwimLaneDir>
```

Note: DB-only states (`blocked`, `cancelled`) have no swim-lane mapping; reverse lookup returns undefined for these. Commands must handle undefined as "not in a swim-lane directory" and fall back to directory scan.

### Shell Script KB Query Pattern (from resolve-plan.sh)

```bash
kb_state_query() {
  local STORY_ID="$1"
  local raw
  raw=$(claude -p \
    "Call storyGetStatus with storyId '${STORY_ID}'. Output ONLY the state string (e.g. 'ready_to_work') or 'null' if not found. No explanation." \
    --allowedTools "mcp__knowledge-base__kb_get_story" \
    --output-format text 2>/dev/null) || true
  echo "$raw"
}

# Usage:
KB_STATE=$(kb_state_query "$STORY_ID")
if [[ -n "$KB_STATE" && "$KB_STATE" != "null" ]]; then
  echo "[KB-FIRST] state from DB: $KB_STATE"
  # Use KB_STATE for state detection
else
  echo "[KB-FIRST FALLBACK] using filesystem for $STORY_ID"
  # Fall back to directory scan
fi
```

### db_state_used Output Field

All modified commands emit `db_state_used` in their result YAML. This field allows callers (setup leaders, CI scripts) to audit whether KB or filesystem was the source of truth for a given invocation.
