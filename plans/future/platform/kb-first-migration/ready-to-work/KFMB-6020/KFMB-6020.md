# KFMB-6020: Dead Code Removal and Documentation Updates — KB-First Architecture Capstone

## Context

The KB-First Migration plan (KFMB-1010 through KFMB-6010) will have, by the time this story executes, fully migrated all story lifecycle management from filesystem-based patterns to KB-exclusive operations. Story state transitions, artifact reads/writes, script state detection, and agent tool calls all route through the Knowledge Base MCP server. No new filesystem story artifacts are produced during normal workflow execution.

Despite the functional migration being complete, residual filesystem-era references survive in three areas:

**Shell script comments**: `generate-stories.sh` and `implement-stories.sh` contain usage comments that reference `stories.index.md` as the discovery source (KFMB-3030 OPP-2 explicitly deferred these to KFMB-6020, identifying specific lines 13/26 and 30/32 respectively). `sync-work-order.py`'s module docstring also describes `stories.index.md` as a data source.

**Agent Output section prose**: The KFMB-5010 lesson (KB entry `fac0d1b6-522d-4765-b3fb-169d35379a48`) documented that AC-11 covered artifact write call bodies but not Output section prose. Bullet points like "written to `_implementation/EVIDENCE.yaml`" remain in migrated agents and mislead future maintainers about where artifacts are stored.

**Reference documentation**: `docs/COMMANDS.md`, `docs/workflow/story-directory-structure.md`, `docs/workflow/phases.md`, and related workflow docs reference filesystem artifact paths, stage directories, and `stories.index.md` as live concepts. A developer reading these docs today would learn the old architecture, not the KB-first one.

Additionally, a migration guide does not yet exist. Engineers joining the project (or revisiting the codebase after the migration) have no consolidated reference explaining what changed, why, and what the new architecture looks like.

**Baseline date**: 2026-02-13. Loaded: yes. No gaps.

**Dependency precondition**: KFMB-6020 MUST NOT begin implementation until KFMB-6010 is at `needs-code-review` or later. The dead code audit depends on knowing exactly which filesystem references survived KFMB-6010.

## Goal

Ensure the codebase contains no misleading dead filesystem story code references and that documentation accurately reflects the KB-first architecture. Author a migration guide (`docs/workflow/kb-first-migration-guide.md`) that transforms this migration from a series of technical changes into understood institutional knowledge.

- Do not modify Orchestrator artifact Zod schemas in `packages/backend/orchestrator/src/artifacts/` — they remain canonical type definitions even after filesystem story output is eliminated.
- Do not modify any functional logic in shell or Python scripts — only comments and docstrings.
- Do not update `packages/backend/database-schema/` or `@repo/db` client API — protected features.
- Do not add new MCP tool registrations or modify the KB MCP server.
- Do not address `sync-work-order.py` `scan_directories()` or `scan_proof_files()` function correctness — these remain valid and are explicitly not in scope.
- Do not write a new KB schema or migration guide for the database layer — the migration guide covers the agent/script/workflow layer only.

## Architecture Notes

This story is a documentation capstone. No architectural decisions are introduced. The core architecture principle being documented is:

**KB-first**: After KFMB-1010 through KFMB-6010, the Knowledge Base (PostgreSQL at port 5433, exposed via MCP tools) is the sole source of truth for story state, artifact content, and workflow state detection. The filesystem retains:
- Worktree directories (`tree/story/{STORY_ID}/`)
- PROOF files (written by dev agents as human-readable evidence)
- Story plan files (`plans/future/*/backlog/{STORY_ID}/`) — structural metadata for planning

All story state that previously lived in `stories.index.md`, stage directory names (`backlog/`, `in-progress/`, `needs-code-review/`), and `_implementation/` YAML artifacts now lives exclusively in KB tables (`stories`, `story_artifacts`, `story_knowledge`).

**Protected areas (do not touch)**:
- `packages/backend/database-schema/` — DB schema definitions
- `@repo/db` client API surface
- `packages/backend/orchestrator/src/artifacts/` — Zod schema definitions (remain canonical even after filesystem removal)
- `apps/api/knowledge-base/` — KB MCP server implementation
