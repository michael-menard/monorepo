# KFMB-2020 — KB-Native Bootstrap Setup Leader

## Context

The `pm-bootstrap-setup-leader` agent (Phase 0 of `/pm-bootstrap-workflow`) validates plan input and produces bootstrap context for subsequent phases. In its current KB Mode (v4.0.0), Step 4 detects whether a previous bootstrap has already run by checking if `{feature_dir}/stories.index.md` exists on disk.

This filesystem check is brittle in two ways:

1. **Divergence risk** (lesson from WINT-1060): The index file can be absent even when stories were previously created in the KB stories table. If stories are created via a KB-native flow without writing `stories.index.md`, the current guard silently misses the collision and allows a duplicate bootstrap run.

2. **TOCTOU race condition** (lesson from APIP-4050): Two concurrent `/pm-bootstrap-workflow` runs on the same `feature_dir` could both pass the `stories.index.md` existence check before either writes it. A KB query via `kb_list_stories` is transaction-isolated and avoids this window.

As the system migrates toward KB-first storage (per this KFMB plan), `stories.index.md` will eventually be eliminated. Relying on its filesystem presence as the collision sentinel creates a dependency on a file being phased out.

**Dependency note**: `kb_list_stories` is already available (no KFMB-1020 dependency from a pure tool-availability standpoint). The dependency on KFMB-1020 is for sequencing (Phase 1 foundational work should complete before Phase 2 agent updates begin) rather than strict technical requirement for this specific story.

## Goal

Replace the filesystem collision check in KB Mode Step 4 of `pm-bootstrap-setup-leader.agent.md` with a `kb_list_stories` MCP tool call, maintaining full signal contract compatibility and adding a graceful fallback for MCP unavailability.

## Non-Goals

- Do not update File Mode behavior — File Mode collision detection via filesystem remains unchanged.
- Do not modify `pm-bootstrap-workflow.md` — that is KFMB-2030's scope.
- Do not modify `pm-bootstrap-generation-leader.agent.md` — that is KFMB-2010's scope (parallel story, different file).
- Do not add new output fields to SETUP-CONTEXT — schema stability is a hard constraint.
- Do not implement a full KB-native bootstrap pipeline end-to-end — that is KFMB-2030's scope.
- Do not remove `stories.index.md` creation from the generation leader — that is KFMB-3020/3030 scope (Phase 3).
- Do not update `kb_list_stories` or `story-crud-operations.ts` — this story only changes how the agent invokes the existing tool.

## Architecture Notes

**Filter parameter selection**: The `kb_list_stories` tool has two relevant filters:
- `feature` — matches the `feature` column (a human-readable description string). NOT suitable for collision detection.
- `epic` — matches the `epic` column (likely stores the plan slug, e.g., "kb-first-migration"). This is the correct filter.

**Pre-implementation verification required (ST-1)**: Before editing the agent, run `kb_list_stories({ epic: "kb-first-migration", limit: 5 })` against the dev KB to confirm the `epic` column value for KFMB-prefixed stories. If the column stores a different value format, adjust the filter parameter in the new Step 4 prose accordingly.

**Fallback design**: The fallback path (AC-4) is safety-critical for transition. When `kb_list_stories` is unavailable:
1. Catch any error (tool not found, connection refused, timeout)
2. Log warning verbatim: `"KB collision check unavailable — falling back to filesystem check"`
3. Check if `{feature_dir}/stories.index.md` exists on disk
4. If exists: `SETUP BLOCKED: "stories.index.md already exists in {feature_dir} — bootstrap already run (filesystem fallback)"`
5. If not exists: proceed to Step 5

**Version bump**: Minor bump (4.0.0 → 4.1.0) — backwards-compatible change within KB Mode only.
