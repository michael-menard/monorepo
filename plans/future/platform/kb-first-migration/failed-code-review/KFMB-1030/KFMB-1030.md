# KFMB-1030: PM Artifact Types and Detail Tables

## Context

The KB database currently supports 13 artifact types for storing implementation workflow artifacts via a jump table (`story_artifacts`) that routes each artifact to a type-specific detail table (established by migration 015). These 13 types cover the dev implementation pipeline: checkpoint, scope, plan, evidence, verification, analysis, context, fix_summary, proof, elaboration, review, qa_gate, and completion_report.

The PM pipeline — which runs before implementation and generates `test-plan.yaml`, `dev-feasibility.yaml`, `uiux-notes.yaml`, and `STORY-SEED.md` for each story — currently writes these outputs to the filesystem under `_pm/` directories. The kb-first migration plan (KFMB series) aims to replace filesystem persistence with direct KB writes.

Before PM agents can write their artifacts to the KB (covered in KFMB-5040), the database and CRUD layer must first be extended to support the 4 new PM artifact types. This story adds that foundation: one new detail table per artifact type, the corresponding Zod enum extensions, the Drizzle schema definitions, and the CRUD routing additions in `artifact-operations.ts`.

The `artifact_type` column in `story_artifacts` is plain `text` — there is no PostgreSQL CHECK constraint to modify. Extension is purely additive in SQL (new tables) and additive in TypeScript (extended Zod enum + new CRUD cases).

**Dependency**: KFMB-1010 (Stories Table Content Columns Migration) must be complete before implementation begins. The two stories touch separate tables — there is no code-level overlap — but the plan specifies KFMB-1010 as a Phase 1 prerequisite.

## Goal

Extend the KB database and CRUD layer to support 4 new PM artifact types (`test_plan`, `dev_feasibility`, `uiux_notes`, `story_seed`), enabling the existing `kb_write_artifact`, `kb_read_artifact`, and `kb_list_artifacts` MCP tools to transparently handle PM pipeline outputs.

## Non-Goals

- Writing PM artifacts to the KB from agents — that is KFMB-5040 (Migrate `_pm/` Writer Agents to `kb_write_artifact`)
- Adding PM artifact types to `packages/backend/database-schema/src/schema/artifacts.ts` — that file covers the main Lego app DB; this story concerns the Knowledge Base DB only
- Modifying any existing artifact type tables or changing existing artifact type Zod values
- Adding HTTP endpoints or MCP tool handler registrations — the existing tools automatically support new types once CRUD routing is updated
- Migrating existing PM YAML files from the filesystem to the KB — that is a later migration story
- Any frontend or app changes
- Adding a PostgreSQL CHECK constraint on `artifact_type` — enforcement is at the Zod layer only

## Architecture Notes

- `artifact_type` column is `text` in PostgreSQL — no `ALTER TYPE` needed; no `CHECK` constraint to extend. Extension is Zod-layer only.
- All 4 new types follow the `{ typedColumns, data }` pattern where `data` stores the full JSONB content for backward compatibility and `typedColumns` promotes queryable fields.
- Recommended typed columns per type:
  - `artifact_test_plans`: `strategy TEXT`, `scope_ui_touched BOOLEAN`, `scope_data_touched BOOLEAN`
  - `artifact_dev_feasibility`: `feasible BOOLEAN`, `confidence TEXT`, `complexity TEXT`
  - `artifact_uiux_notes`: `has_ui_changes BOOLEAN`, `component_count INTEGER`
  - `artifact_story_seeds`: `conflicts_found INTEGER`, `blocking_conflicts INTEGER`, `baseline_loaded BOOLEAN`
- Zod-first types are required: `z.object()` with `z.infer<>` — no TypeScript interfaces.
- No barrel files: all imports from direct source paths.
