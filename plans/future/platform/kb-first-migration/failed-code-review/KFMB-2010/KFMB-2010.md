# KFMB-2010: KB-Native Bootstrap Generation Leader

## Context

The `pm-bootstrap-generation-leader` agent (v4.0.0) currently operates in two modes:

- **KB Mode** (default): Receives `SETUP-CONTEXT` and `ANALYSIS` inline. Writes `story.yaml` and `stories.index.md` to disk, then performs a **raw psql batch INSERT** into the KB `stories` table as a secondary step. Returns SUMMARY inline.
- **File Mode**: Reads `_bootstrap/AGENT-CONTEXT.md` and `_bootstrap/ANALYSIS.yaml` from disk. Writes all output to the feature directory.

The raw psql approach in KB Mode is a known workaround: the agent must know the connection string (`postgresql://kbuser:TestPassword123!@localhost:5433/knowledgebase`), manage a temp SQL file, and handle psql errors directly. This bypasses the MCP abstraction layer entirely.

KFMB-1020 delivers the `kb_create_story` MCP tool. Once it ships, this story rewrites the generation leader to call `kb_create_story` for each story stub via MCP — no raw psql, no temp SQL files, no direct connection string management.

**Dependency**: This story is blocked on KFMB-1020. The `kb_create_story` tool does not exist yet. Do not start implementation until KFMB-1020 ships and the tool interface is confirmed.

## Goal

Rewrite `pm-bootstrap-generation-leader` to:
1. Call `kb_create_story` (via MCP) for each story stub from ANALYSIS — no raw psql
2. Return SUMMARY inline, consistent with current KB Mode contract
3. Remove or deprecate File Mode
4. Explicitly document filesystem output decisions (story.yaml, stories.index.md)

## Architecture Notes

The rewritten agent has a single mode (KB Mode). The dual-mode structure is collapsed:

**Before (v4.0.0)**:
```
KB Mode:
  1. Write story.yaml files to disk
  2. Write stories.index.md to disk
  3. Run psql batch INSERT (raw SQL, temp file, connection string)
  4. Return SUMMARY inline

File Mode:
  1. Read _bootstrap/AGENT-CONTEXT.md
  2. Read _bootstrap/ANALYSIS.yaml
  3. Write story files to feature dir
```

**After (v5.0.0)**:
```
KB Mode (only):
  1. Write story.yaml files to disk (retained pending KFMB-3010)
  2. Write stories.index.md to disk (retained pending KFMB-3010)
  3. Call kb_create_story(story) for each story in ANALYSIS (via MCP)
     - On KB unavailable: log warning, continue, track in SUMMARY
     - On conflict: no-op (idempotent)
  4. Return SUMMARY inline

File Mode: REMOVED (or marked DEPRECATED — see AC-6)
```

**KB degradation**: If `kb_create_story` fails (KB unavailable), the agent logs a warning and continues. The SUMMARY reflects `kb_stories_inserted: 0` with a warning field. This matches the existing degradation behavior.

**Agent model**: Remains haiku. The MCP call loop is concise; no context window growth that would require a larger model.
