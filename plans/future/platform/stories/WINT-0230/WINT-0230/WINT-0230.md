# WINT-0230: Create Unified Model Interface

## Context

The WINT-0220 strategy defines a 4-tier model selection system (Tier 0: Critical/Opus, Tier 1: Complex/Sonnet, Tier 2: Routine/Ollama, Tier 3: Simple/Ollama) to optimize costs while preserving quality. The strategy is documented and validated (qa_verdict: PASS), but there is currently no implementation of the tier-based routing logic.

### Current State

The existing system has:
- **Agent-level assignments** via `model-assignments.ts` (loads agent→model mappings from YAML)
- **Provider adapters** via MODL-0010 (ILLMProvider interface with Ollama/Anthropic/OpenRouter implementations)
- **Static model selection** (agents specify model in frontmatter, no dynamic tier selection)

### Problem

Without a unified model interface that implements the WINT-0220 strategy:
- Agents are statically assigned to models (no tier-based routing)
- No escalation logic for quality/cost/failure scenarios
- Strategy configuration exists but isn't operationalized
- No automatic fallback when Ollama unavailable
- Cost optimization targets (60% reduction) cannot be achieved

### Opportunity

A unified model interface will:
1. **Operationalize WINT-0220 Strategy**: Implement tier-based routing as defined in WINT-0220-STRATEGY.yaml
2. **Enable Cost Optimization**: Route 60% of tasks to free Ollama models (Tier 2/3)
3. **Preserve Quality**: Keep complex reasoning on Claude Sonnet/Opus (Tier 0/1)
4. **Provide Fallback Chains**: Graceful degradation when Ollama unavailable
5. **Support Escalation**: Quality gates, cost budgets, failure retries, human-in-loop triggers
6. **Unblock Downstream Work**: WINT-0260 (cost tracking), WINT-0240 (Ollama fleet), WINT-0250 (escalation automation)

---

## Goal

Create a unified model interface (`UnifiedModelInterface` or `ModelRouter`) that implements the WINT-0220 4-tier strategy, routes agent invocations to appropriate model tiers based on agent name + task characteristics, executes escalation logic for quality/cost/failure scenarios, provides automatic fallback chains, and maintains backward compatibility with existing agent frontmatter.

### Success Criteria

- Unified interface loads and validates WINT-0220-STRATEGY.yaml
- Tier-based model selection works for all 143 agents
- Escalation logic implemented (quality → cost → failure → human triggers)
- Fallback chains handle Ollama unavailability transparently
- Backward compatibility preserved (agents with `model:` frontmatter continue working)
- Integration with MODL-0010 provider adapters verified
- Configuration API exposed for strategy introspection
- 80%+ test coverage with 100% coverage on escalation logic

### AC-3: Escalation Logic Implementation ✅

**Given** escalation triggers defined in strategy,
**When** escalation conditions met (quality gate failure, cost budget exceeded, multiple failures, human-in-loop),
**Then**:
- **Quality escalation**: Retry with next higher tier on gate failure (max 3 retries → Tier 0 or human)
- **Cost de-escalation**: Downgrade non-critical tasks when budget threshold exceeded (80% warning, 95% critical)
- **Failure escalation**: Escalate to Tier 0 or human after 3 failures at current tier
- **Human-in-loop**: Pause workflow on low confidence (<50%), scope violations, security concerns

**Escalation Priority Order** (highest to lowest):
1. Human triggers (always terminate)
2. Failure escalation (max 3 retries → Tier 0)
3. Quality escalation (gate failures → higher tier)
4. Cost de-escalation (non-critical tasks only)

**Termination Guarantees**:
- Max escalation depth = 3 (prevents infinite loops)
- All paths terminate at Tier 0 or human review
- No circular dependencies (enforced by priority order)

**Evidence**:
- Unit test: Quality gate failure → escalate to Tier 1 (from Tier 2)
- Unit test: Cost budget 95% exceeded → de-escalate to Tier 2 (from Tier 1)
- Unit test: 3 failures at Tier 2 → escalate to Tier 0
- Unit test: Low confidence (<50%) → pause for human review
- Unit test: Circular escalation prevented (max depth 3, priority order enforced)
- 100% test coverage on escalation logic (critical path)

## Architecture Notes

### Design Patterns

**Factory Pattern** (ModelRouterFactory):
- Centralizes configuration loading before router instantiation
- Ensures strategy loaded correctly before router created
- Easier to test with dependency injection

**Strategy Pattern** (Escalation Triggers):
- Each escalation trigger (quality/cost/failure/human) implements `EscalationStrategy` interface
- Allows testing triggers in isolation
- Clear priority order enforcement (human > failure > quality > cost)

**Adapter Pattern** (TierAwareProviderAdapter):
- Wraps ILLMProvider from MODL-0010 with tier-based routing layer
- No modifications to base provider interfaces (backward compatible)
- Fallback logic encapsulated in adapter

**Single Responsibility Principle**:
- **StrategyConfigurationLoader**: YAML loading, Zod validation, caching
- **TierSelector**: Agent→tier mapping, legacy model frontmatter
- **EscalationManager**: Escalation triggers, priority order, termination
- **ProviderRouter**: Provider factory integration, fallback chains
