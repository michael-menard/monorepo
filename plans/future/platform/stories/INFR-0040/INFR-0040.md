# INFR-0040: Workflow Events Table + Ingestion

## Context

The orchestrator currently tracks workflow execution in memory and YAML files on disk. There is no persistent, queryable log of workflow events (state transitions, step completions, gaps found, etc.). This blocks downstream telemetry dashboards, learning loops, and cost tracking.

The INFR epic establishes a three-tier persistence architecture:
1. **Operational DB (Postgres)** - state, events, metadata
2. **Artifact Blob Store (MinIO/S3)** - large files
3. **Knowledge Base (pgvector)** - embeddings

INFR-0040 focuses on tier 1: creating the workflow events table and basic ingestion mechanism.

**Problem:**
- No centralized event log for workflow execution
- State transitions not tracked in queryable format
- Cost, tokens, duration not persisted per step
- Gap findings, flow issues, churn events not captured
- TELE-0010 (telemetry stack) blocked waiting for event data
- INFR-0050 (Event SDK) blocked waiting for event schema

**Existing Infrastructure:**
- Drizzle ORM schemas in `packages/backend/database-schema/src/schema/`
- @repo/db client with connection pooling in `packages/backend/db/`
- Orchestrator state machine in `packages/backend/orchestrator/src/state/story-state-machine.ts` (source of state transitions)
- Docker Compose stack with PostgreSQL + Prometheus + Grafana + OTel in `infra/compose.lego-app.yaml`
- @repo/observability with OpenTelemetry tracing

## Goal

Create a `telemetry.workflow_events` table in Postgres with:
- Event type enum (5 core events from INFR PLAN.md)
- ULID event_id for idempotency
- Normalized columns for common queries (event_type, run_id, item_id, workflow_name, agent_role, status)
- JSONB payload for event-specific data
- Append-only, immutable design
- Unique index on event_id to prevent duplicates

Add basic `insertWorkflowEvent()` function that orchestrator nodes can call to write events synchronously (async/queue ingestion comes later in INFR-0050).

## Non-Goals

- Event SDK with hooks/middleware (INFR-0050)
- Async event queue/buffer (INFR-0050)
- Prometheus metrics mapping (TELE-0020)
- Grafana dashboards (TELE-0030)
- Event retention/archival policy (INFR-0060)
- Migrating existing orchestrator runs to event format
- Event replay or backfill mechanisms
- Cross-schema queries joining events with other tables
- Event schema versioning beyond basic event_version field
- Payload validation per event_type (defer to INFR-0050)

## Architecture Notes

### Event Table Design Principles
1. **Append-Only**: Events are immutable, never updated or deleted
2. **Idempotent**: Unique index on event_id prevents duplicates
3. **Schema Separation**: telemetry.* namespace isolated from other schemas (work.*, kb.*, ai.*)
4. **Normalize What You Query**: Frequently queried fields (event_type, run_id, item_id) are top-level columns
5. **JSONB for Flexibility**: Event-specific data goes in payload (avoids rigid schema per event_type)
6. **Event Version Field**: Supports future schema evolution without breaking queries

### Query Patterns Supported
- **Telemetry Dashboard**: "Show all step_completed events in last 24 hours" → index on (event_type, ts)
- **Run Timeline**: "Show all events for run-123" → index on (run_id, ts)
- **Story Audit**: "Show all events for INFR-0040" → index on item_id
- **Workflow Analysis**: "Show all events for dev-implement-story workflow" → index on workflow_name
- **Agent Performance**: "Show all events for dev-implementation-leader" → index on agent_role

### ULID vs UUID Decision
- **Decision**: Use UUID for event_id (see AC-14)
- **Rationale**: Maintain consistency with all existing table patterns in codebase (kb.*, work.*, ai.*)
- **Trade-off**: ULID offers lexicographic sorting but requires text type (slower indexing)
- **Future Path**: Defer ULID to future story if lexicographic sorting becomes critical for telemetry queries
- **Implementation**: Use Drizzle's `uuid('event_id').primaryKey().defaultRandom()` pattern

### Error Handling Strategy
- **Philosophy**: Event logging must not crash orchestrator
- **Implementation**: insertWorkflowEvent catches DB errors, logs warning, continues
- **Trade-off**: Accept occasional event loss over workflow failure
- **Future**: INFR-0050 (Event SDK) will add async queue + retry for higher reliability
