# INFR-0050: Event SDK (Shared Telemetry Hooks)

## Context

INFR-0040 created the `telemetry.workflow_events` table with basic synchronous ingestion via `insertWorkflowEvent()`. INFR-0041 added typed event schemas and helper functions for the 5 core event types (step_started, step_completed, item_state_changed, decision_made, error_occurred).

However, orchestrator nodes must manually call `insertWorkflowEvent()` for each event, leading to:

- **Boilerplate**: Every node duplicates the same event emission pattern
- **Missing Context**: No automatic enrichment with correlation IDs from active OpenTelemetry spans
- **Manual Metadata**: Nodes must manually populate source/emitter/correlation_id fields
- **No Buffering**: Synchronous DB writes may slow down orchestrator execution
- **No Batch Support**: No way to emit multiple events efficiently

### Current Reality (from Baseline 2026-02-13)

**Existing Event Infrastructure**:
- `telemetry.workflow_events` table: UUID primary key, JSONB payload, timestamp columns
- `insertWorkflowEvent()`: Synchronous event insertion function
- Event helper functions: `createStepCompletedEvent()`, etc. (5 event types)
- Zod schemas: Runtime validation for each event payload

**Related Patterns**:
- `@repo/observability`: OpenTelemetry tracing middleware, `trace.getActiveSpan()`
- Drizzle ORM: Batch insert API `db.insert(table).values([...])`
- Resilient error handling: Warn + continue pattern (event logging must not crash orchestrator)

**Dependencies**:
- INFR-0040 (in-qa): Workflow events table and `insertWorkflowEvent()` function exist and are tested. SDK implementation can proceed in parallel with INFR-0040 UAT completion since the dependency is on the table schema (stable), not the completion status.
- INFR-0041 (completed): Typed schemas foundation already available

## Goal

Create a reusable **Event SDK** that provides:

1. **Hook-Based API**: Composable functions that wrap operations and emit events automatically
2. **Auto-Enrichment**: Extract correlation_id from active OpenTelemetry span, auto-populate source/emitter
3. **Buffered Ingestion**: In-memory event buffer with configurable flush interval (default: 5s or 100 events)
4. **Batch Insert Function**: `insertWorkflowEventsBatch()` for efficient bulk writes
5. **Graceful Degradation**: Buffer overflow handling, telemetry downtime resilience, never crash orchestrator
6. **Developer Experience**: Simple API like `withStepTracking(stepName, () => operation())` that auto-emits events

This SDK will be the **canonical way** for orchestrator nodes (and future LangGraph nodes) to emit telemetry events, replacing manual `insertWorkflowEvent()` calls.

## Acceptance Criteria

### AC-1: Create `withStepTracking()` Hook Function
- **Purpose**: Wrap a step operation, auto-emit `step_completed` event on success/error
- **Signature**: `withStepTracking(stepName: string, operation: () => Promise<T>, options?: StepTrackingOptions): Promise<T>`
- **Behavior**:
  - Measure duration (start → end timestamp)
  - Capture tokens/model if provided in options
  - Emit event on completion (success or error)
  - Extract correlation_id from active OTel span if available
  - Return original operation result (transparent wrapper)
- **Error Handling**: Re-throw error after emitting event (caller handles error)
- **Tests**: HOOK-001 (success), HOOK-002 (error), HOOK-003 (duration), HOOK-004 (tokens/model)

### AC-2: Create `withStateTracking()` Hook Function
- **Purpose**: Emit `item_state_changed` event when state transitions occur
- **Signature**: `withStateTracking(itemId: string, fromState: string, toState: string, options?: StateTrackingOptions): Promise<void>`
- **Behavior**: Emit event immediately (not buffered, since state changes are critical)
- **Tests**: HOOK-005 (immediate emission, bypasses buffer)

### AC-3: Create In-Memory Event Buffer
- **Purpose**: Accumulate events before flushing to database in batches
- **Configuration**:
  - `bufferSize` (default: 100) - max events before auto-flush
  - `flushIntervalMs` (default: 5000) - max time between flushes
- **Behavior**:
  - Auto-flush when buffer size reached OR interval elapsed (whichever first)
  - Thread-safe concurrent additions
- **Tests**: BUF-001 (size threshold), BUF-002 (interval), BUF-003 (overflow), BUF-006 (concurrent)

### AC-4: Create `insertWorkflowEventsBatch()` Function
- **Purpose**: Insert multiple events efficiently using Drizzle batch insert API
- **Signature**: `insertWorkflowEventsBatch(events: WorkflowEventInput[]): Promise<void>`
- **Behavior**:
  - Use Drizzle: `db.insert(telemetry.workflow_events).values(events)`
  - Leverage ON CONFLICT DO NOTHING on event_id (from INFR-0040) for idempotency
  - Chunk large batches if >6500 events (PostgreSQL param limit ~65535)
- **Error Handling**: Resilient - catch DB errors, log warnings via @repo/logger, never crash
- **Tests**: BATCH-001 (100 events), BATCH-002 (idempotency), BATCH-003 (DB error), BATCH-004 (chunking)

### AC-5: Auto-Enrich Events with OpenTelemetry Context
- **Behavior**:
  - Extract trace ID from active span (`trace.getActiveSpan()` from @repo/observability)
  - Populate `correlation_id` field automatically
  - If no active span, leave `correlation_id` null
  - Extract at **event creation time**, not flush time (span may end)
- **Integration**: Use @repo/observability's OTel utilities
- **Tests**: OTEL-001 (with span), OTEL-002 (no span), OTEL-003 (timing)

### AC-6: Add Event Buffer Flush on Process Shutdown

- **Cleanup**: Clear flush timer, prevent duplicate flushes
- **Tests**: SHUT-001 (SIGTERM), SHUT-002 (SIGINT), SHUT-003 (timeout), SHUT-004 (deduplication)

### Design Decisions

**Buffer Overflow Strategy**: Drop-oldest + warn (DECIDED)
- **Selected Strategy**: `overflowStrategy: 'drop-oldest'` (default in AC-7 config schema)
- **Rationale**: Graceful degradation; losing old events is acceptable vs. crashing orchestrator or blocking execution
- **Implementation**: When buffer reaches `bufferSize` limit before flush, remove oldest event and log warning via `@repo/logger.warn()` with dropped event count
- **Alternatives Considered**:
  - `error`: Throw error (fail fast) - Rejected: violates orchestrator resilience requirement (event logging must not crash workflow)
  - `block`: Block event emission until flush completes - Rejected: performance impact (could stall orchestrator execution during DB slowness)
- **Configuration**: Strategy is configurable in `TelemetrySdkConfig.overflowStrategy` for advanced use cases, but default 'drop-oldest' recommended for most scenarios

**Flush Timer Management**: `setInterval()` (Node.js built-in)
- Rationale: Simple, no dependencies, sufficient for MVP
- Alternative: Scheduler library (node-schedule) - rejected as over-engineering

**OTel Context Propagation**: Extract at event creation, not flush
- Rationale: Active span may end before flush occurs
- Store trace ID in event object immediately when hook called

**Batch Insert Chunking**: 6500 events per INSERT
- Rationale: PostgreSQL param limit ~65535; 10 fields/event = 65,000 params (safe margin)
- Implementation: `batch-chunker.ts` splits large arrays

**Hook Function Return Value**: Transparent wrapper, return original operation result
- Rationale: SDK should not alter operation behavior
- Alternative: Return event_id for debugging - rejected; use @repo/logger debug mode instead

**Singleton Pattern**: Single SDK instance per process
- Rationale: One event buffer per process simplifies state management
- `initTelemetrySdk()` returns cached instance on subsequent calls
