# INFR-0041: Workflow Event SDK - Typed Schemas & Validation

## Follow-up Context

**Parent Story:** INFR-0040
**Source:** QA Discovery Notes - Deferred Items #2 (Event Payload Validation)
**Original Finding:** "No Zod schemas for event_type-specific payloads. Each of the 5 event types has different payload requirements but these are undocumented. Invalid payloads silently inserted, breaking downstream telemetry queries."
**Category:** Enhancement Opportunity
**Impact:** High (blocks INFR-0050 Event SDK, affects TELE-0010 data quality)
**Effort:** Low (1 story point - similar scope to parent story)

## Context

INFR-0040 created the `telemetry.workflow_events` table with basic synchronous ingestion via `insertWorkflowEvent()`. The table accepts any JSONB payload without validation, and callers must manually construct event objects.

**Problem:**
- **No type safety**: Event payloads are unvalidated JSONB, causing runtime errors if structure is wrong
- **No schema per event_type**: Each of the 5 event types (`item_state_changed`, `step_completed`, `story_changed`, `gap_found`, `flow_issue`) has different payload requirements, but these are undocumented
- **Manual event construction**: Orchestrator nodes manually build event objects, leading to inconsistent field naming and missing data
- **No payload validation**: Invalid payloads silently inserted, breaking downstream telemetry queries
- **Missing metadata**: No correlation_id for distributed tracing, no source metadata for debugging
- **INFR-0050 blocked**: Event SDK story needs typed schemas as foundation

**Current Gaps (from INFR-0040 deferred items):**
1. No Zod schemas for event_type-specific payloads
2. No validation before insertion
3. No source/emitter metadata fields
4. No correlation_id for OpenTelemetry integration
5. Manual event object construction in orchestrator nodes

**Existing Infrastructure:**
- `telemetry.workflow_events` table with 10 columns (from INFR-0040)
- `insertWorkflowEvent()` function for synchronous ingestion (from INFR-0040)
- Drizzle ORM schemas in `packages/backend/database-schema/src/schema/`
- @repo/db client with connection pooling in `packages/backend/db/`
- Orchestrator state machine in `packages/backend/orchestrator/src/state/story-state-machine.ts`

## Goal

Create a **typed event schema library** that:
1. Defines Zod schemas for all 5 event types with typed payloads
2. Adds validation to `insertWorkflowEvent()` to reject malformed events
3. Exports typed helper functions for each event type (e.g., `createItemStateChangedEvent()`)
4. Adds metadata fields: `correlation_id`, `source`, `emitted_by` to table schema
5. Documents event payload structures for downstream consumers (TELE-0010, INFR-0050)

This story provides the **schema foundation** for INFR-0050 (Event SDK with hooks/middleware) and TELE-0010 (telemetry dashboards).

- Mutable event records (events are append-only)

## Architecture Notes

### Schema Design Principles
1. **Event-Type Specific Schemas**: Each event_type has its own Zod schema for payload validation
2. **Discriminated Union**: event_type field determines which schema to apply
3. **Fail-Fast Validation**: Use `.parse()` to throw validation errors immediately
4. **Helper Functions**: Provide typed constructors for each event type to prevent manual object construction
5. **Metadata Enrichment**: New columns support distributed tracing and debugging

### Validation Strategy
- **Pre-Insert Validation**: Validate payload before database insertion
- **Schema Selection**: Use event_type field to select correct Zod schema
- **Error Messages**: Include event_type and field name in validation errors for debugging
- **Optional Fields**: Use `.optional()` for nullable/optional fields in schemas
- **Enum Validation**: Use Zod enums for constrained values (status, severity, etc.)

### Helper Function Design
- **Named Parameters**: Use object destructuring for readability
- **UUID Generation**: Helpers generate event_id internally using crypto.randomUUID()
- **Type Safety**: Return typed WorkflowEventInput objects
- **Validation**: Helpers validate payload before returning
- **Metadata Support**: All helpers accept optional metadata fields (correlation_id, source, emitted_by)

### Migration Strategy
1. **Backward Compatible**: New columns are nullable, existing events unaffected
2. **Migration File**: Single migration adds all 3 metadata columns
3. **No Data Migration**: Existing events keep NULL values for new columns
4. **Future Events**: New events can populate metadata fields as needed
