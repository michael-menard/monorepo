# WINT-0200: Create User Flows Schema with State/Capability Enums

## Context

The WINT autonomous development workflow needs standardized user flow definitions to enable PO (Product Owner) cohesion checks. Currently, there is no formal schema for defining user flows with required states and capabilities, leading to inconsistent flow definitions and inability to validate completeness.

**Relevant Existing Features:**
- **WINT Database Schemas** (`packages/backend/database-schema/src/schema/wint.ts`) - Established pattern for pgEnum definitions
- **Session Management Types** (`packages/backend/mcp-tools/src/session-management/__types__/index.ts`) - Zod validation pattern with `__types__/` structure
- **Examples Framework (WINT-0180)** - In progress, defines storage and schema patterns this story will follow

**Active Dependencies:**
- **WINT-0180 (Examples Framework)** - Blocking dependency that defines storage strategy (database vs filesystem vs hybrid). Current story assumes JSON Schema file creation, but includes Zod schema for runtime validation regardless of storage choice.

**Problem:**
Without a user flows schema:
1. Agents cannot validate user flow definitions programmatically
2. PO cohesion checks cannot verify state coverage (loading, errors, permissions)
3. Capability completeness checks are manual and error-prone
4. Flow complexity is unbounded (risk of mega-flows)

**Proposed Solution:**
Create `user-flows.schema.json` with:
- **Required states enum**: `loading`, `empty`, `validation_error`, `server_error`, `permission_denied`
- **Required capabilities enum**: `create`, `view`, `edit`, `delete`, `upload`, `replace`, `download`
- **Hard constraints**: max 5 flows per feature, max 7 steps per flow
- **Zod schema for runtime validation**: Parallel TypeScript/Zod schema for agent validation

This schema will be consumed by:
- **WINT-0210** (Role Pack Templates) - PO role pack for cohesion checks
- **WINT-4010** (Cohesion Sidecar) - Automated feature completeness validation
- **WINT-4070** (cohesion-prosecutor Agent) - PO role enforcement

## Architecture Notes

### Schema Design Principles

1. **Dual Schema Approach**
   - JSON Schema for file validation and documentation
   - Zod schema for runtime validation in TypeScript code
   - Both schemas must stay in sync

2. **Enum-Based Validation**
   - States and capabilities are closed enums (no arbitrary strings)
   - Ensures consistent terminology across all user flows
   - Enables automated cohesion checks

3. **Complexity Constraints**
   - Max 5 flows per feature prevents feature sprawl
   - Max 7 steps per flow prevents mega-flow complexity
   - Constraints are hard limits, not recommendations

4. **Schema Versioning**
   - `schema_version` field enables evolution tracking
   - Semantic versioning (major.minor.patch)
   - Future versions can add enums without breaking existing flows

### Storage Strategy (TBD)

Depends on WINT-0180 AC-2 decision:
- **Option 1: Filesystem** - User flows stored as JSON files validated against schema
- **Option 2: Database** - JSONB column with pgEnum for states/capabilities
- **Option 3: Hybrid** - Both file and database storage for different use cases

**Current Implementation:** Both JSON Schema and Zod schema to support any storage choice.

### Integration Points

1. **WINT-0210 (Role Pack Templates)**
   - PO role pack will import Zod schema
   - Validate user flows during elaboration phase
   - Check state coverage and capability completeness

2. **WINT-4010 (Cohesion Sidecar)**
   - Automated validation service
   - Runs cohesion checks using this schema
   - Returns findings for PO review

3. **WINT-4070 (cohesion-prosecutor Agent)**
   - PO role agent enforces completeness
   - Uses schema to validate feature cohesion
   - Blocks incomplete features at gates
