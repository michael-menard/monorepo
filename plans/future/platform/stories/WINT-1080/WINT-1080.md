## Context

The WINT platform (Wave 1, WINT-0010) created 6 schema groups in the main app database under the `wint` PostgreSQL schema namespace. Meanwhile, LangGraph operates on a separate knowledge-base PostgreSQL instance (port 5433) with its own workflow tables defined in raw SQL migrations. Both systems track stories, workflow events, and telemetry, but use different database instances, schema designs, and tooling (Drizzle ORM vs. raw SQL).

### Current State

**WINT Schema** (Main App Database - Port 5432, `wint` namespace):
- Location: `packages/backend/database-schema/src/schema/wint.ts`
- ORM: Drizzle ORM v0.44.3 with TypeScript definitions
- 6 Schema Groups:
  1. Story Management (`stories`, `storyStates`, `storyTransitions`, `storyDependencies`)
  2. Context Cache (`contextPacks`, `contextSessions`, `contextCacheHits`)
  3. Telemetry (`agentInvocations`, `agentDecisions`, `agentOutcomes`, `stateTransitions`)
  4. ML Pipeline (`trainingData`, `mlModels`, `modelPredictions`, `modelMetrics`)
  5. Graph Relational (`features`, `capabilities`, `featureRelationships`, `cohesionRules`)
  6. Workflow Tracking (`workflowExecutions`, `workflowCheckpoints`, `workflowAuditLog`)
- Patterns: UUID primary keys with `defaultRandom()`, timestamps, JSONB metadata, Drizzle relations

**LangGraph Schema** (Knowledge Base Database - Port 5433, `public` namespace):
- Location: `apps/api/knowledge-base/src/db/migrations/002_workflow_tables.sql`
- ORM: Raw SQL migrations
- 8 Table Groups:
  1. Story Management (`stories`, `acceptance_criteria`, `story_risks`)
  2. Elaborations (`elaborations`, `gaps`, `follow_ups`)
  3. Implementation (`implementation_plans`)
  4. Verification (`verifications`, `proofs`)
  5. Telemetry (`token_usage`, `feedback`)
  6. Architecture (`adrs`)
  7. Audit (`workflow_events`)
  8. Supporting (`features` with pgvector embeddings)
- Patterns: UUID primary keys with `gen_random_uuid()`, timestamps, pgvector(1536) for semantic search, PostgreSQL enums, views, functions, triggers

### Problem

The current state creates several issues:

1. **Dual Maintenance Burden**: Schema changes require updates in two places with different patterns (Drizzle ORM vs. raw SQL)
2. **Type Fragmentation**: WINT uses Drizzle+Zod types, LangGraph uses raw SQL - no shared TypeScript types
3. **Data Silos**: Story data is split across two databases, making cross-system queries difficult
4. **Inconsistent Enums**: Story state enums differ (`in_qa` vs `uat`, underscores vs hyphens)
5. **Migration Risk**: Future LangGraph integration (WINT-1090) requires schema alignment
6. **Blocked Dependencies**: WINT-1090 (Update LangGraph Repos) and WINT-1100 (Create Shared TypeScript Types) are both blocked by this story

### Constraints

1. **Protected Features**: Do not modify the knowledge-base PostgreSQL instance schema directly - this is a separate database on port 5433
2. **Schema Isolation**: WINT uses `pgSchema('wint')` namespace in main app database (port 5432), LangGraph uses default `public` schema in knowledge-base database (port 5433)
3. **Two Databases**: Main app database vs. knowledge-base database are physically separate instances
4. **Drizzle ORM**: All schema changes must use Drizzle ORM v0.44.3 with drizzle-zod for Zod schema generation
5. **Migration Strategy**: Use Drizzle Kit for all migrations, no manual SQL
6. **Backward Compatibility**: LangGraph is actively used - any schema changes must preserve existing functionality

## Goal

Perform a comprehensive schema diff analysis between WINT and LangGraph schemas, then create a unified schema design that:

1. Identifies overlapping tables and reconciles differences
2. Determines which schema serves as the source of truth for each domain
3. Proposes a migration path to align both systems
4. Documents the unified schema with clear ownership boundaries
5. Ensures backward compatibility with existing LangGraph functionality
6. Creates a foundation for WINT-1100 to generate shared TypeScript types

**Approach**:
- Analyze both schemas table-by-table to identify overlap and divergence
- Create a unified schema specification document
- Define migration strategy (align LangGraph to WINT, merge, or maintain separation)
- Generate migration scripts for schema alignment
- Validate that existing LangGraph queries/functions remain compatible

**Deliverable**: This is primarily an analysis and design story. The output is documentation and a migration plan, not immediate code changes.

- **Data Migration**: This story defines the schema, but does not migrate existing LangGraph data (deferred to WINT-1110)
- **Code Updates**: Do not update LangGraph repository code (deferred to WINT-1090)

## Reuse Plan

### Components

- **WINT schema patterns** from `wint.ts`:
  - Enum definitions: `pgEnum()` pattern
  - Index strategies: Composite indexes, partial indexes
  - Relations: Drizzle relations API for lazy loading
  - UUID primary keys: `defaultRandom()` pattern
  - Timestamps: `defaultNow()` pattern

- **LangGraph state transition functions and triggers**:
  - PostgreSQL functions for workflow logic
  - Triggers for auto-updating timestamps
  - Event logging patterns

- **Drizzle ORM relation patterns** from existing schemas:
  - One-to-many: `relations(stories, ({ many }) => ({ dependencies: many(storyDependencies) }))`
  - Many-to-many: Junction tables with foreign keys
  - Lazy loading: `with` clause for eager loading

### Patterns

- `pgSchema()` for namespace isolation: `export const wintSchema = pgSchema('wint')`
- UUID primary keys: `id: uuid('id').primaryKey().defaultRandom()`
- Timestamps: `created_at: timestamp('created_at').defaultNow()`, `updated_at: timestamp('updated_at').defaultNow()`
- JSONB for flexible metadata: `metadata: jsonb('metadata')`
- Composite indexes: `index('idx_story_state_priority').on(table.state, table.priority)`
- Partial indexes: `index('idx_active_stories').on(table.id).where(sql\`state != 'done'\`)`

### Packages

- **drizzle-orm** v0.44.3 - ORM for schema definition and queries
- **drizzle-zod** - Zod schema generation from Drizzle schemas
- **drizzle-kit** - Migration generation and schema diffing
- **@repo/database-schema** - Central location for unified schema
- **@repo/db** - Database connection pooling (max 1 per Lambda per ADR-005)

## Architecture Notes

### Two Database Strategy

This story maintains the two-database architecture:

1. **Main App Database** (port 5432):
   - WINT schema in `wint` namespace
   - Transactional data, context caching, ML pipeline
   - Drizzle ORM managed

2. **Knowledge Base Database** (port 5433):
   - LangGraph schema in `public` namespace (migrating to align with WINT)
   - Workflow tracking, elaborations, proofs
   - pgvector for semantic search
   - Eventually managed by Drizzle ORM (post-migration)

**Rationale for Separation**:
- Isolation: LangGraph workflow data isolated from main app
- Performance: Separate connection pools, no contention
- Scalability: Independent scaling of each database
- Migration Safety: Staged migration reduces risk

**Future Consolidation** (deferred):
- May merge databases in future if performance allows
- Would require migrating LangGraph tables to main app database
- Out of scope for this story

### Schema Namespace Strategy

- **WINT Schema**: Uses `pgSchema('wint')` namespace in main app database
- **LangGraph Schema**: Uses `public` namespace in knowledge-base database
- **Migration Strategy**: Align LangGraph tables to WINT schema patterns, keep in `public` namespace initially
- **Future**: May migrate LangGraph tables to `wint` namespace if databases merge

### Enum Migration Strategy

Enum migration requires careful handling:

1. **Create Unified Enum**: Define new enum with all states (underscored naming)
2. **Dual Enum Period**: Support both enums during migration (WINT-1110)
3. **Data Migration**: Transform LangGraph enum values (hyphenated → underscored, `uat` → `in_qa`)
4. **Cutover**: Switch LangGraph to use unified enum
5. **Cleanup**: Remove old enum after migration complete

This story defines the unified enum and migration strategy. WINT-1110 executes the data migration.
