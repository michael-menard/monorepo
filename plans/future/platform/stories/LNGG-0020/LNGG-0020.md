# LNGG-0020: Index Management Adapter — stories.index.md Updates

## Context

The platform stories index (`platform.stories.index.md`) is a **markdown table format** tracking 235 stories across 11 epics. The file uses:
- YAML frontmatter for metadata (doc_type, status, dates)
- Markdown headers for wave organization
- Markdown tables with columns: `#`, `Story`, `Title`, `← Depends On`, `Epic`, `Priority`
- Inline status markers like `[x]`, `[~]`, `[ ]` in the Title column
- Special symbols (⚡ for critical, 🎯 for milestones, ✨ for LangGraph work)

### Current State

**LNGG-0010 Status**: In-progress with schema resolution complete and adapter implementation underway. The StoryFileAdapter provides atomic file operations and Zod validation patterns that can be reused.

**Available Components**:
- File utils (atomic writes, safe reads, directory creation) at `packages/backend/orchestrator/src/adapters/utils/file-utils.ts`
- YAML parser utils (gray-matter) at `packages/backend/orchestrator/src/adapters/utils/yaml-parser.ts`
- Error classes at `packages/backend/orchestrator/src/adapters/__types__/index.ts`

**No programmatic way to**:
- Update markdown table portion of index files
- Add new stories to index tables
- Update story status in tables
- Recalculate wave/epic metrics from table data
- Validate table structure (dependencies, cycles)
- Handle concurrent updates safely

### Problem

LangGraph workflows need to:
1. Add new stories to index tables (append row to appropriate wave table)
2. Update story status in tables (change inline markers like `[ ]` → `[~]` → `[x]`)
3. Update story metadata (dependencies, priority, epic)
4. Recalculate wave/epic metrics (count stories by status)
5. Preserve markdown formatting and manual annotations
6. Handle concurrent updates safely (transaction support)

**Critical Discovery**: The original LNGG-002 spec assumed YAML-based index format. The **actual format is markdown tables**, requiring different parsing and update strategies.

### Impact

Without index updates, story status is invisible to team, metrics drift, dependencies break, and workflow automation cannot track progress.

---

## Goal

Create a type-safe `IndexAdapter` class that:
- Parses markdown table format (not YAML frontmatter per-story as original spec assumed)
- Updates table rows by story ID
- Recalculates metrics from table data
- Validates table structure (story ID uniqueness, dependency references)
- Supports atomic updates (read → modify → validate → write)
- Preserves formatting and manual edits outside tables

---

## Non-Goals

**Explicit Out of Scope**:
1. UI/UX changes - This is a backend adapter, no user interface
2. Database persistence - File-based only, no DB writes (WINT-0020 handles DB)
3. Real-time collaboration - No conflict resolution for concurrent edits
4. Version control integration - No git operations, just file writes
5. Index format migration - Keep markdown table format, don't migrate to YAML/JSON
6. Legacy index format support - Only support current markdown table format
7. Cross-epic validation - Only validate within single index file
8. Dependency graph visualization - Only detect cycles, don't render graphs

**Protected Areas (Do Not Touch)**:
- `@repo/db` client package
- Production database schemas
- Orchestrator artifact schemas (read-only reference)
- Existing story files (those are StoryFileAdapter's responsibility)

  by_wave: z.record(z.number().int().min(0)),

})

export type ValidationError = z.infer<typeof ValidationErrorSchema>

export const ValidationResultSchema = z.object({
  valid: z.boolean(),
  errors: z.array(ValidationErrorSchema),
})

export type ValidationResult = z.infer<typeof ValidationResultSchema>

export const StoryIndexSchema = z.object({
  frontmatter: z.object({
    doc_type: z.string(),

### Markdown Table Parsing Strategy

**Option 1: Regex-based** (MVP approach)
- Parse tables using regex patterns
- Pros: Simple, no dependencies
- Cons: Fragile with edge cases, hard to maintain

**Option 2: remark/unified AST** (Recommended for production)
- Use `remark-parse` to build AST
- Use `mdast-util-to-markdown` to serialize
- Pros: Robust, handles edge cases, maintainable
- Cons: Additional dependency, slight complexity

**Decision**: Start with Option 1 for MVP, upgrade to Option 2 if regex proves fragile during testing.

#### Wave Section Detection (Added by elaboration)

**Regex pattern for wave headers**:
```typescript
const WAVE_HEADER_REGEX = /^## (Wave \d+) — (.+?) \((\d+) stories\)$/
```

**Parsing algorithm**:
1. Split markdown content by wave header regex
2. For each section:
   - Extract wave name (e.g., "Wave 2")
   - Extract description (e.g., "Core Infrastructure")
   - Extract expected story count from header
3. Locate markdown table within section (starts with `| # | Story |`)
4. Parse table rows, associating each story with the current wave
5. Validate actual story count matches header count

**Example**:
```markdown
## Wave 2 — Core Infrastructure (15 stories)

| # | Story | Title | ← Depends On | Epic | Priority |
|---|-------|-------|--------------|------|----------|
| 12 | LNGG-0020 | Index Adapter | ← LNGG-0010 | LNGG | P0 |
```

Parsed result:
```typescript
{
  wave: "Wave 2",
  description: "Core Infrastructure",
  expectedStoryCount: 15,
  stories: [
    {
      number: 12,
      story_id: "LNGG-0020",
      title: "Index Adapter",
      depends_on: ["LNGG-0010"],
      epic: "LNGG",
      priority: "P0",
      wave: "Wave 2"
    }
  ]
}
```
