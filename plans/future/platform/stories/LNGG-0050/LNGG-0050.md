# LNGG-0050: KB Writing Adapter

## Context

The orchestrator package currently has KB writing capabilities embedded directly in workflow nodes (`persist-learnings.ts`, `migrate-workflow-knowledge.ts`), but no unified adapter interface. As the LNGG epic builds out LangGraph infrastructure, we need a clean abstraction for KB write operations that:

1. Decouples workflow logic from KB implementation details
2. Provides a consistent interface for all KB write operations
3. Enables dependency injection for testing and flexibility
4. Handles deduplication, error handling, and logging uniformly

**Current State:**

- KB write logic exists in `persist-learnings.ts` (learnings only)
- KB schema defined at `apps/api/knowledge-base/src/db/schema.ts`
- Embedding client exists but not formalized in adapter pattern
- No centralized interface for KB writes across different entry types (lessons, decisions, constraints, etc.)
- Knowledge Base using pgvector (port 5433) with `knowledgeEntries` table
- Vector dimension: 1536 (OpenAI text-embedding-3-small)

**Problem:**

Without a KB Writing Adapter:
- Workflow nodes duplicate KB write logic
- Testing requires mocking multiple KB functions independently
- No consistent deduplication strategy across all KB writes
- Error handling varies between different KB write paths
- Adding new entry types requires modifying multiple files
- No single source of truth for KB write operations

**Proposed Solution:**

Create a KB Writing Adapter at `packages/backend/orchestrator/src/adapters/kb-writer/` that:
1. Exposes a unified `KbWriter` interface with methods for each entry type
2. Handles embedding generation, deduplication, and persistence
3. Provides factory function for dependency injection
4. Supports all KB entry types: lessons, decisions, constraints, runbooks, notes
5. Includes comprehensive error handling and logging
6. Works gracefully when KB is unavailable (fallback mode)

## Goal

Build a reusable, testable KB Writing Adapter that standardizes all Knowledge Base write operations across the orchestrator package, enabling consistent deduplication, error handling, and dependency injection for all KB entry types.

## Non-Goals

## Architecture Notes

**Design Principles:**
1. **Dependency Injection** - All KB dependencies (db, embeddingClient) injected via factory config
2. **Graceful Degradation** - Adapter returns no-op writer when KB unavailable (doesn't crash workflow)
3. **Deduplication First** - Always search for similar content before writing (prevents KB bloat)
4. **Type Safety** - All types defined via Zod schemas with runtime validation
5. **Single Responsibility** - Adapter only handles writes; reads are separate concern

**Factory Pattern:**
```typescript
// Example usage
const kbWriter = createKbWriter({
  kbDeps: {
    db: drizzleDb,
    embeddingClient: openAIClient,
    kbAddFn: kbBulkImport,
    kbSearchFn: kbSearch
  }
})

// Write with automatic deduplication
const result = await kbWriter.addLesson({
  content: "Lesson learned content",
  storyId: "LNGG-0050",
  role: "dev",
  tags: ["pattern", "deduplication"]
})
```

**Error Handling Strategy:**
- KB write failures return error result (don't throw)
- Log errors via `@repo/logger.error()`
- No retry logic initially (defer to reliability story)
- Partial failures in batch writes logged individually

**Deduplication Algorithm:**
1. Generate embedding for new content
2. Search KB for similar embeddings (cosine similarity)
3. If similarity > 0.85, skip write and return "duplicate" result
4. If similarity <= 0.85, proceed with write
5. Return result with metadata (persisted, skipped, duplicate_id)

## Development Feasibility

### Technical Risks

**1. Embedding Generation Performance**
- **Risk**: OpenAI API latency (100-200ms per call) could slow down writes
- **Impact**: Medium - affects write throughput
- **Mitigation**:
  - Initial implementation tolerates latency (not high-throughput use case)
  - Future optimization: batch embedding calls (deferred)
  - Document expected latency in adapter README

**2. KB Deduplication at Scale**
- **Risk**: Similarity search slows down with large KB (>10k entries)
- **Impact**: Medium - affects write performance over time
- **Mitigation**:
  - IVFFlat index already configured on KB
  - Acceptable for initial implementation (<10k entries expected)
  - Monitor query performance in integration tests
  - Optimization deferred to separate performance story

**3. Dependency Injection Complexity**
- **Risk**: Many optional dependencies (db, embeddingClient, kbAddFn, kbSearchFn) increases complexity
- **Impact**: Low - well-contained with Zod validation
- **Mitigation**:
  - Zod schema validation catches missing dependencies early
  - No-op writer provides clear fallback behavior
  - Factory pattern isolates complexity
