# WINT-0010: Create Core Database Schemas (6 schemas)

## Context

The WINT (Workflow Intelligence) epic requires a comprehensive database foundation to track story management, context caching, telemetry, ML pipelines, graph relationships, and workflow execution. This foundational Wave 1 story creates 6 core PostgreSQL schemas that will support the entire WINT infrastructure.

**Current State**: The orchestrator uses file-based YAML artifacts (PLAN.md, CHECKPOINT.yaml, EVIDENCE.yaml, etc.) which lack:
- Queryability for analytics and reporting
- Scalability for large datasets
- Real-time access patterns
- Relational integrity constraints
- Transaction support for atomic updates

**Problem**: To build autonomous development features, we need persistent, queryable storage for:
1. **Story metadata and lifecycle tracking** - Current state is scattered across directory structures and YAML files
2. **Context cache** - No mechanism to cache frequently-used context to reduce token usage
3. **Telemetry and observability** - Agent invocations and decisions are ephemeral, not queryable
4. **ML pipeline data** - No infrastructure for model selection and quality prediction
5. **Graph relationships** - Feature capabilities and cohesion rules are not formalized
6. **Workflow execution state** - Execution tracking relies on file timestamps and manual artifact inspection

**Reality Baseline** (2026-02-13):
- Drizzle ORM v0.44.3 installed and active
- PostgreSQL Aurora in production
- `@repo/db` package provides connection pooling
- `drizzle-zod` available for auto-generating Zod schemas
- `umami.ts` demonstrates `pgSchema()` pattern for namespace isolation
- No active platform stories (no overlap risk)

**Dependencies**: None (Wave 1 story)

**Blocks**: 7 downstream stories (WINT-0020, WINT-0030, WINT-0040, WINT-0060, WINT-0070, AUTO-0010, AUTO-0020)

## Goal

Create 6 isolated database schemas using Drizzle ORM and PostgreSQL, leveraging `pgSchema('wint')` for namespace isolation. Each schema will be self-contained with appropriate indexes, relations, and constraints. The schemas will follow established patterns from existing production schemas but remain isolated to avoid conflicts.

**Success Criteria**:
- All 6 schema groups defined in a single `wint.ts` file
- Schemas isolated via `pgSchema('wint')` namespace
- Zod schemas auto-generated via `drizzle-zod`
- Comprehensive unit tests validate structure and constraints
- Migration files generated via `drizzle-kit`
- Schema design documented in inline comments

## Architecture Notes

### Schema Organization

The WINT schemas will be organized into 6 logical groups:

1. **Story Management** - Core story tracking and lifecycle
2. **Context Cache** - Performance optimization via cached context
3. **Telemetry** - Observability and analytics
4. **ML Pipeline** - Machine learning infrastructure
5. **Graph Relational** - Feature capabilities and cohesion
6. **Workflow Tracking** - Execution state and audit trails

### Namespace Isolation

Using `pgSchema('wint')` provides:
- **Namespace isolation** - No table name conflicts with production schemas
- **Security** - Separate database roles can be assigned per schema
- **Organization** - Clear separation of concerns
- **Migration safety** - WINT migrations don't affect production tables

### Index Strategy

Indexes will be optimized for common query patterns:
- **By story_id** - Most queries filter by story
- **By state/status** - State-based filtering is common
- **By timestamp** - Time-series analytics queries
- **Composite indexes** - For multi-column filters (e.g., `story_id` + `state`)

### Relations Strategy

Drizzle relations will enable:
- **Lazy loading** - Load related data on-demand
- **Type safety** - TypeScript inference for related entities
- **Query optimization** - Automatic JOIN generation

### Zod Schema Generation

Auto-generated Zod schemas provide:
- **Runtime validation** - Validate input before database insertion
- **Type inference** - Single source of truth for types
- **Self-documenting** - Constraints embedded in schema
- **API contracts** - Use for request/response validation

### Technical Considerations

**Schema Isolation**
- `pgSchema('wint')` prevents table name conflicts with production tables
- All queries must reference the `wint` schema explicitly
- Database roles can be scoped to `wint` schema for security

**UUID Primary Keys**
- Use `uuid('id').primaryKey().defaultRandom()` pattern
- Distributed system compatibility (no centralized sequences)
- Performance: UUIDs are larger than integers but provide global uniqueness

**Timestamps**
- All tables include `created_at` and `updated_at`
- Use `timestamp('created_at', { withTimezone: true }).notNull().defaultNow()`
- Enables audit trails and temporal queries

**JSONB Fields**
- Use for flexible metadata (e.g., model hyperparameters, decision context)
- Avoid for structured data with known schema
- PostgreSQL JSONB provides indexing and query support

**Index Design**
- Composite indexes for multi-column filters (e.g., `story_id` + `state`)
- Single-column indexes on foreign keys
- Unique indexes where business rules require uniqueness
- Balance: indexes speed reads but slow writes

**Relations**
- One-to-many: Parent table has `many()` relation, child has `one()`
- Many-to-many: Junction table with foreign keys to both entities
- Lazy loading: Relations loaded only when accessed
