# MODL-0030: Quality Evaluator

## Context

MODL-0020 (Task Contracts & Model Selector) established task-based tier selection, enabling tasks to declare their requirements via contracts and receive optimal model assignments. However, there is currently **no validation mechanism** to assess whether the selected model actually delivered the expected quality.

**Current State:**
- Task contracts define quality requirements: `adequate | good | high | critical`
- Task selector routes tasks to tiers based on complexity, quality, and security
- Model execution happens via `ModelRouter` and provider adapters
- **No feedback loop** - we cannot verify if tier selection was optimal

**Problem:**
Without quality evaluation, we cannot:
- Validate tier selection decisions (was Tier 3 sufficient? Was Tier 0 overkill?)
- Detect over-provisioning (task routed to expensive tier when cheaper tier would work)
- Detect under-provisioning (task failed quality bar, needs higher tier)
- Feed quality data to MODL-0040 (Model Leaderboards)
- Train ML-based selection models (WINT-5xxx)
- Tune task contracts based on actual outcomes

**Examples of Quality Issues:**
1. Task marked `complexity: 'low', qualityRequirement: 'adequate'` routed to Tier 3 (Ollama) produces low-quality output requiring rework → under-provisioned
2. Task marked `qualityRequirement: 'critical'` routed to Tier 0 (Opus) succeeds, but Tier 1 (Sonnet) would have met bar at 50% cost → over-provisioned
3. Security-sensitive task routed to Tier 0/1 outputs code with security flaw → quality validation needed
4. Budget-constrained task de-escalated to Tier 2 produces acceptable quality → optimal tier selection (need data to prove it)

---

## Goal

Create a **Quality Evaluator** that validates model output against task contract quality requirements and identifies tier selection mismatches.

**What We're Building:**
1. `QualityEvaluationSchema` - Zod schema for quality assessment results
2. `evaluateQuality(contract, tier, output)` - Function to score output quality (0-100)
3. Dimension-specific evaluators - Correctness, completeness, coherence, compliance, cost-efficiency
4. Contract mismatch detector - Identifies over/under-provisioned tier selections
5. ModelRouter integration - Optional `evaluateQuality()` method for post-execution validation
6. Configuration - Quality thresholds per requirement level (adequate=60, good=75, high=85, critical=95)

**Quality Dimensions:**
- **Correctness** - Does output satisfy acceptance criteria? (rule-based checks)
- **Completeness** - Are all required elements present? (schema validation)
- **Coherence** - Is reasoning sound and well-structured? (heuristic checks)
- **Compliance** - Does output meet security/quality constraints? (contract validation)
- **Cost-Efficiency** - Could lower tier achieve same quality? (tier comparison)

**How It Works:**
```
Task Contract → Tier Selection → Model Execution → Quality Evaluation
                (MODL-0020)     (ModelRouter)     (MODL-0030 - NEW)
                                                       ↓
                                                Quality Score (0-100)
                                                Dimension Breakdown
                                                Mismatch Detection
                                                Tier Recommendation
                                                       ↓
                                                Logged for MODL-0040
```

**Integration:**
Quality evaluator is invoked **post-execution** to validate model selections. Initially used for logging and metrics (via @repo/logger). Future integration with ModelRouter for dynamic tier adjustment based on historical quality data.

## Non-Goals

- ❌ **ML-Based Quality Prediction** - WINT-5xxx scope. This story uses rule-based evaluation only.
- ❌ **Model Leaderboards** - MODL-0040 scope. Quality evaluator provides data source, leaderboard UI is separate.
- ❌ **Automatic Tier Adjustment** - Follow-up story. This story logs recommendations, doesn't auto-adjust.
- ❌ **Quality Evaluation Persistence** - MODL-0040 scope. MVP logs via @repo/logger, DB persistence deferred.
- ❌ **LLM-as-Judge Evaluation** - Future enhancement. MVP uses rule-based/heuristic evaluation for speed.
- ❌ **Quality Evaluation UI** - AUTO epic scope. Backend API only, no dashboard.
- ❌ **Historical Quality Trends** - MODL-0040 scope. This story provides point-in-time evaluation only.

### AC-2: Quality Scoring Logic ✅

Implement `evaluateQuality(contract, tier, output)` function:
- Accept `TaskContract`, tier (string), output (string)
- Score output against contract quality requirements:
  - `qualityRequirement: 'adequate'` → threshold 60
  - `qualityRequirement: 'good'` → threshold 75
  - `qualityRequirement: 'high'` → threshold 85
  - `qualityRequirement: 'critical'` → threshold 95
- Calculate dimension-specific scores (correctness, completeness, coherence, compliance, cost-efficiency)
- Return overall score as weighted average of dimensions
- Log all scoring decisions with full context via `@repo/logger`

**Pattern:** Follow `readiness-score.ts` base score with deductions/additions pattern

**Tests:** 20+ scoring logic test cases (all thresholds, dimension integration, weighted averaging)

---

### AC-3: Contract Mismatch Detection ✅

Compare achieved quality score against expected quality from contract:

**Over-Provisioning Detection:**
- Score significantly exceeds threshold (e.g., `qualityRequirement: 'good'` (75 threshold) but achieved 95+)
- Lower tier could have met requirements at lower cost
- Flag `contractMismatch: true`, set `recommendation: "tier-2"` (example)

**Under-Provisioning Detection:**
- Score falls below threshold (e.g., `qualityRequirement: 'critical'` (95 threshold) but achieved 85)
- Higher tier needed to meet quality requirements
- Flag `contractMismatch: true`, set `recommendation: "tier-0"` (example)

**Threshold for "Significant" Over-Provisioning:**
- Score exceeds threshold by 20+ points (e.g., adequate=60, achieved 85+)
- Configurable in quality thresholds

**Logging:**
- Log all mismatch detections with context (expected, actual, recommendation)
- Use `logger.warn('contract_mismatch_detected', { ... })`

**Tests:** 10+ contract mismatch test cases (over/under-provisioning, edge cases)

### AC-4: Quality Dimension Evaluators ✅

Implement evaluators for each quality dimension:

**1. Correctness Evaluator** (`evaluateCorrectness(contract, output)`)
- Does output match requirements? (rule-based checks)
- Return score (0-100) with rationale
- Example: Check for required keywords, structure, logic

**2. Completeness Evaluator** (`evaluateCompleteness(contract, output)`)
- Are all required elements present? (schema validation)
- Return score (0-100) with rationale
- Example: Count expected elements vs present elements

**3. Coherence Evaluator** (`evaluateCoherence(contract, output)`)
- Is reasoning sound? (heuristic checks)
- Return score (0-100) with rationale
- Example: Check for logical flow, no contradictions

**4. Compliance Evaluator** (`evaluateCompliance(contract, output)`)
- Meets security/quality constraints? (contract flag validation)
- Return score (0-100) with rationale
- Example: Check security flags, quality constraints from contract

**5. Cost-Efficiency Evaluator** (`evaluateCostEfficiency(contract, tier, score)`)
- Could lower tier achieve same quality? (tier comparison)
- Return score (0-100) with rationale
- Example: If tier-0 achieved 95 but tier-1 threshold is 85, cost-efficiency low

**Each Evaluator:**
- Returns `{ dimension: string, score: number, rationale: string, weight: number }`
- Logs evaluation decision via `@repo/logger`
- Configurable weights per dimension (default: equal weight = 0.2 each)

**Tests:** 25+ dimension evaluator test cases (5 per dimension, edge cases)

const QualityDimensionScoreSchema = z.object({
