## Context

The orchestrator currently persists workflow artifacts (story, checkpoint, plan, scope, evidence, review, qa-verify, etc.) as YAML files on the filesystem. These artifacts are defined with Zod schemas in `packages/backend/orchestrator/src/artifacts/`. The KBAR schema (KBAR-0010) has `artifacts`, `artifact_versions`, and `artifact_content_cache` tables, but these are designed for file sync state, not for storing the actual artifact content in normalized relational form.

**Problem**: Artifacts stored only as YAML files are difficult to query, aggregate, and analyze. We cannot efficiently answer questions like:
- "Show me all stories where AC-3 failed QA"
- "What are the most common blockers across all stories?"
- "Which acceptance criteria take longest to implement?"
- "Generate a trend report on evidence quality over time"

**Solution**: Create Postgres schemas for the 4 core workflow artifact types (story, checkpoint, scope, plan), enabling dual persistence (YAML for human readability + Postgres for querying). This story creates the database schema foundations that INFR-0020 (Artifact Writer/Reader Service) will use to sync artifacts bi-directionally.

**Split Scope**: This story (INFR-0110) handles the 4 core workflow artifacts that drive story execution. Review/QA artifacts (evidence, review, qa-verify) are handled in INFR-0120.

**Reality Baseline** (from 2026-02-13):

## Architecture Notes

### Enum Location Decision

**Decision**: Place `artifact_type_enum` in `public` schema (not `artifacts` schema).

**Rationale**:
1. **Precedent**: WINT and KBAR schemas follow this pattern for cross-namespace enums
2. **Reusability**: KBAR sync tools need to reference artifact types from both `artifacts` and `kbar` schemas
3. **Future-proof**: If other schemas need to reference artifact types, they can without cross-schema dependencies
4. **All 7 types defined upfront**: Avoids ALTER TYPE in INFR-0120, simplifies migration coordination

### Schema Alignment Strategy

**Challenge**: Postgres schema field names must match Zod artifact schema field names exactly to enable INFR-0020 sync.

**Solution**:
- Use snake_case in Postgres (PostgreSQL convention)
- Use camelCase in TypeScript (JavaScript convention)
- Drizzle ORM handles transformation automatically via column aliases
- Document mapping in SCHEMA-REFERENCE.md

**Example**:
```typescript
// Drizzle schema
export const storyArtifactsTable = pgTable('story_artifacts', {
  storyId: uuid('story_id').notNull(), // DB: story_id, TS: storyId
  acceptanceCriteria: jsonb('acceptance_criteria').$type<AC[]>(),
})

// Auto-generated Zod schema
const StoryArtifactInsertSchema = createInsertSchema(storyArtifactsTable)
// Type: { storyId: string, acceptanceCriteria: AC[], ... }
```

### JSONB Denormalization Trade-offs

**Why JSONB arrays instead of normalized tables?**

1. **Co-location**: Acceptance criteria, risks, file changes always loaded together with parent artifact
2. **Size**: Max ~20 ACs per story, ~50 file changes per plan → fits JSONB performance profile (<100KB)
3. **Query patterns**: Most queries filter by story_id, not individual AC fields
4. **Schema evolution**: Zod schemas can evolve without migration for new JSONB fields

**When to normalize instead?**

- Individual items queried frequently (e.g., "show all AC-3 across all stories")
- Items have many-to-many relationships
- Items need separate lifecycle/permissions
- Size exceeds 1MB per JSONB field

**Decision for INFR-0110**: Use JSONB denormalization. Normalize later if query patterns require (tracked in FUTURE-OPPORTUNITIES.md for INFR-0020+).

### Foreign Key Cascade Behavior

**Decision**: Add `ON DELETE CASCADE` to foreign key constraints.

**Rationale**:
- When a story is deleted from `wint.stories`, all related artifacts should be automatically deleted
- Prevents orphaned artifact records
- Story deletion is a rare, intentional operation (not accidental)
- Simplifies cleanup logic in INFR-0020

**Risk**: Story deletion is destructive and irreversible. Mitigation: Document in SCHEMA-REFERENCE.md, add confirmation prompts in future story deletion tools.

### pgSchema Namespace Strategy

**Isolation Rationale**:
- `kbar` schema: File sync state (artifacts, artifact_versions, artifact_content_cache)
- `wint` schema: Workflow runtime state (stories, workflows, sessions)
- `artifacts` schema: Artifact relational data (this story)
- `telemetry` schema: Event tracking (INFR-0040)

**Cross-Namespace Dependencies**:
- `artifacts.story_artifacts.story_id` → `wint.stories.story_id` (FK with ON DELETE CASCADE)
- `artifacts.artifact_type_enum` in `public` schema for shared access

**Benefits**:
- Clear domain boundaries
- Independent migration files
- Easier to reason about data ownership
