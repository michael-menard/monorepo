# KBAR-0010: Database Schema Migrations

## Context

The KBAR (Knowledge Base Artifact Repository) epic requires database tables to store story metadata, artifact metadata, story-artifact relationships, and sync state tracking. Currently, all story and artifact data exists only as YAML files in the filesystem. This story establishes the foundational schema for the KBAR system.

**Problem**: Without database storage, we cannot efficiently:
- Query stories by status, dependencies, or metadata
- Track artifact sync state (file checksum vs DB state)
- Support fast MCP tool queries for story/artifact data
- Maintain referential integrity between stories and artifacts
- Enable agent-driven index generation from DB

**Solution Direction**: Create a new PostgreSQL schema namespace `kbar` with four core table groups following the WINT schema pattern established in migration 0015_messy_sugar_man.sql.

## Goal

Create a complete, production-ready KBAR database schema with:
- Isolated `kbar` schema namespace (following WINT pattern)
- 11 tables across 4 functional groups (stories, artifacts, sync state, index generation)
- Comprehensive indexing for query performance
- Type-safe Zod schemas auto-generated via drizzle-zod
- Drizzle relations for ORM queries
- Successful migration applied to dev environment

### Database Tables

**Story Tables** (3 tables):
- `kbar.stories` - Core story metadata (ID, title, epic, phase, status, priority, metadata JSONB)
- `kbar.story_states` - Story state history and transitions
- `kbar.story_dependencies` - Dependency relationships (blocks, requires, related)

**Artifact Tables** (3 tables):
- `kbar.artifacts` - Artifact metadata (story_id FK, type, path, checksum, status)
- `kbar.artifact_versions` - Version history for auditing
- `kbar.artifact_content_cache` - Optional caching of parsed YAML content

**Sync State Tables** (3 tables):
- `kbar.sync_events` - Track file→DB sync operations
- `kbar.sync_conflicts` - Log sync conflicts for resolution
- `kbar.sync_checkpoints` - Checkpoint state for incremental syncs

**Index Generation Tables** (2 tables):
- `kbar.index_metadata` - Metadata about generated index files
- `kbar.index_entries` - Individual entries in generated indexes

## Architecture Notes

### Schema Isolation
- KBAR tables in dedicated `kbar` schema namespace
- Enums in public schema (reusable across namespaces)
- No cross-schema FK dependencies in MVP
- Pattern matches WINT schema organization

### Cascading Deletes
- `artifacts` → CASCADE on `stories` delete (artifacts belong to story)
- `story_states` → CASCADE on `stories` delete
- `story_dependencies` → CASCADE on `stories` delete
- `artifact_versions` → CASCADE on `artifacts` delete
- Explicit ON DELETE behavior for all FKs

### Timestamp Strategy
- All tables include `created_at` (defaultNow)
- All tables include `updated_at` (defaultNow, updated by trigger or app)
- No soft deletes in MVP (use `deleted_at` if needed in future)

### Index Strategy
- All foreign keys indexed (performance critical)
- Common query fields indexed: epic, phase, status
- Composite indexes deferred to KBAR-0020+ (based on actual query patterns)
- No GIN indexes on JSONB in MVP (defer to KBAR-0080+ based on performance data)

### Type Safety
- Drizzle schema → TypeScript types (single source of truth)
- drizzle-zod → Zod schemas for runtime validation
- Zod-first types per CLAUDE.md requirement
- No manual TypeScript interfaces

### JSONB Metadata Structure

**stories.metadata** (optional fields):

  tags?: string[]                    // Story tags

**artifact_content_cache.content** (parsed YAML):

### Key Technical Decisions

**Decision 1: JSONB vs Separate Columns for Story Metadata**
- ✅ **Chosen**: JSONB metadata column
- Rationale: Flexibility for extensible metadata, no migration needed for new fields
- Trade-off: Slower queries (acceptable for MVP, low volume)

**Decision 2: Artifact Content Caching Strategy**
- ✅ **Chosen**: Cache parsed YAML as JSONB
- Rationale: Fast queries without re-parsing, supports partial updates
- Trade-off: Slightly larger storage (acceptable)

**Decision 3: Sync Conflict Resolution**
- ✅ **Chosen**: Log conflicts, manual resolution
- Rationale: Safer for MVP, auto-resolve can be added later
- Trade-off: Requires manual conflict resolution (acceptable for MVP)

**Decision 4: Index Generation Table Structure**
- ✅ **Chosen**: Hierarchical (metadata + entries)
- Rationale: Separates index metadata from entries, enables efficient queries
- Trade-off: Slightly more complex schema (acceptable)
