# MODL-0040: Model Leaderboards â€” Per-Task Quality/Cost/Latency Tracking with Convergence Detection

## Summary

Implement a leaderboard module in `packages/backend/orchestrator/src/model-selector/` that tracks per-task, per-model quality/cost/latency statistics, detects convergence (when a best model is identified with â‰Ą95% confidence), alerts on quality degradation (>10% drop), and generates human-readable reports. ## Context

MODL-0010 (Provider Adapters), MODL-0020 (Task Contracts and Model Selector), and MODL-0030 (Quality Evaluator) have established the foundational model experimentation infrastructure in `packages/backend/orchestrator/src/`. The quality evaluator produces a `QualityEvaluation` record (0â€“100 composite score across five dimensions) after each model invocation. The tier-based model selector uses Thompson Sampling to pick models dynamically.

**Current State:**
- `evaluateQuality(contract, tier, output) â†’ QualityEvaluation` is implemented and in UAT (MODL-0030)
- There is no persistence layer for run-level results
- There is no aggregation of quality scores across runs
- There is no convergence detection (the model selector never "locks in" a model)
- There is no degradation alert mechanism
- There is no report generation for observability

**Problem:**
Without a leaderboard, the model selection system cannot learn. Each run is isolated â€” Thompson Sampling picks models but accumulates no persistent evidence to reinforce winners. Cost/quality trade-offs are invisible. A regression in a model's performance (quality drops >10%) goes undetected. The LERN and SDLC epics that depend on MODL completion cannot proceed without a working feedback loop.

Implement a YAML-persisted leaderboard for per-task, per-model quality/cost/latency tracking with convergence detection and report generation.

## Architecture Notes

### RunRecord Input Design (Option A â€” Additive)

`QualityEvaluation` (MODL-0030 output) does NOT carry `cost_usd` or `latency_ms`. These live at the provider call layer. The leaderboard accepts a `RunRecord` that wraps `QualityEvaluation` with additional telemetry:

```typescript
const RunRecordSchema = QualityEvaluationSchema.extend({
  cost_usd: z.number().nonnegative(),
  latency_ms: z.number().int().nonnegative(),
  tokens_in: z.number().int().nonnegative().optional(),
  tokens_out: z.number().int().nonnegative().optional(),
})
```

The caller (whoever invokes both the provider and the quality evaluator) assembles the `RunRecord`. This is entirely additive â€” MODL-0030 is not modified.

### Module Structure

```
model-selector/leaderboard.ts â€” exported functions:
  loadLeaderboard(path): Promise<Leaderboard>
  saveLeaderboard(leaderboard, path): Promise<void>
  recordRun(runRecord, path): Promise<LeaderboardEntry>

model-selector/reports.ts â€” exported functions:
  generateSummaryReport(leaderboard): string
  generateByTaskReport(leaderboard, taskId): string
  generateByModelReport(leaderboard, model): string
```

### Convergence Algorithm (MVP Heuristic)

For a given `task_id`, examine all entries for that task:
1. If `total_task_runs < 20` or `best_model_runs < 10` â†’ `exploring`
2. If `quality_gap >= 5.0` (best avg_quality - second_best avg_quality) â†’ `converged` (confidence: 0.95)
3. If `quality_gap >= 3.0` and `best_model_runs >= 8` â†’ `converging` (confidence: 0.80)
4. Otherwise â†’ `exploring` (confidence: `min(0.79, quality_gap / 10)`)
5. Single model for task (no competition): `converged` after `runs_count >= 20`

Document in code: "TODO: Upgrade to Wilson score interval when â‰Ą5 models have run."

### Quality Trend (Rolling Window)

`recent_run_scores: number[]` (max 5 entries) is stored in each `LeaderboardEntry`. On each `recordRun`, the new `qualityScore` is appended and the array is trimmed to last 5. Trend computation:

```
drop_pct = (avg_quality - mean(recent_run_scores)) / avg_quality
if drop_pct > 0.10 â†’ 'degrading'
if drop_pct < -0.10 â†’ 'improving'
else â†’ 'stable'
```

### Circular Import Prevention

`leaderboard.ts` MUST import from `models/__types__/quality-evaluation.ts` (types only) and NOT from `models/quality-evaluator.ts` (implementation). This avoids a circular dependency chain.

### No Module-Level Singletons

Load leaderboard from YAML on demand. Do not maintain a module-level leaderboard object. The caller owns the lifecycle.
