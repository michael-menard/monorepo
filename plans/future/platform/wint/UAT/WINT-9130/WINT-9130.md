# WINT-9130: Document Migration Path — Claude Code to LangGraph Workflow

## Context

Phase 9 (LangGraph Parity) is delivering a complete parallel implementation of the WINT workflow
system using LangGraph TypeScript graphs. By the time WINT-9130 is worked, the following stories
will be complete:

- **WINT-9110** (full workflow graphs): All major WINT workflows implemented as LangGraph graphs
  under `packages/backend/orchestrator/src/graphs/`
- **WINT-9120** (parity test suite): Evidence-backed test results showing which Claude Code
  capabilities are matched by LangGraph and which are pending

Developers who want to adopt the LangGraph workflow — or who need to decide whether to use Claude
Code agents or LangGraph graphs for a given use case — currently have no single authoritative
reference. They must read scattered documents, inspect source code, and infer the comparison
manually.

**Baseline note**: The seed baseline predates Phase 9 work (2026-02-13). WINT-9020 and WINT-9090
have since completed, providing concrete evidence for the feature comparison (42 parity tests,
86% coverage for doc-sync node; Zod schema conversion requirement confirmed in WINT-9090).

---

## Goal

Create `docs/workflow/langgraph-migration.md` as the definitive guide for the Claude Code to
LangGraph transition. The document must be evidence-backed (grounded in WINT-9120 parity test
results), opinionated (tell the reader what to do, not just list options), and cross-referenced
(link to existing workflow docs rather than duplicating content).

---

## Non-Goals

- This story does **not** implement any LangGraph code — documentation only
- This story does **not** backfill incomplete parity items from WINT-9120
- This story does **not** define the LangGraph error handling strategy (that is WINT-9105's ADR)
- This story does **not** update any agent `.agent.md` files or skills
- This story does **not** gate on all Phase 9 stories being complete — only WINT-9120 is a hard
  dependency
- This story does **not** modify any existing LangGraph implementation in
  `packages/backend/orchestrator/` (protected feature)

---

## Architecture Notes

- **Document location**: `docs/workflow/` alongside all other workflow documentation
- **Frontmatter schema**: Must match `docs/workflow/README.md` exactly (version field is
  semver, status must be `active`)
- **Content grounding rule**: Feature comparison table rows must map to a named WINT-9120
  test case. Unknown or speculative capabilities must be marked "Pending WINT-9120"
- **Mermaid usage**: Optional but encouraged for workflow comparison visuals. Diagrams must
  use standard Mermaid syntax compatible with GitHub's renderer
- **Zod note**: Though this is a docs-only story, any YAML schema snippets or code examples
  in the document must use Zod-style notation (`z.object({...})`) for consistency — never
  TypeScript `interface` syntax
- **Versioning**: Use sub-step insertion (`1a`, `1b`) if referencing step numbers in existing
  docs, rather than renumbering (per WINT-1050 lesson to avoid breaking external references)
