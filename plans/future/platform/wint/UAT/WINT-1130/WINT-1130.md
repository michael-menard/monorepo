# WINT-1130: Track Worktree-to-Story Mapping in Database

## Context

**Foundation Phase - Parallel Work Coordination**

WINT-0020 created the core story management schema (stories, storyStates, storyTransitions, storyDependencies). WINT-0090 established MCP tools for story status queries (storyGetStatus, storyUpdateStatus, etc.). Currently, 8 worktree skills exist (wt-new, wt-finish, wt-status, etc.) that handle git operations, but they do NOT track worktrees in the database.

**Problem**: Without database tracking, agents cannot:
1. Detect if a story already has an active worktree in another session
2. Prevent two sessions from working on the same story simultaneously
3. Identify orphaned worktrees (sessions that died without cleanup)
4. Track worktree lifecycle (created, active, merged, abandoned) for workflow analytics
5. Coordinate parallel batch work across multiple worktrees

**Solution**: Add a `worktrees` table to the wint schema and create 4 MCP tools following the WINT-0090/WINT-0110 patterns to enable database-driven coordination of parallel work.

**Reality Baseline** (2026-02-13):
- Stories table exists in unified-wint.ts with id (UUID), storyId, title, state fields
- WINT-0090 (Story Management MCP Tools) completed and in UAT
- WINT-0110 (Session Management MCP Tools) completed
- 8 worktree skills exist but no database tracking
- No worktree-related tables in current schema

## Goal

Enable database-driven coordination of parallel worktree-based development by:
1. Creating `worktrees` table in wint schema with FK to stories
2. Implementing 4 MCP tools for worktree lifecycle management (register, query, list, mark complete)
3. Tracking worktree status (active/merged/abandoned) with timestamps
4. Supporting detection of orphaned worktrees from failed sessions
5. Enabling conflict prevention when multiple sessions work on same story

## Architecture Notes

### Schema Design

**Table: worktrees**

```typescript
export const worktrees = pgTable('worktrees', {
  id: uuid('id').primaryKey().defaultRandom(),
  storyId: uuid('story_id')
    .references(() => stories.id, { onDelete: 'cascade' })
    .notNull(),
  worktreePath: text('worktree_path').notNull(),
  branchName: text('branch_name').notNull(),
  status: worktreeStatusEnum('status').notNull().default('active'),
  createdAt: timestamp('created_at').defaultNow().notNull(),
  updatedAt: timestamp('updated_at').defaultNow().notNull(),
  mergedAt: timestamp('merged_at'),
  abandonedAt: timestamp('abandoned_at'),
  metadata: jsonb('metadata').default({})
}, (table) => ({
  // Enforce one active worktree per story
  uniqueActiveWorktree: uniqueIndex('unique_active_worktree')
    .on(table.storyId, table.status)
    .where(sql`${table.status} = 'active'`)
}))
```

**Key Design Decisions**:

1. **FK with CASCADE**: If story deleted, worktrees auto-delete (orphaned worktrees have no value)
2. **Partial Unique Index**: Prevents concurrent registration of multiple active worktrees for same story
3. **JSONB Metadata**: Flexible schema evolution for future fields (e.g., pr_number, session_id)
4. **Timestamp Fields**: Separate mergedAt/abandonedAt for lifecycle tracking
5. **Default Status**: New records default to 'active'

### MCP Tool Contracts

```

```

**worktree_mark_complete**:
```typescript

Output: { success: true } | null
Error: Zod validation | worktree not found
```

### Concurrency Handling

**Race Condition**: Two sessions registering worktree for same story simultaneously.

**Mitigation**: Partial unique index on `(storyId, status)` where `status='active'` enforces constraint at database level.

**Behavior**:
- First registration succeeds
- Second registration fails with unique constraint violation
- MCP tool catches error, returns null, logs warning

### Error Handling Strategy

**Principle**: Resilient error handling, never throw uncaught exceptions.

**Patterns** (from WINT-0090/WINT-0110):
1. Zod validation at entry → fail fast with clear error
2. FK constraint violations → return null, log warning
3. Not found scenarios → return null (not an error)
4. Unexpected DB errors → return null, log error with context

## Infrastructure Notes

### Migration Strategy

**Forward Migration** (`0024_worktree_tracking.sql`):
```sql
-- Create enum
CREATE TYPE wint.worktree_status AS ENUM ('active', 'merged', 'abandoned');

-- Create table
CREATE TABLE wint.worktrees (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  story_id UUID NOT NULL REFERENCES wint.stories(id) ON DELETE CASCADE,
  worktree_path TEXT NOT NULL,
  branch_name TEXT NOT NULL,
  status wint.worktree_status NOT NULL DEFAULT 'active',
  created_at TIMESTAMP NOT NULL DEFAULT NOW(),
  updated_at TIMESTAMP NOT NULL DEFAULT NOW(),
  merged_at TIMESTAMP,
  abandoned_at TIMESTAMP,
  metadata JSONB DEFAULT '{}'
);

-- Create partial unique index
CREATE UNIQUE INDEX unique_active_worktree
  ON wint.worktrees (story_id, status)
  WHERE status = 'active';

-- Create index for common queries
CREATE INDEX idx_worktrees_story_id ON wint.worktrees(story_id);
CREATE INDEX idx_worktrees_status ON wint.worktrees(status);
```

**Rollback Script** (`0024_worktree_tracking_rollback.sql`):
```sql
DROP INDEX IF EXISTS wint.unique_active_worktree;
DROP INDEX IF EXISTS wint.idx_worktrees_story_id;
DROP INDEX IF EXISTS wint.idx_worktrees_status;
DROP TABLE IF EXISTS wint.worktrees;
DROP TYPE IF EXISTS wint.worktree_status;
```
