# WINT-9140: Validate LangGraph Parity Phase — Execute, Compare, and Sign Off

## Context

Phase 9 of the WINT epic ports all workflow agents from the Claude Code agent system to LangGraph nodes and graphs. Earlier Phase 9 stories completed individual node porting (WINT-9020 doc-sync, WINT-9090 context cache), established the full workflow graphs (WINT-9110), built the parity test suite (WINT-9120), and documented the migration path (WINT-9130).

WINT-9140 is the final gate of Phase 9. Both execution paths — the existing Claude Code agent system and the new LangGraph graph system — must produce identical outputs for identical inputs. This story executes the complete parity validation, surfaces and resolves any remaining deltas, and produces a formal Phase 9 sign-off artifact that closes the LangGraph migration chapter.

This story is read-only from the agent code perspective: no new nodes or graphs are created. The outputs are: (1) a completed parity verification run, (2) a sign-off document, and (3) a KB entry recording Phase 9 completion status.

**Dependency gate:** This story cannot be activated until WINT-9120 (Workflow Parity Test Suite) and WINT-9130 (Migration Documentation) are both in UAT or completed state. The seed and story are generated now for planning purposes.

## Goal

Execute complete representative workflows through both execution paths using the WINT-9120 parity test suite. Record output comparisons, classify any remaining deltas, apply fixes or create follow-up stories for acceptable deviations, and produce a formal Phase 9 sign-off artifact confirming parity has been achieved.

## Non-Goals

- Creating new LangGraph nodes or graphs (all porting done in earlier Phase 9 stories)
- Modifying existing Claude Code agents or `.agent.md` files
- Replacing or deprecating the Claude Code execution path — parity means both paths work; production migration decision is a separate concern
- Full performance benchmarking (dedicated story or Phase 10 scope)
- Fixing parity gaps categorized as acceptable deviations — those go to follow-up stories
- Touching the production database schema (`packages/backend/database-schema/`) — no migrations
- Modifying the `@repo/db` client API

## Architecture Notes

- This story is deliberately read-only from the node/graph code perspective. The parity validation approach is:
  1. Execute `graph.compile()` for each Phase 9 workflow graph (validates compilation)
  2. Run compiled graph routing tests with mock state (validates routing without LLM invocations)
  3. For representative live workflow tests: use real services per ADR-005 (no mocks for live runs)
- The deviation register distinguishes blocking gaps (must fix before sign-off) from acceptable deviations (document + follow-up story). This ensures sign-off is not held hostage by minor, non-blocking differences.
- The sign-off document is Markdown (human narrative + tables), not a Zod-validated YAML — intentional for human readability. Machine-readable portions (deviation register, workflow matrix) should use consistent table structure for downstream parsing if needed.
- ADR-005 (Testing Strategy): UAT must use real services. Live workflow parity runs must use live infrastructure, not mocks.
- ADR-006 (E2E Tests Required in Dev Phase): Backend-only story — E2E tests are exempt.
