# WINT-9020: Create doc-sync LangGraph Node (Native 7-Phase Implementation)

## Context

WINT-0160 (UAT, 2026-02-17) hardened the `doc-sync` agent for production readiness and added a LangGraph Porting Notes section to `.claude/skills/doc-sync/SKILL.md` defining the 7-phase execution contract:

| Phase | Name |
|-------|------|
| Phase 1 | File Discovery (git diff primary, timestamp fallback) |
| Phase 2 | Frontmatter Parsing (file + optional DB merge) |
| Phase 3 | Section Mapping (filename pattern → doc section) |
| Phase 4 | Documentation Updates (surgical table edits in docs/workflow/) |
| Phase 5 | Mermaid Diagram Regeneration (spawns field parsing) |
| Phase 6 | Changelog Entry Drafting (version bump determination) |
| Phase 7 | SYNC-REPORT.md Generation (structured output) |

WINT-9010 (UAT, 2026-02-18) created `@repo/workflow-logic` providing shared business logic functions (`WorkflowStoryStatus`, `getStatusFromDirectory`, `isValidStoryId`, etc.) that both MCP tools and LangGraph nodes can consume.

A node file `packages/backend/orchestrator/src/nodes/workflow/doc-sync.ts` already exists but implements **subprocess delegation** (spawning `claude doc-sync` via `child_process.spawn`) — this is NOT a native TypeScript port of the 7-phase agent logic. WINT-9020 must implement a true native TypeScript port.

**Path Discrepancy (AC-1 — must be resolved in elaboration):** The story index entry specifies `nodes/sync/doc-sync.ts` (a new `sync/` subdirectory). The existing file is at `nodes/workflow/doc-sync.ts`. The elaboration phase must explicitly decide which path to use.

## Architecture Notes

### Subprocess → Native Port Decision

The existing `nodes/workflow/doc-sync.ts` uses:
```typescript
const child = spawn('claude', ['doc-sync', ...args], { shell: true })
```

This is **subprocess delegation**, not a native port. WINT-9020 must implement each of the 7 phases as native TypeScript functions. This means:

- Phase 1: `child_process.exec('git diff HEAD --name-only')` — but only for git diff, not agent delegation
- Phase 2: YAML parsing via `js-yaml` or YAML regex extraction (already used in codebase)
- Phase 4: File editing via `node:fs/promises` readFile/writeFile with regex-based surgery
- Phase 5: String template generation for Mermaid graph syntax

### `createToolNode` Pattern (from delta-detect.ts exemplar)

```typescript
export const docSyncNode = createToolNode(
  'doc_sync',
  async (state: GraphState): Promise<Partial<GraphStateWithDocSync>> => {
    return docSyncImpl(state, {})
  },
)

export function createDocSyncNode(config: Partial<DocSyncConfig> = {}) {
  return createToolNode(
    'doc_sync',
    async (state: GraphState): Promise<Partial<GraphStateWithDocSync>> => {
      return docSyncImpl(state, config)
    },
  )
}
```

### State Extension Pattern

```typescript
export interface GraphStateWithDocSync extends GraphState {
  docSync?: DocSyncResult
}
```

Note: This is a TypeScript `interface` in the existing file — WINT-9020 may retain it as an interface (it extends, not defines) but all Zod schemas must remain Zod-first (`DocSyncConfigSchema`, `DocSyncResultSchema`).
