## Happy Path Tests

### Test HP-1: Successful DB Write — `in-progress` Transition

- **Setup**: Story `WINT-XXXX` exists in `wint.stories` table with `state = 'ready_to_work'`. Executing agent has access to `shimUpdateStoryStatus` MCP tool.
- **Action**: Call `/story-update plans/future/platform/wint WINT-XXXX in-progress`
- **Expected outcome**:
  1. Step 1: Story file located in `ready-to-work/WINT-XXXX/`
  2. Step 2: Worktree cleanup skipped (not `completed` transition)
  3. Step 3 (DB Write): `shimUpdateStoryStatus({ storyId: 'WINT-XXXX', newState: 'in_progress', triggeredBy: 'story-update' })` returns valid record
  4. Step 3.5 (Frontmatter): Story frontmatter `status` updated to `in-progress`
  5. Step 4 (Index): `wint/stories.index.md` `**Status:**` updated to `in-progress`
  6. Step 5 (Result): Returns `db_updated: true`, `file_updated: true`, `index_updated: true`
- **Evidence**: DB row for WINT-XXXX shows `state = 'in_progress'`; story frontmatter shows `status: in-progress`; result YAML includes `db_updated: true`

### Test HP-2: Successful DB Write — `ready-for-qa` Transition with `--no-index`

## Error Cases

### Test EC-1: DB Unavailable → Warning + FS Fallback

- **Setup**: Story exists in filesystem. DB is unavailable (simulated by disconnecting postgres-knowledgebase MCP or using an invalid story ID that causes shimUpdateStoryStatus to return null).
- **Action**: Call `/story-update plans/future/platform/wint WINT-XXXX in-progress`
- **Expected outcome**:
  1. `shimUpdateStoryStatus` returns `null`
  2. Agent emits WARNING: "DB write failed for WINT-XXXX. Falling back to filesystem-only update."
  3. Frontmatter is still updated (FS fallback proceeds)
  4. Index is still updated
  5. Result: `db_updated: false`, `file_updated: true`, `index_updated: true`
- **Evidence**: Result YAML shows `db_updated: false`; WARNING log visible; frontmatter updated

### Test EC-2: Invalid Transition → Guard Returns FAILED, No DB Call

- **Setup**: Story currently at `in-progress`. Attempting transition to `backlog` (not in valid transition table).
- **Action**: Call `/story-update plans/future/platform/wint WINT-XXXX backlog`
- **Expected outcome**:
  1. Transition validation check fires BEFORE any shim call
  2. `shimUpdateStoryStatus` is NOT called
  3. Returns `UPDATE FAILED: Cannot transition from in-progress to backlog`
  4. Frontmatter unchanged, index unchanged, no DB write
- **Evidence**: DB row state unchanged; no shim call recorded; FAILED signal returned

## Risks to Call Out

1. **WINT-1070 race**: If WINT-1070 (deprecate stories.index.md) lands during testing, Step 4 behavior changes. Tests EC-1 through EG-4 assume index write is still active. Tests should note this assumption.

2. **DB state vs command status mismatch**: The DB enum uses `in_qa` for `uat` and `done` for `completed`. Any test verifying DB state must use the DB enum value, not the command string.

3. **shimUpdateStoryStatus null ambiguity**: null can mean "DB unavailable" OR "story not found in DB". The command spec should surface a WARNING in both cases. Tests should verify the warning is emitted regardless of null reason.

4. **Test prerequisite**: WINT-1030 must be complete (wint.stories table populated) before integration tests can run. All integration tests assume DB is seeded with at least one test story.

5. **Missing BLOCKERS**: No blocking issues found. WINT-0120 (Telemetry MCP Tools) is NOT a prerequisite for WINT-1050 (as noted in seed — this was a copy-paste artifact from WINT-1060).
