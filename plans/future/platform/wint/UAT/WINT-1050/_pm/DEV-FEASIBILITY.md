## MVP-Critical Risks (Max 5)

### Risk 1: Status String → DB Enum Mismatch

- **Risk**: The command uses hyphenated strings (`in-progress`, `ready-for-qa`) while the DB uses underscore enums (`in_progress`, `ready_for_qa`). An incorrect mapping table would silently write wrong DB states.
- **Why it blocks MVP**: Every DB write would use a wrong `newState`, making the DB diverge from expected state after migration.
- **Required mitigation**: The mapping table in the command spec must explicitly document all 13 command status values with their DB `newState` values. Canonical reference: `SWIM_LANE_TO_STATE` in `packages/backend/mcp-tools/src/story-compatibility/__types__/index.ts`.

### Risk 2: Execution Order Correctness (Worktree → DB → FS → Index)

- **Risk**: If DB write occurs BEFORE worktree cleanup, and worktree cleanup fails, the DB state would be `done` but the worktree would be in an inconsistent state.
- **Why it blocks MVP**: The `uat → completed` transition (WINT-1150) requires worktree cleanup to complete before the status is finalized in any system.
- **Required mitigation**: The command spec must explicitly order: validate transition → worktree cleanup (if uat→completed) → DB write → frontmatter write → index write. This order is non-negotiable (AC-5).

### Risk 3: shimUpdateStoryStatus Null Handling

- **Risk**: The shim returns `null` on DB failure. If the command spec does not handle null properly, it might fail silently or block the FS update.
- **Why it blocks MVP**: Backward compatibility requires that DB write failure does NOT block the FS fallback path. A `null` return must result in WARNING + continue, not hard failure.
- **Required mitigation**: Command spec Step 3 must explicitly state: "if shimUpdateStoryStatus returns null → emit WARNING → set db_updated: false → proceed to Step 3.5 (FS update)". This matches AC-3.

### Risk 4: Non-Mappable Status Handling Decision

- **Risk**: 8 command statuses have no direct DB equivalent (`created`, `elaboration`, `needs-code-review`, `failed-code-review`, `failed-qa`, `needs-split`, `BLOCKED`, `superseded`). Without an explicit decision, the command spec is ambiguous.
- **Why it blocks MVP**: An agent implementing the command without clear guidance might either (a) crash on unmapped statuses, or (b) silently skip the DB write without logging, leaving the DB stale.
- **Required mitigation**: AC-10 decision documented in the command spec. Recommendation from seed: skip DB write for `needs-split`, `BLOCKED`, `elaboration`, `created`, `needs-code-review`, `failed-code-review`, `failed-qa` (emit note); map `superseded` → `cancelled`. This must be an explicit mapping table entry, not implicit.

## Missing Requirements for MVP

None blocking. The seed recommendations are complete. The command spec writer needs to make one explicit decision per AC-10:

- Decision: map `superseded` → `cancelled` in DB (OR skip). Document the choice.
- Decision: map `BLOCKED` → `blocked` in DB (OR skip). Document the choice.

These decisions must appear explicitly in the mapping table (AC-2). They do NOT require PM sign-off since the seed already provides the recommendation.
