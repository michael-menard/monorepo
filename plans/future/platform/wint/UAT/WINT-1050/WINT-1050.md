# WINT-1050: Update `/story-update` Command to Write Status to Database

**DB-First with FS Frontmatter Backward Compatibility**

---

## Context

The WINT platform has completed the compatibility shim module (WINT-1011/WINT-1012, UAT-verified 2026-02-17) and populated the `core.stories` database table (WINT-1030, UAT-verified 2026-02-16). The `shimUpdateStoryStatus` function provides a stable, DB-only write path with explicit null return on DB failure (no filesystem fallback by design — WINT-1011 AC-2 constraint).

The `/story-update` command (`.claude/commands/story-update.md` v2.1.0) currently updates story status exclusively through YAML frontmatter edits and `stories.index.md` updates. When the command completes, the database remains unaware of the status change. This creates divergence between the filesystem state (the current source of truth) and the database (the target source of truth for Phase 2+ of the WINT migration).

The sibling stories WINT-1040 (story-status → DB) and WINT-1060 (story-move → DB) establish the pattern for augmenting existing commands with DB integration. WINT-1050 follows the same approach for `/story-update`.

**Key behavioral constraint**: If `shimUpdateStoryStatus` returns null (DB unavailable), a WARNING is emitted and the FS frontmatter update proceeds normally. The story update is never blocked by DB unavailability. This is the "DB as primary write, FS as safety net" pattern for Phase 1.

**Scope**: Documentation-only change to a single markdown file. No TypeScript source files are created or modified.

---

## Goal

Modify `.claude/commands/story-update.md` to insert a new Step 3a that calls `shimUpdateStoryStatus` before the existing YAML frontmatter write (now Step 3b). The DB becomes the **primary** status write target; the YAML frontmatter write is retained for Phase 1 backward compatibility. Version bumped to v3.0.0 — this is a major version bump because DB integration is a breaking behavioral change for agents consuming command output.

## Architecture Notes

### DB Write Position

The DB write (Step 3a) is positioned between Step 2 (worktree cleanup) and Step 3b (frontmatter update):

```
Step 1: Locate story file
Step 2: Worktree cleanup (uat → completed transitions only) [WINT-1150, unchanged]
Step 3a: DB write (NEW — shimUpdateStoryStatus)
Step 3b: Update YAML frontmatter (renamed from Step 3, logic unchanged)
Step 4: Update stories.index.md (unchanged, --no-index still suppresses this only)
Step 5: Return result YAML (extended with db_updated field)
```

This ordering satisfies AC-5 (worktree cleanup before DB write) and AC-6 (transition validation — which occurs implicitly at status parse time before Step 1 executes — before DB write).

### Mapping Table Design

The mapping table is inline in the command spec (not referencing TypeScript source at runtime). This is the same pattern used by WINT-1060's story-move.md Step 2.5. The inline table keeps the command self-documenting without requiring agents to reference `SWIM_LANE_TO_STATE` at runtime.

**6 mapped statuses** (from `SWIM_LANE_TO_STATE` + 2 explicit decisions):
- `backlog` → `backlog`
- `ready-to-work` → `ready_to_work`
- `in-progress` → `in_progress`
- `ready-for-qa` → `ready_for_qa`
- `uat` → `in_qa`
- `completed` → `done`
- `BLOCKED` → `blocked` (explicit decision: not in SWIM_LANE_TO_STATE)
- `superseded` → `cancelled` (explicit decision: not in SWIM_LANE_TO_STATE)

**8 unmapped statuses** (skip with reason):
- `created`, `elaboration`, `needs-code-review`, `failed-code-review`, `failed-qa`, `needs-split`

Note: `SWIM_LANE_TO_STATE` maps `UAT` (directory name) to `in_qa`. The command's status value is `uat` (lowercase). The mapping table in the command must handle this case-sensitive distinction.

### Null Return Handling

When `shimUpdateStoryStatus` returns null:
1. Emit WARNING with story ID and "Proceeding with filesystem update only"
2. Set `db_updated = false`
3. Proceed to Step 3b (existing frontmatter update)
4. Do NOT add any compensating FS write — this violates WINT-1011 AC-2 contract

### --no-index Flag Scope

The `--no-index` flag scope is Step 4 only. It does NOT affect:
- Step 3a (DB write) — always executes for mapped statuses when DB is available
- Step 3b (frontmatter update) — always executes
