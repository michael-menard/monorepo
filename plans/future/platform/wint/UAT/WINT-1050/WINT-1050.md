# WINT-1050: Update `/story-update` Command to Write Status to Database via Compatibility Shim

## Context

The WINT platform is migrating story status management from filesystem-based (directory locations + YAML frontmatter) to database-driven (the `wint.stories` table). This migration is progressing in phases:

- **WINT-0090** (uat): Created `storyUpdateStatus` MCP tool for DB writes
- **WINT-1011** (uat): Created `shimUpdateStoryStatus` compatibility shim — DB-first write, null return on failure, NO filesystem fallback on write failure (AC-2 constraint)
- **WINT-1012** (uat): Added `ShimDiagnostics` + 80%+ unit test coverage for all four shim functions
- **WINT-1030** (uat): Populated `wint.stories` table from existing directories

The DB is now populated and the shim is production-ready. **The problem**: the `/story-update` command (`.claude/commands/story-update.md` v2.1.0) currently only updates YAML frontmatter and the stories index when changing story status. The `wint.stories` table is NOT updated, meaning the DB drifts from filesystem state every time an agent calls `/story-update`.

**Conflict to track (non-blocking)**: WINT-1070 (Deprecate stories.index.md) is `created` status and may land while this story is in-progress. If WINT-1070 completes before WINT-1050's index update step runs in production, the index update behavior must be adapted. This story's ACs do NOT depend on WINT-1070 state — the index update (Step 4) is retained but flagged for future removal by WINT-1070.

## Goal

Modify `.claude/commands/story-update.md` to call `shimUpdateStoryStatus` as the primary status write before updating YAML frontmatter. The DB becomes the source of truth for story status. Frontmatter sync is retained for Phase 1 backward compatibility. Version bumped to v3.0.0.

## Non-Goals

- This story does NOT add `--db-only` or `--skip-db` flags
- This story does NOT remove YAML frontmatter updates (retained for Phase 1 backward compat — removal deferred to Phase 7)
- This story does NOT update `story-move.md` (WINT-1060's scope)
- This story does NOT update `story-status.md` (WINT-1040's scope)
- This story does NOT replace the index update step (WINT-1070 owns index deprecation)
- This story does NOT implement telemetry logging (WINT-3070's scope)
- This story does NOT add concurrent update locking at the command layer (`storyUpdateStatus` already uses DB transactions)

## Architecture Notes

### DB-First Write Pattern

The command spec must instruct the executing agent to follow this pattern:

```
Step 3 (DB Write):
  mapped_state = STATUS_TO_DB_STATE_MAP[NEW_STATUS]
  if mapped_state is defined:
    result = shimUpdateStoryStatus({
      storyId: STORY_ID,
      newState: mapped_state,
      triggeredBy: 'story-update',
      reason: <optional if provided>
    })
    if result is null:
      emit WARNING: "DB write failed for {STORY_ID}. Falling back to filesystem-only update."
      db_updated = false
    else:
      db_updated = true
  else:
    emit NOTE: "No DB state for {NEW_STATUS}. Skipping DB write."
    db_updated = false

Step 3.5 (Frontmatter — always runs):
  Update {STORY_ID}.md frontmatter: status: <NEW_STATUS>
```

### Execution Order (Non-Negotiable)

1. Step 1: Locate story file
2. Step 2: Worktree cleanup (if `uat → completed` transition only)
3. Step 3: DB write via `shimUpdateStoryStatus`
4. Step 3.5: Frontmatter update (always)
5. Step 4: Index update (unless `--no-index`)
6. Step 5: Return result

This order ensures:
- Worktree is cleaned before state is changed (WINT-1150 requirement)
- DB is updated before FS (DB is source of truth)
- FS fallback still works if DB is unavailable (backward compat)

### Status → DB State Mapping Table

| Command Status | DB `newState` | Behavior |
|----------------|---------------|----------|
| `backlog` | `backlog` | DB write |
| `created` | — | Skip DB write, emit note |
| `elaboration` | — | Skip DB write, emit note |
| `ready-to-work` | `ready_to_work` | DB write |
| `in-progress` | `in_progress` | DB write |
| `needs-code-review` | — | Skip DB write, emit note |

### Version Strategy

- v2.1.0 → v3.0.0 (major bump: DB integration is a breaking behavioral change for agents reading this spec)
- Rationale: Agents following v2.1.0 would not call the shim; v3.0.0 makes this mandatory

### Backward Compatibility During Phase 1

YAML frontmatter writes are retained for Phase 1. This means:
- DB state is the primary record after a successful write
- Frontmatter state is the secondary/fallback record
- Divergence is possible if DB write succeeds but frontmatter write fails (or vice versa)
- Resolution: accept this short-lived divergence; Phase 7 (WINT-7030) will remove frontmatter writes entirely

**Scenario D: `uat → completed` Order Verification**
- Setup: Story at `uat`; active worktree registered
- Expected order: worktree cleanup completes FIRST, THEN `shimUpdateStoryStatus(newState: 'done')`, THEN frontmatter update

**Scenario E: Non-Mappable Status Skip**
- Setup: Valid transition to `needs-split`
- Expected: DB write skipped, note emitted: "No DB state for needs-split"; `db_updated: false`; frontmatter updated; `UPDATE COMPLETE`

**Scenario F: `BLOCKED` → `blocked` DB Mapping**
- Setup: Story transitioning to `BLOCKED`
- Expected: `shimUpdateStoryStatus({ newState: 'blocked' })` called; `db_updated: true`

Full test plan with setup/action/expected/evidence format is in `_pm/TEST-PLAN.md`.
