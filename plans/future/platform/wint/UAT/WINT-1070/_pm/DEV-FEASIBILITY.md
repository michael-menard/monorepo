# Dev Feasibility Review: WINT-1070 — Generate stories.index.md from Database

## Feasibility Summary

- **Feasible for MVP**: Yes
- **Confidence**: High
- **Why**: This story is architecturally identical to WINT-1030 (`populate-story-status.ts`) in reverse — instead of writing DB from files, it writes a file from DB. The `StoryRepository`, `StoryFileAdapter`, `createDbPool()`, and `findMonorepoRoot()` patterns are already implemented and deployed. The primary technical risk (data completeness gap between DB columns and index fields) is non-blocking because the seed prescribes a hybrid DB-primary / YAML-fallback approach. ## MVP-Critical Risks (Max 5)

### Risk 1: DB schema does not contain all fields needed for index generation

- **Why it blocks MVP**: If `wint.stories` table lacks `phase`, `depends_on`, or `goal` columns, the core per-story section rendering (AC-6) and Ready to Start computation (AC-5) will be incomplete or incorrect.
- **Required mitigation**: Before writing generation code, run `SELECT column_name FROM information_schema.columns WHERE table_schema = 'wint' AND table_name = 'stories'` and cross-reference against index fields. Document the field-source mapping as a constant (`FIELD_SOURCE_MAP`) in the script. For any field confirmed absent from DB, implement YAML fallback. The seed has already identified `depends_on`, `goal` as present in the DB schema; `risk_notes` and freeform `feature` description require YAML fallback.

### Risk 2: Status label mismatch between DB enum and index display format

- **Why it blocks MVP**: The DB uses `ready_to_work` (underscore) but `stories.index.md` displays `ready-to-work` (hyphen). If the label mapping is absent or incorrect, the Progress Summary table (AC-4) will show wrong labels and counts.
- **Required mitigation**: Define an explicit `STATE_TO_DISPLAY_LABEL` constant mapping all `wint.story_state` enum values to their display labels. Test this mapping in unit tests (HPT-4 in test plan).

### Risk 3: `stories.index.md` format regression breaks downstream agents

- **Why it blocks MVP**: Agents and humans that read `stories.index.md` (e.g., `pm-story-generation-leader`, `elab-story`) depend on its structure. If the generated output differs structurally from the manually-authored version, those agents will misparse the file.
- **Required mitigation**: Treat the existing `stories.index.md` as the authoritative format specification. Build the renderer by reverse-engineering the current file format. Include a format-parity check in the `--verify` mode to catch structural regressions.

### Risk 4: Generated `DO NOT EDIT` header conflicts with YAML frontmatter parsers

- **Why it blocks MVP**: Agents that read `stories.index.md` frontmatter via YAML parsers will break if the warning is placed inside the YAML block or creates ambiguous frontmatter delimiters.
- **Required mitigation**: Add the `DO NOT EDIT` warning in two places: (1) as a `generated_by` field in the YAML frontmatter (machine-readable), and (2) as a comment block in the markdown body immediately after the closing `---` frontmatter delimiter (human-readable). Do NOT insert it inside the YAML block itself.

### Risk 5: Concurrent writes from two parallel `--generate` invocations

- **Why it blocks MVP**: If two processes write `stories.index.md` simultaneously, the output could be corrupt.
- **Required mitigation**: Use atomic write pattern: write to a temp file first, then rename to the target path (`fs.rename()` is atomic on most filesystems). Low-priority for initial implementation but required before production use.

## Missing Requirements for MVP

### MR-1: `getAllStories()` method on `StoryRepository`

The seed recommends adding a single `SELECT * FROM wint.stories ORDER BY story_id ASC` method to `StoryRepository` rather than calling `getStoriesByState()` N times. This is more efficient and cleaner.

**Decision text for AC-1**: "Script uses a single `StoryRepository.getAllStories()` call that returns all stories ordered by `story_id`. This method must be added to `StoryRepository` as part of this story."

### MR-2: Field-source mapping constant

The script must document which fields come from DB vs YAML fallback. Without this constant, the hybrid approach is implicit and untestable.

**Decision text for AC-2**: "A `FIELD_SOURCE_MAP` constant (or equivalent) must enumerate each index field and its source (`db`, `yaml_fallback`, `computed`). This drives the field-source breakdown in the generation report."

### MR-3: Handling of stories with `phase: null` in DB

The index is organized by phase. Stories with no phase value must have a defined behavior (omit, or place in catch-all section).

**Decision text**: "Stories with `phase: null` in DB and no phase in YAML frontmatter are logged as skipped with reason 'missing phase'. They are excluded from the generated index but recorded in `generation-report.json`."
