# WINT-1040: Update /story-status Command to Use DB as Primary Source for Story Lookup

## Context

**Milestone reached**: The WINT platform has completed the foundational DB infrastructure required for this story:

- WINT-1011 (Compatibility Shim — Core Functions): **UAT/PASS** 2026-02-17 — `shimGetStoryStatus`, `shimGetStoriesByStatus`, and `shimGetStoriesByFeature` are live and exported from `@repo/mcp-tools`.
- WINT-1012 (Compatibility Shim — Observability): **UAT/PASS** 2026-02-17 — `ShimDiagnostics` opt-in field (`source: db | directory | not_found`) available.
- WINT-1030 (Populate Story Status from Directories): **UAT/PASS** 2026-02-16 — `core.stories` table is populated with current story state.
- WINT-0090 (Story Management MCP Tools): **UAT/PASS** — `story_get_status`, `story_get_by_feature` MCP tools available.

**Problem**: The `/story-status` command (`.claude/commands/story-status.md`, v4.0.0) operates purely from directory structure and `stories.index.md`. When a story's directory location no longer reflects its true state — because WINT-1030 populated the DB and WINT-1020 will eventually flatten the directories — the command gives stale or incorrect results for single-story lookup.

**Solution**: Update the command's "Feature + Story ID" mode to call `shimGetStoryStatus` (via MCP tool `story_get_status`) instead of scanning directories. The shim handles DB-unavailability and directory fallback transparently. Add a "Data Source" section and a DB state → display label mapping table to the command spec. Leave `--depth` and `--deps-order` modes unchanged — they depend on `stories.index.md` for dependency graph data not yet in the DB.

**Scope**: This story modifies exactly one file: `.claude/commands/story-status.md`. It is a documentation/spec update, not a TypeScript implementation.

---

## Goal

Make the `/story-status` command DB-first for single-story lookup, while preserving backward compatibility, maintaining the existing output format, and leaving `--depth` / `--deps-order` modes unchanged.

---

## Non-Goals

- Do NOT migrate `--depth` mode to use the DB — that is WINT-1070 (stories.index.md deprecation)
- Do NOT migrate `--deps-order` mode to use the DB — dependency graph data is not in the DB yet
- Do NOT add TypeScript source files — this story only modifies the command markdown file
- Do NOT implement DB writes — the command is and must remain read-only
- Do NOT remove backward compatibility — agents and humans calling `/story-status` must see no regression
- Do NOT implement the "No Arguments" (all-features summary) mode as DB-first — too broad in scope
- Do NOT modify the `SWIM_LANE_TO_STATE` mapping — it lives in the shim and must not be duplicated
- Do NOT update `stories.index.md` or any story frontmatter files

## Architecture Notes

### DB State → Display Label Mapping

The following mapping must be included in the updated command spec. It is the inverse of `SWIM_LANE_TO_STATE` from the shim, extended with DB-only states:

| DB State Enum | Display Label | Notes |
|---------------|--------------|-------|
| `backlog` | Backlog | In swim-lane `backlog/` or DB `backlog` state |
| `ready_to_work` | Ready to Work | Swim-lane `ready-to-work/` |
| `in_progress` | In Progress | Swim-lane `in-progress/` |
| `ready_for_qa` | Ready for QA | Swim-lane `ready-for-qa/` |
| `in_qa` | In QA / UAT | Swim-lane `UAT/` |
| `done` | Done / Completed | Swim-lane `done/` |
| `blocked` | Blocked | DB-only state — no swim-lane equivalent |
| `cancelled` | Cancelled | DB-only state — no swim-lane equivalent |

States NOT in `SWIM_LANE_TO_STATE` but present as directory-based states (command displays these from directory scan, not DB):
- `elaboration` → Elaboration
- `needs-code-review` → Needs Code Review
- `failed-code-review` → Failed Code Review
- `ready-for-qa` → Ready for QA (maps to `ready_for_qa` DB state)
- `failed-qa` → Failed QA
- `created` → Created

These additional states only reach the command via directory fallback (they are not in the shim's DB mapping). This is acceptable behavior during the migration window.

### MCP Tool Call Pattern

The updated "Feature + Story ID" mode step sequence should follow the `story-update.md` pattern:

```
1. Normalize STORY_ID to uppercase
2. Call: story_get_status({storyId: STORY_ID})
   - On result: derive status label from DB state, display single-story output
   - On null (DB miss): fall back to directory scan (existing logic)
   - On MCP tool unavailable: fall back to directory scan (shim handles this)
3. If directory scan also fails: display "Story not found: {STORY_ID}"
```

### Data Source Section

The updated command spec must include a "Data Source" section that explains:
- DB is now the primary source for the "Feature + Story ID" mode
- `shimGetStoryStatus` (via `story_get_status` MCP tool) is called first
- If the MCP tool is unavailable or returns no result, the command falls back to directory scan
- During the migration window (while WINT-1020 and WINT-1070 are pending), DB and directory may have different state information — DB is authoritative
