# WINT-0170: Add Doc-Sync Gate to Phase/Story Completion

## Context

The WINT epic's `PLAN.md` establishes an architectural decision: "Doc-sync enforcement: Gate check — Phase/story cannot complete until docs current." WINT-0160 (now UAT-verified, qa_verdict: PASS as of 2026-02-17) hardened the `doc-sync` agent to production readiness and explicitly documented the `--check-only` flag as the integration mechanism for this story:

- Exit code **0** → docs are in sync → completion is allowed
- Exit code **1** → docs are out of sync → completion is blocked

Currently, the two primary completion leader agents — `elab-completion-leader` and `qa-verify-completion-leader` — complete phases and stories without verifying that workflow documentation is current. This allows agent/command files to change during a story without their corresponding `docs/workflow/` entries being updated before the story advances.

**Dependency status:** WINT-0160 is in UAT with qa_verdict: PASS. The `--check-only` exit code contract (AC-7 of WINT-0160) is established and stable. WINT-0170 may proceed.

---

## Goal

Modify `elab-completion-leader.agent.md` and `qa-verify-completion-leader.agent.md` to invoke `/doc-sync --check-only` immediately before emitting their completion signals. The gate enforces documentation currency at story advancement without blocking on infrastructure failures.

---

## Non-Goals

- **Do NOT gate the FAIL path** — documentation sync is only enforced when a story advances (exits to UAT or completion). Failed stories are not advancing; adding a gate there would obstruct the feedback loop.
- **Do NOT implement timeout logic** in the completion agents — doc-sync handles its own timeouts internally (30s DB query timeout per WINT-0160). Adding a second timeout layer creates ambiguity.
- **Do NOT modify shared skills** (`/story-move`, `/story-update`, `/index-update`, `/token-log`) — the gate lives in the completion leader agents only, not in shared infrastructure.
- **Do NOT create new agent files** — this story modifies exactly two existing files.
- **Do NOT add the gate to every agent** — only `elab-completion-leader` and `qa-verify-completion-leader` are in scope for Phase 0. Other phase boundaries are deferred.
- **Do NOT implement a LangGraph version** — that belongs to WINT-9020 (doc-sync LangGraph node) which depends on WINT-9010.
- **Do NOT modify other completion agents** (e.g., `dev-implement-implementation-leader`, code-review agents) in this story — only the two named completion leaders.

## Architecture Notes

### Gate Decision Logic

```
invoke /doc-sync --check-only
  ├── exit code 0     → docs in sync  → proceed to completion signal
  ├── exit code 1     → docs out of sync → emit BLOCKED signal (halt)
  └── invocation fails → infrastructure error → emit WARNING (proceed)
```

The critical distinction: "infrastructure failure" (doc-sync could not run) is non-blocking. "exit code 1" (doc-sync ran and found out-of-sync state) is blocking. Agent prose must be unambiguous about these two cases.

### Gate Placement in elab-completion-leader

Current Step 6 = Verify Final State. New gate = Step 6.5 (or Step 7 if renumbered). Completion Signal section follows.

```
Step 6: Verify Final State    ← existing
Step 6.5: Doc-Sync Gate       ← NEW
## Completion Signal          ← existing (unchanged)
```

### Gate Placement in qa-verify-completion-leader (PASS path)

Current PASS path ends: Step 7 (Update Story Status in KB) → Step 8 (Log tokens) → Step 9 (Emit QA PASS). Gate inserts between Step 7 and Step 8:

```
Step 7: Update Story Status in KB   ← existing
Step 7.5: Doc-Sync Gate             ← NEW (PASS path only)
Step 8: Log tokens                  ← existing (renumbered if needed)
Step 9: Emit signal QA PASS         ← existing
```

The FAIL path is not modified.

### Why State Updates Happen Before the Gate

The gate is positioned after all state updates (status change, index update, KB capture) and before the final signal. This is intentional: if docs are out of sync and the gate blocks, the story's state is still correctly recorded in the database and index. The operator can run `/doc-sync`, then re-run the completion leader. On re-run, state updates are idempotent (or can be skipped if already complete). Only the final "done" signal is gated.

### Infrastructure Failure = Non-Blocking

If doc-sync fails to run (MCP unavailable, connection failure, timeout), the gate must not deadlock completion. The `DOC-SYNC COMPLETE (warnings)` signal from doc-sync (file-only fallback when DB unavailable) should be treated as exit code 0 — file-only mode is a graceful degradation, not a failure. The gate must only block on explicit exit code 1.
