# WINT-4110: Create /cohesion-check Command

## Context

The WINT Phase 4 cohesion subsystem has delivered the graph-checker agent (WINT-4060), which queries the feature cohesion graph, applies active cohesion rules, and produces a machine-readable `graph-check-results.json` report. The graph-checker agent declares `spawned_by: cohesion-check-command` in its frontmatter, but no such command file exists yet.

Developers and agents cannot currently trigger a cohesion check without manually constructing a Task tool invocation. There is no discoverable entry point for cohesion verification.

This story creates that entry point: a `.claude/commands/cohesion-check.md` command file following established command conventions.

**Baseline note**: Baseline predates WINT-4010, WINT-4030, WINT-4040, WINT-4050, WINT-4060 deliveries. The graph-checker agent at `.claude/agents/graph-checker.agent.md` is confirmed in-use (WINT-4060 delivered). WINT-4040 (Infer Existing Capabilities) is in `failed-code-review` — does not affect command authoring; only affects graph-checker runtime results (handled via graceful degradation).

---

## Goal

Create a discoverable `/cohesion-check` command file that allows developers and agents to manually trigger the graph-checker agent for a given feature area and receive a clear report of cohesion violations.

---

## Examples

**Example 1 — Clean run, no violations:**
Given a user runs `/cohesion-check plans/future/platform/wint WINT-4110`, the command spawns the graph-checker agent (haiku), which runs Phases 1-4, writes `plans/future/platform/wint/backlog/WINT-4110/_implementation/graph-check-results.json`, and emits `GRAPH-CHECKER COMPLETE`. The command surfaces this to the user and points them to the results file.

**Example 2 — Run with package filter:**
Given a user runs `/cohesion-check plans/future/platform/wint WINT-4110 --package @repo/ui`, the command forwards `packageName: "@repo/ui"` to graph-checker, which scopes its `graph_get_franken_features` call accordingly, and emits `GRAPH-CHECKER COMPLETE` (or `WITH WARNINGS` if graph is empty).

**Example 3 — Degraded run (WINT-4040 not yet stable):**
Given capability data is absent (WINT-4040 still in failed-code-review), graph-checker degrades gracefully, increments `warning_count`, and emits `GRAPH-CHECKER COMPLETE WITH WARNINGS: 1 warnings`. The command surfaces the warning count and tells the user to check `graph-check-results.json` for the degradation warning message.

---

## Edge Cases

**Edge Case 1 — Output directory does not exist:**
If the story directory path is invalid or `_implementation/` cannot be created, graph-checker emits `GRAPH-CHECKER BLOCKED: output directory unknown`. The command relays this signal to the user with the reason.

**Edge Case 2 — Both graph and rules registry empty:**
If graph.features has no rows AND no active cohesion rules exist, graph-checker emits `GRAPH-CHECKER COMPLETE WITH WARNINGS: 2 warnings` (one per degradation condition). The command surfaces the warning count and directs the user to `graph-check-results.json` for the full warnings list.

**Edge Case 3 — Package filter provided but no matching features:**
If `--package @repo/nonexistent` is provided and `graph_get_franken_features({ packageName })` returns an empty array, graph-checker does NOT increment warnings (a package filter was applied, so empty is a valid result). Signal: `GRAPH-CHECKER COMPLETE` with `franken_features_found: 0`.

---

## Non-Goals

- Do NOT implement any TypeScript code, MCP tools, or Lambda handlers
- Do NOT auto-chain cohesion-prosecutor (WINT-4070) or backlog-curator (WINT-4100) — manual invocation only; integration deferred to WINT-4120
- Do NOT create a new output schema — `graph-check-results.json` is defined by the graph-checker agent (WINT-4060)
- Do NOT add auto-run hooks to `dev-implement-story` or `qa-verify-story` — that is WINT-4120's scope
- Do NOT modify the graph-checker agent file — it is already complete (WINT-4060)
- Do NOT add the command to any registry or index unless such a registry already exists

## Architecture Notes

- Command files live at `.claude/commands/{name}.md` — markdown documents only, no build step
- Agent spawning is done via Task tool with `subagent_type: "general-purpose"` and the agent's `.agent.md` path
- The graph-checker agent uses model `haiku` (declared in its frontmatter)
- Graph tools are invoked inside graph-checker via direct TypeScript function imports (ARCH-001 direct-call pattern) — the command does not call them directly
- Output artifact `graph-check-results.json` is defined by graph-checker (WINT-4060) — command only reads and relays it
