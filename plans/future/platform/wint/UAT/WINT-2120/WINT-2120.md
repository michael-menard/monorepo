# WINT-2120: Measure Token Reduction via Cache Hit Rate Benchmark

## Context

WINT-2110 has wired the 5 highest-invocation agent workflows (7 agent files) to call `context_cache_get()` for pre-distilled context packs instead of re-reading source documents and re-querying KB on every invocation. WINT-2030/2040/2050/2060 have populated the `wint.context_packs` table with project conventions, architecture patterns, library snippets, and agent missions content.

We have no validated measurement of whether the cache integration actually achieves the 80% token reduction target. The WINT-2110 AC-6 baselines are rough file-size-derived estimates. The only live observability available is `context_cache_stats()` (hit count, avg hits per pack) and the pack content sizes in the database. Phase 3 telemetry (`agent_invocations` table) is not yet implemented.

**Dependency note**: WINT-2110 is in `needs-code-review` — its AC-6 baseline estimates exist in the story body and possibly as `BASELINE-TOKENS.yaml`. WINT-2040 (agent_missions pack) is in `failed-code-review` — that pack will show zero hits at implementation time. The benchmark is designed to handle both conditions gracefully.

## Goal

Create a benchmark integration test that measures estimated token reduction from the context cache by querying live pack hit counts and content sizes, then produces a `BENCHMARK-RESULTS.yaml` artifact with a PASS/NEEDS-TUNING verdict against the 80% reduction target.

## Examples

**Happy path — partial cache population:**

Given:
- `context_cache_stats({ packType: 'project_conventions' })` returns `{ hitCount: 120, avgHitsPerPack: 24 }`
- `wint.context_packs` for `project_conventions` contains content of ~8,000 characters (~2,000 tokens)
- WINT-2110 baseline for pm-bootstrap agent: 5,000 tokens/invocation

The benchmark computes: estimated tokens saved = 2,000 tokens/hit × 120 hits = 240,000 tokens avoided. Estimated baseline consumed = 5,000 × (120/24 invocations) = 5,000 × 5 = 25,000. Reduction ≈ (240,000 - 25,000) / 240,000 — with the methodology note that proxy measures are directional only.

**Output example (BENCHMARK-RESULTS.yaml):**

```yaml
run_date: "2026-03-07T14:32:00Z"
methodology:
  baseline_source: "WINT-2110 BASELINE-TOKENS.yaml"
  token_proxy: "content_char_count / 4"
  formula: "(pack_token_size * hit_count) / (baseline_input_tokens * invocation_count)"
  limitations: "Proxy measures — not actual LLM invocation logs. Phase 3 telemetry pending."
per_agent_results:
  - agent: pm-bootstrap-workflow
    baseline_tokens: 5200
    estimated_tokens_with_cache: 1040
    reduction_pct: 80
    pack_types_used: [project_conventions, architecture_patterns]
aggregate:
  estimated_reduction_pct: 78
  sample_packs: 3
  excluded_packs: [agent_missions]
  exclusion_note: "agent_missions not yet populated (WINT-2040 in failed-code-review)"
verdict: NEEDS-TUNING
tuning_recommendations:
  - "Populate agent_missions pack (WINT-2040) to complete coverage for pm-dev-feasibility-review agent"
  - "Increase context injection for dev-implement-story agent (currently 60% coverage)"
```

## Edge Cases

**Zero packs populated:** All pack types show zero hits. The benchmark reports all agents as `not_yet_populated`, omits the aggregate reduction percentage (marks as N/A), and emits NEEDS-TUNING verdict with a note that no data was available. The benchmark completes without error.

**Division by zero — empty pack content:** A pack exists in `wint.context_packs` with empty content string. The benchmark detects content_size = 0, excludes that pack from the calculation with a warning logged via `@repo/logger`, and continues. No runtime exception is thrown.

**BASELINE-TOKENS.yaml not found:** WINT-2110 did not produce a machine-readable baseline artifact. The benchmark falls back to hardcoded per-agent estimates from the WINT-2110 AC-6 table, logs a warning (`BASELINE-TOKENS.yaml not found, using hardcoded AC-6 estimates`), and records `baseline_source: hardcoded_ac6_fallback` in the methodology section.

**Only one pack type available:** The benchmark reports reduction for the single populated pack, includes a note in the aggregate section (`based on 1 of N pack types`), and flags the verdict as `NEEDS-TUNING` due to limited sample coverage.

## Non-Goals

- Do not implement Phase 3 telemetry or `agent_invocations` logging — that is WINT-3010/0120 scope
- Do not modify agent `.agent.md` files — WINT-2110 scope
- Do not modify `context_cache_stats` or other MCP tools — consume only
- Do not require actual LLM invocations to measure token usage — use proxy measures (content size + hit count)
- Do not hard-fail CI on the 80% target — advisory result only
- Do not implement cache tuning — this story produces a report; tuning is a follow-on story
- Do not run benchmarks for agents outside the 5 workflows covered by WINT-2110

## Architecture Notes

- Implement as a Vitest integration test file (not a standalone script) to leverage the existing test harness pattern from `performance-benchmarks.integration.test.ts`
- The benchmark runs against real PostgreSQL (ADR-005). Tag or configure as integration-only to avoid accidental execution in unit test runs without a DB.
- The `calculateTokenReduction` function should be a pure function extracted for unit testing independent of DB access — enables the HP-5 unit test to verify math correctness without live data.
- Token proxy calculation: `content_char_count / 4` (approximate chars-per-token ratio). This is a rough estimate; document as such.
- Reduction formula: `(pack_token_size × hit_count) / (baseline_input_tokens × invocation_count)`. Both numerator and denominator are estimates — the result is a directional indicator only.
- Per-agent results should identify model tier (haiku vs. sonnet) where known, since token budget guidelines differ by tier (WINT-2110 opp-3 lesson).
