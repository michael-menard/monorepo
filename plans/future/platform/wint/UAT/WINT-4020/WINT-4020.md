# WINT-4020: Create Rules Registry Sidecar

## Context

The WINT platform has established a sidecar architecture for lightweight services supporting agent workflows. Three sidecars are in place or in progress: role-pack (port 3090, complete), context-pack (port 3091, in failed-qa), and cohesion (port 3092, in needs-code-review via WINT-4010). All Phase 4 sidecars follow the same structural pattern: Node.js http module, Zod-validated schemas, MCP tool wrapper using the direct-call pattern (ARCH-001), housed in `packages/backend/sidecars/`.

Separately, the knowledge base accumulates retrospective learnings — lessons, blockers, patterns — via `kb_add_lesson`. These are currently queryable but not enforceable. There is no mechanism to convert a KB lesson into a gate check, lint rule, or prompt injection that automatically prevents agents from repeating the same mistake. Rules need a lifecycle: proposed (from retro), reviewed, promoted to active, and optionally applied as gate checks, lint rules, or prompt injections.

**Blocking dependency**: This story depends on WINT-2020 (Context Pack Sidecar), which is currently in `failed-qa`. Implementation cannot begin until WINT-2020 reaches UAT.

## Goal

Create a rules registry sidecar at `packages/backend/sidecars/rules-registry/` that provides a structured lifecycle for enforceable rules — from proposal through promotion to active — backed by a dedicated database table and exposed to agents via MCP tool wrappers.

## Examples

**Example 1 — Proposing a rule from a retro learning:**
Given an agent calls `rules_registry_propose({ rule_text: "Do not use TypeScript interfaces — use Zod schemas with z.infer<>", rule_type: "lint", scope: "global", severity: "error", source_story_id: "WINT-4020" })`, the system stores a new rule record with `status=proposed` in the rules table and returns `{ ok: true, data: { id: "<uuid>", status: "proposed", ... } }`.

**Example 2 — Querying active gate rules:**
Given `GET /rules?type=gate&status=active` is sent to port 3093, the system queries the rules table and returns `{ ok: true, data: [<array of active gate rules>] }`, filtered to only type=gate and status=active entries.

**Example 3 — Promoting a proposed rule to active:**
Given a proposed rule with id `abc-123` exists, when `POST /rules/abc-123/promote` is called with body `{ source_story_id: "WINT-4020" }`, the system updates the rule's status to `active` and returns `{ ok: true, data: { id: "abc-123", status: "active", ... } }`.

## Edge Cases

**Edge Case 1 — Duplicate rule detection (case-insensitive):**
When `POST /rules` is called with `rule_text = "  NO BARREL FILES  "` and a non-deprecated rule with `rule_text = "No barrel files"` already exists, the system returns HTTP 409 with `{ ok: false, error: "Conflict: rule already exists", conflicting_ids: ["<existing-rule-uuid>"] }`. Conflict detection normalizes rule_text by lowercasing and trimming whitespace before comparison.

**Edge Case 2 — Re-proposal of deprecated rule is allowed:**
When a rule with `rule_text = "No barrel files"` has `status=deprecated`, and `POST /rules` is called with the same text, the system allows the re-proposal and returns HTTP 201 (deprecated rules do not block new proposals).

**Edge Case 3 — Promote without source reference is rejected:**
When `POST /rules/:id/promote` is called with an empty body (no `source_story_id` or `source_lesson_id`), the system returns HTTP 422 with `{ ok: false, error: "source_story_id or source_lesson_id required to promote" }`.

**Edge Case 4 — Empty rules table returns empty array:**
When no rules exist in the database, `GET /rules` returns HTTP 200 with `{ ok: true, data: [] }` — never a 404.

## Non-Goals

- Do NOT implement rule enforcement (gate execution, lint rule application, prompt injection) — deferred to Phase 5+
- Do NOT modify `packages/backend/database-schema/` protected tables without a migration story
- Do NOT build a UI for rule management — deferred entirely
- Do NOT implement rule versioning beyond proposed/active/deprecated status states
- Do NOT implement automatic rule derivation from KB lessons — this story provides the CRUD layer only
- Do NOT implement semantic similarity conflict detection (pgvector) — deferred to future story; only case-insensitive text match

## Architecture Notes

- **Direct-call pattern (ARCH-001)**: MCP wrappers must import compute functions directly from `@repo/sidecar-rules-registry`. Never call `http://localhost:3093/rules` from within an MCP wrapper.
- **Port registry**: 3090 = role-pack, 3091 = context-pack, 3092 = cohesion, 3093 = rules-registry. Document in `server.ts` comment block.
- **Zod-first types**: All schemas via Zod — `z.infer<typeof Schema>` for derived types. No TypeScript interfaces.
- **No barrel files**: Import directly from source files, not from index.ts re-exports.
- **Conflict detection algorithm**: Normalize rule_text by `rule_text.toLowerCase().trim()` before duplicate check. Query for non-deprecated rules with matching normalized text. Defer pgvector semantic similarity to future story.
- **pnpm workspace**: `packages/backend/sidecars/*` glob auto-covers the new package — no workspace file edits needed.

## Infrastructure Notes

- **New database table**: `rules` table in postgres-knowledgebase via Drizzle migration
  - Schema: `id (uuid pk)`, `rule_text (text not null)`, `rule_type (enum: gate|lint|prompt_injection)`, `scope (text: global|<story_id>|<feature_id>)`, `severity (enum: error|warning|info)`, `status (enum: proposed|active|deprecated)`, `source_story_id (text nullable)`, `source_lesson_id (text nullable)`, `created_at (timestamptz)`, `updated_at (timestamptz)`
- Migration must not modify any protected existing tables
- Sidecar runs as a standalone Node.js process (not AWS Lambda)

## HTTP Contract Plan

```
GET /rules
  Query params: type (gate|lint|prompt_injection), scope (text), status (proposed|active|deprecated)
  Response 200: { ok: true, data: Rule[] }
  Response 400: { ok: false, error: string }

POST /rules
  Body: { rule_text: string, rule_type: "gate"|"lint"|"prompt_injection", scope: string, severity: "error"|"warning"|"info", source_story_id?: string, source_lesson_id?: string }
  Response 201: { ok: true, data: Rule }
  Response 400: { ok: false, error: string }
  Response 409: { ok: false, error: string, conflicting_ids: string[] }

POST /rules/:id/promote
  Body: { source_story_id?: string, source_lesson_id?: string }
  Response 200: { ok: true, data: Rule }
  Response 404: { ok: false, error: string }
  Response 409: { ok: false, error: string }
  Response 422: { ok: false, error: string }

All routes:
  Response 404 (unknown route): { ok: false, error: "Not found" }
```
