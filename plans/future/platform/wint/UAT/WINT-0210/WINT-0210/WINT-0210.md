# WINT-0210: Populate Role Pack Templates for Dev, PO, DA, and QA Roles

## Context

The workflow currently provides no focused, role-specific instruction sets for agents. Each agent spawning session must derive role expectations from full agent files (which are long) or from system prompts (which are generic). This leads to scope creep, vibes-based approvals, and inconsistent enforcement of hard caps.

WINT-0180 will establish the framework (skeleton format, example constraints, injection mechanism). WINT-0190 will create the Patch Queue + Repair Loop schema for dev. WINT-0200 will create the User Flows schema for PO. WINT-0210 uses those foundations to produce the actual role pack files.

Each role pack is a focused markdown file targeting 150-300 tokens injected at agent spawn time. Role packs use the pattern skeleton format from WINT-0180: decision rule, proof requirement, max 2 positive examples, 1 negative example. Four role packs are in scope:

- **dev.md**: Patch Queue pattern (from WINT-0190 schema), Reuse Shared Packages pattern, Repair Loop pattern
- **po.md**: Hard caps (max 5 findings per session, max 2 may be blocking), references user-flows schema (WINT-0200)
- **da.md**: Hard caps (max 5 challenges, cannot challenge blocking items), focused challenge format
- **qa.md**: AC→Evidence trace pattern, ac-trace.json output format specification

The goal is behavior change, not documentation: agents reading these role packs must produce measurably different outputs — fewer mega-patches, fewer vague approvals, fewer uncapped finding lists, consistent evidence trace formatting.

## Architecture Notes

Role pack files are prompt-injection artifacts, not code. They follow the pattern established by `.claude/config/*.yaml` files for role-specific configuration. The key architectural decisions:

1. **Hard caps as MUST NOT, not suggestions**: PO and DA hard caps (max findings, max challenges) must be phrased as MUST NOT to drive mechanical enforcement, not aspiration.

2. **Token budget is a hard constraint**: 150-300 tokens per file. Exceeding 300 tokens means the role pack consumes too much context on spawn. Under 150 tokens means it lacks the actionable specificity to change behavior.

3. **Negative examples are required**: WINT-0180 mandates 1 negative example per role. Without anti-patterns, agents self-define "minimum viable" — typically producing verbose, uncapped outputs.

4. **Schema path references must be resolvable**: dev.md and po.md reference actual schema files from WINT-0190 and WINT-0200. If those schemas are not yet finalized at implementation time, use placeholder paths and do a single-pass update.

5. **No injection mechanism in this story**: Role packs are static markdown files until WINT-2010 (Phase 2) builds the sidecar to inject them at agent spawn time.

- Making role packs too long (exceeding 300 tokens defeats the purpose)
- Making role packs too vague ("be thorough" without specific caps and examples)
- Writing role packs without negative examples
- Writing PO/DA caps as soft guidance ("aim for max 5") rather than hard stops ("MUST NOT exceed 5")
