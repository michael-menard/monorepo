**Status:** backlog

---

## Context

The workflow system currently relies on agents self-reporting stage completion without a centralized validation layer. Agents advance through lifecycle stages (POST_BOOTSTRAP → ELAB_COMPLETE → SCOPE_OK → PATCH_COMPLETE) based on convention — with no enforcement that required proof artifacts exist, are schema-valid, and are non-empty before a stage transition is recorded.

This produces "vibes-based approvals": agents mark stages complete without providing verifiable evidence. Quality gaps surface late in the pipeline (at QA/UAT) when they are expensive to fix. The pipeline phase gate concept exists (APIP-6001) but is scoped to pipeline scripts; it is not available as a reusable service for the workflow agent network.

This story delivers a standalone gatekeeper sidecar that agents can call via MCP to enforce proof-of-work requirements before advancing stages.

---

## Goal

Create a centralized HTTP sidecar service that validates stage-specific proof artifacts, ensuring no stage transition is recorded without verifiable, schema-valid evidence.

---

## Examples

**Example 1 — POST_BOOTSTRAP passes:**
Given a POST to `/gate/check` with `{ stage: "POST_BOOTSTRAP", story_id: "WINT-3010", proof: { checkpoint: { phase: "setup_complete" } } }`, the system returns `{ ok: true, data: { passed: true, stage: "POST_BOOTSTRAP", story_id: "WINT-3010", timestamp: "2026-03-08T..." } }` with HTTP 200.

**Example 2 — ELAB_COMPLETE passes with CONDITIONAL_PASS:**
Given a POST with `{ stage: "ELAB_COMPLETE", story_id: "WINT-3010", proof: { elab: { verdict: "CONDITIONAL_PASS", findings: ["Gap: missing edge case AC"] } } }`, the system returns `{ ok: true, data: { passed: true, ... } }` — both `PASS` and `CONDITIONAL_PASS` are accepted verdicts.

**Example 3 — SCOPE_OK passes with no_scope_files flag:**
Given `{ stage: "SCOPE_OK", proof: { scope: { no_scope_files: true } } }`, the system returns passed:true — the explicit no-scope flag is the accepted alternative to a non-empty `included_files` array.

**Example 4 — PATCH_COMPLETE fails, missing_proofs identified:**
Given `{ stage: "PATCH_COMPLETE", proof: { evidence: { touched_files: 0 } } }`, the system returns `{ ok: false, error: "Gate check failed", missing_proofs: ["evidence.touched_files"] }` with HTTP 422 — zero touched files is an invalid PATCH_COMPLETE proof.

---

## Edge Cases

**Edge Case 1 — Extra proof fields are permitted (passthrough):**
If the proof payload contains extra/unexpected fields beyond the required schema, the gatekeeper strips or ignores them and does not fail. The AC-5 validation is permissive about optional/extra fields and only fails when a *required* field is absent or structurally invalid. This prevents the gate from blocking legitimate completions with enriched proof payloads.

**Edge Case 2 — SCOPE_OK with both empty included_files and no_scope_files:false fails:**
If neither condition for SCOPE_OK is satisfied (included_files is empty array AND no_scope_files is false or absent), the gate correctly returns HTTP 422. This enforces that agents must explicitly declare scope rather than submit an effectively empty proof.

**Edge Case 3 — MCP wrapper handles compute errors gracefully:**
If the underlying gate validation compute function throws or returns null, the MCP `gate_check` wrapper logs a warning via `@repo/logger` and returns null — it does not throw or crash the calling agent's session. This follows the established pattern in `context-pack-get.ts`.

**Edge Case 4 — ELAB_COMPLETE with FAIL verdict is rejected:**
A verdict of `"FAIL"` is not in the accepted set (`PASS | CONDITIONAL_PASS`). Submitting FAIL as an elab verdict returns HTTP 422 with `missing_proofs: ["elab.verdict"]`.

## Architecture Notes

- **Proof Schema Design**: AC-5 uses a `z.discriminatedUnion('stage', [...])` approach where each stage variant defines its own required proof shape. Proof objects use `.passthrough()` or `.strip()` to allow extra fields without failure — permissive about optional fields, strict about required fields.
- **Direct-call MCP wrapper**: The `gate_check` MCP tool imports the compute function from `@repo/gatekeeper-sidecar` directly. This avoids runtime HTTP coupling (lesson OPP-04 from WINT-4010).
- **Port allocation**: 3094 (next in sequence: 3090=role-pack, 3091=context-pack, 3092=cohesion, 3093=rules-registry, 3094=gatekeeper). Port comment block in server.ts is the canonical registry (OPP-03).
- **Auth deferral**: SEC-002 comment in route handler documents intentional VPC-internal auth omission, consistent with context-pack and cohesion.
- **No DB writes**: Gatekeeper validates in-memory proof payloads only. No database tables, no `@repo/db` calls, no schema migrations needed for v1.
- **Graceful degradation**: Gate validation failures log warnings via `@repo/logger` and return structured errors — they never crash the sidecar process.
