# WINT-1070: Generate stories.index.md from Database

## Context

`plans/future/platform/wint/stories.index.md` is a 2,400-line manually-maintained index of all WINT stories. It contains the canonical status, dependency graph, feature descriptions, and risk notes for every story across 9 phases. WINT-1030 (UAT PASS) established the `wint.stories` database table as the authoritative source of story state by scanning story YAML frontmatter and populating the table.

The index file and the database can now drift: any manual edit to `stories.index.md` does not update the database, and any database status update (via WINT-0090 MCP tools) is not reflected in the index. As stories progress through the workflow, the index becomes stale relative to the database.

WINT-1030 explicitly scoped out index generation as a non-goal, noting: "Index file generation — WINT-1070 will generate stories.index.md from database." This story fulfills that deferred commitment.

### Dependency

WINT-1030 must be in UAT (PASS) — confirmed. The `wint.stories` table is populated and `StoryRepository` is deployed.

### Data Completeness Audit (Pre-Implementation Required)

The `wint.stories` table schema contains: `story_id`, `feature_id`, `type`, `state`, `title`, `goal`, `points`, `priority`, `blocked_by`, `depends_on`, `follow_up_from`, `packages`, `surfaces`, `non_goals`, `created_at`, `updated_at`.

Fields present in `stories.index.md` but **not** in `wint.stories`:
- `phase` — YAML frontmatter only (required for section grouping)
- `risk_notes` — YAML frontmatter only (freeform text)
- `feature` freeform description — YAML frontmatter only (long narrative)
- `infrastructure` list — YAML frontmatter only

Fields present in both DB and index:
- `title`, `state` (→ display label), `depends_on`, `goal`

**Hybrid strategy**: DB is primary for `state`, `title`, `goal`, `depends_on`. YAML frontmatter is fallback for `phase`, `risk_notes`, `feature` (freeform), `infrastructure`.

---

## Goal

Create a generation script (`generate-stories-index.ts`) that reads story state from the `wint.stories` database, supplements with YAML frontmatter for fields not stored in the DB, and writes a `DO NOT EDIT` generated version of `stories.index.md` that is structurally identical to the manually-authored version.

After this story, `stories.index.md` becomes a read-only generated artifact. The database is the single source of truth for story state.

---

## Non-Goals

- **Modifying the `wint.stories` schema** — no DDL changes; read-only DB access only
- **Generating indexes for other epics** (KBAR, LNGG, etc.) — scope is WINT only; multi-epic support deferred
- **Real-time regeneration on every DB write** — automation hook deferred; this story delivers the script and `--verify` flag; triggering is handled by humans or a future story
- **Updating story YAML frontmatter files** — script is read-only for both database and filesystem story files
- **Replacing stories.index.md's role as the primary format** — the format is preserved exactly; only authorship changes (machine-generated vs human-written)
- **Enriching the DB schema with new fields** (risk_notes, etc.) — out of scope; YAML fallback covers the gap
- **Protected**: Do not modify `wint.stories` table schema, `@repo/db` client API surface, or existing `StoryRepository` methods (addition of `getAllStories()` is in scope)

## Architecture Notes

### Script Structure

```
packages/backend/orchestrator/src/scripts/
  generate-stories-index.ts           # Main CLI script
  __types__/
    generation.ts                     # Zod schemas: GenerationReport, IndexFrontmatter, StorySection
  __tests__/
    generate-stories-index.test.ts    # Unit tests (pure functions)
    generate-stories-index.integration.test.ts  # Integration tests (real DB)
```

### Key Design Decisions

1. **DB-primary, YAML-fallback**: `wint.stories` is queried first. For fields not in the DB schema (`phase`, `risk_notes`, `feature` freeform, `infrastructure`), `StoryFileAdapter.read()` is called on the story file. YAML fallback failure is fail-soft (story rendered with `—` for missing fields).

2. **`FIELD_SOURCE_MAP` constant**: Explicitly documents which fields come from DB vs YAML at the top of the script. Drives the `field_source_breakdown` in the generation report.

3. **`STATE_TO_DISPLAY_LABEL` constant**: Maps DB enum values (`ready_to_work`) to display labels (`ready-to-work`) for the Progress Summary table and per-story `**Status:**` fields.

4. **Atomic write**: Write to a temp file, then `fs.rename()` to target path to prevent partial writes.

5. **Format parity as constraint**: The generated output must be structurally indistinguishable from the manually-authored file. Treat the existing `stories.index.md` as the format specification.

6. **No double-reads**: If a story's YAML is read for fallback fields, the result is cached in memory for the duration of the run to avoid re-reading the same file.

### Data Flow

```
wint.stories DB  ──→  getAllStories()  ──→  groupStoriesByPhase()
                                                    │
StoryFileAdapter  ──→  YAML fallback  ──────────────┤
  (phase, risk_notes,                               │
   feature, infra)                                  ▼
                                         renderStorySection()
                                                    │
                                         computeProgressSummary()
                                         computeReadyToStart()
                                                    │
                                         renderFrontmatter()
                                                    │
                                         writeIndexFile() [atomic]
                                                    │
                                         writeGenerationReport()
```
