# WINT-4010: Create Cohesion Sidecar

## Context

The WINT workflow needs an automated way to detect Franken-features (features with incomplete CRUD capability coverage) and capability gaps at two points in the workflow lifecycle: (1) post-bootstrap audit of the current graph state, and (2) at gate transitions where a feature's completeness is evaluated before allowing progression.

The graph schema (WINT-0060/WINT-0131) and graph MCP query tools (WINT-0130) already exist and provide the underlying data: `graph.franken_features` view, `graph.capability_coverage` view, and `graph_check_cohesion`/`graph_get_franken_features`/`graph_get_capability_coverage` MCP tools. However, agents currently have to call these MCP tools individually and interpret results. The cohesion sidecar wraps this capability into two focused HTTP endpoints that return structured, actionable cohesion analysis.

The sidecar follows the established pattern from WINT-2010 (role-pack sidecar) and WINT-2020 (context-pack sidecar): Node.js `node:http`, Zod-validated request/response, injectable dependencies, no HTTP framework, internal-only deployment.

---

## Goal

Create a Node.js HTTP sidecar at `packages/backend/sidecars/cohesion/` that exposes two POST endpoints — `/cohesion/audit` (full graph sweep) and `/cohesion/check` (per-feature gate check) — and MCP tool wrappers so agents can detect Franken-features and capability gaps without raw MCP tool chaining.

---

## Examples

**Example 1: Full audit after bootstrap**
- Given: `POST /cohesion/audit` with body `{}` when graph contains 3 features — "wish" (complete: create/read/update/delete), "wint" (incomplete: create/read only), "kfmb" (incomplete: create only)
- The system produces: `{ ok: true, data: { frankenFeatures: [{ featureId: "wint", missingCapabilities: ["update", "delete"] }, { featureId: "kfmb", missingCapabilities: ["read", "update", "delete"] }], coverageSummary: { totalFeatures: 3, completeCount: 1, incompleteCount: 2 } } }`

**Example 2: Gate check for a specific feature**
- Given: `POST /cohesion/check` with body `{ "featureId": "wish" }` when feature "wish" has all four CRUD capabilities mapped in graph
- The system produces: `{ ok: true, data: { featureId: "wish", status: "complete", violations: [], capabilityCoverage: { create: true, read: true, update: true, delete: true } } }`

**Example 3: MCP tool wrapper invocation**
- Given: Agent calls `cohesion_audit()` via MCP
- The system produces: Typed `CohesionAuditResult` with `frankenFeatures[]` and `coverageSummary` — same shape as HTTP endpoint, accessible without constructing raw HTTP requests

---

## Edge Cases

**Edge Case 1: Empty graph (WINT-4030 not yet run)**
- Given: `POST /cohesion/audit` when `graph.features` table is empty (no features populated yet)
- The system produces: `{ ok: true, data: { frankenFeatures: [], coverageSummary: { totalFeatures: 0, completeCount: 0, incompleteCount: 0 } } }` — graceful empty result, never an error

**Edge Case 2: Unknown featureId in cohesion check**
- Given: `POST /cohesion/check` with body `{ "featureId": "nonexistent-feature" }` when no capabilities exist for that feature in the graph
- The system produces: `{ ok: true, data: { featureId: "nonexistent-feature", status: "unknown", violations: [], capabilityCoverage: {} } }` — unknown status instead of error

**Edge Case 3: Invalid input — missing required featureId**
- Given: `POST /cohesion/check` with body `{}` (featureId missing)
- The system produces: `{ ok: false, error: "Invalid input: featureId is required" }` with HTTP 400 — Zod validation rejects before reaching compute function

**Edge Case 4: DB failure during audit**
- Given: `POST /cohesion/audit` when database is unavailable or query throws
- The system produces: `{ ok: false, error: "Internal error" }` with HTTP 500 — error logged via `@repo/logger`, no uncaught exception propagated to caller

## Non-Goals

- Authentication/authorization on sidecar endpoints (deferred per WINT-2020 precedent — internal-only VPC deployment)
- Populating the graph with features (that is WINT-4030's scope)
- Inferring capabilities from existing stories (that is WINT-4040's scope)
- Defining cohesion rules (that is WINT-4050's scope)
- Caching cohesion results (deferred; context-pack sidecar handles caching; cohesion results are computed fresh)
- Frontend UI for cohesion results (out of scope entirely for this epic phase)
- Modifying or extending the graph schema or MCP tools in `packages/backend/mcp-tools/src/graph-query/`

## Architecture Notes

- **No HTTP framework**: Node.js `node:http` only — established by WINT-2010 and WINT-2020; enforced as non-negotiable
- **PORT 3092**: Sequential port assignment (role-pack=3090, context-pack=3091, cohesion=3092); document in `server.ts` comment
- **Pure compute functions**: `computeAudit` and `computeCheck` accept DB as injectable parameter — enables unit testing without a real database
- **Discriminated union HTTP responses**: Always `{ ok: true, data: T }` or `{ ok: false, error: string }` — machine-readable for consuming agents (WINT-4060, WINT-4070)
- **No cross-sidecar imports**: Sidecars are independent packages; copy `sendJson`/`readBody` helpers, do not import them
- **Empty graph handling**: Graph data will be absent until WINT-4030 runs; compute functions must return empty-but-valid results (not errors) when Drizzle queries return empty arrays
- **Canonical implementation order**: (1) `__types__/index.ts` schemas, (2) `compute-audit.ts`, (3) `compute-check.ts`, (4) route handlers, (5) `server.ts`, (6) `index.ts` MCP wrappers, (7) unit tests, (8) integration test

## HTTP Contract

### POST /cohesion/audit
```
Request:
  Content-Type: application/json
  Body: { "packageName"?: string }  (optional filter)

Response 200:
  { "ok": true, "data": { "frankenFeatures": FrankenFeatureItem[], "coverageSummary": { "totalFeatures": number, "completeCount": number, "incompleteCount": number } } }

Response 400 (validation failure):
  { "ok": false, "error": string }

Response 500 (internal error):
  { "ok": false, "error": "Internal error" }
```

### POST /cohesion/check
```
Request:
  Content-Type: application/json
  Body: { "featureId": string }  (required)

Response 200:
  { "ok": true, "data": { "featureId": string, "status": "complete" | "incomplete" | "unknown", "violations": string[], "capabilityCoverage": { "create"?: boolean, "read"?: boolean, "update"?: boolean, "delete"?: boolean, ... } } }

Response 400 (validation failure — missing/invalid featureId):
  { "ok": false, "error": string }

Response 500 (internal error):
  { "ok": false, "error": "Internal error" }
```
