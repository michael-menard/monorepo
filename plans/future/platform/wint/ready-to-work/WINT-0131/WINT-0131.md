# WINT-0131: Add Feature-Capability Linkage to WINT Schema

## Context

WINT-0130 (Create Graph Query MCP Tools) implemented 4 graph query MCP tools, but discovered at implementation time that the `wint.capabilities` table is missing a `featureId` foreign key column linking capabilities to features. This caused 2 of the 4 tools to operate as stubs:

- `graph_get_franken_features` — returns empty array `[]` with a schema-limitation warning
- `graph_get_capability_coverage` — returns `null` with a schema-limitation warning

The other 2 tools (`graph_check_cohesion` and `graph_apply_rules`) are fully functional but also have zero test coverage — all 4 tools deferred their test suites from WINT-0130 due to time-boxing. WINT-0130 had 0% actual coverage against its 80% target (AC-14 unmet).

The schema gap is: the `wint.capabilities` table has no FK to `wint.features`, making it impossible to compute CRUD completeness per feature (Franken-feature detection) or per-feature capability coverage.

**Priority rationale**: P0 because this blocks:
- WINT-4060 (Create graph-checker Agent) — the agent cannot detect Franken-features without functional tools
- WINT-4030 (Populate Graph with Existing Features) — coverage analysis is prerequisite

**Schema source of truth**: `wint.ts` (established by WINT-0130 decision_002). The package index (`index.ts`) re-exports `features`, `capabilities`, `featureRelationships`, and `cohesionRules` from `./wint`. The `unified-wint.ts` capabilities table is NOT exported and must not be modified in this story (deferred to WINT-1100).

**Next migration number**: 0027 (last applied migration is 0026: `0026_wint_1130_worktree_tracking`).

---

## Goal

Complete the WINT-0130 graph query implementation by:

1. Adding a `feature_id uuid NULL REFERENCES wint.features(id) ON DELETE SET NULL` column to `wint.capabilities` via migration 0027
2. Updating the Drizzle schema in `wint.ts` to include the `featureId` column and bidirectional relations
3. Replacing the stub implementations in `graph_get_franken_features` and `graph_get_capability_coverage` with full DB-backed implementations
4. Writing comprehensive unit tests for all 4 graph query MCP tools (the deferred WINT-0130 test suite) achieving 80%+ line coverage

## Architecture Notes

### Schema Column Specification

```typescript
// In wint.ts capabilities table definition:
featureId: uuid('feature_id')
  .references(() => features.id, { onDelete: 'set null' })
```

The column must be:
- Nullable (`featureId` not `featureId.notNull()`)
- FK to `wint.features.id` with `ON DELETE SET NULL` (so deleting a feature does not delete its capabilities)
- Indexed for query performance

### Franken-Feature Detection Logic

A Franken-feature is any feature that has at least one capability linked (featureId is set) but does NOT have all 4 CRUD capability types (`create`, `read`, `update`, `delete`). Features with NO capabilities (all NULL featureId) are also Franken-features per the definition. Since existing seed data (WINT-0080) will have NULL featureId, the tool will initially return all features as Franken-features until WINT-4040 populates the featureId linkage. This is expected behavior.

### Dual-ID Support

Both tools that accept a `featureId` parameter support either:
- A UUID (queries `features.id`)
- A feature name string (queries `features.featureName`)

This pattern is already established in `graph-check-cohesion.ts` via `FeatureIdSchema` and the `or(eq(...), eq(...))` Drizzle query pattern.

### unified-wint.ts Drift Note

After this story, `wint.ts` will have `featureId` on the `capabilities` table but `unified-wint.ts` will not. This is an accepted known state. Add this comment to `unified-wint.ts` capabilities table definition for traceability:
