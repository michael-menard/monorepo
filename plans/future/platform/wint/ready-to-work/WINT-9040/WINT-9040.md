# WINT-9040: Create scope-defender LangGraph Node (`nodes/story/scope-defend.ts`)

## Context

The `scope-defender` agent is a haiku-powered Devil's Advocate worker that challenges non-MVP scope during story elaboration. It currently runs as a Claude Code agent (`scope-defender.agent.md`) and produces `scope-challenges.json` for downstream synthesis.

Phase 9 of the WINT epic ports all Claude Code haiku worker agents to native TypeScript LangGraph nodes. WINT-9040 is the Phase 9 entry for scope-defender, creating a native node at `packages/backend/orchestrator/src/nodes/story/scope-defend.ts` that executes the same 4-phase logical workflow as the agent.

The LangGraph porting notes in `scope-defender.agent.md` define the precise input/output contract. The implementation follows the same structural pattern as `attack.ts` (closest sibling in the same directory) and `doc-sync.ts` (gold standard for `@repo/workflow-logic` import patterns).

**Dependency note**: The WINT-4060 dependency listed in the stories index is an artifact. The scope-defender agent's non-goals explicitly state it does NOT consume graph-checker output. WINT-4060 is not a true implementation dependency for this node. WINT-9010 (`@repo/workflow-logic`) is confirmed deployed and functional.

---

## Goal

Port the scope-defender Claude Code agent to a native TypeScript LangGraph node that executes the same 4-phase Devil's Advocate workflow, writes `scope-challenges.json` to the story directory, and integrates seamlessly into the orchestrator graph.

---

## Examples

**Given**: A story in the orchestrator graph state with `story_id = "WISH-2068"`, `story_brief = { title: "Add export feature", goal: "...", context: "..." }`, and 6 acceptance criteria where AC-1 is marked MVP-critical in `gap_analysis`.

**The system produces**: `{story_dir}/_implementation/scope-challenges.json` with up to 5 challenge objects covering ACs 2-6 (AC-1 excluded as MVP-critical), each with `challenge_type`, `ac_ref`, `rationale`, and `risk_if_deferred` fields. The `truncated` field is `false` since only 5 challenges were produced.

---

**Given**: A story state with 9 non-MVP-critical ACs and `role_pack_path = null`.

**The system produces**: `scope-challenges.json` with exactly 5 challenges (hard cap enforced, `truncated: true`), `warning_count` incremented by 1 for the missing role pack, and the node returns updated state without throwing.

---

## Edge Cases

**Missing optional inputs (gap_analysis = null)**: The node increments `warning_count` by 1 and continues execution without gap_analysis filtering. All ACs are treated as challenge candidates (none are excluded as MVP-critical since there is no gap analysis to read from).

**All ACs are MVP-critical**: When gap_analysis marks every AC as blocking or MVP-critical, the node produces `scope-challenges.json` with `challenges: []` and `truncated: false`. No exception is thrown and the node returns updated state.

## Architecture Notes

**Node factory pattern (from WINT-9030 lesson)**:
```typescript
// CORRECT — use attack.ts pattern exactly
export const scopeDefendNode = createToolNode('scope_defend', async (state: GraphStateWithScopeDefend) => {
  // Phase 1: Load Inputs
  // Phase 2: Identify Challenge Candidates
  // Phase 3: Apply DA Challenges (max 5)
  // Phase 4: Produce Output
  return updateState(state, { ... })
})
```

**Zod-first state extension (from WINT-9090 lesson)**:
```typescript
// CORRECT
const GraphStateWithScopeDefendSchema = GraphStateSchema.extend({
  story_brief: z.object({ title: z.string(), goal: z.string(), context: z.string() }),
  acceptance_criteria: z.array(z.string()),
  gap_analysis: z.object({ ... }).nullable(),
  role_pack_path: z.string().nullable(),
})
type GraphStateWithScopeDefend = z.infer<typeof GraphStateWithScopeDefendSchema>

// WRONG — never use TypeScript interface
interface GraphStateWithScopeDefend { ... }
```

**4-phase execution contract** (from `scope-defender.agent.md` LangGraph Porting Notes):
1. Load Inputs — parse state fields into working context
2. Identify Challenge Candidates — filter ACs, exclude MVP-critical items from gap_analysis
3. Apply DA Challenges — cap at 5, prioritize by `risk_if_deferred`, produce challenge objects
4. Produce Output — write `scope-challenges.json` to `{story_dir}/_implementation/scope-challenges.json`

**DA constraints (non-negotiable)**:
- Scope-defend only reduces or defers — never adds ACs or expands scope
- Maximum 5 challenges regardless of candidate count; set `truncated: true` if capped
- Never challenge items marked blocking or MVP-critical in gap analysis
- Missing optional inputs (`gap_analysis`, `role_pack_path`) increment `warning_count` — do NOT block
