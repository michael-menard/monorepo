# WINT-3070: Update 10 Core Workflow Commands with `telemetry-log` Invocations

## Context

WINT-3020 established the `telemetry-log` skill and the `workflow_log_invocation` MCP tool to capture structured observability data for each agent session. The skill is purpose-built to be fire-and-forget: any return value — including null or an error — must never block the calling workflow.

The 10 primary WINT workflow commands (`pm-bootstrap-workflow`, `elab-epic`, `elab-story`, `dev-implement-story`, `dev-fix-story`, `qa-verify-story`, `pm-refine-story`, `story-status`, `story-update`, `story-move`) currently complete phases without recording any structured observability data. When an agent session ends, all knowledge of which orchestrator ran, what story it processed, and which phase it was in is lost.

This story is the instrumentation step: add fire-and-forget `telemetry-log` invocations to each of the 10 core commands at their phase completion boundaries. Changes are purely additive Markdown edits to existing `.claude/commands/*.md` files — no TypeScript code is written.

**Prerequisite**: WINT-3020 must be merged before UAT. The `telemetry-log` skill and `workflow_log_invocation` MCP tool it produces are required to validate AC-14.

---

## Goal

Add fire-and-forget `telemetry-log` invocations to the 10 core WINT workflow commands so that every agent session produces a structured row in `wint.agent_invocations`, enabling observability of which orchestrators ran, which stories they processed, and which phases they completed.

---

## Examples

**Example 1 — dev-implement-story telemetry call:**
Given `dev-implement-story` is executed against `WINT-4060`, the command's execute-phase completion block now invokes the `telemetry-log` skill with `agentName='dev-implement-story'`, `storyId='WINT-4060'`, `phase='execute'`, `status='success'`. After the command completes, `SELECT * FROM wint.agent_invocations WHERE story_id='WINT-4060'` returns a row with those exact values.

**Example 2 — pm-bootstrap-workflow telemetry call:**
Given `pm-bootstrap-workflow` runs the setup phase for a new epic, the command file's setup-phase completion block invokes `telemetry-log` with `agentName='pm-bootstrap-workflow'`, `phase='setup'`. The call returns (or null if MCP is down), and the command continues without blocking. The primary workflow output (epic plan) is produced regardless of whether the telemetry write succeeded.

---

## Edge Cases

**Edge Case 1 — elab-story.md dual-mode (interactive vs autonomous):**
`elab-story.md` supports two execution modes. Both modes must be covered by a telemetry block — either a shared block placed at a common completion boundary, or mode-specific blocks if the paths diverge before completion. Instrumenting only one mode would leave the other invisible to telemetry.

**Edge Case 2 — story-status.md read-only utility:**
`story-status.md` is a read-only status check with no state transitions. Whether telemetry logging is meaningful for a read-only command is a deliberate design decision. AC-8 requires that the decision be documented explicitly — either a telemetry block is present, or a comment explains why the command is exempt. Silent omission is not acceptable.

**Edge Case 3 — MCP tool unavailable at runtime:**
If `workflow_log_invocation` returns null or throws, the telemetry block must not propagate that failure to the calling workflow. The fire-and-forget constraint means the command's primary output must be produced regardless of telemetry availability.

---

## Non-Goals

- Writing any TypeScript source code (this is a docs-only/Markdown story)
- Modifying the `telemetry-log` skill itself (WINT-3020 scope)
- Instrumenting agent worker files (e.g., `elab-analyst.agent.md`, `dev-execute-leader.agent.md`) — only the 10 named top-level orchestrator command files are in scope
- Adding batching, async queuing, or error escalation to the telemetry calls
- Retroactively backfilling telemetry for past story executions
- Dashboard or query UI for telemetry data (WINT-3060 scope)
- Any changes to the `wint.agent_invocations` table schema

## Architecture Notes

This story is intentionally a docs-only change. The architecture is established by WINT-3020:

- `telemetry-log` skill: fire-and-forget MCP skill invocation
- `workflow_log_invocation` MCP tool: writes to `wint.agent_invocations` table in postgres-knowledgebase
- Data flow: command file → `telemetry-log` skill → `workflow_log_invocation` MCP → `wint.agent_invocations` (port 5433)

**Non-breakage constraint**: All 10 target files are the most frequently invoked orchestrators in the WINT system. Telemetry calls are appended at phase completion boundaries — they are side-effects, not gates. A failure in the telemetry path must never interrupt the primary workflow.

**Insertion pattern**: For each file, locate the phase completion signal (Markdown block or comment signaling phase end — often where `token-log` already appears). Insert the `telemetry-log` skill invocation immediately after `token-log` (or at the same boundary if `token-log` is not present).
