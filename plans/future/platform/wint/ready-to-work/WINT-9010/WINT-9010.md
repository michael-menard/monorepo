# WINT-9010: Create Shared Business Logic Package

## Context

WINT-1100 (Create Shared TypeScript Types) established Zod schemas shared between LangGraph nodes and MCP tools. WINT-9010 takes the next step: extracting **business logic** (not just types) into a shared package so that both execution paths (Claude Code MCP tools and LangGraph nodes) can call the same functions without duplication.

Currently, any business logic that needs to run in both contexts must be implemented twice — once in a Claude Code agent/MCP tool and once in a LangGraph node. This creates drift risk, maintenance overhead, and the possibility of behavioral divergence between the two paths.

The proposed package `packages/backend/workflow-logic/` will contain pure TypeScript functions with no runtime-specific dependencies. MCP tools and LangGraph nodes import from this package rather than duplicating logic.

**Phase 9 context**: This is the first story in the LangGraph Parity phase. WINT-9010 creates the foundation layer that all subsequent WINT-9xxx stories build upon. WINT-9020 through WINT-9050 port individual agents to LangGraph nodes — they all depend on this shared logic package existing first.

---

## Goal

Create `packages/backend/workflow-logic/` as a shared TypeScript package containing business logic functions extracted from existing MCP tools and agents. Both Claude Code MCP tools and LangGraph nodes can import from this package. Single implementation, two execution paths.

---

- Do NOT create LangGraph graphs or nodes (blocked until this package exists)

- Do NOT refactor existing MCP tools to use the new package (only extract; MCP tools may continue using their existing implementations or import from the shared package — that migration is per-story in WINT-9020+)

### Initial Modules to Extract

Based on the existing MCP tools and LangGraph nodes, the following logic candidates are in scope for extraction:

1. **Story status transitions** — logic for determining valid state transitions (e.g., `backlog → elaboration → ready-to-work → in-progress → ...`)
2. **Swim-lane directory mapping** — the mapping used by WINT-1030's compatibility shim (`shimGetStoryStatus` directory fallback)
3. **Story ID parsing/validation** — ID format validation (e.g., `WINT-9010` format rules)

These are the minimal set needed to unblock WINT-9020+ without over-extracting.
