# WINT-3060: Create Telemetry Query Command (`/telemetry`)

## Context

Phase 3 of the WINT platform epic introduces full observability over agent invocations and human decisions. WINT-0040 creates the `wint.agent_invocations`, `wint.hitl_decisions`, and `wint.story_outcomes` tables. WINT-0120 delivers `workflow_get_story_telemetry` as the canonical MCP read interface. WINT-3020/3030/3040 populate those tables during workflow execution.

Currently there is no human-facing interface to review what telemetry has been captured for a given story. Platform operators and developers cannot easily inspect how many agent invocations occurred, what human decisions were made, or what the final outcome verdict was. This makes it impossible to validate that telemetry collection (WINT-3080) is working correctly.

**Dependency Gate:** This story depends on WINT-3010 (Gatekeeper Sidecar), WINT-3030 (telemetry-logger Agent), WINT-3040 (Decision Logging Skill), and WINT-0120 (Telemetry MCP Tools). UAT must not start until WINT-0120 is at UAT. WINT-3040 is a data-enrichment dependency only — empty decisions is valid behavior.

---

## Goal

Create a read-only `/telemetry` command that accepts a `--story STORY_ID` filter and displays a formatted, human-readable summary of agent invocations, HITL decisions, and story outcomes by calling the `workflow_get_story_telemetry` MCP tool.

---

## Examples

**Example 1 — Happy path with full telemetry data:**
- Input: `/telemetry --story WINT-3010`
- Output: Formatted table showing 5 agent invocations (agent_name, status, duration_ms, model_name), 2 HITL decision rows, outcome verdict "passed". Signal: `TELEMETRY RENDERED`

**Example 2 — Custom row limit:**
- Input: `/telemetry --story WINT-0040 --limit 5`
- Output: First 5 invocation rows shown with note "Showing 5 of 12 invocations. Use --limit N to see more." Signal: `TELEMETRY RENDERED`

---

## Edge Cases

**Edge Case 1 — No `--story` flag provided:**
- Input: `/telemetry` (no flags)
- Output: Usage hint displayed (e.g., "Usage: /telemetry --story STORY_ID [--limit N]" with an example). No unfiltered query is attempted. No database call made.

**Edge Case 2 — Story exists but no telemetry logged yet:**
- Input: `/telemetry --story WINT-NEW-01` (story is new, no data in wint.* tables)
- Output: "No telemetry data found for WINT-NEW-01". Signal: `TELEMETRY EMPTY: WINT-NEW-01`. Does not throw, does not display raw null.

**Edge Case 3 — Default row cap with high-volume story:**
- Input: `/telemetry --story WINT-LOAD` (story has 50 invocations, no `--limit` specified)
- Output: First 20 invocation rows shown with advisory "Showing 20 of 50 invocations. Use --limit N to see all."

**Edge Case 4 — MCP tool unavailable:**
- Input: `/telemetry --story WINT-3060` (postgres-knowledgebase unreachable on port 5433)
- Output: "Telemetry service unavailable. Ensure postgres-knowledgebase is running on port 5433." Signal: `TELEMETRY ERROR: service_unavailable`

---

## Non-Goals

- Do NOT create a new MCP tool — `workflow_get_story_telemetry` (WINT-0120) is the data source
- Do NOT add aggregate analytics or trend charts — that scope belongs to WINT-3090 (scoreboard metrics)
- Do NOT query `telemetry.state_transitions` — that table is created by WINT-3100 and is out of scope
- Do NOT display model cost calculations — that belongs to WINT-0260 (model cost tracking)
- Do NOT spawn a subagent for this command — it is a simple read/format operation that runs inline
- Do NOT add write capability (update/delete telemetry) — read-only only
- **Protected**: `apps/api/knowledge-base/` source files must not be modified; DB schema must not be modified

## Acceptance Criteria

- [ ] **AC-1**: `/telemetry` command file exists at `.claude/commands/telemetry.md` with valid frontmatter (`type: utility`, `permission_level: docs-only`, `created`, `updated`, `version`)
- [ ] **AC-2**: `/telemetry --story STORY_ID` calls `workflow_get_story_telemetry({ storyId: STORY_ID })` MCP tool and displays formatted output showing: invocation count, decision count, and outcome verdict (or "no outcome recorded")
- [ ] **AC-3**: When MCP tool returns empty result (no data for storyId), the command displays "No telemetry data found for STORY_ID" — does not throw or display raw null
- [ ] **AC-4**: When MCP tool is unavailable (connection error), the command displays "Telemetry service unavailable. Ensure postgres-knowledgebase is running on port 5433." and exits gracefully
- [ ] **AC-5**: Command accepts `--story STORY_ID` as the primary filter (required for meaningful output); when invoked without `--story`, it displays a usage hint and example rather than attempting an unfiltered query
- [ ] **AC-6**: Invocations section displays `agent_name`, `status` (success/failure), `duration_ms` (if available), and `model_name` (if available) in a formatted table — capped at 20 rows by default
- [ ] **AC-7**: Decisions section displays `decision_type`, `decision_text` (truncated to 80 chars), and `operator_id` in a formatted table
- [ ] **AC-8**: Command documents a `--limit N` optional flag that overrides the default 20-row cap (for power users inspecting high-volume stories)
- [ ] **AC-9**: Command emits exactly one of: `TELEMETRY RENDERED`, `TELEMETRY EMPTY: {storyId}`, or `TELEMETRY ERROR: {reason}` as its completion signal

## Architecture Notes

- **MCP-only read path**: The command calls `workflow_get_story_telemetry({ storyId })` exclusively. No raw SQL, no direct DB connection from command file.
- **Inline execution**: No subagent spawn. Command executes directly in Claude conversation context.
- **Docs-only**: The command file is markdown only. No TypeScript implementation.
- **Port 5433**: postgres-knowledgebase runs on port 5433. The error message must reference this port explicitly (per ADR-005 and seed constraint).
- **Query performance**: Default 20-row cap per section prevents unbounded result sets. `--limit N` allows override.
- **Decisions are soft-dependency data**: If WINT-3040 is not yet live, decisions section returns empty array — this is valid behavior, not an error.
- **WINT-0120 interface**: Before writing the Implementation section (ST-2), dev must read `plans/future/platform/wint/in-progress/WINT-0120/WINT-0120.md` to confirm the final `workflow_get_story_telemetry` parameter/return schema.
