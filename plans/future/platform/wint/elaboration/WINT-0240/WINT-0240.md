# WINT-0240: Configure Ollama Model Fleet

## Context

The WINT system's local-first model strategy (WINT-0220) requires a fleet of Ollama-hosted models to be available on the developer's machine before any workflow agents can execute without incurring cloud API costs. As of 2026-02-20, a partial setup script exists at `packages/backend/orchestrator/setup-ollama.sh` that pulls four models (`deepseek-coder-v2:33b`, `qwen2.5-coder:7b`, `codellama:13b`, `llama3.2:8b`), but this script:

- Is scoped to the orchestrator package rather than the monorepo root
- Pulls `deepseek-coder-v2:33b` (full 33B) without a lite alternative
- Does not match the model set defined in `WINT-0220-STRATEGY.yaml` (which specifies `deepseek-coder-v2:16b` as Tier 2 primary)
- Does not provide a memory-constrained (`--lite`) path for machines with less than 10GB VRAM
- Does not include a standalone health check script
- Does not document VRAM requirements per model in a structured, cross-referenced format
- Does not print environment variable configuration instructions

The `WINT-0220-STRATEGY.yaml` file exists at `packages/backend/orchestrator/docs/WINT-0220-STRATEGY.yaml` (version 1.0.0, effective 2026-02-15) and defines:
- **Tier 2 (Routine Work)**: `deepseek-coder-v2:16b`, `codellama:13b`, `qwen2.5-coder:14b`
- **Tier 3 (Simple Tasks)**: `qwen2.5-coder:7b`, `llama3.2:3b`

The `model-assignments.yaml` at `.claude/config/model-assignments.yaml` lists the Ollama models agents currently expect:
- `ollama:deepseek-coder-v2:33b` (complex code generation, `dev-implement-backend-coder`)
- `ollama:deepseek-coder-v2:16b` (lighter alternative)
- `ollama:qwen2.5-coder:7b` (fast simple tasks)
- `ollama:codellama:13b` (mid-complexity tasks)
- `ollama:llama3.2:8b` (general purpose)

**Dependency status**: WINT-0220 is marked `pending` in the stories index despite the strategy artifacts being available on disk. This story should not move to `ready-to-work` until WINT-0220 is formally completed.

## Goal

Produce three artifacts that together enable one-command setup of the local Ollama model fleet, with a memory-constrained lite path, a standalone CLI health check, and comprehensive VRAM documentation aligned with the WINT-0220 strategy.

- **Creating a runtime HTTP health endpoint (always-on server)** — the health check is a CLI script, not a service

- **Automated VRAM detection and model switching during inference** — that is WINT-0250 (escalation triggers) and future ML pipeline scope
- **Modifying `OllamaProvider` TypeScript code** — this story is scripting and documentation only
- **Adding new models not referenced in WINT-0220 strategy** — fleet composition is dictated by WINT-0220
- **CI/CD integration of health check** — consuming the health check in CI pipelines is a follow-up task
- **`model-strategy.yaml` creation** — that artifact is produced by WINT-0220

## Acceptance Criteria

- [ ] **AC-1: Setup Script Exists and Is Executable**
  - `scripts/setup-ollama-models.sh` exists, is executable (`chmod +x`), and has a shebang line (`#!/usr/bin/env bash` or `#!/bin/bash`)
  - Script begins with `set -e` and validates Ollama installation with `command -v ollama` before any pull attempt
  - Script validates Ollama server is running at `$OLLAMA_BASE_URL` (defaulting to `http://127.0.0.1:11434`) via `curl -s $OLLAMA_BASE_URL/api/tags`
  - Failure messages include install/start instructions (macOS: `brew install ollama`, Linux: curl installer)

- [ ] **AC-2: Full Fleet Pull**
  - Script pulls all models required by the current model-assignments (at minimum: `qwen2.5-coder:7b`, `codellama:13b`, and the Tier 2 coding models from WINT-0220-STRATEGY.yaml)
  - Model names are valid Ollama registry names (verified against `ollama.com/library`)
  - Script confirms each pull with a progress indicator
  - Model list in the script references or is derived from `WINT-0220-STRATEGY.yaml`, not specified independently

- [ ] **AC-3: Memory-Constrained Path**
  - Script accepts a `--lite` flag that skips models requiring >10GB VRAM
  - `--lite` flag is documented in the script's `--help` output
  - The lite model set satisfies the Tier 3 (simple tasks) requirements from WINT-0220 (at minimum `qwen2.5-coder:7b`)

- [ ] **AC-4: Post-Install Verification**
  - After pulling, script runs `ollama list` and checks each required model is present
  - Script reports any missing models with their pull command
  - Script exits with code 0 only if all required models are present (or `--lite` set is complete)
  - Script exits with code 1 if any required model is missing post-pull

- [ ] **AC-5: Health Check Script Exists and Is Correct**
  - `scripts/check-ollama-health.sh` exists, is executable, and returns exit code 0 when healthy
  - Health check queries `GET $OLLAMA_BASE_URL/api/tags` (matching `OllamaProvider.checkAvailability()` logic in `packages/backend/orchestrator/src/providers/ollama.ts`)
  - Health check verifies each model in `model-assignments.yaml` appears in `ollama list` output
  - Health check outputs a structured report (model name, status: present/missing) to stdout
  - Script reports missing models with the pull command to fix
  - Script returns exit code 1 if any required model is missing or server is unreachable

- [ ] **AC-6: VRAM Documentation**
  - `docs/tech-stack/ollama-model-fleet.md` exists and is structured as a table with columns: Model Name, Tier, Ollama Registry Tag, VRAM Required, Use Cases, Notes
  - All models referenced in `model-assignments.yaml` have VRAM entries
  - Documentation distinguishes between full-precision and quantized variants where relevant (e.g., `:33b` vs `:16b`)
  - A "Recommended Minimum" section states the minimum hardware to run the Tier 3 fast fleet

- [ ] **AC-7: Alignment with WINT-0220 Strategy**
  - The model list in `scripts/setup-ollama-models.sh` matches the Ollama-provider models prescribed by `WINT-0220-STRATEGY.yaml`
  - Any discrepancy between the script's model list and `model-assignments.yaml` is documented with rationale
  - The VRAM documentation cross-references `WINT-0220-STRATEGY.yaml` tier assignments

- [ ] **AC-8: Existing Script Superseded**
  - `packages/backend/orchestrator/setup-ollama.sh` either: (a) is removed, (b) updated to delegate to `scripts/setup-ollama-models.sh`, or (c) contains a deprecation notice pointing to the new location
  - No duplication of model pull logic across two scripts

- [ ] **AC-9: Environment Variable Documentation**
  - Setup script `--help` output lists all relevant Ollama env vars: `OLLAMA_BASE_URL`, `OLLAMA_TEMPERATURE`, `OLLAMA_TIMEOUT_MS`, `OLLAMA_ENABLE_FALLBACK`
  - Documentation file includes an "Environment Variables" section with the same list and their defaults (from `OllamaConfigSchema` in `packages/backend/orchestrator/src/providers/ollama.ts`)

## Architecture Notes

1. **Script location convention**: The monorepo uses `scripts/*.sh` for top-level orchestration scripts. The new setup script belongs at `scripts/setup-ollama-models.sh` — do NOT place it in `packages/backend/orchestrator/`.

2. **Single source of truth for model list**: The setup script must reference `WINT-0220-STRATEGY.yaml` as the authoritative model source. If model names are hardcoded in the script, they must be preceded by a comment: `# Source: packages/backend/orchestrator/docs/WINT-0220-STRATEGY.yaml tier 2/3`.

3. **No `eval()` or dynamic execution**: Script must use static model arrays. No dynamic evaluation of YAML (shell scripts cannot parse YAML natively — the model list should be maintained as a shell array aligned with the YAML).

4. **Env var handling**: Always use `${OLLAMA_BASE_URL:-http://127.0.0.1:11434}` pattern for default values. Never hardcode the URL without a fallback.

5. **Health check exit codes**: Follow POSIX convention strictly — exit 0 for healthy, exit 1 for unhealthy. Do not use other exit codes.
