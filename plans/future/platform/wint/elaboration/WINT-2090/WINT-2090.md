# WINT-2090: Implement Session Context Management — session-create and session-inherit Skills

## Context

The WINT platform is in Phase 2 (Context Cache & Sidecars). Phase 2's goal is to achieve an 80% token reduction by introducing shared sidecar services, context caching, and session-based state sharing between leader agents and their spawned workers.

**The current gap**: The `wint.contextSessions` table and its 5 MCP tools (from WINT-0110) are fully implemented on disk. However, there are no Claude Code skills that guide leader agents on how to open sessions or guide worker agents on how to inherit and report into them. Without these skills, the session infrastructure sits idle — the `wint.contextSessions` table accumulates no records and token accounting per workflow is impossible.

**What this story delivers**: Two Claude Code skills — procedural markdown documents that agents follow as step-by-step instructions:

1. **`session-create`** — Called by a leader agent at workflow start. Invokes `sessionCreate` MCP tool to open a record in `wint.contextSessions`. Emits a machine-parseable `SESSION CREATED` output block that the leader captures and injects into worker system prompts.

2. **`session-inherit`** — Called by a worker agent upon receiving a session ID. Invokes `sessionQuery` to verify the session is active. Documents how workers report incremental token usage via `sessionUpdate`. Explicitly forbids workers from calling `sessionComplete` (only the leader that opened the session may close it).

Both skills handle graceful degradation: if the database is unavailable, they emit a warning and allow the workflow to proceed without session tracking — session tracking is a telemetry concern, not a core workflow gate.

**Baseline**: Baseline date 2026-02-13. The baseline captures application-level reality (Drizzle schemas, pgvector KB, Docker Compose) but does not reflect WINT-specific session infrastructure. Session tools from WINT-0110 confirmed present on disk via codebase scan.

## Goal

Enable stateful workflow coordination between leader agents and spawned workers by providing standard skills for opening, inheriting, and closing sessions — activating the `wint.contextSessions` infrastructure from WINT-0110 for use by the first group of high-volume agents.

## Architecture Notes

**Session ownership model**:
- One leader opens → one `session-create` call → one UUID
- N workers inherit → N `session-inherit` calls → N `sessionUpdate` calls (incremental)
- One leader closes → one `sessionComplete` call
- Workers never call `sessionComplete`

**Token update safety**:
- `sessionUpdate` with `mode: 'incremental'` uses SQL arithmetic (`+= value`)
- This is concurrency-safe: multiple workers can call in parallel without race conditions
- `mode: 'absolute'` is explicitly forbidden in skill documentation for worker token reporting

**Graceful degradation contract**:
- `sessionCreate` returns `null` on DB error (already enforced by WINT-0110 implementation)
- Skills must surface this as a **warning**, not an exception
- Downstream workflow steps must not be gated on session tracking

**Skill invocation model**:
- Skills are markdown procedural documents
- Agents follow skill steps manually during their execution — there is no runtime skill executor
- Session IDs are passed between leader and workers via system prompt injection or initial context, not via file writes
