# WINT-9020: Create doc-sync LangGraph Node

## Context

The doc-sync agent (`doc-sync.agent.md` v1.1.0) is production-ready (WINT-0160 UAT PASS). It implements a 7-phase documentation synchronization workflow using git diff-based change detection, YAML frontmatter parsing, database-sourced metadata merge, section mapping, documentation table updates, Mermaid diagram regeneration, changelog drafting, and SYNC-REPORT.md generation.

WINT-0160 added a "LangGraph Porting Notes" section to `SKILL.md` â€” this section documents the canonical interface contract (inputs, 7-phase workflow, outputs, MCP tools) that WINT-9020 must satisfy.

A file `packages/backend/orchestrator/src/nodes/workflow/doc-sync.ts` already exists, but it is a subprocess wrapper that invokes `claude doc-sync` via `child_process.spawn`. It is not a direct port of the agent logic â€” it delegates everything back to the Claude Code CLI. This does not fulfill the WINT-9020 goal of running doc-sync logic natively in TypeScript without the CLI.

WINT-9010 (Create Shared Business Logic Package, UAT PASS) provides `@repo/workflow-logic` with `isValidStoryId`, `WorkflowStoryStatus`, and related utilities that this node may use during Phase 2 frontmatter parsing.

This story creates the direct TypeScript port at `packages/backend/orchestrator/src/nodes/sync/doc-sync.ts` â€” a self-contained LangGraph node that implements all 7 phases natively, enabling LangGraph-orchestrated workflows to run documentation sync without any Claude Code CLI dependency.

---

## Goal

Port the 7-phase doc-sync agent workflow to a TypeScript LangGraph node at `packages/backend/orchestrator/src/nodes/sync/doc-sync.ts`. The node must implement all 7 phases directly (no subprocess), accept `checkOnly` and `force` flags, produce an identical SYNC-REPORT.md structure to the agent, gracefully degrade when the `postgres-knowledgebase` MCP tools are unavailable, and be wrapped with `createToolNode()` for infrastructure concerns.

## Architecture Notes

### MCP Tool Injection Pattern

The MCP tools (`mcp__postgres-knowledgebase__query-workflow-components`, `mcp__postgres-knowledgebase__query-workflow-phases`) are Claude Code MCP tools â€” they are not importable TypeScript modules. The node uses a dependency injection pattern:

```typescript
// In DocSyncConfigSchema:
queryWorkflowComponents: z.function()
  .args(z.object({ component_types: z.array(z.string()), timeout: z.number().optional() }))
  .returns(z.promise(z.array(z.unknown())))
  .optional(),
queryWorkflowPhases: z.function()
  .args(z.object({ timeout: z.number().optional() }))
  .returns(z.promise(z.array(z.unknown())))
  .optional(),
```

When `undefined` (default): file-only mode. When injected by a LangGraph graph that has MCP access: full hybrid mode.

### Identical Outputs Requirement

The SYNC-REPORT.md written by `nodes/sync/doc-sync.ts` must parse correctly via `parseSyncReport()`. Specifically:
- `Total files changed: N` must be present and parseable
- `Total sections updated: N` must be present and parseable
- `Total diagrams regenerated: N` must be present and parseable
- `Manual review items: N` must be present and parseable
- `[DRAFT]` marker for changelog status must be present when changelog was drafted

Test EG-5 is the automated gate for this requirement.

### Deprecation of `nodes/workflow/doc-sync.ts`
