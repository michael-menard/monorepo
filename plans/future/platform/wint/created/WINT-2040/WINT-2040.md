# WINT-2040: Populate Agent Mission Cache

Parse all non-archived `.agent.md` files from `.claude/agents/`, extract mission/scope/signals content, and populate `wint.context_packs` for fast agent-to-agent context retrieval.

---

## Context

The WINT workflow has 142 active agent files in `.claude/agents/`. Each agent has a defined mission, scope, and signals documented in its `.agent.md` body content and YAML frontmatter. Currently, when an agent needs to know what another agent does (e.g., to decide which sub-agent to spawn), it must either load the target agent's full file or operate without that knowledge.

The `wint.context_packs` table (WINT-0010) and `contextCachePut` MCP tool (WINT-0100) provide a database-backed cache for exactly this use case. WINT-0080 already established infrastructure to parse agent frontmatter and the `wint.agents` table stores identity-level data. However, no story has yet extracted the richer mission/scope/signals content from agent files and stored it in the context cache for fast retrieval.

**Problem**: Agent mission context is not available for sub-5-token retrieval. Every invocation that needs another agent's context must either read the filesystem (slow, costs tokens for the full file) or operate blind. At scale with 80%+ token reduction targets (WINT-2120), this is a critical gap.

**Reality baseline**: Loaded from `plans/baselines/BASELINE-REALITY-2026-02-13.md`. Frontmatter parsing deployed (WINT-0080). `wint.context_packs` deployed (WINT-0010). `contextCachePut` implemented and in UAT (WINT-0100). `contextPackTypeEnum` does not yet include `agent_missions` value — migration required.

---

## Goal

Create an `agent-mission-cache-populator` script in `packages/backend/database-schema/src/seed/` that:
1. Discovers all non-archived `.agent.md` files via glob
2. Parses frontmatter reusing `parseFrontmatter` + `extractAgentMetadata` (WINT-0080)
3. Extracts mission/scope/signals from agent body content (H2/H3 sections)
4. Constructs a compact (<300 token) agent mission summary per agent
5. Writes each summary to `wint.context_packs` via direct Drizzle upsert with a deterministic key
6. Reports: total found, cached, skipped, warnings

Also resolves the `agent_missions` pack type gap by adding an enum value via migration.

---

## Non-Goals

- Do NOT build an HTTP endpoint — this is a population script, not a sidecar (that is WINT-2020)
- Do NOT modify the `.agent.md` files themselves
- Do NOT modify the `wint.agents` table schema or seeder (WINT-0080 is a protected story outcome)
- Do NOT build the cache-warming orchestration — that is WINT-2070
- Do NOT implement cache retrieval for agents — that is WINT-2110
- Do NOT deep-index the full agent body content — extract a compact summary only (<300 tokens per entry)
- Do NOT break existing context cache MCP tools or their tests

## Architecture Notes

**Pack type decision**: Use Option A — add `agent_missions` to `contextPackTypeEnum` via a PostgreSQL `ALTER TYPE ... ADD VALUE` migration. This is semantically correct, type-safe, and consistent with how other pack types work. The migration is a single non-blocking `ADD VALUE` statement (PostgreSQL 12+).

**packKey convention**: Agent name derived from filename without `.agent.md` extension. Example: `.claude/agents/pm-bootstrap-workflow.agent.md` → packKey = `pm-bootstrap-workflow`. This is deterministic and matches the naming pattern already used in `wint.agents`.

**Body content extraction approach**: Simple section-based regex extraction targeting H2 (`## Mission`, `## Role`, `## Scope`, `## Signals`) and H3 equivalents. The `gray-matter` library already strips frontmatter and returns the body string — parse body with `extractSection(body, 'Mission')`. Handles missing sections gracefully (returns null). No full Markdown parser needed.

**Content JSONB structure** (target <300 tokens per entry):
```json
{
  "mission": "<truncated at 200 chars>",
  "role": "<optional>",
  "scope": "<optional>",
  "triggers": ["..."],
  "model": "<haiku|sonnet|opus>",
  "permissionLevel": "<read-only|docs-only|restricted|standard>",
  "spawnedBy": ["..."]
}
```

**Script placement**: `packages/backend/database-schema/src/seed/agent-mission-cache-populator.ts`. All dependencies (gray-matter, glob, @repo/logger, Drizzle) already available in that package.

**Source data**: Read directly from `.claude/agents/` filesystem. Do NOT query `wint.agents` table — the DB table lacks the mission/scope body content that is the primary value of this cache.

**DB write approach**: Direct Drizzle upsert (`tx.insert(contextPacks).onConflictDoUpdate(...)`) rather than the `contextCachePut` MCP tool — this is independent of WINT-0100 UAT status and matches the pattern in `agent-seeder.ts`.
