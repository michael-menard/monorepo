## Context

The WINT migration project is transitioning story status tracking from directory-based swim-lane structure (`backlog/`, `ready-to-work/`, `in-progress/`, etc.) to a PostgreSQL database (`core.stories` table via `wint` pgSchema). WINT-0090 (UAT, QA PASS — 144 unit + 6 integration tests, 95% coverage) delivered four MCP tools that query and update story status via the database. WINT-1030 (UAT, QA PASS) populated the database with current story state seeded from directory locations.

However, dozens of agents and commands still operate by reading and writing the filesystem directly. Immediately migrating all agents to DB-only creates a hard cutover risk: if an unmigrated agent updates only the directory, a migrated agent querying only the database will see stale data. The two sources of truth will drift during the migration window (Phase 1 through Phase 7).

This story delivers the TypeScript compatibility shim module — specifically the four core functions — that bridges the gap. The shim encapsulates the DB-first, directory-fallback lookup strategy so that WINT-1040, WINT-1050, and WINT-1060 can update the three story commands using a single, well-tested integration point. The shim is explicitly a temporary artifact — it is scheduled for deletion in WINT-7100 once all 52 agents are migrated.

**Reality Baseline**: 2026-02-13 baseline active. WINT-0090 and WINT-1030 both passed QA after the baseline date, confirming their MCP tools and swim-lane state mapping are stable and production-ready.

## Architecture Notes

### Module Location Decision

**Recommended: Option A — Within `packages/backend/mcp-tools/src/story-compatibility/`**

The shim co-locates with the WINT-0090 tools it wraps. This avoids creating a new package for a temporary artifact, keeps the build pipeline simple, and makes WINT-7100 deletion straightforward (delete the directory, remove the export from `index.ts`).

Option B (new `packages/backend/story-shim/` package) was considered but rejected: adding a new pnpm package with its own Turborepo config, tsconfig, and build pipeline is disproportionate overhead for a module that will be deleted in Phase 7.

### Shim Routing Logic

```
shimGetStoryStatus(storyId):
  1. Validate storyId via StoryIdSchema (throw/return null on invalid)
  2. Try: result = await storyGetStatus(storyId)
  3. If result !== null: return normalize(result, source='db')
  4. Else (DB miss): scan directory for storyId, infer state from parent dir name
  5. If found in directory: return normalize(directoryResult, source='directory')
  6. Else: return null (source='not_found')

shimUpdateStoryStatus(storyId, newState):
  1. Validate inputs
  2. Try: result = await storyUpdateStatus(storyId, newState)
  3. If success: return result
  4. On connection error: log.warn(...), return null
  5. NEVER: fall back to filesystem write

shimGetStoriesByStatus(state) / shimGetStoriesByFeature(epic):
  1. Validate inputs
  2. Try: results = await storyGetByStatus/Feature(...)
  3. If results.length > 0: return normalize(results, source='db')
  4. Else (empty): scan directories, filter by state/epic
  5. Return directory results (source='directory') or [] (source='not_found')
```

### STORIES_ROOT Configuration

Dev must define how the directory fallback locates story directories. Recommended implementation: `process.env.STORIES_ROOT ?? detectMonorepoRoot()` with a documented default. This must be testable (injectable in unit tests via ShimOptions, real path in integration tests).

```typescript
// In __types__/index.ts
const ShimOptionsSchema = z.object({
  storiesRoot: z.string().optional(),  // Injectable for testing; falls back to STORIES_ROOT env var
})

type ShimOptions = z.infer<typeof ShimOptionsSchema>
```

### Directory Scan Strategy

The fallback uses the simplest approach sufficient for correctness: given a `storyId`, locate any directory named `{storyId}/` under the stories root, and infer state from the parent directory's name using the swim-lane mapping. This avoids parsing markdown frontmatter and is resilient to formatting variations.

For `shimGetStoriesByStatus(state)` / `shimGetStoriesByFeature(epic)`, the scan lists directories under the corresponding swim-lane directory (for status) or epic directory (for feature).

**Known gap (documented, not blocking):** `blocked` and `cancelled` are valid `storyStateEnum` values but have no directory swim-lane equivalent. Stories in these states will return `not_found` from directory fallback — correct behavior for Phase 1 since these states are DB-only by design.

### Designed for Deletion

- No external packages import `story-compatibility/` directly — only through the `@repo/mcp-tools` named export
- The delete checklist for WINT-7100 will be: (1) remove `src/story-compatibility/` directory, (2) remove 4 exports from `src/index.ts`, (3) update WINT-1040/1050/1060 importers to call `storyGetStatus` etc. directly
- No database schema changes are required for the shim — it uses existing tables only

## Risks / Edge Cases

1. **WINT-0090 not formally closed**: In UAT status. Implementation can begin; final merge should coordinate with WINT-0090 formal close to ensure function signatures are stable.

2. **blocked/cancelled state gap**: storyStateEnum defines 8 states but swim-lane mapping covers only 6. Stories in blocked/cancelled state with no corresponding directory will return `not_found` from directory fallback. This is correct behavior for Phase 1 (these states are DB-only). Document in `__types__/index.ts` as code comment.

3. **No retry logic on transient DB failures**: One connection error causes immediate fallback to potentially stale directory data. Evaluate single-retry with short backoff in a follow-on story (deferred to Phase 2 when telemetry reveals transient failure frequency).

4. **Directory depth assumption**: Scan assumes story directories are exactly one level under swim-lane directory. Add guard: only directories matching `/^[A-Z]{2,10}-\d{3,4}$/` treated as story directories.
