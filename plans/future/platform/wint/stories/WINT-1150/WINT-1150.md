# WINT-1150: Integrate Worktree Cleanup into Story Completion

## Context

WINT-1130 introduced a `wint.worktrees` table and four MCP tools
(`worktree_register`, `worktree_get_by_story`, `worktree_list_active`,
`worktree_mark_complete`) to track worktree lifecycle in the database. Separately, eight
worktree skills (`wt-new`, `wt-finish`, `wt-status`, etc.) handle git operations but
currently have no database integration.

When a story completes QA (`qa-verify-story` returns PASS), the worktree used for
development is left alive. Developers must manually run `/wt-finish` to merge the
branch, push, clean up the worktree, and update the database record. This manual step is
frequently forgotten or deferred, leading to worktree sprawl and stale branches
accumulating over time. The same gap exists when a story is manually marked `completed`
via `story-update`.

WINT-1140 handles the story start side (worktree creation on `dev-implement-story`).
This story handles the story end side: automatic cleanup on story completion.

**Dependency status**: WINT-1130 is currently in UAT (PASS verdict as of 2026-02-16).
This story may be elaborated now; implementation begins after WINT-1130 is fully
deployed.

---

## Goal

Automatically trigger worktree cleanup when a story completes QA or is manually marked
done. Prevent worktree sprawl and stale branches. If cleanup cannot proceed safely (CI
failing, PR changes requested), defer gracefully and surface an actionable warning
without blocking story completion.

---

## Non-Goals

- Implementing the `wt-finish` skill itself (it already exists)
- Adding auto-merge functionality (wt-finish handles merge decisions)
- Auto-abandoning worktrees on timeout (session timeout detection, deferred to future)
- Batch worktree cleanup (WINT-1170)
- Worktree conflict UI (WINT-1160)
- Integration with dev-implement-story (WINT-1140 handles story start)
- Changes to the `wint.worktrees` database schema (defined in WINT-1130)
- Modification of existing worktree skills beyond invocation

## Acceptance Criteria

- [ ] **AC-1**: When `qa-verify-story` produces a PASS verdict, `qa-verify-completion-leader` calls `worktree_get_by_story` with the story's UUID to check for an active worktree
- [ ] **AC-2**: If no active worktree record exists, the PASS flow continues unchanged (graceful no-op, no warning)
- [ ] **AC-3**: If an active worktree exists, `qa-verify-completion-leader` invokes the `/wt-finish` skill with the worktree's `branchName` and `worktreePath`
- [ ] **AC-4**: If `/wt-finish` succeeds (branch merged, pushed, worktree cleaned up), `worktree_mark_complete` is called with `status: 'merged'`
- [ ] **AC-5**: If `/wt-finish` fails because CI checks are failing, cleanup is deferred and the worktree metadata is updated with `{ cleanup_deferred: true, reason: "ci_failing" }`
- [ ] **AC-6**: If `/wt-finish` fails because the PR has requested changes, cleanup is deferred and metadata updated with `{ cleanup_deferred: true, reason: "pr_review_pending" }`
- [ ] **AC-7**: If cleanup is deferred for any reason, the PASS flow continues and a visible actionable warning is logged: `"WARNING: Worktree '{branchName}' at '{worktreePath}' was not cleaned up. Reason: {reason}. Action: Run /wt-finish {storyId} when ready."`
- [ ] **AC-8**: The `story-update` command triggers the same worktree cleanup check when transitioning a story to `completed` status (manual completion path)
- [ ] **AC-9**: All new code paths use Zod schemas for input/output types (no TypeScript interfaces, per CLAUDE.md)
- [ ] **AC-10**: Tests cover the four critical scenarios: (a) no worktree found (no-op), (b) worktree found + cleanup succeeds, (c) worktree found + CI failing (defer), (d) worktree found + PR changes requested (defer)
- [ ] **AC-11**: Tests verify that story completion (PASS flow and manual `completed` transition) is never blocked if `wt-finish` encounters any error, including timeouts or unexpected failures
- [ ] **AC-12**: If `wt-finish` does not expose machine-readable CI/PR failure reasons (discovered in pre-work), any `wt-finish` failure is treated as `reason: "unknown"` and deferred uniformly — AC-5 and AC-6 specific reasons are only populated if wt-finish returns structured output; the implementation documents this discovery in DECISIONS.yaml before coding
  _Added by autonomous elaboration_
- [ ] **AC-13**: The deferred cleanup path calls `worktree_mark_complete` with `{ status: 'abandoned', metadata: { cleanup_deferred: true, reason: '<reason>' } }` — using `status: 'abandoned'` (not a new status value) to ensure the worktree is no longer listed as active while deferral metadata is preserved
  _Added by autonomous elaboration_

## Architecture Notes

### Cleanup Decision Flow

```
qa-verify PASS (or story-update → completed)
  │
  ▼
worktree_get_by_story(storyId)
  │
  ├─ null / not found ──► skip silently, continue PASS flow
  │
  └─ active worktree found
       │
       ▼
     /wt-finish branchName worktreePath
       │
       ├─ success ──► worktree_mark_complete(status: 'merged') ──► continue PASS flow
       │
       └─ failure
            │
            ├─ reason: ci_failing ──► defer + warn + continue PASS flow
            ├─ reason: pr_review_pending ──► defer + warn + continue PASS flow
            ├─ reason: user_requested ──► defer + warn + continue PASS flow
            └─ reason: unknown/error ──► defer + warn + continue PASS flow (AC-11)
```

### Key Constraint: Haiku Agent Budget

`qa-verify-completion-leader` is haiku-powered. The cleanup logic must remain a
mechanical 3-step conditional (check → invoke → handle). Complex CI/PR status reasoning
must be delegated entirely to the `wt-finish` skill, which surfaces structured failure
reasons.

### Pre-work Required

Before implementing AC-5/AC-6, verify what structured output `wt-finish` currently
returns. If it does not expose `ci_failing` and `pr_review_pending` as machine-readable
failure reasons, adding that output is a prerequisite task (1-2 additional points).
Document finding in DECISIONS.yaml at implementation start.

### Metadata Deferral Format

```json
{
  "cleanup_deferred": true,
  "reason": "ci_failing | pr_review_pending | user_requested | unknown"
}
```

Stored in the `metadata` JSONB column of `wint.worktrees` via `worktree_mark_complete`
or a metadata-update MCP tool call.
