# WINT-9060: Create batch-coordinator LangGraph Graph (`batch-process.ts`)

## Context

The existing `/workflow-batch` Claude Code command orchestrates batch story processing through an agent-based leader pattern. It wraps `/dev-implement-story` with batch mode enabled, queuing decisions by tier, presenting batch reviews at thresholds, and writing `BATCH-DECISIONS.yaml` / `DEFERRED-BACKLOG.yaml` artifacts.

WINT-9060 ports this agent logic to a native LangGraph graph at `packages/backend/orchestrator/src/graphs/batch-process.ts`. The graph orchestrates a queue of stories through the implementation pipeline, handling per-story failures and retries identically to the Claude Code agent — so batch processing works in both execution contexts with equivalent behavior.

The codebase already provides all patterns needed: Send API fan-out in `elab-epic.ts`, injectable stubs for pending dependencies in `backlog-review.ts`, and bounded retry loops in `implementation.ts`. This story composes those patterns into a new batch coordinator graph.

**Dependencies:** WINT-9020 (doc-sync, completed), WINT-9030 (cohesion-prosecutor, pending), WINT-9040 (scope-defender, created), WINT-9050 (evidence-judge, pending). The three pending dependencies are handled via injectable stubs — graph compiles and runs without them.

---

## Goal

Create a LangGraph graph (`batch-process.ts`) that orchestrates a caller-provided list of story IDs through the implementation pipeline in parallel, collecting per-story results, retrying transient failures, and returning a structured batch summary — so batch story processing works natively in the LangGraph orchestrator without requiring the Claude Code agent layer.

## Examples

**Example 1 — Happy path: 3 stories, all succeed**

Given:
```typescript
await runBatchProcess({
  batchId: 'batch-2026-03-08',
  storyIds: ['WINT-4060', 'WINT-4070', 'WINT-4080'],
  config: { maxConcurrency: 3, maxRetriesPerStory: 2, autonomyLevel: 'moderate' },
})
```

The system produces:
```typescript
{
  batchId: 'batch-2026-03-08',
  success: true,
  storiesQueued: 3,
  storiesSucceeded: 3,
  storiesFailed: 0,
  storiesRetried: 0,
  workerResults: [
    { storyId: 'WINT-4060', success: true, retryCount: 0, errors: [] },
    { storyId: 'WINT-4070', success: true, retryCount: 0, errors: [] },
    { storyId: 'WINT-4080', success: true, retryCount: 0, errors: [] },
  ],
  durationMs: 42000,
  completedAt: '2026-03-08T10:05:42.000Z',
  errors: [],
  warnings: [],
}
```

**Example 2 — One story fails, retries once then succeeds**

Given: `storyIds: ['WINT-4090']` where the sub-pipeline throws on attempt 1 but succeeds on attempt 2; `maxRetriesPerStory: 2`

The system produces:
```typescript
{
  storiesQueued: 1,
  storiesSucceeded: 1,
  storiesFailed: 0,
  storiesRetried: 1,
  workerResults: [{ storyId: 'WINT-4090', success: true, retryCount: 1, errors: [] }],
}
```

## Edge Cases

**Edge Case 1 — Injectable nodes absent (WINT-9030/9040/9050 not yet implemented)**

When `cohesionProsecutorNode`, `scopeDefenderNode`, and `evidenceJudgeNode` are not provided in config, the graph skips those nodes with a warning log and continues. The batch result includes warnings like `"cohesion-prosecutor not configured, skipping"` but does not set `success: false` due to missing stubs alone.

**Edge Case 2 — All stories fail after exhausting retries**

When every story in the batch fails `maxRetriesPerStory` times, the graph sets `success: false`, `storiesFailed: N`, and `errors[]` contains per-story failure messages. The graph completes normally (does not throw) — the caller inspects the result to determine partial or full failure.

**Edge Case 3 — Empty storyIds input**

When `storyIds: []`, the dispatcher returns no Send objects and routes directly to `fan_in`. The result is `{ success: true, storiesQueued: 0, storiesSucceeded: 0, storiesFailed: 0 }` with no errors.

## Architecture Notes

**Graph structure:**
```
START --[createBatchProcessDispatcher (Send API)]-->
  story_worker (×N, parallel) -->
    fan_in -->
      complete --> END
                abort --> END (on unrecoverable state)
```

**Key design decisions:**
1. **Dispatcher as conditional edge, not node** — `createBatchProcessDispatcher` returns `Send[]` and is wired via `.addConditionalEdges(START, ...)`. Adding it as a `.addNode()` would break the Send API pattern.
2. **Separate reducers for aggregate vs. array fields** — `overwrite` for scalar fields (storiesSucceeded, storiesFailed), `append` for `workerResults[]` and `errors[]` / `warnings[]`. Never use `overwrite` on a field that multiple parallel workers write.
3. **Injectable stubs are config, not imports** — `cohesionProsecutorNode`, `scopeDefenderNode`, `evidenceJudgeNode` come through `BatchProcessConfigSchema` as `z.unknown().optional()`. Graph nodes check `if (config.cohesionProsecutorNode)` before invoking — skip with warning if absent.
4. **Per-story retryCount in worker state** — Worker nodes carry their own `retryCount` field (not shared state) to avoid reducer conflicts across parallel workers. The fan-in node reads the `retryCount` from each `BatchWorkerResult` to compute `storiesRetried`.
