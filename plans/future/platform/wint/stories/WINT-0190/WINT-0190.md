# WINT-0190: Create Patch Queue Pattern and Schema

## Context

The WINT development workflow currently suffers from mega-patches — single commits that touch UI, API, database schema, tests, and refactors simultaneously. This creates verification failures (hard to confirm each layer is correct), review fatigue, and repair loops that cascade across all touched areas.

There is currently no machine-readable specification for what constitutes an acceptable patch. Agents have no enforced ordering constraint, repair loops are unbounded, and there are no per-patch limits on files changed or diff lines. WINT-0210 (Populate Role Pack Templates) is gated on this story: it cannot complete its Dev role pack (AC-1) or example validation (AC-7) without the artifacts produced here.

The canonical schema pattern for this project is established by `packages/backend/orchestrator/src/schemas/user-flows.schema.json` — the first and only project-authored schema. All structural decisions in this story must follow that pattern.

---

## Goal

Create a machine-readable `patch-plan.schema.json` with enforced constraints (patch ordering enum, max files per patch, max diff lines, repair loop guardrails) and document the Patch Queue and Repair Loop patterns with canonical examples. This unblocks WINT-0210 and establishes the vocabulary that WINT-3010 (Gatekeeper Sidecar) will eventually enforce at runtime.

## Architecture Notes

### JSON Schema Ordering Limitation

JSON Schema draft 2020-12 cannot enforce sequential ordering of enum values within an array. The `patch_type` enum defines the allowed values, but the schema cannot require that `types_schema` patches appear before `api` patches in the `patches` array.

**Design decision:** Ordering is a documented convention, not a schema constraint. The enum array order itself (`types_schema`, `api`, `ui`, `tests`, `cleanup`) communicates the intended sequence. Runtime enforcement is WINT-3010 scope.

### Per-Patch vs Schema-Level Limits

`max_files` and `max_diff_lines` are **per-patch instance fields** (not schema-level defaults) with `maximum` enforcement. This design:
- Makes limits explicit per patch step (developer sees the constraint at authoring time)
- Allows different patch types to have different limits in the future (e.g., a `cleanup` step might allow more files)
- Maintains consistency with how JSON Schema enforces numeric constraints

### `$defs` Reuse Pattern

Following `user-flows.schema.json`, all reusable sub-types go in `$defs`. `PatchStep` and `RepairLoop` are both defined in `$defs` and referenced via `$ref`. This enables future reference from WINT-3010 tooling and WINT-0210 role pack documentation.
