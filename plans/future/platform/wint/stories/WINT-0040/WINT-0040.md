# WINT-0040: Create Telemetry Tables â€” hitl_decisions and story_outcomes

## Context

WINT-0010 established the core database schemas for the wint platform, including the initial telemetry group: `agent_invocations`, `agent_decisions`, `agent_outcomes`, and `state_transitions`. A prior partial implementation (migration `0019_wint_0040_telemetry_columns.sql`) extended these base tables with token tracking, quality metrics, and audit enhancement columns.

Two critical telemetry tables are missing that are required for full observability:

1. **`hitl_decisions`**: Human-in-the-loop (HITL) decisions made during story review or QA phases need to be stored with semantic embeddings so similar past human decisions can be surfaced during future HITL episodes. The existing `agent_decisions` table captures agent-authored decisions; human operator decisions need their own table with pgvector support for semantic similarity retrieval.

2. **`story_outcomes`**: Per-story aggregate outcome records are needed to track the end-to-end result of an autonomous development cycle (quality score, total tokens, review iterations, final verdict). The existing `agent_outcomes` is per-invocation; `story_outcomes` is the story-level rollup needed for ML training data and pipeline health reporting.

Without these two tables, WINT-0120 (Telemetry MCP Tools) cannot be implemented, and the observability goal of the WINT epic cannot be met.

**Prior work note:** The base telemetry tables (`agent_invocations`, `agent_decisions`, `agent_outcomes`, `state_transitions`) are already created and extended. This story adds the two remaining tables only â€” it does NOT re-implement any existing work.

## Goal

Add `hitlDecisions` and `storyOutcomes` Drizzle table definitions to `packages/backend/database-schema/src/schema/wint.ts`, generate and apply the corresponding migration, and export the associated Zod schemas. This enables WINT-0120 to proceed and unlocks full telemetry observability for the autonomous pipeline.

## Non-Goals

- Do NOT seed data into `hitl_decisions` or `story_outcomes`.
- Do NOT modify `agentInvocations`, `agentDecisions`, `agentOutcomes`, or `stateTransitions` â€” those are complete.
- Do NOT add frontend components or API routes â€” pure database schema story.

- Do NOT introduce rollback migrations unless QA process reveals a need.
- Do NOT hand-write the migration SQL for Drizzle-managed tables â€” use `db:generate`.

## Architecture Notes

- **Database boundary**: wint schema tables live in `lego_dev` (port 5432), NOT the KB database (port 5433). This is ARCH-001, confirmed by lesson WINT-1120. Integration tests must use `DATABASE_URL=postgresql://postgres:postgres@localhost:5432/lego_dev`.
- **Drizzle enum compatibility**: If hitlDecisions or storyOutcomes queries filter on any enum-typed column, use `sql\`column::text\`` (raw SQL tag) rather than Drizzle's query builder enum handling â€” lesson APIP-2020.
- **High-volume write pattern**: Both tables are expected to be written frequently. Design for bulk insert, not row-by-row (lesson APIP-3020). Index strategy already accounts for this with high-cardinality-first composite indexes.
- **vector customType**: `wint.ts` currently does not import `vector`. Add `customType` to the drizzle-orm/pg-core import and define a `vector` customType function identical to the one in `apps/api/knowledge-base/src/db/schema.ts`.
- **ivfflat index**: Standard Drizzle index() builder cannot express `USING ivfflat WITH (lists=100)`. After running `db:generate`, manually append the ivfflat index statement to the migration SQL. This is established practice in the knowledge-base migrations.
- **Telemetry hook pre-wire** (lesson WINT-1060): Add JSDoc comment insertion points in both new tables indicating the expected write patterns for WINT-3070 (telemetry-log skill).
