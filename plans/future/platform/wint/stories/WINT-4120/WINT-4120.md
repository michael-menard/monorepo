# WINT-4120: Integrate Cohesion Checks into Workflow

## Context

The WINT Phase 4 adversarial subsystem has produced three detection/evaluation agents:

- `graph-checker` (WINT-4060): queries the feature cohesion graph and active rules, produces `graph-check-results.json`
- `cohesion-prosecutor` (WINT-4070): applies PO-layer reasoning to capability coverage, produces `prosecution-verdict.json`
- `evidence-judge` (WINT-4090): adversarially challenges QA evidence quality, produces `ac-verdict.json`

These agents exist as standalone workers but are not integrated into the automated workflow. Currently, cohesion enforcement is purely manual (the `/cohesion-check` command in WINT-4110 covers the on-demand case). Implementation and QA workflows (`dev-implement-story`, `qa-verify-story`) complete without any automated cohesion or evidence-quality check.

**Dependencies:** WINT-4050 (cohesion rules seed), WINT-4090 (evidence-judge). WINT-4070 (cohesion-prosecutor) is a practical dependency — design integration to degrade gracefully if `prosecution-verdict.json` is absent.

**Baseline:** Gaps exist between the 2026-02-13 baseline and current state — WINT-4010, WINT-4020, WINT-4050, WINT-4060, WINT-4070, and WINT-4090 were all completed after the baseline. Context retrieved directly from story files.

---

## Goal

Automate cohesion and evidence-quality enforcement by adding Phase 2.5 (cohesion check) to `dev-implement-story` and Phase 1.5 (evidence-judge) to `qa-verify-story`, so that every story passes through the Phase 4 adversarial layer as part of the standard workflow, with all failures surfaced as advisories rather than hard blocks in v1.0.

---

## Examples

**Example 1 — Happy path: dev-implement-story with cohesion violations surfaced as advisory**

Given: A story for epic `user-authentication` completes Phase 2 execution. The feature graph has a franken-feature violation (missing `delete` capability). WINT-4030 has run and `wint.rules` has active rules.

Phase 2.5 spawns graph-checker → `graph-check-results.json` is written to `{story_dir}/_implementation/` with 1 violation of severity `warning`. Cohesion-prosecutor spawns → `prosecution-verdict.json` written with `verdict: INCOMPLETE-BLOCKED`. Dev-implement-story surfaces an advisory note in the review artifacts: "Cohesion advisory: feature `user-authentication` missing delete capability." Workflow continues to review phase without emitting FAILED.

**Example 2 — Happy path: qa-verify-story with evidence-judge FAIL surfaced as QA issue**

Given: A story's EVIDENCE.yaml has been written by the verification phase. Evidence-judge evaluates it and determines coverage is insufficient.

Phase 1.5 spawns evidence-judge → `ac-verdict.json` written with `overall_verdict: FAIL`. QA completion phase reads the verdict and includes a QA issue in its report: "Evidence quality FAIL: AC coverage insufficient per evidence-judge." Story is flagged for rework, not silently passed.

---

## Edge Cases

**Edge Case 1 — Empty graph (WINT-4030 not run, pre-Phase-4 story)**

`graph_get_franken_features` returns `[]`. Graph-checker emits `GRAPH-CHECKER COMPLETE WITH WARNINGS: 1 warnings`. Dev-implement-story logs warning `"graph.features empty — WINT-4030 may not have run"` and continues to cohesion-prosecutor spawn. Cohesion-prosecutor receives empty graph data, finds no violations, and produces a clean `prosecution-verdict.json` (no advisory surfaced). Story completes normally with no regression to existing behavior.

**Edge Case 2 — graph-checker BLOCKED, evidence-judge BLOCKED**

If graph-checker emits `GRAPH-CHECKER BLOCKED: {reason}`, dev-implement-story logs `WARNING: cohesion check unavailable — {reason}` and continues. If evidence-judge emits `EVIDENCE-JUDGE BLOCKED: {reason}`, qa-verify-story logs `WARNING: evidence-judge unavailable — {reason}` and continues. Neither blocked state causes the story to fail.

**Edge Case 3 — cohesion-prosecutor absent (WINT-4070 not yet released)**

If cohesion-prosecutor spawn produces `PROSECUTION BLOCKED` or `prosecution-verdict.json` is absent, dev-implement-story logs a warning and does not add an advisory about prosecution verdict. Graph-checker output is still recorded. Workflow continues.

## Architecture Notes

**Phase insertion points:**

- `dev-implement-story`: New **Phase 2.5** block inserted after Phase 2 execution completes (before review phase). Two sequential spawns: graph-checker first, then cohesion-prosecutor (after COMPLETE signal). This mirrors the clean separation pattern established by existing phases.
- `qa-verify-story`: New **Phase 1.5** block inserted after Phase 1 verification-leader completes (before Phase 2 completion-leader). Evidence-judge reads EVIDENCE.yaml that the verification leader just wrote.

**Signal routing:**

| Signal | Routing |
|--------|---------|
| `GRAPH-CHECKER COMPLETE` | spawn cohesion-prosecutor |
| `GRAPH-CHECKER COMPLETE WITH WARNINGS: N` | spawn cohesion-prosecutor; log warning |
| `GRAPH-CHECKER BLOCKED` | log warning; skip to review phase (do not fail) |
| `PROSECUTION COMPLETE` | record verdict; continue to review |
| `PROSECUTION BLOCKED` | log warning; continue to review |
| `EVIDENCE-JUDGE COMPLETE` | read ac-verdict.json; route on overall_verdict |
| `EVIDENCE-JUDGE BLOCKED` | log warning; continue to completion |

**Verdict routing for evidence-judge:**

| overall_verdict | Action |
|----------------|--------|
| `PASS` | No impact; continue |
| `CHALLENGE` | Add advisory to QA report; continue |
| `FAIL` | Surface as QA issue in report; continue (advisory in v1.0) |

**Flag pattern:** Add `--skip-cohesion` flag to dev-implement-story following `--skip-worktree` pattern, for environments where Phase 4 agents are unavailable.
