# WINT-0120: Create Telemetry MCP Tools

## Context

The wint platform has three telemetry tables in the `wint` schema: `agent_invocations`, `hitl_decisions` (added by WINT-0040), and `story_outcomes` (added by WINT-0040). The Drizzle schemas and Zod insert/select schemas for all three are already defined and exported from `@repo/database-schema`. The MCP server (`apps/api/knowledge-base/src/mcp-server/`) already exposes 62 registered tools following a well-established pattern (schema registration in `tool-schemas.ts`, handler wiring in `tool-handlers.ts`).

Currently, agents have no MCP-accessible interface to write telemetry data. Logging agent invocations, HITL decisions, and story outcomes requires direct DB access â€” not available to agents operating via MCP. Similarly, agents cannot query telemetry for a story without a dedicated read tool.

**Dependency:** WINT-0040 must be merged and its migration (`0032_wint_0040_hitl_decisions_story_outcomes.sql`) applied before integration tests for `workflow_log_decision` and `workflow_log_outcome` can pass.

## Goal

Extend the MCP server with four new telemetry tools so agents can write agent invocation records, HITL decision records, and story outcome records, and read a telemetry summary for any story â€” enabling observability and ML training data collection across the wint pipeline.

## Examples

**Example 1 â€” Logging an agent invocation (happy path):**
Given an agent calls `workflow_log_invocation` with `{ invocationId: "inv-abc", agentName: "dev-leader", status: "success", storyId: "WINT-0120", inputTokens: 4200, outputTokens: 1100 }`, the system inserts a row into `wint.agent_invocations` and returns `{ id: "<uuid>" }`.

**Example 2 â€” Logging a story outcome with upsert:**
Given a story completes and calls `workflow_log_outcome` with `{ storyId: "WINT-0120", finalVerdict: "pass", qualityScore: 88 }`, and a prior outcome row already exists for that `storyId`, the system updates the existing row (ON CONFLICT DO UPDATE) and returns the existing row's `{ id: "<uuid>" }` â€” ensuring exactly one outcome row per story.

**Example 3 â€” Reading story telemetry:**
Given calls have been made to log 3 invocations and 1 decision for `WINT-0120`, `workflow_get_story_telemetry({ storyId: "WINT-0120" })` returns `{ invocations: [<3 items>], decisions: [<1 item>], outcome: { finalVerdict: "pass", qualityScore: 88 } }`.

## Edge Cases

**Edge Case 1 â€” Unknown storyId on read:**
When `workflow_get_story_telemetry` is called with a `storyId` that has no records in any telemetry table, the tool returns `{ invocations: [], decisions: [], outcome: null }` â€” no error, no exception.

**Edge Case 2 â€” Partial telemetry (invocations but no outcome):**
When a story has invocations logged but no outcome row yet, `workflow_get_story_telemetry` returns `{ invocations: [<n>], decisions: [], outcome: null }`. The missing outcome is represented as `null`, not an error.

**Edge Case 3 â€” Invalid input rejected by Zod:**
When `workflow_log_invocation` is called with missing required fields (`invocationId` or `status`), the tool returns an MCP error result via `errorToToolResult` â€” no unhandled exception, no partial DB write.

**Edge Case 4 â€” Tool count assertion drift:**
If concurrent stories have added tools since WINT-0120's seed was written, the count in `mcp-integration.test.ts` must be verified against the actual `getToolDefinitions().length` at implementation start â€” not assumed to be 62+4=66.

## Non-Goals

- Embedding generation for `hitlDecisions.embedding` â€” WINT-3040 handles this in Phase 3
- Batching or async queue for high-frequency writes â€” Phase 3 telemetry-log skill (WINT-3010/3020) handles volume; these MCP tools are synchronous single-write
- UI/frontend for telemetry display â€” deferred to WINT-3060
- Modifying any existing telemetry table schema (DDL is frozen post-WINT-0040)
- Writing to the `wint.agent_decisions` table (separate from HITL `hitl_decisions`)
- Adding telemetry tools to `packages/backend/mcp-tools/` â€” they belong in the knowledge-base MCP server which has direct DB access

## Architecture Notes

- **No HTTP calls from handlers** â€” all four tools write/read directly to the Aurora DB via the Drizzle `db` client. No sidecar or Lambda intermediary. (ADR per WINT-4010 lesson)
- **Fire-and-forget intent** â€” log tools (`workflow_log_invocation`, `workflow_log_decision`, `workflow_log_outcome`) are designed to be non-blocking on write errors: catch exceptions, log warning, return graceful error result rather than propagating.
- **Zod-first types** â€” define all input schemas with `z.object(...)`, validate at tool entry with `Schema.parse(input)`, never use TypeScript interfaces.
- **No barrel file imports** â€” import directly from `packages/backend/database-schema/src/schema/wint.ts`.
- **Promise.all for telemetry read** â€” `workflow_get_story_telemetry` should issue all three DB queries in parallel for performance.

## Infrastructure Notes

No infrastructure changes. The knowledge-base MCP server Lambda already has DB access. No new permissions, tables, or migrations required â€” WINT-0040 provides the tables.
