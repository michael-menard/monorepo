# WINT-9120: Create Workflow Parity Test Suite

Prove Functional Equivalence Between Claude Code and LangGraph Execution Paths

---

## Context

The WINT Phase 9 initiative is porting all major Claude Code workflow agents to LangGraph graphs. As of 2026-03-02:

- **WINT-9020** (doc-sync LangGraph node): completed — `packages/backend/orchestrator/src/nodes/sync/doc-sync.ts`
- **WINT-9090** (context cache nodes): completed — `packages/backend/orchestrator/src/nodes/context/`
- **WINT-9010** (shared business logic via `@repo/workflow-logic`): in-qa — types may still shift
- **WINT-9110** (full workflow LangGraph graphs: bootstrap, elab-epic, elab-story, dev-implement, qa-verify, backlog-review): created, not started — depends on WINT-9060, WINT-9070, WINT-9080, WINT-9100

Before the team can confidently cut over from Claude Code orchestration to LangGraph orchestration, a mechanically verifiable proof is required that both execution paths produce identical outputs for identical inputs. Without this, the cutover is a leap of faith. Subtle behavioral differences — different artifact field ordering, missing fields, different retry semantics — could silently break workflows in production.

**Baseline**: BASELINE-REALITY-2026-02-13.md (predates Phase 9 activity; does not capture WINT-9020 completion or WINT-9010 in-qa state).

---

## Goal

Build a parity test suite at `packages/backend/orchestrator/src/__tests__/parity/` that acts as a dual-execution harness. For each workflow, the suite runs the same fixture inputs through both the Claude Code agent path and the LangGraph graph path, then compares outputs using shared Zod artifact schemas. The suite serves as the technical gate for cutover sign-off.

---

## Non-Goals

- Do not implement WINT-9110 workflow graphs — this story only tests them once they exist
- Do not build a live/production parity monitor (separate telemetry story)
- Do not write Playwright/E2E parity tests — this suite is unit/integration level per ADR-005
- Do not modify existing graph implementations — if a graph is not injectable, fix it in the graph's own story scope
- Do not test LLM output content equality — only structural/schema equivalence and state transition parity
- Do not modify production DB schemas, `@repo/db` client, or orchestrator artifact schemas (protected)

## Architecture Notes

**Parity harness design:**
```typescript
// parity-harness.ts — core contract
const ParityResultSchema = z.object({
  verdict: z.enum(['matching', 'divergent', 'error']),
  diff: z.array(z.object({
    fieldPath: z.string(),          // Zod schema path notation
    claudeCodeValue: z.unknown(),
    langGraphValue: z.unknown(),
  })),
  knownDivergences: z.array(z.string()).optional(),
  error: z.object({
    source: z.enum(['claudeCode', 'langGraph']),
    message: z.string(),
  }).optional(),
})
type ParityResult = z.infer<typeof ParityResultSchema>
```

**Graph compilation test pattern** (from existing graph tests):
```typescript
// Graph compile test — fast, no AI calls
const graph = buildStoryCreationGraph({ toolRunner: mockToolRunner })
const compiled = graph.compile()
expect(compiled).toBeDefined()
// Verify routing: inject state, check which node is visited next
```

**Dependency injection for determinism:**
```typescript
// Do NOT call real AI — inject mock toolRunner
const mockToolRunner = vi.fn().mockResolvedValue(fixtureResponse)
const runner = createToolNode({ toolRunner: mockToolRunner })
```

**Timestamp determinism:**
```typescript
beforeEach(() => vi.useFakeTimers())
afterEach(() => vi.useRealTimers())
```

**Known divergence pattern:**
```typescript
it('documents known timestamp format difference', () => {
  const result = runParity(...)
  expect(result.verdict).toBe('divergent')
  expect(result.knownDivergences).toContain('timestamp-format')
  // Test PASSES because divergence is documented and expected
})
```

## Test Plan

**Strategy**: Integration (mocks permitted per ADR-005; parity suite is unit/integration level)

**Coverage target**: >= 80% branch coverage within `packages/backend/orchestrator/src/__tests__/parity/`

**Definition of parity PASS:**
- Both paths parse successfully through the shared Zod artifact schema (structural equivalence)
- All required fields present in both outputs
- Optional fields treated symmetrically (absent in both = matching; absent in one = divergent)
- No real network calls made during test execution
- Verdict is `matching` or test is marked as `knownDivergences`

**Non-determinism handling:**
- AI/LLM responses: injected as fixtures via mockToolRunner (no real AI calls)
- Timestamps: pinned via `vi.useFakeTimers()` before each test
- Random routing: inject deterministic routing functions where graph uses `Math.random()`

**Workflows requiring parity coverage:**
1. doc-sync (WINT-9020 completed — test immediately)
2. story-creation (existing graph — test immediately)
3. elaboration (existing graph — test immediately)
4. bootstrap, elab-epic, elab-story, dev-implement, qa-verify, backlog-review (WINT-9110 — test when ready-for-qa)

See `_pm/test-plan.yaml` for full test case specifications.
