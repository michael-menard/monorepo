# WINT-2010: Create Role Pack Sidecar Service

## Context

The autonomous development workflow spawns multiple specialized agents (pm, dev, qa, po) for each story. Currently, each spawn begins by re-reading long agent instruction files, wasting tokens re-teaching agents their role. WINT-0210 is creating compact 150–300 token role packs in `.claude/prompts/role-packs/`. WINT-2010 exposes those packs as a versioned HTTP sidecar service with an MCP tool wrapper so agents can retrieve their role instructions with a single tool call.

WINT-1120 (Validate Foundation Phase) has confirmed the DB story CRUD, shim fallback, and worktree integration are solid — this story proceeds with Phase 2 sidecar work. No existing sidecar infrastructure exists at `packages/backend/sidecars/` — this package is built from scratch.

**Content dependency**: WINT-0210 authors the role pack files. As of 2026-03-02, WINT-0210 is `ready-to-work` but not yet completed. The sidecar must be built with stub/fixture role pack files and documented to switch to WINT-0210 output at merge time.

## Goal

Stop re-teaching agents their job every spawn. Provide a versioned, cacheable, single-endpoint sidecar service that returns compact (150–300 token) role pack instructions for pm, dev, qa, and po agents.

## Non-Goals

- **Deployment to AWS Lambda** — local sidecar only, not a production Lambda function
- **Dynamic role discovery** — only the 4 explicitly enumerated roles (pm, dev, qa, po) are supported
- **Hot-reload of role pack files** — file cache is invalidated on process restart only (v1 scope)
- **Content authoring** — role pack text is authored by WINT-0210, not this story
- **Authentication/authorization** — internal-only sidecar, no auth required for v1
- **Database storage of role packs** — filesystem is the source of truth; no new DB tables
- **Multi-language or customized role packs** — future ML story scope
- **Frontend serving** — agent-facing sidecar only

## Scope

**New Package:** `packages/backend/sidecars/role-pack/` (name: `@repo/sidecar-role-pack`)

**HTTP Endpoint:**
- `GET /role-pack?role=pm|dev|qa|po&v=X` → `200 { content: "..." }` on success
- Unknown `role` → `404 { error: "Unknown role" }`
- Version specified but unavailable → `404 { error: "Version not found", available: N }`
- Missing `role` query param → `400 { error: "role is required" }`
- Content-Type: `application/json`
- Internal path (not frontend-facing; ADR-001 path schema does not apply)

**MCP Tool:**
- `rolePackGet(input: { role: RoleSchema, version?: number }): Promise<string | null>`
- Zod parse at entry — throws ZodError on invalid input
- Returns content string on success, null on miss

**Package Structure:**

## Architecture Notes

- **Resilient null-return contract**: File reader and MCP tool return `null` on miss, never throw. HTTP layer converts null to 404. This mirrors the established WINT-0100 context cache pattern.
- **Zod-first types**: No TypeScript interfaces anywhere — use `z.infer<typeof Schema>` exclusively per CLAUDE.md.
- **Strict role enum**: `z.enum(['dev', 'po', 'qa', 'da'])` at the Zod layer prevents dynamic role discovery and keeps the allowlist explicit. Roles match actual WINT-0210 output files.
- **Version in frontmatter**: Simple regex (`/^---\n([\s\S]*?)\n---/`) or `gray-matter` extracts `version: N` from the `.md` file. If frontmatter is malformed, log warning and return null version (treat as unversioned).
- **Internal HTTP path**: This sidecar is not frontend-facing. ADR-001 path schema (`/api/v2/...`) does not apply. Simple path `/role-pack` is sufficient.
- **Sidecar process lifecycle**: Runs as a standalone Node.js process. Port configurable via `PORT` env var. No clustering required for v1.

## Infrastructure Notes

- **Package location**: `packages/backend/sidecars/role-pack/` per index entry constraint
- **pnpm workspace**: **MUST add `packages/backend/sidecars/*`** to `pnpm-workspace.yaml` — existing `packages/backend/*` glob only matches one level deep and will NOT discover this package
- **turbo.json**: Verify new package is included in build/test pipeline; add to `packages/**` glob if not already covered

## HTTP Contract Plan

```
GET /role-pack?role={pm|dev|qa|po}&v={optional integer}

Success:
  HTTP 200
  Content-Type: application/json
  Body: { "content": "<role pack text, frontmatter stripped>" }

Invalid role:
  HTTP 404
  Body: { "error": "Unknown role" }

Version not found:
  HTTP 404
  Body: { "error": "Version not found", "available": <N> }

Missing role param:
  HTTP 400
  Body: { "error": "role is required" }

File read error / missing file:
  HTTP 404 (never 500)
  Body: { "error": "Role pack not available" }
  Side effect: logger.warn called with role and error details
```
