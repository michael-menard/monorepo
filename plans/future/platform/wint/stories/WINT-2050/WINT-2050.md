# WINT-2050: Populate Domain Knowledge Cache from ADR-LOG, KB Lessons, and Architecture Docs

## Context

The WINT platform's Phase 2 goal is 80% token reduction across agent invocations. The `wint.context_packs` table (deployed WINT-0010, protected) is the storage layer. WINT-2030 populated project conventions and tech-stack packs. WINT-2040 (in progress) populates agent mission packs.

This story closes the domain knowledge gap: ADRs, lessons learned, blockers, and area-specific patterns from `plans/stories/ADR-LOG.md` and the KB (`knowledgeEntries` pgvector store at port 5433) are currently re-read or re-queried on every agent invocation. These are stable documents that change infrequently, making them ideal candidates for cache population.

**Key architectural complexity**: Unlike WINT-2030, which read only from the filesystem, this story requires dual-database access — reading from the KB pgvector store (port 5433) and writing to lego_dev (port 5432). This is the primary implementation challenge.

**Dependencies resolved**: Actual dependencies are WINT-0010 (table) and WINT-0100 (MCP tools). WINT-2020 and WINT-0030 listed in the original index are not blockers for this story.

## Goal

Create a TypeScript populate script at `packages/backend/mcp-tools/src/scripts/populate-domain-kb.ts` that reads `plans/stories/ADR-LOG.md` and queries the KB for lessons/blockers, then writes discrete, bounded, structured cache entries to `wint.context_packs` via `contextCachePut()`. The script must be idempotent, resilient, and follow the exact same architecture as `populate-project-context.ts`.

## Non-Goals

- Do not add new `contextPackTypeEnum` values to the database schema
- Do not modify `wint.context_packs` table schema (protected by WINT-0010)
- Do not implement the context-pack sidecar HTTP endpoint (WINT-2020 scope)
- Do not implement cache warming scheduling (WINT-2070/WINT-2080 scope)
- Do not cache library patterns (React 19, Vitest API) — WINT-2060 scope
- Do not cache project conventions or tech-stack docs — already done by WINT-2030
- Do not cache agent missions — WINT-2040 scope
- Do not implement automatic doc-change triggers — manual re-population is MVP
- Do not implement a UI or CLI command — script is a Node.js utility only
- Do not store entire raw KB query result sets — always extract bounded structured content
- Do not modify the KB database schema or `knowledgeEntries` table

- [ ] **AC-9**: KB query path is injectable — the KB search function is a parameter with a default implementation, not hardcoded. Unit tests pass a mock `kbQueryFn`; integration tests use the real KB or a mock fixture if KB is not available in CI.

- [ ] **AC-10**: All pack types used are from the existing `contextPackTypeEnum`: `'lessons_learned'` and `'architecture'` only. No new enum values are added.

## Architecture Notes

### Dual-Database Access Pattern

```
Script
  ├── KB read (port 5433):  Drizzle connection via KB_DATABASE_URL
  │     └── knowledgeEntries table
  │           ├── query: lesson-learned tagged entries → group by domain area
  │           └── query: constraint/blocker entries → consolidated blockers pack
  │
  └── lego_dev write (port 5432): contextCachePut() via DATABASE_URL
        └── wint.context_packs (upsert on packType + packKey)
```

### Pack Taxonomy

| packType | packKey | Source | Content Shape |
|----------|---------|--------|---------------|
| `architecture` | `active-adrs` | plans/stories/ADR-LOG.md | `{ summary, adrs: [{id, title, constraint, status}] }` |
| `lessons_learned` | `lessons-backend` | KB query (tag: backend) | `{ summary, lessons: [{title, description, tags}] }` |
| `lessons_learned` | `lessons-frontend` | KB query (tag: frontend) | `{ summary, lessons: [{title, description, tags}] }` |
| `lessons_learned` | `lessons-testing` | KB query (tag: testing) | `{ summary, lessons: [{title, description, tags}] }` |
| `lessons_learned` | `lessons-workflow` | KB query (tag: workflow) | `{ summary, lessons: [{title, description, tags}] }` |
| `lessons_learned` | `blockers-known` | KB query (type: constraint) | `{ summary, blockers: [{title, description}] }` |
