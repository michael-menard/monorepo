# WINT-0100: Create Context Cache MCP Tools

## Context

The WINT autonomous development workflow uses the `wint.contextPacks` table (deployed in WINT-0010) to cache frequently-used context (codebase summaries, architecture decisions, lessons learned, test patterns) and reduce token consumption across agent invocations. The table stores JSONB content with versioning, TTL expiration, and hit tracking.

However, no MCP tools currently exist to:
1. Store context packs when agents generate reusable context
2. Retrieve context packs to inject into agent prompts
3. Invalidate stale context when dependencies change
4. Query cache hit rates for optimization analysis

This blocks context cache population (WINT-2030-2060), cache warming strategies (WINT-2070), and token reduction efforts (target: 80% reduction per WINT-2120).

**Reality Baseline** (as of 2026-02-13):
- `wint.contextPacks` table exists and is deployed (WINT-0010 protected feature)
- `wint.contextSessions` table exists (WINT-0010)
- Session Management MCP Tools completed (WINT-0110) providing pattern template
- Drizzle ORM v0.44.3 active with auto-generated Zod schemas via drizzle-zod
- @repo/db package provides database connection pooling and schema exports
- Zod-first validation patterns established per CLAUDE.md
- No active overlapping work detected

## Goal

Create 4 MCP tools that provide type-safe, validated access to context cache operations:

1. **context_cache_get** - Retrieve cached context by pack type and key
2. **context_cache_put** - Create or update context pack with content
3. **context_cache_invalidate** - Mark context as expired or delete stale entries
4. **context_cache_stats** - Query cache hit rates and effectiveness metrics

Following the established pattern from WINT-0110 (Session Management MCP Tools):
- Zod schemas for input validation
- Drizzle ORM for type-safe database access
- Resilient error handling (log warnings, return null on errors)
- Auto-generated types from drizzle-zod
- Comprehensive test coverage with real PostgreSQL database

## Acceptance Criteria

### AC-1: Implement context_cache_get Tool

**Input Schema**:
```typescript
{
  packType: string (min 1 char),
  packKey: string (min 1 char)
}
```

**Behavior**:
- Query `wint.contextPacks` for matching packType and packKey
- Filter out expired entries (expiresAt <= NOW())
- If found and valid: increment hitCount, update lastHitAt, return pack
- If not found or expired: return null
- On database error: log warning via @repo/logger, return null

**Output**: `SelectContextPack | null`

**Tests**:

- ✅ Cache miss (not found, returns null)
- ✅ Expired pack (expiresAt in past, returns null, hitCount NOT incremented)
- ✅ Hit count increment (concurrent access maintains accuracy)

## Architecture Notes

### Database Operations

**Upsert Pattern** (AC-2):
```typescript
await db.insert(contextPacks)
  .values({ packType, packKey, content, version, expiresAt })
  .onConflictDoUpdate({
    target: [contextPacks.packType, contextPacks.packKey],
    set: { content, version, expiresAt, updatedAt: sql`NOW()` }
  })
```

**Atomic Hit Count Increment** (AC-1):
```typescript
await db.update(contextPacks)
  .set({
    hitCount: sql`${contextPacks.hitCount} + 1`,
    lastHitAt: sql`NOW()`
  })
  .where(and(
    eq(contextPacks.packType, packType),
    eq(contextPacks.packKey, packKey)
  ))
```

**Expiration Filtering** (AC-1):
```typescript
await db.select()
  .from(contextPacks)
  .where(and(
    eq(contextPacks.packType, packType),
    eq(contextPacks.packKey, packKey),
    gt(contextPacks.expiresAt, sql`NOW()`)
  ))
```

## Decisions

### Decision 1: Default TTL Value

**Decision**: Default TTL = 7 days (604800 seconds)

**Rationale**: Balances context freshness vs storage efficiency. Most cached context (codebase summaries, architecture decisions, test patterns) remains valid for 1-2 weeks. Workflows can override via ttl parameter for ephemeral (1 day) or persistent (30 days) content.

**Added by**: Autonomous elaboration

### Decision 2: Content Field Validation Strategy

**Decision**: Use generic `z.record(z.unknown())` for content field validation in input schemas

**Rationale**: The actual database schema has typed JSONB structure (`{ summary?, files?, lessons?, architecture? }`), but MCP tools accept any valid JSONB. Input validation remains flexible while database enforces structure. Future story can add per-packType schema validation if needed.

**Implementation Note**: AC-2 and AC-5 specify `content: z.record(z.unknown())` which is correct for input validation. Database schema enforces specific structure at storage layer.

**Added by**: Autonomous elaboration

### Decision 3: Version Field Type

**Decision**: Use `version?: number` in Zod schemas to match database schema

**Rationale**: Database schema defines `version: integer` field. AC-2 incorrectly shows `version?: string`. Update Zod validation to use `z.number().int().optional()` for type consistency.

**Implementation Note**: AC-5 ContextCachePutInputSchema should use `version: z.number().int().optional()` instead of `z.string().optional()`.

**Added by**: Autonomous elaboration

### Aggregate Query Implementation (AC-4)

**Approach**: Use Drizzle `sql` template for PostgreSQL FILTER syntax

```typescript
const stats = await db.select({
  totalPacks: sql<number>`COUNT(*) FILTER (WHERE expires_at > NOW())`,
  hitCount: sql<number>`SUM(hit_count)`,
  avgHitsPerPack: sql<number>`COALESCE(AVG(hit_count), 0)`,
  expiredCount: sql<number>`COUNT(*) FILTER (WHERE expires_at <= NOW())`
}).from(contextPacks)
```

**Rationale**: FILTER clause provides cleaner separation of active vs expired counts than WHERE + CASE. COALESCE handles NULL from AVG when no rows exist.

### Filter Combination Logic (AC-3)

**Constraint**: If `packKey` provided, `packType` MUST also be provided

**Rationale**: Unique index on (packType, packKey) requires both for specific pack targeting. Querying by packKey alone is ambiguous.

**Implementation**: Validate in Zod schema or throw error if packKey without packType.
