# WINT-0131: Add Feature-Capability Linkage to WINT Schema

## Context

WINT-0130 (Create Graph Query MCP Tools) implemented 4 graph query MCP tools, but discovered at implementation time that the `wint.capabilities` table is missing a `featureId` foreign key column linking capabilities to features. This caused 2 of the 4 tools to operate as stubs:

- `graph_get_franken_features` â€” returns empty array `[]` with a schema-limitation warning
- `graph_get_capability_coverage` â€” returns `null` with a schema-limitation warning

The other 2 tools (`graph_check_cohesion` and `graph_apply_rules`) are fully functional and must not be modified.

The schema gap is: the `wint.capabilities` table has no FK to `wint.features`, making it impossible to compute CRUD completeness per feature (Franken-feature detection) or per-feature capability coverage.

**Priority rationale**: P0 because this blocks:
- WINT-4060 (Create graph-checker Agent) â€” the agent cannot detect Franken-features without functional tools
- WINT-4030 (Populate Graph with Existing Features) â€” coverage analysis is prerequisite

**Schema source of truth**: `wint.ts` (established by WINT-0130 decision_002). `unified-wint.ts` is a spike deliverable (WINT-1080) with known drift; do NOT update it in this story (deferred to WINT-1100).

**Next migration number**: 0027 (last applied migration is 0026: `0026_wint_1130_worktree_tracking`)

## Architecture Notes

### Schema Column Specification

```typescript
// In wint.ts capabilities table definition:
featureId: uuid('feature_id')
  .references(() => features.id, { onDelete: 'set null' })
```

The column must be:
- Nullable (`featureId` not `featureId.notNull()`)
- FK to `wint.features.id` with `ON DELETE SET NULL` (so deleting a feature does not delete its capabilities)
- Indexed for query performance

### Franken-Feature Detection Logic

A Franken-feature is any feature that has at least one capability linked (featureId is set) but does NOT have all 4 CRUD capability types (`create`, `read`, `update`, `delete`). Features with NO capabilities (all NULL featureId) are also Franken-features per the definition, but since existing seed data (WINT-0080) will have NULL featureId, this means the tool will return ALL features as Franken-features until WINT-4040 populates the featureId linkage. This is expected behavior and is documented.

### Dual-ID Support

Both tools that accept a `featureId` parameter support either:
- A UUID (queries `features.id`)
- A feature name string (queries `features.featureName`)

This pattern is already established in `graph-check-cohesion.ts` via `FeatureIdSchema` and the `or(eq(...), eq(...))` Drizzle query pattern.
