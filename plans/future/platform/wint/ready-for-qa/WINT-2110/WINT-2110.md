# WINT-2110: Update 5 High-Volume Agents to Query Context Cache

## Context

The WINT token-reduction initiative has produced four cache population stories (WINT-2030, 2040, 2050, 2060) that write structured context — project conventions, agent missions, domain knowledge, library patterns — into the `wint.context_packs` table. The `context_cache_get` MCP tool (WINT-0100) is UAT-verified and provides a validated retrieval interface.

The five highest-invocation agents in the workflow — `pm-bootstrap-workflow` (setup, analysis, generation leaders), `dev-implement-story` (implementation leader), `elab-story` (elab-analyst), `qa-verify-story` (verification leader), and `dev-fix-story` (fix leader) — currently populate their context windows exclusively through filesystem reads and `kb_search()` calls at invocation time. Each invocation re-reads the same source documents and re-queries the same KB patterns.

The `packTypeValues` Zod array in `packages/backend/mcp-tools/src/context-cache/__types__/index.ts` currently contains 7 pack types. The 4 cache population stories introduce additional pack types (agent_missions, project-conventions, blockers-known, lib-react19, lib-tailwind, lib-zod, lib-vitest) that must be registered before agents can query them.

This story integrates `context_cache_get()` calls into each of the 5 agent workflows so they retrieve pre-distilled context from the cache at startup. The integration is additive: existing `kb_search()` patterns remain; cache provides a fast-path for stable context with graceful fallback to direct reads when cache misses occur.

**AC-5 Audit Result (pre-seeded):** Orchestrator TypeScript nodes reference workflow names (elab-story, qa-verify-story, dev-implement-story) for state machine purposes but do not embed agent instruction content. The 5 target agents have no paired LangGraph TypeScript nodes requiring parallel updates. The AC-5 step should confirm this, then proceed.

## Goal

Integrate `context_cache_get()` calls into the 5 highest-invocation agent workflows, establishing the first measurable reduction in per-invocation token usage and providing the baseline for WINT-2120's 80% reduction target measurement.

## Non-Goals

- Do not replace or remove existing `kb_search()` calls — cache supplements KB, does not replace it
- Do not implement cache warming or population logic — that is WINT-2070/2080
- Do not measure final token reduction — that is WINT-2120
- Do not modify the `wint.context_packs` table schema — protected (WINT-0010)
- Do not modify the context_cache MCP tool implementations — use them as consumers only
- Do not add cache integration to agents outside the 5 listed — defer to a future story
- Do not create new cache pack types — consume what WINT-2030/2040/2050/2060 produce

## Architecture Notes

### Cache Integration Pattern (Per Agent)

Each agent's `## Context Cache Integration` section should follow this structure:

```
## Context Cache Integration

### When to Query

| Trigger | Pack Type | Pack Key | Purpose |
|---------|-----------|----------|---------|
| Workflow start | codebase | main | Project conventions, tech stack |
| Workflow start | architecture | main | ADRs, patterns |

### Call Pattern

context_cache_get({ packType: 'codebase', packKey: 'main' })
  → if null: log warning, continue without cache context
  → if hit: inject content.summary and content.conventions into agent context

### Content Injection Limits

- Inject: summary, conventions (first 5 entries), known_blockers
- Skip: raw_content, full_text, verbose_examples (unbounded size)
- Max injection: ~2000 tokens per pack

### Fallback Behavior

- Cache miss (null): Log "Cache miss for {packType}/{packKey} — proceeding without cache context" via @repo/logger. Continue agent execution.
- Tool error (exception): Catch, log warning, continue. Never block execution.
```

### Graceful Degradation Contract

Cache is a fast-path supplement — never a hard dependency:
- Cache miss → continue with KB search + filesystem reads (existing behavior)
- Cache unavailable → same as cache miss; log warning
- Cache content too large → inject a defined subset only; document the subset in each agent section
