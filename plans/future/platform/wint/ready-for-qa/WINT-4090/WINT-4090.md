# WINT-4090: Create evidence-judge Agent

## Context

The WINT Phase 4 elaboration workflow introduces an adversarial layer to prevent vibes-based approvals in QA. Three roles participate in this layer: Product Owner (cohesion-prosecutor), Devil's Advocate (scope-defender), and Evidence Judge (evidence-judge). The evidence-judge is the QA adversarial agent — a focused, haiku-powered worker that challenges AC verdicts that lack concrete, verifiable proof.

The current QA workflow (`qa-verify-verification-leader`) reads `EVIDENCE.yaml` and marks ACs as PASS or FAIL based on evidence items. However, it does not adversarially challenge the quality of that evidence. An implementer could produce `EVIDENCE.yaml` with vague entries like "implementation appears complete" or "code looks correct" — and the leader would accept them. The evidence-judge exists to reject exactly this.

The sister agent WINT-4080 (scope-defender) follows the same Phase 4 haiku worker pattern and serves as the primary structural template for this story.

---

## Goal

Create `.claude/agents/evidence-judge.agent.md` — a haiku-powered worker agent that evaluates the quality of QA evidence in `EVIDENCE.yaml`, challenges ACs with weak or absent proof, and produces a machine-readable `ac-verdict.json` for downstream consumption by WINT-4120 (workflow integration) and WINT-4140 (Round Table Agent).

### Evidence Strength Classification Rules

The classification rules must be unambiguous to enable a haiku-class model to apply them mechanically. Key design decisions:

| Evidence Type | STRONG Criteria | WEAK Criteria |
|--------------|-----------------|---------------|
| `test` | file path present AND passing_count > 0 | description-only; no file path; count 0 or missing |
| `command` | exact command + result is PASS/FAIL/numeric count | subjective result ("looks good", "no errors noticed") |
| `e2e` | file path present + result counts (passed/failed) | description-only; no file path |
| `http` | path + numeric HTTP status code | description-only; no status code |
| Any type | description must not contain: "appears", "seems", "should", "looks" | any of these subjective words in description → WEAK |

### CHALLENGE vs REJECT Boundary

These thresholds are binary rules, not judgment calls:
- **REJECT**: `evidence_items` is empty OR null → no evidence at all
- **CHALLENGE**: at least one evidence item present, but ALL items are WEAK → insufficient proof
- **ACCEPT**: at least one evidence item is STRONG → proof requirement satisfied
