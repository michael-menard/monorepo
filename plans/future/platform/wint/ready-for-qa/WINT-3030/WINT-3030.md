# WINT-3030: Create `telemetry-logger` Worker Agent — Haiku-Powered Fire-and-Forget Telemetry Invocation

## Context

The WINT Phase 3 telemetry infrastructure provides the plumbing for agent observability:
- **WINT-0040** — Telemetry tables (`wint.agent_invocations`, `wint.hitl_decisions`, `wint.story_outcomes`) — `ready-for-qa`
- **WINT-0120** — MCP tools (`workflow_log_invocation`, `workflow_log_decision`, `workflow_log_outcome`, `workflow_get_story_telemetry`) — `ready-to-work`
- **WINT-3010** — Gatekeeper Sidecar — `ready-for-qa` (direct dependency)
- **WINT-3020** — Invocation Logging Skill (`telemetry-log`) — `needs-code-review` (direct dependency)

Without a dedicated `telemetry-logger` agent, every caller that wants to log telemetry must inline the skill invocation pattern and its error-handling logic. This duplicates boilerplate, increases the risk of latency-inducing mistakes (e.g., accidentally awaiting telemetry before continuing), and makes the fire-and-forget contract invisible to downstream agents.

**Gating requirement**: WINT-3030 must not be started until WINT-3010 QA passes and WINT-3020 code review passes.

---

## Goal

Create a dedicated haiku-powered `telemetry-logger` worker agent that encapsulates the fire-and-forget telemetry invocation pattern, so any workflow orchestrator can log agent telemetry without inlining skill error-handling logic or risking latency.

---

## Examples

**Example 1 — Happy path telemetry log:**
Given a pm-story-generation-leader calls the telemetry-logger agent with payload `{ agentName: "pm-story-generation-leader", storyId: "WINT-3030", phase: "synthesis", status: "complete", inputTokens: 4200, outputTokens: 1800, cachedTokens: 0, durationMs: 12500, modelName: "claude-sonnet-4-6" }`, the telemetry-logger agent invokes the `telemetry-log` skill with those parameters, receives confirmation of DB write, and emits `TELEMETRY-LOGGER COMPLETE`.

**Example 2 — Fire-and-forget on skill unavailability:**
Given the `telemetry-log` skill returns null (WINT-3020 not yet deployed or MCP connection unavailable), the telemetry-logger agent logs a warning (`logger.warn: telemetry-log skill returned null for WINT-3030`) and emits `TELEMETRY-LOGGER COMPLETE WITH WARNINGS: 1` — it does NOT block the caller or propagate the error.

---

## Edge Cases

**Edge Case 1 — Skill returns null:**
The `telemetry-log` skill returns null (MCP tool unavailable, DB write failed, or WINT-3020 not deployed). Agent must emit `logger.warn` with storyId and agentName context, then emit completion signal. The calling workflow must never wait on this result.

**Edge Case 2 — Missing optional fields:**
Caller provides only required fields (agentName, storyId, phase, status) but omits optional fields (errorMessage, invocationId, cachedTokens). Agent passes available fields to skill; skill handles missing optionals. Agent must not fail if optional fields are absent.

**Edge Case 3 — Payload with error context:**
Caller provides `errorMessage: "KB connection timeout"` and `status: "failed"`. Agent passes the full payload including error context to the skill without sanitizing or truncating. The skill's DB schema accepts error messages up to the varchar limit defined in WINT-0040.

## Architecture Notes

- **Model**: `haiku` — low cost, low latency, fast invocation. Do not use sonnet or opus for this agent.
- **Permission level**: `read-only` — agent only calls MCP tools, writes no files.
- **Invocation pattern**: Fire-and-forget. The skill call must never gate the primary workflow. Error handling: skill returns null → `logger.warn` → completion signal. No re-throw.
- **LangGraph Porting Notes** (per WINT-2080): Required section in all haiku worker agents. Must include Input Contract, Execution Contract, Output Contract, Tool Requirements tables.
- **Completion signals**: `TELEMETRY-LOGGER COMPLETE` | `TELEMETRY-LOGGER COMPLETE WITH WARNINGS: {N}` | `TELEMETRY-LOGGER BLOCKED: {reason}`
- **Schema ownership**: Do not redefine the telemetry payload schema inline — it is owned by `telemetry-log/SKILL.md` (WINT-3020) and the DB tables in WINT-0040.
