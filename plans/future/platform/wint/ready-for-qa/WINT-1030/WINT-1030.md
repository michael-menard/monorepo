# WINT-1030: Populate Story Status from Directories

## Context

The WINT platform is transitioning to a database-driven workflow where story status is the source of truth, stored in the `wint.stories` table. Currently, stories exist in lifecycle directories (backlog/, elaboration/, ready-to-work/, in-progress/, ready-for-qa/, UAT/) with status tracked implicitly by directory location.

**WINT-1020** will flatten these directories, adding a `status:` field to YAML frontmatter. However, this leaves the database empty. Story commands (`/story-status`, `/story-update`, `/story-move`) and LangGraph workflows need the database populated to function. Without this initialization, the compatibility shim (WINT-1010) cannot fall back to database queries, and agent workflows will fail.

**Current State:**
- Stories exist in lifecycle directories OR have been flattened with `status:` field
- `wint.stories` table exists (created in WINT-0010) but is empty
- StoryFileAdapter can read YAML frontmatter (deployed in WINT-1020)
- StoryRepository provides database CRUD operations (deployed)
- Story state enum defines valid values: backlog, ready_to_work, in_progress, ready_for_qa, in_qa, blocked, done, cancelled

**Problem:**
After WINT-1020 flattens story directories:
1. Stories will have `status:` field in YAML frontmatter
2. `wint.stories` table will be empty
3. Story commands will not work (no directory structure to scan, no database rows to query)
4. LangGraph workflows cannot track story state transitions
5. Compatibility shim has nothing to fall back to

**Dependencies:**
- **WINT-1020 (Directory Flattening):** Must complete first to ensure status field exists in frontmatter
- **WINT-0020 (Story Management Tables):** wint.stories table already exists from WINT-0010 migration
- **StoryFileAdapter:** Production-ready, used in WINT-1020
- **StoryRepository:** Production-ready database layer

## Goal

Create a population script that initializes the `wint.stories` database table with current story status, enabling database-driven workflows.

**Success Criteria:**
- All stories discovered and processed
- Status correctly inferred from frontmatter (preferred) or directory location (fallback)
- Enum mapping works for all status values (e.g., 'ready-to-work' → 'ready_to_work')
- Malformed YAML skipped gracefully with warnings
- Duplicate story IDs handled without script failure
- Database populated with validated data
- Verification queries confirm data integrity
- Performance: 100+ stories in <60 seconds

**Status Inference:**
```typescript
function inferStatus(story: StoryMetadata, directoryPath: string): StoryState {
  // Priority 1: Frontmatter status field (explicit)
  if (story.frontmatter.status) {
    return mapStatusToState(story.frontmatter.status)
  }

  // Priority 2: Lifecycle directory (implicit)
  const lifecycleDir = extractLifecycleDir(directoryPath)
  return lifecycleToStateMap[lifecycleDir] || 'backlog'
}
```

**Enum Mapping:**
```typescript
function mapStatusToState(status: string): StoryState {
  // Handle hyphen → underscore conversion
  return status.replace(/-/g, '_') as StoryState
}
```

**Duplicate Resolution:**
```typescript
function resolveDuplicates(locations: StoryLocation[]): StoryLocation {
  // Sort by lifecycle rank descending, take most advanced
  return locations.sort((a, b) => b.rank - a.rank)[0]
}

const lifecycleRank = {
  'UAT': 6,
  'ready-for-qa': 5,
  'in-progress': 4,
  'ready-to-work': 3,
  'elaboration': 2,
  'backlog': 1
}
```

### Error Handling Strategy

```typescript
const results = {
  discovered: 0,
  inserted: 0,
  skipped: 0,
  errors: [] as ErrorEntry[]
}

for (const storyPath of storyPaths) {
  results.discovered++

  try {
    const frontmatter = await adapter.readFrontmatter(storyPath)
    const state = inferStatus(frontmatter, storyPath)
    await repo.createStory({ ...frontmatter, state })
    results.inserted++

  } catch (error) {
    results.skipped++
    results.errors.push({
      story_id: extractStoryId(storyPath),
      reason: error.message,
      action: 'skipped'
    })
    // Continue processing (fail-soft)
  }
}
```

### Performance Optimization

**Batch Inserts:**
```typescript
const BATCH_SIZE = 50

for (let i = 0; i < stories.length; i += BATCH_SIZE) {
  const batch = stories.slice(i, i + BATCH_SIZE)
  await db.transaction(async (tx) => {
    for (const story of batch) {
      await repo.createStory(story, tx)
    }
  })
}
```

**Expected Throughput:**

- 200 stories: ~40 seconds
