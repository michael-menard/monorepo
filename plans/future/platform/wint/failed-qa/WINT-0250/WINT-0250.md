# WINT-0250: Define Escalation Rules for Multi-Model Routing (Graduated Chain + Hard Bypass)

## Context

The WINT platform's Phase 0 multi-model strategy (WINT-0220) defines which model tier to use for each task type, and the unified model interface (WINT-0230) provides the routing abstraction. Without escalation rules, the router has no way to automatically promote a task to a more capable (and expensive) tier when the current tier fails or produces low-confidence output. All model requests would either always use the assigned tier — causing silent failures on difficult tasks — or always escalate eagerly to Claude, defeating the cost-reduction goal.

**Problem:** There is no declarative specification of when to escalate from a cheaper model tier to a more expensive one. Workflow agents have no consistent signal to trigger escalation, and escalation events are not logged, making cost analysis and threshold tuning impossible.

**Current State:** Neither `.claude/config/model-strategy.yaml` (WINT-0220) nor `packages/backend/llm-router/` (WINT-0230) exist yet. This story documents the intended future state of the escalation rules layer that sits atop those two foundations. The `.claude/config/` directory does not exist and must be created.

**Downstream Impact:** WINT-0260 (Model Cost Tracking) will consume escalation event logs; the log schema defined in this story must be compatible with WINT-0260 aggregation queries. WINT-0270 (Benchmark Local Models) will produce data that drives threshold tuning — the inline comments in the YAML file must reference WINT-0270 as the expected calibration trigger.

---

## Goal

Create `.claude/config/escalation-rules.yaml` defining a two-class declarative rule system:

1. **Graduated chain rules** — threshold-based triggers that escalate one tier at a time (failure count, confidence score, uncertainty flag)
2. **Hard bypass rules** — categories that skip the chain entirely and go directly to `API-High` (Claude)

The file also defines the escalation log record schema so the unified model interface (WINT-0230) can emit structured escalation events for downstream analysis by WINT-0260 and WINT-0270.

## Architecture Notes

- **Two rule classes, not one.** `graduated_chain` and `hard_bypass` are structurally separate sections. The llm-router (WINT-0230) applies them via distinct code paths — graduated rules check failure counts and confidence scores, bypass rules check task metadata flags. Mixing them in a single list would make the bypass conditions ambiguous.
- **Tier abstraction.** Rules reference tier names, not model names. When WINT-0270 causes a model swap within a tier (e.g., replacing `qwen2.5-coder:32b` with a newer Local-Large model), escalation rules remain unchanged.
- **Log schema is a specification, not a migration.** The `escalation_log_schema` section describes what the llm-router must emit per escalation event. No database table is created by this story — telemetry table creation is WINT-0040's responsibility. WINT-0260 will consume the events once both WINT-0230 (emission) and WINT-0040 (storage) are implemented.
- **One-directional escalation.** Rules define cheaper → more expensive promotion only. De-escalation (expensive → cheaper after successful output) is explicitly deferred to post-MVP. The llm-router must not infer downward movement from this file.
- **`failure_count: 2` covers both Local-Small and Local-Large.** The `from: Local` rule intentionally groups both local tiers. The llm-router tracks failure count per task, not per tier-within-local. If a task fails twice on any Local tier, it escalates to `API-Cheap`.
