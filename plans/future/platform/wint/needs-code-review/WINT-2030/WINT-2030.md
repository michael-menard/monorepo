# WINT-2030: Populate Project Context Cache from CLAUDE.md and Tech-Stack Docs

## Context

The WINT platform's Phase 2 goal is 80% token reduction across agent invocations. The primary mechanism is the context cache: the `wint.context_packs` table (deployed in WINT-0010, protected) stores JSONB context packs that agents retrieve instead of re-reading source documents on every invocation. The MCP tools to write and read this cache — `context_cache_put`, `context_cache_get`, `context_cache_invalidate`, `context_cache_stats` — were delivered by WINT-0100 (UAT-complete as of 2026-02-16).

`CLAUDE.md` and the docs in `docs/tech-stack/` and `docs/testing/` contain project conventions, patterns, and tech-stack decisions that every agent session needs: Zod-first types, no barrel files, `@repo/logger`, component structure, testing strategy, and more. These are currently re-read from disk on every invocation — expensive and token-costly. The context cache infrastructure is live but contains no project knowledge.

**Dependency clarification**: WINT-2020 (Context Pack Sidecar) and WINT-0030 (Context Cache Tables) are listed as dependencies in the index but are not actual blockers. The required table (`wint.context_packs`) was delivered by WINT-0010. `contextCachePut()` operates as a standalone database function without requiring the WINT-2020 HTTP sidecar. This story can proceed independently once WINT-0100 is confirmed complete (it is).

## Goal

Create a TypeScript populate script that reads `CLAUDE.md` and docs in `docs/tech-stack/` and `docs/testing/`, extracts key facts per domain area, and writes 5 discrete cache entries to `wint.context_packs` via `contextCachePut()`. Each entry is a targeted, summarized JSONB pack — not a raw file dump — optimized for agent injection under 2000 tokens per pack.

## Non-Goals

- Do not create new packType enum values — use only existing: `architecture`, `codebase`, `test_patterns`, `story`, `feature`, `epic`, `lessons_learned`
- Do not modify `wint.context_packs` table schema — protected by WINT-0010
- Do not implement the context-pack sidecar HTTP endpoint — that is WINT-2020's scope
- Do not implement cache warming scheduling — that is WINT-2070/WINT-2080's scope
- Do not implement automatic doc-change triggers — manual re-population is sufficient for MVP
- Do not cache agent mission data — that is WINT-2040's scope
- Do not cache domain KB (ADRs, blockers, lessons) — that is WINT-2050's scope
- Do not cache library patterns (React 19, Vitest API) — that is WINT-2060's scope
- Do not implement a UI or CLI command — script is a Node.js utility
- Do not implement token counting — `tokenCount` field may be left null in MVP

## Architecture Notes

**Upsert pattern (from WINT-0100):**
```typescript
await contextCachePut({
  packType: 'architecture',
  packKey: 'project-conventions',
  content: {
    summary: 'Core project conventions for LEGO MOC platform',
    rules: ['Zod-first types — never TypeScript interfaces', 'No barrel files', 'Use @repo/logger, never console.log'],
    patterns: ['Component dir: index.tsx + __tests__/ + __types__/'],
    commands: ['pnpm dev', 'pnpm build', 'pnpm test', 'pnpm check-types'],
  },
  ttl: 2592000, // 30 days
})
```

**Resilient execution loop:**
```typescript
const results = { attempted: 0, succeeded: 0, failed: 0 }
for (const pack of PACK_DEFINITIONS) {
  results.attempted++
  try {
    const content = await extractContent(pack.sourceFile)
    const result = await contextCachePut({ ...pack, content, ttl: 2592000 })
    if (result) results.succeeded++
    else results.failed++
  } catch (error) {
    logger.error('[populate-project-context] Pack write failed', { packKey: pack.packKey, error })
    results.failed++
  }
}
return results
```

**Content extraction strategy:**
- For `CLAUDE.md` (longer doc): section-based extraction by `##` heading — extract each section separately to avoid oversized single entry
- For short tech-stack docs (~40 lines each): full doc extraction into structured fields is viable
- For `docs/testing/`: read `overview.md`, extract by section heading

**packType mapping rationale:**
- `architecture` — for conventions, tech-stack facts, patterns that define how the codebase is built
- `test_patterns` — for testing framework choices, coverage requirements, query conventions
- No new enum values are needed or allowed
