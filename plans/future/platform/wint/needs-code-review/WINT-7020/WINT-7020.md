## Architecture Notes

### Risk Scoring Model

Agents are scored by risk tier using spawn depth and workflow criticality:

| Risk Tier | Criteria | Examples |
|-----------|----------|---------|
| Critical | Leader with 5+ spawn children; top-level orchestrator | pm-story-generation-leader, dev-implement-story |
| High | Leader with 2-4 children or high command trigger frequency | elab-completion-leader, qa-verify-completion-leader |
| Medium | Worker spawned by high-risk leader; functional swim-lane refs | dev-implement-planner, pm-dev-feasibility-review |
| Low | Leaf worker with 1 parent; low reference count | pm-draft-test-plan, pm-uiux-recommendations |

### Batch Grouping Strategy

From `migrate-agents-v3.md` pattern adapted for Phase 7:

1. **Story Management** — commands/agents that read/write story status (highest risk — core CRUD)
2. **Dev Workflow** — dev-implement-* agents (high volume, critical path)
3. **Elab Workflow** — elab-* agents (medium complexity, bounded scope)
4. **QA Workflow** — qa-verify-* agents (quality gate integrity must not degrade)
5. **Review & Reporting** — architect-*, review-*, reporting agents
6. **PM & Planning** — pm-* agents not in story management
7. **Utility & Orphan Candidates** — remaining low-risk or deprecated agents

### Shim Rollback Strategy

The compatibility shim at `packages/backend/mcp-tools/src/story-compatibility/` supports directory fallback. For rollback:
- DB-first write fails → shim falls back to directory operations automatically
- To force directory mode: shim diagnostics field (source: `db | directory | not_found`) can be monitored
- No code change needed for rollback — shim's dual-write is the safety net
