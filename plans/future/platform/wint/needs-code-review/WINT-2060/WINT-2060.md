# WINT-2060: Populate Library Cache

Cache common library patterns (React 19, Tailwind, Zod, Vitest) from docs into `wint.context_packs` for agent reference.

---

## Context

The WINT platform Phase 2 goal is 80% token reduction across agent invocations. The `wint.context_packs` table (protected, WINT-0010) is the storage layer. WINT-2030 populated project conventions and tech-stack packs. WINT-2040 populated agent mission packs. WINT-2050 will populate domain knowledge and ADR packs.

This story closes the library usage pattern gap: when agents generate code using React 19, Tailwind CSS, Zod, or Vitest, they currently have no cached reference for the specific patterns, hook APIs, or idioms used in this codebase. They must either re-read docs, re-read source files, or risk hallucination. At scale (WINT-2120 benchmarks), this is a measurable token cost.

**Problem**: Agent invocations needing library-specific guidance (e.g., "how do we write a Zod schema here?" or "which React hooks are in use?") must re-read CLAUDE.md, tech-stack docs, and real source files on every invocation. These patterns are stable and highly cacheable.

**Reality baseline**: `wint.context_packs` deployed and protected (WINT-0010). `contextCachePut()` UAT-complete (WINT-0100). `populate-project-context.ts` is the direct implementation model (WINT-2030, needs-code-review). The `codebase` packType exists in `contextPackTypeEnum` and is the correct type for library pattern entries.

**Resolved conflicts**:
- Missing `library_cache` packType: Resolved by using existing `codebase` packType — semantically appropriate, avoids migration
- WINT-2020 dependency listed in index: Resolved — write path via `contextCachePut()` is independent of WINT-2020 HTTP sidecar. Actual blocking deps are WINT-0030 (table) + WINT-0100 (MCP tools), both UAT-complete.

---

## Goal

Create a `populate-library-cache.ts` script in `packages/backend/mcp-tools/src/scripts/` using the exact same architecture as `populate-project-context.ts`. Read CLAUDE.md and `docs/tech-stack/*.md`, extract compact structured JSONB packs per library (React 19, Tailwind, Zod, Vitest), and write to `wint.context_packs` via `contextCachePut()`.

Each pack: `packType: 'codebase'`, `packKey: 'lib-{name}'`, `content: { summary, patterns, rules, examples? }`, TTL 30 days.

---

## Examples

**Example 1 — Happy path execution:**

Input: `pnpm tsx packages/backend/mcp-tools/src/scripts/populate-library-cache.ts` (with DATABASE_URL=postgres://...@localhost:5432/lego_dev)

Expected output:
```
[logger] Populating library cache...
[logger] lib-react19: success
[logger] lib-tailwind: success
[logger] lib-zod: success
[logger] lib-vitest: success
[logger] Library cache population complete: { attempted: 4, succeeded: 4, failed: 0 }
```

DB result: `SELECT COUNT(*) FROM wint.context_packs WHERE pack_key IN ('lib-react19','lib-tailwind','lib-zod','lib-vitest') AND pack_type='codebase'` → `4`

---

**Example 2 — Idempotent re-run:**

Input: Run the script a second time (all 4 rows already exist from Example 1)

Expected output: Same logger output as Example 1. `{ attempted: 4, succeeded: 4, failed: 0 }`.

DB result: `SELECT COUNT(*) FROM wint.context_packs WHERE pack_key LIKE 'lib-%'` → `4` (no duplicates — upsert on `(packType, packKey)` composite unique constraint)

---

**Example 3 — Single pack write failure (error isolation):**

Input: `contextCachePut` throws on the `lib-tailwind` write (e.g., transient DB connection drop)

Expected output:
```

[logger] lib-zod: success
[logger] lib-vitest: success
[logger] Library cache population complete: { attempted: 4, succeeded: 3, failed: 1 }
```

Script exits normally (no uncaught exception). The 3 successful packs are persisted.

## Edge Cases

**Edge Case 1 — Source doc temporarily unavailable:**

Condition: `readDoc('docs/tech-stack/frontend.md')` returns `null` (file missing during CI or path shift)

Expected behavior: `extractReact19Patterns` and `extractTailwindPatterns` receive null/empty input. Extraction returns a minimal valid content object (or the pack is skipped with a warning logged). Script does not throw. `result.failed` is incremented for skipped packs. Remaining packs continue normally.

---

**Edge Case 2 — Extracted content exceeds 8000 char JSON size limit:**

Condition: An extraction function naively copies a large section from a verbose doc, producing `JSON.stringify(content).length >= 8000`

Expected behavior: Content is truncated or the extraction function caps patterns/rules arrays at a maximum count before returning. The pack is still written (with capped content). Optionally, a warning is logged. Under no circumstances does the pack write proceed with oversized content that would stress the JSONB column.

---

**Edge Case 3 — Script run on fresh DB (no prior lib-* rows):**

Condition: No `lib-react19`, `lib-tailwind`, `lib-zod`, or `lib-vitest` rows exist yet

Expected behavior: All 4 writes succeed via INSERT path of the upsert. `{ attempted: 4, succeeded: 4, failed: 0 }` returned. Same as Example 1.

---

**Edge Case 4 — Partial prior state (2 of 4 rows exist):**

Condition: `lib-react19` and `lib-zod` already exist in `context_packs` from a previous partial run

Expected behavior: Existing rows are updated via UPDATE path of the upsert. New rows for `lib-tailwind` and `lib-vitest` are inserted. Final count is 4. `{ attempted: 4, succeeded: 4, failed: 0 }`.

## Non-Goals

- Do NOT implement cache-warming orchestration (WINT-2070/2080 scope)
- Do NOT cache agent missions (WINT-2040), project conventions (WINT-2030), or domain knowledge/ADRs (WINT-2050)
- Do NOT build an HTTP endpoint — population script is a Node.js utility only
- Do NOT read from the KB (port 5433) — library patterns come from filesystem docs only
- Do NOT modify `wint.context_packs` table schema (protected by WINT-0010)
- Do NOT add new `contextPackTypeEnum` values — use existing `codebase` packType
- Do NOT cache library changelogs or full API references — extract compact codebase-specific idioms only
- Do NOT implement cache retrieval for agents (WINT-2110 scope)
- Do NOT implement automatic doc-change triggers — manual re-population is MVP
- Do NOT build a CLI command or UI

## Architecture Notes

**Script architecture** (mirrors `populate-project-context.ts` exactly):

```
populate-library-cache.ts
  ├── Constants: MONOREPO_ROOT, TTL_30_DAYS
  ├── PopulateResultSchema (Zod)
  ├── LibraryContentSchema (Zod): { summary, patterns, rules, examples? }
  ├── readDoc(path): string | null
  ├── extractSections(content): Map<string, string>
  ├── extractReact19Patterns(doc: string): LibraryContent
  ├── extractTailwindPatterns(doc: string): LibraryContent
  ├── extractZodPatterns(doc: string): LibraryContent
  ├── extractVitestPatterns(doc: string): LibraryContent
  ├── populateLibraryCache({ contextCachePutFn? }): PopulateResult
  │     ├── reads: CLAUDE.md, docs/tech-stack/*.md
  │     ├── calls: extractX() for each library
  │     ├── writes: contextCachePut({ packType: 'codebase', packKey: 'lib-X', content, ttl })
  │     └── returns: { attempted: 4, succeeded: N, failed: M }
  └── main() — script entry point
```

**Content extraction strategy**: Line-scanning or section-based extraction using `extractSections()`. Do NOT parse full Markdown AST. Each extraction function reads from the appropriate source doc sections:

| Library | Primary source | Section target |
|---------|---------------|----------------|
| React 19 | `docs/tech-stack/frontend.md` | React 19, hooks, component patterns |
| Tailwind CSS | `docs/tech-stack/frontend.md` + `CLAUDE.md` | Tailwind utility classes, token colors |
| Zod | `CLAUDE.md` | Zod-First Types, schema patterns |
| Vitest | `docs/tech-stack/backend.md` | Testing patterns, Vitest setup |

**Injectable dependency for testability**: `populateLibraryCache()` accepts optional `contextCachePutFn` parameter (same pattern as WINT-2050's `kbQueryFn`). Unit tests inject a `vi.fn()` spy; production uses the real `contextCachePut` import.

**packType decision**: Use `'codebase'` — confirmed present in `contextPackTypeEnum` in `packages/backend/database-schema/src/schema/wint.ts`. This avoids any schema migration and aligns semantically with "codebase-specific library patterns."
