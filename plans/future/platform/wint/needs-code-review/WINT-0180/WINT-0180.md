# WINT-0180: Define Examples + Negative Examples Framework

## Context

This story formalizes the implicit structure already established by WINT-0190. Two pattern spec files (`patch-queue-pattern.md` and `repair-loop-pattern.md`) were created under `.claude/prompts/role-packs/_specs/` as part of WINT-0190. These files implicitly follow a pattern that this story makes explicit: a standard for how examples are defined, bounded, and delivered to agents.

Without this framework, WINT-0210 (Populate Role Pack Templates) has no authoritative reference for how to structure the per-role instructions or their examples. The framework establishes the contract that all downstream role pack work must follow.

The baseline (2026-02-13) confirms the `_specs/` directory exists with both pattern files active. No conflicting work is in progress. WINT-0190 owns the `_specs/` files — this story must not modify them.

---

## Goal

Create `.claude/prompts/role-packs/FRAMEWORK.md` — a single authoritative document that defines how examples are structured in role packs. This reduces scope creep and rework by making "minimum viable" patterns explicit with canonical templates. Any agent or developer creating a role pack or pattern spec can consult FRAMEWORK.md to know exactly what is required.

## Architecture Notes

The framework document defines three layers:

1. **Role instruction layer** (`prompts/role-packs/{role}.md`): Per-role instructions capped at 150-300 tokens. These are consumed by agents at spawn time. Format: brief mission statement + 1-2 pattern references.

2. **Pattern spec layer** (`prompts/role-packs/_specs/{pattern}.md`): Detailed pattern specifications with decision rules, examples (max 2 positive + 1 negative), and proof requirements. These are consulted by developers, not injected whole into agent context.

3. **Delivery layer**: How the above files reach agents at runtime:
   - File injection: Read and include `{role}.md` content directly
   - KB MCP: Tag-based retrieval via `kb_search({ tags: ['role-pack', '{role}'] })`
   - Context-pack sidecar: Injection point defined in WINT-2020 (not implemented here)

The pattern skeleton (10-25 lines) constraint exists to enforce "minimum viable" patterns — long patterns become prose documentation, not actionable agent instructions.

2. All 7 required sections present and non-empty
3. Existing `_specs/` files structurally map to framework sections

### Error Cases

- Pattern skeleton > 25 lines: FRAMEWORK.md must document the remediation (split the pattern)
- More than 2 positive examples: FRAMEWORK.md must document the hard cap with rationale
- Prose decision rule: FRAMEWORK.md must include a "DON'T" example or counter-example

### Edge Cases

- Single positive example (allowed — max 2 means "up to 2", not "exactly 2")
- Boundary skeleton lengths (exactly 10 or exactly 25 lines — both inclusive)
- Delivery mechanism precedence order documented

### Risks

- Framework too abstract without a concrete worked example — mitigate by including a complete skeleton example in FRAMEWORK.md
- Token counting requires a tokenizer reference — include line-count proxy as fallback
