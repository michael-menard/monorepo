# WINT-9050: Create evidence-judge LangGraph Node (`nodes/qa/evidence-judge.ts`)

## Context

The evidence-judge Claude Code agent (`evidence-judge.agent.md`) is an adversarial QA worker that evaluates AC evidence quality during Phase 4 QA. It classifies each evidence item as STRONG or WEAK using mechanical rules, derives per-AC verdicts (ACCEPT / CHALLENGE / REJECT), and writes `ac-verdict.json` for downstream consumption.

Phase 9 of the WINT platform initiative ports agent-based logic to native TypeScript LangGraph nodes. WINT-9050 creates the TypeScript equivalent at `packages/backend/orchestrator/src/nodes/qa/evidence-judge.ts`, following the established `createToolNode` pattern used by all existing `nodes/qa/*` nodes. The three core pure functions (`classifyEvidenceStrength`, `deriveAcVerdict`, `deriveOverallVerdict`) are extracted into `packages/backend/workflow-logic/` following the WINT-9010 pattern.

The canonical analogue for this node's structure is `verify-acs.ts`, which performs per-AC iteration with structured log events using `createToolNode`. Evidence-judge follows the same skeleton but replaces model calls with mechanical classification rules.

## Goal

Port the evidence-judge agent's 4-phase evidence classification contract to a native TypeScript LangGraph node so that adversarial AC evidence evaluation runs identically in both the Claude Code workflow and the LangGraph workflow without subprocess delegation.

## Examples

**Example 1 — All STRONG evidence, overall PASS:**
Given a QA graph state with `evidence.acceptance_criteria` containing 3 ACs, each with at least one `test` or `command` type evidence item with objective descriptions, the node produces `ac-verdict.json` at `{story_dir}/_implementation/ac-verdict.json` with `overall_verdict: "PASS"`, all three `ac_verdicts[].verdict: "ACCEPT"`, and `accepted: 3, challenged: 0, rejected: 0`.

**Example 2 — Mixed quality, overall CHALLENGE:**
Given `evidence.acceptance_criteria` with AC-1 having a `test` evidence item (STRONG), AC-2 having only an `http` item with description containing "appears to work" (downgraded to WEAK), and AC-3 having zero evidence items, the node produces `ac-verdict.json` with `overall_verdict: "CHALLENGE"`, `ac_verdicts: [{verdict: "ACCEPT"}, {verdict: "CHALLENGE"}, {verdict: "REJECT"}]`.

## Edge Cases

**Edge Case 1 — Null evidence state (missing EVIDENCE.yaml):**
When `state.evidence` is `null`, the node does not throw. Instead it returns a result with `overall_verdict: "FAIL"`, all AC entries having `verdict: "REJECT"` with `challenge_reason: "no evidence bundle found"`, and a non-empty `warnings` array.

**Edge Case 2 — Subjective language blocklist downgrade:**
An evidence item of type `test` (which is normally classified STRONG) containing the word `"appears"`, `"seems"`, `"should"`, or `"looks"` in its `description` field is downgraded to WEAK. The per-type check runs first; the blocklist check runs second and overrides.

## Non-Goals

- Running or re-executing tests — the node reads `EVIDENCE.yaml` state only; no subprocess invocations.
- Making LLM/model calls — evidence classification is purely mechanical in v1.0.
- Writing to the Knowledge Base or any MCP tool stores.
- Replacing or modifying the existing QA verification leader agent behavior.
- Producing a blocking gate decision — `overall_verdict: FAIL` is a valid output, not a node failure.
- Integrating `evidence-judge` into the LangGraph QA graph routing — that is owned by WINT-4120 and WINT-4140.
- Implementing DB-based evidence retrieval (deferred to after WINT-7060 lands).
- Touching production DB schemas in `packages/backend/database-schema/`.
- Modifying orchestrator artifact schemas in `artifacts/evidence.ts`.

## Architecture Notes

**4-Phase Logical Contract** (from evidence-judge.agent.md — use as implementation scaffold):
1. **Phase 1: Load Inputs** — Read `evidence` from state; parse via `EvidenceItemSchema`; handle null state gracefully
2. **Phase 2: Evaluate Evidence Strength** — For each AC, iterate `evidence_items`; call `classifyEvidenceStrength` per item; accumulate STRONG/WEAK counts
3. **Phase 3: Apply Adversarial Challenge** — Call `deriveAcVerdict` per AC using STRONG/WEAK counts; build per-AC result with `challenge_reason` for non-ACCEPT verdicts
4. **Phase 4: Produce Output** — Call `deriveOverallVerdict`; write `ac-verdict.json` via `fs/promises`; return state update

**Pure Function Extraction Strategy** (WINT-9010 dependency workaround):
- If WINT-9010 is still in-qa: implement classifiers inline in `evidence-judge.ts` as unexported functions first
- After WINT-9010 reaches UAT/completed: extract to `packages/backend/workflow-logic/src/evidence-judge/index.ts` and update exports
- Final state MUST have classifiers in `workflow-logic/` (AC-2, AC-3, AC-4 require this)

**Evidence Strength Classification Rules** (from agent spec):

Per-type baseline classification:
- `test`, `command`, `e2e` → STRONG baseline
- `http`, `file`, `screenshot`, `manual` → WEAK baseline

Subjective language blocklist (applied after per-type check, overrides to WEAK):
- Words: `appears`, `seems`, `should`, `looks`
- If description contains any blocklisted word → WEAK (regardless of type)

**Zod Schemas Required** (Zod-first, no TypeScript interfaces):
- `AcVerdictSchema` — per-AC verdict result
- `EvidenceJudgeResultSchema` — full output with ac_verdicts array
- `EvidenceJudgeOutputSchema` — ac-verdict.json file structure

**Node Middleware Selection:**
- Use `createToolNode` (not `createNode`) — evidence-judge v1.0 makes no model calls; pure computation
- Circuit breaker not needed for this node
