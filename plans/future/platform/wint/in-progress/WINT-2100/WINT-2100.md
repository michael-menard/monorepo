# WINT-2100: Create session-manager Agent

## Context

WINT-0110 (UAT complete) delivered five session management MCP tools — `sessionCreate`, `sessionUpdate`, `sessionComplete`, `sessionQuery`, `sessionCleanup` — backed by the `wint.contextSessions` database table. WINT-2090 (pending) will create `session-create` and `session-inherit` skills that implement the higher-level leader→worker context sharing protocol. Together these form the session infrastructure that Phase 2 agents depend on.

With the session MCP tools live and WINT-2090 skills incoming, there is no dedicated haiku-powered worker agent responsible for session lifecycle orchestration. Without it, leader agents must embed session lifecycle logic inline, creating duplication and making session leak prevention fragile. The index risk note is explicit: "Must prevent session leaks."

The haiku worker agent pattern is well-established in this codebase via `turn-count-metrics-agent.agent.md`. The session lifecycle patterns (create, update, complete, cleanup) are fully documented in `SESSION-MANAGEMENT-TOOLS.md`. This story is specification and wrapping, not new infrastructure.

WINT-9090 will port this agent to a LangGraph node at `packages/backend/orchestrator/src/nodes/context/` for parity. The agent must include a LangGraph porting interface contract section (per pattern established in WINT-0160, validated in WINT-9020) to facilitate the port without reimplementing discovery.

---

## Goal

Create a new haiku-powered worker agent at `.claude/agents/session-manager.agent.md` that handles session lifecycle orchestration on behalf of invoking leader agents:
- Session creation with leaked-session detection and null-return handling
- Session updates with incremental token accumulation and completed-session throw handling
- Session completion with idempotent guard
- Session cleanup with mandatory dry-run preview and explicit opt-in for actual deletion

---

## Non-Goals

- Implementing the `session-create` or `session-inherit` skill logic — that is WINT-2090's scope
- Creating any new MCP tools — the five from WINT-0110 are sufficient
- Adding a `status` column to `wint.contextSessions` — active vs completed is determined by `endedAt IS NULL`
- Integrating with leader agents (context-warmer, batch-coordinator) — this story delivers the agent file only; integration is scoped to each respective leader story
- Implementing the LangGraph node port — that is WINT-9090's scope
- Modifying the `contextSessions` database schema
- Adding session analytics or aggregate queries — that is WINT-3090's scope
- Creating a `/session-manager` CLI command — this agent is always spawned by leaders, never invoked directly

## Architecture Notes

- **Worker agent, not leader**: This agent is always spawned by leader agents. It does NOT spawn sub-agents. Known leaders: context-warmer (WINT-2080), batch-coordinator (WINT-6010).
- **Fire-and-forget null policy**: When `sessionCreate` returns null (DB error), the default policy is warn + continue. This prevents session tracking errors from blocking workflow execution. Policy must be explicitly documented and configurable.
- **Active session detection via endedAt**: `wint.contextSessions` has no status column. Active sessions are identified by `endedAt IS NULL`. The agent must never assume an explicit status field.
- **Incremental token mode**: `sessionUpdate` defaults to `mode: 'incremental'` for concurrent-safe accumulation. This is the correct and required default.
- **Cleanup safety constraint**: `sessionCleanup` defaults to `dryRun: true`. The agent must document that `dryRun: false` must be explicitly passed. This is a hard safety constraint from WINT-0110 implementation.
- **LangGraph porting**: The agent must include a porting interface contract section per the pattern established in WINT-0160 to enable WINT-9090 to port without reimplementing discovery. Target node path: `packages/backend/orchestrator/src/nodes/context/`.
