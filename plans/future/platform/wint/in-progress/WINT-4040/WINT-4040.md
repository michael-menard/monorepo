# WINT-4040: Infer Existing Capabilities from Story History and Populate graph.feature_capabilities

## Context

The WINT graph schema (`graph.capabilities`, `graph.features`) was deployed in WINT-0060 and WINT-0131. The `capabilities.lifecycle_stage` field (`create | read | update | delete`) drives Franken-feature detection in the cohesion sidecar (`computeAudit`, WINT-4010) and the `graph_get_franken_features` MCP tool. As of this story, the `graph.capabilities` table is empty — every feature appears as a Franken-feature with all CRUD stages missing, making all graph analysis meaningless.

WINT-4030 (currently at needs-code-review) populates `graph.features` with the codebase's known features but does not populate `graph.capabilities`. This story fills that gap by inferring capabilities from the corpus of existing story files and seeding the capability graph.

**Prerequisite**: WINT-4030 must be merged and its data verified before this script's integration testing can execute. Unit tests are fully independent via dependency injection.

## Goal

Create a capability inference script that scans existing story YAML files, applies keyword-to-lifecycle-stage heuristics, and populates `graph.capabilities` with feature-linked CRUD capability rows so the cohesion sidecar and graph-checker agents can perform meaningful analysis.

## Examples

**Example 1 — Happy path keyword inference:**
Given a story with title "Add MOC to wishlist" and AC text "user can view and delete saved items":
- Script extracts keywords: `add` → `create`; `view` → `read`; `delete` → `delete`
- Looks up `WISH` epic prefix in `graph.features` → finds UUID `f1a2b3c4-...`
- Inserts 3 rows: `wish-create-inferred`, `wish-read-inferred`, `wish-delete-inferred` all with `feature_id=f1a2b3c4-...`, `lifecycle_stage=create/read/delete`, `capability_type='business'`, `maturity_level='beta'`
- Emits: `{ attempted: 3, succeeded: 3, failed: 0, skipped: 0 }`

**Example 2 — Dry-run mode:**
Given same story and the `--dry-run` flag:
- Script logs table showing would-be inserts without touching the DB
- `SELECT count(*) FROM graph.capabilities` still returns 0 after run
- Dry-run output shows `{ featureName: 'wish', inferredStage: 'create', keyword: 'add', wouldInsert: true }` per entry

## Edge Cases

**Edge Case 1 — Zero features (WINT-4030 not yet merged):**
If `graph.features` contains zero rows at script startup, the script exits immediately with:
`"No features found — run WINT-4030 script first"` and a non-zero exit code — prevents silently producing zero inserts without user awareness.

**Edge Case 2 — Duplicate capability within same feature:**
A story may have both `"add item"` and `"create item"` in its text, both mapping to `create`. On the second insert attempt for `wish-create-inferred`, `ON CONFLICT DO NOTHING` semantics skip the insert. The `skipped` count increments; no error is thrown.

**Edge Case 3 — Epic prefix unknown:**
If a story's epic prefix (e.g., `XYZ`) is not found in `graph.features`, the script logs a warning (`"Feature not found for prefix XYZ — skipping"`) and continues processing remaining stories. `skipped` count increments; the script does not fail.

**Edge Case 4 — Story text with no capability keywords:**
If a story's title, feature, and AC text contain no words from the keyword mapping table, zero capabilities are inferred for that story. The story is processed silently; no error.

## Non-Goals

- Do NOT create new rows in `graph.features` — only read from it (WINT-4030 owns features)
- Do NOT change `capabilities.lifecycle_stage` enum values — must remain `create | read | update | delete`
- Do NOT implement automated re-inference (one-time CLI script, not a daemon or scheduled job)
- Do NOT infer capabilities from UI components or API route files — story text is the only source
- Do NOT modify the cohesion sidecar (WINT-4010) — it is a consumer, not the producer
- Do NOT add database schema migrations — uses existing schema only
- Do NOT treat `upload/replace/download` as separate lifecycle stages beyond the 4 CRUD stages supported by the graph schema

## Architecture Notes

- Script follows the **dependency injection** pattern established in WINT-2030/2050/2060: the main `inferCapabilities()` function accepts an optional `insertFn` parameter. Unit tests pass `vi.fn()` mocks; production code uses real Drizzle client.
- Keyword mapping table implemented as a **Zod-validated lookup object** (not a switch statement) so it is easy to extend in WINT-4050.
- **Parallelization**: Use `Promise.allSettled()` for parallel feature UUID lookups per epic prefix (WINT-2040 opp-3 lesson).
- **Type safety**: All schemas defined with `z.object(...)`, types inferred with `z.infer<>` — no TypeScript `interface` declarations.
- **Security**: All DB writes via Drizzle ORM (parameterized queries) — no raw SQL (ADR-005 / WINT-0130 AC-7).
- The inference is heuristic (`maturity_level: 'beta'`) — the `--validate` output surfaces coverage gaps for human review.
