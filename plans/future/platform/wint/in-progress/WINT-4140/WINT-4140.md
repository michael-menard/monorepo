# WINT-4140: Create Round Table Agent

## Context

The WINT Phase 4 elaboration workflow has two adversarial worker agents delivering conflicting outputs: cohesion-prosecutor (PO role, WINT-4070 — ready-for-qa) produces `cohesion-findings.json` (max 5, max 2 blocking), and scope-defender (DA role, WINT-4080 — UAT) produces `scope-challenges.json` (max 5). These agents intentionally disagree — the prosecutor pushes for completeness, the defender challenges non-MVP scope.

No agent currently resolves these adversarial outputs into a single actionable scope. Without a synthesis step, the elab workflow terminates with two conflicting artifacts and no authoritative final scope for implementation. The conflict resolution is currently manual or undefined.

**Reality constraints:**
- scope-defender is functionally complete (WINT-4080 in UAT); `scope-challenges.json` schema is stable
- cohesion-prosecutor (WINT-4070) is in ready-for-qa; `cohesion-findings.json` schema is provisional pending WINT-4150
- WINT-4150 (Standardize Elab Output Artifacts) is blocked on WINT-4140 and will formalize `final-scope.json` schema
- Agent-file-only story: no TypeScript code; deliverable is `.claude/agents/round-table.agent.md` only
- Model: haiku (per Phase 4 worker agent pattern)

**Lessons applied:**
- WINT-4080 elab gap: scope-defender missing `spawned_by` field — Round Table must include this field (fixes discoverability gap)
- WINT-4080 elab enhancement: `accept-as-mvp` DA challenges should be filtered into an accepted pass-through list, not treated as deferrals
- WINT-4080 elab enhancement: log warning if `schema_version` field absent from input artifacts (schema changes undetectable)
- WKFL-010: QA verification is structural/content only (file inspection); tests exempt for agent-file-only stories

---

## Goal

Create a haiku-powered synthesis agent (`round-table.agent.md`) that mechanically converges PO cohesion findings, DA scope challenges, and elab gaps into a single authoritative `final-scope.json`, explicitly deferring all non-included items to `followups[]` with source attribution.

---

## Examples

**Happy path — full inputs available:**
Given `cohesion-findings.json` with 3 findings (1 blocking), `scope-challenges.json` with 4 challenges (2 defer-to-backlog, 1 reduce-scope, 1 accept-as-mvp), and `elab-gaps.json` with 2 gaps, the Round Table produces a `final-scope.json` where:
- The blocking PO finding and the 2 accepted DA items appear in `final_acs[]` with `source` attribution
- The 2 DA defer-to-backlog items appear in `followups[]` with `source: DA` and `deferral_note` preserved
- The `accept-as-mvp` DA entry appears in an `accepted` pass-through section, not in `followups[]`
- `schema_version: '1.0-draft'` is set; `total_warning_count: 0`

**Happy path — optional inputs missing:**
Given only `scope-challenges.json` present (cohesion-findings.json absent, elab-gaps.json absent), the Round Table proceeds with DA challenges only, increments `total_warning_count: 2` (one per missing optional input), and emits `ROUND-TABLE COMPLETE WITH WARNINGS: 2 warnings`.

---

## Edge Cases

**Conflict: PO marks AC blocking, DA challenges the same AC as defer-to-backlog:**
The AC does NOT appear in `final_acs[]`. It appears in `followups[]` with `source: PO`, `conflict: true`, and the DA `deferral_note` preserved. The Round Table does not resolve the conflict — it surfaces it explicitly for PM grooming decision.

**Missing required input (scope-challenges.json unreadable):**
Round Table emits `ROUND-TABLE BLOCKED: scope-challenges.json missing or unreadable`. PO findings and elab gaps are optional; only `scope-challenges.json` is required.

**DA challenge with `recommendation: accept-as-mvp`:**
Filtered into accepted pass-through list; does NOT appear in `followups[]` as a deferred item. This handles the known schema wart from WINT-4080 elab (wint-4080-enh-5).

**`schema_version` field absent from input artifact:**
Round Table logs a warning noting `schema_version_missing` for that input artifact and proceeds. Does not hard-fail.

## Architecture Notes

- **Model**: haiku — consistent with all Phase 4 worker agents (scope-defender, graph-checker, gap-analytics-agent)
- **Synthesis-only constraint**: The hardest correctness risk. Every phase must include a guard: "If this step would produce an item not present in Phase 1 inputs, STOP and flag as blocked."
- **PO vs DA conflict resolution**: When PO marks an AC `blocking` AND DA recommends `defer-to-backlog` or `reduce-scope` for the same AC, the item goes to `followups[]` with `conflict: true`. Round Table does not adjudicate — it surfaces the conflict.
- **accept-as-mvp handling**: DA challenges with `recommendation: accept-as-mvp` are treated as pass-through accepted items, not deferrals. They appear in the output's `accepted_passthrough[]` section, not `followups[]`. This handles the known wart from WINT-4080 elab (wint-4080-enh-5).
- **schema_version**: Output `final-scope.json` must include `schema_version: '1.0-draft'`. Consumers must treat this as provisional until WINT-4150 delivers the standardized schema.
- **spawned_by**: Must include in YAML frontmatter referencing the elab orchestrator. This fixes the discoverability gap identified in WINT-4080 elab (wint-4080-gap-4).
