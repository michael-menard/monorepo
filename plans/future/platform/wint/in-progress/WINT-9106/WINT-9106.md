# WINT-9106: Implement LangGraph Checkpointer & State Recovery

> **Implementation Gate:** This story MUST NOT enter `in-progress` until WINT-9105 (`completed`) and WINT-0070 (`completed`) are both done. Story generation proceeds speculatively; the ADR and DB migration must exist before coding begins.

## Context

The orchestrator currently uses file-based `CHECKPOINT.yaml` files to record execution state for crash recovery and `resume_from` phase semantics. These files track a phase number (e.g., `resume_from: 3`) and are written to story directories during multi-phase agent execution.

As LangGraph graphs replace agent scripts (Wave 9), file-based checkpointing becomes unviable. A LangGraph graph interrupted mid-node cannot resume from a phase number without knowing the exact LangGraph `thread_id` and serialized state at the moment of interruption. The graph's internal state — node execution history, annotation values, pending edges — cannot be reconstructed from a single integer.

The existing `wint.workflow_checkpoints` table (defined in `packages/backend/database-schema/src/schema/wint.ts`) provides the DB foundation. LangGraph's native `@langchain/langgraph-checkpoint-postgres` package (`PostgresSaver`) provides an idiomatic alternative. WINT-9105 (the LangGraph Error Handling ADR) must resolve this decision before implementation begins.

**Problem:** LangGraph graphs executed by the orchestrator have no crash-safe persistence. If the Lambda process dies mid-node during a multi-hour story elaboration, all progress is lost and the full graph re-executes from the start — expensive in tokens and time.

**Proposed Solution:** Implement a PostgreSQL-backed checkpointer for all orchestrator LangGraph graphs, with: (1) a `Checkpointer` module in `packages/backend/orchestrator/src/checkpointer/` that wraps graph compilation with checkpoint saver injection, (2) a `resume-graph` CLI command, (3) checkpoint TTL/cleanup, and (4) an integration test proving crash-then-resume produces identical final state.

## Goal

Replace file-based `CHECKPOINT.yaml` crash recovery with DB-backed LangGraph thread checkpoint persistence. Enable orchestrator LangGraph graphs to survive process restarts without re-running completed nodes.

## Non-Goals

- Do NOT implement compensating transaction execution — AC-004 only records rollback intent; execution is deferred to a future story.
- Do NOT replace the existing `loadFromDb`/`saveToDb` persistence nodes in `packages/backend/orchestrator/src/nodes/persistence/` — those persist workflow artifacts (elaborations, plans, proofs); the checkpointer is for LangGraph graph state continuity.
- Do NOT implement DB-persisted circuit breaker state (`workflow.circuitBreakerState`) — that is WINT-9107's responsibility.
- Do NOT modify any application-facing API routes or frontend components.
- Do NOT implement multi-graph parallel checkpoint isolation — scope to single-graph thread checkpointing only.
- Do NOT touch `@repo/db` connection pooling configuration — use `getPool()` as-is.

## Architecture Notes

### Checkpointer Approach Decision

**Native `PostgresSaver` (preferred if dependency compatible):**
- `PostgresSaver.setup()` auto-manages migrations via internal `checkpoint_migrations` table
- No manual SQL migration needed for checkpoint tables
- Thread ID model is LangGraph-native (`thread_id` string, `checkpoint_ns` namespace)
- Add to package.json: `pnpm add @langchain/langgraph-checkpoint-postgres --filter @repo/orchestrator`
- Risk: version compatibility with `@langchain/langgraph ^0.2.0` must be verified before committing

**Custom `BaseCheckpointSaver` (fallback if native incompatible):**
- Implement `get()`, `put()`, `list()` against `wint.workflow_checkpoints`
- Use `WorkflowRepository` pattern: `client.query()` with raw SQL, `sql` tag for enum predicates
- Requires new migration `0033_wint_9106_checkpoint_columns.sql` if adding `retry_count`/`error_context` columns
- `state` JSONB column already exists — retry/error context can live inside JSONB without migration

### Connection Pool Constraint

`@repo/db` uses `max: 1` per Lambda. Checkpoint writes must not create a separate pool. If synchronous write-per-node causes p99 > 50ms (measured in ED-1 benchmark), implement configurable async checkpointing (fire-and-forget Promise queue) for non-critical nodes. Side-effecting nodes (AC-004 rollback intent) must remain synchronous.

### State Serialization

`StoryCreationStateAnnotation` extends `GraphStateSchema` (`packages/backend/orchestrator/src/state/graph-state.ts`). All fields must be JSON-serializable. LangGraph state channels with non-serializable reducers (e.g., `Annotation.Root`) must be serialized via the checkpointer's `serde` (serializer/deserializer) interface.

### resume_from Phase Mapping (AC-008)

Legacy `CHECKPOINT.yaml` uses `resume_from: N` (integer phase number). The new module must export:

```typescript
// packages/backend/orchestrator/src/checkpointer/phase-mapping.ts
export const PHASE_TO_CHECKPOINT_MAP: Record<number, string> = {
  1: 'seed-generation',
  2: 'elaboration',
  3: 'feasibility-review',
  // ... map phase integers to LangGraph node names
}
```

Document this mapping in module JSDoc for dev agents consuming this story.
