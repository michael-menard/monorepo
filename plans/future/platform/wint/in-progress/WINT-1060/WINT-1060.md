# WINT-1060: Update /story-move Command to Write Status to Database

## Context

The `/story-move` command (`.claude/commands/story-move.md` v2.0.0) currently moves a story directory between swim-lane folders (`backlog/`, `ready-to-work/`, `in-progress/`, etc.) as its sole mechanism for recording story status changes. This is the filesystem-only model that the WINT project is migrating away from.

WINT-1011 delivered the compatibility shim module (`shimGetStoryStatus`, `shimUpdateStoryStatus`, `shimGetStoriesByStatus`, `shimGetStoriesByFeature`) at `packages/backend/mcp-tools/src/story-compatibility/`. This shim is UAT-verified (35 unit + 10 integration tests, PASS). WINT-1030 populated the `core.stories` database table from the current directory state (UAT-verified: 53 unit tests, PASS). The shim now provides a stable integration point for reading and writing story status to the database with directory fallback.

This story is the third of three sibling command-update stories in Phase 1 (alongside WINT-1040 for `/story-status` and WINT-1050 for `/story-update`). It updates `/story-move` to:
1. Attempt DB-first story locate via `shimGetStoryStatus` (with directory fallback).
2. Write the new status to the database via `shimUpdateStoryStatus` BEFORE the directory move.
3. Continue performing the directory `mv` operation for backward compatibility — unmigrated agents still read the filesystem.
4. If the DB write returns null (DB unavailable), log a warning and proceed with directory move only (fail-safe degradation).
5. Extend the command's return YAML with a `db_updated` field reflecting the DB write outcome.

The `--update-status` flag path (which delegates to `/story-update`) is not modified — WINT-1050 owns that command independently.

---

## Goal

Augment `/story-move` with a database write step that updates `core.stories` before executing the directory `mv`, maintaining full backward compatibility for unmigrated agents by preserving the directory move. This is a Phase 1 transitional change — both the DB record and the filesystem directory are updated on the happy path. Phase 7 (WINT-7030) will remove the directory-move step when all agents are migrated.

## Architecture Notes

### SWIM_LANE_TO_STATE Mapping (Exact — from `__types__/index.ts`)

The following is the authoritative mapping. All agents executing this command MUST use this table, not hardcoded values:

| TO_STAGE argument | DB state (`newState`) | DB-mapped? |
|-------------------|-----------------------|------------|
| `backlog` | `backlog` | YES |
| `ready-to-work` | `ready_to_work` | YES |
| `in-progress` | `in_progress` | YES |
| `ready-for-qa` | `ready_for_qa` | YES |
| `UAT` | `in_qa` | YES |
| `created` | — | NO — skip DB write, proceed with mv |
| `elaboration` | — | NO — skip DB write, proceed with mv |
| `needs-code-review` | — | NO — skip DB write, proceed with mv |
| `failed-code-review` | — | NO — skip DB write, proceed with mv |
| `failed-qa` | — | NO — skip DB write, proceed with mv |

Note: `done` is in SWIM_LANE_TO_STATE but is not a valid stage target for `/story-move`. The QA completion flow handles `done` separately.

### Step Ordering Rationale

DB write happens BEFORE directory mv (Step 2.5 position):
1. Ensures the move is validated before writing to DB (if target directory already exists, we don't write a stale state to DB)
2. If DB write fails (returns null), directory mv still proceeds — this is the fail-safe path
3. If directory mv fails after DB write succeeds, the DB record may be ahead of the filesystem — this is acceptable in Phase 1 (shim's directory fallback compensates)

### --update-status Coordination with WINT-1050

When `--update-status` is provided:
- Step 2.5 DB write is **skipped** (`db_updated: skipped`)
- The command delegates to `/story-update` as before
- WINT-1050 makes `/story-update` DB-aware, so the DB write happens via that path
- This prevents double-write and ensures a single code path owns the DB status for each flag variant

### Backward Compatibility Guarantee

Phase 1 constraint: both DB AND directory are always updated on the happy path. Unmigrated agents read the filesystem; migrated agents read the DB. Both are kept in sync during Phase 1. The `db_updated` field enables future diagnostics without changing external behavior.
