# WINT-4100: Create backlog-curator Agent — Deferred Item Collector and PM Review Surfacer

## Context

The WINT Phase 4 elaboration pipeline generates deferred items through multiple pathways:

- The **scope-defender agent** logs `defer-to-backlog` recommendations in `scope-challenges.json` during story elaboration
- **Agents following the decision-handling protocol** write deferred/moonshot items to KB with tags `["deferred", "moonshot"]` via `kb_add_lesson`
- The **graph-checker agent** (WINT-4060, in elaboration) detects cohesion violations it explicitly does not act on — those remain un-curated until a collector runs

These items accumulate across KB entries and scattered filesystem artifacts but have no automated collection or PM review step. PMs must manually scan agent outputs across multiple story directories to find work that was intentionally deferred — an error-prone, time-consuming process that results in valid deferred work being silently lost between pipeline runs.

No agent is currently responsible for: collecting deferred items across sources, deduplicating them, ranking by recency or impact, and presenting them to the PM in a structured format for backlog grooming decisions.

**Baseline**: 2026-02-13. Baseline predates WINT-4060 (graph-checker agent), WINT-4070 (cohesion-prosecutor), and all Phase 4 feature-cohesion infrastructure. Deferred item sources now exist in agent files and KB entries tagged "deferred".

---

## Goal

Create a `backlog-curator` haiku-powered worker agent that collects deferred items from KB and optional filesystem sources, deduplicates and ranks them, and emits a bounded PM review batch so deferred work surfaces for backlog grooming decisions rather than silently disappearing.

---

## Examples

**Example 1 — Happy path KB query:**
Given the curator runs with `scope: "all"` and KB contains 15 entries tagged `["deferred"]` across various stories, the agent queries KB, deduplicates 3 items that appear in both KB and a `scope-challenges.json` file, and emits `pm-review-batch.json` with the top 10 items (truncated: true) sorted by recency, plus a `PM-REVIEW-REPORT.md` with one section per item showing description, deferral reason, and recommended action (promote-to-story / close / defer-again).

**Example 2 — Scoped run:**
Given the curator runs with `scope: "WINT-4060"` and KB contains 3 entries tagged `["deferred"]` for that story, the agent produces `pm-review-batch.json` with 3 items (truncated: false) and `PM-REVIEW-REPORT.md` scoped to only WINT-4060 deferred items.

**Example 3 — Graceful zero-item run:**
Given the curator runs with `scope: "all"` but KB returns zero entries tagged `["deferred"]` and no `scope-challenges.json` files are found, the agent emits `pm-review-batch.json` with `total_items_found: 0`, `items_in_batch: 0`, and emits `BACKLOG-CURATOR COMPLETE` (not blocked).

---

## Edge Cases

**Edge Case 1 — KB unavailable:**
When KB MCP tools are down (network failure, service unavailable), the agent falls back to scanning `scope_challenges_dir` for `scope-challenges.json` files with `recommendation: defer-to-backlog`. If both KB and filesystem scan fail, the agent emits `BACKLOG-CURATOR COMPLETE WITH WARNINGS: 1 warnings` (not blocked) — curating what it can.

**Edge Case 2 — Duplicate across sources:**
When the same deferral note appears in both a KB entry tagged `["deferred"]` and a `scope-challenges.json` file for the same story (e.g., "add audit trail for deferrals" logged in both sources for WINT-4050), the agent deduplicates using a key of `(story_id + description_hash)` and keeps a single item in the output. The `source` field reflects both origins: `"kb+scope-challenges"`.

**Edge Case 3 — Batch overflow:**
When 25 items qualify but the configurable limit is 10 (default), the agent includes the 10 most recent items, sets `truncated: true`, and records `total_items_found: 25` and `items_in_batch: 10` in `pm-review-batch.json`. The PM-REVIEW-REPORT.md notes the truncation at the top.

**Edge Case 4 — WINT-4060 not yet run:**
When `scope_challenges_dir` is provided but contains no `scope-challenges.json` files (because graph-checker has not run yet), the agent skips the filesystem scan and continues with KB results only. No warning is emitted for this condition — it is the expected state until WINT-4060 ships.

## Architecture Notes

**KB-first design:** The curator queries KB before falling back to filesystem. This aligns with MEMORY.md's directive that KB is the source of truth. Filesystem scan (`scope_challenges_dir`) is an optional supplement, not the primary path.

**Bounded output:** The 10-item default cap on the PM review batch mirrors the scope-defender's 5-challenge cap — both are designed to prevent reviewer fatigue. The cap is configurable via `batch_limit` input.

**Deduplication identity key:** `(story_id + SHA-like hash of deferral description)` — not a full content hash, but a stable identifier derived from the two most discriminating fields. If the same concept appears in KB and in a scope-challenges.json for the same story, they merge to one item with `source: "kb+scope-challenges"`.

**DEFERRED-KB-WRITES.yaml resilience:** Some pipeline runs cannot call KB MCP tools (connectivity issues), resulting in pending write files. The curator's optional `deferred_writes_dir` input handles this: it scans for `DEFERRED-KB-WRITES.yaml` files and includes their `pending_writes` entries in the deferred item collection step.

**Agent model:** haiku — sufficient for document scanning and structured output generation. No reasoning-heavy tasks required.
