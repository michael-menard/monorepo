# WINT-4150: Standardize Elab Output Artifacts — Define Schemas and Enforce at Elab-Complete Gate

## Context

The WINT Phase 4 elaboration workflow produces several machine-readable JSON artifacts from its adversarial agents:
- `cohesion-findings.json` (from cohesion-prosecutor, WINT-4070 — ready-for-qa)
- `scope-challenges.json` (from scope-defender, WINT-4080 — UAT)
- `final-scope.json` (from round-table, WINT-4140 — needs-code-review, uses `schema_version: '1.0-draft'` explicitly awaiting this story)

Plus earlier-phase artifacts:
- `user-flows.json` (already has `UserFlowsSchema` in `packages/backend/orchestrator/src/artifacts/__types__/user-flows.ts`)
- `mvp-slice.json`, `evidence-expectations.json`, `gaps.json` (informal contracts in agent markdown files)

None of the Phase 4 artifacts have formal Zod schemas in the orchestrator artifact package. They exist only as informal contracts documented inside agent `.md` files. This means downstream consumers (round-table agent, future LangGraph nodes in WINT-4120) cannot do schema-validated reads, the elab-complete gate has no machine-enforceable definition of "done", and artifact shape drift is undetectable until runtime failures.

**Dependency**: WINT-4140 (round-table agent) must reach ready-to-work or later before WINT-4150 implementation begins, as its provisional `final-scope.json` schema (`schema_version: '1.0-draft'`) is what WINT-4150 formalizes into `schema_version: '1.0'`.

## Goal

Formalize all elab output artifact schemas as Zod schemas in `packages/backend/orchestrator/src/artifacts/` and add soft artifact validation to the elab-complete gate in `elab-completion-leader.agent.md`, enabling downstream consumers to do schema-validated reads and providing a machine-enforceable definition of elaboration completion.

## Examples

**Example 1 — Schema-validated cohesion findings read:**
Given a `cohesion-findings.json` file written by the cohesion-prosecutor agent with 3 findings (1 blocking), a downstream consumer (round-table agent or LangGraph node) runs `CohesionFindingsSchema.parse(fileContents)`. The parse succeeds and the consumer gets a strongly-typed object with `findings: CohesionFinding[]` (max 5) and `blocking_findings: CohesionFinding[]` (max 2). No defensive null-checks needed.

**Example 2 — Constraint enforcement catches agent output errors:**
Given a buggy cohesion-prosecutor run that emits `blocking_findings` with 3 items (exceeding the max 2 contract), running `CohesionFindingsSchema.parse(fileContents)` throws a ZodError with a clear message: `"blocking_findings: Array must contain at most 2 element(s)"`. The round-table agent surfaces this as an artifact validation error rather than silently consuming malformed data.

**Example 3 — final-scope schema_version upgrade:**
Given a `final-scope.json` produced by WINT-4140's round-table agent using `schema_version: '1.0-draft'`, after WINT-4150 ships the `FinalScopeSchema` requires `z.literal('1.0')`. Implementation of round-table updates the output to emit `schema_version: '1.0'`. Consumers parsing with the new schema get compile-time and runtime assurance of the stable contract.

## Edge Cases

**Edge Case 1 — Elab run predating WINT-4150 (graceful gate degradation):**
A story elaborated before WINT-4150 ships has only `ELAB.yaml` in its `_implementation/` directory — no `cohesion-findings.json`, `scope-challenges.json`, etc. The elab-complete gate added by AC-10 MUST emit warnings for each absent artifact but MUST NOT block the PASS verdict. Retroactive blocking would invalidate all prior elaborations.

**Edge Case 2 — gaps array empty (no blocking gaps found):**
An elab run where the audit produces no gaps results in `gaps.json` with `gaps: []`. The `GapsSchema` array is intentionally unbounded (`z.array(GapItemSchema)` with no `.max()`), unlike cohesion-findings and scope-challenges. An empty array is valid and common.

**Edge Case 3 — UserFlowsSchema re-export collision:**
If a dev naively creates a new `user-flows.ts` in the artifacts root alongside the existing `__types__/user-flows.ts`, two schema definitions diverge. AC-3 explicitly prevents this: the artifacts index must re-export the existing `UserFlowsSchema` from `__types__/user-flows.ts`. No new schema file for user-flows is created.

## Non-Goals

- Do NOT create an API endpoint for elab artifacts — these are filesystem-only artifacts
- Do NOT modify the existing ELAB.yaml schema (`.claude/schemas/elab-schema.md`) — protected golden reference
- Do NOT redefine `UserFlowsSchema` — reference the existing one at `packages/backend/orchestrator/src/artifacts/__types__/user-flows.ts`
- Do NOT make the elab-complete gate blocking for stories that lack the new artifacts — soft validation (warn, do not block) until upstream agents catch up
- Do NOT add a schema for `scope-challenges.json` that conflicts with scope-defender's existing output contract
- Do NOT make breaking changes to existing orchestrator artifacts (EvidenceSchema, ReviewSchema, QaVerifySchema, etc.)
- Do NOT create a LangGraph node (deferred to WINT-4120)
- Do NOT add agent assignment to `model-assignments.yaml`
- Protected: all production DB schemas in `packages/backend/database-schema/`, `@repo/db` client package API surface, Umami analytics schema

## Architecture Notes

### Schema Design Principles

1. **`schema_version` as first field** — all new JSON artifact schemas must have `schema_version` as the first field per `.claude/schemas/elab-schema.md` Validation Rules. This supports future migration tooling.

2. **Bounded arrays at schema level** — `z.array(...).max(N)` enforced in Zod, not just documentation. This makes violations discoverable at parse time, not runtime.

3. **`z.literal()` for version strings** — `final-scope.json` uses `z.literal('1.0')` to hard-pin the schema version, distinguishing it from the WINT-4140 draft (`'1.0-draft'`). Downstream consumers can assert schema stability.

4. **Additive-only index.ts changes** — all exports added to end of existing section groups; no existing exports modified or removed.

5. **No barrel files** — schemas export directly from their `.ts` files; no intermediate `schemas/index.ts` created.
