## Context

WINT-1011 delivers the four core shim functions (`shimGetStoryStatus`, `shimUpdateStoryStatus`, `shimGetStoriesByStatus`, `shimGetStoriesByFeature`) at `packages/backend/mcp-tools/src/story-compatibility/`. This story (WINT-1012) adds two capabilities on top of the core shim:

1. **ShimDiagnostics**: An opt-in observability field that callers can enable to know the data source (`db | directory | not_found`) for each shim call. Critical for monitoring migration completeness during the Phase 1–7 migration window.

2. **Test coverage**: A comprehensive unit test suite achieving minimum 80% coverage across all four test paths, above the global 45% floor. Without this coverage requirement, the shim's correctness under edge conditions (DB unavailable, directory fallback, invalid input) cannot be assured.

Both capabilities depend on the core shim from WINT-1011 being complete before they can be implemented and verified.

**Reality Baseline**: 2026-02-13 baseline active. WINT-0090 and WINT-1030 both passed QA, confirming stable function signatures and swim-lane state mapping.

## Architecture Notes

### ShimDiagnostics Schema

```typescript
// packages/backend/mcp-tools/src/story-compatibility/__types__/index.ts
import { z } from 'zod'

const ShimDataSourceSchema = z.enum(['db', 'directory', 'not_found'])

const ShimDiagnosticsSchema = z.object({
  source: ShimDataSourceSchema,
  queried_at: z.string().datetime(),
})

// Extend ShimOptionsSchema from WINT-1011 to include diagnostics flag:
const ShimOptionsSchema = z.object({
  storiesRoot: z.string().optional(),
  diagnostics: z.boolean().optional().default(false),
})

type ShimDataSource = z.infer<typeof ShimDataSourceSchema>
type ShimDiagnostics = z.infer<typeof ShimDiagnosticsSchema>
type ShimOptions = z.infer<typeof ShimOptionsSchema>
```

### Function Signature Extension

The four core shim functions from WINT-1011 accept an optional `options` parameter. WINT-1012 adds `diagnostics: boolean` to that options object. When `diagnostics: true`, each function appends a `_diagnostics` field to its return value:

```typescript
// Without diagnostics (default):
shimGetStoryStatus('WINT-1011')
// → { id: 'WINT-1011', status: 'backlog', ... }

// With diagnostics enabled:
shimGetStoryStatus('WINT-1011', { diagnostics: true })
// → { id: 'WINT-1011', status: 'backlog', ..., _diagnostics: { source: 'db', queried_at: '2026-02-16T00:00:00Z' } }
```

The `_diagnostics` field is prefixed with `_` to signal it is observability metadata, not part of the canonical output contract. Callers that do not opt in receive no performance or structural overhead.

### Known Future Extension

`ShimDiagnostics` currently captures `source` and `queried_at` but not `fallback_reason` (`db_miss` vs `db_unavailable` vs `empty_result`). This distinction is valuable during migration analysis but excluded from WINT-1012 to keep scope tight. Backwards-compatible extension: add `fallback_reason: z.enum(['db_miss', 'db_unavailable', 'empty_result']).optional()` in Phase 2/3 when Phase 1 telemetry reveals the value of this distinction.

## Risks / Edge Cases

1. **WINT-1011 must be complete first**: ShimDiagnostics cannot be implemented or tested without the core shim functions. Dependency on WINT-1011 is hard.

2. **CI integration test silent skip**: If `DATABASE_URL` is not configured in CI, integration tests silently skip rather than fail with a clear error. Mitigate: add CI pre-flight assertion as part of this story's test infrastructure.

3. **_diagnostics field and output schema conformance**: The `_diagnostics` field must not break WINT-0090 output schema conformance (AC-5 from WINT-1011). Ensure `_diagnostics` is additive and callers without diagnostics enabled receive unchanged output types.

4. **Migration progress telemetry not covered**: ShimDiagnostics captures per-call source but has no structured way to report aggregate migration progress (% of lookups hitting DB vs directory). This metric is deferred to Phase 3 (WINT-3020/WINT-3070).
