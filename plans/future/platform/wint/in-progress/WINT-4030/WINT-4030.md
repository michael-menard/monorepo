# WINT-4030: Populate Graph with Existing Features and Epics

## Context

The WINT graph relational schema was created in WINT-0060 and extended in WINT-0131, but the `graph.features` and `graph.capabilities` tables are empty. The `graph.epics` table does not yet exist. The graph-checker agent (WINT-4060) and cohesion-prosecutor agent (WINT-4070) both depend on this graph being populated before they can produce meaningful results — without data, cohesion checks vacuously pass and provide no insight.

This story closes the gap by:
1. Adding the `graph.epics` table to the schema (Drizzle definition + migration `0036`)
2. Delivering a re-runnable population script that discovers current features from the codebase and inserts seed epics into the graph

**Baseline:** WINT-0060 UAT PASS (graph schema deployed), WINT-0131 UAT PASS (featureId FK on capabilities available), WINT-4010 in needs-code-review, WINT-4020 ready-to-work.

---

## Goal

Create the `graph.epics` table and populate `graph.features` and `graph.epics` with an initial baseline of features discovered from the monorepo codebase and known epics, so that graph-based cohesion checks can operate on real data.

---

## Examples

**Given**: The monorepo has `apps/api/knowledge-base/src/handlers/`, `apps/web/main-app/src/components/`, and `packages/backend/mcp-tools/src/`; the known epics are WINT, KBAR, WISH, BUGF.

**The system produces**: `graph.epics` rows for WINT, KBAR, WISH, BUGF; `graph.features` rows for each discovered handler directory (featureType: `api_endpoint`), each component directory (featureType: `ui_component`), and each backend package `src/` directory (featureType: `service` or `utility`). The script logs `{ attempted: N, succeeded: N, failed: 0 }` for both epics and features.

**Given**: The script is run a second time on the same DB with the same input.

**The system produces**: Identical row counts in `graph.epics` and `graph.features`. All inserts hit the `onConflictDoNothing` path. The script exits cleanly with `{ succeeded: 0, failed: 0 }` for duplicate rows, logging at `logger.warn` level for skips.

---

## Edge Cases

**Edge case 1 — Single insert failure does not abort the run**: If the DB insert for one feature throws (e.g., network error, constraint violation outside featureName), the script catches the error per-item, logs at `logger.error`, increments `failed`, and continues to the next item. The final result accurately reflects partial success.

**Edge case 2 — No directories discovered**: If the monorepo root is wrong or all scan paths return empty directory listings, the script returns `{ features: { attempted: 0, succeeded: 0, failed: 0 } }` without throwing. The epics inserts still proceed.

**Edge case 3 — Duplicate featureName in scan output**: If the directory scanner produces the same path twice (e.g., symlinks), `onConflictDoNothing` on the `features_feature_name_idx` unique index silently absorbs the second insert. Row count is unaffected.

**Edge case 4 — Migration not yet applied**: If the developer runs the script before applying the migration, the INSERT to `graph.epics` fails with a "relation does not exist" error. The error is caught and surfaced via `logger.error`; features inserts (which target the existing `graph.features` table) continue normally.

## Architecture Notes

- The `graph.epics` table follows the same Drizzle/Zod-first pattern as `graph.features` (lines 1257–1302 of `wint.ts`)
- `epicPrefix` (e.g., `'WINT'`) is a unique text field that can be used to join with `kbar.stories.epic` text column for cross-schema queries in future stories
- The population script uses injectable deps (pass functions as parameters) per the `populate-domain-kb.ts` pattern — this enables full unit testability without a live DB in CI (ADR-005 compliance)
- `onConflictDoNothing` is used (not `onConflictDoUpdate`) because epics and features are stable identifiers — if they need updating, a separate migration or manual correction is preferred over silent overwrites
- Migration numbering: `0035_wint_4020_rules_registry.sql` is the current latest → new migration is `0036_wint_4030_graph_epics.sql`
