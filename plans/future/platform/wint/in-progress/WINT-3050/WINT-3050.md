# WINT-3050: Implement Outcome Logging

Add `workflow_log_outcome` calls at story completion points so that every QA verdict (PASS/FAIL) and cancellation/block event persists a structured outcome record in `wint.story_outcomes`.

---

## Context

The wint platform's Phase 3 telemetry infrastructure includes two prerequisites that are either complete or nearly complete:

- **WINT-0040** (ready-for-qa): Provides the `wint.story_outcomes` table via migration `0032_wint_0040_hitl_decisions_story_outcomes.sql`. The Drizzle schema at `packages/backend/database-schema/src/schema/wint.ts:997` exposes `storyOutcomes` with `finalVerdict`, `qualityScore`, `totalInputTokens`, `totalOutputTokens`, `totalCachedTokens`, `estimatedTotalCost`, `reviewIterations`, `qaIterations`, `durationMs`, `primaryBlocker`, and `completedAt`.
- **WINT-0120** (Ready to Work): Provides the `workflow_log_outcome` MCP tool. WINT-3050 has a **hard dependency** on WINT-0120 reaching `needs-code-review` before implementation can begin.

Currently, no workflow agent or skill calls `workflow_log_outcome` at story completion. As a result, `wint.story_outcomes` remains empty even as stories are completed, blocking WINT-3090 (Scoreboard Metrics) and Phase 5 ML prediction which requires a populated training dataset.

Sibling story WINT-3020 (telemetry-log skill, in `needs-code-review`) establishes the call-site injection pattern at agent level; WINT-3050 applies the same approach at story-completion level.

---

## Goal

Populate `wint.story_outcomes` with structured outcome records at every story completion point by injecting non-blocking `workflow_log_outcome` MCP tool calls into the QA verdict flow and at least one additional completion path (dev-phase cancellation or block).

---

## Examples

**Given** a story WINT-TEST-001 completes with a PASS verdict (0 review iterations, 1 QA iteration):
- `qa-verify-completion-leader` calls `workflow_log_outcome({ storyId: "WINT-TEST-001", finalVerdict: "pass", qualityScore: 85, qaIterations: 1, reviewIterations: 0, completedAt: <now> })`
- A row is upserted into `wint.story_outcomes` with `final_verdict='pass'` and `quality_score=85`
- The QA verdict flow continues uninterrupted to `story_update`

**Given** a story WINT-TEST-002 completes with a FAIL verdict, primary blocker "AC-3 missing implementation" (3 review iterations, 2 QA iterations):
- `qa-verify-completion-leader` calls `workflow_log_outcome({ storyId: "WINT-TEST-002", finalVerdict: "fail", qualityScore: 0, reviewIterations: 3, qaIterations: 2, primaryBlocker: "AC-3 missing implementation", completedAt: <now> })`
- A row is upserted with `final_verdict='fail'`, `quality_score=0` (clamped), `primary_blocker='AC-3 missing implementation'`

---

## Edge Cases

**MCP call failure**: If `workflow_log_outcome` returns null or throws, the agent must emit `logger.warn` and continue. The story completion flow (including `story_update`) must not be interrupted or rolled back due to outcome logging failure.

**Quality score floor**: With `reviewIterations=15` and `qaIterations=10`, the raw formula `100 - (15*10) - (10*15) = -200` must be clamped to `max(0, ...)` = 0. Negative quality scores are invalid and must never reach the DB (enforced by `insertStoryOutcomeSchema.refine()`).

**Upsert idempotency**: If `workflow_log_outcome` is called twice for the same story (e.g., partial then final completion), the second call must update the existing row rather than insert a duplicate. This is handled by `ON CONFLICT (story_id) DO UPDATE` inside the tool — implementers must not add application-level duplicate guards.

**Missing optional fields**: `totalInputTokens`, `totalOutputTokens`, `estimatedTotalCost`, and `durationMs` may not be available in all completion contexts. Omit them from the call when unavailable; the schema allows nullable values for all these fields.

## Quality Score Formula

The `qualityScore` field (integer 0-100) is computed using a heuristic formula derived from observable completion signals:

```
qualityScore = max(0, 100 - (reviewIterations * 10) - (qaIterations * 15))
```

**Rationale:**
- Each review iteration (dev-QA loop) deducts 10 points (moderate penalty: rework is costly but recoverable)
- Each QA iteration (QA re-verification cycle) deducts 15 points (higher penalty: QA churn is a stronger quality signal)
- Floor of 0 enforced by `max(0, ...)` and by `insertStoryOutcomeSchema.refine()` (range 0-100)
- A story with 0 review and 0 QA iterations scores 100 (clean first-pass delivery)
- A story with 5 review and 3 QA iterations scores `max(0, 100-50-45) = 5` (near-failing quality)

This formula is consistent with the MODL-0040 lesson: heuristic, deterministic, explainable, zero training data required.

## Architecture Notes

**Fire-and-forget pattern** (required for all outcome log calls):
```
try:
  result = workflow_log_outcome({ storyId, finalVerdict, qualityScore, ... })
  if result is null:
    logger.warn("WINT-3050: outcome log returned null for {storyId}")
catch e:
  logger.warn("WINT-3050: outcome log failed for {storyId}: {e.message}")
// continue story completion flow regardless
```

**Upsert semantics**: `workflow_log_outcome` internally uses `ON CONFLICT (story_id) DO UPDATE` — no application-level deduplication needed. Safe to call multiple times per story.

**qualityScore is computed at call-site**: The agent must compute the score using the AC-1 formula from observable context (`reviewIterations`, `qaIterations`) before passing it to the tool. The tool does not compute it.

**WINT-0120 gate**: Implementation subtask ST-1 must read the final `workflow_log_outcome` parameter contract from the merged WINT-0120 story file before any call-site code is written. Parameter names may differ from the seed's suggestions.
