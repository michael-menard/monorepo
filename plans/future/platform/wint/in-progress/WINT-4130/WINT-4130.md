# WINT-4130: Validate Graph & Cohesion System — Detection Verification Against Known Franken-Features

## Context

The Phase 4 graph & cohesion system consists of: a graph database (`graph.features`, `graph.epics`, `graph.feature_capabilities`), a graph population script (WINT-4030), capability inference (WINT-4040), cohesion rules (WINT-4050), the graph-checker agent (WINT-4060), the cohesion-check command (WINT-4110), and the backlog-curator agent (WINT-4100). As of 2026-03-08, these components are in various states of completion (needs-code-review, failed-code-review, ready-to-work). This story is the integration validation — the first time the whole system is exercised end-to-end against real codebase data.

No end-to-end validation has been run against the graph & cohesion system to confirm that it actually detects the franken-feature patterns it was designed for. The two canonical test cases — "upload without replace" and "create without delete" — are known patterns in the existing codebase, but it has never been confirmed that the graph-checker agent detects them when operating against real graph data populated from the monorepo.

**Dependency chain**: The formal index dependency is WINT-4100, but the actual runtime dependencies for meaningful validation are: WINT-4060 (graph-checker agent), WINT-4030 (graph populated), WINT-4050 (cohesion rules seeded), and WINT-4040 (capability inference — currently in failed-code-review). WINT-4130 is designed to produce partial results with documented gaps when prerequisites are missing rather than blocking on any single prerequisite.

---

## Goal

Verify that the graph-checker agent correctly detects the known "upload without replace" and "create without delete" franken-feature patterns when invoked against real monorepo graph data, and document all findings (including degradation behavior and defects) in structured EVIDENCE.yaml.

---

## Examples

**Given** the graph is populated (WINT-4030 has run) and at least one active cohesion rule exists (WINT-4050 seeded), **when** the graph-checker agent is invoked, **then** `graph-check-results.json` contains `franken_features_found >= 2` and `violations` includes entries for the upload-without-replace and create-without-delete fixture features with a valid `rule_id` and non-empty `actionable_hint`.

**Given** a feature ID that does not exist in `graph.features`, **when** graph-checker is invoked scoped to that feature, **then** it produces `graph-check-results.json` with `violations: []`, `warning_count: 1`, and `warnings: ["graph.features empty — WINT-4030 may not have run"]`, and emits the completion signal `GRAPH-CHECKER COMPLETE WITH WARNINGS: 1 warnings`.

---

## Edge Cases

**WINT-4040 capabilities absent (BUILTIN mode)**: If `graph.feature_capabilities` has no rows for the target features (because WINT-4040 failed code review and was not run), `graph_get_franken_features` returns `[]` due to the inner join. In this case validation falls back to BUILTIN-CRUD-COMPLETENESS structural verification and documents `capability_mode: "builtin"` in EVIDENCE.yaml — the story does NOT block.

**Empty rules registry**: If `rulesRegistryGet({ status: 'active' })` returns `[]` (WINT-4050 rules not seeded), the graph-checker is expected to emit the warning `"No active cohesion rules found — WINT-4050 may not have run"` and fall back to `BUILTIN-CRUD-COMPLETENESS`. Validation confirms this fallback fires correctly rather than silently failing.

## Acceptance Criteria

- [ ] **AC-1 (Pre-flight)**: Validation script or manual checklist confirms: (a) `graph.features` has at least one row for known franken-feature candidates; (b) `rulesRegistryGet({ status: 'active' })` returns at least one active rule; (c) `graph.feature_capabilities` has rows for the target features. Pre-flight state documented in EVIDENCE.yaml.
- [ ] **AC-2 (Known fixture — upload without replace)**: At least one real feature in the monorepo is identified with an "upload" capability but lacking a "replace" capability. This feature is confirmed present in `graph.features` with the upload capability recorded in `graph.feature_capabilities`.
- [ ] **AC-3 (Known fixture — create without delete)**: At least one real feature in the monorepo is identified with a "create" capability but lacking a "delete" capability. This feature is confirmed present in `graph.features` with the create capability recorded.
- [ ] **AC-4 (Detection fires — graph-checker produces violations)**: Invoking the graph-checker (via `/cohesion-check` or direct agent invocation) produces a `graph-check-results.json` where `franken_features_found > 0` and `violations` contains at least one entry for each known franken-feature from AC-2 and AC-3.
- [ ] **AC-5 (Output schema conformance)**: The produced `graph-check-results.json` conforms to the schema documented in `graph-checker.agent.md`: fields `story_id`, `generated_at`, `franken_features_found`, `violations` (array), `warnings` (array), `warning_count` are all present and correctly typed.
- [ ] **AC-6 (Rule-specific detection)**: Each violation entry in `graph-check-results.json` contains a valid `rule_id` (either a rule from the active registry or `"BUILTIN-CRUD-COMPLETENESS"`) and a non-empty `actionable_hint` per the agent schema.
- [ ] **AC-7 (Graceful degradation — empty graph)**: Validation confirms that when graph-checker is invoked against a scoped context with no graph data (e.g., a feature ID not in the graph), it emits `GRAPH-CHECKER COMPLETE WITH WARNINGS: 1 warnings` with warning `"graph.features empty — WINT-4030 may not have run"` and `violations: []`.
- [ ] **AC-8 (Graceful degradation — empty rules)**: Validation confirms that when cohesion rules are absent, graph-checker emits `GRAPH-CHECKER COMPLETE WITH WARNINGS` with the appropriate warning and falls back to `BUILTIN-CRUD-COMPLETENESS`. (May be structural verification of agent spec if DB write access not available.)
- [ ] **AC-9 (Completion signal)**: The graph-checker agent emits exactly one completion signal as its final output line, matching one of: `GRAPH-CHECKER COMPLETE`, `GRAPH-CHECKER COMPLETE WITH WARNINGS: {N} warnings`, or `GRAPH-CHECKER BLOCKED: {reason}`.
- [ ] **AC-10 (Evidence captured)**: All validation outcomes (pre-flight state, fixture confirmation, detection results, schema conformance, degradation results) are recorded in `EVIDENCE.yaml` under an `e2e_tests` section per ADR-006 schema. Defects found are listed under `known_deviations`.
- [ ] **AC-11 (Defect triage)**: Any detection failures (e.g., known franken-feature not detected, output schema mismatch, wrong rule ID) are documented as discrete findings in EVIDENCE.yaml. A recommendation is made for each finding: fix in WINT-4060, fix in WINT-4030, fix in WINT-4050, or defer. No inline fixes are made.

## Architecture Notes

- ADR-001 (ARCH-001 direct-call pattern): All MCP tool invocations use direct TypeScript function imports, not HTTP calls to localhost
- ADR-005: Validation must connect to real DB (dev/staging); no MSW mocking of graph DB
- ADR-006: At least one happy-path run must execute against live resources and be recorded in EVIDENCE.yaml `e2e_tests`
- ADR-002: No ad-hoc schema changes; read-only access only
- `graph_get_franken_features` inner-joins `features` with `capabilities` using `eq(capabilities.featureId, features.id)` — if no capability rows exist for a feature, it will not appear in results. Pre-flight must verify this.
- Degradation behavior is defined in the graph-checker agent spec: empty graph increments `warning_count`, empty rules increments `warning_count`; both paths continue gracefully
