# WINT-3020: Implement Invocation Logging Skill (`telemetry-log`) with `workflow_log_invocation` MCP Tool

## Context

Every agent spawn in the WINT workflow system currently has no persistent record. Leaders spawn workers, workers execute phases, and tokens/latency/outcomes are lost after the session ends. This story is the **Phase 3 telemetry foundation** — deferred here from earlier phases (per WINT-2050 lesson) specifically because the infrastructure (tables from WINT-0040, MCP tooling) now exists.

The `wint.agent_invocations` table is defined in `packages/backend/database-schema/src/schema/wint.ts` (line 658) and is in `ready-for-qa` status via WINT-0040. The `session-create.ts` pattern provides the exact insert model. The `token-log` skill provides the exact skill document template. All canonical references are in place.

**Active dependency note**: WINT-0120 (Create Telemetry MCP Tools) is `pending` as of seed generation. Per the conflict resolution hint, this story may pull the `workflow_log_invocation` MCP tool scope forward — making WINT-0120 partially redundant for that tool, but retaining WINT-0120's remaining scope (`workflow_log_decision`, `workflow_log_outcome`, `workflow_get_story_telemetry`).

---

## Goal

Provide every WINT workflow agent with a fire-and-forget skill (`telemetry-log`) that persists a structured invocation record to `wint.agent_invocations`, enabling Phase 3 observability, cost analysis, and downstream ML training.

---

## Examples

**Example 1 — Successful invocation at end of execution phase:**

Given an agent (`pm-bootstrap-workflow`) completes its execute phase for story WINT-3020 with 1,200 input tokens, 400 output tokens, and 4.2s duration, when the agent calls the `telemetry-log` skill with `status: success`, then one row is inserted into `wint.agent_invocations` with all provided fields populated and `created_at` set to the current timestamp.

**Example 2 — Minimal call with only required fields:**

Given an agent calls the `telemetry-log` skill with only `agentName: 'qa-verify-story'`, `status: 'failure'`, and `invocationId: 'qa-uuid-001'`, then one row is inserted with all optional fields (`storyId`, `phase`, `inputTokens`, etc.) set to null and `cachedTokens` defaulting to 0.

---

## Edge Cases

**Edge Case 1 — MCP tool unavailable (DB down):**

Given the `workflow_log_invocation` MCP tool is called but the Drizzle insert throws (DB connection failure), then the tool's insert function returns `null`, `logger.warn` is called with the error context, and the calling agent receives `null` without an exception being thrown — the primary workflow continues unaffected.

**Edge Case 2 — Duplicate invocationId:**

Given a caller provides an `invocationId` that already exists in `wint.agent_invocations`, then the unique constraint violation is caught in the tool's try/catch, the function returns `null`, and `logger.warn` is called — the caller is never thrown an unhandled error.

---

## Non-Goals

- Batching or async write queue for high-frequency scenarios (deferred to follow-on story; write latency p95 monitored)
- `workflow_log_decision`, `workflow_log_outcome`, `workflow_get_story_telemetry` MCP tools (WINT-0120 scope, not this story)
- Retroactively logging past invocations
- Dashboard or query UI for logged data
- Instrumenting existing agents to call the skill (instrumentation is a follow-on via WINT-3070)
- Frontend changes — this story has no UI impact
- Modification of protected production DB schemas beyond the new `telemetry/` MCP tool module

## Architecture Notes

### Append-Only Write Model
`wint.agent_invocations` is an append-only log. No update or delete operations are acceptable in this story or any follow-on that uses this skill. This is an immutable event log.

### Fire-and-Forget Skill
The `telemetry-log` skill follows the same interaction model as `token-log`: the agent calls the skill, the skill calls the MCP tool, and regardless of the result (success, null, error), the primary workflow continues. The skill is a side-effect, not a gate.

### invocationId Generation
The `invocationId` column has a unique constraint. The skill should recommend callers pass a stable, predictable ID (e.g., `{agentName}-{storyId}-{isoTimestamp}`) for idempotency. If not provided, the tool should generate `crypto.randomUUID()`. This matches the pattern in `session-create.ts`.

### High-Frequency Write Consideration
The `agentInvocations` table already has the correct index strategy from WINT-0040 (`agent_invocations_agent_name_started_at_idx`, `agent_invocations_status_idx`). Synchronous single-row inserts are acceptable for Phase 3 MVP. Batching is deferred — monitor p95 insert latency; threshold is 50ms under load.

### Zod-First Patterns
All input validation uses Zod schemas defined in `__types__/index.ts` alongside the tool implementation file. TypeScript interfaces are not used. `parse()` is called at the entry point of the tool function (fail fast).
