# WINT-4070: Create cohesion-prosecutor Agent (PO Role) — Feature Completeness Enforcer

## Context

The WINT graph infrastructure (WINT-0060, WINT-0131) tracks feature capabilities via `graph.capabilities`. WINT-4040 infers CRUD capabilities from story history. WINT-4050 defines cohesion rules (e.g., "features with 'create' need 'delete'"). WINT-4060 creates a haiku-powered graph-checker that mechanically detects rule violations.

Mechanical rule checking alone cannot distinguish between a legitimately narrow-scoped feature (e.g., a read-only reporting endpoint) and an incomplete feature that was never given a delete operation. A Product Owner lens is needed to apply business judgment at workflow gates — asking "is this feature actually done from a product perspective?" rather than just "does this feature pass all rules?"

**Active in-progress work to be aware of:**

| Story | Status | Relationship |
|-------|--------|--------------|
| WINT-4040 | failed-code-review | Provides capability data this agent reads at runtime — data may be sparse |
| WINT-4050 | needs-code-review | Defines cohesion rules this prosecutor will apply |
| WINT-4030 | needs-code-review | Populates graph.features that backs prosecution queries |
| WINT-4060 | pending | Creates graph-checker agent (haiku) — prosecutor can consume its output |

---

## Goal

Create a sonnet-powered `cohesion-prosecutor` worker agent that applies Product Owner judgment to determine whether a feature's missing CRUD coverage is a genuine completeness gap or an intentional narrow-scope design decision, emitting a structured verdict with justification.

---

## Examples

**Example 1 — Full CRUD coverage:**
Given a story invoking the prosecutor for feature `wishlist-item` where `graph_get_capability_coverage` returns `{ create: 2, read: 3, update: 1, delete: 1 }`, the agent writes `prosecution-verdict.json` with `verdict: "COMPLETE"` and justification "All four CRUD lifecycle stages covered."

**Example 2 — Legitimate read-only feature:**
Given a story invoking the prosecutor for feature `activity-report` where coverage shows `{ create: 0, read: 5, update: 0, delete: 0 }` and the story description says "read-only reporting endpoint", the agent writes `prosecution-verdict.json` with `verdict: "INCOMPLETE-EXCUSED"` and justification "Feature is intentionally read-only per story scope; missing write operations are acceptable."

**Example 3 — Genuine missing operation:**
Given a story invoking the prosecutor for feature `order` where coverage shows `{ create: 2, read: 3, update: 1, delete: 0 }` and the story has no justification for the missing delete, the agent writes `prosecution-verdict.json` with `verdict: "INCOMPLETE-BLOCKED"`, `missing_capabilities: ["delete"]`, and justification "Data-owning feature lacks delete capability with no stated exception."

---

## Edge Cases

**Edge Case 1 — Empty capability data (WINT-4040 not run):**
If `graph_get_capability_coverage` returns all zeros for a feature (or the feature is not found in `graph.capabilities`), the agent must emit `PROSECUTION BLOCKED: no capability data — run WINT-4040 first` and must NOT produce a verdict. Writing a false COMPLETE verdict on empty data is a non-negotiable prohibition.

**Edge Case 2 — Missing optional inputs:**
If `cohesion_rules_path` is not provided, the agent falls back to embedded PO heuristics (all 4 CRUD expected for data-owning features; read-only acceptable for reporting features) and notes `"using embedded heuristics"` in the verdict justification. Verdict is still produced — this is not a blocking condition.

## Architecture Notes

**Role differentiation from WINT-4060 (graph-checker):**
- graph-checker (WINT-4060) = mechanical rule application. Haiku-powered. Detects rule violations programmatically with no judgment.
- cohesion-prosecutor (WINT-4070) = PO judgment layer. Sonnet-powered. Interprets violations, applies business context, decides whether a feature is truly incomplete or legitimately narrow-scoped.

The prosecutor SHOULD consume graph-checker output when available (`graph_checker_output_path`). When graph-checker has not run, the prosecutor calls the same MCP tools directly (`graph_get_capability_coverage`).

**Verdict schema alignment:**
The `prosecution-verdict.json` output schema must be compatible with `GraphCheckCohesionOutputSchema` (`{ status: complete/incomplete/unknown, violations?: string[] }`) for forward compatibility with WINT-4120 (workflow integration). The prosecutor extends this with `verdict` (three-way), `justification`, `missing_capabilities`, and `confidence` fields.

**Execution gate consideration:**
The agent is intended to run during the qa-verify or workflow gate phase. It acts as a gatekeeper before a story is marked done. However, verdict enforcement (actually blocking story promotion) is deferred to WINT-4120.
