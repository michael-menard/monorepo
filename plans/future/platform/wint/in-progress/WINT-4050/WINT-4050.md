# WINT-4050: Seed Initial Cohesion Rules into the Rules Registry

## Context

The WINT platform's cohesion enforcement subsystem has established its foundational infrastructure through prior stories:

- **WINT-4010** delivered the cohesion sidecar (`computeCheck()`) with a hardcoded CRUD completeness check (`CRUD_STAGES = ['create', 'read', 'update', 'delete']`)
- **WINT-4020** delivered the rules registry DB table (`wint.rules`) and the `@repo/sidecar-rules-registry` service exposing `proposeRule()`, `promoteRule()`, and `getRules()`
- **WINT-4030** and **WINT-4040** are in-progress populating the graph with existing features and inferred capabilities (low overlap risk with this story)

**WINT-4120** (prosecutor/judge integration) depends on this story providing a populated, active ruleset. Until rules are seeded and promoted, the cohesion enforcement pipeline has no formal invariants to evaluate against.

The `wint.rules` table has a lifecycle: rules are first `proposed`, then `promoted` to `active` status (requiring `source_story_id` or `source_lesson_id`). This story operationalizes that lifecycle for the initial canonical ruleset.

---

## Goal

Formalize the WINT platform's cohesion requirements by seeding an initial set of active rules into the `wint.rules` table, providing a concrete ruleset that downstream cohesion enforcement stories (WINT-4120) can operate against.

---

## Examples

**Given** a freshly migrated database with an empty `wint.rules` table, **when** `pnpm seed:cohesion-rules` is executed, **then** `getRules({ status: 'active' })` returns exactly 4 rules: the `create→delete` gate rule (severity: error), the `delete→create` gate rule (severity: error), the `update→read` gate rule (severity: warning), and the `upload→replace/update` prompt-injection advisory rule (severity: warning) — all with `source_story_id: 'WINT-4050'`.

**Given** the seed script has already been run once and all 4 rules are active, **when** the seed script is run a second time, **then** `proposeRule()` returns `{ ok: false, conflicting_ids: [...] }` for each existing rule, the seed logs the conflicts gracefully via `@repo/logger`, no new rules are created, and the script exits without error.

---

## Edge Cases

**Upload/replace lifecycle gap:** The `CRUD_STAGES` constant in `compute-check.ts` does not include `upload` or `replace`. A rule requiring `upload→replace` as a `gate` rule cannot be enforced by the current cohesion sidecar without extension. Resolution: store this rule as `rule_type: prompt_injection` (advisory) so it is persisted in the registry without triggering an unenforceable hard gate. The decision must be documented in ELAB (AC-5); if a hard gate is desired, cohesion sidecar extension is deferred to WINT-4120.

**Partial promote failure:** If `proposeRule()` succeeds for all 4 rules but `promoteRule()` fails for one (e.g., missing required field), the seed script must not leave the registry in a silent partial state. The script must log each failure via `@repo/logger` and either retry or report the count of failed promotions so the issue is surfaced.

---

## Non-Goals

- Do NOT modify the `wint.rules` DB schema (protected; delivered by WINT-4020)
- Do NOT extend the cohesion sidecar's `CRUD_STAGES` unless AC-5 explicitly approves — if deferred, that work belongs to WINT-4120
- Do NOT create a UI for rule management (out of scope)
- Do NOT implement the graph-checker agent (WINT-4060) or cohesion-prosecutor agent (WINT-4070)
- Do NOT add rules for features outside the LEGO MOC instructions domain unless clearly generalizable
- Do NOT create a runtime rule engine that dynamically evaluates rules at query time (WINT-4120's concern)

## Architecture Notes

- **Direct-call pattern (ARCH-001):** The seed script calls `proposeRule()` and `promoteRule()` as imported TypeScript functions, not HTTP calls. The rules-registry sidecar HTTP server (if running) is not involved.
- **Discriminated union results:** All calls return `{ ok: true, data: T } | { ok: false, error: string }` — the seed script must inspect `ok` before proceeding to promote.
- **Idempotency contract:** `proposeRule()` returns `{ ok: false, conflicting_ids: [...] }` when a semantically duplicate rule exists. The seed script must treat this as a successful "already exists" case, not an error. `promoteRule()` should be skipped for conflicted rules.
- **Zod-first:** All types defined via `z.object()` + `z.infer<>`. No TypeScript `interface` declarations.
- **No barrel files:** Do not create an `index.ts` in the seeds directory that only re-exports.
