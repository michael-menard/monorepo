# WINT-2070: Implement Cache Warming Strategy — Create `cache-warm` Skill that Invokes All Four Populate Scripts

## Context

The WINT platform Phase 2 goal is 80% token reduction. Four populate scripts exist to warm `wint.context_packs` with project conventions, domain knowledge, library patterns, and agent missions — delivered by WINT-2030, WINT-2040, WINT-2050, and WINT-2060. These scripts were previously invoked manually and independently, with no single entry point to trigger all warm-ups before a high-volume agent workflow session.

Without a unified `cache-warm` skill, agents and workflow scripts have no standard way to ensure all context packs are populated before starting. High-volume operations begin with stale or empty caches, negating Phase 2 token reduction goals. The upcoming `context-warmer` agent (WINT-2080) needs a skill to invoke.

The `cache-warm` skill is a `.claude/skills/cache-warm/SKILL.md` markdown instruction file — not a TypeScript script — following the established `.claude/skills/` format. It provides both human operators and LLM agents a standardized, resilient warm-up sequence.

**Baseline**: Baseline date 2026-02-13 predates WINT-2030/2040/2050/2060 implementations. All four populate scripts are completed. The `cache-warm` skill has been implemented and is at `needs-code-review` status at the time of story creation.

## Goal

Create `.claude/skills/cache-warm/SKILL.md` that orchestrates all four context-pack populate scripts in a resilient sequential execution, treating individual script failures as non-fatal, and producing a structured summary block showing per-script status and total packs written.

## Non-Goals

- Do NOT implement a TypeScript orchestrator script — the skill invokes existing scripts via `pnpm tsx`
- Do NOT implement automatic cache invalidation or doc-change triggers (deferred)
- Do NOT extract `populate-utils.ts` shared module (separate follow-on story — flagged as technical debt after 4 scripts now exist)
- Do NOT modify any populate scripts — read-only from this story's perspective
- Do NOT modify `wint.context_packs` table schema (protected)
- Do NOT add new `contextPackTypeEnum` values (protected)
- Do NOT implement cache scheduling or cron jobs (WINT Phase 3 scope)
- Do NOT implement cache retrieval for agents (WINT-2110 scope)
- Do NOT build a UI or HTTP endpoint

## Architecture Notes

The `cache-warm` skill follows the established `.claude/skills/` pattern:

1. **Instruction file, not code**: SKILL.md is a markdown instruction file executed by an LLM agent or human. It is not a TypeScript orchestrator.

2. **Sequential invocations**: Each script is a standalone `pnpm tsx` Node.js invocation, run in sequence. Parallelization at the skill level is out of scope (identified as a future optimization in WINT-2030 lessons).

3. **Resilient non-fatal design**: Any individual script failure is caught, logged, and reported. The skill continues to the next script. Only a missing `DATABASE_URL` blocks execution entirely.

4. **Two-database model**:
   - `DATABASE_URL` (port 5432, `lego_dev`): required for all scripts that write to `wint.context_packs`
   - `KB_DATABASE_URL` (port 5433, `lego_kb`): required only by `populate-domain-kb.ts` for reading KB lessons and blockers; absence triggers a warning and skip (non-fatal)

5. **Path placement note**: `agent-mission-cache-populator.ts` lives at `packages/backend/database-schema/src/seed/` — different from the other three scripts in `packages/backend/mcp-tools/src/scripts/`. The skill must use the exact correct path.

6. **Follow-on opportunity (non-goal)**: With 4 populate scripts now complete, extracting a shared `populate-utils.ts` module (containing `readDoc()`, `extractSections()`, `PopulateResultSchema`) is now viable. This is explicitly out of scope for WINT-2070 and should be captured as a follow-on refactor story.
