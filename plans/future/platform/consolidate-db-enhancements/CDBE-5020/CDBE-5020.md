# CDBE-5020: Partition `telemetry.workflow_events` Table and Add Partition Management Cron Job

## Context

`telemetry.workflow_events` is the primary telemetry event table in the `lego_dev` database (managed by `packages/backend/db/src/schema.ts`). It is written by the `insertWorkflowEvent()` telemetry SDK and the `batch-insert.ts` batch writer, providing observability for all pipeline events. The table currently has no partitioning — all historical events accumulate in a single heap table. As pipeline throughput increases, time-range queries (the primary access pattern) will degrade to full table scans.

CDBE-5010 (direct dependency, must be merged first) partitions `telemetry.agent_invocations` and `telemetry.token_usage` in a shared maintenance window. CDBE-5020 completes the telemetry partitioning rollout by converting `telemetry.workflow_events` to a monthly range-partitioned table on the `ts` column (timestamptz, NOT NULL DEFAULT NOW()), and implements automated partition management so future monthly partitions are created without manual intervention.

The orchestrator package (`packages/backend/orchestrator/src/cron/`) already has a fully built cron job infrastructure (APIP-3090) with advisory lock support, env-var-based job disabling, and structured logging. This is the preferred mechanism for the partition management job, over pg_cron or Lambda, because it requires no new infrastructure configuration and is already tested.

**Blocking note**: CDBE-5010 must be merged before beginning implementation of CDBE-5020. Elaboration and test planning are unblocked. Both stories should be deployed in the same maintenance window.

---

## Goal

Convert `telemetry.workflow_events` to a monthly range-partitioned table and deploy a cron job that automatically pre-creates future monthly partitions so the table never silently drops inserts due to a missing partition.

---

## Examples

**Example 1 — Happy path: insert routes transparently after migration**
Given the partition migration has been applied to lego_dev and partitions exist for 2026-03, 2026-04, and 2026-05, when `insertWorkflowEvent()` is called with a new event whose `ts` is `2026-04-15T10:00:00Z`, then the insert succeeds without any application code change, and the row appears in `telemetry.workflow_events_y2026m04` when queried directly.

**Example 2 — Happy path: partition manager pre-creates next month's partition**
Given the cron job fires on 2026-03-01 at 00:05 UTC, when `runPartitionManager()` runs with a `monthsAhead` config of 2, then it executes:
```sql
CREATE TABLE IF NOT EXISTS telemetry.workflow_events_y2026m04
  PARTITION OF telemetry.workflow_events
  FOR VALUES FROM ('2026-04-01') TO ('2026-05-01');

CREATE TABLE IF NOT EXISTS telemetry.workflow_events_y2026m05
  PARTITION OF telemetry.workflow_events
  FOR VALUES FROM ('2026-05-01') TO ('2026-06-01');
```
and logs success for each via `@repo/logger`.

## Edge Cases

**Edge Case 1 — December to January year rollover**
When `runPartitionManager()` runs on 2026-12-01 (upcoming months are January and February 2027), it must generate partition name `workflow_events_y2027m01` with range `FROM '2027-01-01' TO '2027-02-01'` — the year must increment correctly. An off-by-one in date arithmetic would create a partition with the wrong range or name, causing inserts in January to fail.

**Edge Case 2 — Insert outside all partition boundaries fails explicitly**
If a row is inserted with `ts = '2024-01-01'` and no partition covers that date range, PostgreSQL raises an error: "no partition of relation 'workflow_events' found for row." This is the correct behavior — it surfaces the problem rather than silently dropping data. The `IF NOT EXISTS` guard on partition pre-creation does not affect this behavior; it only prevents duplicate partition creation errors.

**Edge Case 3 — Idempotent partition pre-creation**
When `runPartitionManager()` is called twice in the same month (e.g., the cron job fires twice due to retry logic), the `CREATE TABLE IF NOT EXISTS ... PARTITION OF ...` statement completes without error on the second run, and the function logs a skip message. No duplicate partition is created.

**Edge Case 4 — Migration idempotency**
If the partition migration SQL is run twice (e.g., accidental re-application), the migration must not error. Guards such as `IF NOT EXISTS` for the new partitioned parent and `IF EXISTS` for the rename step prevent failures on re-run.

## Architecture Notes

- **No Drizzle schema changes**: Drizzle ORM does not support PostgreSQL `PARTITION BY RANGE`. The `workflowEvents` table definition in `schema.ts` continues to reference the parent table — no changes required. Insert/select queries remain identical; the pg driver handles transparent routing to child partitions.
- **Raw SQL migration**: Because `pnpm db:generate` is broken (WINT-0040 — `sets.js` module resolution error) and Drizzle cannot express partitioned tables, the migration must be a hand-authored raw SQL file. Determine at implementation time whether this file belongs in `packages/backend/db/src/migrations/` alongside Drizzle-generated migrations or as a standalone deployment script.
- **Table rewrite strategy**: PostgreSQL does not support `ALTER TABLE ... ADD PARTITION BY`. The idiomatic approach is: (1) `ALTER TABLE telemetry.workflow_events RENAME TO workflow_events_old`, (2) create new `telemetry.workflow_events PARTITION BY RANGE (ts)`, (3) create initial child partitions, (4) `INSERT INTO telemetry.workflow_events SELECT * FROM workflow_events_old`, (5) drop `workflow_events_old`, (6) recreate indexes on parent.
- **Partition naming**: `workflow_events_y<YYYY>m<MM>` (zero-padded month, e.g., `workflow_events_y2026m03`) for predictability and consistent generation.
- **Advisory lock**: `LOCK_KEYS.PARTITION_MANAGER = 42_005` (next available after 42_004). Session-level lock auto-released on pool.end().
- **pg_cron alternative**: Deferred. Confirm pg_cron availability with `SELECT * FROM pg_extension WHERE extname = 'pg_cron'` on Aurora before considering. Node.js cron is the default and recommended approach.
- **Archival non-goal**: Partition detach/archive requires a data retention policy decision. Explicitly out of scope for this story.
- **Foreign keys**: PostgreSQL 12+ supports FK references to partitioned table parents. Verify no FKs reference `telemetry.workflow_events` before migration; if they exist, drop and recreate on the new partitioned parent.
