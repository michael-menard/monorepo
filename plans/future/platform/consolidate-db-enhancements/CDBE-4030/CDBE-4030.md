# CDBE-4030 ‚Äî Embedding Worker Process and Backfill

## Context

CDBE-4020 (in_progress) adds `embedding vector(1536)` columns to five public-schema knowledge tables: `lessons_learned`, `adrs`, `code_standards`, `rules`, and `cohesion_rules`. After CDBE-4020 merges, all existing and future rows on these tables will have `embedding = NULL` until a worker populates them.

The `EmbeddingClient` already handles OpenAI API calls with caching, retry logic, and batch processing for `public.knowledge_entries`. The KB database already has a `pg_notify` trigger pattern in place (`1003_story_notify.sql`). The `pg` library is already a dependency in the knowledge-base package.

There is currently no process that (a) listens for new/updated rows on the five knowledge tables and generates embeddings asynchronously, or (b) backfills the existing rows. Without this worker, semantic search across all KB entity types ‚Äî a prerequisite for retrieval-augmented generation and agent context lookup ‚Äî will never have data to query.

**Implementation is blocked on CDBE-4020 merge.** Elaboration, test planning, and story generation proceed fully.

## Goal

Implement a persistent background worker process that subscribes to PostgreSQL NOTIFY events on the five knowledge tables, generates embeddings via `EmbeddingClient`, writes them back inline, and guarantees eventual consistency via a polling fallback and startup backfill ‚Äî making all KB entity types available for semantic search.

## Examples

**Example 1 ‚Äî NOTIFY path (new row):**
A user runs a KB lesson import that INSERTs 3 rows into `lessons_learned`. The NOTIFY trigger fires for each row, publishing to the `knowledge_embedding_needed` channel with payload `{"table": "lessons_learned", "id": "<uuid>"}`. The worker, which holds a dedicated `pg.Client` LISTEN connection, receives the 3 notifications within 50ms, batches them, calls `EmbeddingClient.generateEmbeddingsBatch()` once with 3 texts, and writes 3 UPDATE statements. All 3 rows now have `embedding IS NOT NULL`.

**Example 2 ‚Äî Polling fallback (durability gap):**
The worker process is restarted due to a deployment. During the 30-second downtime, 5 rows were inserted into `adrs` ‚Äî their NOTIFY events were lost. When the worker restarts, the polling fallback runs within its configured interval (default: 5 minutes), queries `SELECT id, content FROM adrs WHERE embedding IS NULL LIMIT 100`, generates embeddings for the 5 rows, and writes them back. All 5 rows are populated without any manual intervention.

**Example 3 ‚Äî Startup backfill:**
After CDBE-4020 is first applied, 847 existing rows across the five tables have `embedding = NULL`. The worker starts, runs the backfill function, processes rows in batches of 50 with 500ms inter-batch delays, and logs row counts per table. After approximately 9 batches (with delays), all 847 rows have embeddings.

## Edge Cases

**Edge Case 1 ‚Äî Polling overlap guard:**
The polling interval fires every 5 minutes. If a large table has 10,000 NULL rows and the previous polling run took 6 minutes, the new interval fires while the previous run is still executing. The guard detects the active run flag, skips the new run, and logs: "Polling run skipped: previous run still in progress." No duplicate embedding writes occur.

**Edge Case 2 ‚Äî Worker starts before CDBE-4020 applied:**
If the worker is deployed before CDBE-4020's migration is applied, the startup backfill query will fail with a column-not-found error. The worker catches this, logs the error with clear remediation instructions, and exits with a non-zero code to signal the deployment dependency violation.

**Edge Case 3 ‚Äî Batch size trigger vs. time window:**
With `batchWindowMs=50` and `batchSize=25`, if 25 notifications arrive within 10ms, the batch fires immediately on the size limit ‚Äî it does not wait for the remaining 40ms. The time-window trigger is a ceiling, not the only trigger.

**Edge Case 4 ‚Äî Zero NULL rows at backfill:**
All five tables already have non-null embeddings (e.g., backfill previously completed). The backfill function runs, finds 0 NULL rows per table, logs "Backfill complete: 0 rows processed," and returns without making any OpenAI API calls.

## Non-Goals

- Do NOT implement SQS as the primary queue mechanism ‚Äî deferred to a future story. NOTIFY/LISTEN + polling fallback is the MVP durability model.
- Do NOT populate `workflow.story_embeddings` or `workflow.plan_embeddings` ‚Äî those are CDBE-4010's scope.
- Do NOT modify `public.knowledge_entries` embedding logic ‚Äî already managed by existing `EmbeddingClient` usage.
- Do NOT deploy the worker as a Lambda function ‚Äî NOTIFY/LISTEN requires a persistent connection.
- Do NOT change HNSW indexes or the `embedding` column definitions ‚Äî those are CDBE-4020's responsibility.
- Do NOT create `wint.*` or `kbar.*` schema references.
- Do NOT implement a UI or API endpoint for triggering backfill ‚Äî a script is sufficient.
- Do NOT implement cross-table semantic search ‚Äî downstream functionality out of scope.

## Architecture Notes

**LISTEN connection**: Use a raw `pg.Client`, not `getPool()` or `getDbClient()`. Pools recycle connections silently and will drop LISTEN subscriptions without error. Call `client.connect()` once at startup. Register `client.on('error', ...)` with exponential backoff reconnect logic.

**Batching strategy**: Buffer incoming notifications in memory keyed by (table, id). Flush on whichever comes first: (a) `batchWindowMs` has elapsed, or (b) buffer reaches `batchSize`. Deduplicate by id within the buffer window ‚Äî if the same row notifies twice before flush, generate one embedding.

**OpenAI rate limiting**: For the realtime LISTEN path, use default `maxConcurrentRequests` (10). For backfill, instantiate `EmbeddingClient` with `maxConcurrentRequests: 3` to avoid hitting tier limits on large tables.

**Drizzle UPDATE pattern**: `await db.update(<table>).set({ embedding: vectorValue, updatedAt: new Date() }).where(eq(<table>.id, id))`. The `vector` customType's `toDriver()` function serializes correctly for UPDATE (same path as INSERT in CDBE-4020).

**Process model**: The worker is a long-running Node.js process (`run.ts` as entrypoint). It is NOT a Lambda function. Deployment model (standalone process, ECS task, etc.) is outside this story's scope ‚Äî `run.ts` provides the entrypoint; infrastructure provisioning is separate.

## Infrastructure Notes

- **Target DB**: `knowledgebase` on port 5433 ‚Äî not `lego_dev` (port 5432).
- **`pnpm db:generate` is broken** (WINT-0040 / `sets.js` module resolution error) ‚Äî all migration SQL must be authored manually. Do not run `pnpm db:generate`.
- **`CREATE INDEX CONCURRENTLY`** cannot run inside a transaction block. The NOTIFY trigger migration does not add indexes, so this is N/A for ST-1. If any index is ever added in this story's scope, author it outside a transaction.
- **Migration must include DB safety preamble** (DO block) identical to the pattern in `1007_pipe_0020_ghost_state_audit.sql`.
- **`public.` prefix required** on all pgvector operator references (e.g., `public.vector_cosine_ops`). Not applicable to NOTIFY triggers, but noted for any future index work in this story.
