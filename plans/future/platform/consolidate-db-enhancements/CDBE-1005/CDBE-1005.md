# CDBE-1005: Define valid_transitions Lookup Schema and Enumerate All Legal Story State Transitions

## Context

The KB database story state machine currently enforces valid transitions via a hardcoded IF/ELSIF chain inside `workflow.validate_story_state_transition()` (migration 1001). This makes the complete set of legal transitions invisible to queries, tooling, and documentation — implementers must read PL/pgSQL trigger source code to understand what state changes are permitted.

CDBE-1010 (Valid Transitions Lookup Table and State Machine Trigger) is the direct downstream consumer of this story. That trigger needs to JOIN against an authoritative `from_state → to_state` table. Without `workflow.valid_transitions`, CDBE-1010 has no stable reference to query.

Additionally, lessons learned (CDBN-0010) document that KB state machine transitions have historically been a source of confusion, with implementers reinventing the wheel each time they touch state logic. This story permanently resolves that ambiguity by making all legal transitions queryable and peer-reviewed.

**Schema drift known issue**: The `StoryStateSchema` Zod enum in `apps/api/knowledge-base/src/__types__/index.ts` (lines 780–793) does not yet include `created`, `elab`, or `needs_code_review` as valid app-layer values. This story does NOT fix that mismatch — the DB `valid_transitions` table will correctly include all canonical states, but the Zod enum remains out of sync. A companion story should update `StoryStateSchema`; do not expand this story's scope to include that change.

---

## Goal

Create a `workflow.valid_transitions` lookup table populated with all legal story state transition pairs, derived directly from migration 1001's trigger logic, so that downstream stories and tooling have a single queryable authority for the KB state machine.

---

## Examples

**Given** a database with migrations 1000–1004 applied, **the system produces** a `workflow.valid_transitions` table containing rows such as:

| from_state         | to_state          | description                     |
| ------------------ | ----------------- | ------------------------------- |
| NULL               | backlog           | Initial story creation          |
| backlog            | created           | Story picked up for elaboration |
| in_progress        | needs_code_review | Dev submits for review          |
| needs_code_review  | ready_for_qa      | Code review passes              |
| in_qa              | completed         | QA passes                       |
| failed_qa          | in_progress       | Rework after QA failure         |
| (any non-terminal) | cancelled         | Cancel from any active state    |

**Given** a developer running `SELECT * FROM workflow.valid_transitions WHERE from_state = 'in_qa'`, **the system returns** exactly two rows: `in_qa → completed` and `in_qa → failed_qa`, matching the trigger logic in migration 1001.

---

## Edge Cases

**NULL from_state uniqueness**: PostgreSQL UNIQUE constraints treat NULLs as distinct by default — a plain `UNIQUE(from_state, to_state)` will not prevent two rows with `from_state = NULL`. The migration must use either a partial unique index or a functional unique index on `COALESCE(from_state, '__NULL__')` to enforce the constraint correctly.

**Migration slot collision**: Migrations 1000–1003 are confirmed taken at seed time. Slot 1004 is the target, but may be consumed by a concurrent story before implementation. At implementation time, verify 1004 is free; if not, use the next available slot and document the deviation.

**Ghost states must not appear**: States `uat`, `done`, `in_review`, `ready_for_review`, `deferred`, `ready_to_work`, `elaboration`, and `draft` must not appear in any `valid_transitions` row as `from_state` or `to_state`. Migration 1001 has already migrated all rows away from these states and the trigger renders them unreachable.

**`uat` is not a story state**: Per KBAR-0220 lesson, `uat` is a `WorkPhaseSchema` value, not a `StoryStateSchema` terminal state. The terminal state after QA PASS is `completed`. Do not include `uat` as a `to_state`.

- Do not remove ghost states from the `workflow.story_state_enum` PostgreSQL enum — migration 1001 already renders them unreachable.

## Architecture Notes

- `workflow.stories.state` is `text` (not a typed pgEnum column) in Drizzle — enforcement is done entirely via DB trigger. `valid_transitions` columns should also use `TEXT` (not a pgEnum) for `from_state`/`to_state` to stay compatible with future state additions.
- No `valid_transitions` table currently exists anywhere in the KB schema — this is a net-new table.
- The Drizzle schema registration step is required even though the raw SQL migration handles the DDL directly. Skipping it causes type drift between DB and application layer.
- CDBE-1010 will reference this table via trigger logic or FK. This migration must be merged and deployed before CDBE-1010's test suite runs — see CDBE-1030/CDBE-0020 lesson: triggers referencing missing functions/tables abort the entire pgtap transaction.

- **EC-1** — Duplicate NULL from_state row rejected: second `NULL→backlog` insert silently ignored

- **EC-3** — `uat` absent as `to_state`

**Edge cases**:

- **ED-1** — Idempotency: re-running migration leaves count unchanged, no errors
- **ED-2** — `COMMENT ON TABLE` present with `1004:` prefix
- **ED-3** — Table queryable via Drizzle with correct column names

**Manual cases**:

- **MC-1** — Peer review of transition enumeration completeness (side-by-side diff of migration 1001 trigger IF/ELSIF against seed rows; PR comment sign-off)

**Test risks**:

- Transition count hardcoded in pgtap tests may drift if migration 1001 trigger is later amended
- NULL uniqueness constraint approach must be confirmed at implementation time before EC-1 test is written
- Confirm pgtap is available in CI test harness (see CDBE-0020 for established setup pattern)
