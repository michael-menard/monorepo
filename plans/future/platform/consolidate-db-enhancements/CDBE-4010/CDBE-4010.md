# CDBE-4010: pgvector Foundation — `story_embeddings` and `plan_embeddings` Tables with HNSW Indexes

## Context

The knowledgebase database already has the pgvector extension installed (confirmed in the full schema baseline at `999_full_schema_baseline.sql` line 74) and uses `vector(1536)` columns on `workflow.plans`, `workflow.hitl_decisions`, and `public.knowledge_entries`. The `EmbeddingClient` is production-ready and generating embeddings for knowledge entries.

However, `workflow.stories` has no embedding column (it was explicitly removed as "not yet needed" — see `workflow.ts` line 53 comment), and no dedicated normalized embedding tables exist for stories or plans. The `findSimilarStories()` function in `apps/api/knowledge-base/src/search/story-similarity.ts` currently queries `FROM public.stories` (line 51), which is a non-existent table — the stories table lives in `workflow.stories`, not `public`. This pre-existing bug means story similarity search is broken.

CDBE-4020 (embedding columns on knowledge tables) and CDBE-4030 (embedding worker process) both depend on this foundational embedding infrastructure being in place first.

**Migration slot**: 1009 (verified free as of 2026-03-18 — last migration is `1008_kfmb1020_story_content_columns.sql`).

---

## Goal

Create the normalized embedding infrastructure — `workflow.story_embeddings` and `workflow.plan_embeddings` tables with HNSW indexes, an `embedding_section_lookup` reference table, and Drizzle schema registrations — so that CDBE-4020 and CDBE-4030 have a correct, queryable embedding foundation to build on.

---

## Examples

**Example 1 — Inserting a story embedding after migration:**
Given a story `story_id = 'CDBE-4010'` exists in `workflow.stories`, after applying migration 1009, a worker can insert:
```sql
INSERT INTO workflow.story_embeddings (story_id, embedding, model)
VALUES ('CDBE-4010', '[0.01, 0.02, ...]'::vector, 'text-embedding-3-small');
```
The row is stored; a second insert with the same `story_id` is rejected by the UNIQUE constraint.

**Example 2 — Querying similar stories after the fix:**
Given `workflow.story_embeddings` has rows, `findSimilarStories(db, queryVector, 5)` now executes:
```sql
SELECT s.story_id, s.title, s.feature, s.state,
       1 - (se.embedding <=> $1::vector) as similarity_score
FROM workflow.story_embeddings se
JOIN workflow.stories s ON s.story_id = se.story_id
WHERE se.embedding IS NOT NULL
ORDER BY se.embedding <=> $1::vector ASC
LIMIT 5
```
This returns stories ranked by cosine similarity without referencing the non-existent `public.stories` table.

## Architecture Notes

**Separate embedding tables vs. inline columns**: The `workflow.stories` schema comment (`workflow.ts` line 53) explicitly states `embedding` was removed as "not yet needed." The story description and seed specify `workflow.story_embeddings` and `workflow.plan_embeddings` as separate normalized tables. This is the intended architecture — do not add `embedding` directly to `workflow.stories`.

**HNSW vs. ivfflat**: All existing embedding indexes in the codebase use `ivfflat`. This story intentionally introduces `HNSW` for the new tables. HNSW offers higher recall and sub-linear search time at the cost of higher memory usage during build. The migration must document this rationale in a SQL comment. CDBE-4020 and CDBE-4030 must follow the same index type for consistency.

**pgvector extension location**: The extension is installed in the `public` schema. All `vector_cosine_ops` references must use the `public.` prefix: `embedding public.vector_cosine_ops`.

**Embedding dimension**: All existing embedding columns are `vector(1536)`, matching `text-embedding-3-small`. New tables must use the same dimension.

**Public schema for pgvector**: Do not attempt to install pgvector in the `workflow` schema — it must stay in `public`.

---

## Infrastructure Notes

- **Target database**: `knowledgebase` DB at port 5433. The migration safety preamble prevents accidental execution against `lego_dev` (port 5432).
- **Index build**: HNSW indexes on empty tables at migration time are fast. `CONCURRENTLY` overhead is negligible but establishes the safe pattern for future backfill scenarios.
- **pgvector version**: HNSW was added in pgvector 0.5.0. Confirm `SELECT extversion FROM pg_extension WHERE extname = 'vector'` returns >= 0.5.0 before committing to HNSW. If not available, fall back to `ivfflat` and document.
