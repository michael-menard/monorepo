## Context

With the schema prerequisites from CDBE-1050 in place, this story implements the two cascade trigger functions that automate all downstream side effects when a story transitions to `completed` or `cancelled`.

**`story_completion_cascade()`** fires when `NEW.state = 'completed'`. It must:

1. Mark the `story_dependencies` row as resolved (`resolved_at = NOW()`) for this story's entry as a dependency.
2. Mark any worktree row for this story as merged (`merged_at = NOW()`).
3. Soft-delete open `story_assignments` rows (`deleted_at = NOW()` where `deleted_at IS NULL`).

**`story_cancellation_cascade()`** fires when `NEW.state = 'cancelled'`. It must:

1. Soft-delete open `story_assignments` rows (`deleted_at = NOW()`).
2. Soft-delete open `story_blockers` rows (`deleted_at = NOW()`).
3. Flag stories that are blocked by the cancelled story by setting a `blocked_reason` or equivalent marker — **flag only, do NOT auto-advance or auto-resolve them**.

Both functions are wrapped in a single AFTER UPDATE trigger on `workflow.stories` and must be executed inside an explicit transaction. If any sub-operation fails, the entire cascade must roll back. No partial cascade is acceptable.

pgtap integration tests using transaction rollback assertions must be written before this story is considered complete.

## Goal

Automate all downstream side effects of story completion and cancellation so agents do not manage relational state manually, while guaranteeing transactional atomicity — a partial cascade is treated as a failure and rolled back.

## Examples

**Given** story `CDBE-1010` has an open `story_assignments` row (deleted_at IS NULL) and a `story_dependencies` row where it is the depended-on story, **when** its state transitions to `completed`, **then** the `story_dependencies.resolved_at` is set to `NOW()`, the open assignment row has `deleted_at = NOW()`, and any associated worktree has `merged_at = NOW()`.

**Given** story `CDBE-2010` has two open `story_blockers` rows — one indicating `CDBE-3010` is blocked by it — **when** `CDBE-2010` transitions to `cancelled'`, **then** both blocker rows have `deleted_at = NOW()`, the assignment rows for `CDBE-2010` are soft-deleted, and `CDBE-3010` is flagged as blocked by a cancelled story (flag only — its state is NOT changed).

## Edge Cases

**No open assignments:** If a story completes or is cancelled with no open `story_assignments` rows, the UPDATE affects 0 rows. The trigger must not raise an error — 0-row UPDATEs are valid.

**No blocker rows on cancellation:** If a cancelled story has no rows in `story_blockers`, the cancellation cascade completes normally without attempting the flag step for downstream stories.

**No worktree row:** Some stories may not have a worktree (e.g., documentation-only stories). The completion cascade must handle a missing worktree row gracefully — no error, no INSERT.

**Partial failure atomicity:** If soft-deleting assignments succeeds but resolving dependencies fails (e.g., FK violation), the entire cascade must roll back. This is enforced by the explicit transaction wrapping the trigger body. Partial state must never be committed.

**Non-cascade transitions:** The trigger only fires for transitions to `completed` or `cancelled`. All other state changes (`backlog → created`, `ready → in_progress`, etc.) must pass through the trigger without any side effects. Use `IF NEW.state NOT IN ('completed', 'cancelled') THEN RETURN NEW` as the early exit.

**Flag-only on cancellation for blocked stories:** Stories downstream of a cancelled story must be flagged, not auto-resolved. The flag mechanism (a column, a separate table row, or a status note) must be documented in the migration. Auto-advancing a blocked story's state is explicitly out of scope.

## Non-Goals

- Do NOT implement `story_assignments` or `story_blockers` table DDL — those are in CDBE-1050.
- Do NOT auto-advance stories that were blocked by a cancelled story to any new state — flag only.
- Do NOT modify the `validate_story_state_transition` trigger (migration 1001) or the `story_state_history` trigger (migration 1010).
- Do NOT check sibling story completion or auto-advance parent epics — that is a separate feature.
- Do NOT implement API endpoints, service layer, or UI.
- Do NOT implement worktree creation — only worktree state updates (`merged_at` / `abandoned_at`) on existing rows.

## Architecture Notes

- The trigger is AFTER UPDATE (not BEFORE). The cascade reads `NEW.state` after the state change is committed to the row. This ensures the story row is in its final state before side effects run.
- Both cascade functions run inside the same PostgreSQL transaction as the UPDATE that fires them. PostgreSQL guarantees this — trigger functions do not run in isolation. An exception in the trigger body rolls back the entire transaction including the original UPDATE.
- The `SECURITY INVOKER` mode means the trigger runs with the caller's privileges. Since `agent_role` will be the typical caller, `agent_role` must have UPDATE access on `story_dependencies`, `story_assignments`, `story_blockers`, and the worktree table. These grants should be verified against the RLS policies set in CDBE-1050 and any prior migration covering the worktrees table.
- Flag-only behavior for cancellation downstream: the elaboration explicitly prohibited auto-advancing blocked stories. The flag can be implemented as an INSERT into a `workflow.story_events` table (or equivalent audit/event store) rather than a state mutation. This avoids any risk of unintended state transitions.
- The trigger function should call the completion or cancellation cascade based on a CASE or IF/ELSIF on `NEW.state`. A single trigger entry point is preferred over two separate triggers to ensure ordering is deterministic.
