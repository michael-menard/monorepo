# CDBE-2010: `advance_story_state` and `assign_story` Stored Procedures

## Context

Phase 2 of the `consolidate-db-enhancements` feature introduces stored procedures as the
authenticated, atomic entry point for story state transitions and agent assignments.

Previously, agents performing state changes required manual multi-step DB operations:
validate the transition, INSERT into `workflow.story_state_history`, UPDATE
`workflow.stories.state` — each step done independently with no caller authentication.
This creates risk of partial writes, unauthenticated callers, and cascade logic divergence.

**Infrastructure already deployed:**

| Object                                               | Migration | Status               |
| ---------------------------------------------------- | --------- | -------------------- |
| `workflow.valid_transitions` (54-row matrix)         | 1004      | Deployed             |
| `workflow.allowed_agents` + `validate_caller()`      | 1005      | Deployed (CDBE-2005) |
| `workflow.story_state_history` BEFORE INSERT trigger | 1010      | Deployed (CDBE-1010) |
| `workflow.story_assignments` table                   | 1050      | Deployed (CDBE-1050) |
| BEFORE UPDATE trigger on `workflow.stories`          | 1001      | Deployed             |

**Delegation model:** `advance_story_state` inserts into `story_state_history`, which fires
the 1010 BEFORE INSERT trigger. The trigger owns `exited_at` + `duration_seconds` computation
and previous-row closure. The procedure MUST NOT re-implement that logic.

---

## Goal

Give agents a single authenticated, atomic call for story state advancement and assignment,
replacing fragmented manual DB operations and eliminating unauthorized-caller windows.

---

## Examples

**Example 1 — advance_story_state (happy path):**

```sql
-- Caller: pm-story-generation-leader (active in workflow.allowed_agents)
-- Story CDBE-2010 is currently in 'backlog' state
-- workflow.valid_transitions has (backlog → created) row

CALL workflow.advance_story_state(
  p_story_id        => 'CDBE-2010',
  p_to_state        => 'created',
  p_agent_name      => 'pm-story-generation-leader',
  p_reason          => 'Story generation complete',
  p_caller_agent_id => 'pm-story-generation-leader'
);

-- Result:
--   workflow.stories.state = 'created'
--   New story_state_history row: from_state='backlog', to_state='created', exited_at IS NULL
--   Previous open history row: exited_at = NOW(), duration_seconds = <elapsed>
--   (exited_at / duration_seconds set by 1010 BEFORE INSERT trigger — NOT by this procedure)
```

**Example 2 — assign_story (happy path):**

```sql
-- Caller: dev-execute-leader (active in workflow.allowed_agents)
CALL workflow.assign_story(
  p_story_id        => 'CDBE-2010',
  p_agent_name      => 'dev-execute-leader',
  p_phase           => 'implementation',
  p_caller_agent_id => 'dev-execute-leader'
);

-- Result:
--   New workflow.story_assignments row: story_id='CDBE-2010', agent_id='dev-execute-leader',
--   assigned_at=NOW(), deleted_at=NULL
--   If public.agent_invocations is available and a matching invocation exists, linkage recorded.
--   If public.agent_invocations is absent or no invocation found, proceeds without error.
```

## Edge Cases

**Edge Case 1 — unauthorized caller:**

```sql
CALL workflow.advance_story_state(
  p_story_id        => 'CDBE-2010',
  p_to_state        => 'created',
  p_agent_name      => 'rogue-bot',
  p_reason          => 'unauthorized attempt',
  p_caller_agent_id => 'unknown-agent-xyz'
);
-- RAISES: SQLSTATE P0001 — 'Unauthorized caller: unknown-agent-xyz'
-- NO mutations: stories.state unchanged, no story_state_history row inserted
```

**Edge Case 2 — illegal state transition:**

```sql
-- Story is in 'backlog' state; (backlog → completed) is not in valid_transitions
CALL workflow.advance_story_state(
  p_story_id        => 'CDBE-2010',
  p_to_state        => 'completed',
  p_agent_name      => 'pm-story-generation-leader',
  p_reason          => 'test illegal skip',
  p_caller_agent_id => 'pm-story-generation-leader'
);
-- RAISES: SQLSTATE 23514 — 'Illegal state transition: backlog → completed for story CDBE-2010'
-- Transaction aborts atomically — no partial writes survive
```

**Edge Case 3 — public.agent_invocations absent:**

```sql
-- If 999_cleanup_duplicate_tables.sql dropped public.agent_invocations CASCADE,
-- assign_story must not raise. The invocation-linking step is a best-effort operation.
CALL workflow.assign_story('CDBE-2010', 'dev-execute-leader', 'implementation', 'dev-execute-leader');
-- Succeeds. story_assignments row created. Invocation link silently skipped.
```

## Non-Goals

- Do NOT re-implement `exited_at`/`duration_seconds` computation inside `advance_story_state` — the 1010 trigger owns that.
- Do NOT implement `resolve_blocker`, `complete_artifact`, or `ingest_story_from_yaml` — those are CDBE-2020/2030 scope.
- Do NOT implement per-procedure caller authorization using `allowed_procedures` column — MVP validates `active=true` only.
- Do NOT add API endpoints, MCP tool wrappers, or TypeScript service layer changes.
- Do NOT modify `workflow.valid_transitions` data.
- Do NOT add HTTP-level auth — DB-level caller auth via `validate_caller` is the only mechanism here.
- Do NOT implement dynamic agent registration (INSERT into `allowed_agents`) — seed-only per CDBE-2005.

- [ ] **AC-1**: `workflow.advance_story_state(story_id text, to_state text, agent_name text, reason text, caller_agent_id text) RETURNS void` function is created with `CREATE OR REPLACE FUNCTION` in the `workflow` schema, written in `plpgsql`.
- [ ] **AC-2**: `advance_story_state` calls `PERFORM workflow.validate_caller(caller_agent_id)` as its first statement; raises `P0001` if caller is unknown or inactive before any mutations occur.
- [ ] **AC-3**: `advance_story_state` validates the requested transition by querying `workflow.valid_transitions`; raises `SQLSTATE 23514` with a descriptive message if the transition is not found (e.g., `'Illegal state transition: backlog → completed for story CDBE-2010'`).
- [ ] **AC-4**: `advance_story_state` INSERTs a new row into `workflow.story_state_history` with the validated (from_state, to_state) pair and supplied reason. The procedure does NOT set `exited_at` or `duration_seconds` — those are set by the 1010 BEFORE INSERT trigger.
- [ ] **AC-5**: `advance_story_state` UPDATEs `workflow.stories.state = to_state` after the history INSERT succeeds. The BEFORE UPDATE trigger on `workflow.stories` (migration 1001) fires as a second validation — if it rejects the transition, the transaction aborts.
- [ ] **AC-6**: `advance_story_state` is wrapped in an implicit transaction — if any step raises an exception, the entire operation rolls back atomically.
- [ ] **AC-7**: `workflow.assign_story(story_id text, agent_name text, phase text, caller_agent_id text) RETURNS void` function is created with `CREATE OR REPLACE FUNCTION` in the `workflow` schema, written in `plpgsql`.
- [ ] **AC-8**: `assign_story` calls `PERFORM workflow.validate_caller(caller_agent_id)` as its first statement; raises `P0001` if caller is unknown or inactive.
- [ ] **AC-9**: `assign_story` INSERTs a row into `workflow.story_assignments` with `(story_id, agent_id = agent_name, assigned_at = NOW())`.
- [ ] **AC-10**: If `public.agent_invocations` exists and a matching `invocation_id` can be resolved for the calling agent, `assign_story` stores the link (mechanism TBD by dev — may be a column on `story_assignments` or a separate link row). If `public.agent_invocations` does not exist or no active invocation is found, `assign_story` proceeds without error.
- [ ] **AC-11**: `COMMENT ON FUNCTION` is present for both functions, citing the migration number.
- [ ] **AC-12**: Migration is idempotent — second run exits 0 with no errors. Uses `CREATE OR REPLACE FUNCTION` throughout.
- [ ] **AC-13**: pgtap test file is created following the `BEGIN / SELECT plan(N) / ... / SELECT * FROM finish() / ROLLBACK` pattern, covering: `has_function` for both procedures, `throws_ok` for unauthorized caller (P0001), `throws_ok` for illegal transition (23514), `lives_ok` for authorized + legal transition, `lives_ok` for `assign_story` with valid inputs.
- [ ] **AC-14**: Migration grants `EXECUTE` on both functions to `agent_role`.

## Architecture Notes

### Delegation Chain

```
CALL workflow.advance_story_state(story_id, to_state, agent_name, reason, caller_agent_id)
  │
  ├─ PERFORM workflow.validate_caller(caller_agent_id)      ← aborts if unknown/inactive
  │
  ├─ NOT EXISTS check vs. workflow.valid_transitions         ← raises 23514 if illegal
  │
  ├─ INSERT INTO workflow.story_state_history(story_id, event_type, from_state, to_state, reason)
  │     │
  │     └─ [BEFORE INSERT trigger enforce_story_state_history_transition FIRES]
  │           ├─ Second transition validation vs. valid_transitions
  │           └─ Closes previous open row: sets exited_at=NOW(), duration_seconds=<elapsed>
  │                                        [procedure does NOT touch these columns]
  │
  └─ UPDATE workflow.stories SET state = to_state WHERE story_id = p_story_id
        │
        └─ [BEFORE UPDATE trigger on workflow.stories FIRES — migration 1001]
              └─ Third validation layer (hardcoded IF chains — must not drift from valid_transitions)
```

### Cross-Schema Reference

`assign_story` in the `workflow` schema references `public.agent_invocations`. PostgreSQL
permits this, but `agent_role` must have SELECT on `public.agent_invocations`. The migration
must include this grant (or omit the grant gracefully if the table does not exist post-cleanup).

### Security Model

- `validate_caller` uses `SECURITY INVOKER` semantics (no explicit SECURITY DEFINER on the
  stored procedures themselves) — runs with `agent_role` privileges.
- The 1010 trigger function is `SECURITY INVOKER` — the existing `GRANT UPDATE (exited_at, duration_seconds)`
  to `agent_role` (added in migration 1010) covers the transitive UPDATE requirement.
