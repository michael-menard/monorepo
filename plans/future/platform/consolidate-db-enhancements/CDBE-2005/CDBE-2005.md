# CDBE-2005: Stored Procedure Caller Authentication — `allowed_agents` Table

## Context

Phase 2 stored procedures (`advance_story_state`, `assign_story`, `resolve_blocker`,
`complete_artifact`, `ingest_story_from_yaml`) need to verify that their caller is an
authorized agent before performing any mutations. Without a DB-level caller registry, any
process that can connect to the database can invoke these procedures without identity checks.

This story creates `workflow.allowed_agents` — a reference table of authorized agent
identities — and a companion `workflow.validate_caller(caller_agent_id text)` PL/pgSQL
helper function that stored procedures invoke at their entry point to reject unknown callers.
The table is seeded with all known agent identities at migration time.

This table has no dependencies on Phase 1 trigger work and can be deployed independently,
but **must be deployed before any Phase 2 stored procedure migration runs**.

---

## Goal

Create the `workflow.allowed_agents` reference table and `workflow.validate_caller()` helper
function so that Phase 2 stored procedures have a DB-level mechanism to reject unauthorized
callers before performing any mutations.

---

## Examples

**Example 1 — Valid caller proceeds:**
Given `workflow.allowed_agents` contains `agent_id = 'pm-story-seed-agent'` with
`active = true`, when a Phase 2 stored procedure calls
`PERFORM workflow.validate_caller('pm-story-seed-agent')`, the function returns void and
execution continues normally.

**Example 2 — Unknown caller is rejected:**
Given `workflow.allowed_agents` does not contain `agent_id = 'rogue-process'`, when any
code calls `SELECT workflow.validate_caller('rogue-process')`, the function raises
`SQLSTATE 'P0001'` with message `'Unauthorized caller: rogue-process'`, aborting the
calling transaction.

**Example 3 — Deactivated caller is rejected:**
Given `workflow.allowed_agents` contains `agent_id = 'legacy-agent'` with `active = false`,
when a stored procedure calls `PERFORM workflow.validate_caller('legacy-agent')`, the
function raises `SQLSTATE 'P0001'` with message `'Unauthorized caller: legacy-agent'`.

---

## Edge Cases

**Edge Case 1 — NULL caller identity:**
A caller passing `NULL` as `caller_agent_id` must be rejected. Since `agent_id` is the
primary key (NOT NULL), no row will match a NULL lookup, and the function raises
`'Unauthorized caller: <null>'` or equivalent rejection.

**Edge Case 2 — Idempotent re-run:**
Running the migration SQL a second time must not duplicate seed rows or raise errors.
`CREATE TABLE IF NOT EXISTS` guards table creation; `INSERT ... ON CONFLICT DO NOTHING`
(or `INSERT WHERE NOT EXISTS`) guards seed data insertion. Row count after the second run
must equal the count after the first run.

## Architecture Notes

- `workflow.allowed_agents` must live in the `workflow` PostgreSQL schema — consistent with
  all other workflow objects (`workflow.stories`, `workflow.valid_transitions`, etc.).
- `validate_caller()` is a `RETURNS void` function: callers use `PERFORM workflow.validate_caller(...)`.
  On success it is a no-op; on failure it raises an exception that aborts the calling
  transaction.
- No FK constraint from `allowed_agents` to any other table — this is a standalone reference
  table with no cross-schema dependencies.
- The `allowed_procedures` column (text[]) records which stored procedures each agent is
  permitted to call. Phase 2 procedures may use this for fine-grained checks in the future;
  for MVP, `validate_caller` only checks presence + `active = true`.
