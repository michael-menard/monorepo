# CDBE-4020: Embedding Columns on Knowledge Tables

## Context

The knowledgebase database (port 5435) has five knowledge specialty tables — `public.lessons_learned`, `public.adrs`, `public.code_standards`, `public.rules`, and `public.cohesion_rules` — none of which have `embedding` columns today. The `EmbeddingClient` already generates embeddings for `public.knowledge_entries`, and the `public.embedding_cache` table handles deduplication. CDBE-4010 (upstream dependency, currently in "elab" state) establishes the HNSW index pattern on workflow tables and confirms the `vector` customType is reusable.

This story adds `embedding vector(1536)` columns (nullable) and HNSW indexes to all five knowledge tables, enabling the CDBE-4030 embedding worker to backfill and maintain these indexes, and enabling semantic similarity search across all KB entity types for retrieval-augmented generation and agent context lookup.

The implementation is a single SQL migration (manual authoring — `db:generate` has a known failure per WINT-0040), a Drizzle schema update to `kb.ts` (adding `embedding` column to five table definitions), and a pgtap test file verifying schema correctness.

**Blocking Dependency**: CDBE-4020 must not begin implementation (ST-1) until CDBE-4010 is merged. Elaboration is fully unblocked now.

---

## Goal

Add nullable `vector(1536)` embedding columns and HNSW cosine-similarity indexes to the five public-schema knowledge tables so the CDBE-4030 worker can backfill and query them for semantic search across all KB entity types.

---

## Examples

**Example 1 — Happy path migration:**
Given a knowledgebase DB with migrations 1001–1010 applied and pgvector >= 0.5.0 installed, when migration 1011 is applied, then `public.lessons_learned`, `public.adrs`, `public.code_standards`, `public.rules`, and `public.cohesion_rules` each gain an `embedding` column of type `vector(1536)` (nullable), and five HNSW indexes (`lessons_learned_embedding_idx`, etc.) are created — all existing rows retain `embedding = NULL` with no data loss.

**Example 2 — Drizzle type inference:**
Given the five table definitions in `kb.ts` are updated with `embedding: vector('embedding', { dimensions: 1536 })`, when `pnpm check-types --filter @repo/knowledge-base` runs, then the TypeScript compiler exits 0 and the inferred select/insert types for all five tables include an optional `embedding: number[] | null` field.

**Example 3 — Idempotent migration:**
Given migration 1011 has already been applied to the knowledgebase DB, when the migration SQL is run a second time, then all `ADD COLUMN IF NOT EXISTS` and `CREATE INDEX CONCURRENTLY IF NOT EXISTS` guards fire, no errors are raised, and `pg_indexes` still shows exactly one HNSW index per table (no duplicates).

---

## Edge Cases

**Edge Case 1 — Wrong database guard:**
Given the migration SQL is applied against `lego_dev` (port 5432) instead of `knowledgebase`, then the DO block safety preamble raises `EXCEPTION '1011: This migration must run against the ''knowledgebase'' database. Current database: lego_dev'` and halts with no DDL applied.

**Edge Case 2 — pgvector version too old:**
Given the knowledgebase DB has pgvector < 0.5.0 installed (HNSW was added in 0.5.0), when the migration runs, then the version guard DO block raises a clear `EXCEPTION` stating that HNSW requires pgvector >= 0.5.0, preventing a cryptic failure during index creation.

**Edge Case 3 — Empty table HNSW index:**
Given one or more of the five target tables has 0 rows when the migration runs, the pre-flight RAISE NOTICE logs `COUNT = 0` and `CREATE INDEX CONCURRENTLY` still succeeds — HNSW indexes can be created on empty tables without error.

**Edge Case 4 — CONCURRENTLY outside transaction:**
Given the migration SQL is structured incorrectly with `CREATE INDEX CONCURRENTLY` inside a `BEGIN/COMMIT` block, PostgreSQL raises `ERROR: CREATE INDEX CONCURRENTLY cannot run inside a transaction block`. The correct structure has each HNSW index creation as a top-level statement outside any explicit transaction wrapper.

## Non-Goals

- Do NOT create separate embedding tables (like CDBE-4010's normalized `story_embeddings` / `plan_embeddings` approach) — this story adds **inline columns** to the five existing tables.
- Do NOT implement embedding generation triggers, NOTIFY/LISTEN patterns, or OpenAI API calls — that is CDBE-4030's scope.
- Do NOT backfill existing rows with embeddings — that is CDBE-4030's scope.
- Do NOT change the existing ivfflat index on `public.knowledge_entries` — grandfathered, out of scope.
- Do NOT modify the `workflow.*` schema — all five target tables are in the `public` schema.
- Do NOT create or reference `wint.*` or `kbar.*` schemas.
- Do NOT modify `workflow.ts` — only `kb.ts` needs Drizzle schema updates.
- Do NOT attempt `pnpm db:generate` — it fails with a `sets.js` module resolution error (WINT-0040). Author migration SQL manually.

## Architecture Notes

- **HNSW vs. ivfflat**: HNSW chosen for all new embedding indexes in the CDBE-4xxx series. ivfflat is grandfathered only on pre-existing `knowledge_entries` and `workflow.plans` tables. HNSW requires no manual `lists` tuning and has higher recall at low row counts (< 1000). Document rationale in migration SQL comment.
- **Inline columns vs. separate tables**: CDBE-4010 uses normalized embedding tables (`story_embeddings`, `plan_embeddings`). CDBE-4020 deliberately uses inline columns on the five knowledge tables — the tables are small and denormalization reduces query complexity for the CDBE-4030 worker.
- **Nullable embedding columns**: The `embedding` column must be nullable (`NULL` = not yet computed). The NOT NULL constraint would break migration by requiring immediate backfill data — CDBE-4030 handles backfill separately.
- **`public.` prefix on vector operators**: pgvector is installed in the `public` schema. All `vector_cosine_ops` references must use `public.vector_cosine_ops` (not bare `vector_cosine_ops`).
- **No Drizzle functional indexes**: HNSW indexes cannot be expressed in Drizzle table definitions — they must exist in migration SQL only (see lessons learned from CDBE-4010).
