# CDBE-3010: stories_current View ‚Äî Single Query Target for Current Story State

## Context

The `workflow.story_state_history` table (deployed in migration 1010, CDBE-1010 complete) stores state transitions with `exited_at` tracking. An open row (`exited_at IS NULL`) represents a story's current state. The `idx_story_state_history_open_rows` partial index (`story_id, created_at DESC WHERE exited_at IS NULL`) is already deployed and supports efficient open-row lookup.

The existing `workflow.story_details` view (migration 1000) joins `story_state_history` as an aggregated JSONB array (last 20 rows), but does not expose a flat `current_state` / `entered_at` scalar. Every query that needs a story's current state with its `entered_at` timestamp must duplicate the same LATERAL join pattern against `story_state_history`. This is fragile, verbose, and not DRY.

## Goal

Create `workflow.stories_current` as a SQL view that joins all stories to their latest open state history row, exposing `current_state` and `entered_at` as scalar columns without requiring callers to write their own LATERAL join boilerplate.

## Examples

**Example 1 ‚Äî Story with active state history:**
Given a story `CDBE-3010` with one open row in `story_state_history` (`to_state = 'in_progress'`, `exited_at IS NULL`), a query of `SELECT current_state, entered_at FROM workflow.stories_current WHERE story_id = 'CDBE-3010'` returns `current_state = 'in_progress'` and a non-null `entered_at` timestamp ‚Äî no join required by the caller.

**Example 2 ‚Äî Story with multiple transitions (one closed, one open):**
Given a story that transitioned `backlog ‚Üí created ‚Üí in_progress`, the first two rows in `story_state_history` have `exited_at` set (closed). Only the open row (`to_state = 'in_progress'`, `exited_at IS NULL`) is returned by the LATERAL join, so `current_state = 'in_progress'`.

## Edge Cases

**Edge Case 1 ‚Äî Story with no history rows:**
A story that has never had a state transition inserted into `story_state_history` returns `current_state IS NULL` and `entered_at IS NULL`. The LEFT JOIN semantics of the LATERAL guarantee this without error.

**Edge Case 2 ‚Äî Second migration run (idempotency):**
Running `1030_cdbe3010_stories_current_view.sql` a second time exits 0 with no errors. `CREATE OR REPLACE VIEW` is inherently idempotent ‚Äî no `DROP VIEW` guard is needed. The safety preamble only validates the database name; it does not fail if the view already exists.

## Non-Goals

- Do not add an INSTEAD OF UPDATE trigger to `stories_current` ‚Äî the view is read-only; writes go to underlying tables directly.
- Do not add RLS policies directly to the view ‚Äî caller privileges are inherited from `workflow.stories` and `workflow.story_state_history` (SECURITY INVOKER default).
- Do not modify `workflow.story_details` (migration 1000) ‚Äî it remains the wide aggregation view; `stories_current` is the narrow scalar sibling.
- Do not add a `current_state` column to `workflow.stories` itself ‚Äî the view is the correct abstraction.
- Do not expose `from_state` in the view ‚Äî `current_state` + `entered_at` is sufficient.
- Do not create any MCP tool, API handler, or TypeScript service as part of this story.
- Do not create new indexes ‚Äî `idx_story_state_history_open_rows` from migration 1010 is sufficient.

## Architecture Notes

- `workflow.stories_current` is a narrow scalar sibling to `workflow.story_details`. Both views live in the `workflow` schema. The naming convention (`workflow.stories_current`, no `v_` prefix) is consistent with `workflow.story_details` ‚Äî the `v_` prefix is used for utility analytics views (e.g., `workflow.v_plan_churn_summary`).
- The view uses SECURITY INVOKER semantics (default for PostgreSQL views). Callers query with their own privileges. `agent_role`, `lambda_role`, and `reporting_role` already have SELECT on `workflow.story_state_history` (migration 1005 RLS model). No additional grants are needed.
- RLS on `workflow.story_state_history` is SELECT-scoped (USING clause only; no WITH CHECK). The view inherits caller's RLS context without additional configuration.
- The explicit column list in AC-2 is preferred over `s.*` to prevent the view silently growing new columns if `workflow.stories` gains columns in future migrations.
