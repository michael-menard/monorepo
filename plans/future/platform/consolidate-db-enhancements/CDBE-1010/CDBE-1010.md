# CDBE-1010: Valid Transitions INSERT Trigger on story_state_history (with exited_at / duration_seconds columns)

## Context

The `workflow.valid_transitions` lookup table (54 rows, migration 1004) is deployed and authoritative, and RLS policies on `workflow.story_state_history` are live (migration 1005, CDBE-1006 complete). The pgtap harness is operational (CDBE-0010 complete). The `validate_story_state_transition` BEFORE UPDATE trigger on `workflow.stories` (migration 1001) enforces the state machine using hardcoded IF chains; this story adds a second enforcement point directly on `story_state_history` that delegates to the `valid_transitions` table instead of duplicating logic.

The `story_state_history` table currently lacks `exited_at` and `duration_seconds` columns. The story requires adding these columns and implementing a BEFORE INSERT trigger that: validates incoming transitions against `valid_transitions`, rejects illegal ones with SQLSTATE 23514, and closes the previous open row by setting `exited_at = NOW()` and computing `duration_seconds`.

A pre-existing inconsistency exists: the `record_state_transition` trigger (migration 1001) inserts with `event_type = 'state_changed'`, but the `story_state_history.event_type` CHECK constraint only allows `['state_change', 'transition', 'phase_change', 'assignment', 'blocker', 'metadata_version']`. Migration 1010 must resolve this inconsistency.

## Goal

Enforce story state machine rules at the database level by adding duration-tracking columns to `story_state_history` and implementing a BEFORE INSERT trigger that rejects illegal state transitions and records how long each story spent in each state.

## Examples

**Given** a story with an open history row for state `backlog` (inserted 5 minutes ago, `exited_at IS NULL`), **when** a new row is inserted with `from_state='backlog'` and `to_state='created'` (a legal transition), **then** the INSERT succeeds, the previous open row has `exited_at = NOW()` and `duration_seconds ≈ 300`, and the new row is written.

**Given** an attempt to insert a row with `from_state='completed'` and `to_state='in_progress'` (an illegal transition — `completed → in_progress` is not in `valid_transitions`), **when** the INSERT is executed, **then** it raises an exception with SQLSTATE 23514 and a descriptive message identifying the illegal pair.

**Given** a brand-new story with no history rows (initial state recording), **when** a row is inserted with `from_state=NULL` and `to_state='backlog'`, **then** the INSERT succeeds because `NULL → backlog` is present in `valid_transitions` as an `initial_insert` transition.

## Edge Cases

**NULL from_state with no prior rows:** When inserting the first history row for a story (`from_state=NULL`), no previous open row exists to close. The trigger must handle this gracefully without raising an error or attempting an UPDATE that finds no rows.

**Multiple open rows:** If a story somehow has two open rows (both `exited_at IS NULL`), the trigger must close only the most recent one (by `created_at` descending, `LIMIT 1`). The older open row remains open to avoid masking data integrity issues.

**event_type constraint:** The pre-existing `record_state_transition` trigger inserts with `event_type='state_changed'` which is NOT in the CHECK constraint. If the constraint is currently enforced, this pre-existing trigger would be failing on every state change. Migration 1010 must fix this: update existing `state_changed` rows to `state_change` and update the `record_state_transition` function to use `'state_change'`. The new trigger uses `'transition'`.

**RLS interaction:** The trigger function runs as `SECURITY INVOKER` (the PostgreSQL default). When an `agent_role` connection inserts a row, the UPDATE on the previous open row also runs as `agent_role`. Confirm `agent_role` has UPDATE permission on `story_state_history` per migration 1005 RLS policies.

## Non-Goals

## Architecture Notes

- The trigger on `story_state_history` operates as a second enforcement point. The first (primary) is the BEFORE UPDATE trigger on `workflow.stories` (migration 1001). Having both ensures that any direct inserts into `story_state_history` (bypassing the stories table trigger) are also validated.
- Using the `valid_transitions` table for lookup (not hardcoded IF chains) means both enforcement points will automatically stay in sync if the transition matrix is updated — just update the `valid_transitions` rows.
- `SECURITY INVOKER` is the PostgreSQL default for trigger functions, but must be explicitly documented per team convention. This ensures the trigger runs with the caller's privileges, not the function creator's, which is correct behavior under the existing RLS policies.
- The UPDATE to close the previous open row happens inside a BEFORE INSERT trigger. PostgreSQL does NOT re-fire BEFORE INSERT triggers when the trigger itself issues an UPDATE on the same table, so there is no recursion risk.
