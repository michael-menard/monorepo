# CDBE-1006 — Row-Level Security Policies on Workflow Tables Before Trigger Deployment

## Context

The KB database currently runs all operations through a single role (`kbuser`) with no
row-level access controls on any workflow table. The state machine trigger
(`validate_story_state_transition`, migration 1001) runs as its creator's role and
enforces state transitions at the DDL level — but any caller with direct table access can
bypass the trigger entirely by issuing raw SQL, and no role boundary prevents one agent
from reading or modifying another agent's story records.

CDBE-1010 will deploy a high-privilege trigger on `workflow.story_state_history` that
rejects illegal transitions at the DB level. Without RLS, this trigger can be bypassed by
any connection with UPDATE privileges on `workflow.stories`. The three intended callers
(agent Lambda functions, reporting/analytics queries, and background automation agents)
currently have undifferentiated access to every row and every write path — including audit
tables that must be append-only for compliance.

**Critical correction**: The original story description referenced `telemetry.workflow_audit_log`.
No `telemetry` schema exists in the KB database (confirmed schemas: `analytics`, `artifacts`,
`drizzle`, `workflow`). All references in this story target `workflow.workflow_audit_log`.

## Goal

Establish PostgreSQL row-level security on all four core workflow tables — `workflow.stories`,
`workflow.plans`, `workflow.story_state_history`, and `workflow.workflow_audit_log` — by
creating three least-privilege named roles and their corresponding RLS policies, so that
each caller class is restricted to only the operations it is permitted to perform before
the CDBE-1010 trigger is deployed.

## Examples

**Example 1 — reporting_role read access (happy path)**

Given: Migration 1005 is applied. A row exists in `workflow.stories`.
Action: `SET ROLE reporting_role; SELECT * FROM workflow.stories LIMIT 1;`
Result: Query succeeds and returns the row (SELECT is permitted by policy).

Action: `SET ROLE reporting_role; INSERT INTO workflow.stories (...) VALUES (...);`
Result: `ERROR: permission denied for table stories` (42501) — INSERT is not granted.

**Example 2 — lambda_role full DML access (happy path)**

Given: Migration 1005 is applied.
Action: `SET ROLE lambda_role; UPDATE workflow.stories SET state = 'created' WHERE id = <id>;`
Result: UPDATE succeeds — lambda_role has full DML on workflow.stories.

**Example 3 — idempotent re-run (happy path)**

Given: Migration 1005 has already been applied once.
Action: Run `psql $KB_DATABASE_URL -f 1005_workflow_rls.sql` a second time.
Result: Exit 0, no errors. RAISE NOTICE lines confirm objects already exist, skipped.
`pg_policies` count is unchanged — no duplicate policies created.

## Edge Cases

**Edge Case 1 — RLS enabled without policies causes implicit DENY**

If `ALTER TABLE workflow.stories ENABLE ROW LEVEL SECURITY` executes but a subsequent
error prevents policy creation, all non-superuser connections to `workflow.stories` will
receive a silent permission denied on every query. The migration must enable RLS and create
policies atomically in the same DO block. `kbuser`'s BYPASSRLS grant must be applied
before any table's RLS is enabled.

**Edge Case 2 — SECURITY DEFINER trigger bypasses RLS**

The 1001 migration trigger (`validate_story_state_transition`) runs with the privileges
of its creator. If the creator is a superuser, the trigger bypasses RLS on
`workflow.stories` regardless of the active session role. This is expected PostgreSQL
behaviour and is documented in the migration header. The RLS policies protect direct
table access paths (agent, lambda, reporting roles) — the trigger's bypass is a separate
architectural concern, not a defect in this story.

**Edge Case 3 — kbuser superuser status affects BYPASSRLS requirement**

If `kbuser` has `rolsuper = true`, RLS is automatically bypassed and no explicit
`BYPASSRLS` grant is required. The migration includes `ALTER ROLE kbuser BYPASSRLS`
unconditionally — this is harmless for superusers and required for non-superusers.
Implementation must verify `kbuser` superuser status and document the finding in the
migration file comment.

**Edge Case 4 — Migration slot collision from concurrent sibling stories**

Slot 1005 is free as of 2026-03-18. If a concurrent CDBE sibling claims slot 1005 during
implementation, the next free integer must be used and all file references updated
accordingly. Verify at implementation start: `ls apps/api/knowledge-base/src/db/migrations/1005_*.sql`.

## Architecture Notes

**PostgreSQL RLS semantics (critical):**

- `ALTER TABLE ... ENABLE ROW LEVEL SECURITY` without any policies creates an implicit DENY for all non-superusers. Policies MUST be created atomically with (or before) the ENABLE statement.
- `FORCE ROW LEVEL SECURITY` ensures even the table owner is subject to RLS policies — use on all four tables.
- `SECURITY DEFINER` trigger functions bypass RLS on the tables they access. The 1001 trigger is in this category. This is documented but not changed by this story.
- `current_role` (not `current_user`) must be used in policy USING clauses when SET ROLE is in use, as the two differ after role switching.

**Role design rationale:**

- `agent_role`: Automation agents that INSERT new stories and record state history. Cannot DELETE or UPDATE stories (prevents accidental mutation).
- `lambda_role`: Full DML for the Lambda runtime's standard operations. This is the role that will replace `kbuser` at the application layer in a future infrastructure task.
- `reporting_role`: Read-only access for analytics and reporting queries. Cannot mutate any row.
- `kbuser`: Retains BYPASSRLS for backward compatibility during the migration window. Future work will migrate Lambda to connect as `lambda_role`.

**No Drizzle changes required:** RLS policies are not representable in Drizzle ORM schema. No changes to `apps/api/knowledge-base/src/db/schema/workflow.ts` or any other schema file.
