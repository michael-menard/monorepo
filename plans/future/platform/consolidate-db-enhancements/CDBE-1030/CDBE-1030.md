# CDBE-1030: Artifact Write and Plan Archival Cascade Triggers

## Context

Migration 1010 (CDBE-1010, now complete) established the pattern for database-managed state enforcement via BEFORE INSERT trigger on `workflow.story_state_history`. Two analogous automation gaps remain in the workflow schema:

1. The `artifacts.story_artifacts` table stores artifact iterations per story but has no mechanism to mark prior iterations as superseded when a new one is inserted. Queries for the "current" artifact must rely on `iteration DESC` heuristics instead of a dedicated `superseded_at` column.

2. When a `workflow.plan` is archived, its backlog stories are left in `backlog` state indefinitely. Agents must discover and transition them to `deferred` manually, introducing agent-managed relational state that the database should own.

Both gaps are addressed by a single migration following the CDBE-1010 pattern: `SECURITY INVOKER`, `CREATE OR REPLACE FUNCTION`, `DROP TRIGGER IF EXISTS / CREATE TRIGGER`, `ADD COLUMN IF NOT EXISTS`, `COMMENT ON` all objects, and pgtap test coverage.

**Dependency**: CDBE-1010 is complete. The `enforce_story_state_history_transition` BEFORE INSERT trigger (migration 1010) is deployed and validates all `story_state_history` inserts. Trigger 2 in this story will indirectly fire that trigger when bulk-updating stories to `'deferred'` — the `backlog → deferred` transition must exist in `workflow.valid_transitions` (verified by AC-14 pre-condition guard).

---

## Goal

Eliminate agent-managed artifact versioning and plan-story state cleanup by adding two database triggers that automatically supersede prior artifact iterations on insert and defer linked backlog stories when a plan is archived.

---

## Examples

**Example 1 — Artifact versioning (Trigger 1)**

Given: A story `WISH-042` has an existing `artifacts.story_artifacts` row with `artifact_type='dev_proof'`, `iteration=1`, `superseded_at IS NULL`.

When: A new row is inserted with `story_id='WISH-042'`, `artifact_type='dev_proof'`, `iteration=2`.

The system produces:
- The prior row (`iteration=1`) has `superseded_at` set to `NOW()`.
- The new row (`iteration=2`) has `superseded_at IS NULL` (current version).
- `workflow.stories.updated_at` for `WISH-042` is bumped to `NOW()`.

**Example 2 — Plan archival cascade (Trigger 2)**

Given: A plan `platform-q1` has three linked stories: `CDBE-1031` (backlog), `CDBE-1032` (backlog), `CDBE-1033` (in_progress).

When: `UPDATE workflow.plans SET status = 'archived' WHERE plan_slug = 'platform-q1'`.

The system produces:
- `CDBE-1031` and `CDBE-1032` have `state = 'deferred'` (were backlog, now deferred).
- `CDBE-1033` remains `in_progress` (only `backlog` stories are cascaded).

## Edge Cases

**Edge Case 1 — First artifact insert for a given type (Trigger 1 no-op)**

Given: No prior row exists in `artifacts.story_artifacts` for `(story_id='NEW-001', artifact_type='qa_report')`.

When: First row is inserted.

Expected: Trigger executes without error. No prior row is superseded. `workflow.stories.updated_at` is still bumped. The INSERT returns `NEW` normally.

**Edge Case 2 — Plan archived with zero linked backlog stories (Trigger 2 no-op)**

Given: Plan `empty-plan` has no entries in `workflow.plan_story_links`, or all linked stories are in states other than `backlog` (e.g., `ready`, `in_progress`, `UAT`).

When: `UPDATE workflow.plans SET status = 'archived' WHERE plan_slug = 'empty-plan'`.

Expected: Trigger fires, the bulk UPDATE affects 0 rows, no error is raised.

**Edge Case 3 — Non-archived status change (Trigger 2 guard)**

Given: Plan in `active` status with linked backlog stories.

When: `UPDATE workflow.plans SET status = 'in-progress'`.

Expected: Trigger condition `NEW.status = 'archived'` is false. Zero story rows are modified.

**Edge Case 4 — Missing backlog→deferred transition (pre-condition guard)**

Given: `workflow.valid_transitions` does not contain a row for `from_state='backlog'`, `to_state='deferred'`.

When: Migration 1020 is applied.

Expected: The AC-14 pre-condition DO block raises EXCEPTION with a descriptive message before any trigger functions are created. Migration fails fast rather than creating triggers that would cascade-fail at runtime.

## Architecture Notes

- **Trigger 1 ordering in migration**: `ADD COLUMN` → `CREATE INDEX` → `CREATE OR REPLACE FUNCTION` → `DROP / CREATE TRIGGER`. The function references `superseded_at` in an UPDATE statement — PostgreSQL PL/pgSQL validates column references at function execution time (not parse time), but the column must exist when the trigger fires. Adding the column first is still required for correctness.
- **Trigger 2 ENUM comparison**: `workflow.plans.status` is `plan_status_enum` (a PostgreSQL ENUM). Use `NEW.status = 'archived'::plan_status_enum` in the trigger body (explicit cast). `OLD.status IS DISTINCT FROM NEW.status` works correctly on ENUMs.
- **Trigger 2 and the 1010 trigger interaction**: Trigger 2's bulk UPDATE on `workflow.stories.state` will fire the `record_state_transition` function (installed by migration 1010 or earlier), which INSERTs into `workflow.story_state_history`. That INSERT is validated by the `enforce_story_state_history_transition` BEFORE INSERT trigger (migration 1010). If `backlog → deferred` is absent from `workflow.valid_transitions`, every story state update in the cascade will fail with SQLSTATE 23514. AC-14 guards against this at migration time.
- **SECURITY INVOKER**: Both trigger functions use `SECURITY INVOKER` (explicit, not default). Document in `COMMENT ON FUNCTION`. This means the trigger runs with the privileges of the session's current user/role, not the function owner. RLS on `artifacts.story_artifacts` and `workflow.stories` applies. Verify `agent_role` has `GRANT UPDATE (superseded_at) ON artifacts.story_artifacts` (from migration 1005 or add to 1020).
- **Bulk UPDATE scale**: `plan_story_links` join for large plans benefits from `idx_plan_story_links_plan_slug` index. Confirm index exists before assuming efficient cascade. Add to migration comments.

### Critical pgtap Pattern — SAVEPOINT + DO Block (MANDATORY)

Per KB lesson 23ea4ba2 (explicitly names CDBE-1030 as the failure case): all `CREATE TRIGGER` DDL inside pgtap test files must be wrapped in a `SAVEPOINT + DO block` that checks `pg_proc` for the trigger function's existence. Bare `CREATE TRIGGER` statements abort the entire test transaction if the function is absent.

```sql
SAVEPOINT before_trigger_create;
DO $$
BEGIN
  IF EXISTS (SELECT 1 FROM pg_proc WHERE proname = 'supersede_prior_artifact_version' AND pronamespace = 'artifacts'::regnamespace) THEN
    EXECUTE 'CREATE TRIGGER artifact_versions_supersede BEFORE INSERT ON artifacts.story_artifacts FOR EACH ROW EXECUTE FUNCTION artifacts.supersede_prior_artifact_version()';
  END IF;
END $$;
-- If function absent, SAVEPOINT ensures test transaction continues
```
