# CDBE-1050: Cascade Trigger DDL Prerequisites — story_assignments, story_blockers, and story_dependencies.resolved_at

## Split Context

This story is part of a split from CDBE-1020.
- **Original Story:** CDBE-1020 — Story Completion and Cancellation Cascade Triggers
- **Split Reason:** Elaboration identified 5 gaps (2 critical, 3 high). The `story_assignments` and `story_blockers` tables do not exist and cannot be referenced in the cascade triggers. The `story_dependencies.resolved_at` column is likewise absent. These missing DDL objects are a hard blocker for the trigger implementation.
- **This Part:** 1 of 2 (DDL prerequisites only — no trigger logic)
- **Dependency:** Depends on CDBE-1010 only. CDBE-1060 (trigger implementation) depends on this story.

## Context

The completion and cancellation cascade triggers planned in CDBE-1020 must operate on three tables that do not yet exist in the schema:

1. `workflow.story_assignments` — tracks which agents are assigned to a story; the cascade must soft-delete open assignments when a story completes or is cancelled.
2. `workflow.story_blockers` — tracks which stories are blocked by which other stories; the cancellation cascade must flag stories blocked by a cancelled story.
3. `workflow.story_dependencies.resolved_at` — the cascade must mark dependency rows resolved when the depended-on story completes.

Without these three DDL objects, the trigger functions in CDBE-1060 cannot be written or tested. This story creates the tables, adds the column, grants RLS permissions to `agent_role`, and writes pgtap schema-presence tests to verify the objects exist before CDBE-1060 begins implementation.

## Goal

Create the three missing database objects (`story_assignments`, `story_blockers`, `story_dependencies.resolved_at`) with soft-delete and RLS support so the cascade trigger implementation in CDBE-1060 has a valid schema to operate against.

## Examples

**Given** the migration has been applied, **when** an agent inserts a row into `workflow.story_assignments` with `story_id`, `agent_id`, and `assigned_at`, **then** the row is visible to `agent_role` under the RLS policy and the `deleted_at` column defaults to `NULL`.

**Given** the migration has been applied, **when** a pgtap test calls `has_table('workflow', 'story_assignments')`, `has_table('workflow', 'story_blockers')`, and `has_column('workflow', 'story_dependencies', 'resolved_at')`, **then** all three assertions pass.

## Edge Cases

**Idempotency:** If the migration is run a second time (e.g., re-applied in CI), `CREATE TABLE IF NOT EXISTS` and `ADD COLUMN IF NOT EXISTS` must ensure the second run exits with zero errors and no duplicate objects.

**RLS before table existence:** If a prior migration references these tables in an RLS policy before this migration runs, the policy creation will fail. This migration must be sequenced strictly after any migration that references these tables (none currently do, but verify at implementation time).

**story_dependencies.resolved_at on existing rows:** Existing rows in `story_dependencies` will have `resolved_at = NULL` after the column is added. This is intentional — NULL means not yet resolved. No backfill is required.

## Non-Goals

- Do NOT implement `story_completion_cascade()` or `story_cancellation_cascade()` trigger functions — those belong to CDBE-1060.
- Do NOT implement any AFTER UPDATE trigger on `workflow.stories` — that is CDBE-1060 scope.
- Do NOT add foreign key constraints from `story_assignments` or `story_blockers` to external tables not yet confirmed in the schema.
- Do NOT implement API endpoints, service layer, or UI for assignments or blockers.
- Do NOT seed `story_assignments` or `story_blockers` with existing data — these are new tables, empty at creation.

## Architecture Notes

- `story_assignments` and `story_blockers` are append-only with soft-delete via `deleted_at`. The cascade triggers in CDBE-1060 set `deleted_at = NOW()` on open rows rather than hard-deleting them. This preserves audit trail.
- `story_dependencies.resolved_at` is nullable. A NULL value means the dependency is not yet resolved. The cascade trigger sets it to `NOW()` when the depended-on story reaches a completed state.
- Both new tables live in the `workflow` schema, consistent with all other workflow objects.
- RLS policies use `USING (true)` for SELECT/INSERT and `USING (deleted_at IS NULL)` for UPDATE to prevent agents from modifying already soft-deleted rows.
