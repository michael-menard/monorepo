# CDBE-2020: `resolve_blocker` and `complete_artifact` Stored Procedures

## Context

Phase 1 of the consolidate-db-enhancements feature established the trigger and schema foundations for the workflow DB layer: valid transitions, state history enforcement, artifact versioning with supersession (CDBE-1030), plan archival cascades, and the `workflow.story_blockers` / `workflow.story_assignments` tables (migration 1050). Phase 1 also deployed `workflow.allowed_agents` and the `workflow.validate_caller()` entry-point guard (migration 1005 / CDBE-2005).

Phase 2 encapsulates the most common agent mutation patterns as atomic stored procedures. Currently, agents must perform multi-step manual DB operations for blocker resolution and artifact completion — both of which are prone to partial-state failures under concurrent access. CDBE-2020 implements two of these Phase 2 procedures as a single idempotent migration, eliminating these race windows.

The `workflow.story_blockers` table (deployed by migration 1050) has columns `id uuid PK`, `story_id text`, `blocked_by_story_id text`, `created_at timestamptz`, `deleted_at timestamptz`. It does NOT currently have a `resolution_notes` column — this migration adds that column as part of ST-1/ST-2.

---

## Goal

Encapsulate blocker resolution and artifact completion as atomic, caller-validated stored procedures so that agents cannot leave partial state when resolving blockers or recording completed artifacts.

---

## Examples

**Example 1 — Resolving a blocker that unblocks the story:**

Given: story `PIPE-3010` has one active blocker row in `workflow.story_blockers` (id = `<uuid>`, deleted_at = NULL).

Call: `SELECT * FROM workflow.resolve_blocker('<uuid>', 'dev-execute-leader', 'upstream story PIPE-2040 completed');`

Result: The blocker row's `deleted_at` is set to `NOW()`. The function returns `(story_id='PIPE-3010', fully_unblocked=true)` because no remaining active blocker rows exist for `PIPE-3010`.

**Example 2 — Recording a completed artifact atomically:**

Given: story `CDBE-2020` has no existing `plan` artifact for `artifact_name='PLAN.yaml'`, `iteration=0`.

Call: `PERFORM workflow.complete_artifact('CDBE-2020', 'plan', 'PLAN.yaml', 'implementation', 0, 'sha256:abc', 'dev-implement-planner', '{"sections": 4}'::jsonb);`

Result: One row is inserted into `artifacts.story_artifacts`. The `artifact_versions_supersede` BEFORE INSERT trigger fires and supersedes any prior active version for the same `(story_id, artifact_type)`. Calling with the same `(story_id, artifact_type, artifact_name, iteration)` a second time triggers the `ON CONFLICT DO UPDATE` path — the row is updated without creating a duplicate and without re-firing the BEFORE INSERT trigger.

## Edge Cases

**Edge Case 1 — Unknown caller is rejected before any mutation:**

If `caller_agent_id` is not present in `workflow.allowed_agents` with `active=true`, both `resolve_blocker` and `complete_artifact` raise `SQLSTATE P0001` on the very first statement (`PERFORM workflow.validate_caller(caller_agent_id)`). No blocker rows are soft-deleted and no artifact rows are inserted or updated.

**Edge Case 2 — Already-resolved blocker raises an exception, not a silent no-op:**

If `resolve_blocker` is called with a `blocker_id` whose `deleted_at` is already set (the blocker was previously resolved), the function raises an informative exception rather than silently succeeding. This ensures callers are aware of double-resolution attempts and can log or handle them appropriately.

**Edge Case 3 — DO UPDATE path on `complete_artifact` does not re-fire supersession trigger:**

The `artifact_versions_supersede` trigger is a `BEFORE INSERT` trigger. When `complete_artifact` executes its `INSERT ... ON CONFLICT DO UPDATE`, the UPDATE branch does not re-fire the trigger. Supersession only occurs on the initial INSERT attempt (new iteration). Subsequent upserts for the same `(story_id, artifact_type, artifact_name, iteration)` update the row in-place without creating a new supersession event.

## Architecture Notes

- Both procedures use `SECURITY INVOKER` (not `SECURITY DEFINER`) — they run with the caller's privileges, consistent with the established pattern across all CDBE migrations. This means `agent_role` must have appropriate grants on the underlying tables.
- `complete_artifact` uses atomic `INSERT ... ON CONFLICT DO UPDATE` against the `(story_id, artifact_type, artifact_name, iteration)` unique constraint from migration 999 baseline. No advisory locks, no SELECT-before-INSERT.
- `resolve_blocker` returns `RETURNS TABLE (story_id text, fully_unblocked boolean)` using `RETURN NEXT`. The `fully_unblocked` flag is informational — the calling agent is responsible for deciding whether to trigger a state transition via `advance_story_state` (CDBE-2010).
- The `artifact_versions_supersede` BEFORE INSERT trigger fires on the INSERT attempt within `complete_artifact`. When the ON CONFLICT clause routes to DO UPDATE, the trigger does NOT fire again — supersession is only triggered by new insertions.
- `resolution_notes text` column is added to `workflow.story_blockers` by this migration (`ALTER TABLE ... ADD COLUMN IF NOT EXISTS`) — the column did not exist in migration 1050.
