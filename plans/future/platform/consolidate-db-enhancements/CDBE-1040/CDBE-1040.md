# CDBE-1040: Audit Trigger on Stories and Plans (Row-Level UPDATE Audit)

## Context

The `workflow.stories` and `workflow.plans` tables are the primary mutation targets in the pipeline. CDBE-1010 established state enforcement triggers and CDBE-1020/1030 established cascade triggers. No general-purpose audit trail currently captures which columns changed, what values they held, and when the change occurred — only state transitions are tracked via `story_state_history`.

CDBE-1040 closes this gap by adding AFTER UPDATE triggers on both tables that write column-level change records to a new dedicated audit table (`workflow.story_mutation_audit_log`). Sensitive column values are masked before writing.

**Schema correction (from seed conflict analysis):** The original story description referenced `telemetry.workflow_audit_log` — this schema does not exist. The deployed table is `workflow.workflow_audit_log`, but it has an `execution_id uuid NOT NULL` FK constraint that makes it unsuitable for a generic trigger. This story creates a new `workflow.story_mutation_audit_log` table with the correct shape.

**Dependency:** Requires CDBE-1010 (completed).

---

## Goal

Provide an immutable, column-level audit trail of all UPDATE mutations to `workflow.stories` and `workflow.plans`, with automatic redaction of sensitive column values, enabling debugging and compliance reconstruction of any row's history beyond state transitions.

---

## Examples

**Example 1 — Story priority change audited:**
Given a story with `priority = 'medium'`, when `UPDATE workflow.stories SET priority = 'high' WHERE story_id = 'CDBE-1040'` is executed, the system produces one row in `workflow.story_mutation_audit_log` with `table_name = 'workflow.stories'`, `row_id = 'CDBE-1040'`, `column_name = 'priority'`, `old_value = 'medium'`, `new_value = 'high'`.

**Example 2 — Plan status change audited:**
Given a plan with `status = 'draft'`, when `UPDATE workflow.plans SET status = 'active' WHERE plan_slug = 'my-plan'` is executed, the system produces one row in `workflow.story_mutation_audit_log` with `table_name = 'workflow.plans'`, `row_id = 'my-plan'`, `column_name = 'status'`, `old_value = 'draft'`, `new_value = 'active'`.

**Example 3 — Sensitive column redacted:**
Given a story has an `api_key` column updated, when the AFTER UPDATE trigger fires, the system writes `old_value = '[REDACTED]'` and `new_value = '[REDACTED]'` in the audit row — the column name `api_key` is still recorded but neither value is exposed.

**Example 4 — Multiple columns changed in one UPDATE:**
Given `UPDATE workflow.stories SET priority = 'critical', title = 'New Title'`, the system produces two audit rows for that story — one per changed column.

---

## Edge Cases

**Edge Case 1 — Unchanged column produces no audit row:**
When `UPDATE workflow.stories SET priority = 'medium'` is executed and `priority` was already `'medium'`, the trigger detects `OLD.priority IS NOT DISTINCT FROM NEW.priority` and writes zero rows to `workflow.story_mutation_audit_log` for the `priority` column.

**Edge Case 2 — INSERT does not trigger audit:**
When a new story is INSERT-ed into `workflow.stories`, the AFTER UPDATE trigger does not fire (it is FOR EACH ROW AFTER UPDATE only), producing zero audit rows.

**Edge Case 3 — Plan archival cascade triggers auditing of each cascaded story:**
When `workflow.plans` is updated to `status = 'archived'`, the `plan_archival_cascade` trigger (migration 1020) bulk-updates linked `workflow.stories` rows from `backlog` to `deferred`. Each individual story UPDATE fires `audit_story_mutations`, writing one audit row per story for the `state` column change. No recursive loop occurs because `audit_story_mutations` writes to a separate table (`workflow.story_mutation_audit_log`), not back to `workflow.stories`.

**Edge Case 4 — NULL column value before update:**
When a nullable column previously holding `NULL` is updated to a non-NULL value, the audit row records `old_value = NULL` (or text-cast `NULL`) and `new_value = 'new-value'` without crashing.

## Architecture Notes

### New Table: workflow.story_mutation_audit_log

The story creates a new dedicated table rather than inserting into the existing `workflow.workflow_audit_log`. Rationale:

1. `workflow.workflow_audit_log` has `execution_id uuid NOT NULL` FK to `workflow.workflow_executions` — a generic trigger cannot supply this.
2. `workflow.workflow_audit_log` has RLS enabled with `FORCE ROW LEVEL SECURITY`; `agent_role` has no INSERT policy on it.
3. A dedicated table (`workflow.story_mutation_audit_log`) has the correct column shape for row-level auditing and avoids touching an existing production table.

**Proposed schema for workflow.story_mutation_audit_log:**
```sql
CREATE TABLE IF NOT EXISTS workflow.story_mutation_audit_log (
  id            bigserial PRIMARY KEY,
  table_name    text        NOT NULL,
  row_id        text        NOT NULL,
  column_name   text        NOT NULL,
  old_value     text,
  new_value     text,
  changed_at    timestamptz NOT NULL DEFAULT NOW()
);
GRANT INSERT ON workflow.story_mutation_audit_log TO agent_role;
GRANT INSERT ON workflow.story_mutation_audit_log TO lambda_role;
GRANT SELECT ON workflow.story_mutation_audit_log TO agent_role, lambda_role;
```

### Column Diffing Strategy

Use explicit column enumeration in the trigger body (not dynamic `hstore` diff). This is type-safe, requires no extension availability check, and makes the audited column list explicit and reviewable.

```plpgsql
-- Example within trigger body:
IF OLD.priority IS DISTINCT FROM NEW.priority THEN
  INSERT INTO workflow.story_mutation_audit_log (table_name, row_id, column_name, old_value, new_value)
  VALUES ('workflow.stories', NEW.story_id, 'priority',
    workflow.redact_sensitive_value('priority', OLD.priority::text),
    workflow.redact_sensitive_value('priority', NEW.priority::text));
END IF;
```

### Trigger Ordering with plan_archival_cascade

When `workflow.plans.status` is updated to `'archived'`:
1. `plan_archival_cascade` (AFTER UPDATE, 1020) fires → bulk-updates linked stories from `backlog` to `deferred`
2. For each story UPDATE, `audit_story_mutations` (AFTER UPDATE, 1040) fires → writes audit row for `state` column change

This is correct behavior. No recursive risk: `audit_story_mutations` writes to `workflow.story_mutation_audit_log`, not back to `workflow.stories`.
