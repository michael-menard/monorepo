# WINT-0020: Create Story Management Tables

## Context

WINT-0010 established the core WINT database foundation with 6 schema groups in the `wint` PostgreSQL schema namespace. The Story Management schema group includes base tables (`stories`, `storyStates`, `storyTransitions`, `storyDependencies`), but lacks tables for critical story lifecycle capabilities needed by the autonomous development workflow.

**Reality Baseline** (2026-02-13):
- WINT-0010 completed: Core schemas exist in `packages/backend/database-schema/src/schema/wint.ts` with 4 base Story Management tables
- KBAR-0010 completed: KBAR schema provides reference implementation with artifact tracking patterns
- No active platform stories with direct overlap
- Drizzle ORM v0.44.3 active with PostgreSQL Aurora
- No ADR constraints found (ADR-LOG.md missing)

**Current Limitations**:
The existing Story Management schema cannot track:
1. **Story artifacts** - No linkage between stories and their YAML/Markdown artifacts (PLAN.yaml, SCOPE.yaml, EVIDENCE.yaml, etc.)
2. **Story phases** - No granular phase tracking beyond state (need setup, plan, execute, review, qa sub-phases)
3. **Story metadata versioning** - No audit trail for story metadata changes (title, description, ACs)
4. **Story assignments** - No tracking of which agents/users are working on which stories
5. **Story blockers** - No detailed blocker tracking beyond dependency relationships

## Goal

Extend the Story Management schema group with 5 additional tables to enable story lifecycle tracking:
1. `storyArtifacts` - Links stories to filesystem artifacts with checksum-based change detection
2. `storyPhaseHistory` - Granular phase execution tracking (setup → plan → execute → review → qa)
3. `storyMetadataVersions` - Audit trail for story metadata changes with JSONB snapshots
4. `storyAssignments` - Tracks agent/user assignments to stories with phase-level granularity
5. `storyBlockers` - Detailed blocker tracking with type, severity, and resolution metadata

These tables will integrate with existing WINT-0010 tables via foreign key relations and follow established patterns (UUID PKs, timestamps, indexes, Drizzle relations, Zod schemas).

### AC-1: storyArtifacts Table

Extend `wint.ts` with `storyArtifacts` table to link stories to filesystem artifacts.

**Table Structure**:
```typescript
storyArtifacts = wintSchema.table('story_artifacts', {
  id: uuid('id').primaryKey().defaultRandom(),
  storyId: uuid('story_id').notNull().references(() => stories.id, { onDelete: 'cascade' }),
  artifactType: artifactTypeEnum('artifact_type').notNull(),
  filePath: text('file_path').notNull(),
  checksum: varchar('checksum', { length: 64 }).notNull(), // SHA-256 hex string
  lastSyncedAt: timestamp('last_synced_at', { withTimezone: true }),
  createdAt: timestamp('created_at', { withTimezone: true }).notNull().defaultNow(),
  updatedAt: timestamp('updated_at', { withTimezone: true }).notNull().defaultNow()
})
```

**Constraints**:
- Foreign key: `story_id` → `stories.id` (cascade delete)
- Unique constraint: `(story_id, artifact_type)` to prevent duplicate artifact types per story

**Indexes**:
- `story_id` (single-column for FK lookups)
- `artifact_type` (single-column for type filtering)
- `file_path` (single-column for path lookups)
- `(story_id, artifact_type)` (composite for uniqueness enforcement)

## Architecture Notes

### Database Design Decisions

**Schema Namespace**: All tables use existing `wintSchema` (no new pgSchema created) to maintain schema cohesion and simplify cross-table queries.

**Enum Placement**: New enums (artifactTypeEnum, phaseEnum, etc.) are placed in the `public` schema for cross-namespace reusability, following PostgreSQL best practices.

**Foreign Key Strategy**: All tables use FK to `stories.id` with `ON DELETE CASCADE` to ensure referential integrity and automatic cleanup when stories are deleted.

**Indexing Strategy**: Composite indexes prioritize high-cardinality columns first (e.g., `story_id` before `artifact_type`) for optimal query performance.

**JSONB Usage**: `storyMetadataVersions.metadata_snapshot` uses JSONB for flexible schema-less storage of story metadata snapshots, enabling future metadata structure changes without migrations.

**Checksum Format**: SHA-256 checksums stored as 64-character hex strings (VARCHAR(64)) for human readability and debugging, despite binary storage (BYTEA) being more space-efficient. Tradeoff: 32 extra bytes per artifact for better DX.

**Phase Execution Model**: Phase ordering is NOT enforced at database level. LangGraph workflow manages phase transitions. Database allows flexible phase recording for workflow recovery and debugging.

### Scalability Considerations

**Index Overhead**: 15+ indexes across 5 tables will impact write performance at scale (100K+ stories). Mitigation: Monitor slow query logs, consider partial indexes for hot paths.

**JSONB Performance**: Full table scans on JSONB queries will degrade at scale. Mitigation: Add GIN indexes on frequently-queried JSONB paths when usage patterns emerge.

**Cascade Delete Performance**: Deleting stories with hundreds of related records may lock tables. Mitigation: Consider soft delete pattern (deleted_at timestamp) for production use.

**Version History Growth**: `storyMetadataVersions` table will grow faster than other tables (one record per metadata change). Mitigation: Implement retention policy (archive versions > 90 days old).
