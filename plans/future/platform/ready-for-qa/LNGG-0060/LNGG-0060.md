# LNGG-0060: Checkpoint Adapter - Type-safe read/write for CHECKPOINT.yaml workflow state

## Context

The orchestrator workflow system uses CHECKPOINT.yaml files to track phase progression (setup → plan → execute → proof → review → fix → qa-verify → done) and enable deterministic resume after interruption. These files are currently managed manually by agents, leading to schema drift and inconsistent format across 100+ existing checkpoint files.

**Current State**:
- Checkpoint schema defined in `packages/backend/orchestrator/src/artifacts/checkpoint.ts` with 13 core fields
- 100+ CHECKPOINT.yaml files exist across `plans/future/*/` directories (wish, bug-fix, instructions, repackag-app, platform)
- Manual YAML writes by agents lack validation and atomic write protection
- No type-safe read/update operations available
- Resume logic in `idempotency.ts` expects consistent checkpoint format
- Survey of existing files shows some format variations (`e2e_gate`, `moved_to_uat` fields present in some files)

**Problem**: Without a type-safe adapter, agents can write invalid checkpoint data, leading to:
- Resume failures - workflow cannot determine where to continue after interruption
- Schema drift - different agents write different field structures
- Corruption risk - non-atomic writes during interruption leave partial/corrupted state
- No validation feedback - malformed checkpoints discovered at resume time, not write time

**Established Patterns**:
- StoryFileAdapter (LNGG-0010) provides reference pattern for file adapter implementation
- Atomic write pattern using temp file + rename prevents corruption
- Zod schema validation ensures type safety at runtime
- Batch operations for performance when reading multiple files

**Dependencies**:
- **BLOCKING**: LNGG-0010 (Story File Adapter) currently in "fix" phase with TypeScript compilation failures
- This story reuses file-utils.ts, yaml-parser.ts, and error types from LNGG-0010
- Cannot start implementation until LNGG-0010 reaches `completed` status

## Goal

Create a CheckpointAdapter class following the established StoryFileAdapter pattern that provides:
- Type-safe read/write/update operations with Zod validation
- Atomic writes using temp file + rename pattern to prevent corruption
- Phase advancement helpers (updatePhase, advanceToNextPhase)
- Batch read operations for workflow dashboard/metrics
- Backward-compatible with existing CHECKPOINT.yaml files (100+ files in production)

**Success Criteria**: LangGraph persistence nodes can safely checkpoint workflow state to filesystem with guaranteed schema compliance and atomic writes.

### Patterns to Follow

1. **Adapter Class Structure** (from StoryFileAdapter):
```typescript
class CheckpointAdapter {
  async read(filePath: string): Promise<Checkpoint>
  async write(filePath: string, checkpoint: Checkpoint): Promise<void>
  async update(filePath: string, updates: Partial<Checkpoint>): Promise<void>
  async exists(filePath: string): Promise<boolean>
  async readBatch(filePaths: string[]): Promise<BatchReadResult>

  // Convenience helpers
  async advancePhase(filePath: string, completed: Phase, next: Phase): Promise<void>
  async markPhaseBlocked(filePath: string, reason: string): Promise<void>
  async clearBlocked(filePath: string): Promise<void>
}
```

2. **Atomic Write Pattern**:
```typescript
const tempPath = `${filePath}.tmp`
await fs.writeFile(tempPath, content, 'utf-8')
await fs.rename(tempPath, filePath)
// Cleanup temp file on error in finally block
```

3. **Zod Validation Pattern**:
```typescript
const result = CheckpointSchema.safeParse(data)
if (!result.success) {
  throw new ValidationError('Checkpoint validation failed', result.error.errors, filePath)
}
return result.data
```

4. **Error Handling**:
```typescript
try {
  const content = await readFileSafe(filePath)
} catch (error) {
  if (error.code === 'ENOENT') {
    throw new CheckpointNotFoundError(filePath)
  }
  throw new ReadError(filePath, error)
}
```

## Architecture Notes

### Adapter Pattern

CheckpointAdapter follows pure file I/O adapter pattern:
- **No business logic** - Only read/write/validate operations
- **Transport-agnostic** - Works with any file path (local, network, etc.)
- **Type-safe** - Uses Zod for runtime validation
- **Atomic writes** - Prevents corruption via temp file + rename

### Integration with LangGraph

**Consumers**:
1. **Persistence Nodes** (`src/nodes/persistence/`) - Load/save checkpoint state to PostgreSQL
2. **Idempotency Utils** (`src/utils/idempotency.ts`) - Resume logic reads checkpoint to determine next phase
3. **Phase Leaders** (setup, plan, execute, etc.) - Write checkpoint at phase completion

**Usage Pattern**:
```typescript
import { CheckpointAdapter } from '@repo/orchestrator/adapters'

const adapter = new CheckpointAdapter()

// Resume: read checkpoint to determine current phase
const checkpoint = await adapter.read(`${storyDir}/_implementation/CHECKPOINT.yaml`)
const currentPhase = checkpoint.current_phase

// Execute phase...

// On success: advance to next phase
await adapter.advancePhase(
  `${storyDir}/_implementation/CHECKPOINT.yaml`,
  'plan',
  'execute'
)

// On failure: mark blocked
await adapter.markPhaseBlocked(
  `${storyDir}/_implementation/CHECKPOINT.yaml`,
  'Missing API key configuration'
)
```
