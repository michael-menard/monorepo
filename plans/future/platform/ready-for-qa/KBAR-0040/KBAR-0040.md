# KBAR-0040: Artifact Sync Functions

## Context

KBAR-0030 established the `packages/backend/kbar-sync/` package with bidirectional story file sync functions â€” syncing story `.md`/`.yaml` files between the filesystem and the `kbar.stories` database table. That work is currently in `ready-for-code-review` / UAT status.

However, story execution produces many non-story artifact files that are not yet covered by any sync function:

- `_implementation/PLAN.yaml` â€” plan artifact
- `_implementation/SCOPE.yaml` â€” scope artifact
- `_implementation/EVIDENCE.yaml` â€” evidence artifact
- `_implementation/CHECKPOINT.yaml` â€” checkpoint artifact
- `_implementation/DECISIONS.yaml` â€” decisions artifact
- `_implementation/KNOWLEDGE-CONTEXT.yaml` â€” knowledge context artifact
- `_pm/STORY-SEED.md`, `_pm/TEST-PLAN.md`, `_pm/DEV-FEASIBILITY.md` â€” elaboration/test_plan artifacts
- `PROOF-*.md`, `REVIEW.yaml`, `QA-VERIFY.yaml` â€” review artifacts

The `kbar.artifacts`, `kbar.artifactVersions`, `kbar.artifactContentCache`, `kbar.syncCheckpoints`, and `kbar.syncConflicts` tables were designed from the start (KBAR-0010) to hold these non-story artifact types. The `kbarArtifactTypeEnum` already enumerates: `story_file`, `elaboration`, `plan`, `scope`, `evidence`, `review`, `test_plan`, `decisions`, `checkpoint`, `knowledge_context`. No sync functions exist yet for any of these types.

Additionally, KBAR-0030 deferred all batch operations. Running sync one story at a time is insufficient for bootstrapping the database from the existing filesystem state, which contains hundreds of stories with multiple artifacts each.

**Prerequisite**: KBAR-0040 must not start implementation until KBAR-0030 is merged to main. If KBAR-0030 review surfaces changes to `__types__/index.ts` or the package structure, KBAR-0040 must accommodate those changes.

## Goal

Extend `packages/backend/kbar-sync/` with:

1. **Single-artifact sync functions** for each non-story artifact type â€” bidirectional (filesystemâ†’DB and DBâ†’filesystem), following the exact patterns established in KBAR-0030: checksum-based idempotency, graceful error handling, sync event tracking, Zod validation, security-first path handling.

2. **Batch sync operations** that process all artifacts for a story, all artifacts of a given type across all stories, or all artifacts for all stories â€” with checkpoint-based resumption and per-artifact fault isolation.

This gives KBAR-0050 (CLI Sync Commands) the functions it needs to provide a usable `kbar sync artifacts` command, and enables bootstrapping the database from the full filesystem state.

## Acceptance Criteria

- [ ] **AC-1**: Sync non-story artifact from filesystem to database
  - Accept `storyId`, `artifactType` (from `kbarArtifactTypeEnum`, excluding `story_file`), and `filePath`
  - Read file content and compute SHA-256 checksum using `computeChecksum` from `__types__/index.ts`
  - Skip sync if checksum unchanged â€” idempotency (return `syncStatus: 'skipped'`)
  - Insert or update `kbar.artifacts` row with checksum, `syncStatus: 'synced'`, `lastSyncedAt`
  - Insert version record in `kbar.artifactVersions` on content change (sequential version number computed inside transaction, `contentSnapshot`)
  - Create `kbar.syncEvents` record (`eventType: 'artifact_sync'`) with start/end timing
  - Return typed result: `{ success, artifactId, checksum, syncStatus, syncEventId }`

- [ ] **AC-2**: Sync non-story artifact from database to filesystem
  - Accept `storyId`, `artifactType`, and `outputPath`
  - Read artifact content from `kbar.artifactContentCache` if cache is valid (checksum matches `artifacts.checksum` and not expired); otherwise read from `kbar.artifactVersions` latest `contentSnapshot`
  - Increment `hitCount` when cache is used
  - Write to filesystem using atomic write pattern: write to `.tmp` file then `rename()` to final path
  - Update `artifacts.lastSyncedAt`
  - Create `kbar.syncEvents` record
  - Return typed result: `{ success, filePath, syncStatus, syncEventId }`

- [ ] **AC-3**: Cache artifact content on sync
  - After successful filesystemâ†’DB sync (`syncArtifactToDatabase`), upsert parsed YAML/Markdown content into `kbar.artifactContentCache` using Drizzle `.onConflictDoUpdate()` (unique index on `artifactId`)
  - `artifactContentCache.checksum` must match `artifacts.checksum` for cache to be valid
  - `hitCount` incremented on cache reads in `syncArtifactFromDatabase`
  - Cache `expiresAt` set to `now + 24h` (configurable TTL, default 24h)
  - If content fails YAML parsing (malformed file), skip cache upsert gracefully â€” do not fail the overall sync

- [ ] **AC-4**: Batch sync all artifacts for a story
  - Accept `storyId` and optional `artifactTypes` filter (array of `kbarArtifactTypeEnum` values)
  - Discover all artifact files in the story's directory using `ARTIFACT_FILENAME_MAP` (canonical filenameâ†’type mapping defined in `__types__/index.ts`)
  - Sync each artifact independently using `syncArtifactToDatabase` â€” one artifact failure must not abort the batch
  - Aggregate results: `{ filesScanned, filesChanged, filesSkipped, failureCount, conflictsDetected, perArtifactStatus[] }`
  - Create a single `kbar.syncEvents` record for the batch operation (`eventType: 'artifact_sync'`)
  - Return batch result with per-artifact status

- [ ] **AC-5**: Batch sync artifacts by type across all stories
  - Accept `artifactType` (from `kbarArtifactTypeEnum`) and optional `epic` filter string
  - Discover all files matching the artifact type across the story directory tree
  - Use `kbar.syncCheckpoints` to record progress (`checkpointType: 'artifact_type'`) â€” update checkpoint after each successfully processed artifact inside its transaction
  - Allow resumption after interruption: re-running picks up from the last checkpoint and does not re-process already-completed artifacts
  - Return batch result: `{ filesScanned, filesChanged, failureCount, checkpointId }`

- [ ] **AC-6**: Conflict detection for artifacts
  - Create `detectArtifactConflicts` function with the same interface pattern as KBAR-0030's `detectSyncConflicts`
  - Accept `storyId`, `artifactType`, and `filePath`
  - Compute filesystem checksum; compare against `kbar.artifacts.checksum`
  - If mismatch: insert `kbar.syncConflicts` row with `artifactId` FK, return `{ hasConflict: true, artifactId, filesystemChecksum, databaseChecksum, resolutionOptions }`

## Architecture Notes

### Patterns to Follow (Inherited from KBAR-0030)

- **Zod-first types**: All function inputs and outputs defined as Zod schemas; infer TypeScript types via `z.infer<>`. No interfaces.
- **Graceful error handling**: Every function returns a typed result object; never throws to caller; `syncEvents.status` updated to `'failed'` on error.
- **Checksum-based idempotency**: Compare SHA-256 checksum before writing; skip if unchanged.
- **Sync event tracking**: Create pending event at start, update to completed/failed at end, record timing.
- **Atomic file writes**: Write to `.tmp` then `fs.rename()` â€” never write directly to final path.
- **Security-first path handling**: Call `validateFilePath` and `validateNotSymlink` on every path before any I/O.
- **Drizzle ORM transactions**: Wrap multi-table operations in `db.transaction(async tx => { ... })` for atomicity.
- **Import shared utilities**: Use `computeChecksum`, `validateInput`, `normalizeOptionalField` from `__types__/index.ts` â€” do not re-implement.
- **Join queries over N+1**: Use Drizzle `.leftJoin()` to fetch related rows in one query; do not query the same table twice for the same row.
- **Checkpoint-based batch resumption**: Update `kbar.syncCheckpoints` after each artifact inside its transaction.

### Patterns to Avoid (from KBAR-0030 findings)

- Implementing `computeChecksum` locally â€” already exported from `__types__/index.ts`
- Raw SQL strings â€” use Drizzle ORM query builder exclusively
- `as any` casts in transaction callbacks or metadata â€” define Zod schemas, use proper Drizzle types
- Aborting entire batch on first error â€” batch functions must be fault-tolerant
- Assuming artifact file structure without Zod validation â€” parse and validate YAML content
- Ignoring `artifactVersions` table â€” always record version history for audit/rollback
- Skipping path security in batch discovery â€” even discovered paths must be validated

### Version Number Integrity

The `MAX(version) + 1` computation for `artifactVersions` must happen inside the Drizzle transaction to avoid race conditions between concurrent sync calls for the same artifact. Do not read `MAX(version)` before entering the transaction.

### Content Cache Upsert

Use Drizzle's `.onConflictDoUpdate()` for the `artifactContentCache` insert since the table has a unique index on `artifactId`. Never use a `SELECT + INSERT OR UPDATE` pattern.
