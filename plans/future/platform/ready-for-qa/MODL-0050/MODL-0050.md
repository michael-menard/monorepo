# MODL-0050: Add MiniMax Model Provider to LangGraph

## Context

MODL-0010 established a provider adapter system supporting OpenRouter, Ollama, and Anthropic LLM providers. The system uses a pluggable architecture with BaseProvider abstract class (from MODL-0011 refactoring) that implements the Template Method pattern for caching and lifecycle management.

MiniMax is a Chinese AI startup providing natural language processing models used by over 200 million users globally. Their API offers models including abab5.5, abab5.5s, and abab6 families. LangChain JS supports MiniMax via the `@langchain/community` package with the `ChatMinimax` class.

Adding MiniMax as a provider expands model selection options for LangGraph workflows, particularly for Chinese language tasks and cost-optimized alternatives to Western providers.

**Current State**:
- 3 provider adapters exist: OpenRouter, Ollama, Anthropic
- BaseProvider abstract class provides template method pattern
- Provider factory routes models by prefix (e.g., `anthropic/*`, `ollama/*`)
- All providers implement ILLMProvider interface
- Caching strategy: static instance cache, availability cache, config cache

**Problem**:
The orchestrator cannot currently access MiniMax models because:
1. No MiniMax provider adapter exists in `packages/backend/orchestrator/src/providers/`
2. Provider factory doesn't recognize `minimax/*` model prefix
3. No MiniMax configuration schema or environment variable loading
4. No availability checking for MiniMax API endpoint

**Solution**:
Create a MiniMax provider adapter following the established pattern from MODL-0010/MODL-0011:
1. Extend BaseProvider abstract class to inherit caching and lifecycle logic
2. Implement provider-specific methods: `parseModelName()`, `loadConfig()`, `createModel()`, `checkAvailability()`
3. Use `ChatMinimax` from `@langchain/community` for LangChain integration
4. Add factory routing for `minimax/*` prefix
5. Support environment-based configuration via `MINIMAX_API_KEY` and `MINIMAX_GROUP_ID`

This provides MiniMax model access without modifying the established provider abstraction pattern.

## Architecture Notes

### Provider Adapter Pattern

MinimaxProvider follows the established provider adapter pattern from MODL-0010/MODL-0011:

1. **Template Method Pattern**:
   - BaseProvider defines the lifecycle template
   - Subclasses implement provider-specific methods
   - Eliminates ~40 lines of duplicate code per provider

2. **Dependency Injection**:
   - Provider factory returns ILLMProvider interface
   - LangGraph nodes depend on interface, not concrete implementations
   - Enables runtime provider switching

3. **Caching Strategy**:
   - Static instance cache (by configuration hash)
   - Static configuration cache (loaded once per process)
   - Availability cache (TTL-based, default 30s)

### Class Hierarchy

```
ILLMProvider (interface)
    ↑
BaseProvider (abstract class)
    ↑
MinimaxProvider (concrete class)
```

### Factory Routing

```typescript
getProviderForModel(fullModelName)
  → parse prefix
  → route to provider instance
  → return ILLMProvider
```

**Routing Table**:
- `openrouter/*` → OpenRouterProvider
- `ollama/*` → OllamaProvider
- `anthropic/*` → AnthropicProvider
- `minimax/*` → MinimaxProvider (NEW)

### Configuration Flow

```
Environment Variables
  → Zod Schema Validation
  → Configuration Cache
  → Provider Instance
  → ChatMinimax Model
```

### Cache Key Generation

```typescript
config → JSON.stringify → SHA-256 hash → cache key
```

**Security Note**: API keys excluded from hash to prevent leaking sensitive data in logs.

### Availability Checking

```
checkEndpointAvailability(minimaxEndpoint, timeout)
  → HTTP request to health endpoint
  → Cache result for TTL
  → Return boolean
```
