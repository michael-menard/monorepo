# APIP-1020: ChangeSpec Schema Design and Validation Spike

## Context

The autonomous pipeline's implementation loop (APIP-1030), review graph (APIP-1050), QA graph (APIP-1060), change telemetry (APIP-3010), model affinity profiles (APIP-3020), and merge graph (APIP-1070) all consume a shared atomic change contract called a **ChangeSpec**. This schema is the most irreversible decision in Phase 1: once integration code is written against a schema shape, changing it requires cascading updates across 6+ systems.

No ChangeSpec schema exists today. The existing `FileChangeSchema` and `PlanStepSchema` in `packages/backend/orchestrator/src/artifacts/plan.ts` provide field-level inspiration but were designed for a human-facing plan document, not a machine-executable atomic change contract. Building Phase 1 integration code before the schema is validated on real stories is **RISK-002** (high severity) — the entire spike exists to eliminate this risk.

This spike is gated on the schema review acceptance blocker **ENG-001**: ChangeSpec schema review must complete before APIP-1030 elaboration begins.

**Baseline status**: No autonomous pipeline components exist yet. The orchestrator package at `packages/backend/orchestrator/` is actively deployed with elaboration and story-creation graphs — this is the closest existing code this spike builds on.

## Goal

Validate the ChangeSpec schema on 10+ real APIP backlog stories before any integration code is written. Produce:

1. A draft `ChangeSpecSchema` Zod schema with all fields required by 6 consuming systems
2. 10+ YAML decompositions of real backlog stories against the schema
3. Per-decomposition quality metrics (AC count, ambiguous fields, token cost, accuracy verdict)
4. Consumer-perspective validation from all 6 downstream systems
5. A published schema ADR with a verdict (APPROVED / APPROVED WITH CONSTRAINTS / REVISION REQUIRED)

This spike **gates APIP-1030**. No implementation graph code may be written until ADR-002 is published.

## Non-Goals

- Writing the DiffPlannerNode LangGraph node (integration story follows this spike)
- Connecting ChangeSpec output to APIP-1030 (Implementation Graph with Atomic Change Loop)
- Implementing model routing for the diff planner step
- Building any API, Lambda handler, or HTTP endpoint
- Modifying any production database schemas
- Modifying any existing orchestrator graph code
- Evaluating cheap-model quality for decomposition (that is APIP-0040 / RISK-001 scope)

## Architecture Notes

### ChangeSpec Field Inventory (Consuming Systems)

The draft schema must cover all fields required by each downstream system:

| System | Required Fields |
|--------|----------------|
| Implementation loop | `file_path`, `change_type`, `description`, `acceptance_criteria_ref`, `estimated_tokens` |
| Review graph | `change_type`, `risk_level`, `affected_patterns` |
| QA graph | `testability_hint`, `ac_refs`, `test_strategy` |
| Telemetry | `change_type`, `model_used`, `token_cost`, `duration_ms` |
| Model affinity | `change_type`, `file_extension`, `complexity_estimate` |
| Merge graph | `commit_message_hint`, `pr_body_hint`, `related_acs` |

### Discriminated Union Design

ChangeSpec should use `z.discriminatedUnion('change_type', [...])` over a flat schema with optional fields. Exhaustive type-checking in downstream nodes is worth the added schema complexity. Evaluate variants:

- `file_change` — code changes to existing or new source files
- `migration_change` — database schema changes
- `config_change` — configuration or environment changes
- `test_change` — test-only additions or modifications

### Schema Versioning

Follow the `PlanSchema` pattern: `schema: z.literal(1)`. Document in ADR-002:
- Adding optional fields: increment schema literal, update parser to handle both literals during transition
- Removing or renaming fields: major version bump, require coordinated deploy

### Decomposition Prompt Strategy

The spike requires calling a model (Claude Sonnet recommended) with a story's acceptance criteria to get back a list of ChangeSpecs. The prompt template must be documented in the spike output. Reference `packages/backend/orchestrator/src/nodes/elaboration/delta-detect.ts` for how nodes call models with structured output contracts.
