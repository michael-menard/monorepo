# APIP-3080: Parallel Story Concurrency (2-3 Worktrees)

## Context

The autonomous pipeline currently processes one story at a time under the APIP-0020 single-story constraint. The supervisor dispatches a BullMQ job, monitors it to completion, then dequeues the next. As Phase 1 graphs (elaboration, implementation, review, QA) each take meaningful wall-clock time, throughput is bounded by serial execution — one story per graph execution cycle.

With 31 stories in backlog across multiple phases, serial processing creates an unacceptably long queue depth. Provider API rate limits and file-system conflicts are the primary risks to parallel dispatch, not architectural impossibility. BullMQ's native `concurrency` parameter on the Worker constructor is already designed for exactly this use case.

**Phase 0 constraint**: APIP-3080 lifts the single-story constraint only after Phase 0 is complete. This story depends on APIP-0020 (supervisor loop) and APIP-0040 (model router with rate limiting) both being merged and stable.

## Goal

Extend the `PipelineSupervisor` to support 2–3 concurrent stories, each running in an isolated `git worktree`, using BullMQ's native `concurrency` parameter as the primary dispatch mechanism. The `PipelineModelRouter` singleton provides shared rate limiting across all concurrent workers. A `ConcurrencyController` tracks active worktree slots. A `WorktreeConflictDetector` prevents concurrent stories from writing to the same files.

## Non-Goals

- Concurrency higher than 3 worktrees (reserved for Phase 4+ after stability proven)
- Distributed supervisor processes (multiple Node.js processes sharing a queue)
- Automatic concurrency scaling based on system load (static config only)
- `WorktreeConflictDetector` deep file inspection via ChangeSpec diffs (story-level path prefix until APIP-1020 ChangeSpec is stable)
- Secrets management integration (APIP-5004 handles separately)
- Model affinity or smart routing across concurrent stories (APIP-3040 scope)
- UI observability for concurrent story status (APIP-2020 Monitor UI scope)

## Acceptance Criteria

- [ ] **AC-1**: `ConcurrencyConfig` Zod schema defines `maxWorktrees` (1–3 int), `conflictPolicy` ('reject' | 'queue'), and per-worktree circuit breaker thresholds (`failureThreshold`, `recoveryTimeoutMs`); defaults to `maxWorktrees: 1` preserving Phase 0 behavior
- [ ] **AC-2**: `BullMQ Worker` is initialized with `concurrency: config.maxWorktrees`; changing `maxWorktrees` from 1 to 2 causes the supervisor to process two jobs simultaneously without code changes beyond config
- [ ] **AC-3**: Each concurrent story runs in an isolated `git worktree` at `<repo_root>/.worktrees/<storyId>-<timestamp>/`; worktrees are created at dispatch time and removed on story completion/failure
- [ ] **AC-4**: `ConcurrencyController.tryAcquireSlot(storyId)` returns `true` and registers the slot when a free slot is available; returns `false` when all slots are occupied, causing the BullMQ job to be delayed (not rejected)
- [ ] **AC-5**: `WorktreeConflictDetector.checkConflict(incomingStory, activeStories)` returns the conflicting story IDs when two stories have overlapping in-flight file path prefixes; returns empty array when no overlap
- [ ] **AC-6**: When `conflictPolicy: 'reject'`, a job whose story overlaps with an active worktree's file set is nacked and requeued with a delay; a warning is logged via `@repo/logger` with the conflicting story IDs
- [ ] **AC-7**: `PipelineModelRouter` singleton is shared across all concurrent BullMQ worker callbacks; `TokenBucket.consume()` is safe under concurrent invocation (no double-spend or silent bypass) — validated by integration test EC-5
- [ ] **AC-8**: Per-worktree `NodeCircuitBreaker` is instantiated with explicit `failureThreshold` and `recoveryTimeoutMs` from `ConcurrencyConfig`; breaker open on one worktree does not affect other active worktrees
- [ ] **AC-9**: Worktree cleanup on story completion/failure is non-blocking: supervisor calls `git worktree remove`, handles result, logs outcome via `@repo/logger`, and continues dispatch regardless of cleanup success
- [ ] **AC-10**: Integration test validates 2 concurrent stories can be dispatched, processed to their first checkpoint, and completed without interference; uses real Redis (BullMQ) and real git worktrees
- [ ] **AC-11**: Integration test validates that a story cannot dispatch when `maxWorktrees` concurrent slots are occupied; job is delayed and eventually dispatched when a slot frees
- [ ] **AC-12**: `ConcurrencyConfig.maxWorktrees` defaults to `1`; setting to `1` produces behavior identical to APIP-0020's single-story mode (regression test required)

## Architecture Notes

### BullMQ Concurrency (Primary Mechanism)

The `BullMQ Worker` constructor accepts a `concurrency` option that causes it to process N jobs in parallel within the same Node.js process. This is the canonical primitive for multi-story parallel processing. Do NOT implement a bespoke thread pool or use `Promise.all()` as a substitute.

```typescript
// CORRECT
const worker = new Worker('pipeline', processJob, { concurrency: config.maxWorktrees })

// WRONG — do not do this
await Promise.all([processJob(job1), processJob(job2)])
```

### Singleton Router (Shared Rate Limiting)

The `PipelineModelRouter` from APIP-0040 is an in-memory singleton. All concurrent worker callbacks must use the same instance. Do NOT create per-worker router instances — this would bypass shared rate limiting.

```typescript
// All concurrent workers share this instance
const router = PipelineModelRouter.getInstance()
```

### Worktree Path Convention

Use `<repo_root>/.worktrees/<storyId>-<timestamp>/` where timestamp is `Date.now()`. The timestamp prevents stale worktree collisions on retry. Add `.worktrees/` to `.gitignore`.

```
.worktrees/
  APIP-1010-1740000000000/
  APIP-1040-1740000001234/
```

### Non-Blocking Cleanup (WINT-1150 Pattern)

Worktree cleanup must never block story completion. Pattern:

```typescript
// Non-blocking cleanup pattern
try {
  await execGit(['worktree', 'remove', '--force', worktreePath])
  logger.info('Worktree removed', { storyId, worktreePath })
} catch (error) {
  logger.warn('Worktree cleanup failed, continuing', { storyId, error })
  // Do NOT re-throw — slot must be released regardless
} finally {
  concurrencyController.releaseSlot(storyId)
}
```

### ConcurrencyController

```typescript
const ConcurrencyConfigSchema = z.object({
  maxWorktrees: z.number().int().min(1).max(3).default(1),
  conflictPolicy: z.enum(['reject', 'queue']).default('reject'),
  worktreeCircuitBreaker: z.object({
    failureThreshold: z.number().int().min(1).default(2),
    recoveryTimeoutMs: z.number().int().min(1000).default(60000),
  }),
})

type ConcurrencyConfig = z.infer<typeof ConcurrencyConfigSchema>
```

The `ConcurrencyController` maintains a `Map<storyId, worktreeId>` for active slots. `tryAcquireSlot()` returns `false` (causing BullMQ to delay the job) when the map size equals `maxWorktrees`.

### Per-Worktree Circuit Breakers

Each active slot gets its own `NodeCircuitBreaker` instance (from `runner/circuit-breaker.ts`) configured with thresholds from `ConcurrencyConfig.worktreeCircuitBreaker`. Breaker state is scoped to the slot — one slot's breaker opening must not affect other slots.

### WorktreeConflictDetector

Uses story-level path prefix comparison as a conservative proxy. Full ChangeSpec integration is deferred to after APIP-1020. A story's file set is derived from its `story.yaml` surface fields (e.g., `packages/backend/orchestrator/src/supervisor/`). Two stories conflict if their path prefixes overlap.
