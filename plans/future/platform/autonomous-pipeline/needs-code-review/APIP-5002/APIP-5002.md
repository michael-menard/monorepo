# APIP-5002: E2E Test Plan and Playwright Framework Setup for Autonomous Pipeline Critical Path

## Context

The autonomous pipeline's Phase 0 and Phase 1 stories collectively implement the full critical path: BullMQ work enqueue (APIP-0010) → supervisor dispatch (APIP-0020) → elaboration with structuring (APIP-1010) → ChangeSpec planning (APIP-1020) → implementation loop (APIP-1030) → review (APIP-1050) → QA verdict (APIP-1060) → squash-merge (APIP-1070). As of the 2026-02-13 baseline, none of these graphs exist. There is no end-to-end validation mechanism to confirm that a story enqueued into BullMQ actually progresses through all pipeline stages to a successful merge.

Without a structured E2E test harness, integration failures between pipeline stages will only be discovered during manual testing or when a real story gets stuck mid-pipeline. The orchestrator package has unit and integration tests per graph, but no test that exercises the entire queue → supervisor → graph chain end-to-end. Config mismatches between the supervisor dispatch contract and graph entry function signatures, YAML artifact path conventions, and BullMQ job state conventions will not surface until all graphs are manually integrated for the first time.

**Baseline**: 2026-02-13. No pipeline graphs yet exist. The only existing Playwright infrastructure targets the user-facing web application (`apps/web/playwright/`), not headless server-side pipeline processes.

## Goal

Establish a Playwright-based E2E test framework targeting the pipeline (not the web UI) to validate the full critical path from work enqueue through supervisor dispatch, graph execution, and squash-merge. The framework must be designed for phased activation — usable from Phase 0 (queue + supervisor only) and fully exercisable once all 8 dependency graphs are complete.

## Non-Goals

- Building any pipeline graph (elaboration, implementation, review, QA, merge) — this story provides the test harness only
- Implementing the BullMQ queue setup (APIP-0010) or the supervisor (APIP-0020)
- Running the full critical path E2E against real production stories — synthetic fixture only
- Performance or load testing of the pipeline — scope is correctness of the critical path only
- Playwright browser UI tests of any web dashboard (APIP-2020) — that is a separate E2E concern
- UAT-style testing of user-facing web UI features — existing `apps/web/playwright/` handles that
- Modifying `packages/backend/database-schema/` or `@repo/db` client (protected)
- Implementing secrets management (APIP-5004) — E2E suite reads credentials from environment variables only
- Monitoring or operator CLI visibility (APIP-5005, APIP-2020)
- Testing non-happy paths of individual graphs — graph unit/integration tests in `packages/backend/orchestrator/src/graphs/__tests__/` cover those

## Architecture Notes

### Phased Activation Strategy

The E2E framework is designed with two activation tiers:

**Phase 0 E2E** (available after APIP-0010 + APIP-0020):
- `phase0-queue-supervisor.spec.ts` — verifies BullMQ enqueue and supervisor dispatch
- Timeout: 2 minutes
- No graph-level assertions

**Phase 1 E2E** (incrementally activated per graph completion):
- `phase1-elaboration.spec.ts` — activated after APIP-1010 merges; skipIf guard checks elaboration module export
- `phase1-critical-path.spec.ts` — activated after all 8 dependency stories merge; skipIf guard checks all graph module exports
- Timeout: 10 minutes (elaboration), 30 minutes (critical path)

### Skip Condition Pattern

Each Phase 1 spec uses a programmatic module availability check at test start rather than a static skip flag. When the dependency graph merges to main, its export becomes available and the skip condition resolves to `false` automatically:

```typescript
// In phase1-elaboration.spec.ts
test.beforeAll(async () => {
  try {
    await import('@repo/orchestrator/graphs/elaboration')
  } catch {
    test.skip(true, 'SKIPPED: APIP-1010 (structurer node) not yet available')
  }
})
```

### BullMQ Client Lifecycle

The `PipelineTestFixtures` fixture opens two Redis connections in `beforeAll`:
1. `Queue` client (name: `pipeline-e2e`) — for enqueuing test jobs
2. `QueueEvents` client (same queue name) — for `job.waitUntilFinished()` completion signaling

Both must be closed in `afterAll` to prevent connection leaks.

### Port Isolation from QA Graph

APIP-1060 (QA Graph) invokes `pnpm exec playwright test` as a subprocess inside its `run-e2e-tests` node. To prevent conflicts:
- Pipeline E2E Playwright project name: `pipeline-e2e` (distinct from all web app project names)
- `workers: 1` in config prevents concurrent pipeline E2E runs
- Pipeline E2E must be serialized with (not concurrent to) QA graph test executions in CI

### Synthetic Test Story Design

The synthetic story fixture must produce a ChangeSpec whose generated code change passes `pnpm check-types` and `pnpm test` deterministically (not via LLM). Recommended approach: a pre-written code patch that adds a new exported constant and a corresponding Vitest test to a non-critical utility file in `packages/backend/orchestrator/`. The ChangeSpec `instructions` field contains the literal code diff so the implementation graph applies it directly without LLM generation.

### Observable Outputs Only

E2E tests MUST assert only on observable pipeline outputs:
- YAML artifacts on the filesystem (EVIDENCE.yaml, REVIEW.yaml, QA-VERIFY.yaml, MERGE.yaml)
- BullMQ job completion status
- Git log entries (`git log --oneline main --grep={storyId}`)

E2E tests MUST NOT reach into LangGraph checkpoint state directly.
