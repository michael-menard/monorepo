# APIP-4030: Dependency Auditor — Post-Merge Package Analysis, Overlap Detection, and Human-Blocked Queue Routing

## Context

As the autonomous pipeline continuously merges stories that add, remove, or update npm packages, there is no automated mechanism to audit the dependency health of the monorepo. New packages can introduce security vulnerabilities, duplicate functionality already provided by existing packages (e.g., adding `dayjs` when `date-fns` is already installed), bloat the bundle unnecessarily, or become unmaintained over time. Without a dependency auditor, the pipeline's output could silently degrade the supply-chain security and package hygiene of the codebase.

This is a greenfield story — no prior Dependency Auditor work exists in the codebase. All audit logic runs on the dedicated LangGraph server process as a post-merge async cron task; no Lambda or API Gateway involvement.

**Dependencies**:
- **APIP-2010** (Blocked Queue and Notification System) — hard runtime dependency for routing high-risk findings to human review. The auditor's detection logic can be built independently; only the final routing wiring is gated on APIP-2010.
- **APIP-3090** (Cron Job Infrastructure) — required for post-merge trigger scheduling. The `DepAuditCronTask` interface is defined here; plugging into APIP-3090 is a one-line change when it ships.

## Goal

Prevent dependency bloat, vulnerability accumulation, and redundant package introduction without blocking the autonomous pipeline for routine changes. Provide operators with actionable human-review items for high-risk dependency decisions.

## Non-Goals

- Building a UI dashboard for audit findings (deferred to APIP-4070 Weekly Pipeline Health Report or APIP-2020 Monitor UI).
- Auto-remediating vulnerabilities — blocked queue items are for human review only.
- Auditing all packages on every run — only packages changed in the triggering merge are audited.
- Enforcing a license compliance policy — no license scanning in this story.
- Performing SAST or secret detection — handled by APIP-1050 Review Graph security worker.
- Supporting distributed/multi-process environments — single-server assumption per pipeline ADR.
- Modifying any existing wint tables or production DB schemas.
- Implementing the human-review resolution workflow — APIP-2010 owns that.

## Architecture Notes

**Four integrated components**:

1. **`wint.dep_audit_runs` table** — Records each post-merge audit run: which story/commit triggered it, which packages changed, the overall risk level, and a pointer to findings.

2. **`wint.dep_audit_findings` table** — One row per finding per run: package name, finding type (vulnerability/overlap/bundle_bloat/unmaintained), severity (critical/high/medium/low/info), and a structured JSONB payload with details.

3. **`runDepAudit(config, db)` function** — Orchestrates the five analysis steps against changed packages. Returns `DepAuditResultSchema` summary. Includes `pg_try_advisory_lock` concurrency guard (defense-in-depth against duplicate cron runs per APIP-3020 lesson). Follows fire-and-forget semantics — non-blocking on failure.

4. **High-risk routing** — For findings at `critical` or `high` severity (configurable), calls the APIP-2010 blocked queue API. Items deduplicated by `storyId:packageName:findingType`. During development (pre-APIP-2010), use a `DepBlockedQueueClient` stub that logs a warning.

**Equivalence Knowledge Base**:
The package overlap detection heuristic uses a versioned YAML file committed to the repo: `packages/backend/orchestrator/src/config/dep-equivalences.yaml`. This file maps package names to their functional equivalents (e.g., `date-fns ↔ dayjs ↔ moment`, `lodash ↔ ramda`). It is maintainable without code changes, loadable by the pipeline in a future story.

**Injectable Tool Runner Pattern**:
All five analysis functions must accept injectable `toolRunner` / `fetchFn` parameters for testability. No function calls external CLIs or network services directly in unit tests.

**pnpm audit resilience**:
`PnpmAuditOutputSchema` uses `.passthrough()` on unknown fields to survive pnpm major version upgrades. Only the fields actually consumed are strictly validated.
