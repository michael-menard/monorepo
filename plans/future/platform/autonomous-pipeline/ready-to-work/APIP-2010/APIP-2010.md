# APIP-2010: Blocked Queue and Notification System

## Context

Once APIP-0020's supervisor loop is operational, failed pipeline stories silently exhaust BullMQ retries and sit in BullMQ's native `failed` state with no human-visible escalation. There is currently no mechanism to distinguish a job that failed once (transient, will retry) from a job that has exhausted all retry attempts (permanently blocked, needs human intervention).

This story adds explicit `blocked` status semantics to the supervisor process (established by APIP-0020), enabling the pipeline to surface exhausted failures to human operators before they become invisible in BullMQ's failed queue. The blocked state is persisted to a Redis sorted set for queryability by APIP-2020 (Monitor UI) without requiring Bull Board access.

**Dependencies**: APIP-0020 (Supervisor Loop) must be merged before this story begins implementation. APIP-0010 (BullMQ Work Queue Setup) must be merged to confirm `PipelineJobPayload` schema field names and `maxAttempts` configuration.

**Protected areas**: Do NOT touch `packages/backend/database-schema/`, `@repo/db` client, existing orchestrator test suites in `packages/backend/orchestrator/`, or any LangGraph graph signatures.

## Goal

Add two capabilities to the supervisor process:

1. **Blocked-transition logic**: When a BullMQ job's `attemptsMade >= opts.attempts`, call `transitionToBlocked(job)` — an idempotent function that persists a `BlockedJobRecord` to a Redis sorted set and sets a deduplication key to prevent double-notification on supervisor restart.

2. **Configurable webhook notification**: After transitioning to blocked, dispatch a fire-and-forget webhook POST to the configured Slack webhook URL. Notification failure is logged and swallowed — it never crashes the supervisor.

## Non-Goals

- Building a blocked-queue management UI — that is APIP-2020
- Implementing email or push notification channels — schema variants are defined but marked `not_implemented`; only Slack webhook is fully implemented
- Auto-retry of blocked jobs — a human must manually re-enqueue; no automatic re-queue logic
- Modifying `packages/backend/database-schema/` or `@repo/db` client
- Modifying existing LangGraph graph signatures or test suites in `packages/backend/orchestrator/`
- Crash recovery reconciliation for mid-stream LangGraph checkpoints — deferred per APIP-0020 risk notes
- Operator CLI for unblocking/re-queuing stories — that is APIP-5005
- Monitor dashboard — that is APIP-2020
- Graceful shutdown enhancements — that is APIP-2030

## Architecture Notes

**Storage backend decision** (resolved from seed conflict):
- Redis sorted set `pipeline:blocked:jobs` (score = `blockedAt` timestamp in ms) is the chosen storage backend for blocked job records
- Consistent with APIP-0020 constraint: "supervisor has no direct DB writes"
- Readable by APIP-2020 Monitor UI without requiring `@repo/db` or Aurora connection
- Durable under Redis AOF persistence (APIP-0010 responsibility to configure AOF)
- If APIP-2020 requires SQL joins, a `pipeline_blocked_jobs` Aurora table can be added as a separate migration story — do not add Aurora writes to supervisor in this story

**Idempotency strategy**:
- Redis `SET NX notified:{jobId}:{attemptNumber} 1 EX 604800` in the same `MULTI`/`EXEC` pipeline as `ZADD pipeline:blocked:jobs {score} {payload}`
- `MULTI`/`EXEC` ensures both writes succeed or neither does — no partial state
- Notification dispatch only fires if the `SET NX` succeeds (key was new)
- If `SET NX` returns nil (key existed), emit `notification_deduplicated` log event and skip dispatch

**Notification dispatch architecture** (APIP ADR-001 Decision 4):
- Notification HTTP request is made directly from the supervisor process (long-lived Node.js on dedicated server)
- No separate microservice, no Lambda invocation
- `dispatchNotification()` is an injected dependency of `transitionToBlocked()` for testability (pure function of job metadata + injected side-effect functions)

**Schema contract** (APIP ADR-001 Decision 2):
- All types (`BlockedJobRecordSchema`, `NotificationConfigSchema`, `BlockedJobEventSchema`, `WebhookPayloadSchema`) are Zod schemas with `z.infer<>` — no TypeScript interfaces
- `ErrorCategory` imported directly from `packages/backend/orchestrator/src/runner/error-classification.ts` — not duplicated

## Implementation Decisions

### GAP-1: ErrorCategory Zod Compatibility — Decision: Option A (local z.enum mirror)

**Decision date**: 2026-02-25
**Decided by**: elab-autonomous-decider

`error-classification.ts` exports `ErrorCategory` as a raw TypeScript `type` union (not a Zod schema). `BlockedJobRecordSchema` requires `lastErrorCategory` as a Zod field, which cannot directly reference a TS `type`.

**Chosen approach**: Mirror the 8 category values in a local `ErrorCategorySchema = z.enum([...])` inside `apps/api/pipeline/src/supervisor/__types__/index.ts`.

```typescript
// In supervisor/__types__/index.ts
// Mirrors ErrorCategory from packages/backend/orchestrator/src/runner/error-classification.ts
// SOURCE OF TRUTH: error-classification.ts — update this enum if that file adds new categories
const ErrorCategorySchema = z.enum([
  'validation',
  'programming',
  'timeout',
  'cancellation',
  'circuit_open',
  'network',
  'rate_limit',
  'unknown',
])
```

**Drift risk mitigation**: Add the comment referencing the canonical source file as shown above. If `error-classification.ts` adds a new category, the linter will not catch it automatically — the dev must update this enum manually. This is an acceptable trade-off for a stable enum in a controlled codebase.

**Rationale**: Option A is self-contained and requires no coordinated change to `error-classification.ts`. Option B (`z.string().refine()`) loses compile-time enum safety. Option C (adding `ErrorCategorySchema` to the canonical file) is the cleanest long-term solution but requires a coordinated PR outside this story's scope. Option A is sufficient for this story and can be upgraded to Option C in a follow-up.

**No new ACs required** — AC-2 absorbs this decision ("from `ErrorCategory` in `error-classification.ts`" means: mirror those values via local `z.enum`).

### GAP-2: Redis Write Failure Path — Decision: Option A (propagate to BullMQ handler)

**Decision date**: 2026-02-25
**Decided by**: elab-autonomous-decider

The story defines swallow-and-log behavior for webhook failures (AC-7) but was silent on Redis `MULTI`/`EXEC` failures in `transitionToBlocked()`.

**Chosen approach**: Propagate the Redis error upward to the BullMQ Worker `'failed'` event handler. Do NOT swallow Redis errors.

**Implementation contract**:
- `transitionToBlocked(job)` does NOT catch Redis errors — it lets them propagate
- The BullMQ Worker `'failed'` event handler is responsible for catching and logging the error: `logger.error({ event: 'blocked_transition_failed', storyId, stage, error })`
- The job remains in BullMQ's native `failed` state — no `BlockedJobRecord` is written
- The supervisor continues operating normally (the `'failed'` handler does not rethrow)
- On supervisor restart, the same exhausted job may trigger `transitionToBlocked()` again — if Redis has recovered, the `SET NX` deduplication key ensures no duplicate notification

**Rationale**: This is the safest outcome. The job is not silently lost — it remains visible in BullMQ's `failed` queue. The blocked state record is incomplete, but the supervisor continues serving other jobs. Swallowing Redis errors (Option B without rethrow) would make the blocked state silently inconsistent. A retry-once strategy (Option C) was not chosen because `createRedisClient()` already has a built-in `retryStrategy` — double-retrying at the application layer risks masking the underlying connection issue.

**No new ACs required** — AC-1 ("it is idempotent") and AC-3 ("persisted to Redis") absorb this decision. The BullMQ handler's error logging behavior is covered by AC-9 (structured log events).
