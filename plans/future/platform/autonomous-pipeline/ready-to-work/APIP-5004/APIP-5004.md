# APIP-5004: Secrets Engine and API Key Management for Autonomous Pipeline

## Context

The autonomous pipeline's model router (APIP-0040) dispatches work across four LLM provider APIs: Anthropic, OpenRouter, MiniMax, and Ollama. All four provider adapters in `packages/backend/orchestrator/src/providers/` currently read their API keys directly from `process.env` at `loadConfig()` time. There is no abstraction layer, no rotation capability, and no audit trail for key access. A `.env` file containing live keys is the only mechanism today.

An inspection of the codebase confirmed that a live `OPENAI_API_KEY` value is committed in the root `.env` file — exactly the anti-pattern that APIP-5004 must prevent for the autonomous pipeline's model API keys.

For a locally-run development tool, `process.env` key loading is acceptable. For an autonomous pipeline running continuously on a dedicated server (per APIP-5006), it is not: the pipeline needs API keys to be stored securely, retrievable without embedding them in `.env` files on the server filesystem, rotatable without restarting the pipeline process, and auditable.

**Baseline Date**: 2026-02-13. No secrets management infrastructure exists anywhere in the codebase at that date.

## Goal

Introduce a `SecretsClient` module that wraps AWS Secrets Manager for server deployments and falls back to environment variables for local development. Refactor the three provider `loadConfig()` methods (Anthropic, OpenRouter, MiniMax) to use `SecretsClient.get()` instead of reading `process.env` directly. Provide a companion CloudFormation stack for the AWS Secrets Manager resources.

## Non-Goals

- Automated secret rotation (Lambda rotation function) — deferred; MVP rotation is manual
- HashiCorp Vault integration — AWS Secrets Manager is the target for this story
- Secrets management for non-model-API credentials (e.g., database connection strings) — APIP-5004 focuses on model API keys only
- A UI or operator CLI for secret management — that is APIP-5005 scope
- Automatic cache invalidation via AWS Secrets Manager rotation events (SNS/EventBridge) — deferred to Phase 2
- Modifying any protected DB schema packages (`packages/backend/database-schema/`) or the `@repo/db` client
- Modifying the `BaseProvider` abstract interface (`ILLMProvider` or `BaseProvider` class) — no change to the abstract contract

## Architecture Notes

### DECISION: Prefetch Pattern (Option B) — Committed

**Decided**: 2026-02-25 (autonomous elaboration decision)

The async/sync `loadConfig()` question is resolved. **Option B (prefetch pattern) is the committed approach.**

Rationale:
- The Non-Goals explicitly state "Modifying the `BaseProvider` abstract interface — no change to the abstract contract." This rules out Option A (async `loadConfig()`), which would require changing `ILLMProvider.loadConfig(): unknown` and `BaseProvider.getModel()` template method.
- The story already identified Option B as the lower-risk path.

Implementation contract:
- `SecretsClient.prefetch(secretNames: string[]): Promise<void>` — called once at pipeline startup (APIP-5006 scope) before any provider is instantiated; pre-populates the in-memory cache.
- `SecretsClient.getSync(secretName: string): string` — synchronous cache read used by `loadConfig()` methods; throws `Error` if the key was not pre-fetched.
- `loadConfig()` on all three providers remains synchronous — it calls `secretsClient.getSync()`, not `secretsClient.get()`.
- `base.ts` is NOT modified; the `loadConfig(): unknown` signature is unchanged.

### SecretsClient Design

The `SecretsClient` is a module-level singleton. The two-mode design (env fallback vs. AWS) is governed by `SECRETS_ENGINE` env var, validated at instantiation time via Zod:

```typescript
// packages/backend/orchestrator/src/secrets/index.ts

import { z } from 'zod'
import { SecretsManagerClient, GetSecretValueCommand } from '@aws-sdk/client-secrets-manager'
import { logger } from '@repo/logger'

export const SecretsClientConfigSchema = z.object({
  engine: z.enum(['env', 'aws-secrets-manager']).default('env'),
  awsRegion: z.string().optional(),
  cacheTtlMs: z.coerce.number().positive().default(300000), // 5 minutes
})
  .refine(
    config => config.engine !== 'aws-secrets-manager' || !!config.awsRegion,
    { message: 'AWS_REGION is required when SECRETS_ENGINE=aws-secrets-manager' }
  )

export type SecretsClientConfig = z.infer<typeof SecretsClientConfigSchema>

export class SecretsClient {
  private readonly config: SecretsClientConfig
  private readonly cache = new Map<string, { value: string; fetchedAt: number }>()
  private readonly awsClient?: SecretsManagerClient

  constructor(config?: Partial<SecretsClientConfig>) {
    this.config = SecretsClientConfigSchema.parse({
      engine: process.env.SECRETS_ENGINE ?? 'env',
      awsRegion: process.env.AWS_REGION,
      cacheTtlMs: process.env.SECRETS_CACHE_TTL_MS,
      ...config,
    })

    if (this.config.engine === 'aws-secrets-manager') {
      this.awsClient = new SecretsManagerClient({ region: this.config.awsRegion })
    }
  }

  async get(secretName: string): Promise<string> {
    // Check cache
    const cached = this.cache.get(secretName)
    if (cached && Date.now() - cached.fetchedAt < this.config.cacheTtlMs) {
      return cached.value
    }

    // Fetch from source
    const startMs = Date.now()
    let value: string

    if (this.config.engine === 'env') {
      value = process.env[secretName] ?? ''
      if (!value) {
        throw new Error(`${secretName} environment variable is required but not set`)
      }
    } else {
      const response = await this.awsClient!.send(
        new GetSecretValueCommand({ SecretId: secretName })
      )
      value = response.SecretString ?? ''
      if (!value) {
        throw new Error(`Secret '${secretName}' not found or has no value in AWS Secrets Manager`)
      }
    }

    const latencyMs = Date.now() - startMs
    logger.info({ secretName, engine: this.config.engine, latencyMs }, 'secrets: cache-miss fetch')

    this.cache.set(secretName, { value, fetchedAt: Date.now() })
    return value
  }

  flush(): void {
    this.cache.clear()
  }
}

// Module-level singleton
export const secretsClient = new SecretsClient()
```

### Provider Injection Pattern

Each provider's `loadConfig()` method calls `secretsClient.getSync()` instead of reading `process.env` directly. The prefetch pattern (Option B) is committed — see the DECISION record above.

Pipeline startup code (APIP-5006 scope) calls `await secretsClient.prefetch(['ANTHROPIC_API_KEY', 'OPENROUTER_API_KEY', 'MINIMAX_API_KEY'])` before the model router initializes. Once prefetch completes, all subsequent `getSync()` calls in `loadConfig()` read from the in-memory cache synchronously. No change is needed to `ILLMProvider`, `BaseProvider`, or `getModel()`.

**Cross-story note for APIP-5006**: APIP-5006 (server infrastructure) MUST call `secretsClient.prefetch([...])` at pipeline startup before the model router initializes. This is a hard dependency — if prefetch is skipped, the first `getSync()` call in any provider will throw. This requirement must be captured in APIP-5006's implementation notes and subtasks.
