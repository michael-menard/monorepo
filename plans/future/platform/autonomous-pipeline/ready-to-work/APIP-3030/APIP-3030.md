# APIP-3030: Learning-Aware Diff Planner

## Context

The autonomous pipeline's Diff Planner node (introduced in APIP-1030) decomposes a story's change outline into a list of ChangeSpecs — atomic, typed change contracts consumed by the implementation loop, review graph, QA graph, and other downstream systems. As of Phase 1, this decomposition is model-agnostic: the planner does not consider which model will execute the changes or whether that model is known to struggle with particular change patterns.

Phase 3 introduces a learning system: the change telemetry table (APIP-3010) records every model-executed change attempt with outcome, and the Pattern Miner cron (APIP-3020) aggregates this data into per-(model, change_type, file_type) affinity profiles stored in `wint.model_affinity`. Each profile carries a `success_rate`, `avg_attempts`, `avg_tokens`, and a `confidence` level.

**Problem**: Without awareness of model affinity profiles, the Diff Planner may decompose a story in ways that assign weak change types to cheap models — leading to escalation, retry loops, and elevated token costs. RISK-001 (cheap models producing incorrect code that passes micro-verify but fails QA) and RISK-002 (Diff Planner producing poor ChangeSpec decompositions) both implicate this node.

**Dependency gates**:
- APIP-1020 (ChangeSpec Schema Design and Validation Spike) must complete and publish ADR-002 before the Diff Planner's output schema can be finalized. Implementation uses a PlaceholderChangeSpecSchema until then.
- APIP-3020 (Model Affinity Profiles Table and Pattern Miner Cron) must complete and populate `wint.model_affinity` before the node can perform non-trivial routing decisions. Cold-start fallback handles the initial empty state.

## Goal

Update the Diff Planner node to query model affinity profiles at planning time and inject a concise affinity summary into the decomposition prompt — biasing decomposition toward change types the assigned model handles reliably and pre-assigning escalation for known weakness patterns. Reduce escalation rate and improve first-attempt success for cheap models.

## Architecture Notes

### Affinity Profile Query

The query is keyed by `(model_id, change_type, file_type)`. Since the Diff Planner runs before knowing the exact file types in the change, it should query for all `(model_id, change_type)` pairs first, then filter by file_type during prompt assembly. This is preferable to N queries per ChangeSpec.

### Graceful Degradation Chain

```
affinity_query()
  → success, confidence >= MIN → inject affinity summary into prompt
  → success, confidence < MIN  → skip injection, profile_used=false
  → no rows (cold-start)       → model-agnostic fallback, profile_used=false
  → DB error                   → log warning, model-agnostic fallback, profile_used=false
```

All paths must return a valid `DiffPlannerOutput` that passes Zod validation.

### Prompt Injection Structure

The affinity summary injected into the decomposition prompt should be brief and structured:

```
[Model Affinity Note — informational only]
Weak patterns for this model (minimize if possible):
- migration_change / sql: success_rate=0.40 (2/5 attempts)
- config_change / json: success_rate=0.45 (5/11 attempts)

Strong patterns for this model (prefer when decomposition allows):
- file_change / ts: success_rate=0.93 (28/30 attempts)
```

Cap weak patterns at `MAX_WEAK_PATTERNS_INJECTED` (default 5). Cap strong patterns at 3. Sort by success_rate ascending (worst first) for weak; descending for strong.

### Profile Freshness

Profiles are updated by the Pattern Miner cron (APIP-3020, runs daily). The Diff Planner queries the current state at planning time — there is an inherent lag of up to 24 hours. This is acceptable; the affinity summary is advisory, not prescriptive. Document in node JSDoc.

### Token Cost Impact

Injecting affinity summaries increases token cost per planning invocation. With the default cap of 5 weak + 3 strong patterns, the injection adds approximately 80-120 tokens per invocation — acceptable at current planning volume.
