# APIP-0020: Supervisor Loop (Plain TypeScript)

## Context

The autonomous pipeline requires a supervisory process that sits between the BullMQ work queue (APIP-0010) and the LangGraph worker graphs. Without this supervisor, jobs queued by the pipeline have no mechanism for dispatch, monitoring, or status propagation.

The codebase already contains the two worker graphs this supervisor dispatches to:
- `packages/backend/orchestrator/src/graphs/elaboration.ts` — exposes `runElaboration()`
- `packages/backend/orchestrator/src/graphs/story-creation.ts` — exposes `createStoryCreationGraph()`

It also has a circuit breaker pattern (`runner/circuit-breaker.ts`), error classification utilities (`runner/error-classification.ts`), and a Redis client factory (`apps/api/lego-api/core/cache/redis-client.ts`). This story wires those together behind a BullMQ Worker into a coherent supervisor process.

Per APIP ADR-001 Decision 2, this supervisor is a plain TypeScript async process — NOT a LangGraph graph. LangGraph provides genuine checkpointing value for the worker graphs themselves, but the supervisor is a polling/dispatch loop where LangGraph complexity adds no Phase 0 benefit.

Per APIP ADR-001 Decision 4, the supervisor runs as a long-lived Node.js process on the dedicated local server provisioned by APIP-5006. No AWS Lambda.

Phase 0 enforces a single-story-at-a-time constraint via BullMQ `concurrency: 1`. Concurrency expansion is deferred to APIP-3080.

**Active blocking dependency**: APIP-0010 (BullMQ queue) must be completed before full BullMQ Worker wiring. Supervisor dispatch logic and unit tests can be implemented immediately; BullMQ wiring follows APIP-0010 merge.

## Goal

Implement a plain TypeScript `PipelineSupervisor` that:
1. Consumes jobs from the BullMQ work queue (established by APIP-0010)
2. Validates job payloads with Zod schema
3. Derives the LangGraph thread ID using the convention `{storyId}:{stage}:{attemptNumber}`
4. Dispatches to the appropriate LangGraph worker graph (elaboration or story-creation)
5. Enforces wall clock timeout via `Promise.race()` (default: 10 minutes)
6. Applies error classification (PERMANENT → immediate fail; TRANSIENT → BullMQ retry)
7. Protects each worker graph with a per-graph `NodeCircuitBreaker`
8. Logs structured lifecycle events with `@repo/logger`
9. Exposes `start()` and `stop()` for graceful lifecycle management

## Architecture Notes

### Thread ID Convention (ADR — Irreversible)

Thread IDs must follow the format `{storyId}:{stage}:{attemptNumber}` and must be documented as an inline ADR comment in the supervisor code before any coding begins. This convention is used for LangGraph checkpoint reconciliation in crash recovery (APIP-2010). Changing it post-Phase 0 requires checkpoint data migration.

```typescript
// ADR: Thread ID Convention (APIP-0020)
// Format: {storyId}:{stage}:{attemptNumber}
// Example: "APIP-001:elaboration:1"
// IRREVERSIBLE: Changing this convention post-Phase 0 requires LangGraph checkpoint data migration.
const threadId = `${job.data.storyId}:${job.data.stage}:${job.data.attemptNumber}`
```

### Wall Clock Timeout

```typescript
const result = await Promise.race([
  runElaboration(storyId, null, config),
  new Promise<never>((_, reject) =>
    setTimeout(() => reject(new WallClockTimeoutError(stageTimeoutMs)), stageTimeoutMs)
  ),
])
```

On timeout: log the `threadId` so the LangGraph checkpoint can be inspected or abandoned. Do not silently swallow the timeout — it is a signal that the graph is stuck and the checkpoint may be in a partial state.

### Circuit Breaker Per Graph

One `NodeCircuitBreaker` instance per worker graph:
- `elaborationCircuitBreaker` — protects `runElaboration()`
- `storyCreationCircuitBreaker` — protects story-creation graph

When circuit OPEN: re-queue the BullMQ job as delayed (not failed), log `circuit_open` event. This allows the circuit to recover before the job is retried.

### Error Classification Flow

```
graph error thrown
  ↓
isRetryableNodeError(error)
  ↓ true → propagate (BullMQ retry per queue config)
  ↓ false → throw UnretryableError (BullMQ fails job, no retry)
```

Do not treat all errors as retryable. PERMANENT errors (invalid config, schema violations, data corruption) must fail fast.

### Supervisor State Machine

```
Job received → Zod validate → thread ID derived → circuit check
  → [circuit OPEN] → re-queue as delayed → log circuit_open
  → [circuit CLOSED] → dispatch to graph
    → Promise.race([graph, wallClockTimeout])
      → [graph complete] → log completed → BullMQ job complete
      → [timeout] → log timeout → BullMQ job fail (wall_clock_timeout)
      → [graph error] → classify error
        → [PERMANENT] → log failed → BullMQ immediate fail
        → [TRANSIENT] → propagate → BullMQ retry
```

### ST-4: Implement wall clock timeout via Promise.race() with WallClockTimeoutError

**Goal**: Wrap each graph dispatch in `Promise.race([graphInvocation, wallClockTimeout])`. Define `WallClockTimeoutError`. On timeout: log event with `threadId`, mark BullMQ job failed with reason `wall_clock_timeout`. Default timeout: 10 minutes, configurable.

**Implementation notes (two-catch pattern — AC-15)**:
- `WallClockTimeoutError` must be caught in a dedicated branch BEFORE the general error classification catch. Use the following pattern:
  ```typescript
  try {
    await Promise.race([graphInvocation, wallClockTimeout])
  } catch (e) {
    if (e instanceof WallClockTimeoutError) {
      // fail job immediately — do NOT pass to isRetryableNodeError()
    } else {
      // pass to error classifier (ST-5)
    }
  }
  ```
- This ensures timeout errors bypass `isRetryableNodeError()` (which would classify them as `unknown` / retryable, contradicting the intended behavior).

### ST-5: Integrate error classification (PERMANENT vs TRANSIENT) into job processor

**Goal**: Apply `isRetryableNodeError()` from `packages/backend/orchestrator/src/runner/error-classification.ts` to all graph errors. PERMANENT errors cause immediate BullMQ job failure. TRANSIENT errors propagate for BullMQ retry.

**Implementation notes (two-catch pattern — AC-15)**:
- Error classification applies only in the `else` branch of the two-catch pattern established in ST-4. `WallClockTimeoutError` must already be handled before this branch is reached — do not pass timeout errors to `isRetryableNodeError()`.
- `isRetryableNodeError(e)` returning `false` = PERMANENT = fail job immediately with no retry.
- `isRetryableNodeError(e)` returning `true` = TRANSIENT = re-throw so BullMQ handles retry per queue config.
