# APIP-5006: LangGraph Server Infrastructure Baseline

## Context

The autonomous pipeline (APIP) requires all graph execution components to run on a dedicated local server rather than AWS Lambda (APIP ADR-004). This is a critical path story for Phase 0: APIP-0030 (LangGraph Platform Docker Deployment) is explicitly gated on this story's completion per FOLLOW-UPS PLAT-001.

The monorepo has established infrastructure patterns to follow:
- `infra/compose.lego-app.yaml` — the canonical Docker Compose pattern (named volumes, healthchecks, restart policies, `extra_hosts`, one-shot init containers)
- `infra/elasticache/template.yaml` — the CloudFormation standalone IaC pattern (ADR-002)
- `infra/prometheus/prometheus.yml` — the Prometheus scrape target pattern

Existing services claim these ports (must not conflict): 5432 (postgres), 5433 (kb-postgres), 6379 (redis), 9090 (prometheus), 3003 (grafana), 4317/4318 (otel), 9000/9001 (minio). LangGraph Platform uses 8123 (API) and 8124 (Studio UI).

The architecture review established that the LangGraph checkpoint database must use a **dedicated connection pool (max 3)** separate from `@repo/db`, targeting Aurora PostgreSQL. This prevents Aurora `max_connections` exhaustion.

## Goal

Provision and document the dedicated server baseline so APIP-0030 can immediately deploy the LangGraph Platform Docker configuration without additional infrastructure work. Produce:

4. A passing smoke test confirming the server is ready

## Architecture Notes

### Dedicated Server Rationale (APIP ADR-004)

All LangGraph graph execution runs on a dedicated local server (not AWS Lambda) because:
- LangGraph graphs require persistent processes and long-running connections unsuited to Lambda's stateless model
- 2-3 concurrent worktrees (APIP-3080) require sustained compute (min 8 GB RAM)
- Checkpoint database writes benefit from proximity to the Aurora instance

### Connection Pool Isolation

The LangGraph checkpoint connection pool MUST be separate from `@repo/db`:
- `@repo/db` uses `max: 1` per Lambda invocation (serverless model)
- Checkpoint pool: `max: 3`, dedicated user credentials, persistent connection
- Architectural risk if mixed: Aurora `max_connections` exhaustion under concurrent graph execution
- Implementation: separate `pg.Pool` instance in the LangGraph server process, not imported from `@repo/db`

### Port Assignments

Reserved (do not use): 5432, 5433, 6379, 9090, 3003, 4317, 4318, 9000, 9001

LangGraph Platform (verify free on server before deploying APIP-0030):
- 8123: LangGraph API
- 8124: LangGraph Studio UI

Audit command: `ss -tlnp` on target server.

### Docker Compose Conventions

Follow `infra/compose.lego-app.yaml` exactly:
- Named volumes (not bind mounts) for persistent data
- `restart: unless-stopped` on all persistent services
- `restart: "no"` on one-shot init containers
- `healthcheck` on every service with appropriate `CMD` or `CMD-SHELL`
- `extra_hosts: - 'host.docker.internal:host-gateway'` for services that need to reach the host
