# APIP-1030: Implementation Graph with Atomic Change Loop

## Context

The autonomous pipeline (APIP) requires a LangGraph worker graph that takes a validated story with a ChangeSpec plan and executes code changes atomically — one commit per ChangeSpec, with micro-verification before each commit. This is Phase 1's core implementation engine.

**Baseline date**: 2026-02-13. As of baseline:
- No implementation graph, git worktree management, code-generation nodes, or micro-verify toolchain exist.
- APIP-1020 (ChangeSpec Schema Design Spike) and APIP-0040 (PipelineModelRouter) are both in backlog — neither is complete.
- The LangGraph worker graph pattern is established in `packages/backend/orchestrator/src/graphs/` (elaboration.ts, story-creation.ts).

**Critical path position**: This is story 5 of 12 on the critical path. It is blocked by APIP-1020 (which is blocked by APIP-1010) and APIP-0040.

**Sizing warning** (`sizing_warning: true`): This story spans 4 distinct technical domains — LangGraph graph infrastructure, git worktree management, model dispatch integration, and micro-verify toolchain. A split into APIP-1030a (skeleton) / APIP-1030b (change loop) is strongly recommended at elaboration time.

---

## Goal

Build a LangGraph implementation graph (`packages/backend/orchestrator/src/graphs/implementation.ts`) that:
1. Creates an isolated git worktree for each story
2. Loads the story file and ChangeSpec plan into graph state
3. Iterates over ChangeSpecs: dispatch → code-gen → micro-verify → atomic git commit
4. Retries on transient failures (up to configurable maxRetries)
5. Immediately aborts on `BudgetExhaustedError` (PERMANENT, no retry)
6. Produces an `EVIDENCE.yaml` artifact capturing all run data
7. Supports LangGraph checkpoint resume — already-committed ChangeSpecs are skipped

## Acceptance Criteria

- [ ] **AC-1**: A `ChangeSpec` Zod schema (imported from APIP-1020 ADR output) is the typed input contract for the change loop. No integration code proceeds without this schema. (Gate: do not implement AC-5 through AC-8 until APIP-1020 is complete and the ChangeSpec Zod schema is published.)

- [ ] **AC-2**: An `ImplementationGraph` is implemented in `packages/backend/orchestrator/src/graphs/implementation.ts` following the `elaboration.ts` structure: `StateGraph`, `Annotation`-typed state, `createToolNode`-wrapped nodes, `runImplementation(storyId, config)` entry point, and LangGraph checkpointer configured with thread ID `{storyId}:implementation:{attemptNumber}`.

- [ ] **AC-3**: A `create-worktree` node creates an isolated git worktree at a configurable base path (`worktrees/{storyId}`) using `git worktree add`. The worktree path is recorded in graph state. On graph completion (success or failure), the worktree is removed via `git worktree remove --force`. Worktree cleanup never blocks graph result propagation (non-blocking cleanup pattern).

- [ ] **AC-4**: A `load-story` node reads the story file (STORY-SEED.md or elaborated story) and the ChangeSpec plan (produced by APIP-1020's Diff Planner) into graph state. Missing story or ChangeSpec plan produces a typed `LoadError` that transitions to `abort` edge.

- [ ] **AC-5**: A `change-loop` node iterates over ChangeSpecs in order. For each ChangeSpec: (a) calls `PipelineModelRouter.dispatch()` with `storyId`, `agentId: 'implementation-codegen'`, and the ChangeSpec as message context; (b) writes the generated code to the worktree; (c) runs micro-verify (typecheck + scoped test); (d) on micro-verify pass, makes an atomic `git commit` in the worktree with message `feat({storyId}): {changeSpec.id} — {changeSpec.summary}`; (e) on micro-verify fail, applies retry logic (up to configurable `maxRetries`, default: 2) before transitioning to `abort`.

- [ ] **AC-6**: `PipelineModelRouter.dispatch()` throws `BudgetExhaustedError` → change loop treats this as PERMANENT, immediately aborts remaining ChangeSpecs, marks evidence `status: aborted`, and propagates `BudgetExhaustedError` to the BullMQ job processor for permanent failure (no retry).

- [ ] **AC-7**: Micro-verify runs `pnpm check-types --filter <changed-package>` and `pnpm test --filter <changed-package>` scoped to the packages touched by the ChangeSpec. Both commands run inside the worktree path. A micro-verify pass requires both commands to exit 0. Stdout/stderr are captured and included in the evidence `commands_run` array.

- [ ] **AC-8**: Each successful ChangeSpec commit is recorded atomically: commit SHA is captured and stored in graph state under `completedChanges`. On graph restart (LangGraph checkpoint resume), already-committed ChangeSpecs are skipped (idempotent loop — check `completedChanges` before dispatching).

- [ ] **AC-9**: An `evidence-production` node runs after the change loop completes (pass or abort). It writes an `EVIDENCE.yaml` file to the story's feature directory using `EvidenceSchema` + `createEvidence()` / `addTouchedFile()` / `addCommandRun()` helpers. Evidence captures: AC coverage status, touched files per ChangeSpec, micro-verify command results, commit SHAs, token summary (from `PipelineModelRouter` usage), and abort reason if applicable.

- [ ] **AC-10**: The implementation graph logs structured events with `@repo/logger` for each lifecycle transition: `graph_started`, `worktree_created`, `story_loaded`, `change_spec_dispatched`, `micro_verify_passed`, `micro_verify_failed`, `change_committed`, `change_aborted`, `evidence_written`, `graph_completed` — with fields: `storyId`, `changeSpecId`, `provider`, `attemptNumber`, `durationMs`, `tokenUsage` on each event.

- [ ] **AC-11**: Vitest unit tests cover: graph compiles and has expected node/edge structure; change loop routes to commit on micro-verify pass; change loop routes to retry on micro-verify fail; change loop routes to abort after maxRetries; `BudgetExhaustedError` triggers immediate abort (PERMANENT, no retry); already-committed ChangeSpec is skipped on resume; evidence production writes correct `EvidenceSchema`-conformant output.

- [ ] **AC-12**: Vitest integration test (using real git subprocess in `os.tmpdir()` fixture) verifies: `create-worktree` creates an actual git worktree; `git commit` in worktree produces a real commit visible via `git log`; micro-verify runs `pnpm check-types` against a real file change; cleanup removes the worktree. Test must NOT walk the real monorepo — use isolated temp fixture.

- [ ] **AC-13**: Existing orchestrator test suites (`packages/backend/orchestrator`) continue to pass unchanged. The implementation graph does not modify any file in `packages/backend/orchestrator/src/models/`.

## Architecture Notes

### Thread ID Convention
LangGraph checkpoint thread ID: `{storyId}:implementation:{attemptNumber}`

Consistent with the convention established in APIP-0020 (`{storyId}:{stage}:{attempt}`). This must be documented and respected. Changing it post-implementation requires checkpoint data migration.

### IModelDispatch Injection Pattern
The change loop node receives `IModelDispatch` (interface) injected via config/constructor, not the concrete `PipelineModelRouter` directly. This allows:
- Unit tests to mock at the interface boundary
- APIP-1030a (skeleton) to be built before APIP-0040 completes
- Future router swaps without changing graph internals

```typescript
// Interface (new file)
const IModelDispatchSchema = z.object({
  dispatch: z.function()
    .args(z.object({ storyId: z.string(), agentId: z.string(), context: ChangeSpecSchema }))
    .returns(z.promise(ModelResponseSchema))
})
type IModelDispatch = z.infer<typeof IModelDispatchSchema>
```

### Change Loop Conditional Edge Structure
```
change-loop → pass  → commit → (more specs?) → change-loop | evidence-production
           → retry  → (attempts < maxRetries?) → change-loop | abort
           → abort  → evidence-production
```

### ChangeSpec Placeholder Schema
A minimal placeholder Zod schema enables graph skeleton and unit tests before APIP-1020 closes:
```typescript
// packages/backend/orchestrator/src/graphs/change-spec-schema.ts
// PLACEHOLDER — replace with APIP-1020 ADR schema when available
const ChangeSpecSchema = z.object({
  id: z.string(),
  summary: z.string(),
  touchedPackages: z.array(z.string()),
  instructions: z.string(),
})
```

### Atomic Commit Pattern
Each ChangeSpec results in exactly one git commit before the loop advances:
```typescript
execSync(`git -C ${worktreePath} add -A`, { stdio: 'pipe' })
const sha = execSync(
  `git -C ${worktreePath} commit -m "feat(${storyId}): ${changeSpec.id} — ${changeSpec.summary}"`,
  { stdio: 'pipe' }
).toString().trim()
state.completedChanges.push({ id: changeSpec.id, sha })
```

### Zod-First Schema Requirement
All new types (graph state, ChangeSpec, micro-verify result, commit record, LoadError) MUST be `z.object()` schemas with `z.infer<>` type aliases. Zero TypeScript interfaces in graph or node files (per WINT-9090 lesson).

## Infrastructure Notes

- **Local filesystem only**: Per APIP ADR-001 Decision 4, git worktrees run on the local server — no S3 or remote storage. Worktree paths are absolute local paths.
- **Git version requirement**: `git worktree` requires git >= 2.5. Verify on pipeline server before implementation.
- **No Lambda**: This graph runs on the local LangGraph server (APIP-0030), not AWS Lambda.
- **BullMQ integration**: `BudgetExhaustedError` is propagated to the BullMQ job processor (APIP-0020 supervisor) for permanent job failure marking. The graph does not swallow it.
