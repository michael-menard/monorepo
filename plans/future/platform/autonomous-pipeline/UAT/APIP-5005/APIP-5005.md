# APIP-5005: Minimal Operator Visibility CLI

## Context

The autonomous pipeline (APIP) runs as a headless long-lived process on a dedicated local server. During Phase 0-1 development, before the Monitor UI in Phase 2 (APIP-2020), operators have no way to inspect pipeline health without reading raw Redis keys or scanning supervisor log files. This creates a debugging blind spot: when a story gets stuck, times out, or the queue backs up, there is no quick way to assess the situation.

Decision UX-001 pulled this story from Phase 2 to Phase 1 specifically to provide developer-time observability during Phase 1 graph implementation (APIP-1010 through APIP-1070).

This story is part of **Phase 1: Full Worker Graphs**.

**Hard pre-condition**: APIP-0010 (`@repo/pipeline-queue`) must be merged before implementation begins. The CLI's primary data source — `PIPELINE_QUEUE_NAME`, `createPipelineConnection()`, and `PipelineJobDataSchema` — does not exist until APIP-0010 ships.

**Architecture basis**:
- APIP-ADR-001 Decision 2: No HTTP supervisor endpoint in Phase 0-1 (APIP-2030 deferred). CLI queries BullMQ/Redis directly via ioredis.
- APIP-ADR-001 Decision 4: CLI runs on the same local dedicated server as the pipeline components. No cloud execution context.
- ADR-005: Integration tests require real Redis; skip if `REDIS_URL` not set.

## Goal

Deliver a lightweight command-line tool (`npx tsx apps/api/pipeline/src/cli/index.ts <command>`) that operators can invoke to get an immediate text-format snapshot of pipeline health. The tool is a one-shot query tool — not a daemon, not a dashboard.

## Non-Goals

- HTTP server or daemon/watch mode — CLI is one-shot only; polling deferred to APIP-2020 or follow-up
- Modifying supervisor process behavior or adding HTTP endpoints — deferred to APIP-2030
- Writing to the queue (enqueueing, cancelling, requeuing jobs) — read-only only
- Connecting to Aurora PostgreSQL — all pipeline state queried from Redis/BullMQ
- Dashboard UI — APIP-2020
- Importing from `apps/api/lego-api/core/cache/redis-client.ts` — Lambda singleton, incompatible with CLI
- Modifying `packages/backend/database-schema/` or `@repo/db` — protected
- Modifying `packages/backend/orchestrator/` graph signatures — read-only imports only
- Full graph execution history or tracing — deferred to APIP-3010 (Change Telemetry)

## Architecture Notes

**CLI connection lifecycle (required pattern)**:
```typescript
const connection = createPipelineConnection(process.env.REDIS_URL)
const queue = createPipelineQueue(connection)
try {
  // ... query and print
} catch (err) {
  console.error(`Error: Cannot connect to Redis at ${process.env.REDIS_URL}. Is the pipeline server running?`)
  process.exit(1)
} finally {
  await queue.close()
  await connection.quit()
}
```

**Why both `queue.close()` and `connection.quit()`**: BullMQ Queue maintains internal timers and event listeners — `queue.close()` releases those. The underlying ioredis socket is a separate resource — `connection.quit()` closes it. Calling only one leaves the other resource open, causing the process to hang.

**Supervisor state inference**: The supervisor has no HTTP health endpoint (APIP-2030 deferred). CLI infers state by reading BullMQ job records:
- `getActive()` returns > 0 jobs → supervisor is processing
- `getActive()` returns 0 → supervisor is idle
- Last completed/failed jobs from `getCompleted(1)` / `getFailed(1)` provide context

**APIP-1070 stub**: `graph status` command filters jobs by `data.stage` field (from `PipelineJobDataSchema`). Until APIP-1070 ships, no jobs will have `stage: 'merge'`. When merge stage count is 0 and no merge jobs exist, display: `merge: N/A (pending APIP-1070)`. Once APIP-1070 merges jobs with `stage: 'merge'` into BullMQ, the count will naturally show.

**ESLint config requirement**: The project's ESLint configuration enforces `no-console` for application code but must exempt CLI scripts. Add `apps/api/pipeline/src/cli/**/*.ts` to the Node.js files glob in `eslint.config.js` (same pattern used for `scripts/**/*.ts`).

**No `@repo/logger`**: CLI output is operator-facing text (not structured application logs). Use `console.log`/`process.stdout.write` for display output. `@repo/logger` emits structured JSON — inappropriate for a terminal tool.
