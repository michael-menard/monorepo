# APIP-4060: KB Freshness Check and Stale Entry Archival

## Context

The knowledge base at `apps/api/knowledge-base/` accumulates entries over time. Entries reference file paths, story IDs, and code patterns that can become stale as the codebase evolves. Left unchecked, stale KB entries degrade the quality of guidance available to pipeline agents — directing them toward deleted files, deprecated patterns, or advice that no longer matches the codebase.

The `knowledgeEntries` table already has `archived` and `archivedAt` fields (used by WKFL-009), and `kb_update` supports setting these fields. The infrastructure for archival exists; this story adds the scheduled cron worker that identifies and archives stale entries.

**Problem**: There is no automated mechanism to identify KB entries that have become stale: entries that reference files no longer present in the repository, entries older than 90 days that have not been reaffirmed, or entries whose code-pattern advice has drifted. As the pipeline runs more stories and the codebase evolves, this gap compounds.

**Dependency Note**: APIP-3090 (Cron Job Infrastructure) is a hard dependency for scheduling. However, APIP-4060 can be fully implemented and tested in advance. The function is designed as a standalone async function that APIP-3090 registers as a cron handler.

## Goal

Implement `runKbFreshnessCheck` — a standalone async function that scans KB entries older than a configurable age threshold, auto-archives entries referencing confirmed non-existent file paths, flags borderline entries with a `stale-candidate` tag, and returns a structured summary with counts and timing metadata.

## Non-Goals

- Contradiction detection against CLAUDE.md or codebase patterns (semantic comparison — deferred to a follow-up story)
- Implementing the cron schedule or registration (APIP-3090's responsibility)
- Hard-deleting KB entries (only soft-archive via `archived: true`)
- Processing entries already marked `archived: true`
- UI or operator-facing views of the freshness report (APIP-4070 handles reporting)
- Modifying the `knowledgeEntries` schema (archival fields already exist)
- Making network or LLM calls (file-existence checks use local filesystem only)

- [ ] **AC-3**: For each candidate entry that contains a file path reference (heuristic: content matches `/(apps|packages|src)\/[^\s'"`,]+\.(ts|tsx|js|json|md)/g`), the function checks whether that file exists in the repository.

- [ ] **AC-5**: Entries exceeding the age threshold but without a verifiable non-existent file reference are flagged by adding a `stale-candidate` tag via `kb_update({ id, tags: [...existingTags, 'stale-candidate'] })`.
- [ ] **AC-6**: In dry-run mode (`config.dryRun: true`), no mutations are made — the function returns counts of what would be archived/flagged without touching the DB.
- [ ] **AC-7**: The function returns a structured result: `{ archived_count, flagged_count, skipped_count, entries_scanned, duration_ms, dry_run, batches_processed, truncated }`. The `truncated` field is `boolean` (default `false`), set to `true` when processing halts early due to `maxDurationMs` being exceeded (see AC-13).
- [ ] **AC-8**: A Zod schema (`KbFreshnessResultSchema`) exists in `__types__/index.ts` and validates the return value.
- [ ] **AC-9**: A Zod schema (`KbFreshnessConfigSchema`) exists in `__types__/index.ts` and validates the config input with defaults (`staleDays: 90`, `dryRun: false`, `batchSize: 500`).
- [ ] **AC-10**: Unit tests exist in `__tests__/freshness.test.ts` covering: dry-run returns counts without mutations, non-existent file paths are archived, existent file paths are flagged (not archived), entries without file references are skipped, batch processing respects `batchSize`.
- [ ] **AC-11**: The function is exported from `packages/backend/orchestrator/src/nodes/freshness/index.ts` as a named export (no barrel file).
- [ ] **AC-12**: All archival mutations are wrapped in a try/catch — a single entry failure does not abort the full batch; errors are logged via `@repo/logger` and the entry is counted as `skipped`.
- [ ] **AC-13**: The function is time-bounded: if `duration_ms > maxDurationMs` (config, default 5 minutes), processing halts at the current batch boundary and the partial result is returned with a `truncated: true` flag.
- [ ] **AC-14**: The batch scan uses direct Drizzle queries (`db.select().from(knowledgeEntries).where(...).limit(batchSize).offset(offset)`) — NOT `kb_list`. The `kb_list` helper does not support offset-based pagination or the compound `createdAt < cutoff AND archived = false` filter required for this scan. The Architecture Notes batch loop pseudocode is the authoritative implementation reference.
- [ ] **AC-15**: The in-memory batch entry object (returned by the batch `db.select()`) is used directly as `previousEntry` when calling `AuditLogger.logUpdate(entryId, previousEntry, newEntry)`. No separate pre-mutation fetch is needed because the batch select returns full `KnowledgeEntry` rows. The `kb_update` return value serves as `newEntry`.

## Architecture Notes

### Function Signature

```typescript
export async function runKbFreshnessCheck(
  deps: KbFreshnessCheckDeps,   // { db: NodePgDatabase<...>, fs?: { existsSync: (path: string) => boolean } }
  config: KbFreshnessConfig     // validated by KbFreshnessConfigSchema
): Promise<KbFreshnessResult>   // validated by KbFreshnessResultSchema
```

The `fs` dependency is injectable for testability. Default to `node:fs` in production.

### File Path Extraction Heuristic (AC-3)

```typescript
const FILE_PATH_REGEX = /(apps|packages|src)\/[^\s'"`.,\])+\.(ts|tsx|js|json|md)/g
```

- Conservative: only matches paths with explicit file extensions
- Scans the full `content` field of each KB entry
- Multiple matches: process the first confirmed non-existent path (archive trumps flag)
- No match: entry is skipped (not flagged or archived)

### Batch Loop Pattern (from retention-policy.ts)

```
offset = 0
while (true):
  batch = db.select().from(knowledgeEntries)
    .where(createdAt < cutoff AND archived = false)
    .limit(batchSize).offset(offset)
  if batch.length === 0: break
  process(batch)
  offset += batchSize
  if batch.length < batchSize: break
  if Date.now() - startTime > maxDurationMs: return { ...result, truncated: true }
```

### Conservative Archival Policy

- Auto-archive ONLY when file path is explicit and `fs.existsSync` returns false
- Flag with `stale-candidate` tag for all other aged entries (age > staleDays, path exists or no path)
- This ensures false positives (archiving valid guidance) are minimized

### APIP-3090 Integration Contract

`runKbFreshnessCheck` is a pure async function with no LangGraph imports. APIP-3090 wraps it in a cron node:

```typescript
// APIP-3090 will own this adapter:
cronNode.register('monthly-kb-freshness', async () => {
  return runKbFreshnessCheck({ db }, { staleDays: 90, dryRun: false, batchSize: 500 })
})
```
