# APIP-4010: Codebase Health Gate — Snapshot, Drift Detection, and CLEANUP Story Auto-Generation

## Context

As the autonomous pipeline merges stories continuously, there is no automated mechanism to detect
gradual degradation of codebase quality metrics. Lint warnings can accumulate, type errors can creep
in, test coverage can drop, and dead exports can grow — all without any alert to the pipeline operators.
Without a health gate, the pipeline's output could silently degrade the codebase over time.

This story delivers the Phase 4 health gate: an automated snapshot, drift detection, and CLEANUP
story auto-generation system that fires after every 5th merge. The system captures 8 codebase health
metrics, compares them against a stored baseline, and generates prioritized CLEANUP stories when
any metric drifts beyond its configured threshold.

The implementation follows established APIP patterns:
- `wintSchema.table(...)` with drizzle-zod (same as `wint.change_telemetry` in APIP-3010)
- Fire-and-forget insert helper with injectable `db` (same as `writeTelemetry()` in APIP-3010)
- Configurable thresholds in a Zod config object (not hardcoded constants)
- CLEANUP story YAML validated against `StoryArtifactSchema` before writing

**Dependencies**: APIP-1070 (merge-count tracking), APIP-3090 (cron infrastructure), APIP-5007
(SQL migration runner). The DB schema, helpers, and detection logic can be implemented and tested
independently of APIP-3090's cron wiring. AC-10 (cron registration) is deferred until APIP-3090
provides a stable registration interface.

## Goal

Build the Codebase Health Gate as three integrated components plus cron registration:

1. **`wint.codebase_health` table** — Aurora PostgreSQL table storing one snapshot row per health
   check run, with columns for 8 quality metrics plus `merge_number`, `captured_at`, and
   `is_baseline` flag.

2. **`captureHealthSnapshot(config, db)`** — TypeScript function that shells out to CLI tools to
   measure each metric, inserts a row into `wint.codebase_health`, and returns the snapshot.
   Fire-and-forget with respect to the merge pipeline (non-blocking on failure).

3. **`detectDriftAndGenerateCleanup(snapshot, baseline, thresholds)`** — Pure function that
   compares the new snapshot against the most recent baseline row. For any metric exceeding its
   configured threshold delta, generates a CLEANUP story YAML artifact conforming to
   `StoryArtifactSchema` and writes it to the appropriate backlog directory.

4. **LangGraph cron task registration** — Registers the health gate as a cron job in APIP-3090's
   infrastructure, triggered after every 5th merge (deferred until APIP-3090 is ready).

## Non-Goals

- Building a UI dashboard for health metrics (deferred to APIP-4070 Weekly Pipeline Health Report
  or APIP-2020 Monitor UI).
- Auto-remediating detected issues (CLEANUP stories are generated for human/pipeline review,
  not auto-executed).
- Supporting distributed/multi-process cron execution — single-server assumption per ADR-001
  (Phase 4 constraint).
- Defining or implementing the CLEANUP story workflow (how the pipeline processes generated stories).
- Implementing metric collection for platforms other than the local dev server.
- Modifying any existing wint tables or production DB schemas.
- Implementing automatic baseline promotion logic — manual process only (operator sets
  `is_baseline=true` on a known-good row after verification).

## Architecture Notes

### Database Design

`wint.codebase_health` uses a simpler `is_baseline` boolean flag (default false) rather than a
separate `wint.health_baselines` table. This reduces schema complexity while preserving the ability
to mark any snapshot row as a baseline.

**Manual baseline promotion** (documented process, not automated):
```sql
UPDATE wint.codebase_health SET is_baseline = true WHERE id = '<known-good-row-id>';
```
The first-ever snapshot is NOT automatically treated as a baseline. Operators must explicitly
promote a row after verifying it represents a known-good pipeline state. This prevents the
silent-drift-accepted problem.

**Why no automatic baseline promotion**: Automatic promotion creates the risk that a degraded
snapshot becomes the new baseline, masking drift entirely. Calibration is empirical — initial
threshold values and promotion decisions require human judgment.

### Metric Capture Architecture

The 8 metric collector functions are decomposed into small, independently testable units in
`packages/backend/orchestrator/src/nodes/health/collectors/`. Each accepts an injectable
`execFn` parameter:

```typescript
// Example: collectTypeErrors accepts injectable execFn
async function collectTypeErrors(execFn: ExecFn): Promise<number | null>
```

This allows unit tests to mock CLI execution without running real `pnpm check-types:all`.

CLI tool mapping:
| Metric | CLI Tool |
|--------|----------|
| `lint_warnings` | `pnpm lint` (stdout line count) |
| `type_errors` | `pnpm check-types:all` (error count) |
| `any_count` | `pnpm lint` with `@typescript-eslint/no-explicit-any` rule |
| `test_coverage` | Vitest coverage JSON report |
| `circular_deps` | `madge --circular` |
| `bundle_size` | Build output manifest |
| `dead_exports` | `ts-prune` |
| `eslint_disable_count` | `grep -r eslint-disable` |

### Threshold Configuration

`HealthGateThresholdsSchema` is a Zod object with all fields optional and documented defaults:
```typescript
const HealthGateThresholdsSchema = z.object({
  lint_warnings_delta: z.number().default(10),      // Allow up to 10 new lint warnings
  type_errors_delta: z.number().default(0),          // Zero tolerance for new type errors
  any_count_delta: z.number().default(5),            // Allow up to 5 new any usages
  test_coverage_delta: z.number().default(-2),       // Allow up to 2% coverage drop
  circular_deps_delta: z.number().default(0),        // Zero tolerance for new circular deps
  bundle_size_delta: z.number().default(50_000),     // Allow up to 50KB growth
  dead_exports_delta: z.number().default(5),         // Allow up to 5 new dead exports
  eslint_disable_count_delta: z.number().default(2), // Allow up to 2 new eslint-disable
})
```

Initial calibration guidance: Start with these defaults and adjust after observing 3-4 health
gate runs. Note that `type_errors_delta: 0` and `circular_deps_delta: 0` are intentionally
strict — any new type error or circular dependency should trigger a CLEANUP story.

### Concurrency Guard

The health gate cron task should include a `pg_try_advisory_lock` guard (per APIP-3020 pattern)
as defense-in-depth against concurrent cron runs in case APIP-3090 scheduling guarantee ever lapses.

### CLEANUP Story ID Generation

CLEANUP story IDs use the format `APIP-CLEANUP-NNNN`. The implementation must query existing
CLEANUP story directories to determine the next available NNNN sequence number. This lookup
is done synchronously at generation time using `fs.readdirSync` on the backlog directory.

### Operator Log Format

After each health gate run, emit a structured log line (per UI/UX recommendation for APIP-4070):
```
[health-gate] run=complete merge_number={N} metrics_checked=8 metrics_within_threshold={M} cleanup_stories_generated={K} stories=[APIP-CLEANUP-NNNN, ...]
```
Log at INFO level on success, WARN level if any metric collection failed.
