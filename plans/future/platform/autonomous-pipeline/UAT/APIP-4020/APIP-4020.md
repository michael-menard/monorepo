# APIP-4020 — Cohesion Scanner: Weekly Pattern Consistency Scan and Auto-Remediation Story Generator

## Context

The autonomous pipeline (APIP) will produce significant volumes of code changes across the monorepo. Without periodic consistency checks, pattern drift will accumulate gradually — route handlers losing their Zod validation wrappers, React components migrating away from the required directory structure, import conventions diverging from `@repo/*` paths. None of these individually trigger a lint error or type failure, so they do not breach APIP-4010's per-merge health gate.

APIP-4010 (Codebase Health Gate) establishes a `wint.codebase_health` snapshot table triggered after each merge. APIP-4020 adds a **complementary weekly batch scan** that samples the codebase for *structural consistency* specifically, producing a cohesion score and surfacing targeted cleanup stories when the score degrades.

**Dependencies**:
- APIP-4010 must complete first — establishes `wint` schema migration pattern and codebase health table conventions
- APIP-3090 must complete for weekly cron wiring — scanner will function as a standalone trigger until then

**Blocked for implementation** until APIP-4010 is done. All schema design and pure-function work (ST-1, ST-2, ST-3, ST-5) can proceed independently.

## Goal

Implement a weekly LangGraph cron graph that samples the codebase for structural consistency, computes a per-category and composite cohesion score, persists snapshots, and auto-generates targeted cleanup stories in the backlog when any category score drops below its configured threshold — with deduplication to prevent backlog flooding.

## Non-Goals

- Full AST parsing — heuristic regex-based detection is sufficient for v1
- Detecting violations in test files (`__tests__/`) — production source files only
- Scanning `node_modules`, generated files, or `dist/` directories
- Enforcing violations in CI (detection and story-generation only — APIP-4010 handles gating)
- Modifying any existing DB schemas beyond adding `wint.cohesion_snapshots`
- Auto-applying code fixes — generates cleanup *stories*, not automatic edits
- Scanning the entire codebase on every run — sample-based is intentional
- UI for viewing cohesion scores — deferred to APIP-4070 Weekly Pipeline Health Report
- Full TypeScript AST analysis — deferred to v2

## Acceptance Criteria

- [ ] **AC-1**: `CohesionScannerConfigSchema` is defined with Zod and includes: `sampleRatePercent` (default 20), per-category thresholds, story generation toggle, deduplication window in days, and configurable category weights
- [ ] **AC-2**: Pattern detectors are implemented as pure exported functions for each category: route handler structure (Zod validation present, service delegation present), Zod naming convention (`*Schema` suffix on `z.object()`), React directory structure (`index.tsx` + `__tests__/` present), import conventions (`@repo/*` used, no direct shadcn paths)
- [ ] **AC-3**: Each detector returns a `PatternViolationSchema` result: `{ category, filePath, violationType, description, severity, confidence: 'high' | 'medium' | 'low' }`
- [ ] **AC-4**: `CohesionScoreSchema` is computed per category: `{ category, score: 0-100, violationCount, sampleSize, scannedAt }`
- [ ] **AC-5**: Overall `CohesionScanResultSchema` aggregates per-category scores into a weighted composite score with configurable category weights from `CohesionScannerConfigSchema`
- [ ] **AC-6**: Scanner persists a `wint.cohesion_snapshots` row via Drizzle for each completed scan run (score, violation summary, run metadata)
- [ ] **AC-7**: When any category score is below its threshold, a CLEANUP story YAML is written to `plans/future/platform/autonomous-pipeline/backlog/COHCLEAN-{NNNN}/story.yaml` using `createStoryArtifact`; story ID uses `COHCLEAN-{NNNN}` format (auto-incrementing numeric suffix) to comply with `StoryArtifactSchema` ID validation (`/^[A-Z]+-\d+$/`); story title format: `"Cohesion Cleanup: {category} violations ({date})"` with top 5 worst violations in description
- [ ] **AC-8**: Cleanup story generation is idempotent — if a CLEANUP story for the same category was generated within the configured `deduplicationWindowDays`, no new story is created; existing story path is logged instead
- [ ] **AC-9**: LangGraph cron graph is defined with a weekly schedule and integrates into APIP-3090's cron infrastructure once available; standalone manual trigger is available for development use
- [ ] **AC-10**: PostgreSQL advisory lock (`pg_try_advisory_lock(hashtext('cohesion_scanner'))`) is acquired at cron entry; if lock not acquired, run is skipped with a warning log; lock released in `finally` block
- [ ] **AC-11**: Unit tests cover each pure detector function with known-compliant and known-violating file content fixtures; target 70%+ coverage on detector files
- [ ] **AC-12**: Unit tests cover score computation, threshold evaluation, and cleanup story deduplication logic
- [ ] **AC-13**: Integration test: run scanner against a controlled sample directory containing deliberate violations for each category; verify correct scores and that cleanup stories are generated with accurate violation descriptions

## Architecture Notes

### Schema Design

All types are Zod-first. Key schemas:

```typescript
// CohesionScannerConfigSchema — configures the entire scan run
const CohesionScannerConfigSchema = z.object({
  sampleRatePercent: z.number().min(1).max(100).default(20),
  categoryWeights: z.record(z.string(), z.number().min(0).max(1)),
  thresholds: z.record(z.string(), z.number().min(0).max(100)),
  storyGenerationEnabled: z.boolean().default(true),
  deduplicationWindowDays: z.number().int().min(1).default(7),
  pathExcludes: z.array(z.string()).default(['node_modules', 'dist', '__tests__']),
})

// PatternViolationSchema — one violation detected by a heuristic detector
const PatternViolationSchema = z.object({
  category: z.enum(['route-handler', 'zod-naming', 'react-directory', 'import-convention']),
  filePath: z.string(),
  violationType: z.string(),
  description: z.string(),
  severity: z.enum(['high', 'medium', 'low']),
  confidence: z.enum(['high', 'medium', 'low']),
})

// CohesionScoreSchema — per-category computed score
const CohesionScoreSchema = z.object({
  category: z.string(),
  score: z.number().min(0).max(100).nullable(),
  violationCount: z.number().int().min(0),
  sampleSize: z.number().int().min(0),
  scannedAt: z.string().datetime(),
})
```

### Heuristic Detector Design

Follow the `SYSTEM_REF_PATTERNS` + `countSystemReferences` pattern from `structurer.ts`:

```typescript
// Each detector is a pure exported function
export function detectRouteHandlerViolations(
  filePath: string,
  content: string
): PatternViolation[] {
  const violations: PatternViolation[] = []
  const hasZodValidation = /\.parse\(|\.safeParse\(/.test(content)
  const hasServiceDelegation = /await \w+Service\.\w+|await \w+\.execute/.test(content)
  if (!hasZodValidation) {
    violations.push({ category: 'route-handler', filePath, violationType: 'missing-zod-validation', ... })
  }
  return violations
}
```

The `confidence` field allows score weighting: high-confidence violations count at 1.0, low-confidence at 0.3. This enables threshold calibration without changing detection logic.

### Advisory Lock

```typescript
// At graph entry point
const lockAcquired = await db.execute(
  sql`SELECT pg_try_advisory_lock(hashtext('cohesion_scanner'))`
)
if (!lockAcquired.rows[0]?.pg_try_advisory_lock) {
  logger.warn('cohesion_scanner: advisory lock not acquired, skipping run')
  return
}
try {
  // ... scanner execution
} finally {
  await db.execute(sql`SELECT pg_advisory_unlock(hashtext('cohesion_scanner'))`)
}
```

### Cleanup Story Generation

Use `createStoryArtifact` with human-readable content:
- Title: `"Cohesion Cleanup: {category} violations ({YYYY-MM-DD})"`
- Description: lists top 5 worst violations with file paths
- Story ID format: `COHCLEAN-{NNNN}` (auto-incrementing, compliant with `StoryArtifactSchema` regex `/^[A-Z]+-\d+$/`)
- Directory path: `plans/future/platform/autonomous-pipeline/backlog/COHCLEAN-{NNNN}/`
- Deduplication: check for existing `COHCLEAN-*` stories with matching category in description within `deduplicationWindowDays`
