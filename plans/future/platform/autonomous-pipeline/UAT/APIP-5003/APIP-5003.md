# APIP-5003: LangGraph Platform Security Hardening and Network Boundary Documentation

## Context

The autonomous pipeline runs on a dedicated local server (APIP ADR-004) with LangGraph Platform services deployed via Docker Compose (APIP-0030). The LangGraph Platform exposes two services: the Platform API on port 8123 and the Studio UI on port 8124. These services provide internal pipeline execution capabilities that must not be reachable from outside the server's host network.

Architecture decision APIP DECISIONS SEC-001 establishes network isolation as the MVP security boundary. Full authentication and authorization is explicitly deferred to Phase 2 when external access is planned (FOLLOW-UPS SEC-001-full-auth). APIP-0030 establishes the basic port binding for Studio UI (`127.0.0.1:8124:8124`). This story builds on that foundation to document and harden the complete security posture of the LangGraph Platform deployment.

Without explicit documentation and hardening, the LangGraph Platform deployment is an implicit security boundary dependent on correct configuration in APIP-0030. Any future change to the compose file could inadvertently expose services. There is no documented port inventory, no container hardening baseline, and no record of what the network boundary is or how to verify it.

**Implementation Gate**: APIP-5003 compose hardening (AC-2, AC-3, AC-4, AC-5, AC-8) cannot begin until APIP-0030 is complete and `infra/langgraph-server/compose.langgraph-server.yaml` exists with LangGraph Platform services defined. Documentation ACs (AC-1, AC-6, AC-7, AC-9) may be drafted independently.

## Goal

Deliver a `SECURITY.md` document under `infra/langgraph-server/` that formally states the network boundary, port inventory, container hardening configuration, and secrets injection mechanism. Apply Docker container hardening directives to `compose.langgraph-server.yaml`. Provide an operational verification checklist that an operator can run to confirm the security posture is intact after any compose config change.

## Non-Goals

- Implementing authentication or authorization middleware for LangGraph Platform API (port 8123) — deferred to Phase 2 per FOLLOW-UPS SEC-001-full-auth
- Implementing mTLS between containers or between the supervisor and LangGraph Platform — Phase 2 concern
- Implementing the `SecretsClient` library or AWS Secrets Manager CloudFormation resources — that is APIP-5004's scope
- Configuring OS-level firewall rules (iptables/ufw) on the server — if required, document in `SECURITY.md` as a server provisioning prerequisite pointing to APIP-5006; do not implement here unless APIP-5006 ACs explicitly leave this undone
- Provisioning the dedicated server or installing Docker — that is APIP-5006 (prerequisite of APIP-0030 which is prerequisite of this story)
- Deploying or modifying any LangGraph worker graphs — those are APIP-1010 through APIP-1070
- Implementing SAST scanning or security review gates in the code review graph — that is APIP-1050 scope (FOLLOW-UPS SEC-003-story)
- Modifying production DB schemas in `packages/backend/database-schema/` — protected
- Modifying `@repo/db` client package API surface — protected
- Building a full security audit tool or automated compliance scanner

## Architecture Notes

**Security Boundary Model (APIP ADR-004 + SEC-001):**

The dedicated local server is the outermost trust boundary. Docker's host-network isolation provides the MVP security boundary for the autonomous pipeline:
- Ports bound to `127.0.0.1` are reachable only from processes on the same host
- No reverse proxy or external-facing load balancer is configured for LangGraph Platform in Phase 1
- The Supervisor Graph (APIP-0020) communicates with LangGraph Platform API via `127.0.0.1:8123` on the same host — this is the only authorized caller for MVP

**Container Hardening Strategy:**

Apply hardening in order of compatibility risk (lowest risk first):
1. Image tag pinning — zero risk, always apply (AC-5)
2. Port binding to `127.0.0.1` — zero risk, already in APIP-0030 for 8124; add for 8123 (AC-2)
3. Read-only volume mounts for config files — low risk, always apply (AC-4)
4. `cap_drop: [ALL]` — medium risk, verify LangGraph image does not need elevated capabilities (AC-3, ST-1)
5. `read_only: true` filesystem — high compatibility risk, requires ST-1 research first; use `tmpfs:` fallback (AC-3)
6. Non-root `user:` directive — image-dependent, document gap if unsupported (AC-3)

**Secrets Injection Boundary (with APIP-5004):**

```
APIP-5003 owns:                    APIP-5004 owns:
  compose.langgraph-server.yaml      SecretsClient library
  environment:                       AWS Secrets Manager CloudFormation
    ANTHROPIC_API_KEY: ${ANTHROPIC_API_KEY}  → value provided by APIP-5004 at runtime
    OPENROUTER_API_KEY: ${OPENROUTER_API_KEY}
    LANGGRAPH_POSTGRES_URL: ${LANGGRAPH_POSTGRES_URL}
```

APIP-5003 defines the Docker Compose injection mechanism. APIP-5004 provides the runtime values via its `SecretsClient`. The `.env` file is the handoff artifact between the two stories for MVP.

**Phase 2 Auth/Authz Deferral (FOLLOW-UPS SEC-001-full-auth):**

Full authentication and authorization for the LangGraph Platform API and Studio UI is explicitly out of scope for APIP-5003. Phase 2 work will design and implement:
- Bearer token authentication for LangGraph Platform API (port 8123)
- Access control for Studio UI (port 8124)
- mTLS between containers if inter-service trust requires it
- Network policy enforcement via iptables/ufw or Docker network driver
