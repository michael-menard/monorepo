# APIP-4050: Dead Code Reaper — Monthly Cron Analysis and CLEANUP Story Generation

## Context

As the autonomous pipeline continuously merges stories across the monorepo, unexported functions, components with no renderers, files with no importers, and stale `package.json` dependencies accumulate silently. Without an automated removal mechanism, dead code inflates cognitive load, slows TypeScript compilation, and degrades build times across the monorepo's operational lifetime.

This story delivers APIP Phase 4's Dead Code Reaper: a monthly cron job that runs static analysis across the monorepo and generates CLEANUP stories listing verified safe deletions for the pipeline to execute.

**Reality baseline** (2026-02-13):
- `packages/backend/orchestrator/src/graphs/metrics.ts` — canonical `createMetricsGraph()` / `runMetricsCollection()` factory pattern; direct template for this story
- `packages/backend/orchestrator/src/nodes/audit/` — injectable analysis node pattern; proven injectable `execFn` design
- `packages/backend/orchestrator/src/runner/timeout.ts` — `withTimeout()` available and required
- `packages/backend/orchestrator/src/artifacts/story.ts` — `StoryArtifactSchema` available for CLEANUP story validation
- APIP-3090 (Cron Job Infrastructure) — direct dependency; cron wiring (AC-10) is gated until APIP-3090 is complete
- APIP-4010 (Codebase Health Gate) — sibling Phase 4 story; captures `dead_exports` count via `ts-prune`; coordinate to avoid duplicate invocations

**Conflict resolutions**:
- APIP-4010 metric overlap (warning): Dead Code Reaper runs monthly; Health Gate runs every 5th merge. Different schedules minimize duplicate `ts-prune` runs. If sharing becomes desirable, expose `runTsPrune(execFn)` helper for both consumers.
- APIP-3090 dependency (warning): AC-10 (cron registration) is explicitly deferred with `InMemoryCronAdapter` for unit testing. All other ACs are testable independently.

---

## Goal

Implement a monthly cron job that automatically detects dead code across the monorepo (unexported functions, unused files, stale dependencies) using injectable, testable static analysis, micro-verifies each finding via `pnpm check-types`, and produces a validated CLEANUP story YAML artifact listing specific safe deletions for the pipeline to execute — without ever committing changes directly.

---

## Non-Goals

- Automatically committing or merging deletions — the Reaper generates CLEANUP stories for the pipeline to execute; it does not commit changes
- Modifying any existing graph (`elaboration.ts`, `metrics.ts`, `code-audit.ts`, etc.) — protected
- Modifying `@repo/db` client package API surface — protected
- Modifying any production DB schema — no new tables required (file-system + subprocess only)
- Adding HTTP endpoints or operator UI for dead code management — downstream of APIP-2020
- Implementing deletion execution logic — CLEANUP stories are consumed by the existing implementation pipeline
- Scanning for dead CSS classes, unused Tailwind utilities, or frontend-specific dead code — out of scope
- Using AWS services for scheduling — all cron runs on the dedicated server via LangGraph Platform (ADR-001)
- Replacing or duplicating the `dead_exports` metric in APIP-4010 — Health Gate captures the count; Dead Code Reaper provides file-level actionable detail

## Acceptance Criteria

- [ ] **AC-1**: A `DeadCodeReaperConfigSchema` (Zod) exists in `packages/backend/orchestrator/src/nodes/dead-code/` defining: `minAgeDays` (number, default: 30), `excludePatterns` (string array, default: `['**/*.d.ts', '**/generated*', '**/__mocks__/**']`), `maxFindingsPerRun` (number, default: 50), `timeoutMs` (number, default: 300000 — 5 minutes), and `dryRun` (boolean, default: false).

- [ ] **AC-2**: A `scanDeadExports(execFn)` function runs `ts-prune` across the monorepo and returns a `DeadExportFindingSchema[]` array. Each finding includes: `filePath`, `symbolName`, `symbolType` (`function` | `class` | `type` | `variable`), `lineNumber`, `reason` (`unexported-with-no-callers`).

- [ ] **AC-3**: A `scanUnusedFiles(execFn)` function runs TypeScript compiler analysis to identify files with no importers and returns `UnusedFileFindingSchema[]`. Each finding includes: `filePath`, `reason` (`no-importers`).

- [ ] **AC-4**: A `scanUnusedDeps(execFn)` function runs `depcheck` on each `package.json` in the monorepo and returns `UnusedDepFindingSchema[]`. Each finding includes: `packageJsonPath`, `depName`, `depType` (`dependencies` | `devDependencies`).

- [ ] **AC-5**: All three scan functions accept injectable `execFn: (cmd: string) => Promise<string>` parameters and respect `excludePatterns` from the config. Dynamic import patterns (`import(...)`) trigger an automatic exclusion guard — files that are only referenced via dynamic import are NOT flagged.

- [ ] **AC-6**: A `microVerify(finding, execFn)` function performs the micro-verification loop for a single `DeadExportFinding` or `UnusedFileFinding`: (a) writes a temporary deletion to a scratch branch (or simulates via dry-run mode), (b) runs `pnpm check-types`, (c) returns `MicroVerifyResultSchema` with `status: 'safe' | 'false-positive' | 'error'`, `typeCheckOutput` (string), and `durationMs`. In `dryRun: true` mode, the deletion is simulated without writing files.

- [ ] **AC-7**: A `runDeadCodeReaper(config)` entry function orchestrates the full analysis pass: runs all three scans concurrently (up to `maxFindingsPerRun` total), micro-verifies each `safe` candidate, filters out `false-positive` results, and returns a `DeadCodeReaperResultSchema` containing `findings[]`, `verifiedDeletions[]`, `falsePositives[]`, `unverified[]`, and summary counts.

- [ ] **AC-8**: `runDeadCodeReaper` enforces a wall-clock deadline of `config.timeoutMs` via `withTimeout` from `runner/timeout.ts`. On timeout, it returns whatever verified findings were accumulated before the deadline with `status: 'partial'`. **Important**: `withTimeout` throws `NodeTimeoutError` on deadline expiry — it does NOT return a partial result directly. `runDeadCodeReaper` MUST wrap the `withTimeout` call in a `try/catch`, catch `NodeTimeoutError`, and convert it to a `DeadCodeReaperResult` with `status: 'partial'` and whatever `verifiedDeletions` were accumulated before the deadline. This catch-and-convert logic must be explicit in the runner implementation. _Amended by autonomous elaboration_

- [ ] **AC-9**: A `generateCleanupStory(result, config)` function produces a CLEANUP story YAML artifact for each batch of verified deletions. Each story: (a) has ID in `APIP-CLEANUP-NNNN` format (next sequence from `plans/future/platform/autonomous-pipeline/backlog/`), (b) is written to `plans/future/platform/autonomous-pipeline/backlog/APIP-CLEANUP-NNNN/story.yaml`, (c) passes `StoryArtifactSchema.parse()` without errors, (d) lists each specific deletion with its `filePath`, `symbolName`, and micro-verify evidence.

- [ ] **AC-10**: The Dead Code Reaper is registered as a monthly cron job (`0 3 1 * *`) in APIP-3090's cron infrastructure via `buildCronRegistry`. It uses `pg_try_advisory_lock(hashtext('apip_cron_dead-code-reaper'))` at entry to guarantee single-instance execution. Concurrent invocations that cannot acquire the lock exit immediately with `CronRunResult { status: 'skipped' }`. *[Deferred until APIP-3090 provides stable registration interface.]*

- [ ] **AC-11**: Each Dead Code Reaper run produces a structured `CronRunResult` log entry containing: `jobName: 'dead-code-reaper'`, `startedAt`, `completedAt`, `durationMs`, `status` (`success` | `partial` | `failure` | `timeout` | `skipped`), and summary counts (`findingsTotal`, `verifiedDeletions`, `falsePositives`, `cleanupStoriesGenerated`).

- [ ] **AC-12**: Unit tests cover: (a) `DeadCodeReaperConfigSchema` valid/invalid parsing, (b) `scanDeadExports` with mock `execFn` returning `ts-prune` output — verifies finding extraction and `excludePatterns` filtering, (b-bis) `scanUnusedFiles` with mock `execFn` returning compiler output — verifies `UnusedFileFindingSchema[]` extraction and `excludePatterns` filtering (e.g., mock output listing two files where one matches an exclude pattern, asserting only the non-excluded file appears in results), _Added by autonomous elaboration_ (c) `scanUnusedDeps` with mock `execFn` returning `depcheck` JSON — verifies dep extraction, (d) `microVerify` with mock `execFn` returning zero type errors → `status: 'safe'`, (e) `microVerify` with mock `execFn` returning type errors → `status: 'false-positive'`, (f) `runDeadCodeReaper` with `dryRun: true` produces `DeadCodeReaperResultSchema`-conformant result without file writes, (g) `generateCleanupStory` output passes `StoryArtifactSchema.parse()`, (h) timeout path: `runDeadCodeReaper` with very short `timeoutMs` via `vi.useFakeTimers()` → `status: 'partial'`.

- [ ] **AC-13**: `pnpm check-types` and `pnpm lint` pass with no errors or warnings introduced by this story's TypeScript files. All existing tests in `packages/backend/orchestrator/` continue to pass.

- [ ] **AC-14**: CLEANUP story IDs generated by `generateCleanupStory` MUST use a `StoryArtifactSchema`-compliant format satisfying the regex `/^[A-Z]+-\d+$/` (i.e., one uppercase-letter segment, one hyphen, digits only — no compound multi-hyphen forms). The previously described format `APIP-CLEANUP-NNNN` (e.g., `APIP-CLEANUP-0001`) fails this regex and MUST NOT be used. Use `CLEANUP-NNNN` (e.g., `CLEANUP-0001`) as the canonical format — it satisfies `/^[A-Z]+-\d+$/`, is unique, and preserves human readability. AC-9's ID format specification (`APIP-CLEANUP-NNNN`) is superseded by this AC. ID sequencing via `fs.readdirSync` should scan for existing `CLEANUP-NNNN` directories accordingly. `StoryArtifactSchema.parse()` must succeed on every generated story YAML. _Added by autonomous elaboration_

## Architecture Notes

### Module Layout

```
packages/backend/orchestrator/src/nodes/dead-code/
  schemas.ts                    # All Zod-first type definitions
  scanners.ts                   # scanDeadExports, scanUnusedFiles, scanUnusedDeps
  micro-verify.ts               # microVerify with dryRun mode
  runner.ts                     # runDeadCodeReaper orchestration
  cleanup-story-generator.ts    # generateCleanupStory
  __tests__/
    dead-code-reaper.test.ts    # All unit tests (AC-12 subcases a-h)

packages/backend/orchestrator/src/graphs/
  dead-code-reaper.ts           # createDeadCodeReaperGraph() factory (additive)
```

### Key Design Decisions

**Injectable `execFn` pattern** (required — APIP-4010 lesson):
```typescript
// All subprocess-invoking functions follow this signature:
async function scanDeadExports(
  execFn: (cmd: string) => Promise<string>,
  config: DeadCodeReaperConfig
): Promise<DeadExportFinding[]>
```

**`withTimeout` for all expensive operations** (required — APIP-3090 lesson):
```typescript
// Never use createToolNode (10s default kills long-running analysis)
const result = await withTimeout(
  () => runDeadCodeReaper(config),
  { timeoutMs: config.timeoutMs, nodeName: 'dead-code-reaper' }
)
```

**`pg_try_advisory_lock` single-instance guard** (AC-10, deferred):
```typescript
const lockAcquired = await pg_try_advisory_lock(hashtext('apip_cron_dead-code-reaper'))
if (!lockAcquired) {
  return { jobName: 'dead-code-reaper', status: 'skipped', ... }
}
```

**Targeted typecheck for micro-verify** (performance):
- Use `tsc --project packages/{pkg}/tsconfig.json` per affected package
- NOT `pnpm check-types:all` — too slow for per-finding verification

**Dynamic import guard** (false positive mitigation):
- Parse ts-prune output and detect `import(...)` usage in the same file
- Files exclusively referenced via dynamic import are excluded from findings

**CLEANUP story ID sequencing**:
- `fs.readdirSync` on backlog directory to find max existing `APIP-CLEANUP-NNNN` sequence
- Increment by 1 for the new story
- `StoryArtifactSchema.parse()` validation before any write

### False Positive Mitigation Strategy

1. **`excludePatterns` config** (AC-1): Filter out `.d.ts`, generated files, mocks before scanning
2. **Dynamic import guard** (AC-5): Files only referenced via `import(...)` are not flagged
3. **Micro-verify as mandatory guard rail** (AC-6): Every candidate deletion is verified by `pnpm check-types` before being proposed — no deletion listed without a passing typecheck
