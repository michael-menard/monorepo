# APIP-0010: BullMQ Work Queue Setup

## Context

The autonomous pipeline (APIP) requires a durable, restart-safe work queue as the backbone for all pipeline stage transitions — from story intake through elaboration, implementation, review, QA, and merge. Per APIP ADR-001 Decision 1, BullMQ backed by the existing Redis/ElastiCache infrastructure replaces any PostgreSQL queue design.

Currently, no BullMQ infrastructure exists in the monorepo. Redis is already deployed locally via Docker Compose (port 6379, `infra/compose.lego-app.yaml`) with AOF persistence enabled (`--appendonly yes`). The existing `apps/api/lego-api/core/cache/redis-client.ts` demonstrates the ioredis connection pattern but is designed for Lambda singletons and cannot be reused directly for a long-running BullMQ process.

This story is on the critical path: APIP-0020 (Supervisor Loop), APIP-0040 (Model Router), and APIP-5000/5001/5007 all depend on it. The public API of the package created here (queue name, payload schema, factory signature) is a stable contract for downstream consumers.

**Baseline**: 2026-02-13. No BullMQ package exists in the monorepo. No pipeline queue infrastructure. Redis confirmed deployed. No prior BullMQ patterns in the codebase to reuse.

## Goal

Create `packages/backend/pipeline-queue` — a new monorepo package that provides a typed, Zod-validated BullMQ queue factory for the autonomous pipeline, following established package patterns (rate-limiter skeleton, gallery-core schema structure). The package forms the stable queue API contract that APIP-0020 will consume.

## Non-Goals

- Creating any PostgreSQL table or Drizzle schema for queue state (ADR-001 explicitly eliminates `wint.work_queue`)
- Implementing the supervisor loop that consumes the queue (APIP-0020)
- Implementing LangGraph worker graphs (APIP-1010+)
- Deploying BullMQ to the dedicated server (APIP-0020 + APIP-5006 concern)
- Full operator monitoring UI (APIP-2020)
- Changing the existing Redis Docker Compose config (AOF already enabled)
- Modifying `apps/api/lego-api/core/cache/redis-client.ts` (Lambda-specific, protected)
- Modifying `@repo/db` client package (protected)
- Modifying any schemas in `packages/backend/database-schema/` (protected)
- Secrets management for Redis credentials (APIP-5004; use environment variables for now)

## Architecture Notes

**BullMQ connection model**: BullMQ requires at least two separate IORedis connections per queue — one for standard Redis commands and one for blocking `BRPOP` operations. The `Queue` class and `Worker` class each manage their own connections internally. Pass a connection configuration object (or IORedis instance) and BullMQ will handle connection multiplexing. Do NOT pass a shared Lambda singleton.

**Long-running process model**: Per ADR-004, this package runs in a long-running Node.js process (not Lambda). The Lambda singleton pattern in `getRedisClient()` (module-level cached connection) is inappropriate here. Each `createPipelineConnection` call creates a fresh, dedicated IORedis connection.

**Queue name stability**: `PIPELINE_QUEUE_NAME = 'apip-pipeline'` is a contract. Once APIP-0020 uses this name, it cannot change without a Redis key migration. Export as a constant from `src/index.ts`.

**Zod validation at enqueue**: The queue factory should Zod-parse job data at enqueue time (`PipelineJobDataSchema.parse(data)`) to enforce the contract before data enters Redis. This prevents APIP-0020 from receiving malformed jobs.

**No custom retry logic**: BullMQ has built-in retry and backoff natively. Configure via `defaultJobOptions`. Do not build custom retry middleware on top.

## Infrastructure Notes

**Redis**: Already running locally on port 6379 via `infra/compose.lego-app.yaml` with AOF (`--appendonly yes`). No changes required for Phase 0.

**ElastiCache AOF risk (production)**: The local dev Redis has AOF enabled. ElastiCache AOF support depends on the configured tier — it is not available on `cache.t2.micro`. This story does not resolve this risk. Document in README as: "Verify ElastiCache tier supports AOF before production deployment."

**BullMQ ioredis version**: Run `pnpm why ioredis` before adding BullMQ. BullMQ v5+ requires ioredis v5+. Pin compatible versions.

**APIP-5006 dependency note**: APIP-0010 code ships in the monorepo. The runtime execution environment (dedicated server where APIP-0020 worker process runs) is provisioned by APIP-5006. This story creates the package; APIP-0020 + APIP-5006 handle deployment.
