# APIP-1040: Documentation Graph (Post-Merge)

## Context

After a story merges via the Merge Graph (APIP-1070), project documentation drifts unless manually updated. Today all documentation updates are human-authored, creating a bottleneck and a persistent gap between code reality and documentation state.

**Reality baseline (2026-02-13)**: No Documentation Graph exists in the autonomous pipeline. No `DocGraphState` annotation exists. No post-merge trigger mechanism exists in the pipeline yet (merge graph APIP-1070 not yet built). The 6 parallel doc workers must be designed from scratch. The existing `nodes/sync/doc-sync.ts` and `nodes/workflow/doc-sync.ts` nodes provide proven patterns for documentation file discovery and update logic but operate as single-phase nodes, not a 6-worker parallel fan-out graph.

**Active in-progress work that intersects**:
- APIP-0020 (In Elaboration): Supervisor Graph — dispatch interface must accommodate a `doc_graph` worker type for future integration
- APIP-5006 (In Elaboration): LangGraph Server Infrastructure — integration tests require the server
- APIP-5003 (In Elaboration): Security Hardening — git operations in commit node fall within the security boundary

**Constraints**:
- APIP ADR-001 Decision 4: All pipeline components run on the dedicated local server — no Lambda
- APIP ADR-001 Decision 2: Supervisor is plain TypeScript; Documentation Graph is a LangGraph worker graph
- APIP ADR-001 Decision 3: Documentation Graph reads the merge commit diff directly — does NOT depend on ChangeSpec schema

## Goal

Build the **Documentation Graph** — a post-merge LangGraph worker graph that automatically keeps all project documentation in sync with every merged story, without human authoring effort. The graph fans out 6 parallel documentation workers via the LangGraph `Send` API, aggregates results through a Doc Review node, and commits all proposed changes atomically via a single centralized commit node.

## Non-Goals

- Integrating with the Merge Graph (APIP-1070) as a triggered caller — APIP-1040 delivers the graph itself; trigger wiring is deferred to the integration story or APIP-1070
- Sophisticated content quality scoring in Doc Review (beyond success count and KB safety checks) — defer to follow-up if needed
- LLM-generated API documentation from scratch — workers update existing doc files based on diff context; they do not generate complete API docs from zero
- Storybook integration or component rendering for Component Docs Worker — file-based JSDoc updates only
- Deleting or archiving outdated KB entries — KB Sync Worker is append-only in Phase 1
- Changelog publishing (npm publish, GitHub release) — changelog drafting only; publishing is out of scope
- Any UI/dashboard for documentation update status — deferred to APIP-2020
- Operator CLI visibility for doc graph runs — deferred to APIP-5005
- Modifying `packages/backend/database-schema/` or `@repo/db` client — protected
- Migration of existing Claude Code doc-sync agent (`.claude/agents/`) — Documentation Graph is a new parallel track

## Architecture Notes

### Graph Flow

```
START
  → dispatch (reads DocGraphConfig, fans out enabled workers)
    → Send('doc-worker-api-docs', state)
    → Send('doc-worker-component-docs', state)
    → Send('doc-worker-architecture-docs', state)
    → Send('doc-worker-readme-guides', state)
    → Send('doc-worker-kb-sync', state)
    → Send('doc-worker-changelog', state)
  → aggregate (collects all workerResults via reducer: append)
  → doc-review (evaluates results, sets docReviewPassed/commitBlocked)
    → [if commitBlocked] log-blocked → END
    → [if passed] commit → END
```

### Worker Contract

Each of the 6 workers follows this contract:
1. Read `state.mergeEvent.changedFiles` to scope relevant files
2. Call LLM (cheap model via `config.model`) to propose file content updates
3. **Never write to filesystem** — append to `state.proposedFileChanges`
4. Return `DocWorkerResult` — never throw; catch all errors in `result.error`

### KB Sync Worker Deduplication Guard (Mandatory)

```typescript
// Pattern from persist-learnings.ts — replicate exactly
const existing = await config.kbSearchFn({ query: entryContent, limit: 5 })
const tooSimilar = existing.results.some(r => (r.relevance_score ?? 0) >= 0.85)
if (tooSimilar) {
  return { ...result, warnings: ['Entry skipped: similar content exists'] }
}
await config.kbAddFn({ content: entryContent, tags: ['auto-generated', `source:${storyId}`] })
```

### Aggregate Node Sentinel Count

Because LangGraph has no built-in join barrier for `Send`-dispatched nodes, the aggregate node uses a sentinel count:
- Dispatch node sets `state.expectedWorkers = enabledWorkers.length`
- `workerResults` uses `reducer: append` to accumulate as workers complete
- Aggregate node polls `state.workerResults.length >= state.expectedWorkers` to detect completion

### Centralized Commit (Atomicity)

Workers **never** write to filesystem. They write `ProposedFileChange` objects to `state.proposedFileChanges`. Only the commit node applies writes after Doc Review passes. Write-then-rename pattern for atomicity; if any write fails mid-way, log partial failure and do not leave git in dirty state.
