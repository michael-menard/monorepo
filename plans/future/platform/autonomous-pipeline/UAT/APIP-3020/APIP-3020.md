# APIP-3020: Model Affinity Profiles Table and Pattern Miner Cron

## Context

The autonomous pipeline's learning system requires a continuously maintained record of each LLM model's empirical strengths and weaknesses per change type and file type. After APIP-3010 instruments the implementation loop to record raw change attempt telemetry into `wint.change_telemetry`, APIP-3020 is responsible for:

1. **`wint.model_affinity` Aurora PostgreSQL table**: A new Drizzle ORM schema definition in the `wint` namespace that stores aggregated per-`(model, change_type, file_type)` success rates, average attempt counts, average token usage, confidence levels, and trend data. This table is the source of truth for smart routing (APIP-3040), affinity-guided structuring (APIP-3050), and the bake-off engine (APIP-3060).

2. **Pattern Miner LangGraph cron graph**: A LangGraph graph (following the `metrics.ts` pattern) that runs on a schedule and aggregates `change_telemetry` rows into updated `model_affinity` profiles. The aggregation is incremental (watermark-based), computes confidence levels as a structured enum, and handles the cold-start day-1 state (no profiles yet).

No existing `wint.model_affinity` table or Pattern Miner implementation exists in the codebase. The existing wint.ts ML Pipeline tables (`trainingData`, `mlModels`) serve a different purpose (quality prediction, WINT-0050) and must not be conflated with this story's deliverables.

**Upstream dependency**: APIP-3010 must complete before APIP-3020 integration tests can run. APIP-3010 is in backlog with no implementation yet — APIP-3020 may begin schema/migration/unit test work immediately, but integration tests against a real `change_telemetry` table are gated behind APIP-3010.

**Downstream impact**: APIP-3030 (Learning-Aware Diff Planner), APIP-3040 (Learning-Aware Model Router), APIP-3050 (Affinity-Guided Structurer), APIP-3060 (Bake-Off Engine), and APIP-3090 (Cron Infrastructure) all depend on this story.

## Goal

Define the `wint.model_affinity` Drizzle ORM schema and SQL migration, then implement the Pattern Miner as a LangGraph cron graph that incrementally aggregates `wint.change_telemetry` rows into per-`(model, change_type, file_type)` affinity profiles with structured confidence levels and trend data.

## Non-Goals

- Implementing the change telemetry writer or instrumenting the implementation loop — that is APIP-3010
- Implementing model routing that reads from `model_affinity` — that is APIP-3040
- Implementing affinity-guided diff planning or story structuring — APIP-3030 and APIP-3050
- Implementing the bake-off engine for model experiments — APIP-3060
- Implementing the cold-start exploration budget mechanism — APIP-3070 (this story only defines the cold-start data state; the routing fallback logic is APIP-3070)
- Implementing the general cron job scheduling infrastructure — APIP-3090 (this story delivers the first cron graph; the scheduling registration framework is APIP-3090)
- Adding operator UI or dashboard visibility for affinity profiles — APIP-2020 / future
- Modifying any existing WINT ML pipeline tables (`trainingData`, `mlModels`, `modelPredictions`, `modelMetrics`) — those are for quality prediction (WINT-0050), not model affinity routing
- Modifying `@repo/db` client API surface — protected
- Adding any HTTP endpoints — this story has no API surface

## Architecture Notes

### wint.model_affinity Table Design

```
wintSchema.table('model_affinity', {
  id: uuid PK
  model: text NOT NULL           -- LLM identifier (e.g., 'gpt-4o', 'claude-3-5-sonnet')
  change_type: text NOT NULL     -- From change_telemetry.change_type
  file_type: text NOT NULL       -- From change_telemetry.file_type (e.g., '.ts', '.tsx')
  sample_size: integer           -- Total telemetry rows aggregated for this combination
  success_rate: numeric(5,4)     -- 0.0000-1.0000; weighted average across incremental runs
  avg_attempts: numeric          -- Average attempt count per change
  avg_tokens: numeric            -- Average tokens consumed per change
  avg_cost_usd: numeric          -- Average cost per change
  confidence_level: enum         -- 'none' | 'low' | 'medium' | 'high' (pgEnum)
  trend: jsonb                   -- { direction: 'up'|'down'|'stable', delta: number, computed_at: string }
  last_aggregated_at: timestamp  -- Watermark: latest change_telemetry.created_at processed
  created_at: timestamp
  updated_at: timestamp

  UNIQUE INDEX: (model, change_type, file_type)
  INDEX: confidence_level
  INDEX: last_aggregated_at
})
```

### Confidence Thresholds (Named Constants)

```typescript
const CONFIDENCE_THRESHOLDS = {
  NONE: 0,       // sample_size = 0 → 'none'
  LOW_MAX: 19,   // 1–19 → 'low'
  MEDIUM_MAX: 49 // 20–49 → 'medium'
  // 50+ → 'high'
}
```

### Incremental Aggregation Strategy

The Pattern Miner uses `MAX(last_aggregated_at)` from the `model_affinity` table as a global watermark. On each run:
1. Read watermark = `SELECT MAX(last_aggregated_at) FROM wint.model_affinity` (null on cold-start → use epoch)
2. Query `wint.change_telemetry WHERE created_at > watermark GROUP BY (model, change_type, file_type)`
3. For each group: compute `sample_size`, `success_rate`, `avg_attempts`, `avg_tokens`, `avg_cost_usd`
4. Upsert with weighted average formula for `success_rate` on conflict

### Weighted Average Formula for Upsert

On `ON CONFLICT (model, change_type, file_type) DO UPDATE SET`:
```sql
success_rate = (model_affinity.success_rate * model_affinity.sample_size
               + EXCLUDED.success_rate * EXCLUDED.sample_size)
               / (model_affinity.sample_size + EXCLUDED.sample_size)
```
This preserves historical accuracy when incrementally adding new observations.

### LangGraph Graph Structure

Following `metrics.ts` pattern:
- `PatternMinerConfigSchema` (Zod): config for graph (watermark override, dry-run mode)
- `PatternMinerStateAnnotation` (Annotation): graph state tracking rows processed, errors, watermark
- `createPatternMinerGraph()`: factory returning `StateGraph` with nodes: `fetchTelemetry` → `aggregate` → `upsertProfiles` → `logResults`
- `runPatternMiner(config)`: entry point for LangGraph Platform cron invocation; returns structured result with `rows_aggregated`, `profiles_updated`, `durationMs`

### APIP-3090 Compatibility

The Pattern Miner must be structured so APIP-3090 can register it as a cron task without modifying the graph itself. The `runPatternMiner(config)` entry point is the contract. APIP-3090 will call this function on a schedule (e.g., every 15 minutes). The Pattern Miner must be idempotent and safe to run concurrently (though APIP-3090 should ensure single-instance execution).

### Cold-Start Contract

On day 1 with no telemetry data:
- `wint.model_affinity` has zero rows
- Pattern Miner runs without error
- Logs `{ rows_aggregated: 0, profiles_updated: 0, durationMs: <N> }`
- Downstream consumers (APIP-3040, APIP-3050) must check `model_affinity` for rows and handle the empty case with their own fallback logic

### change_telemetry Column Reference (APIP-3010 Contract)

Aggregation SQL is written against APIP-3010's authoritative column list:
- `change_type` — type of change (e.g., 'refactor', 'add-feature', 'fix-bug')
- `file_type` — file extension (e.g., '.ts', '.tsx', '.sql')
- `model` — LLM model identifier
- `outcome` — 'success' | 'failure' (used in `SUM(CASE WHEN outcome = 'success' THEN 1.0 ELSE 0.0 END) / COUNT(*)`)
- `attempts` — attempt count for this change
- `tokens` — tokens consumed
- `cost` — cost in USD
- `created_at` — used as watermark column

If APIP-3010 column names differ slightly in implementation, update the aggregation SQL constant in one place.
