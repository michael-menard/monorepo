# APIP-4040: Test Quality Monitor

## Context

As of February 2026, the autonomous pipeline's QA graph (APIP-1060) enforces a coverage gate, but coverage percentage alone is an unreliable proxy for test effectiveness. The pipeline can generate tests that achieve the coverage threshold by executing code paths without meaningful assertions — superficial tests that pass the gate but miss real regressions.

The orchestrator package at `packages/backend/orchestrator/` has an established pattern for periodic collection graphs (`createMetricsGraph` / `runMetricsCollection` in `src/graphs/metrics.ts`), a `withTimeout` primitive for time-bounding expensive operations, a `code-audit` graph with a `test-coverage` lens, and a `story-creation` graph for auto-generating improvement stories. APIP-3090 (Cron Job Infrastructure, currently in backlog) will deliver the `CronJobRegistry` and `registerCronJobs` infrastructure that APIP-4040 needs for scheduling. All new database tables live in the `wint` pgSchema namespace per the convention established in `packages/backend/database-schema/src/schema/wint.ts`.

APIP-4040 introduces a weekly cron-based monitor that measures multiple dimensions of test quality and auto-generates targeted improvement stories on decay detection.

## Goal

Ensure autonomous pipeline output includes meaningful tests rather than superficial coverage that passes the QA gate but misses real regressions, by measuring assertion density, mutation score, orphaned tests, and critical path coverage floors weekly — and automatically generating test improvement stories when any metric decays below configured thresholds.

## Non-Goals

- Implementing a UI dashboard for test quality trends — deferred to APIP-2020 (Monitor UI)
- Running mutation testing on the entire monorepo — only configurable sampled packages; full-codebase mutation testing is explicitly out of scope due to compute cost
- Modifying any existing orchestrator graphs (elaboration, metrics, story-creation, code-audit) — **protected**
- Modifying any existing `wint` table schemas already in production — only ADD the new `wint.test_quality_snapshots` table
- Replacing the QA gate in APIP-1060 — this monitor is additive, not a replacement
- Adding HTTP endpoints or operator CLI commands — scoped to APIP-2020/APIP-5005
- Real-time test quality alerts — this is a weekly batch job, not a real-time monitor
- Mutation testing for frontend React components — scoped to orchestrator/backend packages only in v1
- Integrating with external mutation testing SaaS (e.g., Stryker Dashboard) — keep fully local

## Architecture Notes

### Graph Structure

Follow the `createMetricsGraph` factory pattern from `graphs/metrics.ts`:

```
TestQualityMonitorState (Annotation)
  ├── assertionDensityResult
  ├── mutationScoreResult
  ├── orphanedTestResult
  ├── criticalPathCoverageResult
  ├── snapshot
  ├── decayResult
  └── generatedStories

createTestQualityMonitorGraph(config)
  → StateGraph with nodes:
    collect-assertion-density → collect-orphaned-tests → collect-critical-path-coverage
    collect-mutation-score (parallel, time-boxed)
    → build-snapshot → detect-decay → generate-stories → persist-snapshot → emit-result

runTestQualityMonitor(config) → entry point
```

### Threshold Configuration

All thresholds are configurable via environment variables to avoid hardcoded constants:

| Config Field | Env Var | Default |
|---|---|---|
| `assertionDensityThreshold` | `TEST_QUALITY_ASSERTION_DENSITY_THRESHOLD` | `2.0` |
| `mutationScoreTarget` | `TEST_QUALITY_MUTATION_SCORE_TARGET` | `0.60` |
| `criticalPathCoverageFloor` | `TEST_QUALITY_CRITICAL_PATH_FLOOR` | `0.80` |
| `mutationTestPackages` | `MUTATION_TEST_PACKAGES` (comma-separated) | `['packages/backend/orchestrator']` |
| `mutationTimeoutMs` | `MUTATION_TIMEOUT_MS` | `300000` |

### Snapshot Persistence

Use the fire-and-forget insert pattern (from APIP-3010):
- Snapshot insert resolves without throwing
- On DB insert failure: log `logger.warn` with `{ jobName, error }` and continue
- Never let snapshot persistence failure cause `CronRunResult.status = 'failure'`

### `withTimeout` Usage (Corrected Signature)

The actual `withTimeout` signature from `runner/timeout.ts` uses a `TimeoutOptions` object, not a bare milliseconds value:

```typescript
// Per-collector time-boxing — do NOT use createToolNode (10s default timeout)
// Use withTimeoutResult for non-throwing fallback behavior (recommended for APIP-4040-B)
import { withTimeout } from '../runner/timeout'

const result = await withTimeout(
  () => runExpensiveCollector(),
  { timeoutMs: config.mutationTimeoutMs, nodeName: 'collect-mutation-score' }
)
// On timeout: NodeTimeoutError is thrown — catch and return fallback
// Alternative: use withTimeoutResult() which returns { result, timedOut } without throwing
```

**Note:** For APIP-4040-B (mutation score collector), prefer `withTimeoutResult()` which returns `TimeoutResult<T>` without throwing — cleaner for fallback-on-timeout patterns.
