# APIP-5007: Database Schema Versioning and Migration Strategy for the Autonomous Pipeline

## Context

The autonomous pipeline (Phase 0–4) is introducing multiple PostgreSQL schemas as it matures: the LangGraph checkpoint store (Phase 0, established by APIP-5001), change telemetry tables (Phase 3, APIP-3010), model affinity profiles (Phase 3, APIP-3020), and cron job infrastructure (Phase 3, APIP-3090). APIP-5001 (currently in ready-for-qa) established a minimal provisional migration runner at `apps/api/pipeline/src/db/` using numbered SQL files — but explicitly designed it as throwaway, pending this story's framework decision.

Without an authoritative migration strategy, each future pipeline story invents its own approach, creating inconsistent versioning, ungated breaking changes in CI, and no rollback safety classification.

The existing production LEGO app schema (`packages/backend/database-schema/`) has a mature migration strategy: Drizzle ORM, `XXXX_description.sql` naming, `schema_versions` metadata table, CI breaking-change detection, and SemVer version tracking. This story either adopts that same strategy for the pipeline schema or explicitly documents where it diverges.

**Key coordination**: APIP-5001's STORY-SEED.md explicitly states it owns the provisional runner and APIP-5007 will formalize or supersede it. This story starts by reading what APIP-5001 actually built, then formalizes or confirms that approach.

**Scope boundary**: BullMQ work queue uses Redis (ADR-001 Decision 1) — no PostgreSQL queue table. APIP-5007 database scope is strictly the LangGraph checkpoint store and future pipeline operational tables (schema_versions as of this story).

## Goal

Define and implement the authoritative migration strategy for the autonomous pipeline database by:
1. Selecting the migration toolchain (Drizzle-kit for pipeline-specific tables + LangGraph `setup()` for checkpoint tables, or plain SQL runner)
2. Establishing the `schema_versions` metadata table for dual-tracking (mirrors production app pattern)
3. Formalizing the APIP-5001 provisional runner into the canonical approach
4. Implementing CI validation gates for pipeline migration PRs
5. Providing `pnpm` commands for the full migration lifecycle

## Non-Goals

- Modifying the BullMQ/Redis work queue — that is APIP-0010 (no PostgreSQL queue table)
- Implementing change telemetry tables (APIP-3010), model affinity tables (APIP-3020), or cron infrastructure tables (APIP-3090) — this story establishes the framework those stories will use
- Modifying the production app schema in `packages/backend/database-schema/` — **protected**
- Modifying `@repo/db` client package — **protected**
- Implementing data retention or archival policies for the checkpoint store
- Implementing a Playwright E2E test for schema migration — no UI surface (ADR-006 does not apply)
- Defining the Operator CLI schema status command — that is APIP-5005
- Supporting multi-tenant or multi-database migration orchestration — single pipeline database in Phase 0
- Adding pipeline tables to `packages/backend/database-schema/src/schema/` — WINT-0080 is in-progress there and the package is protected

## Architecture Notes

**Toolchain Decision** (from seed analysis):
- **Drizzle-kit** for pipeline-specific operational tables that need ORM queries (schema_versions, future Phase 3 telemetry). Drizzle-kit generates SQL from TypeScript schema diffs, maintains `_journal.json`, supports `db:status`.
- **LangGraph `setup()` or its SQL** for LangGraph checkpoint tables specifically. `@langchain/langgraph-checkpoint-postgres` may ship a built-in `setup()` method — verify before writing custom SQL. If `setup()` exists, wrap it in the migration runner and record a `schema_versions` entry for the checkpoint schema version.
- **Boundary**: Checkpoint tables are LangGraph-owned; pipeline operational tables (schema_versions, telemetry) are Drizzle-owned. Document this boundary in MIGRATION-STRATEGY.md.

**Schema Isolation**:
- All pipeline tables in `apps/api/pipeline/src/db/` — never in `packages/backend/database-schema/`
- Pipeline DB client uses an isolated `Pool` (separate from `@repo/db`) with `PIPELINE_DB_*` env vars — established by APIP-5001
- If Drizzle is used, pipeline schema uses its own `pgSchema` namespace or operates in the public schema of the dedicated pipeline database (not shared with production app schema)

**Dual Tracking**:
- Every migration applied by the runner inserts a `schema_versions` record (per SCHEMA-VERSIONING.md pattern)
- For Drizzle migrations: Drizzle journal (`meta/_journal.json`) + `schema_versions` table
- Rollback classification must be present for every migration: `rollback_safe: true/false`, `rollback_sql` where applicable

**Naming Convention**:
- APIP-5001 may have used 3-digit naming (001_) — rename to 4-digit (0001_) as part of this story
- All future pipeline migrations: `XXXX_description.sql` (4-digit, zero-padded, lowercase)
