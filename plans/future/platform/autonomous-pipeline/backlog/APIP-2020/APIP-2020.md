# APIP-2020: Monitor UI v1 — Read-Only Pipeline Dashboard

## Context

The autonomous pipeline (APIP) processes stories through a multi-phase workflow. State is stored in `wint.stories`, token consumption tracked in `wint.token_usage`, and blocked stories are managed by APIP-2010. Currently, operators must query the database directly or use the CLI (APIP-5005) to observe pipeline state.

Operators need real-time visibility into three areas without direct DB access:
1. Current state of the work queue (queued, running, blocked)
2. Per-story token and cost totals to detect runaway spend (RISK-009)
3. Which stories are blocked and why

Without a visual interface, routine monitoring requires technical DB access and is error-prone.

**Dependencies**:
- APIP-2010 (Blocked Queue and Notification System) must be merged before blocked queue data is meaningful in UAT
- APIP-5005 (Minimal Operator Visibility CLI) shares read queries — coordinate shared `PipelineReadService` at hand-off

---

## Goal

Build a read-only web dashboard page hosted in `apps/web/app-dashboard` that gives operators real-time visibility into pipeline state, per-story costs, and blocked stories through three data panels. All data is read-only; no write operations are exposed.

---

## Non-Goals

- No write operations from the dashboard (no retry buttons, no unblock actions, no state transitions) — operator interventions belong in CLI (APIP-5005) or direct DB for v1
- No authentication UI — auth handled at infrastructure/proxy layer or existing session; do not build a new auth flow
- No mobile-optimized layout — operator tooling is desktop-first; responsive polish deferred
- No historical trend charts — per-story cost is current state only; time-series analytics belong in Phase 3 (APIP-3010)
- Do NOT modify `wint.*` schema tables or add new columns
- Do NOT implement APIP-5005 CLI scope — this story is UI only

## Architecture Notes

### Backend: Hono Monitor Domain

Follow the `health` domain pattern exactly:
- `routes.ts` — thin Hono route definitions, delegate to service
- `application/services.ts` — business logic, calls `PipelineReadService`
- `adapters/repositories.ts` — Drizzle read queries against `wint.*` tables

Register at `/monitor` in `apps/api/lego-api/src/app.ts`.

### Shared PipelineReadService

Extract into `packages/backend/orchestrator/src/services/pipeline-read-service.ts`:

```
getPipelineView()   → all stories with state, sorted by state priority
getCostSummary()    → SUM(total_tokens) GROUP BY story_id, JOIN wint.stories for VARCHAR story_id
getBlockedQueue()   → stories WHERE state = 'blocked', include blocked_by, updated_at
```

Both APIP-5005 (CLI) and APIP-2020 (dashboard) must consume this service — never duplicate SQL.

### Token Aggregation Join

`wint.token_usage.story_id` is a UUID FK to `wint.stories.id`. To return human-readable story IDs in the API response, join through `wint.stories`:

```sql
SELECT s.story_id, SUM(tu.total_tokens), tu.phase, tu.model
FROM wint.token_usage tu
JOIN wint.stories s ON tu.story_id = s.id
GROUP BY s.story_id, tu.phase, tu.model
```

The existing `token_usage_story_id_idx` index covers this join. Validate with EXPLAIN before merging (AC-9).

### SSE and Lambda Constraint

API Gateway (Lambda-backed) has a hard 29-second connection timeout. SSE requires a persistent connection. **Resolution**:
- v1 primary mode: polling (`usePipelineMonitor` hook, 30s interval, configurable)
- SSE (`GET /monitor/pipeline/stream`) is implemented but only fully usable when the monitor endpoint is hosted outside Lambda (e.g., on the APIP-0030 LangGraph server or a dedicated process)
- Frontend: EventSource with exponential backoff (3 retries), fallback to polling if SSE repeatedly fails

### Response Shape (Zod schema)

```typescript
const PipelineStorySchema = z.object({
  story_id: z.string(),
  title: z.string(),
  state: z.string(),
  updated_at: z.string().datetime(),
})

const CostSummaryEntrySchema = z.object({
  story_id: z.string(),
  phase: z.string(),
  total_tokens: z.number().int(),
  model: z.string().nullable(),
})

const BlockedStorySchema = z.object({
  story_id: z.string(),
  title: z.string(),
  blocked_by: z.string().nullable(),
  updated_at: z.string().datetime(),
})

const PipelineDashboardResponseSchema = z.object({
  pipeline_view: z.array(PipelineStorySchema),
  cost_summary: z.array(CostSummaryEntrySchema),
  blocked_queue: z.array(BlockedStorySchema),
  generated_at: z.string().datetime(),
})
```
