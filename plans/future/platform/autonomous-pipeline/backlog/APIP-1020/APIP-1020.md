# APIP-1020: ChangeSpec Schema Design and Validation Spike

## Context

The autonomous pipeline's implementation loop (APIP-1030), review graph (APIP-1050), QA graph (APIP-1060), change telemetry (APIP-3010), model affinity profiles (APIP-3020), and merge graph (APIP-1070) all consume a shared atomic change contract called a **ChangeSpec**. This schema is the most irreversible Phase 1 decision: once integration code is written against a schema shape, changing field names or types requires cascading updates across 6+ downstream systems.

No ChangeSpec schema exists in the codebase today. The existing `FileChangeSchema` and `PlanStepSchema` in `packages/backend/orchestrator/src/artifacts/plan.ts` provide field-level inspiration but were designed for a human-facing plan document, not a machine-executable atomic change contract. Building Phase 1 integration code before the schema is validated on real stories is RISK-002 (high severity, affecting APIP-1020, APIP-1030, and APIP-3030).

This spike bypasses integration code entirely. It delivers a validated schema, 10+ story decompositions, quality metrics, and a published ADR that either approves the schema or requires revision — gating all downstream integration work (APIP-1030b).

**Dependency Note (APIP-1010)**: APIP-1020 depends on APIP-1010's `ChangeOutlineItem` format as the primary structured input for decomposition. The spike may proceed with a documented placeholder format if APIP-1010 is still in progress, but ADR-002 must be re-validated against the final APIP-1010 output before it is published.

**Critical Path**: This story is on the 12-story critical path. ADR-002 publication directly unblocks APIP-1030b. Spike execution speed is critical.

## Goal

Design and validate a `ChangeSpecSchema` — the machine-executable atomic change contract for the autonomous pipeline — by:
1. Defining the schema in TypeScript/Zod with versioning and exhaustive type-safety.
2. Validating it empirically against 10+ real APIP backlog stories through manual decomposition.
3. Confirming all 6 downstream consuming systems have the fields they need.
4. Publishing ADR-002 with a clear verdict and cost model.

## Non-Goals

- Writing the DiffPlannerNode LangGraph node (that is APIP-1030 scope).
- Connecting ChangeSpec output to APIP-1030 (Implementation Graph with Atomic Change Loop).
- Implementing model routing for the diff planner step.
- Building any API, Lambda handler, or HTTP endpoint.
- Modifying any production database schemas.
- Modifying any existing orchestrator graph code.
- Evaluating cheap-model quality for decomposition (APIP-0040 / RISK-001 scope — this spike measures cost only).
- Defining the ChangeOutlineItem schema (APIP-1010 scope).
- Operator CLI visibility (APIP-5005 scope).
- Any UI/dashboard changes (APIP-2020 scope).

## Architecture Notes

### ChangeSpec Schema Design

The schema must serve 6 consuming systems. Required fields per system:

| Consuming System | Required Fields |
|-----------------|-----------------|
| Implementation loop | `file_path`, `change_type`, `description`, `acceptance_criteria_ref`, `estimated_tokens` |
| Review graph | `change_type`, `risk_level`, `affected_patterns` |
| QA graph | `testability_hint`, `ac_refs`, `test_strategy` |
| Telemetry | `change_type`, `model_used`, `token_cost`, `duration_ms` |
| Model affinity | `change_type`, `file_extension`, `complexity_estimate` |
| Merge graph | `commit_message_hint`, `pr_body_hint`, `related_acs` |

### Discriminated Union Variants

```typescript
z.discriminatedUnion('change_type', [
  FileChangeSpecSchema,      // file_change — create/modify/delete a source file
  MigrationChangeSpecSchema, // migration_change — database migration file
  ConfigChangeSpecSchema,    // config_change — configuration files
  TestChangeSpecSchema,      // test_change — test file additions/modifications
])
```

Exhaustive coverage of all 4 variants must be confirmed by 10+ real-world decompositions before ADR-002 is published.

### Schema Versioning Strategy

Follow `PlanSchema` pattern:
```typescript
export const ChangeSpecSchema = z.object({
  schema: z.literal(1),  // version pin — bump to z.literal(2) on breaking changes
  change_type: z.enum(['file_change', 'migration_change', 'config_change', 'test_change']),
  // ... shared fields
}).and(z.discriminatedUnion('change_type', [...variants]))
```

Process for adding fields without breaking consumers: add as `z.optional()` in schema v1; document in ADR-002; only make required in schema v2 with migration plan.
