# APIP-5007: Database Schema Versioning and Migration Strategy for APIP Pipeline

## Context

The APIP autonomous pipeline requires a reliable, documented approach to schema evolution before Phase 0 coding begins. This story satisfies PLAT-002 ("Checkpoint schema docs required before Phase 0 completion") from the epic elaboration DECISIONS.yaml.

**Current state**: No dedicated migration tooling or schema namespace exists for the APIP pipeline. The codebase has two established patterns: (1) Drizzle Kit auto-generated migrations for the main app schema (`packages/backend/database-schema/`), and (2) manually-applied sequential numbered SQL files for the Knowledge Base (`apps/api/knowledge-base/src/db/migrations/`). A past incident (KB migration 014) demonstrates that mixing tooling approaches across the same tables causes schema drift requiring costly cleanup migrations — 14 orphaned tables, 6 orphaned functions, and 8 orphaned enum types required a dedicated cleanup pass.

**Scope clarification from ADR-001**: BullMQ/Redis owns queue state — `wint.work_queue` PostgreSQL table is eliminated. This story covers LangGraph worker graph checkpoint storage and pipeline-owned supporting tables only (telemetry stubs, migration version tracking, schema namespace setup). It does not cover the work queue.

**Complicating factor**: LangGraph's `@langchain/langgraph-checkpoint-postgres` package ships its own checkpoint table migration that creates `checkpoints`, `checkpoint_blobs`, `checkpoint_writes`, and `checkpoint_migrations` tables. If this story hand-rolls checkpoint DDL AND LangGraph also creates these tables on first run, there will be conflicts during APIP-0020 implementation. This decision must be resolved explicitly.

**Architecture constraint**: ADR-001 Decision 4 establishes that all pipeline components run on a dedicated local server (Docker Compose), not AWS Lambda. Migration tooling must work in the local Docker Compose environment without requiring Lambda or CI infrastructure.

## Goal

Establish a single, documented migration strategy for pipeline-owned tables before Phase 0 implementation begins. Deliver: an ADR covering both migration tooling choice and LangGraph checkpoint ownership, the baseline `apip` schema DDL, thread ID convention documentation, rollback procedures, and a local development runbook. These artifacts unblock APIP-0020 (Supervisor Graph) and APIP-5001 (Test Database Setup).

## Non-Goals

- Implementing any APIP pipeline business logic or worker graph nodes
- Modifying production schemas in `packages/backend/database-schema/` (protected)
- Modifying the `@repo/db` client package API surface (protected)

## Architecture Notes

**Schema namespace decision**: The `apip` pgSchema namespace is recommended over `wint_pipeline` because:
1. It matches the APIP story prefix convention used throughout the epic
2. `wint_pipeline` would be a sub-namespace under `wint` which implies coupling — the pipeline schema is an independent system
3. Confirm no conflict with existing namespaces (`wint`, `public`, `umami`) via `SELECT schema_name FROM information_schema.schemata`

**LangGraph checkpoint delegation rationale** (recommended approach, confirmed in AC-2 spike):
- LangGraph's `@langchain/langgraph-checkpoint-postgres` creates: `checkpoints`, `checkpoint_blobs`, `checkpoint_writes`, `checkpoint_migrations` tables
- These tables have an internally-managed schema that may change between LangGraph versions
- Hand-rolling these tables creates a maintenance burden and version coupling
- Recommended: let LangGraph own its checkpoint tables via its built-in migration; APIP-5007 owns only the `apip` schema namespace tables
- LangGraph tables will reside in the `public` schema (LangGraph's default) or a configurable namespace — document in ADR-002

**Migration tooling tradeoff**:
- Drizzle Kit: powerful, TypeScript-native, auto-generates diffs, but overhead for a small/slow-evolving schema in a local-only Docker environment
- Sequential SQL scripts (KB pattern): simpler, no tooling dependency, directly portable to any psql client, matches the established KB pattern; requires a lightweight Node.js runner or Docker entrypoint script to apply automatically
- Recommended: sequential SQL scripts (KB pattern) for the `apip` schema — the schema is expected to have few migrations, Docker Compose environment does not benefit from Drizzle's diff generation, and consistency with the KB pattern reduces cognitive overhead for the team

**Connection pool budget** (from seed feasibility guidance):
- LangGraph checkpointer: maintains its own PostgreSQL connection pool (default: varies by configuration)
- BullMQ/Redis: zero PostgreSQL connections
- Pipeline admin/migration connection: one ephemeral connection during migration application
- Total expected: LangGraph pool (2-5 connections) + 0 (BullMQ) + 1 (migration) = 3-6 connections
- Aurora `max_connections` budget: well within limits given the existing `wint` and main app connection pools
