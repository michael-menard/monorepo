# APIP-3010: Change Telemetry Table and Instrumentation

## Context

The autonomous pipeline's learning system (Phase 3) depends on granular per-ChangeSpec-attempt telemetry to mine model affinity patterns and optimize model routing. Without a telemetry record, the system cannot determine which model performs best for which type of file change.

The `wint` schema namespace is established and proven in `packages/backend/database-schema/src/migrations/app/` (worktrees table, stories table, workflow artifacts). The Drizzle ORM + `drizzle-zod` auto-generation pattern is the canonical DB access pattern. APIP-5007 (in-progress) is establishing the migration runner and `apip.schema_migrations` versioning table that governs all new APIP pipeline migrations.

This story has a hard implementation gate: `packages/backend/orchestrator/src/nodes/change-loop.ts` does not exist until APIP-1030 ships. The SQL migration, Drizzle schema, Zod types, and `writeTelemetry()` function can be prepared before APIP-1030 merges. The change-loop instrumentation (AC-7) must wait.

**Dependency chain**: APIP-1010 → APIP-1020 → APIP-1030 → APIP-3010 (AC-7)

## Goal

Deliver two coupled artifacts:

1. A `wint.change_telemetry` Aurora PostgreSQL table (via APIP-5007-registered Drizzle migration) capturing every ChangeSpec attempt — recording `change_type`, `file_type`, `model`, `outcome`, `attempts`, `escalated_to`, `tokens`, `cost`, and `duration_ms`.

2. Instrumentation of the `change-loop` node (built by APIP-1030) to write one telemetry row per ChangeSpec attempt outcome (pass, fail, abort, budget_exhausted) using a fire-and-continue pattern that never blocks or aborts the loop.

## Non-Goals

- Implementing the Pattern Miner or reading telemetry for affinity analysis (APIP-3020)
- Retention/pruning policy for the telemetry table (deferred; documented as follow-up)
- Exposing telemetry via HTTP API or operator UI (APIP-2020, APIP-5005)
- Aggregating telemetry into affinity scores (APIP-3020, APIP-3040)
- Storing full code-gen output or diff content (raw code is in git; telemetry captures outcome metrics only)
- Implementing the change loop itself (APIP-1030 — prerequisite)
- Implementing the ChangeSpec schema (APIP-1020 — prerequisite for enum finalization)
- Modifying `packages/backend/orchestrator/src/models/` (protected)

## Architecture Notes

### Fire-and-Continue Pattern
The `writeTelemetry()` call in the change loop must not propagate failures to the graph's happy path. Default implementation awaits the write for correctness (row committed before advancing). The function catches all errors and calls `logger.warn` — never rethrows. If profiling shows unacceptable latency from the DB write, switch to fire-and-forget (`Promise.resolve(writeTelemetry(...)).catch(err => logger.warn(...))`), but document the tradeoff (risk of row loss on crash).

### DB Injection Pattern
The `db` instance must be injected into the change-loop node via graph config (not imported directly). This follows the `IModelDispatch` injection pattern established in APIP-1030 and decouples unit tests from the real database. The node receives `db` from the graph's `config.configurable` at runtime.

### Zod-First Types
`ChangeTelemetrySchema` must be a `z.object()` schema. The `type ChangeTelemetry = z.infer<typeof ChangeTelemetrySchema>` alias is the only TypeScript type used. No `interface` declarations anywhere in the telemetry module.

### Index Design for Downstream Queries
Two indexes are mandatory at migration time (retrofitting post-scale is risky on Aurora):
- `idx_change_telemetry_story_id` on `(story_id, created_at DESC)` — serves APIP-3020 Pattern Miner's per-story time-series queries
- `idx_change_telemetry_affinity` on `(model, change_type, outcome)` — serves affinity mining aggregations

### ChangeSpec Enum Placeholder
`change_type` and `file_type` columns use `'unknown'` as the initial placeholder enum value. The full enum taxonomy is pending the APIP-1020 ChangeSpec ADR. A follow-up task must update the migration and Zod schema once the ADR is published. Do not block this story on APIP-1020.

## Infrastructure Notes

- **Migration registration**: The SQL migration must be registered through the migration runner and `apip.schema_migrations` versioning table established by APIP-5007. Do not create a parallel migration mechanism.
- **Database**: Aurora PostgreSQL (local dedicated server per APIP ADR-001 Decision 4). Direct DB connection from the change loop process — no Lambda intermediary.
- **High write volume**: At scale, every ChangeSpec attempt generates one row. The indexes defined in AC-4 must be present from day one.
