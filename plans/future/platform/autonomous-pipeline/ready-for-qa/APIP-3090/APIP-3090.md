# APIP-3090: Cron Job Infrastructure

## Context

As of February 2026, all LangGraph worker graphs in the autonomous pipeline are invoked reactively — triggered story-by-story by the supervisor loop. There is no facility for scheduled, periodic invocations of background graphs. The pipeline requires several recurring maintenance and analytics tasks (affinity profile aggregation, metrics collection, code audits, KB compression, doc sync, worktree pruning) that must run on a predictable schedule without human triggering.

APIP-3090 establishes the cron job infrastructure layer on top of the LangGraph Platform (APIP-0030). This is a Phase 3 story in the Learning System & Optimization phase, and it is a hard dependency for multiple Phase 4 quality stories (APIP-4010, APIP-4040, APIP-4050, APIP-4060).

**Hard dependencies:**
- APIP-0030 (LangGraph Platform Docker Deployment) — runtime that executes cron schedule registrations; must be complete before integration testing
- APIP-3020 (Model Affinity Profiles and Pattern Miner) — delivers `runPatternMiner(config)`, the first and most frequent cron job; must be ready before full Pattern Miner registration

**Non-blocking dependency resolution:** The cron scheduler abstraction layer, registry schema, and individual job definitions can be built and unit-tested without a running LangGraph Platform. A `CronSchedulerAdapter` interface backed by a no-op test double enables full unit test coverage before APIP-0030 completes.

## Goal

Establish the cron job infrastructure layer that:
1. Defines a Zod-validated cron job registry schema (`CronJobDefinitionSchema`, `CronScheduleRegistrySchema`, `CronRunResultSchema`)
2. Provides a `registerCronJobs(adapter, registry)` function backed by a `CronSchedulerAdapter` interface (real LangGraph Platform adapter + test double)
3. Registers the initial 6 scheduled jobs with correct intervals, timeout budgets, and single-instance guards
4. Guarantees single-instance execution for Pattern Miner via PostgreSQL advisory locks
5. Enforces per-job wall-clock deadlines via `withTimeout` for expensive jobs
6. Produces structured `CronRunResult` logs for every job execution

## Non-Goals

- Implementing Pattern Miner graph logic — that is APIP-3020; this story only registers `runPatternMiner` as a cron job
- Implementing code audit graph logic — that is a Phase 4 story; this story registers the stub or placeholder

- Implementing doc sync logic — this story registers the stub

- Adding HTTP endpoints or operator UI for cron management — that is APIP-2020/APIP-5005
- Using AWS EventBridge, CloudWatch Events, or any AWS-managed scheduling — all cron runs on the dedicated server via LangGraph Platform (ADR-001 Decision 4)

- Modifying `@repo/db` client package API surface — protected
- Modifying any production DB schema — no new tables in APIP-3090
- Adding operator visibility dashboard — downstream consumer of CronRunResult logs is APIP-2020

## Architecture Notes

### CronSchedulerAdapter Pattern (Testability Gate)

The cron infrastructure uses a `CronSchedulerAdapter` interface to decouple the registry logic from the LangGraph Platform runtime. This enables full unit testing before APIP-0030 is complete:

```typescript
// adapter.ts (Zod-first — no TypeScript interfaces)
const CronSchedulerAdapterSchema = z.object({
  register: z.function().args(CronJobDefinitionSchema).returns(z.promise(z.void())),
  deregister: z.function().args(z.string()).returns(z.promise(z.void())),
})

// InMemoryCronAdapter — for unit tests
// LangGraphPlatformAdapter — for production (requires APIP-0030)
```

### Advisory Lock Pattern (Pattern Miner)

Pattern Miner requires single-instance execution (APIP-3020 contract). Implementation:

```typescript
// In pattern-miner.job.ts
const lockKey = hashtext('apip_cron_pattern-miner') // deterministic from job name
const acquired = await pg.query('SELECT pg_try_advisory_lock($1)', [lockKey])

if (!acquired.rows[0].pg_try_advisory_lock) {
  logger.info('cron.job.skipped', { jobName: 'pattern-miner', reason: 'lock_held' })
  return CronRunResultSchema.parse({ jobName: 'pattern-miner', status: 'skipped', ... })
}

try {
  await withTimeout(runPatternMiner, config.timeoutMs)(config)
} finally {
  await pg.query('SELECT pg_advisory_unlock($1)', [lockKey])
}
```

### Timeout Enforcement

Use `withTimeout` from `runner/timeout.ts` — NOT `createToolNode` — for expensive jobs:
- `createToolNode` has a 10s default that silently kills long-running jobs
- `withTimeout(runFn, timeoutMs)` wraps any async function with a wall-clock deadline
- On timeout: `CronRunResult { status: 'timeout' }` logged; no error propagated to scheduler

### Environment-Aware Registry

```typescript
// registry.ts
function buildCronRegistry(env: NodeJS.ProcessEnv = process.env): CronJobDefinition[] {
  const allJobs = [patternMinerJob, metricsCollectionJob, codeAuditJob, ...]
  return allJobs.filter(job => {
    const envKey = `DISABLE_CRON_JOB_${job.jobName.toUpperCase().replace(/-/g, '_')}`
    return env[envKey] !== 'true'
  })
}
```

## Infrastructure Notes

- **Runtime**: LangGraph Platform (APIP-0030) — dedicated local server; NOT AWS EventBridge or Lambda
- **Cron API**: LangGraph Platform's native cron scheduling facility — no custom cron daemon needed
