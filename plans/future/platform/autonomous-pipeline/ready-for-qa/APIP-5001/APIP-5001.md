# APIP-5001: Test Database Setup for LangGraph Checkpoint Schema

## Context

The autonomous pipeline uses LangGraph for worker graph execution (elaboration, implementation, review, QA, merge). LangGraph requires a PostgreSQL checkpoint store to persist graph state between nodes and across crash-recovery scenarios.

**Scope correction from original story.yaml**: The original story referenced "work queue schema testing." Per ADR-001 Decision 1, BullMQ + Redis replaces PostgreSQL for queue state. There is no `wint.work_queue` PostgreSQL table. This story's scope is exclusively the **LangGraph checkpoint database** — a separate Aurora-compatible PostgreSQL schema used to persist graph execution state.

Before APIP-0020 (supervisor) and APIP-5000 (graph unit testing) can be tested end-to-end with realistic checkpoint state, the checkpoint database must be:

1. Defined with a schema (tables, indexes) matching the LangGraph checkpoint contract
2. Testable in isolation via a local Docker Compose service on a non-conflicting port
3. Migratable via numbered SQL migration files applied in order with an idempotent runner
4. Instrumented with test helpers (`testConnection()`, fixture seeders) for downstream stories

**Relationship to existing patterns**: The KB postgres setup (`apps/api/knowledge-base/`) provides exact canonical templates for all four deliverables. This story reuses those patterns with minimal adaptation.

**Relationship to APIP-5007**: APIP-5001 establishes a minimal provisional migration runner. APIP-5007 will define the production migration framework and may supersede this runner. Design the runner to be explicitly throwaway.

## Architecture Notes

### Pool Isolation (Required)

The pipeline checkpoint DB client MUST use a standalone `Pool` instance — never importing from or composing with `@repo/db`. Rationale: co-mingling pools risks Aurora `max_connections` exhaustion (each isolated service with `max: 3` keeps the total well within Aurora's connection limit).

### Port Allocation

Taken ports: 5432 (Aurora/main), 5433 (KB postgres), 6379 (Redis), 9090, 3003, 4317, 4318, 9000, 9001.
**Checkpoint test DB: port 5434.**

### @langchain/langgraph-checkpoint-postgres Schema Discovery

Before writing custom SQL, dev must check if `@langchain/langgraph-checkpoint-postgres` ships a `setup()` method that handles schema creation. If available, the migration approach becomes: (1) call `setup()` from the runner, (2) wrap it in idempotency tracking. If not available, write the SQL tables manually (checkpoints, checkpoint_blobs, checkpoint_writes, checkpoint_migrations).

### Provisional Migration Runner

APIP-5001's migration runner is explicitly minimal and throwaway. APIP-5007 owns the production migration strategy decision (Alembic vs. plain SQL vs. Drizzle migrations). Design `scripts/apply-migrations.ts` to be a simple Node script reading `*.sql` files in alphabetical order — not a framework. Document in README.md that APIP-5007 will supersede it.

### Test Isolation Strategy

- `fileParallelism: false` in vitest.config.ts — prevents concurrent DB access across test files
- Integration tests must DELETE/TRUNCATE inserted records in `afterEach` or `afterAll`
- Schema unit tests require no DB connection — run in any environment

## Infrastructure Notes

**Docker Compose service** (`infra/langgraph-server/compose.test-db.yaml`):
```yaml
services:
  pipeline-test-db:
    image: postgres:15
    ports:
      - "5434:5432"
    environment:
      POSTGRES_USER: pipelineuser
      POSTGRES_PASSWORD: TestPassword123!
      POSTGRES_DB: pipeline_test
    volumes:
      - pipeline_test_db_data:/var/lib/postgresql/data
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U pipelineuser -d pipeline_test"]
      interval: 5s
      timeout: 5s
      retries: 5

volumes:
  pipeline_test_db_data:
```
