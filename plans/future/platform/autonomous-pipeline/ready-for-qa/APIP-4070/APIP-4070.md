# APIP-4070: Weekly Pipeline Health Report — Sunday Cron Aggregation and Slack Notification

## Context

As the autonomous pipeline accumulates operational data across APIP-3010 (change telemetry), APIP-3020 (model affinity profiles), APIP-4010 (codebase health snapshots), and APIP-2010 (blocked queue records), no mechanism currently surfaces a weekly digest to operators. Without a consolidated view, cost overruns, degraded model performance, and codebase drift can go unnoticed for days.

**Baseline (2026-02-13):**
- Metrics nodes (`packages/backend/orchestrator/src/nodes/metrics/`) and the `MetricsCollector` / `StoryMetricsSchema` (`packages/backend/orchestrator/src/observability/metrics.ts`) are established — per-story cost estimation (`estimatedCost`) and token tracking is available for weekly aggregation.
- YAML artifact persistence pattern (`packages/backend/orchestrator/src/artifacts/`) is established — weekly report follows this pattern for any future DB persistence.
- Knowledge Base (pgvector) is operational — lessons learned and weekly summaries can optionally be written as KB entries.
- Dependency data sources (APIP-2010, APIP-3010, APIP-3020, APIP-4010) are in backlog/ready-to-work and not yet merged. This story's aggregation logic is designed for parallel development using injectable parameters.

## Goal

Introduce a Sunday cron task (`triggerWeeklyReport()`) that aggregates data from the prior 7-day window across four sections — pipeline throughput, total and per-provider costs, model performance, and codebase health delta — then dispatches a formatted Slack notification to operators. The function must be fully operational as a standalone callable before cron wiring.

## Non-Goals

- Building a UI dashboard for weekly reports (that is APIP-2020)
- Implementing email or push notification channels (Slack webhook only; additional channels deferred per APIP-2010)
- Auto-remediating concerns surfaced in the report (advisory only; no story auto-generation)
- Persisting the weekly report summary to a `wint.weekly_reports` DB table (structured log emission is sufficient for MVP; DB persistence is a future enhancement)
- Sending reports for time windows with zero completed stories — graceful skip is sufficient (AC-10)
- Modifying `packages/backend/database-schema/` protected tables or any existing APIP-2010/APIP-3020/APIP-4010 schemas
- Supporting weekly report email delivery (future work)

## Architecture Notes

### Injectable Parameter Design (Critical)

All aggregation functions MUST accept injectable `db` and `window` parameters rather than importing singletons. This enables unit testing without a real database and follows the pattern established in `gap-analytics.ts`.

```typescript
// CORRECT — injectable
async function aggregateThroughput(
  db: DbClient,
  window: { start: Date; end: Date }
): Promise<ThroughputResult | DataUnavailableResult>

// WRONG — singleton import
async function aggregateThroughput(): Promise<ThroughputResult>
```

### Data Unavailable Handling

When any dependency table does not exist (PostgreSQL error code `42P01`) or any query fails, the aggregation function MUST return `{ data_unavailable: true }` rather than throw. `triggerWeeklyReport()` assembles the summary with unavailable sections marked, formats them as "_Data unavailable — dependency story not yet merged_" in the Slack message, and dispatches the partial report (unless ALL sections are unavailable — in that case, emit `weekly_report_skipped` and skip dispatch).

### Empty Window Handling

If ALL aggregation functions return either empty data or `data_unavailable: true` (net zero stories in the window), `triggerWeeklyReport()` emits `logger.warn({ event: 'weekly_report_skipped', reason: 'no_data' })` and returns without dispatching.

### Trend Analysis Threshold

When fewer than `config.minHistoryWeeks` (default: 2) historical report snapshots exist, trend arrows and delta calculations are suppressed. The report notes "insufficient history for trend analysis" in the body. This prevents meaningless delta comparisons on the pipeline's first week of operation.

### Concurrency Guard

`pg_try_advisory_lock` must be acquired at the start of `triggerWeeklyReport()` to prevent duplicate runs if cron fires while a previous invocation is still running (APIP-3020/APIP-4010 established pattern). If lock acquisition fails, emit `logger.info({ event: 'weekly_report_skipped', reason: 'lock_held' })` and return.

### Structured Log Contract

Success: `logger.info({ event: 'weekly_report_dispatched', period, stories_completed, stories_blocked, total_cost_usd, top_concern, top_improvement, sections_available, durationMs })`

Skip (no data): `logger.warn({ event: 'weekly_report_skipped', reason: 'no_data' })`

Skip (lock held): `logger.info({ event: 'weekly_report_skipped', reason: 'lock_held' })`

Dispatch failure (swallowed): `logger.error({ event: 'weekly_report_dispatch_failed', error: error.message })` — do NOT redact webhook URL in error object; redact only in success log.
