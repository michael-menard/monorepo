# LNGG-0070: Integration Test Suite - End-to-End Validation

## Context

All six adapter stories (LNGG-0010 through LNGG-0060) have individual unit and integration tests, but there is no comprehensive test suite validating how these adapters work together in realistic workflow scenarios.

**Current State:**
- ✅ StoryFileAdapter (LNGG-0010) - UAT stage with 80%+ unit test coverage
- ✅ IndexAdapter (LNGG-0020) - UAT stage with transaction support and validation
- ✅ DecisionCallbackRegistry (LNGG-0030) - UAT stage with CLI, Auto, and Noop callbacks
- ✅ StageMovementAdapter (LNGG-0040) - UAT stage with atomic directory operations
- ✅ KBWriterAdapter (LNGG-0050) - UAT stage with deferred write support
- ✅ CheckpointAdapter (LNGG-0060) - Ready-for-QA stage with resume capability

**Problem:**
Without integration tests validating adapter combinations:
- Workflow regressions could occur when individual adapters change
- Interaction bugs between adapters may only be caught in production
- No performance baseline for multi-adapter operations
- Quality comparison vs Claude Code baseline cannot be automated

**Impact:**
The orchestrator needs confidence that:
- Multiple adapters can be used in sequence without conflicts
- State transitions (backlog → ready-to-work → in-progress → UAT) work correctly
- Decision callbacks integrate with file operations
- KB writes don't interfere with story/checkpoint operations
- Performance is acceptable when multiple adapters are used together

---

## Goal

Create a comprehensive integration test suite that validates all adapters working together in realistic workflow scenarios, establishes performance benchmarks, and provides automated quality comparison against Claude Code baseline.

**Success Metrics:**
- All 8 acceptance criteria have passing integration tests
- Code coverage >80% across all adapters (via vitest coverage report)
- Performance benchmarks meet or exceed targets (logged to EVIDENCE.yaml)
- Quality comparison documented with concrete findings

---

## Non-Goals

- **Frontend E2E tests** - This story focuses on backend adapter integration only
- **LangGraph workflow execution** - Actual graph execution is out of scope; integration tests simulate workflow patterns
- **Database integration tests** - No database writes; file-based operations only
- **API endpoint testing** - No HTTP endpoints involved
- **Performance optimization** - Establish baseline only, optimization is future work
- **Load testing** - Focus on single-operation performance, not concurrency
- **New adapter features** - Only test existing adapter functionality
