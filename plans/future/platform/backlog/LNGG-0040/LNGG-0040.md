# LNGG-0040: Stage Movement Adapter

## Summary

Create a type-safe Stage Movement Adapter that updates story status in YAML frontmatter as stories transition between lifecycle stages. This adapter supports the new flat directory structure (WINT-1020) where story status is tracked in frontmatter, not directory location.

**Context:**
LangGraph workflows need to transition stories between lifecycle stages (backlog → elaboration → ready-to-work → in-progress → ready-for-qa → UAT). WINT-1020 fundamentally changed the directory structure from lifecycle-based (backlog/, in-progress/, UAT/) to flat (all stories at epic level) with status tracked in YAML frontmatter.

**Problem:**
Without a stage movement adapter, LangGraph workflows cannot:
- Transition stories between stages in an automated workflow
- Update story status in frontmatter as stories progress
- Validate stage transitions (prevent invalid moves like UAT → backlog)
- Maintain consistency between story location and status field

**Solution:**
Create a Stage Movement Adapter that:
1. Uses StoryFileAdapter (LNGG-0010) for reading/writing story files
2. Updates story status in YAML frontmatter (flat structure approach)
3. Validates stage transitions using a defined DAG
4. Handles errors gracefully (story not found, invalid stage, permission errors)
5. Supports batch operations for parallel processing
6. Works cross-platform (macOS, Linux, Windows)

## Stage Transition Rules

### Valid Forward Transitions
- backlog → elaboration
- backlog → ready-to-work (skip elaboration for simple stories)
- elaboration → ready-to-work
- ready-to-work → in-progress
- in-progress → ready-for-qa
- ready-for-qa → UAT

### Valid Backward Transitions (Rework)
- UAT → in-progress (QA failure requiring fixes)
- Any stage → backlog (deprioritize story)

### Lateral Transitions
- elaboration ↔ backlog (refinement loop)

### Blocked Transitions
- Any forward skip >1 stage (except backlog → ready-to-work)
- UAT → ready-for-qa (must return to in-progress for fixes)

## Technical Approach

### Architecture Decision: Flat Structure with Status Updates (Option B)

**Decision**: The adapter will update the `status` field in YAML frontmatter only. It will NOT move story directories.

**Rationale**:
- WINT-1020 (Story Directory Flattening) is complete and deployed
- All stories are now in flat epic-level directories
- Status is the single source of truth in frontmatter
- Future-proof approach aligned with LangGraph architecture
- Simpler implementation with fewer failure modes

### Components to Reuse

**MUST Reuse:**
1. **StoryFileAdapter** (LNGG-0010) - For reading/writing story YAML files with `update()` method
2. **PathResolver** - For stage directory name resolution and path construction
3. **StoryArtifactSchema** - Already includes `status` field for frontmatter updates
4. **Zod validation patterns** - Established in all adapter schemas

### New Components to Create

**Primary Adapter**:
- `packages/backend/orchestrator/src/adapters/stage-movement-adapter.ts`
  - Main class: `StageMovementAdapter`
  - Methods: `moveStage()`, `batchMoveStage()`, `findStory()`

**Type Definitions** (Zod-first):
- `packages/backend/orchestrator/src/adapters/__types__/stage-types.ts`
  - `StageSchema` - Valid stage names enum
  - `StageTransitionSchema` - Transition graph definition
  - `MoveStageRequestSchema` - Move operation request
  - `MoveStageResultSchema` - Move operation result
  - `BatchMoveStageRequestSchema` - Batch operation request

**Utilities**:
- `packages/backend/orchestrator/src/adapters/utils/stage-validator.ts`
  - `validateTransition()` - Check if transition is valid using DAG
  - `isValidStage()` - Verify stage name
  - `getStageGraph()` - Return stage transition DAG

### Error Handling

Custom error classes with context:
- `StoryNotFoundError` - Story doesn't exist in epic directory
- `InvalidStageError` - Target stage is invalid
- `InvalidTransitionError` - Transition not allowed by DAG
- `StageConflictError` - Story already in target stage
- `FileOperationError` - Permission denied or filesystem error

### Performance Targets

**Single Move**: <100ms
**Batch of 10**: <2s
**Batch of 50**: <5s
**Concurrency Limit**: 10 parallel file operations

## Risks and Mitigations

### MVP-Critical Risks

**Risk 1: StoryFileAdapter API Contract Unclear**
- **Impact**: Implementation blocked if API doesn't support status updates
- **Mitigation**: Review LNGG-0010 implementation to verify `update()` method signature
- **Status**: Low risk - LNGG-0010 is complete with verified API

**Risk 2: Atomic Operation Guarantee**
- **Impact**: Concurrent moves could corrupt story state
- **Mitigation**: Verify StoryFileAdapter uses atomic file operations (temp file + rename)
- **Status**: Low risk - LNGG-0010 designed for atomic operations

**Risk 3: Performance on Large Batches**
- **Impact**: Batch operations on 100+ stories may exceed targets
- **Mitigation**: Implement parallel processing with concurrency limit (10), add performance tests
- **Status**: Medium risk - mitigated by concurrency controls and benchmarks

### Future Risks (Non-MVP)

See `_pm/FUTURE-RISKS.md` for detailed analysis:
- No rollback mechanism for batch failures
- No stage movement history tracking (until WINT-0070)
- No dry-run mode for batch previews
- No custom workflow support per epic
- No notification system for stage changes
