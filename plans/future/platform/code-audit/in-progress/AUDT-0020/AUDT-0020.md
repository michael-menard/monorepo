# AUDT-0020: Polish and Complete Unit Test Coverage for All 9 Audit Lens Nodes

## Context

AUDT-0010 (UAT) delivered the audit graph scaffold ŌĆö LangGraph StateGraph, artifact schemas, scan-scope node, and the nine lens implementations (`lens-security.ts` through `lens-code-quality.ts`). All 9 lens nodes were scaffolded as part of that foundation and all 9 have corresponding test files in `packages/backend/orchestrator/src/nodes/audit/__tests__/`.

The existing test files were written during the scaffold phase. They have good happy-path and basic positive/negative coverage, but they need systematic review against the acceptance criteria: minimum 3 known-positive fixtures and 3 known-negative fixtures per lens, `LensResultSchema` compliance on every test, `by_severity` consistency assertions, severity calibration tests (production path vs test file path), and consistent edge-case coverage for empty files, empty `targetFiles` arrays, non-existent paths, and binary file content.

**Reality correction**: The story index describes this as "Add unit tests with fixtures" ŌĆö in reality, all 9 test files already exist. The actual scope is polishing and gap-filling the existing tests to meet the acceptance criteria consistently across all 9 lenses.

**Dependency**: AUDT-0010 is confirmed UAT. No other active stories touch `nodes/audit/` or lens files. AUDT-0030 (Audit Orchestration Nodes) is blocked on this story.

## Architecture Notes

- All 9 lens `run()` functions accept `CodeAuditState` (from `graphs/code-audit.ts`) and return `Promise<LensResult>`.
- `LensResultSchema` is the compliance target: `{ lens, total_findings, by_severity: { critical, high, medium, low }, findings: AuditFinding[] }`.
- `AuditFindingSchema` required fields: `id`, `lens`, `severity`, `confidence`, `title`, `file`, `evidence`, `remediation`, `status`. Optional: `description`, `lines`.
- `AuditLensSchema` enum values: `'security' | 'duplication' | 'react' | 'typescript' | 'a11y' | 'ui-ux' | 'performance' | 'test-coverage' | 'code-quality'`. Note: accessibility lens returns `'a11y'`, not `'accessibility'`.
- `calibrateSeverity()` in `lens-typescript.ts` is the only lens with explicit test-path severity downgrade. Other lenses scope by file extension or directory prefix, not by severity calibration.
- Binary file behavior: `readFile(path, 'utf-8')` on binary content returns garbled text without throwing. Regex patterns are unlikely to match binary noise. No implementation guard is required at this scope.
- Huge file behavior: No file-size gate is implemented. For the current monorepo (largest source files <300KB), this is acceptable. Defer file-size gating to AUDT-0030 or a dedicated tech-debt story.
- Test file exclusion inconsistency: `lens-typescript.ts` checks `/test/` as a path segment in addition to `__tests__`, `.test.`, `.spec.` ŌĆö other lenses only check the latter three. This divergence is intentional and documented; do not normalize.
