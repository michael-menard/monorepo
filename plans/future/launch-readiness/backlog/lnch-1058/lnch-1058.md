# lnch-1058: reset-password App Operational Runbook

## Context

Phase 3 (Launch readiness).

`apps/web/reset-password/` is a standalone web application (not a Module Federation remote) that handles the Cognito-driven password reset flow. Users arrive here via a one-time link emailed by Cognito; the app collects their new password and submits it to Cognito. Because this app is accessed by email link rather than navigated to within the shell, it must be independently deployable and must work regardless of the shell's state.

The app is built with React 19 and deployed to its own CloudFront distribution and S3 bucket. Its failure modes are distinct from the gallery micro-frontends: the app must handle expired or already-used reset tokens gracefully, and any deployment error here directly prevents users from recovering access to their accounts — making it a high-stakes surface to get right.

## Goal

A verified runbook at `docs/operations/runbooks/reset-password.md` covering deployment, rollback, health checks, environment variable inventory, and the failure modes specific to the Cognito password reset flow — including token expiry edge cases.

## Non-Goals

- Cognito password policy and reset email template configuration (auth/identity runbook).
- Email delivery pipeline (SES configuration runbook).
- CloudFront distribution setup (infra runbook).
- Module Federation shell (this app is standalone — no MF dependency).

## Scope

**Output file:** `docs/operations/runbooks/reset-password.md`

**Runbook sections to write:**

### 1. App Overview
- Purpose: standalone password reset form; accepts Cognito `code` + `username` URL params, submits new password to Cognito
- Architecture: standalone React SPA (no Module Federation); served from its own CloudFront distribution
- CloudFront distribution ID and S3 bucket per environment
- Upstream dependencies: Cognito User Pool (confirmForgotPassword API); no backend API calls beyond Cognito SDK
- Entry point: linked from Cognito-sent email — URL format: `https://reset.example.com?code={code}&username={username}`

### 2. Environment Variables
| Variable | Purpose | Example |
|---|---|---|
| `VITE_COGNITO_USER_POOL_ID` | Cognito pool for password reset | `us-east-1_XXXXXX` |
| `VITE_COGNITO_CLIENT_ID` | Cognito app client (must match email template config) | `abc123` |
| `VITE_COGNITO_REGION` | AWS region of the Cognito pool | `us-east-1` |
| `VITE_MAIN_APP_URL` | Redirect target after successful reset | `https://app.example.com/login` |
| `VITE_ENVIRONMENT` | Environment tag | `prod` |

Note: this app has no `VITE_API_BASE_URL` — it communicates directly with Cognito via the AWS SDK, not through the platform API.

### 3. Deployment Procedure
1. Pre-deploy checklist:
   - Confirm Cognito User Pool ID and Client ID are correct in SSM for this environment
   - Confirm `VITE_MAIN_APP_URL` points to the correct post-reset redirect target
   - Confirm Cognito email template's reset link URL matches this app's CloudFront domain
2. Build: `pnpm turbo run build --filter=reset-password`
3. S3 sync — note: since this is a standalone SPA (no MF), all assets go to the same bucket with standard SPA routing config (all paths serve `index.html`)
4. CloudFront invalidation: `/*` (SPA routing requires full invalidation)
5. Post-deploy smoke test:
   - Navigate directly to the app root — confirm the form renders (no blank screen)
   - Simulate an invalid token visit: `?code=bad&username=test@example.com` — confirm a clear error message, not an uncaught exception
   - Using a staging test account, trigger a real reset flow end-to-end

### 4. Rollback Procedure
1. Restore previous S3 objects via versioning
2. Invalidate CloudFront `/*`
3. Verify form renders and Cognito connectivity works
4. Critical: if a rollback is triggered because the Cognito Client ID was wrong, also update SSM and redeploy — do not leave a broken Client ID in SSM

### 5. Health Check
- URL: `https://{reset-cdn-domain}/` — HTTP 200, page contains the password reset form (check for form element in page source)
- Cognito connectivity: confirm the Cognito SDK can reach the User Pool (no CORS or network error in browser console on page load)
- Token validation: confirm that an expired token shows a user-facing error, not a raw JS exception

### 6. Common Errors and Remediation

| Error | Likely Cause | Remediation |
|---|---|---|
| "Invalid verification code" shown immediately on page load | URL params (`code`/`username`) missing or malformed | Check that Cognito email template URL format matches what the app expects |
| "Code expired" even for fresh reset requests | Cognito reset code TTL is very short (default 1 hour); check if emails are delayed | Check SES delivery latency; advise users to use reset link within 1 hour |
| Form submits but nothing happens | Cognito Client ID mismatch | Verify `VITE_COGNITO_CLIENT_ID` matches the client configured in the Cognito User Pool |
| CORS error in browser console on Cognito SDK call | Cognito pool in wrong region or incorrect endpoint | Verify `VITE_COGNITO_REGION` and `VITE_COGNITO_USER_POOL_ID` |
| 404 on direct URL visit to `/` or any path | S3 bucket not configured for SPA routing (CloudFront error pages not set to return `index.html`) | Verify CloudFront custom error response: 403/404 -> `/index.html` with 200 status |
| Blank screen after successful reset (redirect fails) | `VITE_MAIN_APP_URL` pointing to wrong domain | Verify env var; update SSM and redeploy |
