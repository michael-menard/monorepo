# lnch-1021: OpenSearch Operations Runbook

## Context

Phase 3 (Before general availability).

Amazon OpenSearch powers search functionality for the MOC gallery. Search issues (stale index, degraded cluster health, shard failures) affect user-facing search quality. Engineers need to know how to check cluster health, trigger re-indexing, and perform basic operations without deep Elasticsearch/OpenSearch expertise.

## Goal

Write an OpenSearch operations runbook at `docs/operations/runbooks/opensearch-operations.md` covering routine and incident-level OpenSearch operations.

## Non-Goals

- Do not document OpenSearch cluster provisioning.
- Do not document search relevance tuning.
- Do not document Lambda-to-OpenSearch connectivity (lnch-1019 covers VPC).

## Scope

**Output file:** `docs/operations/runbooks/opensearch-operations.md`

**Content outline:**

1. **Overview** — domain name per environment, index naming convention, cluster topology
2. **Checking cluster health** — AWS console path, health API endpoint (`GET /_cluster/health`), green/yellow/red meaning
3. **Index operations**
   - Listing indices and their sizes
   - Triggering a full re-index
   - Checking indexing lag
4. **Shard management** — checking unassigned shards, forced shard allocation
5. **Common issues**
   - Yellow cluster (replica not assigned) — causes, resolution
   - Red cluster (primary shard missing) — escalation trigger
   - High indexing latency — diagnosis and remediation
6. **Snapshot and restore** — automated snapshots, manual snapshot, restore to new domain
7. **Scaling** — instance type change, adding data nodes
8. **Access control** — fine-grained access control, domain access policies
9. **Monitoring** — key CloudWatch metrics for OpenSearch

## Acceptance Criteria

- [ ] Re-indexing procedure is step-by-step with expected duration
