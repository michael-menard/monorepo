# lnch-1010: API Rollback Runbook

## Context

Phase 1 (Critical path — must be complete before launch).

Lambda's versioning and alias system enables fast rollbacks, but the process requires knowing the correct alias names, version identifiers, and verification steps. Without this runbook, an on-call engineer responding to a production incident at 2am has no documented path to rollback, increasing MTTR significantly. This runbook pairs directly with lnch-1009 (Deployment) and lnch-1024 (On-Call Playbook).

## Goal

Write a step-by-step rollback runbook at `docs/operations/runbooks/api-rollback.md` that any on-call engineer can execute under pressure to revert the API to the last known good version.

## Non-Goals

- Do not document forward deployments (lnch-1009).
- Do not document database rollbacks — schema rollbacks are a separate, complex topic.
- Do not document frontend rollback (lnch-1012).

## Scope

**Output file:** `docs/operations/runbooks/api-rollback.md`

**Content outline:**

1. **Overview** — when to use this runbook, rollback mechanism (Lambda aliases/versions or CDK rollback)
2. **Decision criteria** — when to rollback vs hotfix (error rate threshold, latency SLO breach, data integrity issue)
3. **Prerequisites** — AWS access, correct IAM role, identify previous good version
4. **Finding the previous good version** — AWS console path, CLI command to list Lambda versions
5. **Step-by-step rollback**
   a. Update Lambda alias to point to previous version (specific CLI command)
   b. Verify alias update propagated
   c. Run smoke tests (link to smoke test section in lnch-1009)
   d. Monitor error rates for 5 minutes
6. **Database compatibility** — what to check if the rolled-back version has different schema expectations
7. **Communicating the rollback** — incident channel notification template
8. **Post-rollback** — create incident ticket, schedule post-mortem (lnch-1026), root cause investigation
9. **Escalation** — if rollback does not resolve the issue, escalation path (lnch-1028)
