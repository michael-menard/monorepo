# lnch-1053: app-dashboard Operational Runbook

## Context

Phase 3 (Launch readiness).

`apps/web/app-dashboard/` is the internal operations dashboard — it surfaces metrics, content moderation queues, user management, and platform health indicators for internal team use. Unlike the public-facing micro-frontends, the dashboard is gated behind admin-only Cognito groups and does not serve end users. It is deployed as a Module Federation remote, served from its own S3 bucket and CloudFront distribution, and consumed by the shell application.

Because the dashboard controls sensitive internal operations (user bans, content approval, KB writes), an outage or misconfiguration here has a different risk profile than a public gallery going down. This runbook documents how to deploy, roll back, and diagnose issues specific to `app-dashboard` — independent of the shared infra runbook.

## Goal

A verified runbook at `docs/operations/runbooks/app-dashboard.md` covering all operational procedures specific to the internal dashboard micro-frontend: deployment, rollback, health checks, environment variable inventory, and the most common failure modes encountered during and after a deploy.

## Non-Goals

- CloudFront distribution setup and WAF configuration (covered in the infra runbook).
- Cognito user pool management and admin group provisioning (covered in the auth runbook).
- Shared Module Federation shell configuration (covered in the main-app runbook).
- Monitoring dashboards and alerting threshold definitions (covered in lnch-1020).

## Scope

**Output file:** `docs/operations/runbooks/app-dashboard.md`

**Runbook sections to write:**

### 1. App Overview
- Purpose and target audience (internal admins only)
- Architecture: Module Federation remote, consumed by shell at runtime
- CloudFront distribution ID and S3 bucket name (environment-specific)
- Access control: Cognito admin group requirement, how the shell enforces it

### 2. Environment Variables
| Variable | Purpose | Example |
|---|---|---|
| `VITE_API_BASE_URL` | Internal API gateway endpoint | `https://api-internal.example.com` |
| `VITE_COGNITO_USER_POOL_ID` | Cognito pool for admin auth | `us-east-1_XXXXXX` |
| `VITE_COGNITO_CLIENT_ID` | Cognito app client | `abc123` |
| `VITE_MF_REMOTE_URL` | Self-referential MF entry URL | `https://dashboard.cdn.example.com/remoteEntry.js` |
| `VITE_ENVIRONMENT` | Environment tag (`staging`, `prod`) | `prod` |

Document where these are stored (SSM Parameter Store paths), who has write access, and how they are injected at build time.

### 3. Deployment Procedure
1. Pre-deploy checklist (confirm no active moderation sessions, notify team in Slack #ops)
2. Trigger deployment: `pnpm turbo run build --filter=app-dashboard` then CDK deploy command
3. S3 sync command and CloudFront invalidation command
4. Post-deploy smoke test: verify `remoteEntry.js` is reachable, load dashboard in staging, confirm admin-gated routes are accessible with a test admin account
5. Deployment verification checklist

### 4. Rollback Procedure
1. Identify the last known-good S3 object version (versioning is enabled)
2. Restore previous `remoteEntry.js` and asset bundle from S3 version history
3. Issue a targeted CloudFront invalidation for `/*`
4. Verify rollback by loading dashboard and confirming previous build hash in browser dev tools
5. Post-rollback incident note: log in the ops channel with timestamp and reason

### 5. Health Check
- URL: `https://{dashboard-cdn-domain}/remoteEntry.js` — must return HTTP 200 with `Content-Type: application/javascript`
- Shell integration check: load the main app shell as an admin user, navigate to `/dashboard`, confirm the remote loads without federation error in the browser console
- API connectivity: confirm the internal API health endpoint is reachable from the browser (`/health`)

### 6. Common Errors and Remediation

| Error | Likely Cause | Remediation |
|---|---|---|
| `Failed to load remoteEntry.js` in shell console | S3 sync incomplete or CloudFront cache stale | Run CloudFront invalidation; verify S3 object exists |
| Admin routes redirect to 403 | Cognito group claim missing in JWT | Check user is in the `admins` Cognito group; re-login to refresh token |
| Blank dashboard screen with no console errors | Module Federation version mismatch with shell | Redeploy shell and dashboard in the same pipeline run |
| API calls return 401 | Internal API Gateway authorizer rejecting admin token | Verify `VITE_API_BASE_URL` points to internal (not public) APIGW |
| Dashboard loads but data tables are empty | Lambda cold start timeout on first load | Wait 30 s and reload; if persistent, check Lambda concurrency limits |

### 7. Contacts and Escalation
- Primary on-call: platform team Slack `#ops-escalation`
- CDK stack owner for infra changes
- Cognito admin group management: identity team
