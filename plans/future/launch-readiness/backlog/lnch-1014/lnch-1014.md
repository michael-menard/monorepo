# lnch-1014: Database Operations Runbook

## Context

Phase 2 (Before soft launch).

Aurora PostgreSQL is the primary transactional database. Day-to-day operations include connecting to the database, running migrations, checking connection pool health, and monitoring query performance. Without a runbook, routine operations (especially migrations in production) carry unnecessary risk. This is distinct from Aurora infrastructure operations (lnch-1020), which covers cluster-level AWS operations.

## Goal

Write a database operations runbook at `docs/operations/runbooks/database-operations.md` covering routine database tasks that application engineers perform as part of deployments and operations.

## Non-Goals

- Do not document Aurora cluster provisioning or scaling (lnch-1020).
- Do not document database troubleshooting/incident response (lnch-1015).
- Do not document the database schema design.

## Scope

**Output file:** `docs/operations/runbooks/database-operations.md`

**Content outline:**

1. **Overview** — Aurora PostgreSQL cluster info, environments (dev/staging/prod), connection details location
2. **Connecting to the database** — bastion host or AWS Systems Manager Session Manager procedure, psql connection string format
3. **Running migrations**
   - Pre-migration checklist (backup, maintenance window, notify team)
   - Migration command (ORM-specific, e.g., Drizzle/Prisma migrate command)
   - Verifying migration success
   - Rolling back a migration
4. **Checking connection pool health** — metrics to watch, how to see active connections
5. **Slow query investigation** — pg_stat_activity, CloudWatch query insights
6. **Backup and restore** — automated backup schedule, how to initiate a manual snapshot, point-in-time restore procedure
7. **Read replica usage** — when to route reads to replica, connection string differences
8. **Granting/revoking database access** — role management procedure
9. **Escalation** — when to escalate to lnch-1015 (troubleshooting) or lnch-1020 (Aurora infra)

## Acceptance Criteria

- [ ] Runbook exists at `docs/operations/runbooks/database-operations.md`
- [ ] Migration procedure is step-by-step with actual commands
- [ ] Connection procedure works for each environment (dev, staging, prod)
- [ ] Backup and restore steps are verified against actual Aurora configuration
- [ ] Reviewed and approved by the engineer who owns database operations
- [ ] A non-original author performs a test migration on staging using the runbook
