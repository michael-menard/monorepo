# lnch-1042: Loading State Consistency

## Context

Phase 2 (Before soft launch).

Loading states across the app are inconsistent ‚Äî some areas show spinners, some show skeleton screens, some show nothing. Inconsistency creates a jarring user experience and makes the app feel unpolished. This is a quick win (Size S) because a shared skeleton/loading component already exists or can be quickly built, and applying it consistently is mostly a search-and-replace exercise.

## Goal

Audit all loading states in the main app and implement a consistent, skeleton-first loading pattern across all data-fetching components.

## Non-Goals

- Do not implement optimistic UI updates (separate pattern).
- Do not change data fetching strategy or caching.
- Do not implement loading states for the dashboard app.

## Scope

**Output locations:**
- `apps/web/main-app/src/components/Skeleton/` ‚Äî skeleton loading components
- Updates to all components that fetch data asynchronously

**Implementation requirements:**

1. **Skeleton components** ‚Äî `SkeletonText`, `SkeletonCard`, `SkeletonGalleryGrid` ‚Äî shimmer animation, matches layout of the actual content
2. **Loading pattern rule** ‚Äî all data-fetching components use skeleton screens, not spinners, for initial load
3. **Spinner usage** ‚Äî spinners reserved for action-triggered loading (button submit, form save), not page/component load
4. **Loading state inventory** ‚Äî audit all async data areas, categorize current behavior
5. **Apply skeletons** ‚Äî replace all initial-load spinners/blanks with appropriate skeleton variants
6. **Consistent loading duration** ‚Äî if data loads in <200ms, do not flash skeleton (use a minimum display time or delay)
7. **Error state after loading** ‚Äî all skeleton states gracefully transition to error state if loading fails (links to lnch-1040)

## Acceptance Criteria

- [ ] `Skeleton` component family exists (`SkeletonText`, `SkeletonCard`, `SkeletonGalleryGrid` at minimum)
- [ ] All initial data-load states in the main app use skeleton screens
- [ ] No spinner is shown for initial page/component load (only for action-triggered loading)
- [ ] Skeleton components pass axe-core scan (aria-busy or aria-label on loading regions)
- [ ] Loading state inventory is documented
- [ ] Vitest tests cover skeleton components
- [ ] Reviewed by the product designer for visual consistency
