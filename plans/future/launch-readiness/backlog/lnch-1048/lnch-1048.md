# lnch-1048: User Feedback/Support Channel

## Context

Phase 2 (Before soft launch).

Users need a way to report bugs, request features, and get help. Without a feedback mechanism, dissatisfied users will silently churn, and the team will have no structured way to learn from early users. A simple in-app feedback widget and a visible support contact mechanism are essential before soft launch when feedback will be most valuable.

## Goal

Implement an in-app feedback and support mechanism that allows users to submit feedback and access support, integrated into the main app without requiring a full customer support platform.

## Non-Goals

- Do not implement a full customer support ticketing system (scope this to a simple feedback widget).
- Do not build custom feedback analytics — route to an existing tool.
- Do not implement a live chat widget (separate effort if needed).

## Scope

**Output locations:**
- `apps/web/main-app/src/components/FeedbackWidget/` — feedback collection component
- `apps/web/main-app/src/components/SupportLink/` — support contact component
- Integration with chosen feedback destination (e.g., email, GitHub Issues, Canny, or similar)

**Implementation requirements:**

1. **Feedback widget** — accessible via a persistent "Give feedback" button (floating or in footer/navigation)
2. **Feedback form** — type (bug/feature/other), description (required), optional screenshot attachment, email (pre-filled if logged in)
3. **Submission** — route feedback to chosen tool (decision needed: email, GitHub Issues, Canny, etc.)
4. **Confirmation** — success message after submission, expected response time
5. **Support link** — "Get help" link visible in navigation or footer, links to support email or FAQ
6. **Bug report shortcut** — on error pages (lnch-1047) and error states (lnch-1040), "Report a bug" link pre-fills feedback form with context
7. **Accessibility** — fully keyboard navigable, screen reader accessible
8. **Rate limiting** — basic protection against spam submissions

**Note:** The implementer must confirm the feedback destination tool with the product/support team before building. This is a Tier 2 decision (preference).

## Acceptance Criteria

- [ ] Feedback widget is accessible from every page in the app
- [ ] Feedback form submits successfully and routes to the chosen tool
- [ ] User receives confirmation message after submitting feedback
- [ ] "Report a bug" link on error pages pre-fills the feedback form
- [ ] Widget is keyboard navigable and passes axe-core scan
- [ ] Rate limiting prevents spam submissions
- [ ] Vitest tests cover feedback form validation and submission
- [ ] Reviewed by the product manager for feedback routing and copy
