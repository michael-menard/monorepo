# lnch-1015: Database Troubleshooting Runbook

## Context

Phase 2 (Before soft launch).

Database issues are among the most high-impact incidents. Connection exhaustion, lock contention, replication lag, and storage capacity issues can all cause service degradation or outages. On-call engineers need a clear troubleshooting guide that helps them diagnose and resolve database problems quickly, even without deep PostgreSQL expertise. This complements the database operations runbook (lnch-1014) and the Aurora infrastructure runbook (lnch-1020).

## Goal

Write a database troubleshooting runbook at `docs/operations/runbooks/database-troubleshooting.md` covering incident-level database problems with systematic diagnosis procedures.

## Non-Goals

- Do not document routine operations (lnch-1014).
- Do not document Aurora cluster-level operations (lnch-1020).
- Do not document Lambda connection handling — that is in lnch-1013.

## Scope

**Output file:** `docs/operations/runbooks/database-troubleshooting.md`

**Content outline:**

1. **Overview** — how to confirm the database is the root cause (vs Lambda, network)
2. **Connection exhaustion**
   - Symptoms (Lambda timeout, "too many connections" error)
   - Diagnosis: checking `pg_stat_activity`, connection counts per Lambda function
   - Resolution: killing idle connections, connection pool tuning, failover to read replica
3. **Lock contention and deadlocks**
   - Symptoms: query timeouts on writes
   - Diagnosis: `pg_locks`, `pg_stat_activity` blocked queries
   - Resolution: identifying and killing blocking queries
4. **High CPU / slow queries**
   - Symptoms: CloudWatch RDS CPU alarm
   - Diagnosis: Performance Insights, slow query log, `EXPLAIN ANALYZE`
   - Resolution: index recommendations, query optimization
5. **Replication lag** — symptoms, checking lag metric, impact on read replicas
6. **Storage capacity alert** — checking free storage, cleanup options, requesting storage increase
7. **Failover procedure** — when Aurora auto-failover triggers, what to expect, how to verify new primary
8. **Recovery from failed migration** — how to identify the damage, rollback steps
9. **Escalation** — when to engage AWS Support, DBA escalation path
