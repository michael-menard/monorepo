# lnch-1060: knowledge-base MCP Server Operational Runbook

## Context

Phase 3 (Launch readiness).

`apps/api/knowledge-base/` is a Node.js MCP (Model Context Protocol) server that runs as an AWS Lambda function. It is distinct from the main platform API: rather than serving the web front-end, it provides a structured knowledge retrieval interface consumed by AI agents (including the Claude-based workflow agents used in the platform's internal tooling). It exposes KB read/write operations over the MCP protocol, backed by the platform database and vector store.

Because the knowledge-base MCP server is an internal service (not user-facing) but is a hard dependency for the AI agent workflows that power features like recommendations and content generation, an outage here can degrade AI-assisted features across the platform without any direct user-visible error in the web apps. This makes it harder to detect and diagnose than a web app outage — a dedicated runbook is essential.

## Goal

A verified runbook at `docs/operations/runbooks/knowledge-base.md` covering deployment, rollback, health checks, environment variable inventory, and failure modes specific to the knowledge-base MCP Lambda server — including MCP protocol-level diagnostics and vector store connectivity checks.

## Non-Goals

- Vector store (embedding database) provisioning (infra runbook).
- MCP client configuration in AI agent workflows (AI/agent runbook).
- Main platform API deployment (main API runbook).
- Knowledge base content ingestion scripts (data platform runbook).

**Runbook sections to write:**

### 1. Service Overview
- Purpose: MCP server providing structured knowledge retrieval and writes for AI agents
- Runtime: Node.js Lambda function (not a web app — no CloudFront or S3 bucket)
- Invocation: API Gateway HTTP endpoint or Lambda function URL, consumed by MCP clients (AI agents)
- Upstream dependencies: Aurora PostgreSQL (structured KB data), vector store (semantic search), S3 (KB artifact storage)
- Consumers: internal AI agent workflows, Claude-based agents in the platform toolchain

### 3. Deployment Procedure
1. Pre-deploy checklist:
   - Confirm DB connectivity from Lambda VPC config (check VPC security group allows Lambda -> Aurora)
   - Confirm vector store is healthy (check its status endpoint)
   - Confirm no AI agent workflows are actively running a large KB operation (check CloudWatch for in-flight Lambda executions)
2. Build: `pnpm turbo run build --filter=knowledge-base`
3. Deploy via CDK or Serverless Framework (document the exact deploy command per environment):
   - `npx cdk deploy KnowledgeBaseStack --context env=prod`
4. Verify Lambda function update: check AWS Console that the function's code hash updated and the new version is the `$LATEST` alias target
5. Post-deploy smoke test:
   - Invoke the Lambda directly with a minimal MCP `initialize` request (document the test payload)
   - Confirm the response includes the correct `serverInfo.name` (`platform-knowledge-base`)
   - Invoke a `tools/list` MCP request — confirm the expected tools are listed
   - Invoke a simple `kb/read` tool call — confirm it returns a valid KB entry

### 4. Rollback Procedure
Lambda versioning enables instant rollback without re-deployment:
1. Identify the previous Lambda version number (visible in the AWS Console or via `aws lambda list-versions-by-function`)
2. Update the alias (`prod` or `staging`) to point to the previous version:
   `aws lambda update-alias --function-name knowledge-base-{env} --name live --function-version {prev-version}`
3. Invoke the `initialize` smoke test to confirm the rollback is live
4. Log rollback with version numbers and reason in ops channel
5. Do NOT delete the failed version — keep it for post-incident analysis

### 5. Health Checks
- Lambda reachability: `aws lambda invoke --function-name knowledge-base-{env} --payload '{...mcp-init...}' /tmp/out.json` — exit code 0, response body is valid MCP response
- DB connectivity: invoke a `kb/read` tool and confirm data is returned (not a timeout or connection error)
- Vector store connectivity: invoke a semantic search tool call and confirm embedding lookup succeeds
- CloudWatch: confirm the function's error rate is below threshold; p99 duration is within Lambda timeout margin

### 6. Common Errors and Remediation

| Error | Likely Cause | Remediation |
|---|---|---|
| Lambda times out on all invocations | DB or vector store connection not established (cold start + VPC networking delay) | Check VPC security groups; verify Lambda is in the correct subnets with Aurora access |
| `Connection refused` to Aurora | DB endpoint changed (e.g. after failover) or security group misconfigured | Update `DB_HOST` in Lambda env or SSM; check security group inbound rules on Aurora |
| MCP `initialize` returns unknown server name | Wrong Lambda version is active (alias pointing to wrong version) | Check alias target; update alias to correct version |
| `tools/list` returns empty tool set | Lambda code did not bundle the tools registry (build artifact issue) | Check build output; verify all tool modules are included in the Lambda zip |
| Vector store semantic search returns no results | Vector store index is stale or embeddings not generated for new KB entries | Trigger a re-embedding job; check vector store index status |
| S3 artifact read returns 403 | Lambda execution role missing S3 permissions or wrong bucket name | Check IAM role policy attached to Lambda; verify `S3_ARTIFACT_BUCKET` env var |
| High error rate after deploy with no obvious cause | Lambda initialisation error (module resolution failure) | Check CloudWatch Lambda init logs for module-not-found errors; this is usually a missing build step |

### 7. MCP Protocol Diagnostics

When AI agents report KB issues, standard MCP debugging steps:
1. Confirm the agent's MCP client is pointing to the correct Lambda function URL for the environment
2. Use the AWS CLI to directly invoke the Lambda with a raw MCP JSON-RPC request to isolate whether the issue is in the server or the client
3. Check CloudWatch Logs Insights for the Lambda log group: `aws logs start-query ...` (document the useful query patterns for KB errors)
4. Confirm the MCP protocol version matches between client and server (breaking change risk on updates)
