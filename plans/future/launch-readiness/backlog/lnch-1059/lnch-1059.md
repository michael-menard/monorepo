# lnch-1059: user-settings App Operational Runbook

## Context

Phase 3 (Launch readiness).

`apps/web/user-settings/` is an authenticated micro-frontend where users manage their profile, notification preferences, linked accounts, and account deletion. Like the wishlist gallery it is 100% authenticated — unauthenticated access must redirect to login. It is built with React 19, TanStack Router, and Module Federation, and deployed to its own CloudFront distribution and S3 bucket.

The user-settings app has elevated sensitivity compared to other micro-frontends because it controls account-level operations: profile updates, password change initiation (redirecting to the reset-password app), and account deletion. A runbook is required to ensure safe deployment and rapid incident recovery, particularly because a broken settings app can leave users unable to update security-sensitive profile data.

## Goal

A verified runbook at `docs/operations/runbooks/user-settings.md` covering deployment, rollback, health checks, environment variable inventory, and failure modes specific to the authenticated user-settings micro-frontend — with emphasis on write-path safety and auth guard correctness.

## Non-Goals

- Cognito password change flow (covered by the reset-password runbook, lnch-1058).
- User profile data schema and API (backend runbook).
- CloudFront distribution provisioning (infra runbook).

## Scope

**Output file:** `docs/operations/runbooks/user-settings.md`

**Runbook sections to write:**

### 1. App Overview
- Purpose: profile management, notification preferences, linked accounts, account deletion
- Architecture: Module Federation remote; shell loads it at `/settings`, requires auth
- CloudFront distribution ID and S3 bucket per environment
- Upstream dependencies: user profile API (read + write), Cognito (for auth and email/password change), notification preferences API

### 2. Environment Variables
| Variable | Purpose | Example |
|---|---|---|
| `VITE_API_BASE_URL` | User profile and preferences API | `https://api.example.com` |
| `VITE_COGNITO_USER_POOL_ID` | Auth (required) | `us-east-1_XXXXXX` |
| `VITE_COGNITO_CLIENT_ID` | Cognito app client | `abc123` |
| `VITE_RESET_PASSWORD_URL` | URL of the reset-password standalone app | `https://reset.example.com` |
| `VITE_IMAGE_CDN_BASE_URL` | Avatar/profile image CDN | `https://images.example.com` |
| `VITE_MF_REMOTE_URL` | Module Federation entry URL | `https://settings.cdn.example.com/remoteEntry.js` |
| `VITE_ENVIRONMENT` | Environment tag | `prod` |

### 3. Deployment Procedure
1. Pre-deploy checklist:
   - Verify user profile API health
   - Confirm `VITE_RESET_PASSWORD_URL` points to the correct reset-password app domain for this environment (staging vs prod)
   - Confirm no account deletion requests are actively processing (check for in-flight delete jobs)
2. Build: `pnpm turbo run build --filter=user-settings`
3. S3 sync with cache-control headers (no-cache for `remoteEntry.js`)
4. CloudFront invalidation for `remoteEntry.js` and `index.html`
5. Post-deploy smoke test:
   - Attempt unauthenticated access to `/settings` — confirm redirect to login
   - Log in as a test user — confirm settings page loads with profile data populated
   - Update display name — confirm save succeeds and persists on reload
   - Navigate to "Change Password" — confirm redirect to the reset-password app URL

### 4. Rollback Procedure
1. Restore previous S3 objects via versioning
2. Invalidate CloudFront
3. Smoke test: authenticated user can load settings, update a field, and trigger the change-password redirect
4. Log rollback event with reason and timestamp in ops channel

### 5. Health Check
- URL: `https://{settings-cdn-domain}/remoteEntry.js` — HTTP 200
- Auth redirect: unauthenticated visit to `/settings` redirects to login (HTTP 302 or client-side redirect, not blank screen)
- Profile load: authenticated test user sees their profile data populated in the form
- Write path: display name update persists across a page reload
- Change-password link: redirects to the correct reset-password app URL

### 6. Common Errors and Remediation

| Error | Likely Cause | Remediation |
|---|---|---|
| Settings page loads but profile fields are blank | User profile API returning empty or 404 for the authenticated user | Check API logs for the user's ID; verify the user record exists in DB |
| Profile save returns 400 | API validation rejecting the payload (e.g. display name too long) | Check API Lambda logs; surface validation error to user in UI |
| Account deletion gets stuck (spinner indefinitely) | Account deletion job Lambda timed out or DLQ has failed messages | Check deletion Lambda logs; check DLQ for failed messages; do not retry blindly — escalate |
| "Change Password" link broken | `VITE_RESET_PASSWORD_URL` set to wrong environment's URL | Verify env var in SSM; fix and redeploy |
| Avatar upload fails | Profile image CDN or S3 upload Lambda issue | Check image CDN and upload Lambda logs |
| Unauthenticated users see settings content briefly before redirect | Auth guard fires after initial render (flash of content) | Confirm auth check is synchronous in router guard; add loading state if needed |
| Module Federation fails to load | Shell MF config references wrong remote URL | Verify `VITE_MF_REMOTE_URL` and shell's MF configuration |

### 7. Account Deletion Incident Protocol
Account deletion is irreversible. If a deletion is initiated unexpectedly or in error:
1. Immediately check the deletion job queue for in-flight jobs
2. If the job has not completed, contact the backend team to halt it — document the Lambda function and DLQ name
3. If the job completed, escalate to the data team — account recovery from backup may be possible within the retention window
4. Notify the affected user directly

### 8. Contacts and Escalation
- Platform on-call: `#ops-escalation`
- User profile API: backend team
- Account deletion / data recovery: data platform team
- Auth: identity team
