# lnch-1017: Rate Limiting Adjustments Runbook

## Context

Phase 3 (Before general availability).

API Gateway and Lambda have configurable rate limits that protect the backend from abuse and unintended traffic spikes. However, overly aggressive rate limits will also block legitimate users, especially during launch when traffic patterns are unpredictable. Engineers need to know how to observe current rate limiting behavior, identify false positives (legitimate users being rate limited), and adjust limits safely.

## Goal

Write a runbook at `docs/operations/runbooks/rate-limiting.md` documenting how to monitor, diagnose, and adjust API rate limits in API Gateway and application-level rate limiting.

## Non-Goals

- Do not document WAF rules (separate security concern).
- Do not redesign the rate limiting architecture.
- Do not document Cognito throttling (covered in lnch-1022).

## Scope

**Output file:** `docs/operations/runbooks/rate-limiting.md`

**Content outline:**

1. **Overview** — rate limiting layers (API Gateway throttling, Lambda reserved concurrency, application-level limits)
2. **Current limit values** — table of limits per environment and endpoint type
3. **Identifying rate limit events** — CloudWatch metrics (4xx spikes, throttle count), log patterns
4. **Distinguishing abuse from legitimate traffic** — IP patterns, user agent analysis, Cognito user correlation
5. **Adjusting API Gateway limits** — AWS Console path, CLI command to update throttle settings
6. **Adjusting Lambda reserved concurrency** — when to change, how to change
7. **Temporary limit increase procedure** — for planned launch events or traffic spikes
8. **Rolling back limit changes** — restoring previous values
9. **Alerting** — what alarms fire on rate limit events, thresholds
10. **Escalation** — if abuse is confirmed, escalation to security team

## Acceptance Criteria

- [ ] Runbook exists at `docs/operations/runbooks/rate-limiting.md`

- [ ] Abuse vs legitimate traffic distinction guidance is actionable
