# lnch-1012: Frontend Rollback Runbook

## Context

Phase 1 (Critical path — must be complete before launch).

Frontend rollback is more complex than API rollback because static assets are immutable once uploaded, but CloudFront distributions cache aggressively. Rolling back the frontend means either re-deploying the previous build artifacts or pointing the distribution to a versioned S3 path. Module Federation remote entry URLs add another layer — rolling back the shell app without rolling back remote modules (or vice versa) can cause runtime errors for users.

## Goal

Write a frontend rollback runbook at `docs/operations/runbooks/frontend-rollback.md` that provides a safe, ordered procedure for reverting the frontend to a previously known-good state.

## Non-Goals

- Do not document API rollback (lnch-1010).
- Do not document forward deployments (lnch-1011).
- Do not document Module Federation architecture in depth (lnch-1018).

## Scope

**Output file:** `docs/operations/runbooks/frontend-rollback.md`

**Content outline:**

1. **Overview** — rollback mechanism (versioned S3 paths, re-deploy previous build), Module Federation ordering concern
2. **Decision criteria** — when to rollback frontend vs wait for a hotfix
3. **Prerequisites** — identify previous good build version/tag, AWS access, distribution ID
4. **Finding the previous deployment** — S3 versioning path convention, deployment tag in git
5. **Step-by-step rollback**
   a. Identify previous good artifact (S3 path or git tag)
   b. Re-deploy previous artifact to S3 (command)
   c. Invalidate CloudFront cache for affected paths
   d. Verify Module Federation remote entry URLs resolve correctly
   e. Smoke test critical pages
6. **Module Federation rollback order** — which remote to roll back first if only a subset is broken
7. **Communicating the rollback** — incident channel template
8. **Post-rollback actions** — incident ticket, post-mortem trigger (lnch-1026)
9. **Escalation** — link to lnch-1028
