# lnch-1020: Aurora Operations Runbook

## Context

Phase 2 (Before soft launch).

Aurora PostgreSQL is managed at the AWS cluster level with operations distinct from application-level database operations (lnch-1014). Cluster-level operations include scaling, maintenance windows, parameter group changes, failover testing, and snapshot management. These operations require AWS console/CLI access and carry risk if done incorrectly. This runbook is needed before soft launch when real user data will be at stake.

## Goal

Write an Aurora cluster operations runbook at `docs/operations/runbooks/aurora-operations.md` covering all routine and emergency cluster-level operations.

## Non-Goals

- Do not document application-level database operations (lnch-1014).
- Do not document database troubleshooting incidents (lnch-1015).
- Do not document Aurora provisioning from scratch.

## Scope

**Output file:** `docs/operations/runbooks/aurora-operations.md`

**Content outline:**

1. **Overview** — cluster identifiers per environment, instance types, Aurora version, writer/reader endpoints
2. **Scheduled maintenance windows** — when they are, what happens, how to prepare
3. **Manual failover procedure** — when to trigger, steps, verification, expected downtime
4. **Scaling operations**
   - Vertical scaling (instance type change) — procedure, maintenance window required, steps
   - Aurora Serverless auto-scaling — current ACU min/max settings, how to change
5. **Parameter group changes** — which parameters require reboot, change procedure
6. **Snapshot operations** — manual snapshot, automated backup schedule, cross-region copy
7. **Point-in-time restore** — when to use, steps, creating a new cluster from restore
8. **Major version upgrade procedure** — pre-upgrade checks, upgrade steps, rollback plan
9. **Storage operations** — monitoring free storage, storage auto-scaling behavior
10. **IAM database authentication** — if used, how to manage

## Acceptance Criteria

- [ ] Runbook exists at `docs/operations/runbooks/aurora-operations.md`
- [ ] Cluster identifiers for each environment are documented (or reference to where they are stored)
- [ ] Failover procedure is tested in staging and confirmed accurate
- [ ] Snapshot and restore procedures are verified against actual Aurora configuration
- [ ] Reviewed by the infra/DBA engineer
