# lnch-1043: Accessibility Audit (WCAG 2.1 AA)

## Context

Phase 1 (Critical path — must be complete before launch).

WCAG 2.1 AA compliance is a legal requirement in many jurisdictions and a moral obligation. The main app has not been formally audited. Known risk areas include: custom interactive components (galleries, MOC editors, dropdowns) that may lack proper ARIA roles, keyboard navigation gaps, insufficient color contrast with the LEGO sky/teal palette, and missing alt text on user-uploaded MOC images. This is the largest effort in the UX workstream (Size L).

## Goal

Conduct a comprehensive WCAG 2.1 AA audit of the main app and remediate all critical and high-impact violations. Implement axe-core automated checks in the test suite to prevent regressions.

## Non-Goals

- Do not audit the dashboard app (out of scope for this story).
- Do not target WCAG 2.1 AAA compliance.
- Do not redesign the visual design — work within the existing design system.

## Scope

**Output locations:**
- `apps/web/main-app/src/` — fixes distributed across existing components
- `apps/web/main-app/src/test/` — axe-core test integration
- `docs/operations/` — audit report (optional but recommended)

**WCAG 2.1 AA criteria to audit (minimum):**

| Criterion | Requirement | Risk Level |
|-----------|-------------|------------|
| 1.1.1 Non-text Content | Alt text for all images | High (user-uploaded MOC images) |
| 1.3.1 Info and Relationships | Semantic HTML structure | Medium |
| 1.4.3 Contrast (Minimum) | 4.5:1 for text, 3:1 for UI components | High (sky/teal palette) |
| 1.4.11 Non-text Contrast | 3:1 for UI components and graphics | Medium |
| 2.1.1 Keyboard | All interactive elements keyboard accessible | High |
| 2.1.2 No Keyboard Trap | Focus never gets stuck | High |
| 2.4.3 Focus Order | Logical tab order through page | Medium |
| 2.4.7 Focus Visible | Visible focus indicator on all interactive elements | High |
| 3.3.1 Error Identification | Form errors identify the field | Medium |
| 3.3.2 Labels or Instructions | All form controls labeled | Medium |
| 4.1.2 Name, Role, Value | ARIA for custom widgets | High |

**Implementation approach:**
1. Run automated axe-core scan on all routes, generate violation report
2. Manual keyboard-only navigation walkthrough of all critical flows
3. Screen reader test (VoiceOver/NVDA) on critical flows
4. Color contrast check with Tailwind palette
5. Remediate all Critical and High violations
6. Document any Medium violations deferred with timeline
7. Add axe-core to Vitest test suite for regression prevention

## Acceptance Criteria

- [ ] Automated axe-core scan reports zero Critical violations on all main app routes
- [ ] Manual keyboard navigation covers all critical user flows without traps or gaps
- [ ] All images (including user-uploaded) have meaningful alt text or are decorative (empty alt)
- [ ] Color contrast passes 4.5:1 for all body text, 3:1 for UI components
- [ ] All form controls have associated labels
- [ ] Focus indicator is visible on all interactive elements in the design system palette
- [ ] axe-core integrated into Vitest suite and runs on CI
- [ ] Audit report documents all violations found and their remediation status
- [ ] Reviewed by a person with screen reader experience
