# lnch-1045: Session Expiry & Re-auth UX

## Context

Phase 1 (Critical path — must be complete before launch).

Cognito tokens expire and users will be silently logged out. Currently, when this happens, the app likely shows a cryptic auth error or silently fails. The correct behavior is: warn the user before expiry, gracefully handle the expiry with a re-authentication prompt that preserves their current context (page and unsaved form data where possible), and redirect them back after successful re-auth. A broken session experience is one of the most trust-damaging UX failures.

## Goal

Implement proper session expiry detection, warning, and re-authentication flow in the main app that is seamless and preserves user context.

## Non-Goals

- Do not change Cognito token lifetime configuration.
- Do not implement "remember me" persistent sessions (separate security decision).
- Do not implement session management for the dashboard app in this story.

## Scope

**Output locations:**
- `apps/web/main-app/src/components/SessionExpiry/` — session expiry UI components
- `apps/web/main-app/src/hooks/useSession/` — session monitoring hook
- Updates to auth/token refresh logic in the app

**Implementation requirements:**

1. **Token refresh** — automatic silent token refresh before expiry using Cognito refresh token
2. **Expiry warning** — 5 minutes before session expires, show a dismissible toast/banner: "Your session expires in 5 minutes. Stay signed in?"
3. **Stay signed in** — clicking "Stay signed in" triggers a silent token refresh
4. **Session expired modal** — when the token expires without refresh, show a modal: "Your session has expired. Sign in to continue." with sign-in button
5. **Context preservation** — store current URL in session storage before redirecting to login; restore after re-auth
6. **Unsaved data warning** — if the user has unsaved form data, warn before redirect: "You have unsaved changes. Sign in to save them." (best effort)
7. **API 401 handler** — when any API call returns 401, trigger re-auth flow (catches edge cases)
8. **Accessibility** — session expiry modal is announced to screen readers (role="alertdialog")

**Component structure:**
```
SessionExpiry/
  index.tsx          # SessionExpiryModal
  SessionWarning.tsx # Warning toast/banner
  __tests__/
  __types__/
useSession/
  index.ts           # Hook for session monitoring
  __tests__/
```

## Acceptance Criteria

- [ ] Silent token refresh prevents expiry for active users
- [ ] Warning toast appears 5 minutes before session expiry
- [ ] "Stay signed in" button successfully refreshes the session
- [ ] Expired session shows modal with sign-in option (not a broken page)
- [ ] After re-auth, user is redirected back to their previous URL
- [ ] API 401 responses trigger the re-auth flow
- [ ] Session expiry modal has `role="alertdialog"` and is keyboard navigable
- [ ] Vitest tests cover warning and expiry scenarios using mocked Cognito tokens
- [ ] Playwright E2E test covers the full expiry and re-auth flow (linked to lnch-1052)
