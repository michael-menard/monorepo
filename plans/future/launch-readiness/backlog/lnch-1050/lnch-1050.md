# lnch-1050: Confirmation & Success Feedback

## Context

Phase 3 (Before general availability).

Users currently receive no consistent feedback after completing important actions — creating a MOC, saving changes, deleting an item, or updating their profile. Without feedback, users don't know if their action succeeded, which leads to double-submissions, confusion, and support requests. This is a quick win (Size S) — a toast/notification system and consistent success patterns can be implemented rapidly.

## Goal

Implement a consistent toast/notification system for success and confirmation feedback, and apply it to all significant user actions in the main app.

## Non-Goals

- Do not build a full notification center (separate feature).
- Do not implement error messages here (lnch-1040 covers error feedback).
- Do not implement optimistic UI updates.

## Scope

**Output locations:**
- `apps/web/main-app/src/components/Toast/` — toast notification component
- Updates to all action handlers that lack success feedback

**Implementation requirements:**

1. **Toast component** — success, info, warning variants; auto-dismisses after 4 seconds; manually dismissible; stacks if multiple toasts
2. **Toast provider** — context provider at app root, `useToast` hook for triggering toasts
3. **Success actions to cover**
   - MOC created: "Your MOC has been published!"
   - MOC saved/updated: "Changes saved"
   - MOC deleted: "MOC deleted" (with undo option within 5 seconds if feasible)
   - Profile updated: "Profile updated successfully"
   - File uploaded: "File uploaded successfully"
   - Password changed: "Password updated"
   - Account settings saved: "Settings saved"
4. **Destructive confirmations** — before delete actions, show a confirmation dialog (not a toast): "Delete this MOC? This cannot be undone." [Delete] [Cancel]
5. **Accessibility** — toasts use `role="status"` for non-urgent, `role="alert"` for urgent; focus not moved to toast

**Component structure:**
```
Toast/
  index.tsx         # Toast component
  ToastProvider.tsx # Context provider
  useToast.ts       # Hook
  __tests__/
  __types__/
```

## Acceptance Criteria

- [ ] Toast component exists with success, info, and warning variants
- [ ] All 7 success actions have corresponding toast notifications
- [ ] Delete actions have a confirmation dialog before proceeding
- [ ] Toasts auto-dismiss after 4 seconds and are manually dismissible
- [ ] Toasts use correct ARIA roles (role="status" or role="alert")
- [ ] Toasts are announced by screen readers without moving focus
- [ ] Vitest tests cover toast display and auto-dismiss behavior
- [ ] Reviewed by the product designer for copy and behavior
