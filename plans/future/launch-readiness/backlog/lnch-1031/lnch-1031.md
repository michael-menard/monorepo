# lnch-1031: Metrics Reference Guide

## Context

Phase 3 (Before general availability).

Engineers monitoring the system need to know what each metric means, what normal values look like, and when to be concerned. Without a reference guide, every engineer builds their own mental model of "normal" independently, making it hard to communicate about metrics and harder to spot subtle degradation. This guide acts as the authoritative reference for all system metrics.

## Goal

Write a metrics reference guide at `docs/operations/monitoring/metrics-reference.md` that documents every key metric, its meaning, normal ranges, and thresholds for concern.

## Non-Goals

- Do not document CloudWatch dashboard navigation (lnch-1029).
- Do not document alarm response procedures (lnch-1030).
- Do not document business metrics or analytics (separate product concern).

## Scope

**Output file:** `docs/operations/monitoring/metrics-reference.md`

**Content outline:**

1. **Overview** — how to use this guide, metric source reference
2. **API / Lambda metrics table**
   | Metric | Source | Unit | Normal Range | Warning | Critical |
   |--------|--------|------|--------------|---------|----------|
   | Invocation count | Lambda | count/min | varies | - | - |
   | Error rate | Lambda | % | <1% | >2% | >5% |
   | Duration p50/p95/p99 | Lambda | ms | TBD | TBD | TBD |
   | Throttles | Lambda | count | 0 | >0 | >10/min |
   | Concurrent executions | Lambda | count | varies | 80% limit | 90% limit |
3. **API Gateway metrics table** — request count, 4xx rate, 5xx rate, latency
4. **Aurora PostgreSQL metrics table** — CPU, connections, IOPS, free storage, replica lag
5. **OpenSearch metrics table** — cluster status, JVM memory, indexing rate, search latency
6. **CloudFront metrics table** — request count, error rate, cache hit ratio, origin latency
7. **Custom application metrics** — any application-level metrics emitted via CloudWatch custom namespaces
8. **SLO definitions** — formal SLO targets for key metrics
