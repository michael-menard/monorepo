# lnch-1040: Error Message Standardization

## Context

Phase 1 (Critical path — must be complete before launch).

Error messages across the application are currently inconsistent — some show raw API error codes, some show generic "Something went wrong" messages, and some show nothing at all. Poor error messages destroy user trust and generate unnecessary support requests. Users need to understand what went wrong and what action they can take. This story audits all error states and implements a consistent error message system.

## Goal

Audit all error states in the main app and implement a standardized error message system with user-friendly copy, appropriate context, and actionable recovery paths.

## Non-Goals

- Do not redesign the error display components from scratch if a suitable pattern already exists.
- Do not handle server-side error message generation (that is an API concern).
- Do not implement a full error boundary hierarchy (may be separate).

## Scope

**Output locations:**
- `apps/web/main-app/src/components/ErrorMessage/` — standardized error message component
- `apps/web/main-app/src/components/ErrorBoundary/` — React error boundary component
- `apps/web/main-app/src/utils/errors/` — error categorization utilities
- Update all existing error display sites to use the standardized component

**Implementation requirements:**

1. **Error message component** — accepts error type, displays appropriate user-facing copy and recovery action
2. **Error categories and copy**
   - Network errors: "Check your connection and try again" + retry button
   - Auth errors (401): "You need to sign in" + sign-in link
   - Permission errors (403): "You don't have permission to do that" + back link
   - Not found (404): "We couldn't find that" + home link
   - Validation errors: field-level with specific constraint message
   - Server errors (500): "Something went wrong on our end. We've been notified." + retry
   - Rate limit (429): "You're doing that too fast. Please wait a moment." + auto-retry countdown
3. **Error boundary** — catches unexpected React errors, renders a safe fallback
4. **Audit of existing error states** — identify and list all current error display sites, update each to use the standard component
5. **Error logging** — errors above validation level should be logged via `@repo/logger`
6. **Zod schema for error types** — typed error categorization

## Acceptance Criteria

- [ ] Standardized `ErrorMessage` component exists and is used at all error display sites
- [ ] All 7 error categories have user-friendly copy (no raw error codes shown to users)
- [ ] Network errors have a retry button that works
- [ ] Auth errors redirect to sign-in correctly
- [ ] React ErrorBoundary catches uncaught errors and renders a safe fallback
- [ ] Error messages pass axe-core scan (role="alert" where appropriate)
- [ ] Vitest tests cover all error categories
- [ ] No raw API error messages appear in any user-visible state (verified by QA walkthrough)
