# HSKP-2004: Serverless MCP Server

## Context

The monorepo's API is defined in `serverless.yml` â€” each Lambda function has a name, HTTP path, method, IAM permissions, environment variables, and other configuration. When Claude Code adds a new endpoint or modifies an existing one, it needs accurate information about what already exists to avoid naming conflicts, path collisions, and incorrect IAM configuration.

The Serverless MCP Server parses `serverless.yml` and exposes its contents as queryable MCP tools. It handles the variable interpolation complexity of Serverless Framework (`${self:...}`, `${env:...}`, `${file(...)}`) so Claude Code receives fully-resolved values.

This story depends on HSKP-2002 (MCP Server Infrastructure).

---

## Goal

Build a Serverless MCP Server that exposes three tools for querying Lambda function configuration from `serverless.yml`, with full variable reference resolution and sub-200ms cached responses.

---

## Non-Goals

- Deploying or modifying `serverless.yml`
- Validating `serverless.yml` beyond what is needed for parsing
- Supporting Serverless Framework plugins that add custom variable sources
- Supporting non-Serverless deployment configurations (CDK, SAM, etc.)

---

## Scope

### MCP Tools

**`list_functions`**
- Returns all Lambda function names defined in `serverless.yml`
- Includes the resolved HTTP path and method for each function
- Response shape: `{ functions: Array<{ name: string, path: string, method: string }> }`

**`get_function_config`**
- Accepts a function name
- Returns complete function configuration: handler path, runtime, memory, timeout, environment variables (resolved), layers, VPC config
- Resolves `${self:...}`, `${env:...}`, and `${file(...)}` variable references
- Response shape: `{ function: string, config: FunctionConfig }`

**`get_iam_permissions`**
- Accepts a function name
- Returns the IAM statements attached to that function (or the global provider IAM if the function inherits it)
- Resolves any variable references within IAM statements
- Response shape: `{ function: string, statements: Array<IamStatement> }`

### Variable Resolution
The parser must handle these Serverless Framework variable patterns:
- `${self:provider.region}` â€” references other values in the same `serverless.yml`
- `${env:SOME_VAR}` â€” references environment variables (substituted from `.env` or process environment)
- `${file(./config/vars.yml):someKey}` â€” references values from external files

If a variable cannot be resolved (e.g., `${env:SECRET}` not set), the raw reference string is returned rather than throwing an error, and a warning is logged.

### Server Location
`tools/mcp-servers/serverless/`

### Caching
- All three tools are cached using `@repo/mcp-shared` cache
- Cache is invalidated by file-watching on `serverless.yml` and any `${file(...)}` referenced files
- Target: sub-200ms for all cached responses

## Implementation Notes

- Parse `serverless.yml` using `js-yaml` â€” do not use the Serverless Framework CLI programmatically (too heavy)
- Implement variable resolution as a recursive post-processing pass over the parsed YAML object
- Self-references (`${self:...}`) can be resolved with a simple deep-get on the parsed object â€” handle circular references defensively
- For `${env:...}`, load `.env` file at server startup using `dotenv` (if present) as a fallback for unset env vars
- File references (`${file(...)}`) should support both YAML and JSON files
