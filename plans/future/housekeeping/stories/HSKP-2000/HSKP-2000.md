# HSKP-2000: API Service Extraction

## Context

All 45 Lambda handlers currently contain mixed concerns: business logic, AWS-specific event parsing, and response formatting are tightly coupled inside each handler. This makes local development without Lambda impossible, complicates testing (tests must simulate API Gateway events), and couples every handler to the AWS transport layer.

The goal of this story is to apply the Hexagonal Architecture pattern (Ports & Adapters) to the API layer â€” separating domain services from transport adapters so that the same business logic can run in Lambda, an Express server, or any future runtime without modification.

This is the foundational story for API Portability. HSKP-2001 (Express Local Dev Server) depends on this story being complete.

---

## Goal

Extract business logic from Lambda handlers into transport-agnostic service and handler packages. Introduce three new packages that replace direct Lambda-coupled code with clean separation of concerns.

---

## Non-Goals

- Running the API locally (covered by HSKP-2001)
- Changes to the database schema or Drizzle models
- Changes to the frontend API client
- Moving to a different deployment model â€” Lambda remains the production target
- Migrating 100% of all endpoints in this story â€” focus is MOC, Gallery, Wishlist, and Parts List domains

---

## Scope

### New Packages

**`@repo/api-services`** â€” Domain layer
- Pure TypeScript business logic, no AWS dependencies
- Services for MOC, Gallery, Wishlist, and Parts List domains
- Accepts a `HandlerContext` (db connection, user identity, config) injected at the boundary
- Has no knowledge of HTTP or Lambda

**`@repo/api-handlers`** â€” Application layer
- Unified `Request` / `Response` handler functions using web-standard interfaces
- Handler signature:
  ```typescript
  export async function handleListMocs(
    request: Request,
    ctx: HandlerContext
  ): Promise<Response>
  ```
- Calls services from `@repo/api-services`
- Shared utilities: error classes, pagination helpers, response formatters

**`@repo/lambda-adapter`** â€” Infrastructure layer
- Converts API Gateway events to `Request` objects
- Converts handler `Response` objects back to API Gateway response format
- Extracts user identity from Cognito authorizer context
- Lambda entry points become thin wrappers: parse event â†’ call handler â†’ format response

### Migration Scope
- Update all 45 Lambda handlers across MOC, Gallery, Wishlist, and Parts List domains to use the new packages
- Shared utilities (error classes, pagination, response formatting) extracted to `@repo/api-handlers`

### Handler Pattern
```typescript
// Before: Lambda-coupled handler
export const handler = async (event: APIGatewayProxyEvent) => {
  const userId = event.requestContext.authorizer?.claims?.sub
  // ... business logic mixed in
  return { statusCode: 200, body: JSON.stringify(result) }
}

// After: Thin Lambda wrapper
export const handler = lambdaAdapter(handleListMocs)

// Portable handler in @repo/api-handlers
export async function handleListMocs(
  request: Request,
  ctx: HandlerContext
): Promise<Response> {
  // pure business logic, no AWS
}
```

## Implementation Notes

- Migrate one domain at a time (suggest: Wishlist first since HSKP-2007/2008 touch it, then MOC, Gallery, Parts List)
- Use feature flags or staged deployment if risk of regression is high
- The `Request` and `Response` types should use the global Web API types (available in Node 18+), not custom interfaces
- `HandlerContext` must be injected at the Lambda boundary â€” never imported as a singleton inside handlers
