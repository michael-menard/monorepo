# HSKP-2006: Scaffold Feature Skill

## Context

HSKP-2005 generates a single endpoint. Most product features require multiple endpoints, a database migration, and a full frontend slice (components, RTK hooks, routes). Doing this manually — even with the `/scaffold-endpoint` skill — requires coordinating many moving pieces and making consistent naming and architecture decisions across the full vertical.

The `/scaffold-feature` skill goes one level higher: given a feature description, it generates the complete vertical slice — backend handlers, schemas, serverless config, frontend components, RTK hooks, and optionally a database migration — as a coherent, interconnected set of files.

This story depends on HSKP-2002, HSKP-2003, HSKP-2004 (MCP servers) and HSKP-2005 (Endpoint Skill, which establishes the single-endpoint generation patterns that this story composes).

---

## Goal

Implement a `/scaffold-feature` Claude Code slash command that generates a complete vertical slice of a new feature from a plain-English description, presenting all generated files as a diff for developer approval before writing, with an incremental apply option.

---

## Non-Goals

- Auto-merging the generated code into existing files without developer review
- Generating tests that require real database connections (generated tests use mocks)
- Generating infrastructure beyond `serverless.yml` entries (no CDK, no Terraform)
- Replacing the developer's judgment on architecture — the skill generates a starting point, not a finished product

---

## Scope

### Command Interface
```
/scaffold-feature "wishlist priority sorting with drag-drop reorder"
/scaffold-feature "MOC soft delete with restore capability"
/scaffold-feature "gallery image lightbox with keyboard navigation"
```

### Generation Layers

**API Layer** (calls `/scaffold-endpoint` patterns for each endpoint)
- Lambda handlers for all CRUD/action endpoints implied by the feature
- Zod request/response schemas per endpoint
- `serverless.yml` entries for all functions
- Handler integration tests

**Frontend Layer**
- React feature components in the appropriate `apps/web/*` application
- RTK Query hooks for all generated endpoints
- Route registration (if the feature introduces a new page)
- Component unit tests

**Shared Layer**
- Zod schemas shared between frontend and backend in `@repo/api-client`
- TypeScript types derived from those schemas

**Database Layer** (optional, prompted)
- Drizzle migration file if the feature requires schema changes
- New Drizzle table definitions or column additions
- Updated Drizzle relations

### Generation Workflow
1. **Parse feature description** — identify: entities involved, CRUD operations needed, UI entry points, whether DB changes are required
2. **Query MCP for context**:
   - `list_tables`, `get_table_schema`, `get_relations` from Drizzle MCP for existing data model context
   - `list_functions` from Serverless MCP to understand existing endpoints and avoid conflicts
3. **Generate implementation plan** — present an ordered list of all files to be generated with a brief description of each, before generating any code. Developer confirms or adjusts the plan.
4. **Generate code** — produce all files in the confirmed plan
5. **Validate** — run `tsc --noEmit` on all generated TypeScript files
6. **Present diff** — show all generated files as a unified diff
7. **Apply** — either apply all at once or incrementally (layer by layer: DB → API → Frontend)

### Incremental Apply Option
The developer can choose to apply the generated output in layers:
- Apply DB migration first, review, then apply API layer, then Frontend layer
- Useful for large features where the developer wants to commit and test incrementally

## Acceptance Criteria

### Intent Parsing
- [ ] **AC1**: The skill identifies all CRUD operations implied by the feature description
- [ ] **AC2**: The skill identifies entities involved and maps them to existing Drizzle tables (via MCP)
- [ ] **AC3**: The skill correctly identifies whether a DB migration is likely needed and prompts the developer

### Implementation Plan
- [ ] **AC4**: Before generating any code, the skill presents a file-by-file plan listing every artifact to be created
- [ ] **AC5**: The developer can modify the plan (e.g., "skip the migration, I'll do it manually") before generation proceeds
- [ ] **AC6**: The plan includes estimated implementation order (DB → API → Frontend)

### API Layer
- [ ] **AC7**: All handlers follow the portable pattern from HSKP-2000/2005
- [ ] **AC8**: Zod schemas are generated for every endpoint's request and response
- [ ] **AC9**: `serverless.yml` entries are generated for every function with no path conflicts
- [ ] **AC10**: Handler tests are generated for all endpoints (happy path + validation + auth)

### Frontend Layer
- [ ] **AC11**: React components are generated in the correct `apps/web/*` directory for the feature domain
- [ ] **AC12**: Components follow the component directory structure: `ComponentName/index.tsx`, `__tests__/`, `__types__/`
- [ ] **AC13**: RTK Query hooks are generated for all API endpoints
- [ ] **AC14**: If the feature requires a new page/route, the route is registered in the application router
- [ ] **AC15**: Component tests are generated (basic render + primary user interaction)

### Shared Layer
- [ ] **AC16**: Shared Zod schemas that are used by both backend and frontend are placed in `@repo/api-client`
- [ ] **AC17**: No type duplication — frontend and backend reference the same schema type

### Database Layer
- [ ] **AC18**: If the developer confirms a migration is needed, a Drizzle migration file is generated
- [ ] **AC19**: New table definitions follow the existing Drizzle schema patterns in the codebase
- [ ] **AC20**: Relations are added to the Drizzle relations file for any new foreign keys

### Diff Approval and Incremental Apply
- [ ] **AC21**: All generated files are presented as a unified diff before any file is written
- [ ] **AC22**: Developer can choose "apply all" or "apply by layer"
- [ ] **AC23**: If the developer declines, no files are modified
- [ ] **AC24**: After applying, the skill reports how many files were created/modified

### Type Validation
- [ ] **AC25**: All generated TypeScript files pass `tsc --noEmit` before being presented
- [ ] **AC26**: Type errors are reported with file and line number, and the skill attempts one self-correction pass

### Quality
- [ ] **AC27**: Skill agent file (`scaffold-feature.agent.md`) is documented with usage examples and an explanation of the generation layers
- [ ] **AC28**: A sample invocation (`/scaffold-feature "MOC soft delete with restore capability"`) produces all expected artifacts across API, Frontend, and DB layers

---

## Implementation Notes

- The skill reuses the single-endpoint generation logic from HSKP-2005 for each API endpoint — avoid duplicating the generation logic
- The implementation plan step is critical for developer trust — do not skip it or make it opt-in
- For large features, the total diff can be very large (20+ files). Consider paginating the diff presentation by layer
- The skill should detect if the feature description overlaps with an existing feature (via `list_functions`) and warn the developer rather than generating conflicting files
