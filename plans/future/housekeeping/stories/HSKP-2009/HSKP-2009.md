# HSKP-2009: Wishlist Handler Integration Tests

## Context

There are zero integration tests for the 8 wishlist API handlers. The project requires a minimum of 45% test coverage, and the wishlist domain is a significant portion of the API surface area. Without integration tests, handler logic — input validation, authorization, database interactions, error cases — is untested and any regression goes undetected until it reaches production.

This story depends on HSKP-2008 (Consolidate Wishlist Schemas) being complete so that handlers use the canonical schemas from `@repo/api-client`. All tests should import schemas from the same source as the handlers.

---

## Goal

Create integration tests for all 8 wishlist handler functions, covering happy path, input validation errors, and authorization errors for each handler. Bring wishlist domain test coverage to or above the 45% project minimum.

---

## Non-Goals

- E2E tests for the wishlist UI (those belong in Playwright tests)
- Unit tests for individual utility functions (those are separate from handler integration tests)
- Performance or load tests
- Tests that require a real running database — all database interactions are mocked

---

## Scope

### Handlers to Test

| Handler | File | HTTP |
|---------|------|------|
| create-item | `apps/api/endpoints/wishlist/create-item/handler.ts` | POST /wishlist/items |
| list | `apps/api/endpoints/wishlist/list/handler.ts` | GET /wishlist/items |
| get-item | `apps/api/endpoints/wishlist/get-item/handler.ts` | GET /wishlist/items/:id |
| update-item | `apps/api/endpoints/wishlist/update-item/handler.ts` | PUT /wishlist/items/:id |
| delete-item | `apps/api/endpoints/wishlist/delete-item/handler.ts` | DELETE /wishlist/items/:id |
| reorder | `apps/api/endpoints/wishlist/reorder/handler.ts` | POST /wishlist/items/reorder |
| search | `apps/api/endpoints/wishlist/search/handler.ts` | GET /wishlist/items/search |
| upload-image | `apps/api/endpoints/wishlist/upload-image/handler.ts` | POST /wishlist/items/:id/image |

### Test Strategy
- Use Vitest for all tests
- Mock the database layer (Drizzle client) using `vi.mock()` or a test double injected via `HandlerContext`
- Mock authentication by constructing a `HandlerContext` with a test `userId`
- Do not mock the handler itself — test the actual handler function

### Test Cases Per Handler
Each handler requires at minimum:
1. **Happy path** — valid input, mocked DB returns success, assert correct response status and body
2. **Validation error** — invalid/missing required input, assert 400 response with error detail
3. **Authorization error** — missing `userId` in context or attempt to access another user's resource, assert 401 or 403

### Test File Location
`apps/api/endpoints/wishlist/{handler-name}/__tests__/{handler-name}.test.ts`

## Implementation Notes

- The `HandlerContext` mock should be created as a shared test fixture:
  ```typescript
  function createTestContext(overrides?: Partial<HandlerContext>): HandlerContext {
    return {
      db: createMockDb(),
      userId: 'test-user-id',
      logger,
      ...overrides,
    }
  }
  ```
- Database mock should return typed results that match the Drizzle select schema to avoid false type errors in tests
- For `upload-image`, mock the S3 client as well — the handler should not make real S3 calls in unit/integration tests
- After HSKP-2000 lands, tests should use the portable handler signature directly rather than wrapping in a Lambda event
