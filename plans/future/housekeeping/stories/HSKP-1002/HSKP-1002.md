# HSKP-1002: Reset Password Tests

## Context

After a user requests a password reset (HSKP-1001), Cognito sends a verification code to their email. The user then navigates to the reset password page, enters the verification code along with a new password, and completes the reset. This flow requires testing for code validation, password strength requirements, and error handling for invalid or expired codes.

This is a sub-story of HSKP-1000 (Auth E2E Test Suite). It depends on HSKP-1001 to ensure the full forgot + reset journey is coherent.

---

## Goal

Implement a complete set of Playwright E2E tests for the reset password flow, covering UI element presence, verification code validation, password strength enforcement, the happy path (successful password reset), and error handling for invalid and expired verification codes.

---

## Non-Goals

- Testing the forgot password form (covered by HSKP-1001)
- Testing email delivery of the verification code
- Unit or integration tests for the ResetPassword component
- Testing Cognito token expiry timing at the infrastructure level

---

## Scope

### Test Scenarios

**UI Element Verification**
- Page renders with verification code input, new password input, confirm password input, and submit button

**Verification Code Validation**
- Empty code shows required field error
- Incorrectly formatted code shows validation error
- Expired or invalid code returns an appropriate error message after submission

**Password Strength Requirements**
- Password too short is rejected with a clear error
- Password missing required character classes (uppercase, number, symbol) is rejected
- Matching valid passwords pass validation

**Happy Path**
- Valid verification code + strong matching passwords resets the password successfully
- User is redirected to login (or shown a success message) after reset

**Error Handling**
- Invalid code (wrong digits) returns an error message
- Expired code returns a specific expired-code error message

- `apps/web/playwright/features/auth/reset-password.feature`
- `apps/web/playwright/steps/reset-password.steps.ts`

## Acceptance Criteria

### UI Elements
- [ ] **AC1**: Reset password page renders a verification code input field
- [ ] **AC2**: New password input field is present
- [ ] **AC3**: Confirm password input field is present
- [ ] **AC4**: Submit button is present and clearly labelled

### Verification Code Validation
- [ ] **AC5**: Submitting with an empty verification code shows a required field error
- [ ] **AC6**: Submitting with a code that does not match Cognito's format shows a validation error
- [ ] **AC7**: Submitting with an expired/invalid code from Cognito shows a user-readable error message

### Password Strength
- [ ] **AC8**: Password shorter than the minimum length (e.g., 8 characters) is rejected with an error
- [ ] **AC9**: Password missing a required character class shows the appropriate strength error
- [ ] **AC10**: Confirm password not matching the new password shows a mismatch error
- [ ] **AC11**: A strong, matching password pair passes all client-side validation

### Happy Path
- [ ] **AC12**: Entering a valid Cognito verification code and strong matching passwords completes the reset
- [ ] **AC13**: After successful reset the user is redirected to login or shown a success confirmation
- [ ] **AC14**: The test uses a real Cognito test user and a real verification code obtained via admin SDK (`adminResetUserPassword` / `adminInitiateAuthCode`)
- [ ] **AC15**: Test user is deleted via Cognito admin utility after the test completes

### Error Handling
- [ ] **AC16**: Submitting an incorrect verification code shows an "Invalid code" error message
- [ ] **AC17**: Submitting an expired verification code shows an "Expired code" or equivalent error message
- [ ] **AC18**: Errors are displayed inline or as notifications — the form is not reset on error

### Test Quality
- [ ] **AC19**: All selectors use semantic queries (`getByRole`, `getByLabelText`, `getByText`)
- [ ] **AC20**: Tests pass in CI with no flakiness over 3 consecutive runs

---

## Implementation Notes

- Obtaining a real verification code in tests requires using the Cognito admin SDK to trigger a code and then intercept it — check whether `adminResetUserPassword` can be combined with a custom message Lambda or alternative approach
- If real code interception is not feasible in the test environment, use `adminSetUserPassword` to set the password directly and test only the UI validation paths; document the limitation
- Password strength requirements must match the Cognito User Pool password policy configured for the environment
