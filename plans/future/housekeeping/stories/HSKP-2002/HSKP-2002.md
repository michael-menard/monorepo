# HSKP-2002: MCP Server Infrastructure

## Context

Claude Code (and other AI assistants supporting the Model Context Protocol) can call MCP servers to retrieve structured codebase context — database schemas, serverless configuration, API contracts, etc. This dramatically improves code generation quality by giving the AI accurate, up-to-date information instead of relying on pattern matching from training data.

This story creates the shared foundation that all MCP servers in the monorepo will build on. HSKP-2003 (Drizzle MCP) and HSKP-2004 (Serverless MCP) both depend on the infrastructure established here.

---

## Goal

Build a shared MCP server infrastructure library, server factory pattern, and Claude Code integration so that subsequent MCP servers (Drizzle, Serverless) can be implemented quickly with consistent error handling, caching, and registration.

---

## Non-Goals

- Implementing the Drizzle or Serverless MCP tools themselves (HSKP-2003, HSKP-2004)
- Building an MCP proxy or aggregator
- Supporting MCP clients other than Claude Code at this stage
- Real-time streaming responses — request/response is sufficient

---

## Scope

### Shared Library (`tools/mcp-servers/shared/`)

**Server Factory**
- `createMcpServer(config)` factory function that wires up tool registration, lifecycle hooks, and error handling
- Consistent JSON response envelope across all tools
- Tool registration helper with typed input/output schemas (Zod)

**Error Handling Hierarchy**
- `McpError` base class
- `McpToolError` — tool-level errors (bad input, not found)
- `McpInternalError` — server-level errors (DB unavailable, file not found)
- All errors serialise to a consistent JSON structure Claude Code can parse

**Response Caching**
- In-memory LRU cache keyed by tool name + serialised input
- File-watching invalidation: cache is busted when the relevant source file changes (e.g., schema file, `serverless.yml`)
- Configurable TTL per tool, defaulting to 60 seconds
- Target: sub-200ms for all cached responses

**Logging**
- Structured logging via `@repo/logger` for all tool invocations (tool name, duration, cache hit/miss)

### Claude Code Integration
- `.claude/settings.json` updated to register all MCP servers
- MCP server processes launched as sidecar processes via the Claude Code MCP configuration
- Each server listens on a configurable stdio or socket transport

### Directory Structure
```
tools/
  mcp-servers/
    shared/
      src/
        factory.ts          # createMcpServer
        errors.ts           # McpError hierarchy
        cache.ts            # LRU cache with file-watching
        logger.ts           # Structured logging wrapper
        types.ts            # Shared Zod schemas
      package.json          # @repo/mcp-shared
    drizzle/                # HSKP-2003
    serverless/             # HSKP-2004
```

## Implementation Notes

- Use the `@modelcontextprotocol/sdk` npm package for the MCP server implementation — do not implement the protocol from scratch
- File watching can use `chokidar` (already likely in the monorepo) or Node's built-in `fs.watch`
- The LRU cache can use `lru-cache` package — lightweight and well-maintained
- Transport: stdio is the simplest MCP transport and is sufficient for Claude Code integration
