# HSKP-1000: Auth E2E Test Suite - Validation & Completion

## Context

The authentication flows (signup, login, email verification, forgot password, reset password, logout) are critical paths that must not regress. Playwright E2E tests exist for some flows but coverage is incomplete. A bug (HSKP-1003, now fixed) was causing login form submissions to fail in E2E tests due to a selector strategy mismatch and a Radix Checkbox registration issue with react-hook-form.

This is the parent orchestrating story for the auth E2E effort. It coordinates validation of the existing test suite and implementation of missing test coverage via sub-stories HSKP-1001 and HSKP-1002.

**Key Files:**
- `apps/web/playwright/features/auth/*.feature`
- `apps/web/playwright/steps/*.steps.ts`
- `apps/web/playwright/utils/cognito-admin.ts`

---

## Goal

Achieve 100% Playwright E2E test coverage for all authentication flows: signup, login, email verification, forgot password, reset password, and logout. Ensure proper test user cleanup via Cognito admin utilities so tests remain isolated and repeatable.

---

## Non-Goals

- Unit or integration tests for auth components (covered separately)
- Testing third-party OAuth flows (out of scope for this epic)
- Load or performance testing of auth endpoints
- Testing Cognito internal behavior — only the UI flows are tested

---

## Scope

### Validate Existing Tests
- Confirm signup, login, and email-verification tests pass with the HSKP-1003 fix applied
- Ensure selectors use `getByRole` and `getByLabelText` consistently (no brittle CSS selectors)
- Verify test user cleanup runs via Cognito admin utilities after each test

### Implement Missing Tests
- Forgot password flow (delegated to HSKP-1001)
- Reset password flow (delegated to HSKP-1002)
- Logout flow (inline with this story)

### Test Infrastructure
- Cognito admin utility (`cognito-admin.ts`) must support user creation, confirmation, and deletion
- All test users created during E2E runs must be cleaned up in `afterEach` / `afterAll` hooks

---

## Acceptance Criteria

### Existing Tests
- [ ] **AC1**: Signup happy path passes end-to-end with verified test user cleanup
- [ ] **AC2**: Login happy path passes using `getByRole` / `getByLabelText` selectors (no CSS selectors)
- [ ] **AC3**: Email verification flow passes including Cognito confirmation step
- [ ] **AC4**: All existing auth tests pass in CI with no flakiness over 3 consecutive runs

### Logout Tests
- [ ] **AC5**: Authenticated user clicks logout and is redirected to the login page
- [ ] **AC6**: After logout, navigating to a protected route redirects to login
- [ ] **AC7**: Logout test cleans up the test user via Cognito admin utility

### Sub-Story Completion
- [ ] **AC8**: HSKP-1001 (Forgot Password Tests) completed and all its AC pass
- [ ] **AC9**: HSKP-1002 (Reset Password Tests) completed and all its AC pass

### Test Quality
- [ ] **AC10**: No test uses CSS class selectors — all use semantic queries (`getByRole`, `getByLabelText`, `getByText`)
- [ ] **AC11**: Each test creates its own isolated test user and deletes it on completion
- [ ] **AC12**: All auth E2E tests pass in the CI pipeline

---

## Implementation Notes

- The HSKP-1003 fix (Controller-based Checkbox for react-hook-form) must be confirmed in place before validating login tests
- Cognito admin utilities should use the Admin SDK (`adminCreateUser`, `adminSetUserPassword`, `adminDeleteUser`) to bypass the normal confirmation flow in test setup
- Avoid sharing test users across tests to prevent ordering dependencies
