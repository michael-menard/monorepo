# HSKP-1003: Bug - Login Form Not Submitting in E2E Tests

## Context

During development of the auth E2E test suite (HSKP-1000), login form submissions were silently failing in Playwright tests. The form appeared to accept input but the submit action had no effect, causing tests to time out waiting for navigation after form submission.

This bug is now **fixed**. This story documents the root cause and resolution for future reference.

---

## Goal

Identify and fix the root cause of login form submission failures in E2E tests so that HSKP-1000, HSKP-1001, and HSKP-1002 can proceed with a reliable test foundation.

---

## Non-Goals

- Changes to authentication business logic
- Changes to the Cognito integration
- Any UI/UX redesign of the login form

---

## Scope

### Root Cause Analysis

Two distinct issues were identified:

**Issue 1: Selector Strategy Mismatch**
The Playwright tests were using CSS class selectors (e.g., `.login-checkbox`) to interact with the "Remember Me" checkbox. The component's CSS classes changed during a refactor, silently breaking the selector. The checkbox click was not registering, leaving react-hook-form in an invalid state that blocked submission.

**Issue 2: Radix Checkbox + react-hook-form Registration**
The "Remember Me" checkbox used a raw Radix `Checkbox` component wired directly to react-hook-form's `register()`. Radix Checkbox does not render a native `<input type="checkbox">`, so `register()` could not detect value changes, causing the field to remain in an uncontrolled state. This meant react-hook-form's `isValid` guard blocked submission even when credentials were correct.

### Resolution Applied

1. **Selector fix**: All Playwright selectors updated to use `getByRole('checkbox', { name: /remember/i })` — semantic, resilient to CSS class changes.

2. **Controller fix**: Wrapped Radix `Checkbox` in react-hook-form's `<Controller>` component, providing explicit `value` and `onChange` wiring. This gives react-hook-form correct field state and unblocks form submission.

## Acceptance Criteria

All criteria are already met (story is completed):

## Implementation Notes

- The `<Controller>` wrapper pattern should be applied to all other Radix form primitives (Select, RadioGroup, etc.) if they are used inside react-hook-form — check `apps/web` for similar patterns
- Document the selector strategy in the Playwright test conventions: always prefer `getByRole` over CSS selectors
