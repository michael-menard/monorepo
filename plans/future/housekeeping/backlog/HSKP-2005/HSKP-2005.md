# HSKP-2005: Scaffold Endpoint Skill

## Context

Creating a new API endpoint currently requires touching multiple files: Lambda handler, Zod schemas, test file, `serverless.yml` entry, and RTK Query hook. This is repetitive, error-prone, and requires knowledge of where each piece lives. Developers frequently miss one of the pieces (e.g., forget to add the `serverless.yml` entry or RTK hook).

With the MCP servers in place (HSKP-2002, HSKP-2003, HSKP-2004), Claude Code now has accurate knowledge of the database schema, existing handler patterns, and current `serverless.yml` structure. This story builds a `/scaffold-endpoint` skill that leverages that context to generate a complete, correct endpoint from a plain-English description.

This story depends on HSKP-2002 (MCP Infrastructure), HSKP-2003 (Drizzle MCP), and HSKP-2004 (Serverless MCP).

---

## Goal

Implement a `/scaffold-endpoint` Claude Code slash command that generates a complete, working Lambda endpoint — handler, Zod schemas, test file, `serverless.yml` entry, and RTK Query hook — from a single natural language description, and presents the generated files as a diff for developer approval before writing.

---

## Non-Goals

- Generating frontend UI components (covered by HSKP-2006)
- Generating database migrations (covered by HSKP-2006)
- Generating endpoints for non-Lambda runtimes
- Automatically committing or deploying generated files
- Generating multi-endpoint features (that is HSKP-2006 scope)

---

## Scope

### Command Interface
```
/scaffold-endpoint "list all wishlist items for current user"
/scaffold-endpoint "bulk delete wishlist items by IDs"
/scaffold-endpoint "get MOC by slug with related parts list"
```

### Generation Workflow
1. **Parse intent** — extract HTTP method, resource, filters, authentication requirements from the description
2. **Query MCP for context**:
   - `list_tables` and `get_table_schema` from Drizzle MCP to identify relevant tables and columns
   - `list_functions` from Serverless MCP to check for existing similar endpoints and avoid path collisions
   - `get_relations` for relevant tables to understand joins needed
3. **Generate code**:
   - Lambda handler (`apps/api/endpoints/{domain}/{name}/handler.ts`) using the portable handler pattern from HSKP-2000
   - Zod request/response schemas (`apps/api/endpoints/{domain}/{name}/schemas.ts`)
   - Test file with happy path, validation error, and auth error fixtures
   - `serverless.yml` entry (function definition + HTTP event)
   - RTK Query hook (`packages/core/api-client/src/hooks/{domain}/{name}.ts`)
4. **Validate** — run `tsc --noEmit` on the generated files to catch type errors before presenting
5. **Present diff** — show the developer all generated files as a unified diff for review and approval

### Skill Definition
- Skill agent file: `.claude/agents/scaffold-endpoint.agent.md`
- Registered as a slash command in Claude Code

## Acceptance Criteria

### Intent Parsing
- [ ] **AC1**: The skill correctly identifies the HTTP method (GET, POST, PUT, DELETE, PATCH) from the description
- [ ] **AC2**: The skill identifies the primary resource/domain from the description
- [ ] **AC3**: The skill identifies whether authentication is required (default: yes, unless description says "public")

### MCP Context Usage
- [ ] **AC4**: The skill calls `get_table_schema` on the relevant table before generating the handler
- [ ] **AC5**: The skill calls `list_functions` to check for path conflicts before generating the `serverless.yml` entry
- [ ] **AC6**: The skill uses `get_relations` when the description implies a join (e.g., "with related parts list")

### Generated Handler
- [ ] **AC7**: Generated handler follows the portable handler pattern: `(request: Request, ctx: HandlerContext) => Promise<Response>`
- [ ] **AC8**: Generated handler uses `@repo/api-services` for database access (not raw Drizzle in the handler)
- [ ] **AC9**: Generated handler has correct authentication guard
- [ ] **AC10**: Generated handler returns structured error responses using shared error classes from `@repo/api-handlers`

### Generated Schemas
- [ ] **AC11**: Zod schemas are generated for the request (path params, query params, body) and the success response
- [ ] **AC12**: Schema field types match the Drizzle schema types returned by `get_table_schema`
- [ ] **AC13**: Schemas follow the Zod-first pattern (no TypeScript interfaces)

### Generated Test File
- [ ] **AC14**: Test file includes a happy path test with a mocked `HandlerContext`
- [ ] **AC15**: Test file includes a validation error test (invalid request input)
- [ ] **AC16**: Test file includes an auth error test (missing/invalid userId)
- [ ] **AC17**: Generated tests are valid Vitest test files

### Generated serverless.yml Entry
- [ ] **AC18**: Function entry includes: function name, handler path, HTTP path, HTTP method, authorizer
- [ ] **AC19**: Function name follows the existing naming convention in `serverless.yml`
- [ ] **AC20**: HTTP path does not conflict with any existing function path (checked via `list_functions`)

### Generated RTK Query Hook
- [ ] **AC21**: RTK Query mutation or query hook is generated matching the endpoint's HTTP method
- [ ] **AC22**: Hook is typed with the generated Zod response schema type
- [ ] **AC23**: Hook file follows the existing api-client hook patterns

### Diff Approval Workflow
- [ ] **AC24**: All generated files are presented as a unified diff before any file is written
- [ ] **AC25**: The developer must explicitly confirm before files are written to disk
- [ ] **AC26**: If the developer declines, no files are modified

### Type Validation
- [ ] **AC27**: Generated files pass `tsc --noEmit` before being presented to the developer
- [ ] **AC28**: If type errors are found in generated code, the skill reports them and attempts one self-correction pass before presenting

### Quality
- [ ] **AC29**: Skill agent file (`scaffold-endpoint.agent.md`) is documented with usage examples
- [ ] **AC30**: A sample invocation (`/scaffold-endpoint "list all wishlist items for current user"`) produces all 5 output artifacts correctly

---

## Implementation Notes

- The skill is implemented as a Claude Code agent (`.agent.md` file), not as application code — it orchestrates Claude's own code generation capabilities
- MCP calls are made inline within the agent's reasoning step
- The `tsc --noEmit` validation step should run in a temp directory to avoid polluting the workspace with partially written files
- If the developer requests changes after seeing the diff, the skill should be able to regenerate with the feedback incorporated
