# HSKP-1001: Forgot Password Tests

## Context

The forgot password flow allows users who cannot remember their credentials to request a password reset email from Cognito. This flow consists of a page where users enter their email address and submit a reset request. While the flow exists in the application, there are no Playwright E2E tests covering it.

This is a sub-story of HSKP-1000 (Auth E2E Test Suite). The fix from HSKP-1003 ensures the login form works correctly in E2E context; this story focuses specifically on the forgot password page.

---

## Goal

Implement a complete set of Playwright E2E tests for the forgot password flow, covering UI element presence, input validation, the happy path (successful reset request), navigation, and button state management.

---

## Non-Goals

- Testing the actual email delivery (Cognito sends the email; we cannot intercept it in E2E)
- Testing the reset password form that the user sees after clicking the email link (covered by HSKP-1002)
- Unit or integration tests for the ForgotPassword component

---

## Scope

### Test Scenarios

**UI Element Verification**
- Page renders with expected heading, email input, and submit button

**Email Validation**
- Empty submission shows required field error
- Invalid email format shows validation error
- Valid email format is accepted

**Happy Path**
- Valid email submission triggers Cognito reset request
- Confirmation message displayed to user

**Navigation**
- "Back to Login" link navigates to the login page
- Page is accessible from the login page via a link

**Button State**
- Submit button is disabled while the request is in flight
- Submit button re-enables after response (success or error)

### Files to Create
- `apps/web/playwright/features/auth/forgot-password.feature`
- `apps/web/playwright/steps/forgot-password.steps.ts`

## Acceptance Criteria

### UI Elements
- [ ] **AC1**: Forgot password page renders a heading identifying the flow (e.g., "Forgot Password" or similar)
- [ ] **AC2**: Email input field is present and accepts text input
- [ ] **AC3**: Submit button is present and labelled clearly

### Validation
- [ ] **AC4**: Submitting with empty email shows a required field error message
- [ ] **AC5**: Submitting with an invalid email format (e.g., "notanemail") shows an email validation error
- [ ] **AC6**: A valid email format (e.g., "user@example.com") passes client-side validation

### Happy Path
- [ ] **AC7**: Entering a valid email and submitting shows a confirmation message (e.g., "Check your email")
- [ ] **AC8**: The test uses a real Cognito test user created via the admin utility so the reset request succeeds
- [ ] **AC9**: Test user is deleted via Cognito admin utility after the test completes

### Navigation
- [ ] **AC10**: A link to "Back to Login" (or equivalent) is present and navigates to the login page
- [ ] **AC11**: The login page contains a link to the forgot password page

### Button State
- [ ] **AC12**: Submit button shows a loading/disabled state while the Cognito request is in flight
- [ ] **AC13**: Button returns to enabled state after a response is received

### Test Quality
- [ ] **AC14**: All selectors use semantic queries (`getByRole`, `getByLabelText`, `getByText`)
- [ ] **AC15**: Tests pass in CI with no flakiness over 3 consecutive runs

---

## Implementation Notes

- Use `cognitoAdmin.createUser()` in `beforeEach` and `cognitoAdmin.deleteUser()` in `afterEach` to manage test user lifecycle
- The happy path test needs a real user in Cognito so the "ForgotPassword" API call does not return `UserNotFoundException`
- Button loading state can be detected by checking `aria-busy="true"` or a disabled attribute on the submit button
