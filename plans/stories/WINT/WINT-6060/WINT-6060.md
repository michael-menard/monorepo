# WINT-6060: Create weekly-analyst Agent (with kb_get_scoreboard MCP registration sub-task)

## Context

The WINT workflow platform has three analytics MCP tools currently registered and callable: `kb_get_token_summary`, `kb_get_bottleneck_analysis`, and `kb_get_churn_analysis`. A fourth tool, `kb_get_scoreboard`, was implemented as part of WINT-3090 in `apps/api/knowledge-base/src/crud-operations/analytics-operations.ts` (lines 392-427) and its input schema `KbGetScoreboardInputSchema` is defined at line 82. However, the tool is absent from `tool-schemas.ts` and `tool-handlers.ts` — it cannot be called by agents.

This story delivers two tightly coupled artifacts:
1. MCP registration of `kb_get_scoreboard` (~50 lines of boilerplate following the established `kb_get_churn_analysis` pattern)
2. Creation of `.claude/agents/weekly-analyst.agent.md` — a sonnet leader agent that calls all four analytics tools plus `kb_search` (for retro lessons), aggregates the results into a structured `WeeklyAnalysis` KB artifact, and surfaces trend summaries, anomaly detection, actionable recommendations, and week-over-week comparisons

The agent is the analytical engine that WINT-6070 (Weekly Report Command) will invoke. It bridges raw telemetry data with human-consumable weekly insights.

A prior elaboration attempt for this story failed with the blocker: "kb_get_scoreboard not MCP-registered — requires prerequisite story WINT-6055". WINT-6055 does not exist in the KB. The resolution is to include MCP registration as a sub-task within this story's scope, eliminating the phantom dependency.

---

## Goal

Register `kb_get_scoreboard` as a callable MCP tool and create the `weekly-analyst.agent.md` leader agent that aggregates all four analytics tools plus KB retro lessons into a structured weekly workflow health artifact.

## Examples

**Example 1 — MCP registration (AC-1, AC-2):**
Given: `kb_get_scoreboard` exists in `analytics-operations.ts` but is absent from `tool-schemas.ts` and `tool-handlers.ts`.
After: An agent calling `kb_get_scoreboard({ start_date: "2026-03-17", end_date: "2026-03-23" })` receives a scoreboard JSON object with `throughput`, `cycle_time`, `first_pass_success`, `cost_efficiency`, and `agent_reliability` fields.

**Example 2 — Agent execution (AC-7, AC-8):**
Given: A user invokes the `weekly-analyst` agent with `week_start_date: "2026-03-17"` and `week_end_date: "2026-03-23"`.
After: The agent calls all four analytics tools with that date range, queries KB for retro lessons, synthesizes results, finds the prior `WeeklyAnalysis` artifact for week-over-week comparison, and writes a new `WeeklyAnalysis` artifact to KB via `kb_write_artifact` containing all 13 required fields.

**Example 3 — Week-over-week comparison (AC-9):**
Given: A `WeeklyAnalysis` KB artifact exists for the prior week.
After: The agent includes a `week_over_week` section in the new artifact showing delta values for throughput, first-pass rate, and cycle time compared to the prior week.

## Edge Cases

**Edge Case 1 — No prior WeeklyAnalysis artifact:**
If no prior `WeeklyAnalysis` artifact is found via `kb_search({ artifact_type: "WeeklyAnalysis" })`, the agent skips the week-over-week comparison section and notes "First week analyzed — no prior baseline available" in the artifact.

**Edge Case 2 — kb_get_scoreboard invoked with no date range:**
All three input parameters (`start_date`, `end_date`, `feature`) are optional. If the agent calls `kb_get_scoreboard({})`, the tool returns a scoreboard across all time. The MCP schema reflects `z.coerce.date().optional()` for date fields.

**Edge Case 3 — Analytics tool returns empty data:**
If any analytics tool returns zero results (e.g., no completed stories in the week), the agent populates the corresponding artifact field with a zero/empty value and notes the gap in the `anomalies` field, rather than failing or omitting the field.

## Architecture Notes

- `kb_get_scoreboard` function signature: `kb_get_scoreboard(deps: AnalyticsDeps, input: KbGetScoreboardInput)` returning composite scoreboard with 5 metric groups.
- `KbGetScoreboardInputSchema` is a Zod schema with three optional fields: `start_date: z.coerce.date().optional()`, `end_date: z.coerce.date().optional()`, `feature: z.string().optional()`.
- The MCP handler (`handleKbGetScoreboard`) must follow the same structure as `handleKbGetChurnAnalysis`: parse input with schema, call function, return `{ content: [{ type: 'text', text: JSON.stringify(result, null, 2) }] }`.
- Leader agents declare `type: leader` and `permission_level: orchestrator` in frontmatter. The `kb_tools` frontmatter lists every MCP tool the agent invokes.
- Output artifacts go to KB via `kb_write_artifact`, not to filesystem paths (per KB lesson fac0d1b6).
- Agent files are exempt from the 45% coverage threshold (KB lesson 6bc29a85). Tests are content-inspection only.
