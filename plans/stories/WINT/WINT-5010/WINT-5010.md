# WINT-5010: Create HiTL Interview Sidecar

## Context

The workflow system captures human-in-the-loop decisions at QA gates, code reviews, and story
approvals via `workflow.hitl_decisions` (Phase 3 telemetry, WINT-3xxx). These records capture
binary decision outcomes but lack the structured interview data — rationale, confidence, alternatives
considered, and risk assessment — needed to train ML classification models.

Existing sidecars at `packages/backend/sidecars/` follow a consistent pattern established by
role-pack (WINT-2010), context-pack (WINT-2020), cohesion (WINT-4010), and rules-registry
(WINT-4020). The ML pipeline MCP tools (WINT-0140) provide `trainingDataIngest` for writing to
`workflow.training_data`.

Port assignment: ports 3090–3093 are taken. This sidecar defaults to port 3094.

WINT-5020 (Classification Agent) and WINT-5040 (ML Training Data Collection) are downstream
dependents — they require structured labeled rows that this sidecar produces.

---

## Goal

Create a sidecar service that captures structured operator interview answers at workflow decision
points and writes them as labeled training examples to `workflow.training_data` for downstream ML
model training.

## Examples

**Example 1 — QA gate interview captured:**
```
Input (POST /hitl-interview):
{
  "storyId": "WINT-5099",
  "phase": "qa",
  "decisionType": "qa_gate",
  "interviewAnswers": {
    "rationale": "All acceptance criteria passed; edge case tests green",
    "confidence": 0.92,
    "alternativesConsidered": "Requested additional test coverage before approving",
    "riskAssessment": "Low — no data-layer changes in this story"
  },
  "featureVector": {
    "storyId": "WINT-5099",
    "phase": "qa",
    "decisionType": "qa_gate",
    "storyComplexityScore": 3,
    "agentPhase": "qa_gate",
    "decisionContext": "10 ACs, all green; 67% coverage"
  }
}

Output (HTTP 200):
{ "success": true, "trainingRowId": "a1b2c3d4-..." }

Side effect: workflow.training_data row inserted with
  dataType = "hitl_interview",
  features = { storyId, phase, decisionType, storyComplexityScore, agentPhase, decisionContext },
  labels = { rationale, confidence, alternativesConsidered, riskAssessment },
  storyId = "WINT-5099"
```

**Example 2 — MCP tool called directly by LLM agent:**
```
Input (MCP tool call):
hitlInterview({
  storyId: "WINT-5099",
  phase: "review",
  decisionType: "code_review"
})
// Agent provides structured answers inline via its context

Output: { success: true, trainingRowId: "..." }
// Compute function called directly — no HTTP self-call (ARCH-001)
```

## Edge Cases

**Edge Case 1 — No prior hitl_decisions context for storyId:**
The sidecar queries `workflow.hitl_decisions` for context enrichment using storyId + decisionType.
If no matching rows exist, `decisionContext` defaults to an empty string (not null/undefined).
The training row is still written successfully.

**Edge Case 2 — trainingDataIngest returns null (DB failure):**
`trainingDataIngest` returns null when the DB insert fails. The HTTP handler must detect this and
respond with HTTP 500 `{ "error": "Failed to write training row" }` and log a warning via
`@repo/logger`. The sidecar must NOT retry — callers are responsible for retry logic.

**Edge Case 3 — confidence boundary values:**
`confidence` is a float in [0, 1]. Values of exactly 0 or 1 are valid and accepted. Values outside
this range (e.g., -0.1 or 1.1) are rejected by `InterviewAnswersSchema` with a 422 response.

**Edge Case 4 — All three decisionType enum values must work:**
`qa_gate`, `code_review`, and `story_approval` are the only valid values. Any other string (e.g.,
`"unknown_type"`) fails Zod validation and returns 422.

## Non-Goals

- Writing to `workflow.hitl_decisions` — this sidecar only reads from it for context enrichment
- Providing a web UI for operators — operator prompting is MCP-tool-driven (LLM agent context)
- Training or evaluating the ML model itself — that is WINT-5020's scope
- Marking training rows as validated — handled by `trainingDataMarkValidated` (WINT-0140)
- Real-time streaming or websocket interface
- Authentication/authorization for the HTTP endpoint
- Querying `hitl_decisions` by anything other than storyId + decisionType

## Architecture Notes

### ARCH-001: MCP Tool Must Call Compute Function Directly

Per the cohesion sidecar precedent, the `hitlInterview` MCP tool in
`src/mcp-tools/hitl-interview.ts` must call the compute function `performHitlInterview` directly
via import — NOT by issuing an HTTP request to `localhost:3094`. This allows unit testing without
requiring the HTTP server to be running.

```typescript
// CORRECT (ARCH-001)
import { performHitlInterview } from '../hitl-interview.js'
export async function hitlInterview(input) {
  return performHitlInterview(input)
}

// WRONG — do not do this
const res = await fetch('http://localhost:3094/hitl-interview', ...)
```

### ARCH-002: No Cross-Sidecar HTTP Calls

Sidecars must not issue HTTP calls to other sidecars. Any shared compute must be done through
shared packages (`@repo/mcp-tools`, `@repo/db`, etc.).

### ARCH-003: Port Registry

```
3090 — role-pack         (WINT-2010)
3091 — context-pack      (WINT-2020)
3092 — cohesion          (WINT-4010)
3093 — rules-registry    (WINT-4020)
3094 — hitl-interview    (WINT-5010)  ← this story
```

### ARCH-004: Zod Validation Before DB Write

`InterviewAnswersSchema.parse()` and `FeatureVectorSchema.parse()` must be called in the compute
function before `trainingDataIngest` is invoked. Any validation failure must surface as a 422
(HTTP) or thrown error (MCP tool) — never reach the DB with unvalidated data.

### ARCH-005: hitl_decisions is READ-ONLY in this sidecar

The sidecar queries `workflow.hitl_decisions` using a SELECT filtered by `storyId` and
`decisionType` to populate `decisionContext` in the feature vector. It performs no INSERT, UPDATE,
or DELETE on that table under any circumstances.

```json
    "rationale": "string (required, min 1)",

    "storyComplexityScore": 3,
    "agentPhase": "qa_gate",

}
```

**Responses:**

| Status | Body | Condition |
|--------|------|-----------|
| 200 | `{ "success": true, "trainingRowId": "<uuid>" }` | Training row written |
| 400 | `{ "error": "..." }` | Malformed JSON body |
| 404 | `{ "error": "Not found" }` | Unknown path |
| 405 | `{ "error": "Method not allowed" }` | Non-POST method |
| 422 | `{ "error": "Validation error", "details": [...] }` | Zod schema failure |
| 500 | `{ "error": "Failed to write training row" }` | trainingDataIngest returns null |
