# WINT-9010: Create Shared Business Logic Package — Extend @repo/workflow-logic with Decision, Artifact, Context, and Token Modules

## Context

The `@repo/workflow-logic` package currently provides shared status transitions, evidence classification, story ID validation, and directory mapping — pure functions with Zod schemas and zero runtime dependencies. It is already a proven cross-environment package: both Claude Code agents (via direct import) and LangGraph nodes (e.g., `nodes/qa/evidence-judge.ts`) consume it successfully.

However, four additional categories of business logic remain stranded: either as prose in `.claude/agents/_shared/` markdown files, or as inline logic inside individual LangGraph nodes without extraction into the shared package. These are:

1. **Decision classification** — the 5-tier system (Clarification, Preference, Ambiguous, Destructive, External) currently defined only in `decision-handling.md` as prose
2. **Artifact management strategies** — rules governing which artifact type maps to which phase, and validation of artifact/phase combinations; currently prose in `kb-integration.md`
3. **KB context retrieval helpers** — canonical query string patterns for domain/task/role combinations; currently prose in `kb-integration.md`
4. **Token tracking** — the in/out accounting protocol defined in `token-tracking.md`; currently prose only

No active stories are modifying `packages/backend/workflow-logic/` or `.claude/agents/_shared/`, so there is no overlap risk with current in-progress work (WINT-8060, WINT-3090).

---

## Goal

Extend `@repo/workflow-logic` with four new pure-function modules — `decision/`, `artifact/`, `context/`, and `token/` — to establish a single authoritative TypeScript source of truth that both Claude agents and LangGraph nodes can import, eliminating behavioral drift between prose-guided and code-driven execution environments.

---

## Examples

**Decision classification:**
Given a description `"drop the users table"`, `classifyDecisionTier()` returns `"destructive"`. Given `"rename this file"`, it returns `"clarification"`. Combined with `shouldEscalate("destructive", "aggressive")`, the result is `true` — confirming that destructive decisions always escalate regardless of autonomy level.

**Token summary formatting:**
Given `{ inputTokens: 12000, outputTokens: 3200, model: "claude-sonnet-4-6", agentName: "dev-agent" }`, `formatTokenSummary()` returns `"In: ~12000 / Out: ~3200"` — the canonical format consumed by the `/token-log` command.

**Artifact phase validation:**
Given `getArtifactPhase("pm_test_plan")` returns `"planning"`, and `isValidArtifactForPhase("pm_test_plan", "planning")` returns `true`, while `isValidArtifactForPhase("pm_test_plan", "development")` returns `false`.

**Context query building:**
Given `buildContextQuery("wishlist", "story-generation", "pm")` returns a canonical query string like `"wishlist story-generation patterns pm"` ready for `kb_search`.

---

## Edge Cases

**classifyDecisionTier with ambiguous input:** An empty string or a description that matches no tier keyword should return a safe default tier (`"clarification"`) without throwing — unknown descriptions should not crash callers.

**estimateTokenCount with multi-byte characters:** Unicode characters (e.g., emoji, accented letters) occupy more than 1 byte each. The implementation must use `Buffer.byteLength()` or equivalent rather than `string.length` to avoid under-estimating token count.

**shouldEscalate with Tier 4 (destructive) at any autonomy level:** The escalation matrix in `decision-handling.md` specifies that Tier 4 always escalates regardless of autonomy level. The pure function must encode this as an unconditional rule, not a lookup that could accidentally return `false` for an unrecognized autonomy value.

**z.record() single-argument form:** Using `z.record(z.unknown())` compiles to an error in strict mode (KB lesson INFR-0040). Any map/record type in new modules must use `z.record(z.string(), valueSchema)`.

---

## Non-Goals

- Do NOT port LangGraph node infrastructure (state types, `createToolNode`, circuit breakers) into `workflow-logic`
- Do NOT move or replace `_shared/*.md` prose files — they remain the source of truth for human agents; `workflow-logic` is the TypeScript encoding
- Do NOT extract the full decision-handling runtime flow (batch mode queue, escalation formatting, KB logging calls) — only the pure classification predicates
- Do NOT add DB access, HTTP calls, or file I/O to any `workflow-logic` module
- Do NOT create new barrel files (`index.ts` that purely re-exports from multiple sub-modules without adding anything)
- Do NOT modify existing `workflow-logic` modules unless needed to integrate new exports into `index.ts`
- Do NOT update the orchestrator LangGraph nodes to consume the new modules — that is a follow-on story (WINT-9020 or similar)

- [ ] **AC-8**: `pnpm test` passes for `@repo/workflow-logic` with at minimum 60% coverage for the new modules.

- [ ] **AC-9**: `pnpm check-types` passes for both `@repo/workflow-logic` and `@repo/orchestrator` (confirming no import breaks).

- [ ] **AC-10**: `@repo/workflow-logic`'s `package.json` `dependencies` list does not grow — no new runtime dependencies added.

## Architecture Notes

### Module Pattern (from evidence-judge/ template)

Each new module follows this structure:

```
src/{module}/index.ts
  // ====================================================================
  // Schemas (Zod, at the top)
  // ====================================================================
  export const FooSchema = z.enum([...])
  export type Foo = z.infer<typeof FooSchema>

  // ====================================================================
  // Constants / decision tables
  // ====================================================================
  const FOO_TABLE = { ... } as const

  // ====================================================================
  // Pure functions
  // ====================================================================
  export function doFoo(input: Foo): Bar { ... }
```

### Scope Boundary for decision/ Module

Extract from `_shared/decision-handling.md`:
- **EXTRACT**: tier enum (5 values), escalation matrix table (`Tier x AutonomyLevel → boolean`), batch mode decision flags
- **DO NOT EXTRACT**: KB logging calls, `AskUserQuestion` invocation, batch queue management, escalation message formatting

### Zero-Dependency Constraint

`@repo/workflow-logic` must remain dependency-free of: MCP SDK, LangGraph, AWS services, DB clients. `zod` is already a dependency. `@repo/logger` is already a dependency (available but not required for pure modules).

### z.record() Rule (KB lesson INFR-0040)

Always use `z.record(z.string(), valueSchema)` — never `z.record(valueSchema)` (single-argument form fails compilation in strict mode).
