# WINT-6070: Create /weekly-report Skill (CLI Command for Weekly Analyst Agent)

## Context

The `weekly-analyst.agent.md` agent (delivered by WINT-6060) aggregates telemetry from all four analytics tools (`kb_get_scoreboard`, `kb_get_bottleneck_analysis`, `kb_get_churn_analysis`, `kb_get_token_summary`) plus KB retro lessons into a structured `WeeklyAnalysis` KB artifact. The agent is the analytical engine, but it has no user-facing entry point ŌĆö it can only be invoked by other agents that know its invocation contract.

This story creates the thin CLI wrapper skill at `.claude/skills/weekly-report/SKILL.md` that allows operators to trigger the weekly analysis from the Claude Code command line with a single command. It follows the same delegation pattern established by `/batch-process` (WINT-6020): thin CLI wrapper ŌåÆ read agent file ŌåÆ build CONTEXT block ŌåÆ spawn foreground Task ŌåÆ await signal ŌåÆ surface result.

The deliverable is a single Markdown file. No TypeScript source files, no DB migrations, no npm packages, no frontend components.

**Prerequisite**: WINT-6060 must complete before this skill can be exercised end-to-end. `weekly-analyst.agent.md` already exists on disk; `kb_get_scoreboard` MCP registration must be confirmed per WINT-6060 AC-1 and AC-2.

---

## Goal

Create a `/weekly-report` CLI skill that allows operators to trigger the weekly analytics agent from the command line and receive a formatted markdown report derived from the persisted `WeeklyAnalysis` KB artifact.

---

## Examples

**Example 1: Current week report (no args)**

Given: Operator runs `/weekly-report` on 2026-03-23 (Monday).

The skill derives `week_of = "2026-03-23"` (today is already Monday), reads `weekly-analyst.agent.md`, spawns the agent in foreground mode with `week_of: "2026-03-23"`, awaits the `WEEKLY ANALYSIS COMPLETE: 2026-03-23 ŌĆö 2 anomalies, 3 recommendations` signal, calls `kb_read_artifact({ story_id: "SYSTEM", artifact_type: "analysis", artifact_name: "WEEKLY-ANALYSIS-2026-03-23" })`, and renders:

```
## Weekly Report: 2026-03-23

### Scoreboard Snapshot

| Metric                     | This Week | Prior Week | Delta |
| -------------------------- | --------- | ---------- | ----- |
| Throughput (stories/week)  | 3.5       | 4.0        | -0.5  |
| Avg Cycle Time (days)      | 2.1       | 1.8        | +0.3  |
| First-Pass Rate            | 71%       | 75%        | -4%   |
| Avg Cost Per Story ($)     | 0.042     | 0.038      | +0.004|

### Anomalies (2)
...

### Recommendations (3)
...
```

**Example 2: Specific week, feature-scoped**

Given: Operator runs `/weekly-report --week=2026-03-17 --feature=WISH`.

The skill uses the explicit `week_of = "2026-03-17"` and passes `feature: "WISH"` to the agent CONTEXT block. The agent filters all analytics queries to the WISH feature prefix. The rendered report shows only WISH-scoped metrics.

**Example 3: Force regeneration of existing analysis**

Given: Operator runs `/weekly-report --week=2026-03-17 --force`.

The skill passes `force: true` in the CONTEXT block. The agent overwrites the existing `WEEKLY-ANALYSIS-2026-03-17` artifact even if it already exists. Useful for rerunning with updated data after backfill.

## Edge Cases

**Edge Case 1: Agent returns WEEKLY ANALYSIS FAILED**

Given: The agent encounters an error (e.g., `kb_get_scoreboard` MCP not registered because WINT-6060 is incomplete).

The skill receives `WEEKLY ANALYSIS FAILED: kb_get_scoreboard is not a registered MCP tool`. It surfaces the full failure message verbatim to the operator. It does NOT call `kb_read_artifact`. It does NOT crash. No partial report is shown.

**Edge Case 2: `kb_read_artifact` returns null after WEEKLY ANALYSIS COMPLETE**

Given: The agent returns `WEEKLY ANALYSIS COMPLETE: 2026-03-23 ŌĆö 0 anomalies, 1 recommendation` but a subsequent `kb_read_artifact` call returns null (e.g., artifact was written to a different name or KB is temporarily unavailable).

The skill shows:

```
Weekly analysis completed: WEEKLY ANALYSIS COMPLETE: 2026-03-23 ŌĆö 0 anomalies, 1 recommendation

WeeklyAnalysis artifact could not be loaded for display ŌĆö check KB directly.
Run: kb_read_artifact({ story_id: "SYSTEM", artifact_type: "analysis", artifact_name: "WEEKLY-ANALYSIS-2026-03-23" })
```

**Edge Case 3: --week omitted and today is a Monday**

Given: Operator runs `/weekly-report` on 2026-03-23 (a Monday). `Date.getDay()` returns 1 (Monday = 1 in JS 0-indexed Sunday-start system).

Formula: `daysToSubtract = (day + 6) % 7 = (1 + 6) % 7 = 0`. The derived `week_of` is today's date, not the previous Monday. This is the correct behaviour ŌĆö the skill derives the most recent Monday, which today IS.

**Edge Case 4: anomalies array is empty**

Given: The artifact's `anomalies` array is `[]`.

The anomalies section renders:

```
### Anomalies
No anomalies detected.
```

Rather than an empty section or omitted section.

## Architecture Notes

- Skills are Markdown files only. No TypeScript source files are produced.
- The analytical logic lives entirely in `weekly-analyst.agent.md`. The skill is purely orchestration.
- `kb_read_artifact` MUST be called in the main conversation, not inside a Task tool sub-agent. Sub-agents spawned from skills do not have reliable MCP access (lesson from WINT-6020).
- Token logging must not be duplicated. `weekly-analyst.agent.md` handles telemetry internally. The skill must include a non-negotiable explicitly prohibiting a second `/token-log` call (lesson from WINT-6030).
- The `--week` date derivation algorithm must handle the Monday edge case: when today is Monday, `daysToSubtract = (day + 6) % 7 = 0`, so `week_of = today`. JS `Date.getDay()` is 0-indexed Sunday-start (Sunday=0, Monday=1).
- All WeeklyAnalysis output lives in KB artifacts, never in flat files.
