# ORCH-4020: E2E Pipeline Tests — Bootstrap to Story Creation to Elaboration

## Context

The LangGraph orchestrator package contains 23 graph types covering the full autonomous pipeline. As of 2026-03-18, these graphs have never been run end-to-end. ORCH-4020 is the capstone test story for the `autonomous-pipeline-test-plan` feature — it validates that `bootstrap → story_creation → elaboration` compose correctly as a multi-graph pipeline when LLM providers and KB persistence are mocked out.

**Predecessor chain**: ORCH-1010 (mock factories) → ORCH-2010 (elaboration edge routing unit tests) → ORCH-3010 (elaboration delta cluster integration) → ORCH-4010 (full graph tests for elab + story-creation individually) → **ORCH-4020** (E2E three-graph pipeline chain).

**Problem**: No test exists that exercises the full pipeline from `runBootstrap` through `runStoryCreation` to `runElaboration`. The integration between these three graphs — particularly the data handoff from `StoryCreationResult.synthesizedStory` to `runElaboration`'s `currentStory` parameter — has never been verified. A first automated proof-of-life run against ORCH stories cannot be trusted until this handoff is confirmed testable.

**Active in-progress work**:

- ORCH-2010: Unit tests for elaboration.ts edge routing (low overlap)
- ORCH-3010: Integration tests for elaboration delta cluster (low overlap — different file)
- ORCH-4010: Full graph tests for elaboration and story-creation individually (medium — direct predecessor; implement ORCH-4020 after ORCH-4010 merges)

---

## Goal

Establish a Vitest integration test suite that proves the three-graph pipeline (`runBootstrap` → `runStoryCreation` → `runElaboration`) composes correctly with all external I/O (LLM and DB) mocked, providing an automated proof-of-life for the autonomous pipeline before any live run.

---

## Examples

**Example 1 — Bootstrap handoff (happy path)**:
Given `runBootstrap({ storyId: 'ORCH-TEST-001', storyRequest: createPipelineTestStoryRequest(), config: { persistToDb: false, requireHiTL: false } })` with `story-creation.js` mocked at module level to return a valid `StoryCreationResult` shape, the system returns `BootstrapResult` with `success: true`, `storyId === 'ORCH-TEST-001'`, `storyCreationResult` non-null, and `errors: []`.

**Example 2 — Story-creation → elaboration chain (happy path)**:
Given `runStoryCreation(createPipelineTestStoryRequest(), null, { persistToDb: false, requireHiTL: false, autoApprovalThreshold: 0 })` with story-creation graph mocked to return a valid `SynthesizedStory`, feeding that story to `runElaboration(synthesizedStory, null, { persistToDb: false, recalculateReadiness: false })` produces `ElaborationResult` with `success: true`.

## Edge Cases

**Edge Case 1 — Bootstrap error path**:
Given `runBootstrap` with the story-creation mock configured to throw `Error('story-creation failure')`, the function returns `BootstrapResult` with `success: false` and a non-empty `errors` array. No uncaught exception propagates.

**Edge Case 2 — Empty storyId**:
Given `runBootstrap({ storyId: '', ... })`, the initialize node detects the missing storyId and returns `BootstrapResult` with `success: false` and `errors: ['No story ID provided for bootstrap']`. No uncaught exception.

**Edge Case 3 — synthesizedStory null from story-creation** _(covered indirectly via bootstrap error path)_:
The null-synthesizedStory scenario is covered by EC-1 (bootstrap with throwing story-creation mock): when story-creation fails, `BootstrapResult.storyCreationResult` is null and `success: false`. A direct call `runElaboration(null, ...)` is a TypeScript type error (parameter is non-nullable `SynthesizedStory`) and MUST NOT be written as a standalone test. EC-3 is satisfied by the bootstrap error path in AC-4. (Updated by autonomous elaboration: GAP-2 resolution.)

**Edge Case 4 — 10-second timing boundary**:
All integration tests in the suite complete within 10 seconds total when all external I/O (LLM, DB) is mocked. No test timeout occurs.

## Architecture Notes

**Key discovery from source inspection**: `StoryCreationConfigSchema` does not include an `llmProvider` injectable field. LLM-dependent nodes (seed, fanout-pm, fanout-ux, fanout-qa, attack, synthesize) call LLM internally without config-level injection. This means AC-3's two-graph chain test must use one of two strategies:

1. **Preferred**: Mock the entire `story-creation.js` module at the top level (same as AC-2 bootstrap strategy). This tests graph handoff shape, not internal story-creation logic. Story-creation's internal behavior is covered by ORCH-4010.
2. **Fallback**: Use `vi.mock` for each individual LLM-dependent node file — higher maintenance burden.

The preferred strategy means the test file may need `vi.resetModules()` between test groups, or splitting into two files:

- `pipeline-e2e-bootstrap.integration.test.ts` (bootstrap handoff with story-creation mocked)
- `pipeline-e2e-chain.integration.test.ts` (story-creation mocked → elaboration chain)

Dev agent should read `bootstrap.ts` lines 184–238 (`createRunStoryCreationNode`) and `bootstrap.test.ts` for the exact `vi.mock` structure before writing tests.

**Config flags**:

- `persistToDb: false` — disables `load_from_db`, `save_to_db`, `persist_learnings` nodes
- `requireHiTL: false` — bypasses HiTL pause in bootstrap config
- `autoApprovalThreshold: 0` — ensures story-creation skips HiTL by auto-approving at any readiness score
- `recalculateReadiness: false` — skips `update_readiness` node in elaboration

**BootstrapResult shape** (from source):

```typescript
{
  storyId: string           // min(1)
  success: boolean
  storyCreationResult: unknown | null
  durationMs: number
  completedAt: string       // datetime
  errors: string[]
  warnings: string[]
}
```
