# PIPE-2020: Dispatch Router: Test Coverage for Dev/Review/QA Branches + Type-Cast Elimination

## Context

The `dispatch-router.ts` supervisor module has five pipeline stage branches (`elaboration`, `story-creation`, `implementation`, `review`, `qa`). The first two stages were implemented in APIP-0020 and have thorough test coverage (634-line test file, all error paths covered). The remaining three stages — `implementation`, `review`, and `qa` — were added in subsequent stories and are fully written in `dispatch-router.ts` lines 223–244, but have **zero test coverage** in `dispatch-router.test.ts`.

Additionally, the review branch contains an unsafe Zod type cast (`as StorySnapshotPayload & { worktreePath?: string; featureDir?: string }`) that bypasses type safety because `ReviewJobDataSchema.payload` does not include `worktreePath` and `featureDir` fields. PIPE-2010 will add `ReviewPayloadSchema` to `@repo/pipeline-queue` to enable elimination of this cast.

## Goal

Extend `dispatch-router.test.ts` to provide comprehensive test coverage for the three untested dispatch branches (`implementation`, `review`, `qa`), extend `createMockRunners()` to satisfy the full `GraphRunners` interface, and — after PIPE-2010 is merged — eliminate the unsafe type cast in the review branch by using the typed `ReviewPayload` directly.

## Examples

**Example 1: Implementation job routes to runDevImplement()**

Given a BullMQ job enqueued with:
```json
{
  "storyId": "PIPE-2020",
  "stage": "implementation",
  "attemptNumber": 1,
  "payload": {}
}
```

The system produces:
- `dispatchJob()` calls `runners.runDevImplement({ storyId: 'PIPE-2020', attempt: 1 })`
- `runners.runElaboration`, `runners.runStoryCreation`, `runners.runReview`, `runners.runQAVerify` are never called
- A `completed` log event is emitted containing `storyId`, `stage: 'implementation'`, `threadId`, `attemptNumber`, and `durationMs`

**Example 2: Review job routes to runReview() with payload fallbacks**

Given a BullMQ job enqueued with:
```json
{
  "storyId": "PIPE-2020",
  "stage": "review",
  "attemptNumber": 2,
  "payload": {
    "storyId": "PIPE-2020",
    "title": "Dispatch Router Review Branch",
    "worktreePath": "/tmp/worktrees/PIPE-2020",
    "featureDir": "plans/future/platform/pipeline-orchestrator-activation"
  }
}
```

The system produces:
- `dispatchJob()` calls `runners.runReview({ storyId: 'PIPE-2020', worktreePath: '/tmp/worktrees/PIPE-2020', featureDir: 'plans/future/platform/pipeline-orchestrator-activation', attempt: 2 })`
- No other runners are called
- `completed` log event emitted with all required fields

## Edge Cases

**Edge Case 1: Hanging runner triggers wall clock timeout**

Given a `qa` job where `runners.runQAVerify` is a `vi.fn()` that never resolves, and `stageTimeoutMs` is set to 1000ms, and timers are advanced by 2000ms:

The system produces:
- `dispatchJob()` rejects with a `wall_clock_timeout` error
- A `timeout` lifecycle log event is emitted with the correct `threadId` for the `qa` stage

**Edge Case 2: Circuit breaker trips for implementation branch**

Given the circuit breaker `failureThreshold` is set to 1, and one `implementation` dispatch has already failed, and a second `implementation` job is dispatched:

The system produces:
- `job.moveToDelayed()` is called for the implementation circuit (not the elaboration circuit)
- A `circuit_open` lifecycle log event is emitted

**Edge Case 3: Review job without worktreePath falls back to empty string**

Given a `review` job payload without `worktreePath` or `featureDir` fields:

The system produces:
- `runners.runReview` is called with `worktreePath: ''` and `featureDir: 'plans/future/platform'`
- No error is thrown — the fallback values are used silently

## Architecture Notes

### Injectable Runners Pattern

`dispatchJob(job, data, config, runners?)` accepts an optional `runners` parameter that, when provided, overrides the production `loadGraphRunners()` call. This is the established test seam. Do NOT use `vi.mock('./graph-loader')` — the injectable parameter is the correct approach per APIP-0020 learnings.

### createMockRunners() Extension

The current function returns only `runElaboration` and `runStoryCreation` stubs. TypeScript will flag incomplete `GraphRunners` objects when the three new runners are absent. The extension adds:

```typescript
const devImplementSpy = vi.fn().mockResolvedValue(MockDevImplementResult)
const reviewSpy = vi.fn().mockResolvedValue(MockReviewResult)
const qaVerifySpy = vi.fn().mockResolvedValue(MockQaResult)

return {
  runElaboration: elaborationSpy,
  runStoryCreation: storyCreationSpy,
  runDevImplement: devImplementSpy,
  runReview: reviewSpy,
  runQAVerify: qaVerifySpy,
  elaborationSpy,
  storyCreationSpy,
  devImplementSpy,
  reviewSpy,
  qaVerifySpy,
  ...overrides,
}
```

### Thread ID Convention (IRREVERSIBLE ADR)

Thread IDs follow the format `{storyId}:{stage}:{attemptNumber}`. Tests asserting on `threadId` must use this exact format (e.g., `'PIPE-2020:implementation:1'`). This convention is serialized in LangGraph checkpoints and cannot be changed post-Phase 0.

### Type Cast Removal (Gated on PIPE-2010)

Before (unsafe — current state):
```typescript
const payload = data.payload as StorySnapshotPayload & {
  worktreePath?: string
  featureDir?: string
}
graphPromise = graphRunners.runReview({
  storyId: data.storyId,
  worktreePath: payload.worktreePath ?? '',
  featureDir: payload.featureDir ?? 'plans/future/platform',
  attempt: data.attemptNumber,
})
```

After (typed — post PIPE-2010):
```typescript
// data.payload is now typed as ReviewPayload — no cast needed
graphPromise = graphRunners.runReview({
  storyId: data.storyId,
  worktreePath: data.payload.worktreePath ?? '',
  featureDir: data.payload.featureDir ?? 'plans/future/platform',
  attempt: data.attemptNumber,
})
```

### worktreePath Fallback Behavior

The review branch uses `payload.worktreePath ?? ''` and `payload.featureDir ?? 'plans/future/platform'`. Tests must cover both the with-payload and without-payload variants to ensure the fallback paths are exercised. The hardcoded `'plans/future/platform'` default is a known maintainability risk (PIPE-2010/Opp-2) but extraction is out of scope here.
