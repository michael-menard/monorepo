# PIPE-2040: Finish-Before-New-Start and Double-Dispatch Prevention

## Context

The `SchedulerLoop` class in `apps/api/pipeline/src/scheduler/index.ts` implements two partial guards today:

1. **Structural in-progress filter**: `getEligibleStories()` queries `WHERE s.state = 'ready'` — stories already in `in_progress` are never returned for dispatch.
2. **Atomic KB advance (F006)**: `dispatchStory()` calls `kb_update_story_status` to advance state to `in_progress` BEFORE calling `queue.add()`. If the KB advance returns `updated: false`, the BullMQ enqueue is skipped.
3. **Finish-before-new-start priority (F005)**: `applyFinishBeforeNewStart()` reorders the dispatch queue so stories from plans with `in_progress` siblings are promoted to the front.

Two gaps remain:

1. **No BullMQ-level deduplication**: `queue.add()` in `dispatchStory()` does not pass a `jobId`. Without `jobId`, a crash between KB advance and BullMQ enqueue could leave a story in `in_progress` KB state with no BullMQ job — it would never be re-dispatched. With `jobId`, a recovery retry can safely re-call `queue.add()` with the same ID and BullMQ will silently deduplicate.
2. **Finish-before-new-start is priority ordering only, not enforcement**: `applyFinishBeforeNewStart()` promotes in_progress sibling stories to the front but does not prevent new-plan stories from being dispatched in the same cycle.

---

## Goal

Harden the scheduler's dispatch guarantees by adding BullMQ `jobId` deduplication to prevent ghost jobs on crash recovery, and adding a configurable strict finish-before-new-start enforcement mode that defers new-plan work while existing plans have in_progress stories.

---

## Examples

**Example 1 — jobId deduplication on crash recovery:**
Given a story `PIPE-099` was KB-advanced to `in_progress` but the process crashed before `queue.add()` completed. On restart, the scheduler detects no active BullMQ job and calls `dispatchStory()` again. With `jobId: 'PIPE-099:implementation:1'`, `queue.add()` succeeds and produces a single job (BullMQ deduplicates if a job with that ID already exists). Without `jobId`, a second enqueue would result in a duplicate job running concurrently.

**Example 2 — strictFinishBeforeNewStart enforcement:**
Given plan-X has story STORY-A currently `in_progress` and story STORY-B is `ready`. Plan-Y has story STORY-C that is `ready` with no in_progress siblings. With `strictFinishBeforeNewStart: true`, only STORY-B is eligible in the current poll cycle (it belongs to a plan with active work). STORY-C is deferred to the next cycle. With `strictFinishBeforeNewStart: false` (default), both STORY-B and STORY-C are dispatched in the same cycle.

---

## Edge Cases

**Edge case 1 — F006 double-dispatch prevention (existing, must be preserved):**
If `kb_update_story_status` returns `{ updated: false }` (another process already advanced the story to `in_progress`), `dispatchStory()` returns early without calling `queue.add()`. The new `jobId` addition must not alter this early return path.

**Edge case 2 — strictFinishBeforeNewStart with no in_progress work anywhere:**
If `strictFinishBeforeNewStart: true` is configured but no plan has any `in_progress` stories, all eligible `ready` stories from any plan are dispatched normally. The strict filter only activates when at least one plan has active in_progress work.

## Architecture Notes

**jobId deduplication strategy:**
BullMQ supports job ID-based deduplication: if a job with the same `jobId` is already active, waiting, or delayed in the queue, a new `add()` call with the same `jobId` is silently ignored. The `jobId` format `{storyId}:implementation:{attemptNumber}` aligns with the existing thread ID convention established in APIP-0020 (post-Phase 0, IRREVERSIBLE). This provides cross-process crash recovery safety complementing the KB-level F006 guard.

**strictFinishBeforeNewStart implementation approach:**
The existing `applyFinishBeforeNewStart()` method (lines 195–231 of `scheduler/index.ts`) already has the `hasSiblingInProgress` Set built from a raw SQL query. The strict filter can reuse this data: if `hasSiblingInProgress.size > 0` (any eligible story has an in_progress plan sibling), filter the eligible list to only those in `hasSiblingInProgress`. This avoids a second DB query and makes `strictFinishBeforeNewStart` a deterministic filter on top of the existing reordering logic.

**Why not change the SQL filter:**
The `getEligibleStories()` SQL `WHERE s.state = 'ready'` is the primary structural guard and is correct. TypeScript-level filtering on top of the SQL results is appropriate for policy enforcement that may evolve (e.g., toggling strictness via config) without requiring SQL migrations.
