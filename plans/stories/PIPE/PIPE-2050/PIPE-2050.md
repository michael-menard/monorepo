# PIPE-2050: Scheduler Loop: Review and QA Dispatch Branches

## Context

The `SchedulerLoop` in `apps/api/pipeline/src/scheduler/index.ts` currently polls for stories in `state = 'ready'` and dispatches them as `implementation` stage jobs to BullMQ. The implementation dispatch path is complete and hardened: F006 atomic state advance before enqueue, jobId deduplication (PIPE-2040), finish-before-new-start ordering (PIPE-2040), and strict-finish filtering.

However, the scheduler has no mechanism to advance stories past the implementation phase. After a story completes implementation (advancing to `needs_code_review`) and a code review artifact is submitted (advancing to `ready_for_qa`), there is no automated process to enqueue a `review` stage job. Similarly, after a story reaches `in_qa`, no `qa` stage job is enqueued. The `dispatch-router.ts` in the supervisor already routes `review` and `qa` stage jobs to `runReview()` and `runQAVerify()` respectively — but it only receives those jobs if something enqueues them.

The pipeline stalls after implementation. This story closes that gap.

**Dependencies confirmed complete**: PIPE-2010 (`ReviewPayloadSchema`/`QaPayloadSchema` in `@repo/pipeline-queue`), PIPE-2020 (`dispatch-router.ts` review/QA routing), PIPE-2040 (`strictFinishBeforeNewStart` config field, jobId pattern).

**Not yet complete**: PIPE-2030 (completion/failure callbacks) — not a blocker for this story but required for full end-to-end pipeline flow in PIPE-4010.

---

## Goal

Extend `SchedulerLoop.runOnce()` to poll for stories in `ready_for_qa` and `in_qa` states and dispatch them as `review` and `qa` stage BullMQ jobs respectively, following the same F006 atomic dispatch pattern already established for implementation dispatch.

---

## Examples

**Example 1 — Review dispatch happy path**:
Given a story `PIPE-099` is in state `ready_for_qa` with no unresolved dependencies, and `maxConcurrent = 5` with 2 active jobs, when `runOnce()` is called, the scheduler calls `kb_update_story_status({ story_id: 'PIPE-099', state: 'in_qa' })` first, then calls `queue.add('story-PIPE-099', reviewJobData, { jobId: 'PIPE-099:review:1' })`. The story is now in `in_qa` with a queued review job.

**Example 2 — QA dispatch happy path**:
Given a story `PIPE-100` is in state `in_qa` (already advanced by a prior review dispatch or callback) and the QA dispatch branch polls `in_qa` stories, when `runOnce()` is called, the scheduler advances state and calls `queue.add('story-PIPE-100', qaJobData, { jobId: 'PIPE-100:qa:1' })`.

**Example 3 — Multi-stage dispatch in one cycle**:
Given three stories simultaneously eligible — one `ready`, one `ready_for_qa`, one at QA-trigger state — and `maxConcurrent = 10`, `runOnce()` dispatches all three in a single cycle: one as `implementation`, one as `review`, one as `qa`.

---

## Edge Cases

**Edge Case 1 — F006 preservation for review dispatch**:
If `kb_update_story_status` returns `updated: false` for a `ready_for_qa` story (another process already advanced it), `dispatchReviewStory()` logs a warning and returns without calling `queue.add()`. The story remains in its current state and will be re-evaluated in the next poll cycle.

**Edge Case 2 — Slot ceiling shared across all stages**:
If `maxConcurrent = 3` and 3 implementation jobs are already in-flight, `slotsAvailable = 0` and `runOnce()` returns early — no review or QA stories are dispatched regardless of eligibility. All stages share a single BullMQ capacity ceiling.

**Edge Case 3 — strictFinishBeforeNewStart applies equally to review/QA**:
With `strictFinishBeforeNewStart: true` and an in-progress implementation story from Plan A, a new `ready_for_qa` story from Plan B (with no in-progress siblings) is deferred — the strict filter applies to all dispatch stages uniformly.

**Edge Case 4 — Zero eligible stories for any stage**:
If no stories are in `ready_for_qa` or the QA-trigger state, `runOnce()` completes without error. Each dispatch branch handles empty eligible lists gracefully.

## Architecture Notes

**F006 Atomic Dispatch Contract (IRREVERSIBLE)**:
All dispatch paths — implementation, review, and QA — MUST advance KB state before calling `queue.add()`. If KB advance returns `updated: false`, skip enqueue. This prevents double-dispatch on crash recovery.

**State Machine Clarification**:
The seed AC-2 mentions advancing to `'in_review'` which is NOT a canonical state. The correct canonical progression is:
- Review dispatch: `ready_for_qa → in_qa` (advance before enqueuing review job)
- QA dispatch trigger: stories in `in_qa` that need `qa` stage job dispatched

The implementation must verify this against `dispatch-router.ts` behavior before finalizing. The canonical state machine (`packages/backend/orchestrator/src/state/story-state-machine.ts`) is authoritative.

**Thread ID Convention (IRREVERSIBLE ADR APIP-0020)**:
Format `{storyId}:{stage}:{attemptNumber}` — all stages. Review: `PIPE-099:review:1`. QA: `PIPE-099:qa:1`.

**Shared Slot Ceiling**:
`maxConcurrent` applies to total BullMQ in-flight count (active + waiting). All three dispatch stages (implementation, review, QA) share this ceiling. Slots consumed by implementation dispatch are subtracted from slots available for review/QA dispatch in the same `runOnce()` cycle.

**worktreePath Default**:
`ReviewPayload.worktreePath` and `featureDir` are optional in the schema. Pass `''` (empty string). `dispatch-router.ts` already defaults to `''` / `'plans/future/platform'` when these are absent. The scheduler is NOT responsible for worktree allocation.

**Supervisor Pattern**:
The scheduler only enqueues. `dispatch-router.ts` handles stage routing and graph execution. The scheduler has no direct import of or dependency on the supervisor.
