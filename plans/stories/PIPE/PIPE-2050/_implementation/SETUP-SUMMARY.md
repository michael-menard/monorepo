## Key Implementation Notes

### Story Overview
The SchedulerLoop currently polls for `ready` stories and dispatches them as `implementation` stage jobs. After implementation completes and a code review is submitted, the story advances to `ready_for_qa`. However, there is no mechanism to dispatch `review` stage jobs, and later no mechanism to dispatch `qa` stage jobs. The pipeline stalls after implementation.

This story extends SchedulerLoop to add:
1. Review dispatch branch: polls `ready_for_qa`, advances to `in_qa`, enqueues review stage job
2. QA dispatch branch: polls `in_qa`, advances state, enqueues qa stage job
3. Multi-stage dispatch in a single cycle with concurrent limits

### Critical Patterns
- **F006 Contract**: KB state MUST advance BEFORE BullMQ enqueue (atomic dispatch)
- **Thread ID Format**: `{storyId}:{stage}:{attemptNumber}` (e.g., `PIPE-099:review:1`)
- **Finish-Before-New-Start**: Must apply to all dispatch branches equally
- **SQL Pattern**: Use existing Drizzle CTE pattern from `getEligibleStories()`
