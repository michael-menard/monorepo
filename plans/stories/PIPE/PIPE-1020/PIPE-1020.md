# PIPE-1020 — Link APIP Stories to autonomous-pipeline Plan

## Context

The `autonomous-pipeline` plan exists in the KB with `status: implemented` and `storyPrefix: APIP`.
As of 2026-03-19, a `kb_list_stories(plan_slug: 'autonomous-pipeline')` query may return up to 43
linked stories — however, this depends on whether all APIP stories were imported with a `plan_slug`
parameter at creation time (which would have auto-created `spawned_from` links). If any APIP stories
were bulk-imported by PIPE-1010 without a `plan_slug` parameter, those stories exist in the KB
`stories` table but have no corresponding `plan_story_links` row. This creates an invisible gap:
schedulers querying `kb_list_stories(plan_slug: 'autonomous-pipeline')` will miss those orphaned stories.

PIPE-1020 is a verification-then-link chore: audit whether all APIP stories in KB are properly
connected to the `autonomous-pipeline` plan, and insert any missing `plan_story_links` rows using
the established `ON CONFLICT DO NOTHING` idempotency pattern.

**Dependency**: PIPE-1020 must not execute before PIPE-1010 (`Bulk Import APIP Stories from Worktree
to KB`) is in `completed` state. The `plan_story_links` table has a foreign key on `story_id`
referencing the `stories` table — inserts will fail for any APIP story that does not yet exist in KB.

---

## Goal

Ensure every APIP story in the KB has a `plan_story_links` row connecting it to the
`autonomous-pipeline` plan, so schedulers and plan queries return a complete and accurate story set.

---

## Examples

**Given** PIPE-1010 is completed and 43 APIP stories exist in KB, but `plan_story_links` contains
0 rows for `plan_slug='autonomous-pipeline'`, **the system produces** 43 newly inserted rows with
`link_type='spawned_from'`, and a post-execution count report shows `newly_linked=43, gap_resolved=true`.

**Given** `plan_story_links` already contains 43 rows linking all APIP stories to `autonomous-pipeline`
(no gap), **the system produces** 0 new rows inserted, and the count report shows
`already_linked=43, newly_linked=0, gap_resolved=true` — a successful idempotency (verification-only) pass.

**Given** a partial gap where 30 of 43 APIP stories are linked, **the system produces** exactly 13
new rows inserted using `ON CONFLICT (plan_slug, story_id) DO NOTHING`, and the post-execution
count reflects `newly_linked=13, already_linked=30, gap_resolved=true`.

---

## Edge Cases

**Idempotent re-run**: If the linking operation is run a second time after a successful first run,
the `ON CONFLICT (plan_slug, story_id) DO NOTHING` constraint ensures 0 rows are inserted. The
operation is safe to re-run at any time.

**Cancelled story inclusion (APIP-1030)**: APIP-1030 is in `cancelled` state. Default recommendation:
include it in the plan link set. Plan links record plan participation and history, not just active
story status. A cancelled story was still generated as part of the autonomous-pipeline plan's
lifecycle and should appear in `plan_story_links`.

**Premature execution (PIPE-1010 not complete)**: If APIP stories don't exist in the `stories`
table, the `story_id` FK constraint on `plan_story_links` will reject every insert. The executor
must run a pre-condition check (`kb_list_stories(feature: 'platform')` filtered to APIP prefix) and
abort with a clear message if APIP stories are not found. Do not attempt inserts against missing rows.

**Already fully linked (AC-3 idempotency path)**: If the audit finds gap_count = 0, the story
completes as a verification-only pass with no writes. This is a valid success outcome — the pre-existing
43 links may already be complete.

## Architecture Notes

- PIPE-1020 is a **data management story** — no implementation code is written.
- All work happens via KB MCP queries and direct SQL (or migration-script pattern).
- `plan_story_links` has a unique constraint on `(plan_slug, story_id)` — always use `ON CONFLICT DO NOTHING`.
- Use `link_type = 'spawned_from'` (not `'mentioned'`) — APIP stories were generated as part of the
  autonomous-pipeline plan's lifecycle, matching the semantic intent of `spawned_from`.
- The `migrate-plans-to-kb.ts` script uses `'mentioned'` — PIPE-1020 intentionally differs because
  the APIP stories were plan-spawned, not merely referenced in plan documentation.
- The executor should prefer a bulk SQL approach over 43 individual `kb_create_story` calls — each
  MCP call is a network round-trip and the bulk pattern is established in the migration script.
