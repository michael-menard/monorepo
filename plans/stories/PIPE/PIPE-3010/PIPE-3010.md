# PIPE-3010: Wire Execute Node to LLM Invocation in dev-implement.ts

## Context

`dev-implement.ts` is the LangGraph graph that drives autonomous story implementation. It has a complete graph topology (`initialize → load_plan → execute → review_subgraph → collect_evidence → save_to_db → complete`) and a working state annotation that includes model escalation fields (`modelTier`, `iterationCount`, `maxIterations`, `shouldEscalate`).

However, `createExecuteNode` is a one-line stub that sets `executeComplete: true` and emits a `WINT-9070` warning. The change-loop node (`createChangeLoopNode`) is fully implemented in `nodes/change-loop.ts` and handles per-ChangeSpec dispatch → code-gen → micro-verify → atomic git commit via an injectable `IModelDispatch`. The graph is fully wired but the execute node does none of the actual work.

Every story routed through `dev-implement.ts` currently produces no code changes. The stub execute node immediately marks execution complete with no file changes, no model calls, and no real evidence. The stub `collect_evidence` node saves a fake proof with path `"stub"` that will mislead the review phase downstream.

**Active Sibling Work:**
- PIPE-3020: Worktree Isolation for `elab-story.ts` — strictly separate scope, do NOT touch `elab-story.ts`
- PIPE-2010: Unified BullMQ Job Payload Schema — `worktreePath` in BullMQ payload becomes real post-PIPE-3010

---

## Goal

Replace the stub `createExecuteNode` in `dev-implement.ts` with a real implementation that invokes `IModelDispatch` for each `ChangeSpec` from the loaded plan, delegates the per-spec loop to `createChangeLoopNode`, tracks `completedChanges` in state, drives Sonnet→Opus→blocked escalation via `shouldEscalate`, and replaces the stub `collect_evidence` node with real evidence built from `completedChanges`.

---

## Examples

**Given** a `DevImplementConfig` with an injectable `IModelDispatch` mock and a `planContent` containing one `ChangeSpec` targeting `packages/backend/orchestrator/src/graphs/dev-implement.ts`, **the system** calls `modelDispatch.dispatch` once, delegates the file-write/micro-verify/commit loop to `createChangeLoopNode`, records one `CommitRecord` in `state.completedChanges`, and returns `executeComplete: true`.

**Given** the `collect_evidence` node runs after a successful execute with one `CommitRecord` in `completedChanges`, **the system** calls `workflowRepo.saveProof` with an `Evidence` record whose `touchedFiles` array contains the real file path from the commit record — no `"stub"` path, no `"WINT-9080"` reference.

---

## Edge Cases

**Given** a `DevImplementConfig` with no `modelDispatch` injected, **the execute node** returns `executeComplete: true` with an empty `completedChanges` array and emits a warning — it does not throw or crash the graph. This graceful degradation ensures the graph remains functional during staged rollout.

**Given** `state.iterationCount >= state.maxIterations` when `afterExecute` is evaluated, **the conditional edge** routes to `complete` with `workflowSuccess: false` (abort-to-blocked) instead of proceeding to `review_subgraph` or `collect_evidence`. The story is marked blocked rather than silently succeeding.

**Given** `planContent` is present but cannot be parsed as a `ChangeSpec[]` (invalid schema, null, unknown format), **the execute node** sets `changeSpecs = []`, emits a warning about the parse failure, and returns `executeComplete: true` with empty `completedChanges` — it does not throw.

## Architecture Notes

- **Injection pattern**: `DevImplementConfig.modelDispatch` mirrors `ImplementationGraphConfig.modelDispatch` — same Zod-custom pattern. Do NOT hard-wire ModelRouter calls inside the execute node.
- **State extension pattern**: Follow `ImplementationGraphStateAnnotation` in `implementation.ts` for field names and Annotation types. Use `overwrite` reducer for all new fields.
- **Escalation wiring**: `shouldEscalate` returns `'proceed' | 'escalate_to_opus' | 'abort_to_blocked'`. Execute loop updates `iterationCount` and `modelTier` between retries. `afterExecute` must handle `abort_to_blocked` → route to `complete` with `workflowSuccess: false`.
- **planContent parsing**: `planContent` in state is typed as `unknown`. Extract `ChangeSpec[]` using `z.array(ChangeSpecSchema).safeParse()`. On failure, `changeSpecs = []` + warn.
- **worktreePath derivation**: Use `path.join(config.worktreePath, storyId)` for the story-specific path (consistent with elab-story.ts pattern). This story does NOT create/delete worktrees — it only reads the path.
- **No barrel files** (CLAUDE.md rule).
- **Structured log events**: emit `execute_started`, `execute_complete`, `execute_escalate` with `storyId`, `durationMs`, `modelTier`.
- **ARCH-001**: All subprocess invocations use `child_process.spawn` wrapped in Promise — already enforced inside `change-loop.ts`. Execute node does not call subprocesses directly.
- **ARCH-002**: All conditional edge functions exported for test access.
