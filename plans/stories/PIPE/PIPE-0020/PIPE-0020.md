# PIPE-0020: Ghost State Data Migration — Verify and Record Audit Trail for 1001 Ghost-State UPDATEs

## Context

Migration 1001 (`1001_canonical_story_states.sql`) established the canonical 13-state story lifecycle model. As part of its Section 2, it ran bare `UPDATE workflow.stories SET state = '...' WHERE state = '...'` statements to migrate 8 ghost states to their canonical equivalents:

| Ghost State        | Canonical Equivalent |
| ------------------ | -------------------- |
| `uat`              | `completed`          |
| `done`             | `completed`          |
| `in_review`        | `needs_code_review`  |
| `ready_for_review` | `needs_code_review`  |
| `deferred`         | `cancelled`          |
| `ready_to_work`    | `ready`              |
| `elaboration`      | `elab`               |
| `draft`            | `backlog`            |

These UPDATEs ran **before** the state history recording trigger (`record_state_transition`) was created in Section 5 of the same migration. This means every story that was in a ghost state at the time migration 1001 ran has **no corresponding audit row** in `workflow.story_state_history`. The audit trail for the original mass migration is missing.

Additionally, migration 1001's `record_state_transition()` trigger uses `event_type = 'state_changed'` (with trailing 'd'), while the `story_state_history` table's check constraint only permits: `state_change`, `transition`, `phase_change`, `assignment`, `blocker`, `metadata_version`. This is a pre-existing defect in 1001's trigger that PIPE-0020 must not replicate.

PIPE-0010 (completed) fixed the TypeScript/Zod enum model so that ghost state values are no longer expressible in application code. PIPE-0020 is the database complement: ensuring no ghost-state rows remain in `workflow.stories` and backfilling the missing audit trail in `story_state_history`.

PIPE-1010 (downstream) depends on all stories being in canonical states before artifact gate enforcement is added.

---

## Goal

Deliver an idempotent SQL migration that eliminates any remaining ghost-state rows in `workflow.stories` and backfills the missing `story_state_history` audit records for all ghost-state transitions that migration 1001 performed without history recording.

## Edge Cases

**Edge Case 1 — enforce_state_transition trigger blocks direct ghost-state UPDATEs:**

Ghost state values (e.g. `uat`, `ready_to_work`) are not in the trigger's valid transition table. A direct `UPDATE workflow.stories SET state = 'completed' WHERE state = 'uat'` will raise `EXCEPTION 'Invalid state transition: uat → completed'`. The migration must disable the `enforce_state_transition` trigger for the scope of its UPDATEs (within the same transaction), then re-enable it after.

**Edge Case 2 — Duplicate history rows on second migration run:**

If the migration runs twice (e.g. re-deployed in a rollback-and-retry scenario), the history INSERT must not create duplicate rows. The INSERT must use `ON CONFLICT DO NOTHING` or a pre-flight `NOT EXISTS` guard to ensure idempotency. The conflict target must be an existing unique constraint or a WHERE NOT EXISTS subquery scoped to `metadata->>'migrated_by' = 'PIPE-0020'` for the given story.

**Edge Case 3 — event_type constraint enforcement on existing rows:**

The `story_state_history` check constraint on `event_type` was added after migration 1001 ran. PostgreSQL does not retroactively enforce check constraints against existing rows. This means existing rows with `event_type = 'state_changed'` (inserted by the 1001 trigger) do not violate the constraint today. However, any new INSERT (including PIPE-0020's backfill inserts) IS subject to the constraint. PIPE-0020 must use `'state_change'` (no trailing 'd') for all new inserts or it will fail with a check constraint violation.

## Architecture Notes

### Trigger Bypass Strategy

Migration 1001's `enforce_state_transition` trigger will block direct UPDATEs of ghost-state rows because ghost values (e.g. `uat`, `ready_to_work`) are not recognized as valid `from_state` values in the trigger's conditional logic. The migration must wrap its UPDATEs with trigger disable/enable within the same transaction:

```sql
BEGIN;
ALTER TABLE workflow.stories DISABLE TRIGGER enforce_state_transition;
-- ghost-state UPDATEs here
UPDATE workflow.stories SET state = 'completed' WHERE state = 'uat';
-- ... other ghost mappings ...
ALTER TABLE workflow.stories ENABLE TRIGGER enforce_state_transition;
COMMIT;
```

This is safe: the trigger exists to enforce forward transitions on live application writes. The migration is a one-time correction operation.

### event_type Constraint Compliance

The `story_state_history` table check constraint (from `workflow.ts` schema):

- Allowed values: `state_change`, `transition`, `phase_change`, `assignment`, `blocker`, `metadata_version`
- NOT allowed: `state_changed` (trailing 'd') — this is what migration 1001's trigger uses (pre-existing defect)

All PIPE-0020 inserts MUST use `event_type = 'state_change'`.

### Metadata JSONB Structure

All PIPE-0020 audit INSERT rows must include a `metadata` JSONB object to distinguish migration-sourced rows from organic state transitions:

```json
{
  "migrated_by": "PIPE-0020",
  "migration_date": "<NOW()::text>",
  "original_state": "<ghost_state_value>"
}
```

This makes migration-sourced rows queryable via `metadata->>'migrated_by' = 'PIPE-0020'`.

### Findings to Document (AC-10)

**Finding: event_type constraint mismatch — migrations 1001 AND 1003 both use 'state_changed'**

- Migration 1001's `record_state_transition()` function uses: `event_type = 'state_changed'` (trailing 'd')
- Migration 1003's `story_notify` function (line 13) also uses: `event_type = 'state_changed'` (trailing 'd')
- `story_state_history` check constraint requires: `event_type = ANY (ARRAY['state_change','transition','phase_change','assignment','blocker','metadata_version'])`
- Implication: both the 1001 trigger and the 1003 notify function insert with an invalid value per the current constraint. Since the constraint was likely added after those migrations ran (or was added as NOT VALID), existing rows with `'state_changed'` are not retroactively invalidated.
- Recommendation: Create a follow-on story to fix BOTH `record_state_transition()` (migration 1001) AND the `story_notify` function (migration 1003) to use `'state_change'`, and optionally backfill existing `'state_changed'` rows. Do not fix either inline in PIPE-0020 unless dev-feasibility review explicitly approves the scope expansion.
