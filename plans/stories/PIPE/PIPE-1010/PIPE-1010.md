# PIPE-1010: Bulk Import APIP Stories from Worktree to KB

## Context

PIPE-0010 established a clean 13-state canonical `StoryStateSchema` in the DB. PIPE-0020 migrated all existing ghost-state rows (uat, ready_to_work, in_review) to valid canonical values. The KB is now ready to receive new story records.

40+ APIP (autonomous-pipeline) stories exist exclusively in the `tree/story-cdbn-2011` git worktree under `plans/future/platform/autonomous-pipeline/`. Zero of these stories exist in the KB. PIPE-1020 (link APIP stories to the autonomous-pipeline plan) and PIPE-1040 (validate APIP-1031/1032 code on main) both depend on APIP stories being present in the KB. The broader pipeline-orchestrator-activation plan cannot proceed until this import is complete.

This story is a pure data migration executed by an agent calling `kb_create_story` in a loop — no application code changes required.

## Goal

Import all 41 distinct APIP stories from the autonomous-pipeline worktree into the KB as canonical story records, with ghost states mapped to valid StoryStateSchema values, so that downstream PIPE stories can query and link against the full APIP corpus.

## Examples

**Example 1 — UAT-stage story import:**
Given APIP-0010 exists in the `UAT/` stage directory of the worktree with title "BullMQ Work Queue Setup", calling `kb_create_story({ story_id: "APIP-0010", state: "completed", title: "BullMQ Work Queue Setup", feature: "apip", epic: "autonomous-pipeline" })` succeeds and `kb_list_stories({ feature: "apip" })` subsequently returns a record for APIP-0010 with `state: "completed"`.

**Example 2 — Ghost state mapping applied before insert:**
Given APIP-6003 is in the `ready-to-work/` stage directory (worktree status string: `ready-to-work`), the mapping table converts this to `ready` before the `kb_create_story` call. The call succeeds and APIP-6003 is stored with `state: "ready"` — not the ghost value `ready_to_work`.

**Example 3 — Idempotent re-run:**
Given the full import loop has already run (41 stories in KB), re-running the identical loop of 41 `kb_create_story` calls produces zero errors (upsert semantics prevent duplicates) and `kb_list_stories({ feature: "apip", limit: 100 })` still returns exactly 41 APIP stories.

## Edge Cases

**Edge Case 1 — Story exists in both `stories/` archive and a stage directory:**
APIP-0010 has a copy in `stories/APIP-0010/` (status: uat in story.yaml) AND in `UAT/APIP-0010/`. Stage directory position is authoritative. The import uses state `completed` (from UAT/ position), not the raw `uat` string from story.yaml. This prevents ghost state values from sneaking in via the story archive copies.

**Edge Case 2 — Story without story.yaml:**
APIP-0020 has no `story.yaml` in the worktree. Its metadata must be sourced from `.md` frontmatter (`PROOF-APIP-0020.md` or `APIP-0020.md`). If no .md file provides a title, a fallback title is inferred from the directory name. State is `completed` (derived from proof artifacts, as documented in the seed inventory table).

**Edge Case 3 — APIP-1010 and APIP-1030 must NOT be imported:**
These two story IDs were superseded (APIP-1010 by APIP-1020; APIP-1030 split into APIP-1031/1032) and have no directory in the worktree. The import loop skips them entirely. If referenced in a plan index, they are treated as obsolete and excluded.

## Architecture Notes

This story produces no application code changes. It is an agent-driven operational procedure:

1. Read state mapping table from seed (pre-computed, verified against StoryStateSchema).
2. For each of the 41 stories in the inventory table: apply mapping, call `kb_create_story`.
3. Post-import: call `kb_list_stories({ feature: "apip", limit: 100 })` and assert total >= 41.
4. Spot-check 5 stories (one per non-completed state) to verify state values.
5. Re-run full loop to confirm idempotency.

**State determination hierarchy** (authoritative → fallback):
`Stage directory position > story.yaml status field > .md frontmatter status > inferred default`

**Key constraint**: `kb_create_story` uses partial-merge upsert semantics by `story_id`. If a story already exists, fields that are already set are NOT overwritten. Re-runs are safe.
