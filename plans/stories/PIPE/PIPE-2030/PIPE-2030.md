# PIPE-2030: Fix F004 KB Callback Bugs: Invalid `qa` State Mapping and Verify `failed` State Transition

## Context

Story PIPE-2030 was originally written to wire BullMQ `worker.on('completed')` and `worker.on('failed')` callbacks to advance KB story state via `kb_update_story_status`. This wiring was implemented as gap fix F004 in commit `5a3341f0` (2026-03-16). The F004 implementation is present and functional in `apps/api/pipeline/src/supervisor/index.ts`.

**This story is a bug fix**, not a new implementation. The original description is stale. Two correctness defects were discovered in the F004 implementation:

1. `STAGE_TO_NEXT_STATE['qa'] = 'UAT'` — `'UAT'` is not a valid `StoryStateSchema` state. Zod validation in `kb_update_story_status` rejects this silently (caught by `.catch()` — `logger.warn`). QA-completed stories never advance their state in the KB.

2. `worker.on('failed')` on final failure sets only `blocked_reason` without explicitly setting `state: 'blocked'`. The comment in the code reads "Set story to blocked in KB on final failure" but the `state` field is missing from the call, meaning no state transition occurs.

## Goal

Fix the two defects in the F004 BullMQ KB callback implementation so that all stage completion events correctly advance KB story state, and final job failures correctly set stories to the `blocked` state.

## Examples

**Example 1 — QA stage completion (the primary bug):**

Given a BullMQ job completes with `data = { storyId: "PIPE-2030", stage: "qa" }`, the supervisor's `STAGE_TO_NEXT_STATE` map returns `'completed'` (not `'UAT'`), and `kb_update_story_status` is called with `{ story_id: "PIPE-2030", state: "completed" }`. Zod validation passes. The story advances to `completed` state in the KB.

**Example 2 — Final failure (the secondary bug):**

Given a BullMQ job fails with `attemptsMade = 3` and `opts.attempts = 3` (final attempt exhausted), the supervisor calls `kb_update_story_status` with `{ story_id: "PIPE-2030", state: "blocked", blocked_reason: "Agent timeout after 3 attempts" }`. The story transitions to `blocked` state and records the failure reason.

## Edge Cases

**Edge Case 1 — kbDeps is null:**

When `KB_DB_PASSWORD` is not set, `this.kbDeps` is null. Both the `completed` and `failed` handlers skip the `kb_update_story_status` call entirely. No error is thrown. The null-guard condition `if (storyId && stage && this.kbDeps)` prevents the call.

**Edge Case 2 — Transient failure (not final):**

When a job fails with `attemptsMade = 1` and `opts.attempts = 3`, `isFinalFailure` is false. The `kb_update_story_status` call is NOT made. Only final failures (all retry attempts exhausted) trigger the KB state update.

**Edge Case 3 — Unknown stage:**

When `job.data.stage` does not appear in `STAGE_TO_NEXT_STATE` (e.g., `"unknown_stage"`), `nextState` is undefined. The guard condition `if (nextState)` prevents the KB call. No error is thrown.

## Acceptance Criteria

- [ ] **AC-1**: `STAGE_TO_NEXT_STATE` in `apps/api/pipeline/src/supervisor/index.ts` maps all four stages (`elaboration`, `implementation`, `review`, `qa`) to valid `StoryStateSchema` states. Specifically, `qa` must NOT map to `'UAT'`. Valid mapping: `{ elaboration: 'ready', implementation: 'needs_code_review', review: 'ready_for_qa', qa: 'completed' }`.

- [ ] **AC-2**: All four mapped states (`ready`, `needs_code_review`, `ready_for_qa`, `completed`) can be passed to `kb_update_story_status` without Zod validation errors. The `as any` cast on the state field may be removed once all values are valid enum members.

- [ ] **AC-3**: `worker.on('completed')` calls `kb_update_story_status` with the correct `state` value for each stage, using the fire-and-forget `.catch()` pattern with `logger.warn`.

- [ ] **AC-4**: `worker.on('failed')` on final failure: the call to `kb_update_story_status` includes `state: 'blocked'` in addition to `blocked_reason`. If deliberate omission is chosen instead, a code comment documents the intentional decision with justification.

- [ ] **AC-5**: Unit tests exist for the `completed` callback path: mock `kbDeps`, trigger the event handler, assert `kb_update_story_status` was called with the expected `state` value for at least the `qa` stage (verifying `'completed'`, not `'UAT'`) and the `implementation` stage (verifying `'needs_code_review'`).

- [ ] **AC-6**: Unit tests exist for the `failed` callback path: mock `kbDeps`, simulate final failure (`attemptsMade >= attempts`), assert `kb_update_story_status` was called with `state: 'blocked'` and a non-empty `blocked_reason`. Also verify transient failure (non-final) does NOT call `kb_update_story_status`.

- [ ] **AC-7**: All new and existing supervisor tests pass with `pnpm test --filter @repo/pipeline`.

- [ ] **AC-8**: No `wint.*` schema references are introduced in any modified or created file.

## Architecture Notes

- The PipelineSupervisor is a long-lived Node.js process (not a Lambda). BullMQ event handlers (`worker.on`) must remain non-blocking. All KB calls are fire-and-forget.
- `StoryStateSchema` is the canonical source of truth for valid state strings. The definitive list as of this writing: `backlog`, `created`, `elab`, `ready`, `in_progress`, `needs_code_review`, `ready_for_qa`, `in_qa`, `completed`, `failed_code_review`, `failed_qa`, `blocked`, `cancelled`. `'UAT'` and `'uat'` are NOT valid states.
- The `kbDeps` field is nullable by design — when `KB_DB_PASSWORD` is absent, `kbDeps` is null and all KB callbacks must be skipped. This is already correctly implemented in both handlers.
- The `as any` cast on the state field (`state: nextState as any`) was a workaround for the invalid `'UAT'` value. After fixing the mapping, this cast should be revisited and removed if all values are now valid enum members.
