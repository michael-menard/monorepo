# PIPE-2010: Unified BullMQ Job Payload Schema for Dev, Review, and QA Dispatch

## Context

The `@repo/pipeline-queue` package defines `PipelineJobDataSchema`, a Zod discriminated union that covers all BullMQ pipeline job stages (`elaboration`, `story-creation`, `implementation`, `review`, `qa`). A prior fix (F002) consolidated the supervisor to re-export from this canonical package. However, three residual schema gaps prevent the dispatch router from being fully type-safe across all stages:

1. **Review payload missing worktree fields**: `ReviewJobDataSchema.payload` is typed as `StorySnapshotPayload` (only `storyId`, `title`, `description`, `feature`, `state`). But `dispatch-router.ts` requires `worktreePath` and `featureDir` to invoke `runReview()` — these are added via an unsafe `as StorySnapshotPayload & { worktreePath?: string; featureDir?: string }` type cast at lines 228–235.

2. **Local parallel payload stubs**: `SynthesizedStoryPayloadSchema` and `StoryRequestPayloadSchema` exist in `apps/api/pipeline/src/supervisor/__types__/index.ts` as local informal documentation of elaboration/story-creation payloads. Their relationship to the canonical `@repo/pipeline-queue` types must be explicitly documented (kept as local documentation, not duplicated).

3. **QA payload shape unverified**: `QaJobDataSchema.payload` currently uses `StorySnapshotPayload`. The actual `RunQAVerifyFn` signature confirms QA only requires `storyId` and `attempt` — the payload shape needs to be explicitly documented as `QaPayloadSchema` even if the shape matches the existing snapshot.

**Sequencing constraint**: PIPE-2020 (Dispatch Router: dev/review/QA branches) directly consumes `ReviewJobDataSchema` and `QaJobDataSchema`. This story MUST be merged before PIPE-2020 implementation begins to avoid the dispatch router working around the same type-cast pattern.

**Relevant files (ground truth from codebase scan)**:

- `packages/backend/pipeline-queue/src/__types__/index.ts` — canonical schema (primary edit target)
- `apps/api/pipeline/src/supervisor/dispatch-router.ts` lines 227–236 — unsafe cast site
- `packages/backend/orchestrator/src/nodes/review/types.ts` — `ReviewWorkerInputSchema` confirms `worktreePath: z.string().min(1)` and `featureDir: z.string().min(1)` are required by the review graph
- `apps/api/pipeline/src/supervisor/__types__/index.ts` — local payload stubs + re-export comment

---

## Goal

Extend `PipelineJobDataSchema` in `@repo/pipeline-queue` to formalize the review and QA payload shapes — including the `worktreePath` and `featureDir` fields needed by the review graph — and eliminate the unsafe type cast in `dispatch-router.ts`, making all five pipeline stage dispatch paths fully type-safe at the Zod schema boundary.

## Examples

**Example 1: Valid review job with worktree context**

Given a BullMQ job enqueued with:

```json
{
  "storyId": "PIPE-2020",
  "stage": "review",
  "attemptNumber": 1,
  "payload": {
    "storyId": "PIPE-2020",
    "title": "Dispatch Router Review Branch",
    "worktreePath": "/tmp/worktrees/PIPE-2020",
    "featureDir": "plans/future/platform/pipeline-orchestrator-activation"
  }
}
```

The system produces:

- `PipelineJobDataSchema.parse()` succeeds
- `result.payload.worktreePath === '/tmp/worktrees/PIPE-2020'`
- `dispatch-router.ts` passes `worktreePath` to `runReview()` without any type cast

**Example 2: Valid review job without optional worktree fields**

Given a BullMQ job enqueued with:

```json
{
  "storyId": "PIPE-2020",
  "stage": "review",
  "attemptNumber": 1,
  "payload": {
    "storyId": "PIPE-2020",
    "title": "Dispatch Router Review Branch"
  }
}
```

The system produces:

- `PipelineJobDataSchema.parse()` succeeds
- `result.payload.worktreePath === undefined`
- `dispatch-router.ts` falls back to `worktreePath ?? ''` (existing behavior preserved)

## Edge Cases

**Edge Case 1: Missing required payload.storyId in review job**

Given:

```json
{
  "storyId": "PIPE-2020",
  "stage": "review",
  "attemptNumber": 1,
  "payload": { "title": "Missing storyId" }
}
```

The system produces:

- `PipelineJobDataSchema.parse()` throws `ZodError`
- Error path includes `payload.storyId`
- `createPipelineQueue().add()` rejects the job before it reaches BullMQ

**Edge Case 2: worktreePath provided as non-string**

Given:

```json
{
  "storyId": "PIPE-2020",
  "stage": "review",
  "attemptNumber": 1,
  "payload": { "storyId": "PIPE-2020", "title": "T", "worktreePath": 123 }
}
```

The system produces:

- `PipelineJobDataSchema.parse()` throws `ZodError`
- Error path includes `payload.worktreePath`

## Non-Goals

- Do not change the `stage` discriminant values (`elaboration`, `story-creation`, `implementation`, `review`, `qa`) — these are serialized in BullMQ jobs and encoded in the thread ID convention `{storyId}:{stage}:{attemptNumber}` (IRREVERSIBLE ADR).
- Do not implement the dispatch router itself — that is PIPE-2020.
- Do not implement scheduler dispatch for review/QA stages — that is PIPE-2050.
- Do not change `ElaborationJobDataSchema.payload` or `StoryCreationJobDataSchema.payload` from `z.record(z.unknown())` — these stages pass complex orchestrator graph input objects; the opaque type is intentional until ORCH stories formalize those boundaries.
- Do not modify the `PIPELINE_QUEUE_NAME` constant or connection factory.
- Do not add new BullMQ stage types beyond the existing five.
- Do not remove `SynthesizedStoryPayloadSchema` or `StoryRequestPayloadSchema` from supervisor `__types__` — they serve as local documentation of elaboration/story-creation payload expectations and are not duplicating canonical pipeline-queue types.

## Architecture Notes

### ReviewPayloadSchema Design

```typescript
// Proposed addition to packages/backend/pipeline-queue/src/__types__/index.ts

export const ReviewPayloadSchema = StorySnapshotPayloadSchema.extend({
  // Required by runReview() in dispatch-router; optional in BullMQ payload
  // because the scheduler may enqueue before worktree is allocated.
  // dispatch-router defaults to '' if absent.
  worktreePath: z.string().optional(),
  featureDir: z.string().optional(),
})

export type ReviewPayload = z.infer<typeof ReviewPayloadSchema>
```

### QaPayloadSchema Design

```typescript
// RunQAVerifyFn only needs storyId + attempt (from job-level fields, not payload).
// Payload is StorySnapshotPayload for context. Define as explicit alias.
export const QaPayloadSchema = StorySnapshotPayloadSchema

export type QaPayload = z.infer<typeof QaPayloadSchema>
```

### Dispatch-Router Fix

Before (unsafe):

```typescript
const payload = data.payload as StorySnapshotPayload & {
  worktreePath?: string
  featureDir?: string
}
graphPromise = graphRunners.runReview({
  storyId: data.storyId,
  worktreePath: payload.worktreePath ?? '',
  featureDir: payload.featureDir ?? 'plans/future/platform',
  attempt: data.attemptNumber,
})
```

After (typed):

```typescript
// data.payload is now typed as ReviewPayload — no cast needed
graphPromise = graphRunners.runReview({
  storyId: data.storyId,
  worktreePath: data.payload.worktreePath ?? '',
  featureDir: data.payload.featureDir ?? 'plans/future/platform',
  attempt: data.attemptNumber,
})
```

### passthrough() Note

`StorySnapshotPayloadSchema` uses `.passthrough()` so extra fields are not rejected. `ReviewPayloadSchema.extend()` inherits this behavior — `worktreePath`/`featureDir` are promoted from "pass-through extras" to "explicitly typed optional fields." Do NOT call `.strict()` on `ReviewPayloadSchema` — it would break existing BullMQ jobs that include extra fields.

### Local Payload Stubs in Supervisor **types**

`SynthesizedStoryPayloadSchema` and `StoryRequestPayloadSchema` in `apps/api/pipeline/src/supervisor/__types__/index.ts` are retained as local documentation. They document the expected shapes for `elaboration` and `story-creation` payloads respectively, and are used by dispatch-router.ts casts for those stages. They are NOT moved to `@repo/pipeline-queue` because the elaboration/story-creation stages intentionally use `z.record(z.unknown())` in the canonical schema until ORCH stories formalize those graph input contracts.
