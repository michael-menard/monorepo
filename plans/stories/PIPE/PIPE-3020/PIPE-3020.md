# PIPE-3020 — Worktree Isolation for Graph Execution: Wire Real Git Worktree Lifecycle in elab-story.ts

## Context

`elab-story.ts` already has the full worktree lifecycle structure in place — `worktree_setup` and `worktree_teardown` nodes exist as named nodes in the compiled graph, and the conditional edge logic guarantees teardown always runs (AC-3). However, both nodes are stubs:

- `createWorktreeSetupNode` (lines ~202–215) assigns a fake path (`${baseDir}/${storyId}`) and emits a warning referencing `WINT-9060`
- `createWorktreeTeardownNode` (lines ~289–304) is a no-op that emits the same `WINT-9060` stub warning

The real git worktree creation logic already exists in `packages/backend/orchestrator/src/nodes/create-worktree.ts` (from APIP-1031), and the real worktree registration tool exists in `packages/backend/mcp-tools/src/worktree-management/worktree-register.ts` (from WINT-1130). Every graph execution that runs through `elab-story.ts` currently gets a fake worktree path — any node that attempts filesystem access at `worktreePath` fails silently or produces incorrect results, and the `workflow.worktrees` table has no record of these executions.

**Predecessor stories**: APIP-1031 (implemented create-worktree node), WINT-1130 (implemented worktree-register MCP tool)
**Coordination**: PIPE-3010 also touches `dev-implement.ts` execute node — this story is scoped strictly to `elab-story.ts`. Do not touch `dev-implement.ts`.

---

## Goal

Replace the stub worktree node factories in `elab-story.ts` with real implementations that invoke `git worktree add` via the injectable gitRunner pattern, register the worktree in the KB database, and clean up properly on teardown so every elaboration graph execution runs in a real, isolated filesystem workspace.

---

## Examples

**Given** an elaboration graph is invoked for story `PIPE-9999` with `worktreeBaseDir: "/tmp/worktrees"`:

- **Setup path**: `createWorktreeSetupNode` calls `git worktree add -b elab/PIPE-9999 /tmp/worktrees/PIPE-9999`, calls `worktreeRegister({ storyId: "PIPE-9999", worktreePath: "/tmp/worktrees/PIPE-9999", branchName: "elab/PIPE-9999" })`, and returns `{ worktreePath: "/tmp/worktrees/PIPE-9999", worktreeSetup: true }`.
- **Teardown path**: `createWorktreeTeardownNode` looks up the worktree record via `worktreeGetByStory({ storyId: "PIPE-9999" })`, calls `git worktree remove --force /tmp/worktrees/PIPE-9999`, calls `worktreeMarkComplete({ worktreeId: "<uuid>", status: "abandoned" })`, and returns `{ worktreeTornDown: true }` without modifying `workflowSuccess`.

---

## Edge Cases

**Registration failure (non-blocking)**: When `worktreeRegister` returns `null` (MCP tool error), `createWorktreeSetupNode` emits a `logger.warn` but continues — `worktreeSetup` remains `true`. The elaboration subgraph can still run; only observability is degraded.

**Git worktree add failure (hard fail)**: When `git worktree add` exits non-zero or throws, `createWorktreeSetupNode` returns `{ worktreeSetup: false, worktreePath: null, errors: [<message>] }`. The graph routes to teardown via `afterElabStoryInitialize` since `workflowComplete` may be set by the initialize node; teardown must not crash when `worktreeSetup === false` (matches existing guard on line 292).

**Teardown with no KB record**: When `worktreeGetByStory` returns null (no record found), teardown skips `worktreeMarkComplete` with a `logger.warn` and returns `{ worktreeTornDown: true }` — teardown never sets `workflowSuccess: false`.

**Teardown git failure**: When `git worktree remove --force` fails, teardown emits a `logger.warn` and returns `{ worktreeTornDown: true }` — matching the `cleanup-worktree.ts` warn-only contract. `workflowSuccess` is NOT changed.

## Acceptance Criteria

- [ ] **AC-1**: `createWorktreeSetupNode` in `elab-story.ts` calls `git worktree add -b elab/{storyId} {worktreePath}` (using injectable `gitRunner`) when invoked. On success, `state.worktreePath` contains the real filesystem path and `state.worktreeSetup === true`.

- [ ] **AC-2**: `createWorktreeSetupNode` calls `worktreeRegister({ storyId, worktreePath, branchName })` after successful git worktree creation. If registration fails (returns null), a warning is emitted but setup is not considered failed — `worktreeSetup` remains true.

- [ ] **AC-3**: `createWorktreeTeardownNode` calls `git worktree remove --force {worktreePath}` on teardown. Errors are emitted as warnings only — teardown does NOT change `workflowSuccess` or `workflowComplete`. `state.worktreeTornDown === true` regardless of teardown errors (matches `cleanup-worktree.ts` contract).

- [ ] **AC-4**: `createWorktreeTeardownNode` calls `worktreeMarkComplete` with `status: 'abandoned'` after git cleanup, using the worktree ID retrieved via `worktreeGetByStory({ storyId })`. If no worktree record is found, teardown skips the mark-complete call with a warning.

- [ ] **AC-5**: If `git worktree add` fails (non-zero exit code or thrown error), `createWorktreeSetupNode` returns `{ worktreeSetup: false, worktreePath: null, errors: [...] }`. The existing graph routing (via `afterElabStoryInitialize`) handles this gracefully — teardown still runs per AC-3.

- [ ] **AC-6**: `worktreeBaseDir` from `ElabStoryConfig` is used to compute the worktree path — hardcoded `/tmp/worktrees` is not used directly. The config schema default (`/tmp/worktrees`) applies when not overridden.

- [ ] **AC-7**: The no-longer-needed `WINT-9060` warning strings are removed from both node factory functions.

- [ ] **AC-8**: Unit tests cover: real `git worktree add` args called with injectable mock (HP), `worktreeSetup: true` on success (HP), `worktreeSetup: false` on git failure (EP), teardown emits warning-only on cleanup error (EP), teardown calls `worktreeMarkComplete` (HP).

- [ ] **AC-9**: Integration test (new `elab-story-worktree.integration.test.ts`) verifies the full setup → teardown lifecycle in an isolated `os.tmpdir()` git repo with an initial commit.

- [ ] **AC-10**: `pnpm test --filter @repo/orchestrator` passes with zero regressions. `pnpm check-types` passes with zero errors.

## Architecture Notes

**Branch naming convention**: Use `elab/{storyId}` prefix (e.g., `elab/PIPE-3020`) to distinguish elaboration worktrees from implementation worktrees (`impl/` prefix used by `dev-implement.ts` and `create-worktree.ts`).

**Injectable gitRunner**: Factory signatures must follow: `createWorktreeSetupNode(opts?: { gitRunner?: GitRunner; repoRoot?: string })`. The `defaultGitRunner` from `create-worktree.ts` is the runtime default; tests inject a `vi.fn()` mock.

**Teardown sequencing**: Teardown should be async/awaited (not fire-and-forget void IIFE). Inside the elab-story graph, teardown is a deliberate unconditional node — the `afterElaborationSubgraph` edge always routes to `worktree_teardown`. The void IIFE pattern in `create-worktree.ts` was designed for opportunistic cleanup outside the graph structure; inside the graph, awaited teardown is correct.

**Registration failure handling**: `worktreeRegister` returns `null` on error (it does not throw). The setup node must check for `null` and emit a warning, not fail setup.

**wint.* avoidance**: `worktreeRegister` uses the KB `worktrees` table via `@repo/mcp-tools`. Do NOT use `WorkflowRepository` or `wint.*` schema for any worktree tracking in this story.

**ADR compliance**:
- ADR (d0b64e4d): Injectable gitRunner over execSync — all git-invoking nodes must use injectable spawn-based gitRunner
- elab-story.ts AC-3: `afterElaborationSubgraph` already routes to `worktree_teardown` unconditionally; teardown implementation must never skip on error paths
